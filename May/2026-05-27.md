#### | HTTP 인증 헤더

HTTP는 **Stateless** 하기 때문에 서버는 클라이언트가 누구인지 기억하지 못한다.
따라서 보호된 리소스(프로필, 결제 정보 등)에 접근할 때는 **매 요청마다 신원을 증명**해야 한다.

이 때 사용하는 것이 **HTTP 인증 헤더**다.

| 헤더 | 방향 | 역할 |
|---|---|---|
| `WWW-Authenticate` | 서버 → 클라이언트 (Response) | "이 리소스는 보호 구역이야. 인증이 필요해." (`401 Unauthorized`) |
| `Authorization` | 클라이언트 → 서버 (Request) | 신분증을 헤더에 담아 제출 |

---

##### 1. Basic Authentication

- 아이디 + 패스워드를 `Base64`로 인코딩해 헤더에 담아 전송
- ⚠️ **암호화가 아니다** — 인코딩은 규칙대로 치환하는 것이며, 디코딩이 가능한 평문

```
Authorization: Basic Aszxperasdf=
```

- 매 요청마다 서버가 DB에서 아이디/패스워드를 직접 조회해 검증 → **서버 자원 부담, 병목 위험**
- 반드시 **HTTPS** 환경에서만 사용
- ✅ 구현이 간단해 토이 프로젝트나 내부용 API에 적합

---

##### 2. Digest Authentication

Basic 인증의 보안 취약점(비밀번호 평문 전송)을 개선한 방식.
비밀번호 대신 **해시값**을 보낸다.

**핵심 개념**

| 용어 | 의미 |
|---|---|
| **Nonce** | 서버가 클라이언트에게 발급하는 일회성 난수 (딱 한 번만 유효) |
| **Realm** | 서버가 관리하는 보안 공간의 범위 (예: `users`, `posts`) |

**인증 흐름**

```
① 클라이언트 → 보호된 리소스 요청
② 서버 → 401 응답 + Nonce, Realm 발급
③ 클라이언트 → (비밀번호 + Nonce + Realm)을 해시로 구워 Authorization 헤더에 담아 전송
④ 서버 → 동일하게 해시 연산 후 비교 → 인증 완료
```

- 보안성은 Basic보다 높지만, 여전히 **매 요청마다 서버가 검증**해야 하는 부담 존재

---

##### 3. Bearer Token ← 현대 웹 표준

기존 방식의 공통 문제 : 매 요청마다 서버가 비밀번호를 직접 검증

> **"이 토큰을 소지한 요청은 인증·인가된 것으로 한다"** = Bearer Token
> 마치 무기명 콘서트 티켓처럼, 티켓(토큰)이 위조되지 않았으면 OK

```
Authorization: Bearer mF_9.B5f-4.1JpM
```

**Bearer Token 인증 흐름**

```
① 최초 로그인 시 (Body에 ID/PW 전송)
② 서버에서 인증 수행
③ 토큰 발급
④ 클라이언트에게 토큰 전달
⑤ 이후 요청 시 토큰을 헤더에 담아 전송 → 서버는 DB 조회 없이 토큰만 검증
```

**장점**
- 비밀번호는 최초 로그인 때 딱 1번만 전송
- 모바일 앱, 다른 도메인 등 범용적으로 동작 (쿠키 기반 세션의 한계 해소)
- CSRF 공격 방어

**단점**
- 토큰이 탈취되면 만료 전까지 무효화 불가
- 토큰 보관 위치가 까다로움

| 저장 위치 | 위험 요소 |
|---|---|
| Storage (localStorage 등) | XSS 공격으로 탈취 가능 |
| Cookie | CSRF 공격에 취약 |

#### | JWT (JSON Web Token)

일반 Bearer 토큰(불투명 토큰)은 의미 없는 무작위 문자열 → 서버가 토큰을 저장하고 조회해야 함
→ 결국 서버 자원 낭비

> 💡 **토큰 자체에 유저 정보가 담겨 있고, 위변조 여부까지 스스로 증명할 수 있다면?**
> → 서버가 DB를 조회할 필요가 없다 → **JWT 탄생**

**JWT란?** 당사자 간에 정보를 JSON 객체로 안전하고 가볍게 전송하기 위한 웹 표준 개방형 규격
마치 **홀로그램 인쇄 티켓**처럼, 내부 내용과 발급자의 서명이 함께 담겨 있다.

---

##### JWT 구조

`.`(점) 으로 세 영역이 구분된다.

```
eyJhbGciOiJIUzI1NiJ9  .  eyJ1c2VySWQiOjEwfQ  .  SflKxwRJSMeKKF2QT4fwpMeJf36P
       Header                   Payload                    Signature
```

**1. Header (헤더)**
```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```
토큰 타입과 서명에 사용된 알고리즘 정보

**2. Payload (페이로드)**
```json
{
  "userId": 10,
  "email": "ffsf@naver.com",
  "name": "Lulu",
  "role": "admin",
  "iat": 1435353000,
  "exp": 1435354800
}
```
- JWT에 담을 실제 정보
- Base64 인코딩 → **평문이나 다름없음**
- ⚠️ 주민등록번호, 카드 번호, 비밀번호 등 민감한 정보는 절대 넣지 말 것

**3. Signature (서명)**

```
HMACSHA256(
  base64(Header) + "." + base64(Payload),
  서버만 아는 비밀키
)
```
- 해커가 Payload를 수정하면 Signature가 달라짐 → 위변조 탐지 가능

---

##### JWT를 쓰는 이유

1. **완전한 무상태성 + 수평 확장성**
   - 토큰 안에 필요한 정보가 모두 담겨 있어 서버가 세션 상태를 유지할 필요 없음
   - 서버를 여러 대로 늘려도(스케일 아웃) Redis 같은 세션 공유 스토리지가 필요 없음

2. **DB 조회 최소화**
   - DB는 하드디스크 기반 → 가장 느린 I/O
   - 모든 요청이 DB 조회로 몰리면 병목 발생 → JWT는 서명 검증만으로 인증 완료

##### JWT의 단점

1. **토큰 무효화가 어렵다**
   - 발급된 토큰은 클라이언트가 보관 → 서버가 임의로 만료 불가
   - 탈취 시 만료 시간까지 계속 사용 가능

2. **네트워크 대역폭 낭비**
   - Payload가 많을수록 토큰 길이 증가 → 매 API 요청마다 수 KB 전송 → 서버 비용 증가

#### | Access Token & Refresh Token

**보안과 편의성의 딜레마**

- Access Token 수명을 15분으로 짧게 → 탈취 위험 감소 But 15분마다 로그인 재요청 → 불편
- 수명을 길게 → 편하지만 탈취 위험 증가

**해결책 : 두 가지 토큰으로 역할 분리**

| 구분 | Access Token | Refresh Token |
|---|---|---|
| **역할** | 실제 API 리소스에 접근하기 위한 출입증 | Access Token을 새로 발급받기 위한 인증서 |
| **수명** | 15분 ~ 30분 (짧게) | 2주 ~ 1달 (길게) |
| **저장 위치** | 프론트엔드 메모리 또는 LocalStorage / HttpOnly 쿠키 | 서버 측 저장소(Redis 등) 또는 클라이언트 HttpOnly 쿠키 |
| **전송 빈도** | 매 API 요청마다 항상 전송 | Access Token 만료 시에만 전송 |

> 🎡 **놀이공원 비유**
> - Access Token → 15분짜리 일회용 탑승권 (주머니에 넣고 다님. 탈취돼도 15분 후 쓸모없음)
> - Refresh Token → 가방에 넣은 연간 회원권 (탑승권 만료 시 새 탑승권 발급용)

#### | RTR (Refresh Token Rotation)

Refresh Token이 탈취되면 장기간 악용 가능 → 이를 방어하기 위한 전략

> **"Refresh Token은 한 번 쓰면 폐기하고, 새것으로 교체한다"**

**정상 흐름**

```
① Access Token(A1) 만료
② 클라이언트가 Refresh Token(R1)으로 재발급 요청
③ 서버: R1 유효 확인 → R1 폐기
④ 새 Access Token(A2) + 새 Refresh Token(R2) 발급
⑤ 서버: R2를 저장소에 저장
```

**해커가 R1을 탈취한 경우**

```
① 해커가 R1으로 재발급 요청 → 서버가 R2 발급, R1 폐기
② 이후 진짜 유저가 R1으로 재발급 요청
③ 서버: "R1이 없는데?" → 비정상 접근으로 판단
④ 해당 유저의 모든 Refresh Token 즉시 폐기
⑤ 해커가 갖고 있던 R2도 무효화 → 공격 차단
```

#### | Layered Architecture (레이어드 아키텍처)

스프링 애플리케이션의 책임을 계층별로 분리한 구조.
**코드 중복 방지 + 책임 분리**가 목적이다.

```
클라이언트
    ↕ 요청/응답
Controller   ← 외부 입출력, API 스펙 처리
    ↕ DTO
Service      ← 비즈니스 로직 처리
    ↕ Entity
Repository   ← DB 접근, CRUD 처리
    ↕ SQL
  Database
```

##### 각 계층의 역할

**Controller**
- 클라이언트 요청/응답 담당
- 계층 간 데이터를 **DTO**로 주고받아 레이어별 필요한 데이터 형태를 분리

**Service**
- 핵심 비즈니스 로직 수행

**Repository**
- DB 접근 및 CRUD 처리
- RDB는 관계 지향, 자바는 객체 지향 → 구조 차이 존재
- 이 간극을 메우기 위해 RDB 테이블 구조를 자바 객체로 표현한 것이 **Entity**

**Entity에 들어가는 것**
- 테이블 정의
- 컬럼 매핑
- 키 정의

**DTO (Data Transfer Object)**
- 계층 간 전달할 데이터 구조를 정의하는 객체
- 레이어별로 필요한 데이터가 다르기 때문에, 동일한 양식으로 전달되도록 보장

**미들웨어 (필터 / 인터셉터)**
- 클라이언트가 보낸 데이터가 올바른지 검증
- 계층 간 전달 데이터 적절성 확인
- JWT 인증 처리에도 활용

### 🗨️ 오늘의 회고

- Basic → Digest → Bearer Token → JWT 순서로 인증 방식이 발전한 흐름을 보니, 각 방식이 왜 나왔고 어떤 한계를 해결하려 했는지 맥락이 명확하게 이해됐다.
- JWT가 "무상태성"을 완성하는 핵심 기술이라는 것, 그리고 그 때문에 토큰 무효화 문제가 생기고 RTR로 이어진다는 흐름이 인상적이었다.
- Layered Architecture는 스프링 MVC에서 배운 Controller/Service/Repository 구조와 이어지는 내용이라 비교적 자연스럽게 이해됐다. DTO와 Entity가 왜 분리되는지도 납득이 됐다.
- 위클리 챌린지에 시간을 더 쏟기 위해 내일부터 일정 조정이 필요하다.
