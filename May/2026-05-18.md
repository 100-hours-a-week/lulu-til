#### | 웹의 구조 및 이해

##### 웹 1.0
- **시기** : 1990년대 초반
- **핵심 키워드** : 정보 제공, 읽기 전용, 단방향 소통
- **특징**
  - 웹 사이트의 목적이 오직 정보 제공
  - 사용자는 직접 내용을 추가하거나 변경할 수 없음

##### 웹 2.0
- **시기** : 2000년대 초반 ~ 현재
- **핵심 키워드** : 사용자 참여, 공유, 양방향 소통, 사용자 제작 콘텐츠(UGC)
- **특징**
  - 사용자가 단순히 정보를 받아보는 것에서 벗어나 직접 정보를 제작하고 공유
  - 블로그, 싸이월드, 페이스북, 인스타그램, 유튜브 등이 대표적 사례

##### 웹 3.0
- **시기** : 아직 본격적으로 도래하지 않음
- **핵심 키워드** : 개인화, 신뢰성, 지능화, 탈중앙화
- **특징**
  - **맞춤형 콘텐츠** : AI(유튜브 추천, 넷플릭스 추천), 머신러닝 기반의 개인화 서비스
  - **탈중앙화** : 데이터가 중앙 집중식 서버가 아닌 다수의 컴퓨터에 분산 저장 → 개인이 자신의 데이터를 직접 관리 가능, 보안성 향상
  - **경량 AI 모델** : 모바일 디바이스에서 구동 가능한 경량화 추세 → 개인 데이터 주권 강화
  - **보안 강화** : 웹을 통해 금융, 자산, 개인정보까지 처리하는 시대 → 보안의 중요성 증대

> 💡 **개발자로서 흐름을 읽어야 하는 이유**
> 블록체인 → SaaS → 1인 유니콘(바이브 코딩)처럼 기술 트렌드는 계속 변한다.

#### | 클라이언트-서버 구조

> 🍽️ **레스토랑 비유**
> - 클라이언트 = 손님 (음식을 요청하고 소비하는 주체)
> - 서버 = 주방장/식당 (음식을 만들어 제공하는 주체)
> - 네트워크/프로토콜 = 메뉴판 (소통의 규칙)

##### 클라이언트 (Client)
- **정의** : 서비스를 요청하는 단말기 또는 소프트웨어. 사용자와 맞닿아 있어 **유저 에이전트(User Agent)** 라고도 부름
- **핵심 역할**
  - **요청의 시작점** : 네트워크 통신은 항상 클라이언트가 먼저 요청(Request)을 보냄
  - **사용자 인터페이스 구현** : 서버로부터 전달받은 HTML, CSS, JS 데이터를 브라우저를 통해 화면에 렌더링
  - **컴퓨팅 자원 사용** : 애니메이션, 버튼 클릭, 이벤트 처리 등 클라이언트 측 연산은 사용자 기기의 CPU와 메모리를 활용
- **실제 기술 예시** : 크롬·사파리·엣지 등 웹 브라우저, 카카오톡·인스타그램 앱, Postman 등 API 테스트 도구

##### 서버 (Server)
- **정의** : 네트워크를 통해 클라이언트에게 서비스나 자원을 제공하는 컴퓨터 시스템 또는 프로그램
- **특징** : 클라이언트의 요청이 언제 올지 모르기 때문에 24시간 특정 포트를 열어두고 대기
- **핵심 역할**
  - **수동적 대기 & 응답** : 스스로 통신을 시작하지 않고, 요청이 오면 비즈니스 로직을 수행 후 결과 반환
  - **비즈니스 로직 처리** : 회원가입 시 이메일 중복 체크처럼 서비스의 핵심 로직 처리 (회사의 대외비)
  - **중앙 집중식 자원 관리 & 보안** : 데이터 무결성 유지, 권한 없는 사용자 접근 통제
- **실제 기술 예시**

| 서버 종류 | 역할 | 예시 |
|---|---|---|
| Web Server | 정적 파일(HTML, 이미지) 빠르게 처리 | Nginx, Apache |
| WAS (Web Application Server) | 로그인, 회원가입 등 동적 비즈니스 로직 처리 | Tomcat, Node.js, Spring Boot |
| Database Server | 데이터 저장 및 관리 | MySQL, PostgreSQL, MariaDB |

##### 왜 클라이언트와 서버를 분리했을까?

원래는 하나였다. 1970~80년대에는 **메인프레임(거대한 중앙 컴퓨터)** 과 더미 터미널이 전부였는데, 비용이 엄청났고 병목 현상도 심했다.

1980년대 후반 개인용 컴퓨터(PC)가 보급되면서 변화가 생겼다.

- 중앙 컴퓨터가 하던 일 중 **화면 그리기, 사용자 입력 처리** 를 PC에 위임
- 중앙 컴퓨터는 **비즈니스 로직, 데이터 저장** 에 집중
- 이 분리 구조가 바로 **클라이언트-서버 아키텍처**

**장점**
- 비용 절감 : 클라이언트의 컴퓨팅 파워 활용
- UX 향상 : 클라이언트가 뼈대(UI)를 미리 갖추고, 서버 데이터만 끼워 화면 생성
- 확장 용이 : 사용자가 늘어날수록 클라이언트 자원도 함께 늘어남 (서버 부하 분산)
- 유지보수 편의

#### | HTTP

##### HTTP란?
**HyperText Transfer Protocol**의 약자.

- **하이퍼텍스트(HyperText)** : 단순한 글을 넘어 이미지, 링크, 서식 등이 포함된 구조화된 텍스트
- HTML(HyperText Markup Language)이 하이퍼텍스트의 대표적인 표현 방식
- HTTP는 HTML, CSS, JS, 이미지 등의 하이퍼텍스트를 웹 서버와 클라이언트 사이에서 전송하기 위한 **약속(프로토콜)**

##### 왜 사용해야 하는가?
표준 규약이 없다면 사람마다 제각각의 방식으로 데이터를 전송 → 혼돈

HTTP는 OS나 브라우저 종류와 상관없이 전 세계가 공통으로 사용하는 네트워크 약속이다.

#### | HTTP Message, Method, Status Code, URL

##### HTTP Message 구조

클라이언트와 서버 간 통신의 기본 단위. 크게 4개의 층으로 구성된다.

| 구성 요소 | 설명 |
|---|---|
| **시작 줄 (Start Line)** | 요청 방식, 요청 주소, HTTP 버전 등 요청/응답의 개요 |
| **헤더 (Header)** | 편지 봉투에 적힌 부가 정보 (메타데이터). 브라우저 버전, 데이터 크기, 데이터 종류 등 |
| **빈 줄 (Empty Line)** | 헤더와 바디를 구분하는 구분자 (`<CRLF>`) |
| **바디 (Body / Payload)** | 실제 알맹이 내용. 로그인 시 ID/PW, 응답 시 HTML 등 |

#### | HTTP Method

클라이언트가 서버에 "이 데이터로 무엇을 해줘" 라고 할 때의 **동사 역할**

| 메서드 | 역할 |
|---|---|
| `GET` | 데이터 조회 (가져오기) |
| `POST` | 새로운 데이터 생성 (등록) |
| `PUT` | 기존 데이터 전체 수정 (대체) |
| `PATCH` | 기존 데이터 일부 수정 |
| `DELETE` | 데이터 삭제 |

##### 왜 메서드를 나눴을까?

웹 트래픽이 폭발적으로 증가하면서 두 가지 문제가 생겼다.

1. **캐싱(Caching) 최적화**
   - 수억 명이 구글 뉴스를 읽을 때, 서버가 모든 요청을 다 열어봐야 한다면 CPU 부하가 막심
   - `GET` 요청임을 미리 알면 캐싱 서버에서 바로 응답 가능 → 본 서버 부하 절감

2. **보안 강화**
   - 데이터를 수정/삭제하는 요청(`PUT`, `DELETE`)을 미리 구분해두면
   - 중간 네트워크 장비에서 요청을 열어보지 않고도 "일반 유저가 왜 DELETE 요청?" 하고 빠르게 차단 가능

> 💡 요청을 열어보지 않고도 **안전한지, 캐시해도 되는지, 차단해야 하는지** 를 분류하기 위한 것!

#### | HTTP Status Code

서버가 클라이언트의 요청 결과를 명확하게 전달하기 위한 숫자 코드

| 범위 | 의미 |
|---|---|
| `1xx` | 정보 메시지 |
| `2xx` | 성공 |
| `3xx` | 리다이렉션 |
| `4xx` | 클라이언트 오류 |
| `5xx` | 서버 오류 |

> ⚠️ **500번대 에러**에서 서버가 상세한 실패 이유를 클라이언트에 노출하지 않는 이유?
> 예를 들어 로그인 실패 시 "비밀번호만 틀렸어요"라고 알려주면, 해커에게 "해당 ID는 존재한다"는 힌트를 주는 셈이 됨 → 보안상 상세 정보를 숨기는 것이 원칙!

#### | HTTP URL

**URL (Uniform Resource Locator)** : 리소스(웹에서 제공하는 모든 것)의 위치와 종류를 나타내는 주소

##### URL 구성 요소

```
https://www.example.kr:80/path/to/example.html?key1=value1&key2=value2
│─────│  │──────────────│ │────────────────────│ │──────────────────────│
스키마      도메인:포트           리소스 경로               매개변수(쿼리스트링)
```

| 구성 요소 | 설명 | 예시 |
|---|---|---|
| 스키마 | 사용할 프로토콜 | `http`, `https` |
| 도메인:포트 | 요청할 웹 서버와 게이트 번호 | `www.example.kr:80` |
| 리소스 경로 | 서버 내 리소스의 위치 | `/path/to/example.html` |
| 매개변수 | 추가 정보 전달 (`?`로 시작, `&`로 구분) | `?key1=value1&key2=value2` |

> 포트는 보통 숨겨두지만, nmap 같은 포트 스캐닝 도구로 확인 가능하다.

#### | Query String vs Path Variable

클라이언트가 URL로 서버에 추가 정보를 전달하는 두 가지 방법

##### Query String (쿼리 스트링)
- URL 끝에 `?`를 붙여 전달하는 추가 조건
- **정렬, 필터링** 에 사용

```
https://shop.naver.com/shoplist?category=tshirt&size=100
```
→ 쇼핑리스트 중 **티셔츠 카테고리, 사이즈 100** 조건으로 필터링

##### Path Variable (패스 베리어블)
- URL 경로 자체에 변수 값을 포함
- 특정 **리소스(자원)를 식별** 할 때 사용

```
https://naver.com/users/135/profile-img
```
→ 135번 유저의 프로필 이미지라는 **리소스 자체를 특정**

| 구분 | Query String | Path Variable |
|---|---|---|
| **사용 목적** | 필터링, 정렬 등 추가 조건 | 특정 리소스 식별 |
| **예시** | `?category=tshirt&size=100` | `/users/135/profile-img` |

#### | REST API

HTTP의 장점을 최대한 살린 통신 설계 규격

##### HTTP 기본 요소
- **리소스** : 요청 대상 (무엇에 대해)
- **메서드** : 행위 (어떻게 할지)
- **메시지** : 전달 내용 (어떤 데이터로)

> 예) "유저를 생성할래요. 유저 데이터는 이거입니다"
> → 리소스: 유저, 메서드: 생성(POST), 메시지: 유저 데이터

##### REST API 핵심 규칙

1. **URI는 자원을 명사로 표현한다**
   - ❌ `/getUsers` → ⭕ `/users`
   - 행위는 HTTP 메서드로만 표현한다

2. **`/`는 계층 관계를 나타낼 때 사용한다**

3. **자원에 대한 행위는 HTTP Method로 표현한다**

```
회원 생성       →  POST   /users
루루 회원 이름 변경  →  PATCH  /users/lulu  {"name":"Charlie"}
특정 회원 삭제   →  DELETE /users/{user_id}
```

> ⚠️ **규칙은 팀바팀!** 핵심 규칙 외에는 사내 명명 규칙, 팀 컨벤션에 따라 달라진다.
> 예를 들어 회원 삭제 API에서 30일 이내 복구 약관이 있다면 `PATCH`(소프트 딜리트)를 쓸 수도 있다.
> 중요한 것은 "논리"가 있는 선택을 하는 것!

#### | Spring Framework

**자바 애플리케이션에서 객체 생성과 의존성 관리를 자동화해 구조를 단순하고 유연하게 만드는 프레임워크(뼈대)**

##### 왜 Spring이 탄생했나?

- `JSP` : 복잡하고, 재사용성 부족, 비즈니스 로직 혼재
- `EJB` : 높은 개발 난이도, XML 기반 복잡한 배포 설정, 테스트 어려움
- "자바인데 왜 자바의 객체지향 특성을 못 살리지?" 라는 문제의식 → **Spring 탄생**

##### Spring의 핵심 특징

**IoC (Inversion of Control) - 제어 역전**

객체의 생성과 생명 주기를 개발자가 아닌 **스프링 컨테이너**가 관리

- 개발자가 직접 `new`로 객체를 생성하면?
  - 강한 결합도 → 테스트/유지보수 어려움
- IoC를 통해 개발자는 "이런 객체들이 필요해요"만 선언
- 스프링 컨테이너가 알아서 필요한 객체들을 주입

**DI (Dependency Injection) - 의존성 주입**

객체가 필요로 하는 다른 객체를 직접 생성하지 않고, **외부(컨테이너)에서 만들어서 주입**해주는 방식

의존성 주입 방식 3가지:
- 필드 주입
- 생성자 주입 ← 권장
- 세터 주입

**AOP (Aspect-Oriented Programming) - 관점 지향 프로그래밍**

공통 관심사와 핵심 관심사를 분리

| 구분 | 설명 | 예시 |
|---|---|---|
| **공통 관심사** | 모든 서비스에서 반복되는 기능 | 로그 기록, 트랜잭션, 보안, 성능 측정 |
| **핵심 관심사** | 서비스별로 다른 비즈니스 로직 | 회원가입, 주문 처리 등 |

**POJO (Plain Old Java Object)**

프레임워크나 라이브러리에 종속되지 않는 순수한 자바 객체

- 프레임워크 독립적 → 유연성 높음
- 단위 테스트 용이
- 필요한 필드와 메서드만으로 구성 → 간결함

#### | 스프링 컨테이너

**Spring Bean** : 스프링 컨테이너가 직접 생성하고 관리까지 책임지는 객체

##### 빈 등록 방법

1. **컴포넌트 스캐닝** (일반적으로 많이 사용)
   - `@Component`, `@Service`, `@Repository` 어노테이션 부착

2. **수동 등록**
   - 설정 클래스에 `@Configuration` + `@Bean` 어노테이션이 붙은 메서드 작성

### 🗨️ 오늘의 회고

- 웹의 역사부터 HTTP, REST API, 스프링까지 개발의 큰 흐름을 한 번에 배웠다.
  왜 이런 구조가 생겼는지 "히스토리"를 이해하면서 공부하니 훨씬 잘 와닿는다.
- REST API는 규칙이 있지만 팀마다 다를 수 있다는 점, "논리"가 있는 선택이 중요하다는 게 인상적이었다.
- Spring의 IoC/DI 개념은 글로 읽으면 이해되는데, 실제 코드로 작성할 때 어떻게 쓰이는지 예제를 더 찾아봐야겠다.
- JD는 기업의 "희망사항" → 완벽하게 갖추지 않아도 지원할 수 있다는 말을 기억해두자!
