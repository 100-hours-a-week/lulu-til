#### | 상태(State)와 연결(Connection)

##### Stateful vs Stateless

| 구분 | Stateful | Stateless |
|---|---|---|
| **개념** | 서버가 클라이언트의 상태를 기억 | 매 요청을 독립적으로 처리, 상태 기억 없음 |
| **비유** | "매번 아아 드시는 노민솔 고객님이시죠?" | "처음 뵙겠습니다. 원하시는 메뉴 골라주세요." |
| **장점** | 맞춤형 서비스, 복잡한 흐름 제어 가능 | 서버 부담 적음, 확장 용이, 처리 속도 빠름 |
| **단점** | 서버 메모리 부담, 분산 시스템에서 처리 어려움 | 요청마다 보낼 정보 많음, 맞춤형 서비스 어려움 |
| **사용 예** | 게임, 채팅 | HTTP, REST API |

> 💡 **HTTP는 Stateless** — 동시에 들어오는 수많은 요청을 빠르게 처리하기 위해 상태를 기억하지 않는다.

---

##### Connection Oriented vs Connectionless

| 구분 | Connection Oriented | Connectionless |
|---|---|---|
| **비유** | 전화 | 문자 |
| **방식** | 통신 중 연결 유지 | 요청-응답 후 연결 종료, 반복 |
| **장점** | 실시간성 보장 | 적은 자원으로 많은 요청 처리 |
| **단점** | 연결 유지에 자원 소모, 동시 연결 수 제한 | 매번 연결을 새로 맺어야 함 |
| **사용 예** | 영상통화, 온라인 게임 | HTTP |

> 💡 **HTTP는 Stateless + Connectionless** — 이 두 가지 특성 덕분에 수많은 사용자의 요청을 빠르고 효율적으로 처리할 수 있다.

#### | 인증(Authentication)과 인가(Authorization)

헷갈리기 쉬운 두 개념이지만 명확하게 다르다.

| 구분 | 인증 (Authentication) | 인가 (Authorization) |
|---|---|---|
| **질문** | "너 누구야?" | "너 이거 해도 돼?" |
| **의미** | 사용자의 신원을 확인하는 보안 절차 | 인증된 사용자에게 특정 권한을 부여하는 절차 |
| **예시** | 로그인 (ID/PW 확인) | 학생회장이 시험지를 수령할 수 있는 권한 |

> ⚠️ 인증 없이 인가는 없다. 항상 인증이 먼저 이루어진다.

#### | 쿠키 (Cookie)

**웹 사이트가 사용자의 브라우저에 저장하는 작은 기록 정보 파일**

##### 왜 쿠키를 사용하나?
서버 자원은 비싸다. 굳이 서버에 저장할 필요 없는 정보는 **브라우저에 저장**하는 것이 효율적.

##### 주로 저장하는 것들
- 방문 기록
- 장바구니에 담은 항목
- 언어 설정

> ⚠️ 쿠키는 브라우저에 평문으로 노출되기 때문에, **민감한 정보(비밀번호 등)는 저장하면 안 된다.**

---

#### | 세션 (Session)

**사용자 정보를 서버 측에서 관리해 보안을 강화하고 맞춤형 서비스를 제공하는 방식**

- 세션 저장소는 메모리, DB 등 서버가 관리하는 어떤 형태도 가능

##### 세션 + 쿠키 인증 흐름

```
① 클라이언트 → "안녕, 나는 A야" (ID/PW 전송)
② 서버 → 인증 확인 후 세션에 사용자 정보 저장
③ 서버 → 클라이언트에게 세션 ID 발급 (쿠키에 저장)
④ 이후 요청 → 쿠키의 세션 ID만 보고 바로 인증 처리
```

> 💡 매번 ID/PW를 다시 보내지 않아도 되니 효율적이고, 실제 정보는 서버에 있으므로 쿠키보다 보안이 높다.

#### | 프레임워크 vs 라이브러리

| 구분 | 프레임워크 | 라이브러리 |
|---|---|---|
| **핵심 가치** | 일관성 | 전문성 |
| **개념** | 개발의 기본 뼈대/규칙/구조를 정해놓음 | 특정 기능을 개발자가 필요할 때 호출해 사용 |
| **제어 주체** | 프레임워크가 흐름을 제어 (제어의 역전) | 개발자가 직접 호출해서 제어 |
| **자율성** | 낮음 (규약을 따라야 함) | 높음 |
| **예시** | Spring, Express | React, Lodash |

> 💡 **제어의 역전(IoC)** 관점에서 보면, 프레임워크는 "내가 네 코드를 불러서 실행할게", 라이브러리는 "네가 나를 불러서 써"의 차이다.

#### | API (Application Programming Interface)

**두 시스템 사이의 경계에서 상호 소통을 위해 만들어진 접점**

- 이미 만들어진 다른 애플리케이션의 기능을 가져다 쓸 수 있게 해줌
- 서로 다른 시스템 간 데이터 교환과 기능 공유가 가능

**API Docs (API 문서)**
- API 사용 설명서
- 내부 개발자용 내부 문서와 외부 공개 문서로 나뉨
- 프론트엔드 ↔ 백엔드 협업의 핵심 소통 수단

#### | 페이징 (Paging)

**한정된 자원에서 데이터를 적당한 크기로 나눠 페이지 단위로 처리하는 것**
주로 게시판, 카페, SNS 피드처럼 콘텐츠가 많은 곳에 사용된다.

##### 표현 방식

| 방식 | 설명 | 주요 사용처 |
|---|---|---|
| **페이지 베이스** | 페이지 단위로 끊어 이동 (`?page=1`) | 게시판, 카페 |
| **인피니티 스크롤** | 스크롤이 맨 아래 도달하면 다음 페이지 자동 요청 | SNS (인스타그램, 트위터) |

##### 구현 방식

**Offset-based**
- `offset`(건너뛸 수) + `limit`(가져올 양)으로 데이터 조회
- ✅ 구현이 간단하고 특정 페이지로 바로 이동 가능
- ❌ 데이터가 자주 변경되면 **중복/누락** 발생 가능

```
offset=0, limit=10  → 1~10번 데이터
offset=10, limit=10 → 11~20번 데이터
```

**Key-based**
- 데이터의 고유 키(인덱스)를 기준으로 다음 데이터를 조회
- ✅ 대용량 데이터에서도 조회 속도 빠름, 데이터 추가/삭제에도 안정적
- ❌ 특정 페이지로 바로 이동 불가, 순차 탐색만 가능

| 세부 방식 | 특징 |
|---|---|
| **Cursor-based** | 마지막으로 본 데이터의 key 값을 직렬화해 다음 조회 기준으로 사용 |
| **Token-based** | 마지막으로 본 위치를 토큰으로 저장해 다음 요청 시 사용 |

### 🗨️ 오늘의 회고

- Stateless + Connectionless라는 HTTP의 두 가지 특성이 왜 그렇게 설계되었는지 맥락을 이해하니, 쿠키/세션이 왜 필요한지까지 자연스럽게 연결됐다.
- 인증과 인가는 단어가 비슷해서 헷갈렸는데, "신원 확인 → 권한 부여" 순서로 정리하니 명확해졌다.
- 페이징은 단순한 기능처럼 보이지만 offset-based의 중복 문제, key-based의 순차 탐색 제한 등 실무 트레이드오프가 있다는 점이 흥미로웠다.
- 교재 추가 학습 우선순위 문제를 내일부터는 반드시 개선해야겠다.
