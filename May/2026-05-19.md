#### | Servlet

**자바 언어로 작성되어 웹 서버에서 클라이언트 요청을 받아 처리하고, 동적인 웹 콘텐츠를 생성해 응답하는 서버 측 프로그램**

초기 웹 개발에서는 URL 요청마다 별도의 서블릿이 필요했다.

```
"Post를 알고 싶어요" → Post 서블릿
"User를 알고 싶어요" → User 서블릿
...
```

요청 종류가 늘어날수록 서블릿도 폭발적으로 증가 → 관리가 어려워짐

##### Dispatcher Servlet

이 문제를 해결하기 위해 등장한 **"안내 전문 데스크 매니저"**

- 모든 외부 요청을 가장 먼저 혼자 수신
- 일을 직접 처리하지 않고, 요청을 확인해 적절한 컨트롤러로 연결
- **Handler Mapping** 을 통해 요청을 전달할 컨트롤러를 탐색

#### | 스프링 MVC

##### MVC 패턴이란?

소프트웨어를 **Model**, **View**, **Controller** 세 가지 역할로 나눠 개발하는 디자인 아키텍처.
코드의 복잡성을 줄이고 책임을 분리하기 위해 사용한다.

| 구성 요소 | 역할 | 예시 |
|---|---|---|
| **Model** | 데이터 및 비즈니스 로직 처리 | 회원가입 나이 검증, DB 저장/조회 |
| **View** | 사용자가 보는 화면/결과물 | HTML, Thymeleaf, JSON |
| **Controller** | URL 단위로 요청을 받고 Model ↔ View 연결 | `GET /posts` 요청 처리 |

> 💡 요즘은 View를 서버가 아닌 **클라이언트(프론트엔드)** 에서 직접 그리는 경우가 많다.
> 서버는 JSON만 내려주고, 화면 렌더링은 React·Vue 등 프론트가 담당 → 처리 속도 향상!

##### 스프링 MVC 내부 흐름

```
클라이언트 요청 (GET /posts)
        ↓
Dispatcher Servlet  →  모든 요청을 가장 먼저 수신
        ↓
Handler Mapping     →  "이 요청은 어떤 컨트롤러 담당이지?" 탐색
        ↓
Handler Adapter     →  해당 컨트롤러 호출
        ↓
Controller          →  어떤 비즈니스 로직을 수행할지 결정
        ↓
Model               →  비즈니스 로직 수행 + 결과 데이터를 담아둠
        ↓
View Resolver       →  결과를 어떤 형태로 줄지 결정 (HTML? JSON?)
        ↓
Dispatcher Servlet  →  최종 응답을 클라이언트에 전달
```

#### | HTTPS

##### 왜 HTTPS가 등장했나?

초기 웹(HTTP)은 정보를 **보기만** 하는 형식이라 보안이 크게 중요하지 않았다.
하지만 지금은 금융, 개인정보, 자산 등 중요한 정보를 처리하게 되면서 보안이 필수가 됐다.

```
HTTP + Secure = HTTPS
```

##### HTTPS의 핵심

1. **암호화** : 데이터를 키로 암호화해 중간에 탈취되어도 내용을 알 수 없게
2. **인증** : 통신 상대방의 신원을 확인해 사칭을 방지

---

##### 암호화의 두 가지 방식

**대칭키 암호화**
- 암호화와 복호화에 **동일한 키 1개** 사용
- 장점 : 데이터 전달이 빠름
- 단점 : 키 자체를 상대방에게 안전하게 전달하기 어려움

**비대칭키 암호화**
- **공개키(Public Key)** 로 암호화, **개인키(Private Key)** 로 복호화
- 장점 : 공개키는 누구에게나 공유해도 되므로 키 교환이 안전
- 단점 : 대칭키보다 처리 속도가 느림

> 💡 **전자 서명** : 개인키로 암호화하고 공개키로 복호화하는 방식.
> "이 데이터는 내(서버/기관)가 발급한 게 맞다"는 것을 증명하는 용도로 사용.

| 구분 | 대칭키 | 비대칭키 |
|---|---|---|
| **키 개수** | 1개 | 공개키 + 개인키 |
| **속도** | 빠름 | 느림 |
| **키 전달 안전성** | 낮음 | 높음 |
| **주요 용도** | 실제 데이터 통신 | 키 교환, 전자서명 |

##### HTTPS 통신 과정 (핵심)

> **HTTPS 통신의 핵심 = 클라이언트의 대칭키를 안전하게 주고받는 과정**

실제 데이터는 빠른 **대칭키**로 통신하되,
그 대칭키를 처음 교환할 때만 안전한 **비대칭키**를 활용하는 방식.

##### 인증서 (SSL/TLS Certificate)

서버의 신원을 보증하는 디지털 문서. 아래 정보가 담겨 있다.

- **서버의 공개키** : 클라이언트가 대칭키를 암호화해 서버에 전달할 때 사용
- **인증 기관(CA) 정보** : 이 인증서를 발급한 신뢰할 수 있는 기관
- **인증 기관의 전자 서명** : 인증 기관의 개인키로 서명 → "이 서버는 우리가 검증한 서버가 맞다"

### 🗨️ 오늘의 회고

- Dispatcher Servlet이 왜 필요한지, 요청이 들어왔을 때 내부에서 어떤 흐름으로 처리되는지 이해했다. 흐름도를 그려보니 각 컴포넌트의 역할이 훨씬 명확하게 보였다.
- HTTPS는 단순히 "암호화된 HTTP"라고만 알고 있었는데, 대칭키와 비대칭키를 조합해서 속도와 보안을 동시에 잡는 구조라는 걸 알게 됐다.
- 과제인 HTTP 내용 정리를 빠르게 마무리하고 교재 추가 학습도 병행해야겠다.
