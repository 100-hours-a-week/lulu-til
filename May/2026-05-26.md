#### | 데이터베이스 (Database)

##### 파일 시스템의 한계

초기에는 기능별로 하나의 파일을 만들어 데이터를 관리했다. 하지만 규모가 커지면서 다음과 같은 문제가 생겼다.

- 데이터 중복 기록
- 파일이 커질수록 탐색이 어려움
- 보안·권한 제어 부족
- 동시 수정 불가
- 데이터 무결성 보장 안 됨
- OS 간 호환성 문제

##### 데이터베이스가 해결한 것들

| 문제 | 데이터베이스의 해결 방법 |
|---|---|
| 중복/불일치 | 중앙 집중 관리 |
| 느린 탐색 | 인덱스 자료구조로 빠른 검색 |
| 보안 취약 | 사용자/DB별 권한 제어 |
| 동시 수정 충돌 | 트랜잭션과 락(Lock)으로 동시성 제어 |
| 잘못된 데이터 | 무결성 제약으로 규칙 강제 |
| OS 비호환 | 모든 OS에서 동작하는 시스템 제공 |

##### 관계형 데이터베이스 (RDBMS)

1970년 에드거 F. 코드가 제안한 **관계형 모델**에 기초한 데이터베이스.
데이터를 **행(Row)과 열(Column)** 로 구성된 테이블 형태로 저장한다.

**테이블 설계 순서**
1. **엔티티 정의** : 저장할 대상(개체) 결정
2. **속성 정의** : 각 엔티티가 가질 데이터 항목 결정
3. **관계 정의** : 엔티티 간의 관계 설계

**SQL (Structured Query Language)**

| 분류 | 이름 | 명령어 예시 |
|---|---|---|
| **DDL** | 데이터 정의어 | `CREATE`, `ALTER`, `DROP` |
| **DML** | 데이터 조작어 | `SELECT`, `INSERT`, `UPDATE`, `DELETE` |
| **DCL** | 데이터 제어어 | `GRANT`, `REVOKE` |

#### | ERD (Entity-Relationship Diagram)

**개체(Entity)와 관계(Relationship)를 한눈에 파악할 수 있도록 그린 도표**

설계 단계별 흐름:
1. **정보 모델링** : 현실 세계의 데이터 간 관계 구상
2. **논리적 모델링** : 업무 중심으로 개념적 데이터 모델 구체화

##### 까마귀발 표기법 (Crow's Foot Notation)

관계의 최솟값과 최댓값을 선과 기호로 표현하는 방식

| 관계 종류 | 의미 |
|---|---|
| **Zero** | 0개 또는 그 이상 (선택적 관계) |
| **One** | 반드시 1개 |
| **Many** | 1개 이상, 여러 개 가능 |
| **Zero or Many** | 0개 또는 여러 개 (선택적, 복수 가능) |
| **One or Many** | 최소 1개 이상, 여러 개 가능 |
| **One and Only One** | 반드시 딱 1개 (최솟값·최댓값 모두 1) |

> 예) 교사와 강좌 엔티티에서, 각 교사가 단 하나의 언어만 가르친다면 → **One and Only One**

#### | 식별 관계 vs 비식별 관계

| 구분 | 식별 관계 | 비식별 관계 |
|---|---|---|
| **개념** | 부모가 없으면 자식이 존재할 수 없음 | 부모가 없어도 자식이 독립적으로 존재 가능 |
| **PK 사용** | 부모의 PK를 자식의 PK로 사용 | 부모의 PK를 FK로만 사용 |
| **예시** | 주문 상세(주문 없으면 존재 불가) | 회원의 선택 프로필(회원 탈퇴해도 보존 가능) |

#### | 인덱스 (Index)

**테이블 조회 속도를 높이기 위한 자료구조**

책의 목차처럼, 원하는 데이터를 빠르게 찾을 수 있도록 별도의 색인을 만들어두는 것.

---

##### Hash Index

키 값을 해시 함수로 변환해 특정 버킷 주소에 매핑하는 방식

```
입력값(key) → Hash Function → 고정 길이 주소값 → 실제 데이터 위치
```

| 항목 | 내용 |
|---|---|
| **장점** | 단일 키 탐색 속도가 가장 빠름 (O(1)) |
| **단점** | 범위 검색·정렬 불가, 해시 충돌 발생 가능 |
| **주요 사용처** | In-memory DB, key-value 기반 DB (Redis, MongoDB 등) |

> 💡 MySQL InnoDB는 자주 접근되는 키 값을 내부적으로 해시 인덱스로 캐싱해 B+Tree 탐색을 생략하기도 한다.

---

##### Tree Index

노드의 계층 구조(부모-자식 관계)를 활용해 정렬된 값을 탐색하는 방식

- 왼쪽이 작고 오른쪽이 큰 비교 규칙 사용
- 한쪽으로만 치우친 **우편향 트리(right-skewed tree)** 가 되면 탐색 성능 저하

**B-Tree (Balanced Tree)**
- 모든 리프 노드가 동일한 깊이 유지 → 어떤 데이터든 일정한 탐색 시간
- 각 노드에 키 + 데이터(주소값)를 함께 저장
- ❌ 범위 검색 시 트리를 위아래로 반복 탐색 → Disk I/O 증가

**B+Tree** ← MySQL InnoDB 기본 자료구조

- **리프 노드에만 실제 데이터(주소값) 저장**, 내부 노드는 탐색용 키만 보관
- 리프 노드들이 **링크드 리스트로 연결** → 범위 검색 시 트리 재탐색 없이 순차 접근 가능

**B-Tree vs B+Tree 핵심 비교**

| 항목 | B-Tree | B+Tree |
|---|---|---|
| **노드 저장 내용** | 모든 노드에 키 + 데이터 | 내부 노드: 키만 / 리프 노드: 키 + 데이터 |
| **한 페이지(16KB)당 노드 수** | 적음 (4개, 데이터가 무거우므로) | 많음 (16개, 키만 저장하므로) |
| **Fan-out (자식 노드 수)** | 낮음 → 트리 깊이 깊어짐 | 높음 → 트리 깊이 얕아짐 |
| **범위 검색** | 트리 왕복 탐색 필요, Disk I/O 많음 | 리프 노드 링크드 리스트로 순차 접근 |
| **주요 사용처** | 일부 파일시스템 | MySQL InnoDB |

> 💡 **Fan-out이 높을수록** 하나의 노드가 더 많은 자식을 가질 수 있어 트리 높이가 낮아지고, 탐색 단계가 줄어든다.

#### | Full Text Index (전문 검색 인덱스)

**일반 텍스트 문장을 단어 단위로 쪼개 검색할 수 있게 하는 인덱스**

정렬 가능한 값(숫자, 날짜 등)은 B+Tree로 인덱싱이 쉽지만, **문장은 정렬 기준이 없다**.
→ 문장을 단어 단위로 분해해 각 단어가 어느 문서에 있는지 역방향으로 매핑하는 **Inverted Index(역색인)** 방식 사용

##### 파서(Parser) 종류

| 파서 | 방식 | 장점 | 단점 |
|---|---|---|---|
| **Stopword** | 공백·문장부호 기준으로 분리 | 구현 단순 | 한국어 조사 처리 안 됨 |
| **N-gram** | 지정된 글자 수 단위로 분리 | 한국어 포함 다양한 언어 지원 | 인덱스 크기 매우 커짐, 오검색 확률 높음 |
| **MeCab** | 형태소 분석 기반 분리 | 한국어 노이즈 감소, 인덱스 가벼움 | 동의어·문맥 처리 어려움, 대용량 트래픽 취약 |

##### Full Text Index의 한계

- 새 글이 삽입/수정될 때마다 역색인 재생성 필요 → 쓰기 비용 높음
- 어디까지나 **간이 기능** — 고급 검색 기능은 부족

> 💡 **한국어 대규모 검색**에는 **Elasticsearch** 같은 별도 전문 검색 엔진을 구축하는 것이 권장된다.

#### | 트랜잭션 (Transaction)

**한 번에 수행해야 할 연산들을 묶어 All or Nothing으로 처리하는 단위**

데이터베이스의 무결성과 일관성을 보장하기 위해 사용한다.

> 예) 송금 (찰리 → 보이드, 10만원)
> 1. 찰리 계좌에 10만원 있는지 확인
> 2. 보이드 계좌가 정상 상태인지 확인
> 3. 찰리 계좌에서 10만원 차감
> 4. 보이드 계좌에 10만원 추가
>
> → 4단계 중 하나라도 실패하면 전체 롤백

##### 트랜잭션 상태 흐름

```
Active (활성)
    ↓
Partially Committed (부분 완료) ← 커밋 명령 도착
    ↓                    ↓
Committed (완료)      Failed (실패)
(영구 반영)               ↓
                    Aborted (철회)
                    (이전 상태로 복구)
```

##### 무결성 vs 일관성

| 개념 | 의미 | 예시 |
|---|---|---|
| **무결성 (Integrity)** | 데이터에 결함이 없는 상태 | NOT NULL 컬럼에 null이 들어가지 않음 |
| **일관성 (Consistency)** | 작업 전후 데이터가 모순 없이 규칙을 준수 | 송금 전후 총 잔액 합계가 동일 |

---

#### | ACID

트랜잭션이 안전하게 수행된다는 것을 보장하기 위한 4가지 원칙

| 원칙 | 이름 | 의미 |
|---|---|---|
| **A** | Atomicity (원자성) | 트랜잭션은 완전히 수행되거나 전혀 수행되지 않아야 함 (All or Nothing) |
| **C** | Consistency (일관성) | 처리 결과가 항상 일관성과 정합성을 보장해야 함 |
| **I** | Isolation (격리성) | 동시에 실행되는 트랜잭션들은 서로 영향을 주지 않아야 함 |
| **D** | Durability (지속성) | 완료된 트랜잭션의 결과는 장애가 발생해도 영구 반영됨 |

> 예) **Isolation** : 비행기 좌석을 내가 구매하는 도중 다른 사람이 같은 좌석을 구매하려 해도, 내 트랜잭션에 영향을 주지 않음
>
> 예) **Durability** : 은행 서버가 꺼져도 루루의 계좌 잔액은 그대로 유지됨

##### Isolation Level (격리 수준)

트랜잭션 고립의 정도를 단계별로 정의한 기준.

> ⚠️ 격리 수준이 높아질수록 데이터 정합성은 높아지지만, 동시 처리량(Throughput)은 낮아진다. 성능과 정합성 사이의 트레이드오프를 상황에 맞게 선택해야 한다.

### 🗨️ 오늘의 회고

- 파일 시스템 → 데이터베이스 등장 배경을 먼저 이해하니, 인덱스와 트랜잭션이 왜 필요한지 자연스럽게 연결됐다.
- B-Tree vs B+Tree는 "노드에 데이터를 같이 저장하느냐"는 단순한 차이가 Fan-out, 범위 검색 성능까지 연쇄적으로 영향을 준다는 게 흥미로웠다. InnoDB가 B+Tree를 쓰는 이유가 납득됐다.
- Full Text Index는 한국어 특성상 한계가 명확하고, 실무에서는 Elasticsearch 같은 전문 검색 엔진을 따로 두는 이유가 이해됐다.
- ACID의 Isolation Level은 개념만 잡았는데, 각 레벨(READ UNCOMMITTED ~ SERIALIZABLE)별 동작과 발생할 수 있는 문제(Dirty Read, Phantom Read 등)는 추가로 공부해봐야겠다.
