#### | NoSQL

**Not Only SQL** — SQL만을 사용하지 않는 여러 유형의 데이터베이스

##### 왜 NoSQL이 등장했나?

기존 RDB는 정형화된 스키마와 강한 규칙을 갖고 있어, 아래와 같은 현대 서비스의 요구를 충족하기 어려워졌다.

| RDB의 한계 | NoSQL의 이점 |
|---|---|
| 스키마가 고정적 → 다양한 데이터 형태 수용 어려움 | 유연한 스키마 → 비정형 데이터 저장 가능 |
| 수직 확장(Scale-up) 중심 → 비용 증가 | 수평 확장(Scale-out) 용이 |
| 동시 대규모 쓰기에 약함 | 대규모 쓰기/읽기 처리에 최적화 |
| 복잡한 JOIN이 많아질수록 성능 저하 | 단순 구조로 빠른 읽기/쓰기 |

---

##### NoSQL의 4가지 유형

**1. Key-Value Store**

단순한 키-값 한 쌍으로 데이터를 저장하는 방식

```
"session:user123" → { userId: 123, role: "admin", exp: 1735000000 }
"token:abc123"    → { valid: true, userId: 456 }
```

- ✅ 인메모리 구조 + 해시 테이블 기반 → 매우 빠른 읽기/쓰기
- ✅ 구조가 단순해 수평 확장 용이
- ❌ value 내부 구조를 DB가 이해하지 못해 복잡한 쿼리 어려움
- **주 사용처** : 세션, 캐시, 토큰, 사용자 설정 등 속도에 민감한 곳
- **대표 DB** : Redis, Aerospike

**2. Document Store**

JSON, XML 같은 문서(Document) 단위로 데이터를 저장하는 방식

```json
{
  "_id": "user_001",
  "name": "Lulu",
  "email": "lulu@example.com",
  "address": { "city": "Seoul", "zip": "04524" },
  "tags": ["admin", "vip"]
}
```

- ✅ 스키마가 자유로워 구조 변경이 유연
- ✅ 문서 단위로 조회/수정 가능
- ❌ 데이터 간 관계성이 약함
- **주 사용처** : 전자상거래 플랫폼, 소셜 미디어
- **대표 DB** : MongoDB, CouchDB

**3. Column-Oriented Store**

테이블 구조처럼 보이지만, 행(Row)이 아닌 **컬럼(Column) 단위**로 데이터를 저장하는 방식

> 💡 RDB는 **Row-Oriented** (행 단위로 저장) → 한 행 전체를 읽어야 함
> Column Store는 **Column-Oriented** (컬럼 단위로 저장) → 필요한 컬럼만 빠르게 읽을 수 있음

- ✅ 대용량 데이터·빅데이터 처리에 적합
- ✅ 특정 컬럼만 조회할 때 읽기 성능 우수
- ❌ 스키마 설계가 복잡, 잘못 설계하면 비효율
- **주 사용처** : 로그 수집·분석, 시계열 데이터, 추천 시스템
- **대표 DB** : Cassandra

**4. Graph Store**

**노드(Node)와 엣지(Edge)** 구조로 데이터를 저장 → 관계(Relationship) 중심

```
(Lulu) --[팔로우]--> (Charlie)
(Charlie) --[좋아요]--> (게시물 A)
(Lulu) --[구매]--> (상품 B)
```

- ✅ 관계 기반 조회에 최적화 → 소셜 네트워크, 추천 시스템에서 강력
- ❌ 트랜잭션·대용량 처리가 복잡
- ❌ 관계를 유지해야 해서 수평 확장 어려움
- **주 사용처** : 소셜 그래프, 추천 엔진, 지식 그래프
- **대표 DB** : Neo4j, Amazon Neptune

---

##### NoSQL 유형 한눈에 비교

| 유형 | 저장 방식 | 강점 | 주 사용처 | 대표 DB |
|---|---|---|---|---|
| Key-Value | 키-값 쌍 | 빠른 읽기/쓰기 | 캐시, 세션, 토큰 | Redis |
| Document | JSON/XML 문서 | 유연한 스키마 | 전자상거래, SNS | MongoDB |
| Column-Oriented | 컬럼 단위 | 대용량 분석 | 로그, 시계열 | Cassandra |
| Graph | 노드-엣지 | 관계 탐색 | 소셜, 추천 | Neo4j |

#### | 데이터베이스 확장

사용자가 늘어날수록 DB 처리량도 폭증한다.
DB에서의 작업은 크게 **쓰기(Write)** 와 **읽기(Read)** 로 나뉘며, 이를 분산 처리하기 위한 기법들이 있다.

---

##### 파티셔닝 (Partitioning)

**하나의 테이블을 논리적으로 분할해 관리하는 기법**

**수직 파티셔닝 (Vertical Partitioning)**

하나의 테이블에서 **컬럼을 분리**해 자주 조회되는 컬럼과 그렇지 않은 컬럼을 나눈다.

```
[기존 users 테이블]
id | name | email | profile_img | bio | last_login | created_at

↓ 수직 파티셔닝

[users_basic]          [users_detail]
id | name | email      id | profile_img | bio | last_login
(자주 조회)             (가끔 조회)
```

- ✅ 자주 조회되는 데이터의 I/O 감소, 캐싱 효율 증가
- ❌ JOIN 필요도가 늘어 쿼리 복잡도 증가, 트랜잭션 관리 어려움

**수평 파티셔닝 (Horizontal Partitioning)**

테이블 스키마는 동일하게 유지하면서 **행(Row)을 기준으로 분리**한다.

```
[분리 기준 예시]
범위(Range)  : 2024년 데이터 / 2025년 데이터
리스트(List) : 서울 / 부산 / 제주
해시(Hash)   : user_id % 4 → 파티션 0, 1, 2, 3 균등 분배
```

- ✅ 쓰기 성능 향상, 아카이빙/백업 용이
- ❌ 인덱스 관리 어려움, 데이터 불균형 가능, 파티션 간 쿼리 복잡

##### 복제 (Replication)

**동일한 데이터를 여러 서버에 복제해 읽기 부하를 분산하는 기법**

```
          쓰기
클라이언트 ──→  Master DB
                  │ 복제
          ┌───────┼───────┐
        Slave1  Slave2  Slave3
          ↑       ↑       ↑
         읽기     읽기     읽기
```

- **Master** : 쓰기 담당
- **Slave** : 읽기 담당 (읽기 트래픽이 늘어나면 Slave만 추가하면 됨)
- 읽기가 압도적으로 많은 서비스에 효과적

#### | Server Cache

**자주 사용되는 데이터를 임시로 저장해두고, 동일한 요청이 오면 저장된 결과를 바로 반환하는 기법**

트래픽이 늘어날수록 반복되는 요청을 매번 DB까지 가서 처리하는 건 비효율적이다.
→ **"처리한 결과를 저장해두고, 같은 요청이 오면 이걸 줄게"**

```
[캐시 없을 때]
클라이언트 → WAS → DB 조회 → 응답  (매번 DB I/O 발생)

[캐시 있을 때]
클라이언트 → WAS → Cache Hit? → 응답  (DB 접근 없음)
                        ↓ Miss
                       DB 조회 → Cache에 저장 → 응답
```

> 💡 **Cache Hit** : 캐시에 원하는 데이터가 있어 바로 반환
> **Cache Miss** : 캐시에 없어서 DB까지 조회 후 캐시에 저장

자주 조회되지만 자주 변경되지 않는 데이터를 캐싱하면 DB 부하를 크게 줄일 수 있다.

---

##### Local Cache vs Global Cache

캐시는 저장 위치와 공유 범위에 따라 두 가지로 나뉜다. (아래 표에서 인스턴스 = WAS)

| 구분 | Local Cache | Global Cache |
|---|---|---|
| **정의** | 인스턴스에 독립적으로 존재 | 여러 인스턴스에서 공유하는 캐시 |
| **장점** | 데이터 접근 속도 빠름, 추가 네트워크 지연 없음 | 데이터 일관성과 공유, 동일한 캐시 데이터 사용 |
| **단점** | 스케일 아웃 시 일관성 유지 힘듦, 각 서버 캐시가 독립적으로 작동 | 네트워크 지연 가능성, 로컬에 비해 상대적으로 느림 |
| **예시** | 단일 서버 세션 정보, 임시 계산 결과 | 멀티 서버 세션 정보, 상품 정보·가격 등 공유 정보 |

---

##### Spring Local Cache 코드 예시 — 인기글

```java
@Service
public class PostService {

    // 인기 게시글 캐싱
    @Cacheable(value = "popularPosts")
    public List<Post> getPopularPosts() {
        return postRepository.findTop10ByOrderByLikesDesc();
    }

    // 유튜브 메타데이터 캐싱
    @Cacheable(value = "youtubeMeta", key = "#videoId")
    public YoutubeMeta fetchYoutubeMeta(String videoId) {
        return youtubeApiClient.getMeta(videoId);
    }
}
```

##### Spring Global Cache 코드 예시 — 인기글

```java
@Service
public class VideoService {

    @Cacheable(value = "youtubeCache", key = "#videoId", unless = "#result == null")
    public String getVideoInfo(String videoId) {
        System.out.println("실제 API 호출 중..");
        // ~~ DB 조회 ~~
        return "VideoInfo for " + videoId;
    }

    @CacheEvict(value = "youtubeCache", key = "#videoId")
    public void clearVideoCache(String videoId) {
        System.out.println("캐시 삭제: " + videoId);
    }
}
```

> 💡 `@Cacheable` : 메서드 결과를 캐시에 저장. 동일한 키로 재요청 시 메서드를 실행하지 않고 캐시값을 바로 반환
> `@CacheEvict` : 특정 키의 캐시를 삭제. 데이터가 변경될 때 캐시를 무효화하는 데 사용

### 🗨️ 오늘의 회고

- NoSQL은 "RDB가 나쁜 게 아니라, 상황에 맞는 도구를 쓰는 것"이라는 관점이 중요하다는 걸 느꼈다. 유형별로 강점이 다르니 실무에서 어떤 DB를 선택하는지도 관심 있게 봐야겠다.
- 파티셔닝의 수직/수평 차이가 처음엔 헷갈렸는데, "컬럼을 나누냐 / 행을 나누냐"로 정리하니 명확해졌다.
- 캐시, 복제, 파티셔닝이 모두 "DB 병목을 어떻게 줄이냐"는 같은 문제를 다른 방향에서 해결한다는 게 흥미로웠다.
