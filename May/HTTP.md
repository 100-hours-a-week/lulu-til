# 📡 HTTP (HyperText Transfer Protocol)

## HTTP란?

> 서버와 클라이언트가 서로 데이터를 주고 받기 위해 사용되는 **통신 규약**

- 웹 문서 간에 링크를 통해 연결할 수 있는 프로토콜
- 여러 종류의 데이터를 주고 받을 수 있다
  - HTML, TEXT
  - 이미지 / 음성 / 영상 파일
  - JSON, XML
  - 기타 등등

---

## 📜 HTTP의 역사

### HTTP/0.9 (1991년)

- 이후 버전과 구분하기 위해 **0.9 버전**이라고 불림
- 헤더 없음
- `GET` 메서드만 지원
- HTML 문서만 전송 가능한 단순한 프로토콜

---

### HTTP/1.0 (1996년)

**특징**

- `GET` 외에 `POST`, `HEAD` 메서드 추가
- 요청과 응답에 **헤더** 추가
  - 요청 헤더 : HTTP 버전 정보 추가
  - 응답 헤더 : 상태코드 추가(서버 상태 표시), `content-type` 추가(HTML 외 다양한 파일 전송 가능)
- **단일 연결** : 요청-응답 쌍마다 새로운 TCP 연결 사용
  - 매번 새로운 연결로 인한 성능 저하 및 서버 부하 비용 증가

---

### HTTP/1.1 (1997년)

**특징**

- **지속적인 연결 (Persistent Connection)** : `Keep-Alive` Header를 통해 지정한 timeout 동안 TCP 연결을 유지하여 요청-응답 처리 가능
- **파이프라이닝 (Pipelining)** : 하나의 커넥션에서 응답을 기다리지 않고 순차적인 여러 요청을 연속적으로 보내 그 순서에 맞춰 응답을 받는 방식 → 지연 시간 감소
- **HOL (Head Of Line) Blocking** : 클라이언트가 요청을 순서대로 보내더라도 TCP 프로토콜상 서버는 순차적으로 처리해야 함. 우선순위 요청의 응답 시간이 길어지면 후순위 요청의 응답 시간도 함께 길어지는 현상
- `PUT`, `DELETE`, `OPTIONS`, `TRACE` 등 메서드 추가
- **캐시 제어 메커니즘** : `Cache-Control` 헤더를 통해 클라이언트가 데이터를 캐싱하게 하여 불필요한 데이터 전송 감소
- **Host Header** : `host` 헤더 필드를 통해 가상 호스팅 지원

> 💡 우리가 아는 대부분의 HTTP 기능은 **1.1에 구현**된 것이고, 2와 3에서는 **성능 개선**에 초점이 맞춰져 있다.

---

### HTTP/2.0 (2015년)

**특징**

- **이진(Binary) 프로토콜** : HTTP/1.1의 텍스트 기반 전송 방식에서 데이터를 **바이너리로 변환**하여 전송 → 파싱 속도 향상, 오류 발생 가능성 감소
- **서버 푸시 (Server Push)** : 서버가 클라이언트의 요청을 예상하여 필요한 데이터를 미리 전송
  - 예) HTML 문서 요청 시, 그 문서가 사용하는 이미지 · CSS · JS 파일 등을 서버가 스스로 파악해 브라우저 캐시에 미리 push → 페이지 로딩 속도 향상
- **헤더 압축 (Header Data Compression)** : HTTP 메시지의 헤더를 압축하여 전송
  - 각 요청/응답마다 같은 데이터를 재전송하는 대신, 클라이언트와 서버 양측에서 이전에 보낸 `key-value` 쌍을 저장한 **헤더 테이블** 사용
  - 헤더 테이블은 커넥션이 살아 있는 동안 계속 존재하며, 양측에서 점진적으로 업데이트
  - 각 신규 헤더 `key-value` 쌍은 기존 테이블에 추가되거나 이전 값을 대체
  - 두 번째 요청부터는 변경된 헤더만 전송 → 매 요청마다 오버헤드를 대폭 감소
- **응답 다중화 (Multiplexing)** : 각 요청별로 **스트림(Stream)** 을 만들어 프레임(Frame)을 주고 받음 → 각 요청이 독립적으로 응답되어 **HOL Blocking 문제 해결**
- **Stream 우선순위 (Stream Prioritization)** : 멀티플렉싱으로 여러 요청/응답이 뒤섞이면서 패킷 순서가 혼잡해짐 → **우선순위 지정 트리**를 사용하여 서버의 Stream 처리 우선순위 지정 가능. 서버는 우선순위가 높은 응답이 먼저 전달되도록 대역폭을 설정

---

### HTTP/3.0 (2019년 ~ 진행중)

**HTTP/2.0의 한계**

1. HTTP는 여전히 **TCP 기반** 위에서 동작하기 때문에, TCP 자체의 핸드셰이크 과정에서 발생하는 지연시간 존재
2. TCP는 패킷이 유실되거나 오류가 있을 때 재전송하는데, 이 재전송 패킷에 지연이 발생하면 결국 **HOL Blocking 문제** 재발생
3. 이를 해결하기 위해 구글이 새로운 **UDP 기반 QUIC 프로토콜**을 고안 → TCP/IP 4계층에서도 동작하도록 설계한 것이 **HTTP/3.0**

**특징**

- `UDP + 패킷 재전송 + 혼잡제어 + 흐름제어` 등을 직접 구현한 = **QUIC**
- 다중 독립 Stream을 지원하며 손실된 패킷의 **빠른 복구** 가능
- **빠른 연결 설정** : QUIC는 `0-RTT` 연결 설정을 지원하며, 첫 번째 데이터 패킷이 연결 설정과 동시에 전송됨 → 3-way handshake를 대체하여 초기 지연 시간을 크게 단축
- **HOL Blocking 완전 해결** : HTTP/2에서 패킷 손실 시 대기해야 하는 HOLB 문제를 완전 해결

## 🏗️ HTTP 통신 구조

HTTP 통신은 **클라이언트(Front-End)** 와 **서버(Back-End)** 로 나뉘어진 구조

```
클라이언트 ──[ 요청(Request) ]──▶ 서버
클라이언트 ◀──[ 응답(Response) ]── 서버
```

**왜 클라이언트와 서버를 분리해야 하는가?**

각자의 역할에만 집중할 수 있기 때문이다.

| 역할 | 담당 |
|------|------|
| **클라이언트** | 복잡한 비즈니스 로직이나 데이터를 다룰 필요 없이 UI를 그리는 데 집중 |
| **서버** | 복잡한 비즈니스 로직이나 데이터를 다루는 데 집중. 트래픽 폭주로 고도화가 필요한 경우 클라이언트는 신경쓰지 않고 서버만 개선 가능 |

## 🔄 HTTP의 무상태성 (Stateless)

> **무상태(Stateless)** : 클라이언트와 서버 사이에 **상태를 유지하지 않는다**

- 서버가 클라이언트의 상태를 보존하지 않음
- 예) 홈페이지에서 회원 로그인을 하고 페이지를 이동했을 때 또 로그인을 요구하는 경우 → 서버는 클라이언트의 상태를 보존하지 않기 때문에 해당 클라이언트가 회원인지 알 수 없음
- 따라서 무상태 환경에서는 **회원 정보를 클라이언트가 토큰 형태로 보유**하면서, 서버와 통신할 때 실어 보내 인증하는 방식을 사용

| 구분 | 내용 |
|------|------|
| **장점** | 서버 확장성 높음 (Scale Out). 클라이언트가 상태 정보를 갖고 있으므로 아무 서버나 호출해도 되어 서버의 수평 확장에 유리 |
| **단점** | 클라이언트가 추가 데이터 전송 (데이터를 많이 사용) |

---

## 🔌 HTTP의 비연결성 (Connectionless)

> HTTP는 기본적으로 **연결을 유지하지 않는 모델**

- 서버와 클라이언트의 커넥션 연결을 지속하지 않음
- 클라이언트가 서버에 요청하고 응답을 받으면 바로 **TCP/IP 연결을 끊음**
- 1시간 동안 수천 명 이상이 서비스를 사용해도 실제 서버에서 동시에 처리하는 요청은 수십 개 이하로 적음
  - 예) 웹 브라우저 검색 페이지에서 검색 버튼만 연타하면서 이용하지는 않는 것처럼
- 비연결성 덕분에 **서버 자원을 매우 효율적으로 사용** 가능

**무상태성 vs 비연결성 차이**

| 개념 | 설명 |
|------|------|
| **무상태성 (Stateless)** | 클라이언트와 서버 간에 상태 정보를 보유하지 않아, 클라이언트가 상태 정보를 일일이 HTTP에 실어 요청해야 하는 것 |
| **비연결성 (Connectionless)** | 클라이언트와 서버 간에 네트워크 연결 자체가 끊어져 단절되는 것 |

---

## 📊 HTTP 상태 코드

> 클라이언트가 보낸 요청의 처리 상태를 응답에서 알려주는 기능 (100~500번대 숫자)

| 코드 | 분류 | 설명 |
|------|------|------|
| **1xx** | 정보 | 요청을 받았으며 프로세스를 계속 진행 |
| **2xx** | 성공 | 요청을 성공적으로 받았으며 인식했고 수용 |
| **3xx** | 리다이렉션 | 요청 완료를 위해 추가 작업 조치가 필요 |
| **4xx** | 클라이언트 오류 | 요청의 문법이 잘못되었거나 요청을 처리할 수 없음 |
| **5xx** | 서버 오류 | 서버가 명백히 유효한 요청에 대한 충족을 실패 |

## 📨 HTTP 메시지

> 클라이언트와 서버 간에 오가는 **요청**과 **응답**에 사용되는 요소

### HTTP 메시지의 기본 구조

```
┌─────────────────────────┐
│      시작 라인           │  (Start Line)
├─────────────────────────┤
│        헤더              │  (Header)
├─────────────────────────┤
│      공백 라인           │  (Empty Line = CRLF)
├─────────────────────────┤
│        바디              │  (Message Body)
└─────────────────────────┘
```

### HTTP 요청 메시지 구조

| 구성 요소 | 내용 |
|-----------|------|
| **시작 라인** | `Method` (GET / POST / PUT / DELETE 등) + `URL` (요청 대상 경로) + `Version` (사용된 HTTP 버전) |
| **헤더** | HTTP 전송에 필요한 모든 부가 정보 (메시지 크기, 압축 여부, 인증, 브라우저 정보, 서버 정보, 캐시 등) |
| **공백 라인** | 헤더와 바디를 구분하기 위한 라인 |
| **바디** | 실제 전송할 데이터 (HTML 문서, 이미지, 영상, JSON 등) |

### HTTP 응답 메시지 구조

| 구성 요소 | 내용 |
|-----------|------|
| **시작 라인** | `Version` (사용된 HTTP 버전) + `Status Code` (성공/실패 숫자 코드) + `Status Message` (사람이 이해할 수 있는 결과 텍스트) |
| **헤더** | HTTP 전송에 필요한 모든 부가 정보 (메시지 크기, 압축 여부, 인증, 브라우저 정보, 서버 정보, 캐시 등) |
| **공백 라인** | 헤더와 바디를 구분하기 위한 라인 |
| **바디** | 전송받은 데이터 |

## ⚙️ HTTP 메서드

> 클라이언트와 서버 사이의 **요청(Request)** 과 **응답(Response)** 데이터를 전송하는 방식.  
> 서버에 주어진 리소스에 수행하길 원하는 행동, 서버가 수행해야 할 동작을 지정하는 요청을 보내는 방법.

### 주요 메서드

| 메서드 | 동작 | 설명 |
|--------|------|------|
| `GET` | Read | 리소스 조회 |
| `POST` | Create | 요청 데이터 처리, 주로 등록에 사용 |
| `PUT` | Update | 리소스를 대체(덮어쓰기), 해당 리소스가 없으면 생성 |
| `PATCH` | Update | 리소스 **일부** 변경 (PUT이 전체 변경, PATCH는 일부 변경) |
| `DELETE` | Delete | 리소스 삭제 |

### 기타 메서드

| 메서드 | 설명 |
|--------|------|
| `HEAD` | GET과 동일하지만 메시지 Body를 제외하고, 상태 줄과 헤더만 반환 |
| `OPTIONS` | 대상 리소스에 대한 통신 가능 옵션(메서드)을 설명 (주로 CORS에서 사용) |
| `CONNECT` | 대상 자원으로 식별되는 서버에 대한 터널을 설정 |
| `TRACE` | 대상 리소스에 대한 경로를 따라 메시지 루프백 테스트를 수행 |

---

### 메서드 상세 설명

#### `GET`
- 리소스 **조회** 메서드 (Read)
- 서버에 전달하고 싶은 데이터는 **쿼리스트링**을 통해 전달
- 메시지 바디를 사용한 데이터 전달도 가능하지만, 서버에서 지원하지 않는 곳이 많아 권장하지 않음
- 조회 시 POST도 사용할 수 있지만, GET 메서드는 **캐싱이 가능**하기 때문에 GET 사용이 유리

#### `POST`
- 전달한 데이터 **처리 / 생성** 요청 메서드 (Create)
- **메시지 바디**를 통해 서버로 요청 데이터를 전달하면 서버는 요청 데이터를 처리하여 업데이트
- 주로 신규 리소스 등록, 프로세스 처리에 사용
- 데이터를 GET 하는데 JSON으로 조회 데이터를 넘겨야 하는 애매한 경우에도 POST 사용

#### `PUT`
- 리소스를 **대체(수정)** 하는 메서드 (Update)
- 요청 메시지에 리소스가 있으면 **덮어쓰고**, 없으면 **새로 생성**
- 클라이언트가 리소스의 **구체적인 전체 경로**를 지정해 보내야 함

#### `PATCH`
- 리소스의 **일부**를 변경하는 메서드 (Update)
- PATCH를 지원하지 않는 서버에서는 `POST`를 대신 사용 가능

#### `DELETE`
- 리소스를 **제거**하는 메서드 (Delete)
- 상태 코드는 대부분 `200`을 사용하고, 상황에 따라 `204` 사용

#### `HEAD`
- GET과 동일하지만 서버에서 **Body를 반환하지 않음**
- 응답의 상태 코드만 확인할 때처럼 리소스를 받지 않고 **찾기만 원할 때** 사용 (검사 용도)
- 서버의 응답 헤더를 통해 리소스가 수정되었는지 확인 가능

#### `OPTIONS`
- **예비 요청(Preflight)** 에 사용되는 HTTP 메서드
- 예비 요청 : 본 요청을 하기 전에 안전한지 **미리 검사**하는 것
- 서버의 지원 가능한 HTTP 메서드와 출처를 응답받아 **CORS 정책을 검사**하기 위한 요청

#### `CONNECT`
- 클라이언트와 서버 간의 데이터 전송을 위한 **터널을 설정**하는 메서드
- 주로 **프록시 서버**와 함께 사용되며, 보안 연결(HTTPS)을 중계할 수 있도록 터널을 생성하는 데 이용
- 프록시 서버가 클라이언트와 서버 간의 직접적인 연결을 설정하여 암호화된 데이터를 중계할 수 있게 함

#### `TRACE`
- 일종의 **검사용** 메서드
- 서버에 도달했을 때의 최종 패킷 요청 내용을 응답받을 수 있음
- 요청의 최종 수신자는 반드시 송신자에게 `200 (OK)` 응답의 Body로 수신한 메시지를 반송해야 함
- 최초 클라이언트의 요청에는 **Body가 포함될 수 없음**
