#### | OOP (Object-Oriented Programming)

**현실 세계의 객체 개념을 기반으로, 데이터를 속성(필드)과 기능(메서드)으로 묶어 프로그램을 설계하는 패러다임**

##### OOP 없이 절차지향으로 개발하면?

규모가 커질수록 유지보수가 급격히 어려워진다.

> 예) "결제 엔진을 변경해주세요" 요청이 들어왔을 때
> - 결제 로직이 주문, 배송, 장바구니 등 여기저기 함수로 흩어져 있다면
> - 연관된 코드를 전부 찾아서 고쳐야 함 → 작업 시간 폭증, 사이드 이펙트 위험

절차지향은 기능들이 서로 얽혀 있어 **한 곳의 변경이 다른 곳에 영향을 주는 구조**다.

##### OOP는 무엇인가?

> **"누가 이 일을 책임질래?"** 라는 기준으로 나눈 프로그래밍 기법

처리 순서가 아닌, **역할과 책임을 가진 주체(객체)들의 협력**으로 프로그램을 구성한다.

```
엔진 객체    → "시동을 걸고 바퀴에 회전을 주는 것"만 책임
              (연료가 가솔린이든, 전기든 상관없음)

브레이크 객체 → "바퀴에 제동을 거는 것"만 책임
              (디스크 브레이크든, 드럼 브레이크든 상관없음)

자동차 객체  → 직접 시동을 걸거나 제동하지 않음
              엔진과 브레이크를 조합하는 역할만 담당
              → 부품이 바뀌어도 자동차 객체는 수정할 필요 없음
```

##### 스프링 부트 계층에도 OOP가 적용되어 있다

| 계층 | 책임 |
|---|---|
| `Controller` | 웹 요청 흐름 처리 |
| `Service` | 비즈니스 로직 수행 |
| `Repository` | 데이터 접근 처리 |

> 💡 OOP에서 중요한 것은 **"객체는 이 일을 책임지는 주체"** 라는 관점이다.
> 대규모 서비스를 안정적으로 유지보수할 수 있게 하는 근본적인 설계 기법!

#### | SOLID 원칙

유지보수성과 확장성을 높이기 위해 만들어진 **다섯 가지 객체지향 설계 원칙**

| 원칙 | 이름 | 한 줄 요약 |
|---|---|---|
| **S** | SRP (단일 책임 원칙) | 클래스를 수정해야 하는 이유는 한 가지여야 한다 |
| **O** | OCP (개방-폐쇄 원칙) | 확장에는 열려 있고, 기존 코드 수정에는 닫혀 있어야 한다 |
| **L** | LSP (리스코프 치환 원칙) | 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다 |
| **I** | ISP (인터페이스 분리 원칙) | 내가 안 쓰는 거대한 인터페이스보다 잘게 쪼갠 인터페이스를 써라 |
| **D** | DIP (의존관계 역전 원칙) | 구체적인 것이 아닌 추상적인 것에 의존해라 |

> 💡 **S(SRP)와 O(OCP)를 잘 지키면 나머지 원칙들은 대체로 따라온다!**

---

##### SRP (Single Responsibility Principle) - 단일 책임 원칙

> 클래스를 수정해야 하는 이유가 **오직 한 가지**여야 한다

수정될 이유가 다른 기능들은 서로 분리해야 한다.

**SRP 적용 시 얻는 것**
- **수정 범위 최소화** : 변경이 생겨도 해당 클래스만 수정
- **가독성 향상** : 코드가 하나의 역할만 하므로 읽기 쉬움
  - 내가 작성한 코드는 다른 개발자들도 보는 공식 문서가 된다

**스프링에서의 SRP 적용**

| 계층 | 수정해야 하는 이유 |
|---|---|
| `Controller` | 웹 요청 주소나 API 스펙이 바뀔 때 |
| `Service` | 환불 정책, 할인 규칙 등 비즈니스 룰이 바뀔 때 |
| `Repository` | DB 종류가 바뀔 때 |

> ⚠️ 모든 코드를 극단적으로 쪼개야 한다는 뜻은 아니다.
> 설계에는 정답이 없고, 팀·프로젝트·중요도에 따라 적절한 수준을 찾는 것이 중요하다.

---

##### OCP (Open Closed Principle) - 개방-폐쇄 원칙

> - **Open** : 새로운 기능 추가/확장에는 항상 열려 있어야 한다
> - **Closed** : 그로 인해 기존 코드가 수정되는 것에는 닫혀 있어야 한다

**새 기능을 추가할 때, 잘 돌아가고 있는 기존 코드는 건드리지 마라**

```
🖥️ 마우스를 교체할 때
→ 포트에서 마우스를 뽑고 새 마우스를 꽂으면 끝
→ 본체 뚜껑을 열고 메인보드를 건드릴 필요가 없다
```

---

##### DIP (Dependency Inversion Principle) - 의존관계 역전 원칙

> 상위 모듈(정책/핵심 규칙)이 하위 모듈(세부 구현)에 **직접 의존하지 않고**,
> 양쪽 모두 **추상화(인터페이스)에 의존**해야 한다

**상위 모듈이 하위 모듈에 직접 의존하면?**
- 하위 모듈(세부 기술)이 변경될 때마다 상위 모듈(핵심 비즈니스)도 함께 수정해야 함
- 코드의 주도권을 잃게 됨

**OCP + DIP 적용 예시 — 파일 저장소**

```
<<Interface>>
FileStorage              ← "파일을 저장한다"는 원칙만 정의
    ↑               ↑
LocalFileStorage    CloudFileStorage
(로컬에 저장)       (클라우드에 저장)
```

저장 방식이 바뀌어도 비즈니스 로직은 `FileStorage` 인터페이스만 바라보면 되므로 수정 불필요!

#### | JVM (Java Virtual Machine)

**자바 프로그램이 OS에 상관없이 실행될 수 있도록 하는 가상 머신**

- 낮은 이식성 문제를 해결 → "Write Once, Run Anywhere"
- 바이트 코드를 각 OS에 맞는 기계어로 번역
- 메모리 자동 관리 (Garbage Collection)

##### 자바 프로그램 실행 과정

```
main.java (자바 소스코드)
      ↓ javac (자바 컴파일러)
main.class (바이트 코드)
      ↓ JVM
기계어 (OS에 맞게 번역 후 실행)
```

---

##### 클래스 로더 (Class Loader)

프로그램 실행에 필요한 `.class` 파일을 JVM에 가져오는 역할

- 프로그램 시작 시 모든 파일을 한 번에 로드하지 않음
- **필요할 때(Runtime)** 해당 클래스만 동적으로 로드 → 메모리 효율

**바이트 코드 내부 구성**

| 구성 요소 | 설명 |
|---|---|
| **Constant Pool** | 클래스 이름, 메서드 이름, 문자열 등 메타데이터를 담는 내부 테이블 |
| **Default Constructor** | 생성자를 작성하지 않으면 컴파일러가 자동으로 생성 |
| **메서드 정보** | 스택 슬롯 수, 지역 변수 슬롯 수 등 실행에 필요한 정보 |

**클래스 로드 과정**

```
Loading      → .class 파일을 JVM 메모리에 로드
    ↓
Linking
  ├ Verifying  → JVM 명세에 맞는지 검증
  ├ Preparing  → 클래스에 필요한 메모리 할당
  └ Resolving  → 심볼릭 레퍼런스 → 다이렉트 레퍼런스로 변환
    ↓
Initialization → static 필드 등 클래스 변수를 설정된 값으로 초기화
```

---

##### 실행 엔진 (Execution Engine)

클래스 로더가 올려놓은 바이트 코드를 기계어로 해석해 실행

- **Interpreter** : 바이트 코드를 한 줄씩 기계어로 해석
- **JIT Compiler** : 자주 호출되는 코드를 미리 기계어로 컴파일해 캐싱 → 반복 실행 속도 향상

---

##### 런타임 데이터 영역 (Runtime Data Areas)

JVM이 자바 애플리케이션 실행 시 사용하는 메모리 공간

| 영역 | 특징 | 저장 내용 |
|---|---|---|
| **메서드 영역** | JVM 실행부터 종료까지 유지 | 클래스 정보, 클래스 변수, Constant Pool |
| **힙 영역** | 런타임 시 동적으로 할당 | `new`로 생성된 객체, 인스턴스 변수 |
| **스택 영역** | 스레드마다 독립 생성, LIFO 구조 | 기본형 변수, 메서드 실행 정보(스택 프레임) |
| **PC 레지스터** | 스레드마다 독립 생성 | 현재 실행 중인 JVM 명령어 주소 |
| **네이티브 메서드 스택** | C/C++/어셈블리 코드 실행 시 사용 | 네이티브 메서드 실행 정보 |

> 💡 **스택 프레임** : 메서드 호출마다 하나씩 생성되며, 해당 메서드 실행에 필요한 모든 정보를 담고 있다. 메서드가 종료되면 스택에서 제거된다.

### 🗨️ 오늘의 회고

- OOP → SOLID → JVM 순서로 배우면서 "왜 이렇게 설계하는가"에 대한 맥락이 자연스럽게 연결됐다.
- SOLID 원칙은 이름만 외우는 게 아니라, **SRP는 스프링 계층 구조**, **OCP·DIP는 인터페이스 설계**에서 실제로 어떻게 적용되는지 예시를 통해 이해하는 게 핵심인 것 같다.
- JVM 메모리 구조(메서드 영역, 힙, 스택)는 이후 멀티스레드나 GC를 공부할 때 다시 쓰일 개념이니 확실히 정리해두기.
