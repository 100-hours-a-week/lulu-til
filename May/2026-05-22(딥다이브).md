Spring의 핵심 요소는 **애플리케이션 수준에서의 인프라 지원**이다. 엔터프라이즈 애플리케이션의 "기반 구조"에 초점을 맞춰 개발팀이 특정 배포 환경에 얽매이지 않고 **비즈니스 로직에 집중**할 수 있게 한다.

| **사용 언어** | Kotlin, Groovy, 동적 프로그래밍 언어 |

## 🔄 Spring IoC (Inversion of Control) 컨테이너

> Spring 프레임워크에서 **객체를 생성하고 관리하고 의존성을 책임지는** 컨테이너

- POJO의 생성, 초기화, 서비스, 소멸에 대한 권한을 가진다
- 개발자들이 직접 POJO를 생성할 수 있지만 **컨테이너에게 맡긴다**
- 덕분에 개발자는 **비즈니스 로직에 집중** 가능
- 객체 생성 코드가 없으므로 **TDD가 용이**

### IoC의 분류

#### DL (Dependency Lookup)
저장소에 저장되어 있는 Bean에 접근하기 위해 컨테이너가 제공하는 API를 이용하여 Bean을 Lookup하는 것

#### DI (Dependency Injection)
각 클래스 간의 의존관계를 빈 설정 정보를 바탕으로 컨테이너가 **자동으로 연결**해주는 것

| 주입 방식 | 설명 |
|-----------|------|
| **Setter Injection** | 수정자 주입 |
| **Constructor Injection** | 생성자 주입 |
| **Method Injection** | 필드 주입 |

> 💡 의존성 주입(DI)은 제어의 역전(IoC)의 특수한 형태로, 객체가 생성자 인자, 팩토리 메서드 인자, 또는 객체 인스턴스가 생성된 후 설정되는 속성을 통해서만 의존성을 정의하는 방식이다. IoC 컨테이너는 빈을 생성할 때 이러한 의존성을 주입한다. 이 과정은 빈 자체가 클래스의 직접 생성이나 서비스 로케이터 패턴을 사용하여 의존성을 제어하는 방식의 **근본적인 반대 개념**이다. (그래서 제어의 역전이라는 이름이 붙었다)

### IoC 컨테이너의 기반 패키지

- `org.springframework.beans`
- `org.springframework.context`

#### BeanFactory 인터페이스
- 모든 유형의 객체를 관리할 수 있는 **고급 구성 메커니즘** 제공
- 구성 프레임워크와 기본 기능 제공
- Bean을 등록 / 생성 / 조회 / 반환 관리
- 빈을 생성하고 분배하는 책임을 진다
- Bean을 조회할 수 있는 `getBean()` 메서드가 정의되어 있음

#### ApplicationContext 인터페이스
- `BeanFactory`의 **하위 인터페이스**
- 기업 특화 기능 제공
- Spring IoC 컨테이너를 나타내며 빈의 인스턴스화, 구성 및 조립을 담당
- BeanFactory보다 추가로 제공하는 기능
  - 국제화가 지원되는 텍스트 메시지 관리
  - 이미지 같은 파일 자원을 로드할 수 있는 포괄적인 방법 제공
  - 리스너로 등록된 빈에게 이벤트 발생을 알려줌

### Bean이란?

> Spring에서 애플리케이션의 핵심을 이루고 **Spring IoC 컨테이너에 의해 관리되는 객체**

- Spring IoC 컨테이너에 의해 인스턴스화, 구성 및 관리되는 객체
- 빈은 애플리케이션에 있는 여러 객체 중 하나
- 빈과 빈 사이의 의존 관계는 컨테이너에서 사용하는 **구성 메타데이터**에 반영

## 🧩 Microservice

> 하나의 애플리케이션이 **느슨하게 결합되고 독립적으로 배포 가능한** 여러 개의 작은 서비스로 구성된 클라우드 네이티브 아키텍처 방식

- 데이터베이스 및 데이터 관리 모델을 포함하는 **자체 기술 스택** 보유
- REST API, 이벤트 스트리밍, 메시지 브로커의 조합을 통해 서로 통신
- 비즈니스 기능에 따라 구성되며, **경계가 있는 컨텍스트**로 서비스를 구분

### 마이크로서비스의 특징

#### 자율성
- 각 구성 요소 서비스는 다른 서비스의 기능에 영향을 주지 않으면서 독립적으로 개발, 배포, 운영, 확장 가능
- 서비스가 코드 또는 구현을 다른 서비스와 공유할 필요 없음
- 개별 구성 요소 간의 통신은 **잘 정의된 API**를 통해 이루어짐

#### 전문성
- 각 서비스는 일련의 기능을 위해 설계되며 **특정 문제 해결**에 중점
- 서비스가 복잡해지면 더 작은 서비스로 분할 가능

### 마이크로서비스의 이점

| 이점 | 내용 |
|------|------|
| **민첩성** | 독립적인 소규모 팀 조직을 육성, 개발 주기 시간 단축 |
| **유연한 확장성** | 각 서비스를 독립적으로 확장, 필요한 인프라 규모 적절히 조절 가능 |
| **손쉬운 배포** | CI/CD를 통해 새 아이디어를 손쉽게 시험하고 롤백 가능 |
| **기술적 자유** | 특정 문제 해결에 가장 적합한 도구를 자유롭게 선택 |
| **재사용 가능한 코드** | 잘 정의된 소규모 모듈로 분할하여 기능을 여러 용도로 활용 |
| **복원력** | 서비스가 독립적이므로 실패에 대한 애플리케이션의 저항성 증가 |

## 📄 Project Object Model (POM)

> Maven의 **기본 작업 단위**

- 프로젝트에 대한 정보와 Maven이 프로젝트를 빌드하는 데 사용하는 **구성 세부 정보**를 포함하는 XML 파일
- `pom.xml` 파일에 프로젝트 관리 및 빌드에 필요한 환경 설정, 저장소 서버 위치 지정, 의존성 관리 등의 정보를 저장

## 🔧 Maven과 Gradle

> 프로젝트에서 작성한 코드와 xml, yml, properties, jar 파일을 JVM이나 Tomcat이 인식할 수 있도록 **패키징해주는 빌드 자동화 도구**

앱 개발 시 다양한 외부 라이브러리를 빌드 도구 설정 파일에 **필요한 라이브러리 종류, 버전, 종속성을 명시**하여 다운로드해주는 도구를 **빌드 관리 도구**라고 한다.

### Maven

- Java용 프로젝트 관리 도구 (Apache의 Ant 대안으로 만들어짐)
- 프로젝트, 빌드 순서, 외부 라이브러리 종속성 관계를 `pom.xml`에 명시
- 외부 저장소에서 필요한 라이브러리와 플러그인을 설치 후 **로컬 캐시에 저장**
- `setting.xml`을 통해 중앙저장소에서 로컬저장소로 다운로드하는 **저장소 경로 변경 가능**

### Gradle

- Maven과 Ant의 개념들을 사용하는 대안으로 나온 프로젝트 관리 도구
- Groovy 언어(JVM에서 사용되는 스크립트 언어로 Java와 유사)를 사용하는 **DSL(Domain Specific Language)** 사용
- XML 파일보다 **코드가 간결**하다는 장점

### Maven vs Gradle 비교

| 구분 | Maven | Gradle |
|------|-------|--------|
| **모델** | 고정적이고 선형적인 단계 기반 | 작업 의존성(Configuration Injection) 기반 |
| **빌드 속도** | 상대적으로 느림 | 이미 빌드된 작업은 재실행하지 않아 **빌드 시간 단축** |
| **캐시** | 일반 캐시 | **concurrent에 안전한 캐시** 허용 |

### 코드 비교

**Maven → `pom.xml`**
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-devtools</artifactId>
    <scope>runtime</scope>
    <optional>true</optional>
</dependency>
```

**Gradle → `build.gradle`**
```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web'
    implementation 'org.springframework.boot:spring-boot-starter-thymeleaf'
    implementation 'org.springframework.boot:spring-boot-starter'
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
}
```

## 🚀 Spring Boot

### Spring Boot란?

> Spring 기반 애플리케이션을 생성하기 위한 **오픈 소스 Java 기반 프레임워크**

- Spring의 모든 도구와 기술을 확장하여 **빠른 애플리케이션 개발**을 가능하게 하는 더 사용하기 쉬운 프레임워크
- 의존성 주입, 국제화, 데이터 바인딩, 타입 변환, AOP 등의 핵심 기술 보유
- 테스트, 모의, MVC 웹 애플리케이션, 원격, 작업 스케줄링 등 필요한 모든 것을 갖춤

### Spring vs Spring Boot 비교

| 구분 | Spring | Spring Boot |
|------|--------|-------------|
| **설정** | 모듈과 라이브러리를 직접 선택·설치·구성해야 함 | 프레임워크가 설치 및 구성 선택을 대신 처리 |
| **시작 속도** | 코딩 시작 전 세팅에 상당한 시간 필요 | 훨씬 빠르게 시작 가능 |
| **커스터마이징** | 거의 무한히 구성 가능 | 일부 커스텀 기능 포기 가능성 있으나 대부분 문제 없음 |
| **적합한 사례** | 매우 크고 복잡한 프로젝트, 에지 케이스 | 웹 개발 및 마이크로서비스 프로젝트 |

> 💡 Spring은 **모든 종류의 도구가 들어있는 큰 공구 상자**, Spring Boot는 **특정 작업에 필요한 도구만 담은 작은 공구 상자**로 비유할 수 있다.

### Spring Boot의 작동 방식

Spring Boot는 **DI(Dependency Injection)** 및 **IoC(Inversion of Control)** 를 사용하여 Java 프로그램에 기능을 추가한다.

```
개발자 → 프로그램 논리 처리
Spring Boot → 이벤트 루프 및 메시지 전달 제공
IoC 컨테이너 → 객체와 기능에 대한 의존성 주입을 효율적으로 처리
```

- **DI** : 외부 프레임워크(Spring Boot)가 엔터프라이즈 애플리케이션에 객체 및 기능을 제공하므로, 개발자는 이를 직접 구축할 필요 없음
- **IoC** : 프로그램의 제어 흐름이 외부 소스에서 제공됨. Spring의 특별한 IoC 컨테이너는 개발자의 노력을 최소화하면서 의존성 주입을 효율적으로 처리
- **Java 주석(@Annotation)** : `@` 기호가 붙은 태그로 코드에 메타데이터를 제공하여 기능 추가에 활용

**4계층 아키텍처**
```
HTTP 요청 / JSON 데이터 전송
        ↕
   비즈니스 논리 (프로그램 로직)
        ↕
   데이터 객체 표현
        ↕
   외부 데이터베이스 실제 표현
```

> 이 아키텍처는 마이크로서비스 구축에 이상적이며, MVC 설계 패턴에도 매핑 가능

### Spring Boot의 장점

- 개발 가속화
- 간소화된 구성
- 마이크로서비스를 위한 기본 제공 기능
- 종속성 관리
- 생산 지원 기능
- 원활한 통합

### Spring Boot 주요 기능

| 기능 | 설명 |
|------|------|
| **자동 구성** | 독단적 접근 방식에 따라 모듈 및 외부 라이브러리를 자동으로 결정하고 구성. 선택 및 구성에 드는 시간 절약 |
| **독립 실행형 JAR** | 런타임 환경에 직접 배포할 수 있는 독립형 실행 JAR 파일 생성. 모든 Java 클래스, 메타데이터, 이미지 및 리소스 포함 |
| **내장 웹 서버** | HTTP 서버 필요 시 자동으로 설치 및 구성. Tomcat, Jetty, Undertow 사용 가능 |
| **Spring Boot Starters** | 모든 라이브러리와 종속성을 단일 구성 파일로 묶어 프로세스 단순화. 코드 양 감소, 누락 방지 |
| **외부화된 구성** | 런타임 환경에 특화된 명령줄 인자, 속성 파일, 환경 변수 등을 단일 파일로 캡슐화하여 `@` 주석을 통해 액세스 |
| **Spring Boot 개발 도구** | 오류 관리, 로깅, 원격 실행, 클래스 경로 관리 등을 지원하는 추가 도구 포함. 필요에 따라 선택적 사용 가능 |
| **Spring Boot Actuator** | 런타임 중 모니터링 가능한 엔드포인트 정의. HTTP 또는 JMX를 사용하여 활성화/비활성화/노출. 상태, 메모리, 예외 등 모니터링 |
| **간소화된 종속성 관리** | 종속성을 수동으로 구성하지 않아도 됨. 필요한 모듈과 라이브러리를 자동 구성하고 독립 실행형 JAR로 패키징 |
| **생산 지원 기능** | Spring Boot Starters(구성 패키징), 독립 실행형 JAR(배포), Spring Boot Actuators(런타임 상태 모니터링) |

## 📝 결론 : Spring Boot의 의존성 주입 구조가 객체 생성 책임과 계층 간 결합도에 미치는 영향

> Spring Boot의 의존성 주입(DI) 구조는 **객체 생성의 책임을 프레임워크(IoC 컨테이너)로 전담**시키고, 컴포넌트 간의 **결합도를 획기적으로 낮춰** 유연하고 테스트하기 쉬운 애플리케이션을 구축할 수 있게 한다.

---

### 1. 스프링 프레임워크와 의존성 주입(DI)

의존성 주입이란, 객체의 **생성과 사용의 관심을 분리**하여 객체 간의 결합도를 낮추고, 유연성 및 재사용성을 높이는 방법이다.

객체를 직접 생성하는 대신 **외부에서 생성된 객체를 주입받음**으로써:
- 객체 간의 결합도 감소
- 코드의 재사용성 및 유지보수성 향상

Spring 프레임워크는 이를 위해 **IoC 컨테이너**를 제공하며, 이 컨테이너는 객체의 생명 주기를 관리하고 필요한 의존성을 자동으로 주입한다.

---

### 2. 의존성 주입 방법과 장단점

#### ⭐ 생성자 주입 (Constructor Injection) — 가장 권장

```java
@Service
public class UserService {
    private final UserRepository userRepository;

    // 생성자 주입
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}
```

- 모든 의존성이 생성자를 통해 주입되므로, 객체 생성 시 **모든 의존성이 준비된 상태**로 시작
- 의존성 주입이 누락되면 **컴파일 시점에 에러 발생** → 런타임 에러 방지
- **불변성 보장**, **순환 의존성 방지**

#### 세터 주입 (Setter Injection)
- **선택적 의존성**을 다룰 때 유용

#### 필드 주입 (Field Injection)
- 테스트 코드나 설정 코드에서 **간편하게 사용** 가능
- ⚠️ **단점** : Spring 컨테이너 없이는 객체 생성 및 의존성 주입이 어려워 **테스트 코드 작성이 어려움**, 순환 의존성 문제 발생 가능

> ✅ 따라서, 스프링 프레임워크에서는 가능한 **생성자 주입을 사용하는 것을 권장**한다.

---

### 3. 실제 프로젝트에서의 의존성 주입 적용 사례

웹 애플리케이션 개발 시, **컨트롤러 → 서비스 → 리포지토리** 계층 간의 의존성을 주입하여 각 계층의 결합도를 낮추고 코드의 재사용성을 높일 수 있다.

```
Controller
    ↓ (DI)
Service
    ↓ (DI)
Repository
    ↓
Database
```

- 각 계층은 **인터페이스를 통한 의존성만**을 가지게 되므로, 구현체가 변경되어도 다른 계층에 영향을 주지 않음
- 테스트 코드 작성 시 실제 구현체 대신 **모의 객체(Mock Object)** 를 주입하여 각 컴포넌트의 독립적인 단위 테스트 가능
- 테스트 대상 외의 의존성을 모의 객체로 대체할 수 있어 **테스트 복잡성 감소, 신뢰성 향상**

---

### 4. 의존성 주입의 중요성 및 주의점

**의존성 주입을 적절히 사용하면:**
- 코드 간의 **결합도 감소**
- 변경에 **유연하게 대응** 가능
- 코드의 **재사용성 향상**
- 애플리케이션의 **테스트 용이성 증가**

**⚠️ 주의해야 할 점:**
- 너무 많은 의존성을 주입하게 되면 **코드 복잡성 증가**
- 각 컴포넌트 간의 의존성이 복잡하게 얽히면 **애플리케이션 구조 이해 및 유지보수가 어려워짐**
- **순환 의존성** 문제 발생 가능

> ✅ 따라서, 의존성 주입을 사용할 때는 **의존성의 관리와 구조의 단순성을 유지**하는 것이 중요하다.
