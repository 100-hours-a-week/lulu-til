## 🚀 시작하기 전에

### 1. `/api` 를 붙인 이유

#### 1.1. 웹 화면(Frontend)과 데이터(Backend)의 주소 구분

하나의 서버나 도메인(`example.com`)에서 웹 페이지도 보여주고 데이터도 처리할 때, 주소가 뒤섞이면 헷갈릴 수 있다.

| 주소 | 역할 |
|------|------|
| `/users` | 유저들이 보는 **사용자 목록 화면 (HTML)** |
| `/api/users` | 프론트엔드 코드가 서버에서 **사용자 데이터 (JSON)** 를 받아오기 위한 주소 |

이렇게 명확하게 구분해 두면 주소만 봐도 **"아, 이건 화면이 아니라 데이터를 주고받는 통로구나"** 라고 직관적으로 알 수 있다.

#### 1.2. 서버 서빙(Routing) 및 배포의 편리함

실무에서는 프론트엔드(Vue.js, React 등)와 백엔드(Spring, Node.js 등)를 같은 서버에 띄우거나, 앞단에 Nginx 같은 웹 서버를 두고 무중단 배포를 하곤 한다.

이때 Nginx 설정에 아래처럼 한 줄로 깔끔하게 처리할 수 있다.

```
"/api가 들어오는 모든 요청" → 백엔드 서버
"나머지 요청"               → 프론트엔드 빌드 파일 (HTML/JS)
```

> ⚠️ 만약 `/api`가 없으면 `/login`, `/users`, `/products` 등 **모든 백엔드 주소를 Nginx에 일일이 등록**해야 하므로 관리가 힘들어진다.

#### 1.3. API 버전 관리 (Version Control)

나중에 서비스가 커져서 로그인 방식을 완전히 뜯어고쳐야 할 때, 기존 앱 사용자들이 구버전 API를 쓰고 있다면 주소를 갑자기 바꿀 수 없다. 이때 `/api` 뒤에 버전을 붙여서 우아하게 해결한다.

| 주소 | 설명 |
|------|------|
| `/api/v1/login` | 기존 로그인 방식 (유지) |
| `/api/v2/login` | 새로 개발한 로그인 방식 |

### 2. 응답 메시지 설계 원칙

| 상황 | 응답 내용 |
|------|-----------|
| ✅ **응답 성공** | `data` 필드 안에 데이터를 담아 전송 |
| ❌ **응답 실패** | `message` 전송 + `null` 값의 `data` 필드 전송 |

**설계 이유**

1. 응답에 성공한 경우, 클라이언트 측에서 메시지를 확인할 필요가 없으므로 `message`를 전송하지 않도록 설계
2. 응답에 실패한 경우, 클라이언트 측에서 실패 이유를 확인할 수 있도록 `message`를 전송하도록 설계
3. `data` 필드가 `null`인지 아닌지에 따라 응답 성공/실패를 구분하기 위해 항상 `data` 필드를 포함

---

### 3. 기본 가정

> 화면 이동은 **클라이언트 측에서 처리**하고, 요청에 대한 **데이터만 서버에서 전송**한다고 가정하였습니다.

## 📌 API 설계

### 1. 로그인

```
POST /api/login
```

**설계 이유**

- `POST /api/users/login`과 고민하였으나, 유저가 아닌 **"로그인 정보"를 "등록"** 하는 것이기 때문에 `POST /api/login`으로 설계
- **JWT 기반** 로그인 동작을 구현했다고 가정하고 로그인 성공 시 토큰을 전송하도록 설계
- 잘못된 request를 보내는 경우와 **로그인 정보에 해당하는 유저가 없는 경우**를 분리하여 설계
- 클라이언트 측에서 올바른 아이디와 패스워드에 대한 검증을 진행한다고 가정하였기 때문에, 유저가 없는 경우만 실패로 간주하여 `"non_existent_user"` 메시지를 응답
- 회원 정보를 조회할 때 필요한 `user_id`를 반환하도록 설계

---

### 2. 회원 가입

```
POST /api/users
```

**설계 이유**

- `POST /api/users/signup`과 고민하였으나, `POST /api/users`만으로 **"회원가입 = 유저 등록"** 이라는 뜻을 표현할 수 있기 때문에 해당 방식으로 설계
- 회원가입 버튼 클릭 시, **이메일과 닉네임에 대한 중복 여부를 체크**한다고 가정
- 잘못된 request인 경우와 **중복된 이메일/닉네임이 존재하는 경우**를 분리하여 응답하도록 설계
- 에러 코드 구분

| 상황 | `code` 필드 값 |
|------|----------------|
| 중복된 이메일 존재 | `-1` |
| 중복된 닉네임 존재 | `-2` |

- `code` 필드의 에러 코드(`-1`, `-2`)는 클라이언트 측에서 **상수로 등록**하여 코드값마다 다른 처리(helper text 표시)를 한다고 가정
- 이메일 중복 여부를 먼저 확인하고, 중복된 경우 바로 응답한다고 가정하여 **이메일과 닉네임이 둘 다 중복된 경우는 분리하여 처리하지 않음**
- 회원가입 시각은 **서버 측에서 측정하여 저장**한다고 가정

---

### 3. 게시글 목록 조회

```
GET /api/posts
```

**설계 이유**

- 게시글 **"들"** 을 조회하는 것이기 때문에 복수형 `posts`로 설계
- 요청 시, 요청하는 `page`와 조회할 게시글 개수(`size`)를 전송하여 서버에서 페이지네이션 처리 가능하도록 설계
- 응답 필드 구성

| 필드 | 설명 |
|------|------|
| `page` | 조회한 게시글들의 페이지 번호 |
| `size` | 조회한 게시글의 개수 |
| `hasNext` | 다음 게시글 존재 여부 |
| `data.posts` | 게시글 데이터 목록 |

- `posts` 필드 안에 `post_id`를 포함한 이유 : 게시글 클릭 시 `post_id`를 통해 해당 게시글을 상세 조회하기 위함

---

### 4. 게시글 상세 조회

```
GET /api/posts/{post_id}
```

**설계 이유**

- `post_id`에 해당하는 게시글을 조회하는 것이기 때문에 Path Variable로 설계
- **잘못된 request인 경우**와 **게시글이 삭제된 경우** (목록 조회까지는 있었지만, 상세 조회 시 삭제된 경우)를 분리하여 응답하도록 설계

---

### 5. 게시글 작성

```
POST /api/posts
```

**설계 이유**

- 게시글을 **생성**하는 것이기 때문에 `POST /api/posts`로 설계
- 게시글 생성 성공 시, **Created를 의미하는 `201` 상태 코드**를 반환하도록 설계
- 게시글 작성 시간 측정 및 좋아요/조회수/댓글 수를 **0으로 세팅**하는 것은 서버 측에서 처리한다고 가정

---

### 6. 게시글 수정

```
PUT /api/posts/{post_id}
```

**설계 이유**

- 게시글 수정은 일부 수정이 아닌 **내용 덮어쓰기의 개념**에 가깝다고 판단하여 `PUT` 메서드로 설계
- 제목과 내용에 대한 유효성 검증은 **클라이언트 측에서 처리**한다고 가정

---

### 7. 게시글 삭제

```
DELETE /api/posts/{post_id}
```

**설계 이유**

- 몇 번 게시글을 삭제한다는 정보만 전달하면 되기 때문에 **Request Body를 작성하지 않음**

---

### 8. 좋아요 등록 / 취소

```
PUT /api/posts/{post_id}/likes
```

**설계 이유**

- 몇 번 게시글의 좋아요를 **"수정"** 하는 것이기 때문에 `PUT` 메서드로 설계
- 좋아요 등록(추가)인지 취소인지만 전달하여 서버에서 좋아요 개수를 업데이트(`+1 / -1`)한다고 가정
- 좋아요 등록/취소 성공 시, `GET /api/posts/{post_id}` API를 호출하여 게시글을 다시 조회(새로고침)한다고 가정
- **잘못된 request인 경우**와 **좋아요를 누른 시점에 이미 게시글이 삭제된 경우**를 분리하여 설계

---

### 9. 조회수 등록

```
PUT /api/posts/{post_id}/views
```

**설계 이유**

- 몇 번 게시글의 조회수를 **"수정"** 하는 것이기 때문에 `PUT` 메서드로 설계
- 좋아요와 달리 조회수는 **취소할 수 없기 때문에 Request Body를 작성하지 않음**
- 서버에서 조회수를 `+1` 업데이트 한다고 가정
- 조회수 추가 성공 시, `GET /api/posts/{post_id}` API를 호출하여 게시글을 다시 조회(새로고침)한다고 가정
- **잘못된 request인 경우**와 **조회수가 추가되는 시점에 이미 게시글이 삭제된 경우**를 분리하여 설계

---

### 10. 댓글 등록

```
POST /api/posts/{post_id}/comments
```

**설계 이유**

- 몇 번 게시글에 댓글을 **생성**하는 것이기 때문에 해당 방식으로 설계
- `comment_id`와 작성 시간은 **서버 측에서 생성**한다고 가정
- 댓글 등록 성공 시, `GET /api/posts/{post_id}` API를 호출하여 게시글을 다시 조회(새로고침)한다고 가정
- **잘못된 request인 경우**와 **댓글이 등록되는 시점에 이미 게시글이 삭제된 경우**를 분리하여 설계

---

### 11. 댓글 수정

```
PUT /api/comments/{comment_id}
```

**설계 이유**

- 몇 번 댓글을 수정하는 것이기 때문에 `PUT /api/comments/{comment_id}`로 설계
- `comment_id`로 댓글을 구분할 수 있기 때문에 **`post_id`는 URL에 포함시키지 않음**
- 댓글 수정 성공 시, `GET /api/posts/{post_id}` API를 호출하여 게시글을 다시 조회(새로고침)한다고 가정
- **잘못된 request인 경우**와 **댓글이 수정되는 시점에 이미 게시글이 삭제된 경우**를 분리하여 설계

---

### 12. 댓글 삭제

```
DELETE /api/comments/{comment_id}
```

**설계 이유**

- 몇 번 댓글을 삭제하는 것이기 때문에 `DELETE /api/comments/{comment_id}`로 설계
- 댓글 삭제 성공 시, `GET /api/posts/{post_id}` API를 호출하여 게시글을 다시 조회(새로고침)한다고 가정
- **잘못된 request인 경우**와 **댓글이 삭제되는 시점에 이미 게시글이 삭제된 경우**를 분리하여 설계

---

### 13. 회원 정보 조회

```
GET /api/users/{user_id}
```

**설계 이유**

- `user_id`에 해당하는 회원을 조회하는 것이기 때문에 `GET /api/users/{user_id}`로 설계

---

### 14. 회원 정보 수정

```
PUT /api/users/{user_id}
```

**설계 이유**

- `user_id`에 해당하는 회원을 수정하는 것이기 때문에 `PUT /api/users/{user_id}`로 설계
- 회원 정보는 **닉네임과 프로필 이미지만 수정 가능**하기 때문에 해당 정보만 Request Body에 포함

---

### 15. 회원 탈퇴

```
DELETE /api/users/{user_id}
```

**설계 이유**

- `user_id`에 해당하는 회원을 삭제하는 것이기 때문에 `DELETE /api/users/{user_id}`로 설계

---

### 16. 비밀번호 수정

```
PUT /api/users/{user_id}/password
```

**설계 이유**

- `user_id`에 해당하는 회원의 비밀번호를 수정하는 것이기 때문에 `PUT /api/users/{user_id}/password`로 설계
- 회원 정보 수정 API와 공통으로 묶어 처리하는 것을 고민했지만, **중복으로 데이터가 업데이트되는 것을 방지**하기 위해 API를 분리
