#### | Controller

**클러스터의 현재 상태를 지속적으로 모니터링하고, 사용자가 정의한 목표 상태(Desired State)와 일치하도록 조치를 취하는 자동화된 관리 루프**

```
Kubernetes Objects                                   System Resources
┌─────────────┐                                       ┌─────────────┐
│    Pods     │◀───────── Update ────┐   ┌── Watch ──▶│  Containers │
│  Services   │                       │   │            │   Volume    │
└─────────────┘                       ▼   │            │  iptables   │
       │                          Controller           │   rules     │
       └──────────── Watch ───────────┘   └── Update ─▶└─────────────┘
```

##### 컨트롤러가 없다면?

지금까지는 파드로 직접 작업을 수행해왔다 — 파드 안에 컨테이너를 배치하고, 그 파드를 직접 생성·수정·삭제하는 방식이다.

- 하지만 **파드는 일시적인(ephemeral) 리소스**다.
  - 애플리케이션 내부에서 오류가 발생하거나
  - 노드가 죽거나
  - 업데이트가 필요하거나
  - 그 외 다양한 이유로 파드는 언제든 교체되거나 삭제되는 것이 기본값이다.
- 이런 파드를 사람이 직접 생성해서 관리한다면?
  - 파드가 사라지면 그걸 **직접 다시 만들어야** 한다.
  - 파드가 사라진 사실 자체를 **알아채기 어렵다.**
  - 파드를 자동으로 다시 만들어줄 **상위 관리자가 없는 상태**가 된다.

> ⚠️ 실제 현업에서 쿠버네티스를 사용한다고 하면, 파드를 직접 생성하는 경우는 거의 없다. **Deployment, StatefulSet, Job, CronJob** 같은 컨트롤러가 파드를 대신 관리한다.

##### Desired State vs Current State

컨트롤러를 이해하려면 "원하는 상태"와 "현재 상태" 두 가지 개념을 먼저 알아야 한다.

| 구분 | 설명 |
|---|---|
| **Desired State (원하는 상태)** | 운영자가 컨트롤러의 YAML 파일(`spec.replicas: 3` 등)에 적어 둔, 클러스터가 도달해야 할 목표 상태 |
| **Current State (현재 상태)** | 클러스터에 실제로 존재하는 상태 (예: `spec.replicas: 3`인데 실제로는 파드가 2개만 떠 있는 상태) |

두 상태가 일치하지 않으면(예: 원하는 파드 수 3개, 현재 파드 수 2개) 컨트롤러가 이 차이를 감지하고 부족한 파드를 새로 생성한다. 반대로 파드가 초과되어 있다면 삭제하기도 한다.

##### Control Loop — Reconciliation

```
        ┌──────────────────────┐
        │   두 상태가 일치하는가?  │
        └──────────┬───────────┘
             예 ───┴─── 아니오
              │              │
       현재 상태 유지    리소스 생성·수정·삭제
              │              │
              └──────┬───────┘
                     ▼
      Desired State ⇄ Controller ⇄ Current State
```

컨트롤러는 이 **Control Loop**를 주기적으로 돌면서 두 상태를 비교하고, 다르면 맞춰주는 동작을 반복한다 — 이 과정을 **Reconciliation(조정)**이라고 부른다.

> 💡 백엔드 개발에서 말하는 "Controller"(라우팅, 요청 제어 역할)와는 완전히 다른 개념이다. k8s의 Controller는 **상태를 맞추는 자동화 루프**를 의미한다.

##### Controller의 핵심 역할

**1. 원하는 상태를 유지**

```yaml
spec:
  replicas: 3
```

- 이건 "파드를 3개 만들어라"라는 **일회성 명령이 아니다.**
- "앞으로도 클러스터 내에 이 파드를 3개로 **유지**해라"라는 **선언적 명령**이다.
- 파드가 하나 삭제되면 → 컨트롤러가 제어 루프로 감지 → 다시 원하는 상태로 맞춘다.

**2. Auto Healing (자동 복구)**

```
정상 상태
Pod A / Pod B / Pod C
   ↓
Pod B 삭제
   ↓
Pod A / Pod C
   ↓
Controller가 부족한 Pod 감지
   ↓
Pod D 생성
   ↓
다시 Pod 3개 유지
```

정상 상태가 아니어서 삭제된 파드를 컨트롤러가 감지하고, 다시 원하는 상태로 만들기 위해 새 파드를 생성한다 — 이게 자동 복구다.

**3. 자동 확장/축소**

```
replicas: 3
   ↓
replicas: 5로 변경
   ↓
Controller가 Pod 2개 추가 생성
```

사용자가 원하는 `replicas` 숫자를 바꾸면, 컨트롤러가 자동으로 파드를 늘리거나 줄인다.

**4. 쿠버네티스의 선언적 관리**

| 방식 | 특징 |
|---|---|
| **명령형(Imperative)** | "Pod을 하나 생성한다. Pod이 죽으면 다시 생성한다. Pod 수가 부족하면 추가한다..." → 상황에 따라 사람이 직접 명령을 내림 |
| **선언형(Declarative)** | "나는 파드가 3개면 좋겠어" → 최종 상태만 선언하면, 이후의 생성·삭제·복구는 컨트롤러가 알아서 수행 |

> 운영자는 상태를 **정의**만 하면 되고, k8s 시스템이 알아서 그 상태를 맞춰간다.

##### Pod 직접 생성 vs Controller로 Pod 관리

| 구분 | Pod 직접 생성 | Controller로 Pod 관리 |
|---|---|---|
| Pod 생성 | 사용자가 직접 생성 | Controller가 Pod 템플릿으로 생성 |
| Pod 삭제 시 | 그대로 사라짐 | 새로운 Pod 자동 생성 |
| 복제본 유지 | 지원하지 않음 | 지정한 수량 유지 |
| 확장·축소 | Pod을 직접 추가·삭제 | 목표 수량만 변경 |
| 장애 복구 | 운영자가 직접 처리 | Controller가 자동 처리 |
| 운영 적합성 | 단순 테스트·학습 | 지속적으로 운영할 애플리케이션 |

##### Controller YAML 구조

```yaml
spec:
  replicas: 3

  selector:
    matchLabels:
      app: web

  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:1.27
```

| 필드 | 설명 |
|---|---|
| **replicas** | 유지할 Pod의 수 |
| **selector** | Controller가 관리할 Pod을 찾는 조건 |
| **template** | 새로운 Pod을 만들 때 사용할 설계도 |

```
Selector → 관리 대상 Pod 검색 → 현재 Pod 수 확인 → Pod 부족 → Template을 사용하여 새 Pod 생성
```

> 💡 **Label과 Selector가 핵심이다.** 컨트롤러는 모든 파드를 관리하지 않는다. 자신이 관리할 파드의 라벨을 정해두고, Selector로 그 라벨을 가진 파드를 검색한다.

##### Controller는 모든 걸 직접 수행할까?

컨트롤러는 모든 작업을 직접 수행하지 않는다. 주로 **API Server를 통해 클러스터 상태를 확인**하고, 필요한 리소스의 생성·수정·삭제를 **요청**할 뿐이다.

```
사용자 ── YAML 적용 ──▶ API Server ── 상태 저장 ──▶ etcd
                            ▲                          │
                            │                          │(watch)
                     필요한 변경 요청                  Controller
                            │                          │
              새로운 리소스 생성 또는 수정 ◀────────────┘
```

파드가 부족할 때 실제 흐름:

1. 컨트롤러가 현재 노드 상태 / 현재 파드 수 / 응답을 확인
2. "파드가 부족한데?" → API Server에 Pod Template대로 "이렇게 만들어주세요" 요청
3. API Server → **Scheduler**에게 "이 파드를 배치할 노드를 골라주세요" 요청 (etcd의 노드 상태 참고)
4. Scheduler가 워커노드 1번을 추천
5. API Server → 워커노드 1번의 **kubelet**에게 명령 전달
6. kubelet → **Container Runtime**에게 "컨테이너 만들어주세요" 요청

##### Controller 종류

| Controller | 주요 목적 |
|---|---|
| **ReplicaSet** | 지정한 수의 동일한 Pod 복제본 유지 |
| **Deployment** | ReplicaSet을 관리하며 배포, 업데이트, 롤백 수행 |
| **StatefulSet** | 고유한 이름과 스토리지가 필요한 상태 기반 애플리케이션 관리 |
| **DaemonSet** | 조건을 만족하는 각 노드에 Pod을 실행 |
| **Job** | 완료를 목적으로 하는 일회성 작업 실행 |
| **CronJob** | 정해진 일정에 따라 Job을 반복 실행 |

관리 대상이나 목적은 다르지만, **기본 원리는 동일**하다 — Desired State를 정의하고, Control Loop로 Current State와 비교해서 맞춘다.

##### 왜 컨트롤러가 필요한가 (Stateless 애플리케이션 관점)

- k8s를 사용한다는 건 보통 서비스 규모가 커졌다는 의미다.
- 이벤트 발생 시 트래픽의 고점과 저점 차이가 크고, WAS의 처리량이 급격히 늘어나는 경우가 많다.
- 파드 몇몇에 장애가 발생할 수 있다.
  - **리소스 문제**: 로그 적체, 메모리 부족
  - **애플리케이션 문제**: 데드락(코드 레벨), 커넥션 점유, DB 커넥션 끊김
- → 이런 상황을 사람이 일일이 대응하기는 불가능하므로 **컨트롤러**가 필요하다.

##### Q&A — 컨트롤러는 클러스터 단위? NS 단위?

- 종류에 따라 제어 범위가 다르다.
  - **ReplicaSet, Deployment** → **네임스페이스(NS) 단위**로 관리
  - **Node Controller** → **클러스터 단위**로 관리

##### Q&A — 컨트롤러 매니저는 뭘 하나?

- **컨트롤러**는 실제 업무(Reconciliation)를 담당한다.
- **컨트롤러 매니저(kube-controller-manager)**는 여러 컨트롤러를 모아서 운영·실행하는 것을 매니지먼트해주는 상위 프로세스다.

#### | ReplicaSet

**특정 수의 Pod 복제본을 항상 유지하는 데 사용되는 컨트롤러**

라벨 셀렉터로 관리할 Pod를 찾아, 사용자가 지정한 개수의 Pod가 항상 실행되도록 유지하는 Controller다.

```
원하는 상태: Pod 3개
현재 상태: Pod 3개
        ↓
    Pod 하나 삭제
        ↓
현재 상태: Pod 2개
        ↓
ReplicaSet이 차이 감지
        ↓
   새로운 Pod 1개 생성
        ↓
현재 상태: Pod 3개
```

##### ReplicaSet은 언제 사용할까?

**Stateless Application에 적합**

```
사용자 요청
   │
   ▼
Service / Load Balancer  ──▶  Pod 1 / Pod 2 / Pod 3 (동일한 요청 처리 가능)
                                     │
                                     ▼
                    공유 저장소 / 외부 상태 저장소
                    (Database / Redis / Object Storage)
```

- 내부에 상태(state)를 가지고 있지 않기 때문에, 어떤 파드가 요청을 받아도 동일하게 처리 가능
- 상태는 외부 저장소(DB, Redis 등)에 접근해서 관리
- 애플리케이션이 여러 Pod로 배포되어도 동일한 역할을 수행하며, **Pod의 이름은 중요하지 않다**
- ReplicaSet은 Label ← Selector로 대상을 찾고, `replicas`에 지정된 숫자와 **개수만 비교**하기 때문에 이런 구조에 잘 맞는다

**반대로 Stateful Application에는 부적합**
- Pod 안에 고유한 ID가 있어야 하고, 상태(state)가 하나의 Pod 안에 컨테이너와 함께 들어있기 때문에
- Pod의 역할도, 이름도 모르고 오직 "개수"만 신경 쓰는 ReplicaSet으로는 관리가 불가능
- 이런 경우엔 **StatefulSet** 컨트롤러를 사용한다

##### ReplicaSet YAML 구조

```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: web-replicaset

spec:
  replicas: 3

  selector:
    matchLabels:
      app: web

  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: web
          image: nginx:1.27
          ports:
            - containerPort: 80
```

##### ⚠️ 라벨 사용 시 주의점 — 기존 Pod도 관리 대상이 될 수 있다

이미 아래처럼 `app: web` 라벨을 가진 파드가 떠 있는 상태에서,

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: existing-pod
  labels:
    app: web
spec:
  containers:
    - name: web
      image: nginx:1.27
```

같은 `app: web`을 selector로 갖는 ReplicaSet을 만들면?

```yaml
selector:
  matchLabels:
    app: web
```

→ **기존에 있던 파드도 그대로 관리 대상에 포함된다.** 컨트롤러는 라벨만 보고 판단하기 때문이다.

> ⚠️ 이게 왜 위험하냐면, **컨트롤러가 삭제되면 관리하고 있던 파드들도 함께 삭제되기 때문**이다. 의도치 않게 기존 파드를 편입시켰다가 통째로 날려버릴 수 있다.

##### ReplicaSet의 한계

ReplicaSet은 Pod의 **복제본 수 유지에만 관심**이 있다.

```
# image: nginx:1.27
image: nginx:1.28
```

- 이미지 버전을 바꿔서 `apply` 해도, **기존에 실행 중이던 파드가 자동으로 새 버전으로 재배포되지는 않는다.**
- 새롭게 만들어지는 파드만 새 버전의 템플릿을 사용할 뿐, 이미 떠 있는 파드는 그대로 유지된다.
- 파드가 언젠가 재생성될 때까지 기다리는 방식으로는 운영자가 배포를 컨트롤하기 어렵다.
- 실제 애플리케이션을 운영하려면 복제본 수 유지 이상의 기능이 필요하다:
  - 새로운 버전 배포
  - 기존 파드와 새 파드의 점진적 교체
  - 배포 진행 상태 확인
  - 이전 버전으로 롤백
  - 배포 이력 관리
  - → 이걸 해결하는 게 **Deployment**다 (내부적으로 ReplicaSet을 갖고 있으면서, 배포 관련 기능을 추가한 상위 컨트롤러)

##### 헷갈릴 수 있는 부분 Q&A

**1. ReplicaSet이 파드를 "수리"한다?**
→ 아니다. 애초에 쿠버네티스는 고장난 파드를 고쳐서 쓰지 않는다. 문제가 있으면 **삭제하고 새로 만든다.**

**2. ReplicaSet은 파드가 부족할 때만 동작한다?**
→ 아니다. **수를 "맞추는" 것**이지 늘리는 것만 하는 게 아니다. 파드가 원하는 수보다 많으면 초과된 파드를 제거하기도 한다.

**3. Probe와 Controller는 같은 역할을 하나?**
→ 관리하는 **레벨이 다르다.**
| 구분 | 복구 레벨 | 예시 |
|---|---|---|
| **Probe** (예: Liveness Probe) | **Container 레벨** 복구 | `kubectl get pods -o wide` → 같은 pod id, restart count만 증가 |
| **Controller** | **Pod 레벨** 복구 | Pod가 삭제되면 새로운 Pod(새로운 pod id)를 만듦 |

보통 이 둘은 **함께 사용**된다.

**4. ReplicaSet이 트래픽을 분산하는가?**
→ 아니다. ReplicaSet은 Label로 매치되는 파드를 보고, 복제본 수가 부족하면 만드는 역할만 한다 (`template`으로 `label: app=web`인 파드를 생성). **트래픽 분산은 Service의 역할**이다 (ClusterIP Service가 `label: app=web`을 가진 파드들을 EndpointSlice로 관리하며 분산).

##### 정리

| 질문 | 답변 |
|---|---|
| ReplicaSet이란? | 지정된 수의 Pod 복제본을 유지하는 Controller |
| 원하는 Pod 수는 어디에 작성? | `spec.replicas` |
| 관리할 Pod은 어떻게 찾는가? | `spec.selector`와 Pod Label |
| Pod이 부족하면? | `spec.template`으로 새 Pod 생성 |
| Pod이 많으면? | 초과된 Pod 제거 |
| 삭제된 Pod 자체가 복구되는가? | 아니요. 새로운 Pod이 생성됨 |
| 트래픽을 분산하는가? | 아니요. 트래픽 전달은 Service의 역할 |
| 실무에서 직접 사용하는가? | 일반적으로 Deployment를 통해 간접적으로 사용 |

---

#### | Deployment

**ReplicaSet을 관리하여 Pod의 개수뿐 아니라 애플리케이션의 배포, 업데이트, 롤백까지 자동으로 관리하는 상위 Controller**

- ReplicaSet ← 같은 Pod 개수를 유지하는 컨트롤러
- Deployment
  - ReplicaSet을 이용하고
  - 배포·업데이트·롤백을 자동으로 관리하는 컨트롤러
  - `pod.yaml`, `replicaset.yaml`을 따로 만들지 않고 `deployment.yaml` 파일 하나로 관리

##### 왜 ReplicaSet만으로는 부족한가

```
v1 애플리케이션 실행 중
   ↓
새로운 기능 개발
   ↓
v2 이미지 배포
   ↓
기존 Pod을 v2 Pod으로 교체
```

이걸 직접 한다면?
1. v1 ReplicaSet 축소
2. v2 ReplicaSet 생성
3. v2 ReplicaSet 확장
4. 배포 상태 확인
5. 문제 발생 시 v1 복구

→ **Deployment는 이 과정을 자동화해준다.**

```
1. Deployment의 이미지 버전 변경 (yaml)
   ↓
2. Deployment가 새 ReplicaSet 생성
   ↓
3. 새 ReplicaSet 확장
   ↓
4. 기존 ReplicaSet 축소
   ↓
5. 새로운 버전으로 교체
```

##### Deployment의 실제 구조

Deployment가 파드를 **직접** 관리하지는 않는다. 실제로는 계층 구조로 위임한다.

```
Deployment   (배포 관련 컨트롤러)
    │
    ▼
ReplicaSet   (복제본 컨트롤러)
    │
    ▼
Pod Template → Pod → Container
```

##### Deployment를 사용하면 해결되는 것들

| 운영 문제 | 해결 방법 |
|---|---|
| Pod을 여러 개 유지해야 함 | ReplicaSet을 통해 복제본 유지 |
| 새로운 버전을 배포해야 함 | 새로운 ReplicaSet 생성 |
| 서비스 중단을 줄여야 함 | RollingUpdate로 순차 교체 |
| 배포에 문제가 발생함 | 이전 Revision으로 롤백 |
| Pod 수를 변경해야 함 | `replicas` 값을 변경해 확장·축소 |
| 배포 진행 상태를 확인해야 함 | Rollout 상태 제공 |
| 이전 버전 기록이 필요함 | 이전 ReplicaSet을 Revision으로 보관 |

##### Deployment의 생성 흐름

사용자가 `deployment.yaml`을 만들고 `apply` 명령을 실행하면:

```
1. 사용자가 Deployment 생성
   ↓
2. Deployment가 ReplicaSet 생성
   ↓
3. ReplicaSet이 Pod 생성
   ↓
4. Pod 내부에서 컨테이너 실행
```

```bash
# shell master node
kubectl apply -f web-deployment.yaml
```

```
# 생성되는 리소스
Deployment: web-deployment
    └── ReplicaSet: web-deployment-7f8b9c
            ├── Pod: web-deployment-7f8b9c-a1b2c
            ├── Pod: web-deployment-7f8b9c-d3e4f
            └── Pod: web-deployment-7f8b9c-g5h6i
```

##### Deployment YAML 구조

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-deployment

spec:
  replicas: 3

  selector:
    matchLabels:
      app: web

  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: web
          image: nginx:1.27
          ports:
            - containerPort: 80
```

##### Deployment로 업데이트하기

이미지 버전을 올린다고 가정해보자.

```yaml
spec:
  containers:
    - name: web
      image: nginx:1.27  # 여기 부분이 변경된다
      ports:
        - containerPort: 80
```

```bash
kubectl apply -f nginx-dep.yaml
```

- 변경된 YAML 파일을 apply하면
- Deployment가 Pod Template의 변경 여부를 확인하고
- 템플릿 안의 내용이 변경되면 **Deployment는 새로운 ReplicaSet을 생성**한다

##### 템플릿이 변경되면 항상 ReplicaSet이 새로 만들어질까?

`deployment.yaml`이 어떻게 변하든 항상 새로운 ReplicaSet이 만들어지는 것은 아니다.

**핵심 기준: Pod Template이 변경되었는가**

새로운 ReplicaSet이 생성되는 대표적인 변경 사항:
- 컨테이너 이미지 변경
- 환경 변수 변경
- Pod Label 변경
- ConfigMap 또는 Secret 참조 변경
- Volume 설정 변경
- Probe 설정 변경
- 리소스 requests와 limits 변경

반대로 **복제본 수(`replicas`) 변경은 ReplicaSet에 영향을 주지 않는다.**
- `replicas: 3 → 5`로 바꿔도 기존 ReplicaSet은 그대로 유지되고 복제본 수만 조정된다.

##### Deployment 배포 전략

| 전략 | 동작 |
|---|---|
| **RollingUpdate** | 기존 Pod과 새로운 Pod을 점진적으로 교체 |
| **Recreate** | 기존 Pod을 모두 삭제한 후 새로운 Pod 생성 |

**1. RollingUpdate**

기존 Pod을 한 번에 모두 종료하지 않고, 새로운 Pod로 조금씩 교체하는 배포 전략이다.

```
기존 버전 Pod
v1  v1  v1
      ↓
새 Pod 하나 생성
v1  v1  v1  v2
      ↓
기존 Pod 하나 제거
v1  v1  v2
      ↓
반복
v1  v2  v2
      ↓
배포 완료
v2  v2  v2
```

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-deployment
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 1
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    # ...
```

| 옵션 | 의미 |
|---|---|
| **maxSurge** | 업데이트 중 원하는 복제본 수를 초과해서 추가로 생성할 수 있는 파드 수. 예: `replicas: 3`, `maxSurge: 2`면 롤링 업데이트 중 파드 최대 개수는 5까지 허용 |
| **maxUnavailable** | 업데이트 중 사용할 수 없는 상태를 허용하는 최대 파드 수. 예: 파드가 3개이고 `maxUnavailable: 1`이면, 롤링 업데이트 중 최소 2개 이상은 항상 사용 가능해야 함 |

**RollingUpdate가 항상 무중단일까?**

→ 그렇지 않다. RollingUpdate는 어디까지나 **서비스 중단을 "줄이는" 전략**이지, 무조건 무중단을 보장하지는 않는다. 진짜 무중단 배포를 하려면 아래를 함께 고려해야 한다.

- 새로운 Pod이 정상적으로 실행되어야 함 → **Probe 설정**
- 애플리케이션이 요청을 받을 준비가 되어 있어야 함 → **Probe 설정**
- Readiness Probe가 적절히 설정되어야 함 → **Probe 설정**
- Service가 정상 Pod만 대상으로 요청을 전달해야 함 → **Probe 설정**
- 기존 버전과 새 버전이 **동시에 동작**할 수 있어야 함
  - **데이터베이스 스키마가 양쪽 버전과 호환되어야 함**
  - 무중단 배포는 가능한데, v1/v2를 동시에 운영해야 하는 상황이 생김
  - 클라이언트(핸드폰 앱)는 버전에 따라 v1 API, v2 API를 각각 호출할 수도 있다
  - 실제로는 배포 후 2주 내에 전체 유저 중 80%가 업데이트를 따라오지만, 1년이 지나도 5%는 절대 업데이트하지 않는 경우도 생긴다

**2. Recreate**

기존 파드를 모두 날리고, 새로운 파드를 생성하는 배포 전략이다.

```
기존 상태
v1  v1  v1
   ↓
기존 Pod 모두 삭제
 —   —   —
   ↓
새로운 Pod 생성
v2  v2  v2
```

→ **서비스 중단이 당연히 발생한다.**

**Recreate를 고려하는 경우**
- 개발 또는 테스트 환경
- 짧은 서비스 중단이 허용되는 환경
- 두 버전이 동시에 실행되면 안 되는 애플리케이션
- 동일 리소스에 대한 동시 접근이 허용되지 않는 경우
- 기존 버전을 완전히 종료한 뒤 새 버전을 실행해야 하는 경우

**RollingUpdate vs Recreate**

| 구분 | RollingUpdate | Recreate |
|---|---|---|
| 기존 Pod 처리 | 점진적으로 삭제 | 모두 먼저 삭제 |
| 새로운 Pod 생성 | 점진적으로 생성 | 기존 Pod 삭제 후 생성 |
| 서비스 중단 가능성 | 상대적으로 낮음 | 높음 |
| 버전 공존 | 배포 중 공존 | 공존하지 않음 |
| 운영 복잡도 | 상대적으로 높음 | 단순함 |
| 주요 사용 환경 | 일반적인 운영 환경 | 개발·테스트 또는 특수한 애플리케이션 |

##### Deployment에서 확인해야 하는 항목

```bash
kubectl get deployment
```

```
NAME             READY   UP-TO-DATE   AVAILABLE   AGE
web-deployment   3/3     3            3           10m
```

| 항목 | 설명 |
|---|---|
| **READY** | 준비된 Pod 수 / 원하는 Pod 수 |
| **UP-TO-DATE** | 최신 Pod Template으로 생성된 Pod 수 |
| **AVAILABLE** | 사용 가능한 상태의 Pod 수 |
| **AGE** | Deployment가 생성된 이후 경과 시간 |

##### Deployment 사용 시 ReplicaSet이 여러 개 남는 이유

```bash
kubectl get replicaset
```

```
NAME                        DESIRED   CURRENT   READY
web-deployment-7f8b9c       0         0         0
web-deployment-84d7f6       0         0         0
web-deployment-95e8a7       3         3         3
```

- 업데이트할 때마다 Deployment는 **새로운 ReplicaSet을 만들고, 기존 ReplicaSet은 바로 삭제하지 않는다.**
- **롤백을 위해서** 이전 ReplicaSet들을 Revision 기록으로 보관해두는 것이다 (그러니까 굳이굳이 수동으로 날리지 말 것).

##### 실무에서 Deployment 사용하기

**1. Stateless App은 Deployment를 사용해야 한다**
- WAS, WS, FE 서버, auth 서버 등

**2. 이미지에 `latest` 태그를 사용하면 안 된다**
- `latest` 태그로는 이미지가 바뀐 것을 Deployment가 감지할 수 없기 때문
- `sha` 값이나 semantic version(`v1.0.3`) 등 명시적인 태그를 사용해야 한다

**3. Controller는 Probe와 함께 사용해야 한다**
- 복제본으로 안정성을 확보해도, 새로 만들어진 Pod가 바로 트래픽을 받을 준비가 된 건 아니다
- Spring 컨테이너처럼 준비될 때까지 시간이 걸리는 애플리케이션은 **Readiness Probe**와 반드시 함께 써야 한다

**4. 배포 후에는 배포 상태를 확인해야 한다**
- YAML이 정상적으로 변경되어도 배포가 항상 성공하는 것은 아니다
- 헬스 체크(Probe)를 잘 설정해두거나, 아래 명령으로 롤아웃 상태를 직접 확인한다

```bash
kubectl rollout status deployment/web-deployment
```

> 💡 참고로 배포에 문제가 생겼을 땐 `kubectl rollout undo deployment/web-deployment` 명령으로 직전 Revision으로 즉시 롤백할 수 있다.

##### 정리

| 정리 | 개념 |
|---|---|
| Deployment란? | ReplicaSet을 관리해 애플리케이션의 배포와 업데이트를 자동화하는 Controller |
| Deployment가 직접 관리하는 주요 리소스는? | ReplicaSet |
| ReplicaSet은 무엇을 관리하는가? | Pod |
| ReplicaSet과 가장 큰 차이는? | Deployment는 업데이트, Rollout, 롤백을 관리 |
| 이미지가 변경되면 어떻게 되는가? | 새로운 ReplicaSet을 생성하여 Pod을 교체 |

### 🗨️ 오늘의 회고

- 10주차 위클리 챌린지를 진행하며 Deployment를 먼저 사용했는데, 알고 보니 그건 11주차 위클리 챌린지 과제였고 아직 배우지 않은 부분이었다. 서칭만으로 학습하고 진행했던 부분이라 순서가 꼬였는데, 앞으로는 아직 배우지 않은 개념을 먼저 가져다 쓰기보다 진도에 맞춰 꼼꼼히 확인하면서 진행해야겠다는 걸 다시 느꼈다.
- 오늘은 그 연장선에서 **Controller**의 개념부터 제대로 짚었다. "컨트롤러가 없으면 파드를 사람이 직접 만들고, 사라지면 또 직접 만들어야 한다"는 설명이 와닿았다 — 결국 컨트롤러는 **Desired State와 Current State를 계속 비교해서 맞추는 자동화 루프**라는 게 핵심이었다. Control Loop, Reconciliation 같은 용어도 그림으로 정리하니 훨씬 명확해졌다.
- **ReplicaSet**은 "복제본 개수 유지"에만 관심이 있다는 점이 인상적이었다. 개수만 보고 라벨로 대상을 찾다 보니, 기존에 같은 라벨을 가진 파드가 있으면 의도치 않게 관리 대상에 편입될 수 있다는 주의사항도 실무적으로 꼭 기억해야겠다. 특히 "컨트롤러가 삭제되면 관리하던 파드도 같이 삭제된다"는 부분은 무심코 지나치면 사고로 이어질 수 있을 것 같다.
- ReplicaSet만으로는 이미지 버전을 바꿔도 기존 파드가 자동으로 교체되지 않는다는 한계가, 왜 **Deployment**가 필요한지로 자연스럽게 이어졌다. Deployment가 내부적으로 새로운 ReplicaSet을 만들고 기존 ReplicaSet을 점진적으로 축소하는 방식으로 배포를 관리한다는 구조(Deployment → ReplicaSet → Pod → Container)를 이해하니, 지난주에 배운 Service·Probe와도 자연스럽게 연결됐다.
- **RollingUpdate가 항상 무중단 배포를 보장하지는 않는다**는 점도 새롭게 배운 부분이다. Readiness Probe 설정, 신구 버전 동시 운영, DB 스키마 호환성까지 함께 고려해야 진짜 무중단 배포가 가능하다는 걸 알고 나니, 지난주 "Readiness Probe가 없으면 무중단 배포를 할 수 없다"던 문장이 오늘 배운 Deployment 내용과 정확히 맞물렸다.
