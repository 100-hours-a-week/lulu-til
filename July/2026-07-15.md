#### | Service

**파드(Pod) 집합에 대한 네트워크 엔드포인트를 생성하고, 클러스터 내부 또는 외부에서 해당 파드들에 안정적으로 접근할 수 있도록 지원하는 쿠버네티스 리소스**

```
                        Deployment
                            │
                        ReplicaSet
                            │
        ┌───────────────────┼───────────────────┐
   Pod (app:A)          Pod (app:A)          Pod (app:A)
   10.10.10.1           10.10.10.2           10.10.10.3

Selector(app=A) → Service(1.1.1.1) ──▶ 위 세 파드로 트래픽 전달
```

##### 왜 필요한가

- **파드는 항상 IP가 변경**되고, 실행되고 있는 노드도 완벽하게 예측할 수 없다.
- 파드는 언제든지 생성되고, 언제든지 삭제된다.
- 그래서 클라이언트나 내부 서비스가 **특정 파드의 IP로 직접 접근하는 것은 안정적이지 않다** — IP가 계속 바뀌기 때문이다.
- **Service는 이런 파드들 앞에서 오랫동안 유지되는 가상의 IP 주소와 DNS 이름을 제공**한다. 즉 파드의 IP 문제를 서비스가 해결해준다.

##### 서비스로 내부 연결을 해결하는 방법

```
frontend (user browser)
└── Ingress (https://api.domain.com/users/me)  (LB, RP)
      └── backend-service  (내가 요청을 전달해야 하는 파드들)
              ├── 10.244.1.10
              ├── 10.244.2.15
              └── 10.244.2.16
```

파드가 교체되더라도 `backend-service:8080`만 부르면 된다. 쿠버네티스에서는 서비스가 **일관된 DNS 이름**을 갖고 있어서, 클러스터 내부에서는 IP 주소를 직접 쓰지 않고 **서비스 이름으로 접근**해서 트래픽을 전달한다.

##### 서비스의 기본 역할

> ⚠️ 서비스의 기본 역할은 "외부에 공개하는 엔드포인트"가 아니라, **클러스터 내부에서 계속 변경되는 파드들에게 안정적으로 접근하게 하는 것**이다. 흔히 외부 요청을 처리하는 용도로 오해하기 쉽지만 그렇지 않다.

```
frontend Pod
   │
   │  backend-service (http://backend-service:8080)
   ▼
ClusterIP Service
   ├── backend Pod A
   └── backend Pod B
```

> 💡 **비유하자면 회사의 전화 체계**와 비슷하다. 개발팀 대표 번호(02-000-0100)를 두고, 사원 1번(02-000-0101), 사원 2번(02-000-0102) 같은 사내 내선을 관리한다고 해보자. 사원이 퇴사하고 입사할 때마다 번호가 바뀐다면 매번 그 개인 번호로 요청하기가 어렵다. 그래서 그냥 **개발팀 대표 번호**로 전화한다 — 대표 번호 뒤에서 실제로 응대하는 사람은 바뀌어도 상관없다. Service가 바로 이 "대표 번호" 역할을 한다.

##### Service의 핵심 역할

**1. 안정적인 접근 주소 제공**

일반적으로 가장 많이 사용하는 서비스 타입이 **ClusterIP Service**다. 클러스터 내부에서 사용할 수 있는 가상의 IP가 만들어진다.

```
# Backend Service
http://backend-service:8080
10.244.6.4
```

보통은 이 IP를 직접 쓰지 않고, 대부분 **주소(서비스 이름)** 를 사용해서 연결한다.

**네임스페이스**는 클러스터 안에서 분리되어 있는 가상의 공간이다.
- 같은 네임스페이스에서는 `http://backend-service:8080`처럼 호출한다.
- 다른 네임스페이스에서는 `http://backend-service.default:8080`처럼 네임스페이스를 명시해야 한다.

**서비스의 DNS 구조**

```
[서비스명].[네임스페이스].svc.cluster.local
```

**2. 여러 파드를 하나의 대상으로 추상화**

```
backend-a: 10.244.1.10
backend-b: 10.244.2.15
backend-c: 10.244.2.16

# 여러 개의 주소를 갖고 있는 파드들을
# 하나의 서비스로 추상화

# 클라이언트 입장에서는
backend-service:8080
```

**3. 대상 Pod의 변경을 자동으로 반영**

Service는 **Selector**를 통해 대상을 결정한다.
- 조건(label AND 연산)에 맞는 새로운 파드가 생성되면, 자동으로 Service의 대상에 포함된다.
- 조건에 맞던 파드가 삭제되어 조건이 더 이상 맞지 않게 되면, 서비스 대상에서 자동으로 제외된다.

**4. 여러 파드로 트래픽을 전달할 수 있다** — 자연스럽게 간단한 **로드밸런싱**이 된다.

```
frontend Pod
   │
   │  backend-service
   ▼
ClusterIP Service
   ├── backend Pod A
   ├── backend Pod B
   └── backend Pod C
```

> 💡 단, **L4 수준 — 네트워크 레이어 수준에서의 로드밸런싱**만 가능하다. 엄청 복잡한 로드밸런싱 로직이 동작하는 것은 아니다.

##### 서비스는 L4 수준 리소스

서비스는 기본적으로 **IP 주소 + 포트**를 기준으로 트래픽을 전달한다 (IP, Port, TCP, UDP). **HTTP 요청 내용 같은 걸 보고 트래픽을 분산하지는 않는다** — HTTP는 응용 계층(L7 Layer)에서 처리되는 영역이기 때문이다. Host domain이나 URL path를 보고 트래픽을 전달하는 건 L7 계층의 역할이다.

> ⚠️ 정확하게 "1:1:1로 트래픽이 나뉜다"라고 단순하게 봐서는 안 된다. 트래픽 정책, 세션 설정 등에 따라 실제 분배 방식은 달라질 수 있다.

##### Service 동작 구조

```
클라이언트 Pod ──▶ backend-service:8080 ──▶ Service(backend-service, ClusterIP: 10.96.10.20)
                                                    │
                                                    ▼
                                               EndpointSlice (10.244.1.10, 10.244.2.15)
                                                    │
                              ┌─────────────────────┼─────────────────────┐
                          Pod A                  Pod B                 Pod C
                    app=community            app=community          tier=frontend
                     tier=backend              tier=backend          (조건 불일치 → 제외)
```

1. Pod에 Label을 붙인다.
2. Service에 Selector를 작성한다.
3. Selector와 일치하는 Pod를 찾는다.
4. 해당 Pod의 IP를 EndpointSlice에 기록한다.
5. Service로 들어온 트래픽을 backend Pod로 전달한다.

##### EndpointSlice란?

**서비스가 현재 연결할 수 있는 실제 파드 IP 목록을 저장하는 리소스**다.

- **Service**: 만들어지면 서비스 자체에 IP(ClusterIP)가 부여된다.
- **EndpointSlice**: 서비스 뒤에 어떤 파드들이 연결되어야 하는지, 즉 연결 대상인 파드들의 IP 리스트를 실제로 들고 있는 리소스다.

> 💡 예전 버전에서는 `Endpoint`라는 리소스를 사용했지만, 쿠버네티스 1.3x 버전대부터 `EndpointSlice`로 바뀌었다.

##### Service YAML 구조

```yaml
apiVersion: v1
kind: Service
metadata:
  name: community-service
spec:
  selector:
    app: community
    tier: backend
  ports:
    - protocol: TCP
      port: 80
      targetPort: 3000
```

| 필드 | 설명 |
|---|---|
| **port** | 클러스터 내부에서 서비스가 요청을 받을 때 사용하는 포트 |
| **targetPort** | 연결 대상 파드의 컨테이너에서 사용하는 포트 — Pod YAML의 `containers.ports.containerPort`와 매칭되어야 함 |

##### Service 타입

쿠버네티스 서비스에는 몇 가지 타입이 있으며, 대표적인 타입은 다음과 같다. **실무에서는 거의 ClusterIP를 사용**한다.

| 타입 | 접근 범위 | 주요 사용 목적 |
|---|---|---|
| **ClusterIP** | 클러스터 내부 | 내부 애플리케이션 간 통신 |
| **NodePort** | Node의 IP와 포트를 통해 외부 접근 | 실습 또는 제한된 외부 노출 |
| **LoadBalancer** | 외부 Load Balancer를 통해 접근 | 클라우드 환경의 외부 서비스 |

**ClusterIP**
클러스터 내부에서만 접근할 수 있는 기본 서비스 타입이다. 서비스를 선언할 때 타입을 적지 않으면 자동으로 ClusterIP형으로 생성된다. 클러스터 IP는 클러스터 내부에서 서비스를 통해 노출되고, 내부에 있는 파드들에게 요청을 전달한다. 보통 내부 마이크로서비스 간 통신(frontend → backend, backend → database)에 사용된다. **ClusterIP형 서비스만 만들어두면 외부에서는 접근이 불가능하다.**

**NodePort**
각 노드에 특정 포트를 열어서 클러스터 외부에서 Service로 접근할 수 있게 하는 타입이다.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: community-nodeport
spec:
  type: NodePort
  selector:
    app: community
    tier: backend
  ports:
    - port: 80
      targetPort: 3000
      nodePort: 31000
```

각 노드에서 같은 포트를 노출하고, `NodeIP:Port번호` 형식으로 외부에서 접근할 수 있게 하는 방식이다 (ClusterIP 방식에 노드 포트를 추가로 열어둔 것). 주로 실습, 간단한 테스트 환경, 내부 네트워크에 대한 임시 접근 등에 사용된다.

> ⚠️ 왜 NodePort를 자주 사용하지 않을까? `https://35.30.11.1:31000`처럼 **모든 유저에게 우리 인프라의 퍼블릭 IP + 포트 번호가 그대로 노출**되기 때문에, 관리 측면과 보안 측면에서 좋지 않다.

**LoadBalancer**
클라우드 제공자의 외부 Load Balancer를 연결해서 서비스를 외부에 노출하는 타입이다.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: community-loadbalancer
spec:
  type: LoadBalancer
  selector:
    app: community
    tier: backend
  ports:
    - port: 80
      targetPort: 3000
```

```
Internet
   │
   ▼
Cloud Load Balancer (ALB)
   │
   ▼
Service
   ├── Pod A
   └── Pod B
```

> ⚠️ `kubeadm`으로 쿠버네티스를 직접 설치한 경우에는 `type: LoadBalancer`를 적어도 **클라우드와 연동되어 있지 않기 때문에 사용할 수 없다.** (EKS처럼 클라우드가 통합 관리하는 환경에서만 정상 동작한다.)

##### Service 타입의 선택 기준

```
클러스터 내부 통신인가?
    │
    ├── Yes → ClusterIP
    │
    └── No
          │
          ├── 간단한 실습·임시 노출 → NodePort
          │
          └── 클라우드 외부 노출 → LoadBalancer (EKS)
```

##### 서비스가 하지 않는 일

1. **서비스는 파드를 생성하지 않는다** — Service는 파드를 직접 만들지 않고, 이미 만들어진 파드 중에서 **조건에 맞는 파드에게 트래픽을 전달하는 역할**만 가진다.
2. **서비스는 URL 경로로 라우팅하지 않는다** — 이건 L7 계층의 다른 리소스(**API Gateway**, **Ingress**)가 처리해야 하는 영역이다.

##### 실무 팁

1. **서비스가 동작하지 않으면 엔드포인트 → DNS 문제를 먼저 확인해라** — Service는 고정된 이름과 가상 IP만 제공하는 역할일 뿐이다. 실제로 서비스가 연결이 안 될 때 그 원인을 서비스 자체에서 찾기는 힘들고, EndpointSlice나 CoreDNS/LocalDNS라는 파드 쪽에서 문제를 찾는 경우가 많다.
2. **Service의 Target Port와 Pod의 Container Port가 일치해야 한다** — 포트가 다르면 요청이 실패한다.
3. **Selector를 변경할 때는 조심해야 한다** — Selector가 바뀌면 요청이 전달되는 파드 집합이 통째로 달라져서, 일시적으로 처리량이 급감하는 경우가 있다.
4. **NodePort는 실습 위주로만 사용하고, 운영 환경에서는 사용하지 않는다.**

##### 정리

| 구분 | 내용 |
|---|---|
| 한 줄 정리 | Service는 변경되는 여러 Pod 앞에 안정적인 접근 주소를 제공하고 요청을 전달하는 리소스입니다. |
| 해결하는 문제 | Pod가 재생성되면서 이름과 IP가 변경되는 문제 |
| Pod 선택 방법 | Label Selector |
| 실제 backend 목록 | EndpointSlice |
| 내부 접근 주소 | ClusterIP와 Service DNS |

#### | Probe

**컨테이너의 상태를 주기적으로 진단하는 메커니즘**

Probe는 **kubelet이 컨테이너 안의 애플리케이션 상태를 주기적으로 검사**하고, 검사 결과에 따라 컨테이너를 재시작하거나 Service 트래픽에서 제외하는 기능이다.

```
Service
  ├──/health, /ready──▶ Pod
  └──/Liveness failed → restart / Readiness failed──▶ Pod
```

프로브를 설정하면 애플리케이션의 상태를 주기적으로 점검하고, 필요한 조치를 자동으로 수행해서 **안정성 + 가용성을 보장하는 핵심 요소**다.

##### Probe가 필요한 이유

컨테이너가 실행되고 있다고 해서(Running 상태) 정상적으로 요청을 처리할 수 있다는 보장은 없다. 애플리케이션 내부에서
- 데드락에 걸렸거나
- 내부 스레드가 멈췄거나
- 무한 루프에 걸렸을 수 있다.

그래서 쿠버네티스가 파드에게 물어봐야 하는 질문이 세 가지 있다.

| 확인해야 하는 상태 | 확인 질문 |
|---|---|
| **시작 상태** | 애플리케이션이 정상적으로 시작되었는가? |
| **생존 상태** | 애플리케이션이 멈추지 않고 계속 동작하고 있는가? |
| **준비 상태** | 지금 사용자 요청을 받아도 되는가? |

##### 프로브의 종류

| 종류 | 역할 |
|---|---|
| **Startup Probe** | 애플리케이션의 초기화 과정이 완료되었는지 확인. 스프링, 데이터베이스처럼 초기화 시간이 긴 애플리케이션에서 사용 |
| **Liveness Probe** | 앱이 정상적으로 실행 중인지 지속적으로 확인. 비정상 상태로 확인되면 자동 복구(재시작) 트리거를 실행 |
| **Readiness Probe** | 앱이 사용자 요청을 처리할 준비가 되었는지 점검. 준비되지 않은 상태에서 트래픽이 유입되지 않도록 방지 |

##### 프로브의 진단 방법

프로브는 다음 세 가지 방법 중 하나를 사용한다.

1. **httpGet** (가장 흔한 방식) — 특정 URL 경로로 HTTP 요청을 보내고, 응답이 2xx라면 정상으로 판단.
2. **tcpSocket** — 특정 포트에 TCP 연결을 시도하고, 성공적으로 연결되면 정상으로 판단. 소켓 연결이 필요한 파드에서 사용.
3. **exec** — 컨테이너 내부에서 특정 명령어를 실행하고, 종료 코드가 0이면 정상으로 판단 (예: `cat /tmp/healthy` 명령이 실행됐을 때 해당 파일이 존재하면 OK).

##### 사용 이유

컨테이너 환경에서는 이 컨테이너가 제대로 동작하는지 확인하려면 API 콜을 날리거나, 내부 로그를 보거나, 모니터링 시스템을 확인해야 하는 **수동적인 작업**이 필요하고, 이런 수동 작업은 대응 속도가 느리다. **프로브는 이런 수동 작업을 자동화**해준다.

프로브의 역할:
- **상태 점검** — 앱 상태를 감지하고 이상하면 kubelet이 재시작
- **트래픽 제어** — 준비가 안 된 앱에는 트래픽을 주지 않음
- **자동 복구** — 비정상 상태를 자동으로 복구하거나 재시작

##### kubelet과 Probe의 관계

프로브는 컨테이너 안에 설치된 별도의 프로그램이 아니다. **워커 노드에 상주하는 kubelet이 주기적으로 컨테이너의 문을 두드려 보는 것**이 바로 probe다.

```
kubelet → (probe 요청) → Pod
                            │
              컨테이너가 응답하거나 못하거나
                            │
프로브가 실패한 걸 kubelet이 감지
                            │
kubelet: "이 컨테이너가 정상이 아닌데?"
                            │
재시작 명령을 kubelet이 직접 내림
```

```
[Kubelet] → Liveness Probe 실행(health 등 엔드포인트 요청)
   │
   ├── 200 OK → 컨테이너 정상 동작 유지
   │
   └── Timeout/500 → 컨테이너 비정상 상태로 판단
                          │
                          ▼
              Kubelet이 컨테이너 재시작 (restart policy에 따라 수행)
                          │
                          ▼
              새로운 컨테이너 생성 후 Probe 재검사 시작
```

##### Probe 사용 방법

> 💡 `spec.containers.readinessProbe`, `livenessProbe`, `startupProbe` 필드에 프로브를 정의하여 컨테이너 상태를 주기적으로 검사하도록 설정한다.

#### | Liveness Probe

**컨테이너 생존 확인**

설정된 방식으로 라이브니스 프로브가 실패하면, 컨테이너가 죽었다고 판단하고 kubelet이 해당 컨테이너의 재시작을 시도한다. 즉 **죽은 앱을 감지하고, 다시 살려주는 역할**이다.

핵심 역할: **데드락 탐지** — 프로그램이 겉으로는 살아 있는 것처럼 보이지만 사실은 멈춰 있을 때, 앱의 무응답 상태를 감지하는 방법이다.

##### 동작 방식

**1. 헬스 체크**
- Liveness Probe는 애플리케이션의 특정 엔드포인트를 정기적으로 호출하거나, 명령어 또는 TCP 소켓으로 상태를 점검한다.
- 일반적인 HTTP 요청을 통해 상태를 확인하고, HTTP 응답 코드로 결과를 판단한다 (예: 200이면 정상, 500이면 에러).

**2. 설정 요소**
- **초기 지연 시간(initialDelaySeconds)** — 컨테이너 시작 후 첫 번째 상태 점검을 수행하기까지 대기하는 시간
- **주기(periodSeconds)** — 상태 점검 사이의 간격
- **타임아웃(timeoutSeconds)** — 응답을 기다리는 시간
- **실패 임계값(failureThreshold)** — 특정 횟수 이상 상태 점검에 실패하면 컨테이너 재시작

**3. 장애 발생 시 처리 과정**
- 특정 엔드포인트가 정상적인 상태 코드를 반환하지 못하면, 쿠버네티스는 장애로 인식한다.
- 설정된 횟수만큼 연속으로 점검에 실패하면, 쿠버네티스는 컨테이너를 재시작한다.

##### 구현 예시 (HTTP 기반)

```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 10
  failureThreshold: 3
```

##### 구현 시 주의 사항

**1. 로직을 단순화한다**
라이브니스 프로브는 주기적으로 요청되므로, 점검 로직이 복잡하면 점검 자체가 리소스 낭비가 된다. 엔드포인트는 경량화해서 설계해야 한다 (DB 쿼리를 날리거나 특정 값을 계산하지 말고, 단순하게 200 OK만 리턴하는 식).

**2. 일시적 장애(Flapping)를 고려한다**
`periodSeconds`, `failureThreshold`를 잘 설계해야 한다. Probe 주기를 너무 짧게 설정하면, 앱이 잠깐 바빠서 응답이 늦었을 뿐인데 죽여버리고, 다시 살아나면 또 죽이는 식으로 **죽였다 살렸다를 반복(Flapping)** 하게 된다. 이런 플래핑이 발생하면 가용성과 처리량이 상당히 나빠진다.

**실무적인 튜닝 기준**

| 설정 | 튜닝 가이드 |
|---|---|
| `initialDelaySeconds` | 앱 부팅 시간보다 여유 있게. 특히 StartupProbe가 없다면 좀 더 길게 설정하는 것이 좋음 |
| `failureThreshold` | 보통 3~5번 정도를 기준으로 설정 |
| `periodSeconds` | 주기를 너무 짧게 하지 말고, 10~30초 정도로 조금 길게 하는 것이 좋음 |
| `timeoutSeconds` | 일시적으로 트래픽이 몰리면 응답 시간이 지연될 수 있기 때문에 여유롭게 설정하는 것이 좋음 |

#### | Readiness Probe

**컨테이너가 요청을 받을 준비가 되어 있는지 판단**

준비가 되어 있지 않다고 판단하면 해당 Pod를 **Service에서 제거**한다. 애플리케이션이 살아 있어도(Liveness는 통과) 아직 초기화 중이거나 일시적으로 요청을 처리할 수 없는 상태일 수 있는데, 이럴 때 **일시적으로 외부 요청을 못 받도록 처리**하는 것이 Readiness Probe의 역할이다 — 서비스에서 제외시키는 방식으로, 구체적으로는 **EndpointSlice를 수정**한다.

레디니스 프로브를 통과해야만 해당 파드의 IP 주소가 Service의 EndpointSlice 목록에 추가된다.

> 💡 결론적으로, **레디니스 프로브가 없으면 무중단 배포를 할 수 없다.** 새로운 버전의 파드가 준비 상태가 된 이후에야 트래픽을 받아야 하고, Auto Scaling이 동작할 때도 새롭게 추가된 파드가 준비 상태가 된 이후에 서비스에 올라가야 하기 때문이다.

##### 주요 특징

1. **주기적 점검** — 정의된 주기와 방식에 따라 컨테이너 상태를 점검
2. **준비 상태 제어** — 컨테이너가 준비되지 않았으면 외부 트래픽이 유입되지 않도록 함
3. **다양한 점검 방식 지원** — HTTP GET, 명령 실행, TCP 소켓 등

##### 레디니스 프로브가 구분하는 상태

- 컨테이너가 준비 상태로 전환되기 전까지, 서비스는 해당 파드를 엔드포인트로 삼지 않는다.
- **Ready 상태**: 컨테이너가 외부 요청을 처리할 준비가 된 상태
- **NotReady 상태**: 컨테이너가 외부 요청을 처리할 준비가 되지 않은 상태

---

#### | Liveness Probe vs Readiness Probe

| 구분 | Liveness Probe | Readiness Probe |
|---|---|---|
| 목적 | 지속적인 **생존 확인** (Is the app healthy?) | **트래픽 처리 준비 확인** (Is the app ready for traffic?) |
| 실패 시 행동 | 컨테이너를 **재시작 (Restart)** | 서비스 목록에서 **제외 (Stop Traffic)** |
| 사용 사례 | 데드락(Deadlock), 무한 루프 발생 시 | 롤링 업데이트 또는 DB 연결 대기 중 |
| 비유 | 심장 박동 확인 (멈추면 심폐소생술) | 영업 준비 완료 확인 (안 되면 문 닫음) |

> 💡 실무에서는 이 두 프로브를 보통 함께 사용한다. WAS(Web Application Server)에는 두 가지 모두 필요하다 — 재시작을 해야 하는 데드락 상황도 있고, 롤링 업데이트를 해야 하거나 일시적으로 스레드 풀이 가득 차는 상황도 있기 때문이다.

---

#### | Startup Probe

**초기화가 오래 걸리는 컨테이너에게 사용**

실무적으로는 **Liveness Probe의 오작동을 막는 용도**로도 많이 사용된다. 애플리케이션 중에는 초기화까지 오래 걸리는 경우가 있다 (예: 스프링, MySQL — 스프링부트 프로젝트가 커지면 초기 구동 시간이 2~3분씩 걸리는 경우도 있다).

**Liveness Probe는 기본적으로 10초 단위로 실행**되는데, 초기화가 진행 중인 상태에서 Liveness Probe가 그대로 동작하면 "계속 살리고 죽이고, 살리고 죽이고"를 반복하게 된다. **Startup Probe는 앱이 초기화될 때까지 Liveness Probe를 잠시 멈춰주는 역할**을 한다. 초기화 시간이 긴 앱들에서 많이 사용된다.

##### 프로브 타입 정리

1. **Startup Probe** — 부팅되는 시간 동안 Liveness Probe가 컨테이너를 건드리지 못하게 지켜준다.
2. **Liveness Probe** — 앱이 응답하지 못하는 상태를 감지하고 데드락을 감지한다. 응답하지 못하면 재시작한다.
3. **Readiness Probe** — 앱이 일시적으로 트래픽을 받지 못하는 상태라면, 잠시 동안 외부 트래픽을 받지 못하도록 한다.

##### Web 서비스에서 선택하는 표준 조합

| 파드 종류 | Startup | Liveness (생존) | Readiness (준비) | 핵심 전략 |
|---|---|---|---|---|
| **Frontend** | X | HTTP (`/`) | HTTP (`/`) | 단순하게 (켜지면 바로 투입) |
| **Backend** | O (느리다면) | HTTP (`/live`, DB 체크 금지) | HTTP (`/ready`, DB 체크 필수) | 이원화 (살아있는 것과 일할 준비된 것을 구분) |
| **Database** | O | TCP / CMD (보수적 설정) | CMD (`SELECT 1`) | 신중하게 (재시작은 최후의 수단) |

#### | QoS Classes

**Pod의 리소스 관리와 스케줄링을 위한 분류 체계**

Pod의 CPU, 메모리 `requests`와 `limits` 설정을 기준으로 쿠버네티스가 Pod를 분류하고, **노드 자원이 부족할 때 어떤 Pod를 우선 보호하거나 퇴출할지 판단하는 기준**이다. 쿠버네티스는 파드를 다음 세 가지 QoS Class 중 하나로 **자동 분류**한다.

> 💡 QoS Class는 사용자가 직접 이름으로 설정하는 것이 아니라, Pod에 작성한 CPU·Memory 설정을 보고 **쿠버네티스가 자동으로 결정**한다.

| QoS Class | 핵심 특징 |
|---|---|
| **Guaranteed** | 필요한 자원을 명확하게 확보하고 가장 강하게 보호받는 Pod |
| **Burstable** | 최소 자원은 요청하지만, 여유가 있다면 추가 자원을 사용할 수 있는 Pod |
| **BestEffort** | 자원을 요청하지 않고 노드의 남는 자원을 사용하는 Pod |

##### QoS Class를 왜 사용하는가?

하나의 워커 노드에서는 여러 파드가 CPU, Memory, 디스크, 네트워크 등의 자원을 함께 사용한다. 노드에 자원이 충분하면 별문제가 없지만, **자원이 부족해지면 실행되고 있는 모든 파드를 계속 실행하지 못하는 상황**이 생긴다.

```
Worker Node
├── 사용자 API Pod
├── 관리자 API Pod
├── 로그 처리 Pod
├── 배치 작업 Pod
└── 테스트용 Pod
```

이런 노드 상태에서 메모리가 부족해지면, 쿠버네티스는 어떤 파드를 먼저 종료해야 할까? 모든 파드를 무작위로 종료해버리면 비즈니스 로직적으로 중요한 파드가 먼저 중지될 수도 있다. **QoS Class**는 다음을 가능하게 해준다.
- 중요한 파드들을 상대적으로 보호
- 자원을 많이 사용하는 파드를 식별
- 노드 전체가 멈추기 전에 일부 파드를 미리 퇴출
- CPU·Memory 자원 사용량을 예측 가능하도록 관리
- 하나의 파드가 노드 자원을 과도하게 사용하는 문제 방지

> ⚠️ QoS Class는 **서비스의 중요도를 나타내는 값이 아니다.** 어디까지나 **리소스 설정에 따라 파드의 보호 수준을 결정하는 분류 체계**일 뿐이다.

##### Requests와 Limits

**Requests** — Pod를 실행하기 위해 쿠버네티스에게 요청하는 **최소한의 자원**

```yaml
resources:
  requests:
    cpu: "250m"
    memory: "256Mi"
```

스케줄러가 파드를 어느 노드에 배치할지 판단할 때 사용된다 — "적어도 requests에 적힌 자원량 이상이 있는 노드에 배치해주세요"라는 의미로, 해당 자원을 수용할 수 있는 노드를 찾아서 그곳에 배치한다.

**Limits** — 컨테이너가 사용할 수 있는 **최대한의 자원 한도**

배치가 된 파드가 노드의 자원을 사용할 때 사용할 수 있는 최대치다. CPU와 Memory는 Limit 초과 시 동작이 다르다.

| 리소스 | Limit 초과 시 동작 |
|---|---|
| **CPU** | 컨테이너를 종료하지 않고 CPU 사용을 제한함 |
| **Memory** | 메모리를 더 사용할 수 없으면 컨테이너가 **OOM Kill**될 수 있음 |

CPU는 요청을 나눠서 조금 덜 줄 수 있는 반면, 메모리는 한 번 할당된 이후에는 회수하기가 어렵다. 즉 **CPU는 최소 배분이 가능**하고, **Memory는 컨테이너가 사용할 수 있는 상한선**으로 동작한다.

##### QoS Class 종류

**1. Guaranteed**

```
Requests ═══ Limits
```

컨테이너의 CPU, Memory의 **Request와 Limit을 동일하게 설정**한 파드가 여기 해당한다.

- 분류 조건: 파드 안의 **모든** 컨테이너가 CPU와 Memory 모두에서 이 조건을 맞춰야 한다.
- 특징:
  - 스케줄러는 요청한 자원을 확보할 수 있는 노드에만 파드를 배치 (예: Request = Mem 512MB)
  - 동시에 해당 파드가 사용할 자원도 Limit = Mem 512MB로 동일
  - 실행 중에 요청량보다 더 많은 자원을 애초에 사용할 수 없음
  - 노드 자원 부족 상황에서 **상대적으로 가장 강하게 보호**됨
  - 중요한 시스템, 안정적인 성능이 필요한 파드들을 배치

**언제 사용할까?**
- 결제, 인증처럼 **처리량보다 안정성**이 우선되는 시스템
- 급격한 처리량 변화를 파드의 자원 할당으로 처리하는 시스템에는 어울리지 않음 (예: 실시간 랭킹)
- 예상 자원 사용량이 비교적 명확한 앱
- 장애가 발생하면 비즈니스 로직에 심각한 문제가 생기는 경우

> ⚠️ 주의 사항: **Guaranteed QoS라고 해서 "절대 종료가 안 된다"는 뜻은 아니다.** 다음과 같은 경우에는 여전히 종료될 수 있다.
> - 컨테이너가 애초에 메모리가 부족한 경우
> - 노드 장애가 생기는 경우
> - 노드가 종료되는 경우
> - 관리자가 파드를 지우는 경우
> - **더 낮은 우선순위를 갖고 있는 파드를 모두 제거해도, 필요한 자원량이 확보되지 않는 경우** (예: 1순위 파드 A(Guaranteed)를 우선 가져가야 하는데, 하나의 노드에 모든 파드가 전부 Guaranteed라면 QoS Class만으로는 퇴출 순서를 정할 수 없다)
>
> 이런 경우 kubelet이 파드를 퇴출할 때 추가로 고려하는 기준: **1) 실제 사용량이 requests를 초과했는가? 2) Pod Priority가 더 낮은가? 3) Request 대비 현재 자원 사용량이 더 적은 파드를 먼저 퇴출**

**2. Burstable**

```
Requests < Limits
```

일부 자원은 보장받는데, 노드에 여유가 있으면 요청량보다 더 많은 자원을 사용할 수 있는 파드다.

- 분류 조건: 최소 하나의 컨테이너에 CPU 또는 메모리에 Request나 Limit이 설정되어 있으면 된다 (Guaranteed 조건을 충족하지 않는 나머지 경우).
- 평상시에는 request 기준으로 실행하고, 노드에 여유가 있으면 Limit까지 가져와서 사용한다. 일부만 설정해도 Burstable로 분류된다.

```yaml
resources:
  requests:
    memory: "256Mi"
```

> 💡 `limit`을 아예 지정하지 않아도 Burstable로 분류된다. 요청량이 늘어나면 노드에 자원 여유가 있는 만큼 더 가져다 쓴다.

**특징**
- Requests를 설정하면 그 값을 기준으로 스케줄링된다.
- 여유 자원이 있으면 리퀘스트보다 많이 쓸 수 있다.
- Guaranteed보다는 자원 활용이 여유롭고 유연하다.
- BestEffort보다는 상대적으로 보호받는다.
- **일반적인 WAS는 Burstable 클래스로 설정하는 경우가 많다.**

**언제 사용?** WAS, Spring, Node.js API처럼 request를 설정하고 limit은 잡지 않거나 여유롭게 설정하는 경우 — 노드의 최소 자원량은 보장받되, 트래픽에 따라 자원 사용량이 늘어나는 시스템에 적합하다.

**3. BestEffort**

```
(Requests, Limits 모두 미설정)
```

CPU, Memory 모두 Request와 Limit을 설정하지 않은, 노드에 남는 자원만으로 동작하는 파드다.

- 분류 조건: 아무것도 설정하지 않으면 자동으로 BestEffort가 된다.
- 특징:
  - 스케줄러에 요청 사항이 없다.
  - 노드의 남은 자원을 자유롭게 사용한다.
  - 다른 파드들이 QoS Class가 더 높기 때문에 우선 보호되며, **BestEffort가 가장 먼저 퇴출**된다.
  - 메모리가 부족할 때 OOM이 발생할 가능성이 크다.

**언제 사용?** 실패해도 문제없는 파드들 — 테스트 파드, 일회성 파드, 디버깅 파드, 개발 환경, 중간에 실패해도 다시 시도하면 같은 결과가 나오는 보조 작업(통계성 작업) 등.

##### Pod는 언제 퇴출되는가? — Eviction

**Eviction**은 노드를 안정적으로 유지하기 위해 kubelet이 실행 중인 파드를 노드에서 종료시키는 동작이다. 워커 노드 자원이 부족하면 노드 전체 파드에 문제가 생길 수 있기 때문에, 전체 파드가 죽기 전에 **미리 자원을 확보**한다.

```
노드 자원 부족 감지
      ↓
불필요한 이미지와 종료된 컨테이너 정리
      ↓
자원 부족이 계속됨
      ↓
퇴출 대상 Pod 선정
      ↓
Pod 종료
      ↓
노드 자원 확보
      ↓
API Server 보고
      ↓
종료된 파드가 다시 실행되어야 하면?
      ↓
다시 스케줄링
      ↓
다른 노드로 배치
```

##### 무엇이 부족하면 퇴출되는가?

1. **메모리**
2. **디스크 공간 부족**
   - 노드의 파일 시스템
   - 컨테이너 이미지 저장 공간
   - 컨테이너 로그
3. **PID (프로세스 ID) 부족**
   - 노드에서 생성할 수 있는 프로세스 ID가 부족한 경우
   - 애플리케이션이 너무 많은 프로세스를 만들면 발생
   - 노드별로 할당할 수 있는 PID 개수가 정해져 있음

##### QoS Classes 정리

| 핵심 개념 | 내용 |
|---|---|
| QoS Class | CPU·메모리 Requests와 Limits를 기준으로 Pod를 자동 분류 |
| Guaranteed | 모든 컨테이너의 CPU·메모리 Request와 Limit이 동일 |
| Burstable | 일부 자원을 설정했지만 Guaranteed 조건은 충족하지 않음 |
| BestEffort | CPU·메모리 Requests와 Limits가 모두 없음 |
| 주요 목적 | 노드 자원 부족 상황에서 Pod 보호와 퇴출 판단 |
| 실제 퇴출 판단 | Request 초과 여부 → Pod Priority → Request 대비 사용량 |

### 🗨️ 오늘의 회고

- 어제까지는 쿠버네티스 복습과 파드 설계까지 완료했는데, 실제로 아는 것이 부족해서 EC2에 노드를 어떻게 배치해야 할지 막막했다. 오늘 Service, Probe, QoS Classes까지 배우고 나니 "파드를 안정적으로 운영한다"는 게 구체적으로 어떤 요소들로 구성되는지 조금씩 그림이 잡히는 느낌이다.
- **Service**의 핵심이 "외부 노출"이 아니라 **"클러스터 내부에서 계속 바뀌는 파드에게 안정적으로 접근하는 방법"** 이라는 걸 명확히 짚고 넘어간 게 오늘 가장 큰 수확이었다. 회사 대표 전화번호 비유 덕분에, 개별 파드 IP가 아니라 서비스 이름으로 접근해야 하는 이유가 확실히 이해됐다. Selector → EndpointSlice → 실제 Pod로 이어지는 흐름도 그림으로 정리하니 훨씬 명확해졌다.
- **Probe** 파트에서는 "컨테이너가 Running 상태라고 해서 정상 처리가 가능한 게 아니다"라는 전제가 인상적이었다. 특히 Liveness Probe와 Readiness Probe의 차이 — **재시작할 것이냐(Restart) vs 트래픽만 잠깐 끊을 것이냐(Stop Traffic)** — 를 "심장 박동 확인 vs 영업 준비 완료 확인"이라는 비유로 정리하니 헷갈리지 않게 됐다. 특히 **"Readiness Probe가 없으면 무중단 배포를 할 수 없다"**는 문장이 지난주에 배운 무중단 배포 내용과 바로 연결됐다.
- Startup Probe가 "Liveness Probe의 오작동을 막는 용도"라는 설명도 실용적이었다. 초기화가 오래 걸리는 스프링/DB 앱에서 Liveness Probe가 10초 주기로 계속 재시작을 반복하는 상황(Flapping)을 막아주는 안전장치라는 걸 알고 나니, 왜 세 프로브를 함께 설계해야 하는지 납득이 갔다.
- **QoS Classes**는 처음 접하는 개념이라 조금 낯설었지만, "노드 자원이 부족할 때 누구를 먼저 지킬 것인가"라는 관점으로 보니 이해가 쉬웠다. Guaranteed(결제·인증처럼 안정성이 최우선인 시스템), Burstable(일반적인 WAS), BestEffort(테스트·일회성 작업)로 구분해서 실제 사용 사례와 함께 정리하니 감이 잡혔다. 특히 "Guaranteed도 절대 안 죽는 건 아니다"라는 주의사항을 놓치지 않고 짚어야겠다.
- 오늘 목표였던 "EC2에 쿠버네티스 클러스터 구축"은 아직 진행 중이다. 개념 정리는 꽤 탄탄해진 것 같으니, 내일은 이 개념들을 실제 실습에 적용하면서 막혔던 부분들을 풀어나가야겠다.
