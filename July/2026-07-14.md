#### | Pod

**하나 이상의 컨테이너를 그룹화하여 실행 환경과 네트워크 자원, 스토리지를 공유하는 가장 작은 배포 단위**

쿠버네티스에서는 컨테이너를 워커 노드에 직접 배포하지 않는다. 컨테이너를 **파드**라는 실행 단위로 감싸고, 파드를 워커 노드에 배치하는 구조다.

##### 파드가 필요한 이유 — "그냥 컨테이너로 실행하면 안 되나?"

Docker에서는 `docker run ~~` 명령으로 바로 컨테이너를 만들어 실행했는데, 쿠버네티스는 왜 굳이 파드라는 단위를 하나 더 두는 걸까? 이를 이해하려면 **노드와 파드의 관계**를 먼저 봐야 한다.

- 워커 노드 = 우리의 파드가 실행되는 공간
- 쿠버네티스는 컨테이너를 소를 다루듯 "가축"처럼 취급한다. 그래서 EC2에 애완동물처럼 이름을 붙이지 않고, **역할 + 숫자**로 관리한다 (예: 역할 = 워커, 숫자 = 01, 02…).
- 워커 노드에는 어떤 컨테이너든 다 들어올 수 있다 — "프론트용 워커노드", "백엔드용 워커노드"처럼 나누지 않고 그냥 하나의 워커 노드로 관리한다.

워커 1에서 백엔드 컨테이너를 **직접** 실행한다고 가정해보면, 컨테이너가 1개일 때는 그럭저럭 동작할 수 있다. 하지만 **실행해야 하는 최소한의 컨테이너 개수가 2개 이상**이 되는 순간부터 문제가 생긴다. 컨테이너가 자동으로 배치되길 바란다면, "자동으로 배치되려면 최소한 무엇까지 있어야 동작한다"는 **정의(단위)** 가 필요해진다 — 그 최소 배포 단위가 바로 파드다.

##### 파드와 컨테이너의 관계

```
Worker Node
└── Pod
      └── Container (Spring Boot / Nginx / Node.js)
```

실무적으로는 **파드 하나에 컨테이너 하나**를 두는 경우가 많다. 파드는 컨테이너보다 한 단계 바깥에 있는 개념이다 — 파드 자체가 애플리케이션을 실행하는 게 아니라, **애플리케이션은 컨테이너에서 실행**되고 파드는 그 컨테이너(들)를 감싸는 **쿠버네티스의 관리 단위**다.

```
Pod (쿠버네티스 관리 단위)  →  Container (실제 애플리케이션 실행)  →  Application Process (java / node / nginx)
```

##### 파드를 쓰면 얻는 것

파드 내부의 컨테이너들끼리는 다음을 함께 공유한다.
- 네트워크
- IP 주소
- 포트
- 볼륨
- 생성 주기

이 중에서도 **가장 빠른 통신 방식은 로컬 호스트 통신**이다. 만약 컨테이너가 다른 컨테이너와 통신해야 하는 경우, 같은 파드 안에 있다면 이 로컬 호스트 통신을 활용할 수 있다.

##### 파드의 네트워크

**1. Pod마다 IP가 할당된다** — 쿠버네티스에서는 일반적으로 파드 단위로 IP를 할당한다.

**2. 같은 파드의 컨테이너는 IP를 공유한다** — 하나의 파드에 여러 컨테이너가 있다면, 그 컨테이너들은 동일한 네트워크 공간을 공유한다.

```
Pod IP: 10.244.1.20
├── Main Container (port 8080)  ←── localhost ──→  Helper Container (port 9090)
```

> ⚠️ 로컬호스트 통신이 가능한 건 장점이지만, 같은 파드 안의 컨테이너들은 **동일한 포트를 사용할 수는 없다.**

##### 파드 볼륨

파드에는 내부 컨테이너들이 공유할 수 있는 볼륨이 존재한다. 메인 컨테이너가 볼륨에 작성한 값을 헬퍼 컨테이너가 읽을 수 있다.

> 💡 **중앙 집중형 로깅 시스템**에서 이 구조가 많이 사용된다. 메인 컨테이너가 백엔드 컨테이너인 경우, 요청이 들어오면 볼륨에 로그를 남기고, 헬퍼 컨테이너가 특정 주기마다 쌓인 로그를 중앙 집중 로그 저장소로 전송하는 역할을 한다.

##### 파드 유형

**Pod 유형 1 — 단일 컨테이너 구조**

```
Pod
└── Spring Boot Container
```

실무적으로 가장 흔한 구조이며, 쿠버네티스의 장점을 가장 많이 가져가는 구조다.

> 왜 프론트와 백엔드를 같은 파드에 넣지 않을까? 애플리케이션 파드를 만들어 프론트와 백엔드 컨테이너를 함께 넣으면 다음 문제가 생긴다.
> - **장애 파악이 힘들다**
> - **독립적으로 확장이 불가능하다** — 쿠버네티스의 배포 단위는 파드이므로, 스케일 아웃(처리량이 더 필요해서 컨테이너를 늘려야 하는 상황)이 필요할 때 애플리케이션 파드 전체를 늘려야 한다. 우리가 실제로 필요한 처리량은 백엔드 처리인데, 프론트엔드 컨테이너도 함께 늘어나 버린다.
>   - 실제 필요한 구성: FE 컨테이너 3개, BE 컨테이너 10개
>   - 프론트/백엔드를 한 파드로 묶었다면: FE 컨테이너 10개, BE 컨테이너 10개 (불필요한 FE 컨테이너가 함께 늘어남)

**Pod 유형 2 — 멀티 컨테이너 패턴 (사이드카 패턴)**

```
Pod
├── Main Container   — 애플리케이션 (비즈니스 로직)
└── Helper Container — 보조 기능 (로깅, 포맷팅 등)
```

파드에는 여러 컨테이너를 넣을 수 있는데, 보통 비즈니스 로직을 담당하는 컨테이너는 하나만 두는 경우가 많고, 여기에 로깅·포맷팅 같은 **보조 기능 컨테이너**를 함께 넣는 경우가 있다. 이런 구조(사이드카 패턴)를 **멀티 컨테이너 패턴**이라고 부른다.

> ⚠️ 멀티 컨테이너 패턴에서 주의해야 할 사항: 파드 안에서는 네트워크 공간과 볼륨을 공유하기 때문에 **포트 충돌**, **볼륨 충돌**에 유의해야 하고, 무엇보다 **생명주기가 항상 같다**는 점을 고려해야 한다. 예를 들어 트래픽이 스파이크 형태라 파드가 빠르게 늘어나고 빠르게 제거되어야 하는 상황인데, 헬퍼 컨테이너의 초기화나 종료 처리에 한참 시간이 걸린다면 전체 파드의 생성·삭제 속도에 영향을 줄 수 있다.

##### Pod는 일시적인 리소스다 (매우 중요‼️)

파드는 **언제든지 삭제되고, 언제든지 다시 만들어질 수 있다.** 파드는 영구적으로 유지되지 않는다.

파드가 사라지는 대표적인 상황:
- 사용자가 `delete` 명령을 실행했을 때
- 워커 노드에 장애가 발생했을 때
- 컨테이너가 반복적으로 장애를 겪어 상위 관리 리소스가 새로운 파드로 교체할 때
- 노드 유지보수가 필요할 때
- Auto Scaling이 걸릴 때 — 처리량이 많이 필요해서 파드를 늘렸다가, 필요 처리량이 줄면 파드를 지움

새로운 처리량이 필요할 때 쿠버네티스는 **기존 파드에 자원을 추가로 할당하는 방식이 아니라, 그냥 새로운 파드를 만드는 방식**으로 대응한다.

> ⚠️ **새로 생성된 파드는 이전 파드와 완전히 다른 존재다.** 똑같은 이미지, 똑같은 라벨, 똑같은 규칙을 적용하더라도 그렇다. 새로운 파드는
> - 고유한 Pod 식별자를 새로 갖는다
> - IP 주소가 다르다
> - 실행되는 워커 노드도 스케줄링에 따라 달라질 수 있다
> - 내부에서 사용하는 볼륨(컨테이너가 가진 임시 쓰기 레이어)도 다르다
>
> 그래서 다음 두 가지는 반드시 기억해야 한다.
> - **파드 내부에 중요한 데이터를 저장하면 안 된다.**
> - **Pod의 요청을 고정 IP로 하면 안 된다.**

##### 파드가 생성되는 과정

```
사용자 → API Server → Scheduler → Worker Node의 kubelet → Container Runtime

1. 사용자가 API Server에 Pod 생성 요청
2. API Server가 Pod 정보를 저장 (etcd)
3. Scheduler가 실행할 Worker Node를 결정
4. Worker Node의 kubelet이 자신에게 할당된 Pod를 확인
5. kubelet이 Container Runtime에 컨테이너 이미지 실행 요청
6. Container Runtime이 이미지를 다운로드하고 컨테이너 생성
7. kubelet이 API Server에 Pod 상태를 보고
```

##### 파드 YAML 구조

일반적으로 쿠버네티스의 대부분의 자원은 YAML 파일로 관리한다.

```yaml
apiVersion: v1
kind: Pod

metadata:
  name: express-pod

spec:
  containers:
    - name: express-container
      image: adapterz/k8s-pod:latest
      ports:
        - containerPort: 3000
```

| 필드 | 설명 |
|---|---|
| **apiVersion** | 쿠버네티스 Docs에서 API 버저닝을 해둔 값. Pod는 v1 버전이라 `v1`을 사용 |
| **kind** | 쿠버네티스 리소스의 종류 |
| **metadata** | 리소스를 식별할 때 사용하는 공간 |
| **spec** | 파드가 어떤 모습으로 실행되어야 하는지 정의하는 부분. 파드 내부에서 실행할 컨테이너, 이미지, 포트 등의 정보를 작성 |
| **spec.containers** | 파드 내부에서 실행할 컨테이너 목록 |
| **container.name** | 컨테이너 이름 |
| **container.image** | 컨테이너가 사용할 이미지 |
| **container.ports** | 포트 정보 |

##### Label

**쿠버네티스 오브젝트에 붙이는 키-값(Key-Value) 형태의 이름표**

쿠버네티스에서는 라벨이 상당히 중요하다. 라벨을 사용하면 애플리케이션, 역할, 실행 환경, 버전 같은 기준으로 쿠버네티스 오브젝트들을 구분할 수 있다.

```yaml
app: community
env: production
tier: backend
version: v1
```

##### 라벨을 왜 사용하는가

쿠버네티스는 리소스를 **동적으로 생성**하고, 그 동적으로 생성된 리소스들은 각각 별도의 IP를 가진다. 즉 **Pod의 IP와 식별자는 고정되지 않는다.** 그래서 리소스를 이름이 아닌 "속성(라벨)"으로 관리하기 위해 라벨을 사용한다.

예를 들어 express 파드가 10개로 늘어나면 IP도 10개가 되며, 이 IP들을 미리 예측하거나 고정시킬 수 없다. 그런데 만약 "express 서비스에서 백엔드 서비스 중 v1인 것들을 v2로 업데이트해줘"처럼 **특정 조건을 만족하는 리소스들만** 다뤄야 하는 요구사항이 생긴다면, 이런 조건들을 **미리 라벨로 붙여놓는 것**으로 해결한다.

##### 라벨 구성

`키: 값` 형태로 구성된다.

##### 실무에서 많이 사용하는 라벨

| 라벨 | 의미 | 예시 값 |
|---|---|---|
| **app** | 리소스가 어떤 애플리케이션에 속해 있는지 구분 | `community` |
| **tier** | 애플리케이션 내부에서 어떤 계층에 속해 있는지 나타냄 (팀에 따라 `layer`로 표현하기도 함) | `frontend`, `backend`, `data` |
| **env** | 어떤 실행 환경에 속하는지 | `dev`, `staging`, `production` |
| **version** | 버전 정보 | `v1`, `v2`, `1.0.1` |

> 💡 클러스터 구성 방식에 따라 네임스페이스 전략도 달라진다. 하나의 클러스터 안에서 여러 네임스페이스로 나누는 경우, 어떤 팀은 **환경 기준**으로 나누고(`prod-ns`, `dev-ns`), 어떤 팀은 **애플리케이션 기준**으로 나눈다(`was-ns`, `data-ns`, `payment-ns`). 여러 클러스터에서 다시 여러 네임스페이스로 나누는 구성도 존재한다.

라벨은 파드에서만 사용하는 것이 아니라, **Pod, Node, Service, ConfigMap, Secret, Namespace** 등 다양한 쿠버네티스 오브젝트에 붙일 수 있다.

##### 실무에서 라벨을 잘 사용하는 방법

1. **라벨명이 명확해야 한다** — `type` 같은 라벨은 "무슨 타입인지" 알 수 없다. 의미를 명확히 전달할 수 있는 이름을 사용해야 한다.
2. **라벨명은 팀 내에서 규칙을 정해놓고 사용하는 것이 일반적** — `app`, `application`, `service`처럼 비슷한 의미의 이름을 혼용하면 관리가 어려워지므로, 미리 규칙을 정해둔다.
3. **너무 자주 변할 수 있는 상태는 라벨로 지정하지 않는다** — `cpu-usage`, `instance-id`처럼 계속 바뀌는 값은 라벨로 적합하지 않다. 가끔 변하는 정도라면 라벨로 붙일 수 있지만, 변경 빈도가 너무 높다면 굳이 라벨로 관리하지 않는다.

##### Selector

**라벨을 기준으로 조건에 맞는 쿠버네티스 오브젝트를 선택하는 방법**

- **라벨(Label)** = 리소스에 붙이는 이름표
- **셀렉터(Selector)** = 그 라벨을 기준으로 리소스를 선택하는 방법

```yaml
metadata:
  labels:
    app: community
    tier: backend

# ...

selector:
  app: community
  tier: backend
```

셀렉터를 사용하는 이유는 라벨을 사용하는 이유와 동일하다 — 동적으로 생성되는 리소스를 이름이 아니라 속성으로 관리하기 위함이다.

##### 셀렉터의 주요 특징

- **라벨 기반으로 선택**한다.
- **동적으로 갱신**된다 — 라벨이 붙은 파드가 새롭게 늘어나면 셀렉터의 결과도 함께 갱신된다. 조건에 맞는 리소스가 생성되거나 삭제되면 선택 결과도 자동으로 바뀐다.
- **여러 조건을 함께 사용**할 수 있으며, 기본적으로 조건들은 **AND**로 결합된다.
- 리소스끼리 서로 연결할 때 많이 사용된다. 대표적으로 **Service**(쿠버네티스에서 도메인 역할을 해주는 리소스)가 파드와 연결될 때 기본적으로 셀렉터를 사용한다 (`service.yaml`).

### 🗨️ 오늘의 회고

- 어제 회고에서 남겼던 것처럼, 강의를 들을 땐 이해가 됐는데 막상 용어를 정리하다 보니 헷갈리는 부분들이 조금씩 드러났다. 특히 **Pod와 Container의 관계**를 "Pod는 관리 단위, Container는 실제 실행 단위"로 명확히 구분해서 정리하니 훨씬 또렷해졌다.
- 오늘 배운 것 중 가장 인상 깊었던 부분은 **"쿠버네티스는 자원을 추가로 할당하지 않고, 새로운 파드를 만든다"**는 지점이다. 새로 생성된 파드는 똑같은 이미지·라벨을 써도 "완전히 다른 존재"라는 설명을 보고, 왜 파드에 중요한 데이터를 저장하면 안 되는지, 왜 고정 IP로 요청을 보내면 안 되는지가 자연스럽게 이해됐다. 이건 이번 주 초에 배운 "쿠버네티스는 컨테이너를 가축으로 취급한다"는 비유와도 정확히 연결되는 부분이었다.
- 멀티 컨테이너 패턴(사이드카 패턴)을 배우면서, "왜 프론트/백엔드를 한 파드에 안 넣나요?"라는 질문에 대한 답 — 장애 파악과 독립적인 확장(스케일 아웃) 문제 — 이 특히 실무적으로 와닿았다. FE 3개·BE 10개가 필요한데 한 파드로 묶으면 불필요하게 FE도 10개가 되어버린다는 구체적인 숫자 예시 덕분에 이해가 훨씬 쉬웠다.
- Label과 Selector 파트에서는 "왜 라벨이 필요한가"의 핵심이 **"Pod의 IP와 이름이 고정되지 않기 때문"** 이라는 걸 알게 됐다. 리소스를 이름이 아니라 속성으로 관리한다는 개념이 이후에 배울 Service, Deployment와도 계속 연결될 것 같아서, 라벨 네이밍 규칙(app/tier/env/version)을 지금부터 익숙하게 만들어둬야겠다.
- 아직 쿠버네티스를 실제로 다뤄본 적이 없어서 10주차 위클리 챌린지 진행 속도가 더딜 것 같다는 예상을 했었는데, 오늘 Kubectl 명령어와 Pod YAML 구조까지 정리하고 나니 실습을 시작할 최소한의 기반은 잡힌 것 같다. 전주차 과제 보완은 예정대로 뒤로 미루고, 내일부터 교재·실습 복습에 집중할 예정이다.
