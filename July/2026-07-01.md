#### | Bridge Network

**컨테이너들이 동일한 호스트 내에서 서로 통신할 수 있도록 연결해주는 도커의 기본 가상 네트워크**

브릿지 네트워크는 **한 대의 도커 호스트 안에서** 여러 컨테이너를 연결하는 **가상 내부 네트워크**다.

##### "그냥 인터넷이랑 통신하면 되는 거 아닌가?"

실제 서비스는 하나의 프로그램으로 구성되지 않는다. 컨테이너들 사이에서도 다양한 통신이 일어난다.

| 통신 주체 | 목적 |
|---|---|
| 사용자 → Nginx | 웹사이트 또는 API 요청 |
| Nginx → API | 리버스 프록시 요청 전달 |
| API → MySQL | 데이터 조회 및 저장 |
| API → 외부 인터넷 | 외부 API 호출, 패키지 다운로드 등 |

이때 **모든 컨테이너를 그냥 호스트가 사용하는 네트워크에 연결**하면 어떻게 될까?

- 어떤 컨테이너가 어떤 포트를 사용하는지 관리하기 힘들어짐
- MySQL 같은 보안성이 중요한 컨테이너가 외부 인터넷에 노출될 수 있음
- 컨테이너는 삭제되고 재생성이 자주 되는 구조라 IP를 새롭게 할당받게 되고, 연결 설정이 깨질 수 있음
- 여러 프로젝트가 하나의 호스트에서 실행 중일 때, 내부 프로젝트들이 서로 의도치 않게 통신할 수 있음 (브로드캐스트)

> 💡 이러한 문제들 때문에, 도커는 **컨테이너마다 독립적인 네트워크 환경을 만들고** 필요한 컨테이너끼리만 같은 네트워크를 연결할 수 있도록 한다. 이것이 **Bridge Network**다.

##### 이해해야 하는 개념

**1. Host**

도커 엔진이 실행되는 실제 서버, 리눅스 머신 등을 뜻한다. (EC2, 내 컴퓨터 등)

- 도커 호스트는 컨테이너를 실행하고 관리하는 기반 컴퓨터다.
- 컨테이너는 호스트의 CPU, Memory, Network 자원을 함께 공유해서 사용하지만, 가능한 서로 분리된 환경에서 실행된다.

**2. 네트워크 인터페이스**

네트워크 통신을 위한 출입구.

| 구분 | 예시 | 의미 |
|---|---|---|
| 물리 네트워크 인터페이스 | `eth0`, `ens5`, `enp0s3` | 실제 서버가 VPC, 인터넷, 사내망과 연결되는 출입구 |
| 가상 네트워크 인터페이스 | 컨테이너 내부의 `eth0` | 컨테이너가 네트워크를 사용하는 출입구 |
| Docker 브리지 인터페이스 | `docker0`, `br-...` | 여러 컨테이너의 가상 인터페이스를 연결하는 가상 스위치 |

**3. 컨테이너는 자신의 네트워크 공간을 가진다**

브릿지 네트워크에 연결된 컨테이너는 다음 정보를 각자 가진다.
- 자신의 네트워크 인터페이스
- 자신의 IP 주소
- 자신의 기본 게이트웨이
- 자신의 라우팅 테이블
- DNS 설정

```
브릿지 네트워크 안에서:

Nginx 컨테이너     → eth0: 172.18.0.2
API 컨테이너       → eth0: 172.18.0.3
MySQL 컨테이너     → eth0: 172.18.0.4
```

> 💡 브릿지 네트워크 안에 연결된 컨테이너는 네트워크별 IP 주소를 **자동으로 할당**받는다(컨테이너가 생성될 때). 각 네트워크에는 서브넷 + 기본 게이트웨이 설정이 되어 있고, **컨테이너 IP는 생성될 때마다 변경**될 수 있다.

##### Bridge Network의 구조

브릿지 네트워크는 실제 네트워크 장비를 통한 네트워크가 아니라, **리눅스 + 도커 소프트웨어로 만든 가상의 네트워크**다.

| 구성 요소 | 역할 | 비유 |
|---|---|---|
| 컨테이너 `eth0` | 컨테이너 내부 네트워크 출입구 | 각 집의 현관문 |
| `veth pair` | 컨테이너와 호스트를 연결하는 가상 케이블 | 집과 단지를 연결하는 케이블 |
| Docker Bridge | 같은 네트워크의 컨테이너를 연결하는 가상 스위치 | 아파트 단지 내부 스위치 |
| Host NIC | 호스트가 외부 네트워크와 통신하는 실제 인터페이스 | 단지 밖으로 나가는 출입구 |

```
Docker Host: EC2 Ubuntu

MySQL 컨테이너      API 컨테이너       Nginx 컨테이너
eth0:172.18.0.4    eth0:172.18.0.3    eth0:172.18.0.2
     ↓ veth pair         ↓ veth pair        ↓ veth pair
              Docker Bridge (docker0 또는 사용자 정의 브리지)
                            ↓
                  호스트 네트워크 스택
                  (라우팅 · NAT · 방화벽 규칙)
                            ↓
                  호스트 물리 NIC (eth0 또는 ens5)
                            ↓
                    인터넷 또는 AWS VPC
```

> ⚠️ 컨테이너 내부에서 보이는 `eth0`이 실제로 호스트의 NIC에 직접 연결되는 게 아니다. 컨테이너는 **먼저 도커 브릿지에 연결**되고, 외부 통신이 필요한 경우 호스트의 라우팅, NAT, 방화벽 규칙을 통과해서 실제 네트워크 인터페이스로 나가게 된다.

##### 같은 브릿지 네트워크 내부에서 통신하는 경우

```
Nginx 컨테이너 → http://api:8080 요청
→ Docker Bridge (api 컨테이너로 전달)
→ API 컨테이너 (HTTP 응답)
→ Docker Bridge (HTTP 응답 전달)
→ Nginx 컨테이너
```

- Nginx는 API 컨테이너의 **IP를 알 필요가 없다.** 브릿지 네트워크에서 컨테이너 이름 ↔ 컨테이너 IP를 DNS에 등록해서 관리해주기 때문이다.
- 같은 네트워크라면 `-p` 옵션(포트 노출)이 필요 없다.
- API 외부 포트를 열지 않으면 접근 범위가 다음처럼 제한된다.

```
Nginx      → api:8080   ✅ 가능 (같은 브릿지 네트워크)
호스트     → api:8080   ✅ 가능 (호스트에서 접근)
외부 사용자 → api:8080  ❌ 불가능 (외부 포트 미노출)
```

##### 기본 Bridge vs 사용자 정의 Bridge

도커를 설치하면 기본으로 브릿지가 자동 생성된다. 컨테이너를 실행할 때 별도로 `--network` 옵션을 주지 않으면 기본 브릿지가 할당되는데, 실무에서는 **기본 브릿지보다 사용자 정의 브릿지를 더 선호**한다.

| 비교 항목 | 기본 `bridge` 네트워크 | 사용자 정의 브릿지 네트워크 |
|---|---|---|
| 생성 방식 | Docker 설치 시 자동 생성 | `docker network create`로 생성 |
| 컨테이너 이름 DNS | 기본적으로 지원하지 않음 | 지원 |
| 네트워크 분리 | 모든 기본 컨테이너가 같은 네트워크에 들어갈 수 있음 | 필요한 서비스끼리만 묶을 수 있음 |
| 실행 중 연결·해제 | 제한적 | `docker network connect`, `disconnect` 가능 |
| 실무 권장도 | 단순 테스트용 | 서비스 단위 구성에 권장 |

사용자 정의 브릿지 네트워크를 선호하는 이유 정리:
- **자동으로 DNS** 지원 (컨테이너 이름으로 통신 가능)
- **격리 수준이 더 높다** (프로젝트별 통신 범위 분리)
- **실행 중에도** 컨테이너를 네트워크에 연결하거나 분리할 수 있다

##### 네트워크 명령어

```bash
# 네트워크 생성
docker network create app-net
# 명시적으로 bridge 드라이버 지정 (디폴트가 bridge 드라이버)
docker network create --driver bridge app-net

# 만들고 나서 네트워크 스펙 확인
docker network inspect app-net

# 실행 중인 컨테이너를 네트워크에 연결
docker network connect app-net nginx-3

# 연결 해제
docker network disconnect app-net nginx-3
```

`inspect`로 볼 수 있는 것:

| 항목 | 의미 |
|---|---|
| `Driver` | 네트워크 드라이버 종류 |
| `Subnet` | 컨테이너에 할당할 IP 범위 |
| `Gateway` | 컨테이너의 기본 게이트웨이 |
| `Containers` | 연결된 컨테이너 목록 |
| `IPv4Address` | 각 컨테이너에 할당된 IP |

##### 실무적인 구조 패턴

**① Nginx - API - MySQL 분리**

Spring API 컨테이너를 두 개의 네트워크에 동시에 할당해서, 네트워크 계층별 가시성을 세밀하게 제어한다.

```
backend-net:   Spring API ←→ mysql:3306 ←→ MySQL

frontend-net:  사용자 → HTTPS:443 → Nginx → api:8080 → Spring API
```

- Nginx는 API를 볼 수 있음
- API는 DB를 볼 수 있음
- **Nginx는 DB를 볼 수 없음** (서로 다른 네트워크이므로)

**② 사이드카 패턴**

메인 컨테이너(Spring API)와 사이드카 컨테이너(로그 수집 Agent)를 함께 두는 패턴. 로그, 모니터링, 프록시, 보안, 포매팅 등의 **공통 관심사를 분리**하는 방식이다.

```
app-net (사용자 정의 Bridge):

Spring API → 로그 파일 또는 stdout → 로그 수집 Agent → 로그 전송 → 로그 저장소
```

**③ 워커 패턴**

사용자 요청을 바로 처리하지 않고, API, 메시지 브로커 등을 별도의 네트워크에 분리해서 **비동기로 작업을 처리**하는 패턴.

```
worker-net:
  Spring API → 작업 등록 → RabbitMQ / Redis
  Worker     → 작업 조회 → RabbitMQ / Redis

frontend-net:
  사용자 → Nginx → Spring API
```

##### 실무적으로 잘 사용하기

**① 기본 브릿지보다 사용자 정의 브릿지를 사용하자**
- 컨테이너 이름 DNS 지원
- 프로젝트별 통신 범위 분리
- 컨테이너 IP가 바뀌어도 자동으로 DNS가 처리해줌
- 실행 중에도 컨테이너를 네트워크에 연결·분리 가능

**② 컨테이너 IP 값을 설정에 하드코딩하지 않는다**

```yaml
# ❌ 하드코딩 - 나쁜 예
# application.yml 또는 .env 에
DB_HOST = 172.24.0.3
```

```yaml
# ✅ 권장 패턴 - DNS로 적기
DB_HOST = mysql
# Bridge DNS에 등록되어 있으니까
```

컨테이너 IP는 생성될 때마다 바뀔 수 있다. **컨테이너 이름**으로 통신하는 것이 올바른 방식이다.

##### Bridge Network 정리

- **한 도커 호스트 안에서** 컨테이너를 연결하는 가상 내부 네트워크
- 컨테이너는 각자 IP, Network Interface를 가진 **독립적인 네트워크 환경**을 가짐
- **같은 브릿지 네트워크에 있는 컨테이너끼리는 내부 통신 가능**
- 외부에 공개할 서비스만 `-p`로 호스트 포트랑 연결
- 기본 브릿지보다는 **사용자 정의 브릿지를 사용하는 게 좋다**

#### | Docker Network Driver

도커는 다양한 네트워크 드라이버를 제공한다.

| 드라이버 | 특징 |
|---|---|
| `bridge` | 기본값. 한 호스트 내 컨테이너 연결 |
| `host` | 거의 안 씀 |
| `overlay` | 쓰긴 하는데 직접 구현하지는 않음 |
| `macvlan` | 볼 일이 거의 없긴 함 |

##### 1. Bridge Driver

도커 호스트 내에서 컨테이너를 **가상의 스위치로 연결**하는 네트워크 드라이버. (앞서 설명한 Bridge Network가 이 드라이버를 사용한다)

```
사용자 → EC2 IP:80 또는 도메인:443
       → Nginx(172.18.0.2) → api:8080
       → Spring API(172.18.0.3) → mysql:3306
       → MySQL(172.18.0.4) [사용자 정의 Bridge: app-net]
```

##### 2. Host Network

컨테이너가 **별도의 네트워크를 만들지 않고 호스트의 네트워크 공간을 직접 공유**하는 방식. 컨테이너가 별도의 IP 주소를 할당받지 않고, **호스트의 IP 주소를 그대로 사용**한다.

| 구분 | Bridge | Host |
|---|---|---|
| 컨테이너 | 172.18.0.2:80 | 별도 IP 없음 |
| 호스트 | 10.0.1.10:8080 → 컨테이너 80 | 10.0.1.10:80 |
| 외부 | 10.0.1.10:8080 → 컨테이너 80 | 10.0.1.10:80 → 컨테이너 프로세스 |

장점:
- 브릿지 네트워크보다 네트워크 경로가 단순
- NAT, 포트 매핑이 필요 없음
- 호스트에서 실행 중인 서비스가 로컬호스트로 직접 접근해야 하는 경우에 유용

단점:
- 컨테이너와 호스트의 네트워크 격리가 사라진다
- 같은 포트를 사용할 수 없다 (호스트가 점유 중인 포트는 컨테이너가 사용하지 못함)
- 여러 서비스가 한 서버에 공존하는 경우에는 일반적으로 사용하기 어렵다

##### 3. Overlay Network

**여러 도커 호스트에 흩어져 있는 컨테이너를 하나의 가상 내부망처럼 연결**하는 네트워크. 브릿지는 한 대의 호스트에서만 동작하지만, 컨테이너가 여러 개의 호스트로 분리되어 있으면 브릿지로는 연결이 안 된다.

왜 사용하냐:
- 여러 EC2에 분산된 서비스 컨테이너를 연결할 수 있다
- 서비스가 다른 호스트로 이동해도 같은 오버레이 네트워크를 쓰면 통신 가능

> ⚠️ Overlay 네트워크를 쓰려면 **Docker Swarm**을 먼저 구성해야 한다.
> ```bash
> docker swarm init
> docker swarm join
> ```
> Swarm을 하려면 몇 가지 포트를 열어야 하는 요구사항이 있다.

| 포트 | 프로토콜 | 용도 |
|---|---|---|
| 2377 | TCP | Swarm 관리 제어 통신 |
| 7946 | TCP / UDP | 노드 간 통신 |
| 4789 | UDP | Overlay 데이터 경로 |

**Overlay vs Bridge 비교**

| 구분 | Bridge | Overlay |
|---|---|---|
| 통신 범위 | 한 대의 Docker Host | 여러 Docker Host |
| 주 사용 환경 | Compose, 단일 EC2 | Swarm, 다중 서버 |
| 관리 복잡도 | 낮음 | 높음 |
| 네트워크 구성 | 로컬 가상 브리지 | 여러 호스트 위의 분산 가상망 |

##### 4. Macvlan Driver (볼 일이 거의 없음)

컨테이너마다 별도의 MAC 주소를 부여해서 컨테이너가 마치 **물리 네트워크에 직접 연결된 것처럼** 보이게 하는 드라이버. 레거시 애플리케이션이나 레거시 물리 장비가 있는 경우에만 보통 사용하며, 거의 볼 일이 없다.

---

#### | Reverse Proxy

**클라이언트의 요청을 대신 받아 서버에 전달하고, 서버의 응답을 클라이언트로 반환하는 중개 역할을 하는 서버**

대표적인 구현체: **ALB**, **Nginx**

사용 이유:
- **보안 강화**: 실제 서버를 외부에 노출하지 않음
- **로드 밸런싱**: 트래픽을 여러 서버에 분산
- **캐싱**: 정적 리소스 등을 캐시해서 응답 속도 향상

---

### 🗨️ 오늘의 회고

- 오프라인 이후 수면 시간이 줄어서 체력적으로 벅찬 하루였다. 그래도 교육장에서 늦게까지 학습하며 위클리 챌린지 설계를 최대한 진행했다.
- Bridge Network가 "컨테이너마다 독립적인 네트워크 환경을 만드는 이유"부터 시작해서 veth pair, Docker Bridge, Host NIC까지 물리적인 구조로 풀어내 주니, 지금까지 막연하게 "컨테이너들이 서로 통신한다"고만 이해했던 부분이 훨씬 명확해졌다.
- 특히 **컨테이너 내부의 eth0이 실제로 호스트 NIC에 직접 연결되는 게 아니라, 브릿지를 먼저 거친다**는 사실이 인상적이었다. 당연하게 생각하고 지나쳤던 부분인데, 구조를 알고 나니 "왜 -p 옵션이 필요한지", "왜 같은 네트워크에서는 포트를 열지 않아도 통신이 되는지"가 자연스럽게 연결됐다.
- 기본 브릿지 vs 사용자 정의 브릿지에서 **DNS 자동 지원 여부**가 핵심 차이라는 점이 와닿았다. 컨테이너 IP는 재생성될 때마다 바뀌니, IP 대신 컨테이너 이름으로 통신하는 것이 훨씬 안전하고 유지보수도 쉽다. `DB_HOST = 172.24.0.3` 대신 `DB_HOST = mysql`로 적는 습관을 지금부터 들여야겠다.
- 사이드카, 워커 패턴은 Bridge Network를 활용한 실무 설계 패턴인데, 결국 "어떤 컨테이너가 무엇을 볼 수 있는가"를 네트워크로 명시적으로 설계하는 방식이라는 게 인상 깊었다. Nginx가 DB를 볼 수 없게 만드는 것도 코드가 아니라 **네트워크 구성**으로 강제한다는 점이 새로웠다.
- Overlay Network는 Docker Swarm과 세트라는 것, Swarm은 직접 구현하진 않지만 다중 서버 환경에서 브릿지의 한계를 넘기 위해 필요하다는 큰 그림을 이해하는 데 의미가 있었다.
