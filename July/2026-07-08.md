#### | ECR (Elastic Container Registry)

**AWS에서 제공하는 컨테이너 이미지 저장소 서비스로, 사용자가 직접 서버를 구축·운영하지 않아도 이미지 저장, 버전 관리, 접근 제어, 보안 스캔까지 자동으로 처리되는 완전 관리형 레지스트리**

한마디로 **AWS 도커 이미지 저장소**이며, AWS 서비스이기 때문에 다음과 같은 특징을 가진다.
- **IAM 권한 제어** — 누가 이미지를 push/pull 할 수 있는지 세밀하게 통제
- **이미지 보안 취약점 스캔** — 이미지를 올리면 자동으로 스캔
- **ECS, EKS 같은 AWS 서비스와 자연스러운 연동**
  - ECS → 컨테이너 실행 환경 관리형 서비스
  - EKS → 쿠버네티스 마스터 관리형 서비스
- **이미지 수명주기(Lifecycle) 관리**

##### 컨테이너 이미지 관련 용어 정리

| 용어 | 설명 |
|---|---|
| **Image** | 도커에서 애플리케이션 실행에 필요한 코드, 라이브러리, 설정을 묶은 실행 패키지 |
| **Registry** | 이미지를 저장하는 저장소 (ECR, Docker Hub, GitHub Container Registry 등) |
| **Repository** | 특정 이미지를 이름 기준으로 버전별로 모아두는 공간. Registry 안의 "폴더" 같은 개념 (예: ECR에서 `backend` Repository를 만들면 백엔드 이미지들이 그 안에 모임) |
| **Tag** | 같은 이미지에서 버전을 구분하는 데 사용하는 표기 (예: `my-backend:1.0.0`) |

##### 태그(Tag)의 종류

| 태그 유형 | 설명 |
|---|---|
| `latest` | 가장 최근에 올라간 태그 |
| `1.0.0` 같은 버전 태그 | 사람이 읽기 쉬운 버전 표기 |
| 커밋 SHA 값 태그 | 어떤 커밋에서 빌드됐는지 정확히 추적 가능 |

##### ECR 이미지 주소 구조

```
123456789012.dkr.ecr.ap-northeast-2.amazonaws.com/my-backend:v1.0.0
     │                    │                          │         │
  AWS 계정 ID            리전                  ECR 리포지토리 이름   태그명
```

##### ECR을 왜 사용하는가?

핵심 목적은 **컨테이너 이미지를 AWS 환경 안에서 안전하게 저장하고, AWS 서비스가 빠르게 가져와 실행할 수 있도록 하기 위함**이다. ECR은 AWS 서비스이기 때문에 외부에서 접근할 때는 IAM 권한을 통해 제약을 걸 수 있고, AWS 서비스만 사용하는 환경이라면 연동성이 훨씬 좋다.

**1. Docker Image를 보관하기 위해**
애플리케이션을 도커로 배포하려면 빌드가 완료된 이미지가 필요하다. 이 이미지를 안전하게 보관하는 곳이 ECR이다.

**2. ECS·EC2·EKS 같은 AWS 내부 컴퓨팅 리소스와 연결하기 위해서**
ECR은 AWS 내부 서비스와 연결하기 편하고, 비용 측면에서도 유리하다. AWS는 **내부 네트워크 통신 비용이 외부 네트워크 통신 비용보다 저렴**하다.
- 예: 이미지 크기가 1.2GB인 경우, Private Subnet의 EC2(WAS)가 이미지를 받아올 때 NAT Gateway를 거치면 컴퓨팅 비용 + 네트워크 비용이 함께 발생한다.

**3. 보안 스캔**
ECR은 이미지를 올리면 자동으로 보안 취약점을 스캔해준다.
- **Basic Scanning** — 이미지를 업로드하면 자동으로 실행되는 기본 스캐닝
- **AWS Inspector** — 취약점을 더 깊이 조사하는 서비스

> 💡 실무에서는 CI/CD 단계별로 다른 작업을 수행한다. PR을 올릴 때 하는 작업과 main 브랜치에 merge 될 때 하는 작업이 다른 것처럼, 보안 스캔 시점도 파이프라인 단계에 맞춰 나눠서 운영한다.

##### 실무에서 ECR을 사용하는 방식

개발자가 로컬에서 ECR로 직접 이미지를 push하는 방식이 아니라, **CI/CD 파이프라인이 자동으로 push/pull을 수행**하는 것이 일반적이다.

```
개발자 → GitHub → GitHub Actions/CodeBuild → Amazon ECR → ECS/EKS/EC2

1. 개발자가 코드 push
2. GitHub Actions Workflow 실행
3. 테스트 실행
4. Docker Image build
5. ECR에 image push
6. ECS/EKS/EC2가 image pull
7. 새 버전 배포
```

EC2에서 이미지를 직접 가져올 때는 다음과 같은 흐름을 사용한다.

```shell
# ECR 로그인
aws ecr get-login-password --region ap-northeast-2 \
  | docker login --username AWS --password-stdin \
  123456789012.dkr.ecr.ap-northeast-2.amazonaws.com

# 이미지 가져오기
docker pull 123456789012.dkr.ecr.ap-northeast-2.amazonaws.com/my-backend:20260708-a1b2c3d

# 컨테이너 실행
docker run -d --name my-backend -p 8080:8080 \
  123456789012.dkr.ecr.ap-northeast-2.amazonaws.com/my-backend:20260708-a1b2c3d
```

> ⚠️ 이 흐름을 그대로 반복 적용하다 보면 **이미지가 계속 쌓이는 문제**와 **중단 배포 문제**가 함께 발생할 수 있다. 이는 곧이어 정리한 "ECR 사용 시 주의점"과 "무중단 배포" 개념으로 이어진다.

##### ECR 사용할 때 주의하면 좋은 것들

**1. `latest` 태그만 사용하지 않는다**
`latest`만 사용하면 지금 내가 받은 이미지가 명확히 어떤 버전인지 알 수 없다. 여러 개발자가 동시에 개발하는 상황에서는 "내가 마지막에 빌드한 버전"이라는 보장이 되지 않는다. → 커밋 SHA나 버전 태그처럼 **명확하게 구분되는 태그**를 함께 사용해야 한다.

**2. ECR도 비용이 나가는 서비스라는 것을 잊지 않는다**
저장하는 건 결국 이미지이고, CI/CD 파이프라인이 돌 때마다 1.2GB 같은 이미지가 계속 저장된다.
> 💡 실제 사례: 4년 정도 운영한 어떤 회사의 ECR 비용이 월 20만 원 수준까지 쌓인 경우가 있었다. → **Lifecycle Policy**를 설정해서 "N일 지나면 삭제" 또는 "최근 N개 이미지만 보관" 같은 규칙을 반드시 적용해야 한다.

**3. IAM 권한을 최소한으로 분리해서 부여한다**
ECR 권한은 크게 `push`와 `pull`로 나뉜다.
- 실행 서버(EC2, ECS 등)는 **pull 권한만**
- CI/CD 서버는 **push 권한**
이렇게 역할별로 권한을 최소화해서 부여하는 것이 안전하다.

**4. ECR 로그인은 영구 로그인이 아니다**
Docker Hub는 한 번 로그인하면 데스크톱이 유지되는 동안 로그인 상태가 유지되지만, **ECR CLI 로그인은 인증 토큰 유효 시간이 12시간**으로 제한되어 있다. 그래서 CI/CD 스크립트에는 배포할 때마다 ECR 로그인 과정을 항상 포함시켜야 한다.

##### Private Subnet에서 ECR에 접근하는 방법

Private Subnet은 인터넷 연결이 없는 네트워크이고, ECR은 VPC 내부에 있는 서비스가 아니다. 따라서 접근 방법은 두 가지다.

| 방법 | 설명 |
|---|---|
| **NAT Gateway** | 인터넷을 경유해서 ECR에 접근. 비용(NAT 사용료 + 트래픽 비용)이 발생 |
| **VPC Endpoint** | NAT나 인터넷망 없이 **AWS 내부망을 통해서만** ECR과 통신. 비용·보안 측면에서 유리 |

#### | Code Deploy

**AWS에서 제공하는 자동화된 애플리케이션 배포 서비스**

> 🤔 "GitHub Actions에서 배포까지 다 하면 되는데, 왜 굳이 Code Deploy를 쓰지?"라는 의문이 들 수 있다. Code Deploy는 **"어떤 방식으로, 어떤 서비스에, 어떤 순서로 배포할 것인가"를 AWS 안에서 (반)자동화해주는 서비스**다.

##### GitHub Actions만으로 배포할 때의 흐름

```
1. 코드 푸시
2. GitHub Actions 러너가 동작
   a. 코드 풀
   b. 테스트
   c. 도커 빌드
   d. ECR 푸시
   e. SSH로 배포 대상 서버에 직접 접속
   f. docker pull
   g. docker run -d --name ~~
```

Code Deploy를 도입하면 위 흐름 중 **e ~ g 단계(실제 서버 접속·이미지 반영·컨테이너 재실행)를 Code Deploy가 대신 담당**하게 된다.

##### Code Deploy를 사용했을 때 얻는 이점

**1. 서버가 여러 대일 때 반복 작업 문제가 크게 줄어든다**
- 배포 대상이 EC2 10대 정도라면 SSH로 직접 처리해볼 만하지만, 1000대라면 병렬로 처리하더라도 여전히 SSH 접속이 필요해 관리가 힘들어진다.
- 만약 **VM이 동적으로 생성되는 환경**(Auto Scaling Group)이라면 문제는 더 커진다. 예를 들어 EC2가 2대 → 10대로 늘어나는 상황에서, 기존 2대는 IP를 알고 있지만 **새로 생긴 8대는 배포 시점에 IP를 어떻게 알아낼지**가 애매해진다. Code Deploy는 이런 동적 인프라에 대한 배포를 태그 기반으로 처리해준다.

**2. 배포 실패 시 자동 대응이 조금 더 강화된다**
배포가 실패하면 롤백을 해야 하는데, 이때 헬스 체크·메트릭·롤백 대상 이미지 같은 값들은 결국 우리가 설정하는 값이다. Code Deploy는 "배포가 실패했을 때 어떻게 할 것인가, 어떤 이미지로 롤백할 것인가"를 표준화된 방식으로 다뤄준다.

##### CI/CD 전체 구조 속에서 Code Deploy의 위치

```
[ CI ]              [ Artifact ]         [ CD ]           [ Running Service ]
Build & Test  →   jar / zip / image  →   Deploy    →      실제 실행 중인 서비스
```

##### Code Deploy의 구성 요소

| 구성 요소 | 설명 |
|---|---|
| **Application** | Code Deploy에서 사용하는 배포 단위 (예: 백엔드 서비스, 프론트엔드 서비스) |
| **Deployment Group** | 어디에 배포할지 정하는 그룹 |
| **Revision** | 실제로 배포할 결과물(패키지, 버전) |
| **AppSpec** | 배포를 어떻게 진행할지 알려주는 배포 설명서 |
| **Deployment Configuration** | 배포 속도와 방식을 정하는 설정 |

**Deployment Group**은 EC2를 하나하나 지정하는 방식이 아니라, 보통 **태그(K-V Tag)** 조합으로 관리한다.

```
key: app,  value: was
key: env,  value: prod
```

→ "이 태그 조합을 가진 대상에게만 배포해줘" 라는 방식으로 동작하기 때문에, **EC2를 만들 때 태그 설정을 반드시 해두어야** 배포 대상에서 누락되지 않는다.

**AppSpec**은 배포 순서를 정의하는 설명서다. 예시는 다음과 같다.

```yaml
hooks:
  BeforeInstall: 기존 앱 정리
  ApplicationStart: 새 앱 실행
  ValidateService: 정상 실행 확인
```

**Deployment Configuration**에서는 배포 전략(Blue/Green, Rolling 등)과 배포 속도를 지정할 수 있다.
- **AllAtOnce** — 한 번에 다 배포하기
- **OneAtATime** — 한 번에 하나씩 배포하기

##### GitHub Actions vs Code Deploy

| 구분 | GitHub Actions | Code Deploy |
|---|---|---|
| 성격 | 범용 자동화 도구 | AWS 배포 자동화 서비스 |
| 주 사용처 | 테스트, 빌드, 스크립트 실행, 외부 도구 연결 | EC2/ECS/Lambda 배포 관리 |
| 배포 전략 | 직접 구현해야 함 | AWS 배포 전략을 설정값으로 사용 |
| AWS 연동 | Credentials 설정 필요 | AWS 서비스와 네이티브로 연동 |
| 적합한 사용처 | 간단한 배포, 전체 워크플로우 제어 | AWS 리소스 대상 안정적 배포 |

> 💡 실무에서는 이 둘을 함께 사용하는 경우가 많다. **GitHub Actions**로 Git Trigger 기반의 CI(이미지 빌드까지)를 처리하고, **Code Deploy**가 AWS 리소스로의 실제 배포를 담당하는 조합이 일반적이다.

##### Code Deploy의 단점

1. **설정할 것이 많다** — 초기 세팅에 시간이 꽤 걸린다.
2. **Agent 문제** — 배포 대상 EC2에 Code Deploy Agent를 설치해야 하고, 이 에이전트가 항상 실행 중이어야 배포가 가능하다. 그만큼 관리 포인트가 하나 더 늘어난다.

##### Code Deploy의 장점

- 배포 전략을 직접 구현하지 않아도 되고, **표준화된 방식**으로 관리할 수 있다.
- 배포 타이밍 조절도 우리가 직접 스크립트로 짜지 않아도 된다.
- EC2 같은 동적으로 생성/삭제되는 자원에 배포하기 편하다.

##### 언제 Code Deploy를 선택하는가?

1. EC2 배포를 표준화하고 싶고, 수동 SSH 배포를 없애고 싶을 때
2. 여러 EC2가 동적으로 생성·삭제되는 환경일 때
3. AWS 중심으로 CD 파이프라인을 구성할 때
4. 배포 전략(Blue/Green, Rolling 등)을 좀 더 다양하게 선택하고 싶을 때

### 🗨️ 오늘의 회고

- 과제 수정을 오늘까지 마무리하고 내일부터 9주차 위클리 챌린지를 진행할 계획이다. 로컬에서 이미지를 빌드하고 ECR에서 pull 받아 컨테이너를 실행하는 방식을 직접 해보니, 예상보다 속도가 훨씬 빨라서 놀랐다. 어제 배운 무중단 배포 개념과 연결해서 생각해보니, ECR·Code Deploy 같은 관리형 서비스들이 왜 존재하는지가 더 잘 이해됐다.
- ECR이 단순한 "이미지 저장소"가 아니라 **IAM 권한, 보안 스캔, AWS 내부망 연동**까지 포함하는 서비스라는 걸 알게 됐다. 특히 "AWS 내부 네트워크 통신 비용이 외부보다 저렴하다"는 이유로 ECR을 쓴다는 관점은 신선했다 — 단순히 편의성 때문이 아니라 **비용 구조 자체가 다르다**는 걸 처음 인지했다.
- `latest` 태그만 사용하면 안 되는 이유, ECR도 결국 돈이 나가는 서비스라 Lifecycle Policy가 필요하다는 점, 그리고 ECR 로그인 토큰이 12시간짜리라는 점은 실무에서 바로 마주칠 법한 함정들이라 특히 기억해둬야겠다.
- Code Deploy 파트에서는 "GitHub Actions로도 배포가 되는데 왜 굳이?"라는 질문에 대한 답이 명확했다. **서버가 수백~수천 대이거나 Auto Scaling으로 동적으로 생기는 환경에서는 SSH 기반 배포 자체가 한계에 부딪힌다**는 설명이 와닿았다. 태그 기반으로 Deployment Group을 관리한다는 개념도 어제 배운 무중단 배포의 "재현성" 요구사항과 자연스럽게 연결됐다.
- GitHub Actions와 Code Deploy를 "이거 아니면 저거"가 아니라 **CI는 GitHub Actions, CD는 Code Deploy**로 역할을 나눠서 함께 쓰는 조합이 실무에 가깝다는 정리가 오늘 배운 내용 중 가장 실용적으로 느껴졌다.
