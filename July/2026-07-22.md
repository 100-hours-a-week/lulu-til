#### | Job

**한 번 실행하고 정상적으로 끝나야 하는 작업을 관리하는 컨트롤러**

| 구분 | Deployment | Job |
|---|---|---|
| 목표 | 애플리케이션을 **계속 실행 상태로 유지** | 작업을 **한 번 성공**시키는 것이 목표 |
| 특징 | 업데이트, 롤백 등 지속적인 운영을 처리 | 애플리케이션이 한 번 실행되고 종료 |

##### Job을 왜 쓸까? 그냥 Pod로 apply하면 안 되나?

일회성 작업을 **일반 Pod로 실행하면 문제가 생길 수 있다.**

- 작업이 실패하면 → **다시 실행해야 함** (직접 재시도 처리를 해줘야 함)
- 작업이 실패한 것을 **파드 로그를 계속 쳐다보고 있어야 함** (수동 모니터링)
- 중간에 실패했을 때 **핸들링하기도 까다로워진다**
- **병렬로 처리**해야 하는 경우 여러 개의 파드를 직접 띄우고 관리해야 함

**일회성 작업의 예시**
- 데이터베이스 마이그레이션 (예: MySQL의 chatting 테이블을 MongoDB로 옮겨야 하는 작업)
- 파드 하나로만 처리하기에는 시간이 너무 오래 걸리는 경우도 있다

##### Job의 역할

- Job은 **작업이 성공할 때까지 해당 작업에 사용되는 Pod을 관리**한다.
- 파드들의 **성공 횟수를 관리**한다.
- 성공 횟수가 특정 지정한 수가 되면, **작업이 성공했다고 판단**한다.
- → 일회성 작업을 **안정적이고 자동화된 방식**으로 처리할 수 있게 해준다.

##### Job의 동작 방식

```
Job 생성
   ↓
Pod 생성
   ↓
컨테이너에서 작업 실행
   ↓
종료 코드 확인
   ↓
   ├── 0        → 성공
   └── 0이 아님 → 실패 및 재시도
```

> ⚠️ 코드에서 `try-catch` 구문을 쓸 때, `catch` 이후에 **에러 로그만 찍고 넘어가면 종료 코드가 0으로 판단**되어 버린다. 실제로는 실패한 작업인데 Job 입장에서는 "성공"으로 잘못 인식할 수 있으므로, 실패를 감지했을 때는 반드시 0이 아닌 종료 코드로 프로세스를 종료시켜야 한다.

##### Job YAML 구조

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: example-job
spec:
  completions: 3     # 총 작업 완료 수 (3개의 Pod 실행 필요)
  parallelism: 2      # 동시에 실행할 Pod 수 (병렬 실행)
  template:
    metadata:
      name: example-pod
    spec:
      containers:
        - name: example-container
          image: busybox
          command: ["sh", "-c", "echo Hello World! && sleep 30"]
      restartPolicy: OnFailure   # 실패 시 재시도
```

##### 핵심 설정

| 설정 | 의미 |
|---|---|
| **completions** | 전체 작업이 완료되기 위해 필요한 **성공 횟수** |
| **parallelism** | 동시에 실행할 수 있는 **파드 수** |
| **restartPolicy** | 재시도 할지 말지 정하는 정책 |

**restartPolicy 옵션**

| 값 | 동작 |
|---|---|
| **OnFailure** | 실패하면 재시도 (`backoffLimit`으로 몇 번 재시도할지 지정) |
| **Never** | 실패하면 중지 (재시도하지 않음) |

##### 실무적으로 사용하는 예시

- 데이터베이스 초기화
- 데이터베이스 스키마 마이그레이션
- 일회성 데이터 변환
- 대량 파일 처리
- 테스트 데이터 생성
- 배포 전 사전 작업

##### concurrencyPolicy

**이전 Job이 끝나지 않았는데, 다음 실행 시간이 됐을 때** 어떻게 처리할지를 정하는 옵션이다.

| 설정 | 동작 |
|---|---|
| **Allow** | 이전 Job이 실행 중이어도 새 Job 실행 |
| **Forbid** | 이전 Job이 실행 중이면 이번 실행을 건너뜀 |
| **Replace** | 이전 Job을 종료하고 새로운 Job 실행 |

#### | Networking

**쿠버네티스 Networking은 클러스터 안의 Pod와 Service가 서로 통신하고, 필요에 따라 클러스터 외부와도 연결될 수 있도록 네트워크 경로를 구성하고 관리하는 체계**

##### 쿠버네티스에서 네트워크가 필요한 이유

```
Frontend Pod → Backend Pod → Database Pod
Backend Pod → Redis
Backend Pod → 외부 결제 API
   └ 백엔드 파드는 워커노드에 배치
       └ 몇 번 포트를 열어야 하는데요? 어떤 노드의?
외부 사용자 → Frontend 애플리케이션
   └ 프론트엔드 파드 IP를 모름
운영자 → Kubernetes API Server
```

- 파드는 확장성 때문에, **메인 비즈니스 로직이 하나인 컨테이너**를 담는 단위로 여러 개에 나눠져서 배치된다.
- 파드는 **주기적으로 생성·파괴**되고, 그때마다 **IP가 달라진다.**
- 쿠버네티스는 단순히 컨테이너를 실행하는 것을 넘어서, **파드들이 서로 통신하는 경로를 잘 잡아주는 것이 매우 중요하다.**

##### 네트워크 동작 방식에 따른 구조

1. 같은 Pod 내부의 컨테이너 간 통신
2. Pod와 Pod 간 통신
3. Pod와 Service 간 통신
4. 클러스터 내부와 외부 네트워크 간 통신

---

##### 1. 같은 Pod 내부의 컨테이너 간 통신

같은 파드 안에서는 **네트워크 공간을 공유**한다 — `localhost`를 쓰는 것과 같다.

| 구분 | 내용 |
|---|---|
| 특징 | 포트가 겹치면 안 된다 |
| 장점 | 통신 속도가 엄청 빠름 |
| 단점 | 포트 공유 문제 (같은 포트를 두 컨테이너가 동시에 못 씀) |

##### 2. Pod와 Pod 간 통신

파드는 모두 **고유한 IP**를 가지고 있고, 서로 같은 노드에 있을 수도 있고 다른 노드에 있을 수도 있다.

**같은 노드에 있는 Pod 간 통신**

```
Pod 1 Network              Pod 2 Network
   eth0                        eth0
    │                            │
  veth0                       veth1
      \                       /
          cbr0 (브릿지)
             │
            eth0
             │
        Root Network
```

```
Worker Node A
Pod A 10.244.1.10 → Pod B 10.244.1.11
```

→ **내부 브릿지 네트워크(cbr0)로 통신**한다.

**서로 다른 노드 간 Pod끼리 통신**

```
Node 1 Root Network              Node 2 Root Network
Pod1  Pod2                        Pod3  Pod4
 └veth┘cbr0┘eth0                  eth0┘cbr0┘veth┘
              └─────── Route Table ────────┘
```

- **TCP 프로토콜을 UDP 프로토콜로 감싸고**, UDP를 통해 노드 간 통신한 뒤, 목적지 노드에서 TCP를 까서 파드 간 통신을 완성한다 → 이 방식이 **VXLAN**이다.
- 서로 다른 노드에 있는 파드끼리 통신하려면 다음 세 가지를 알아야 한다.
  1. 목적지 파드가 **어느 노드**에 있는지
  2. 목적지 파드 네트워크로 **어떻게 전달**할지
  3. 노드 사이의 패킷을 **어떤 방식으로 운반**할지 (VXLAN, IP in IP, BGP, eBPF)
- 실제 네트워크 구성은 **CNI 플러그인**(Flannel, Calico, Cilium 등)이 담당한다.
- **쿠버네티스는 자체적으로 모든 패킷을 전달하지 않는다** — CNI 플러그인이 구현해놓은 네트워크 구성을 사용할 뿐이다. 쿠버네티스는 어디까지나 **오케스트레이션 프로그램**이지, 네트워크 프로그램이 아니다.

##### 3. Pod와 Service 간 통신

- 파드는 IP·노드가 계속 변경되므로, **서비스라는 대표번호 하나를 세워두고 그 뒤에서 내부로 전달**한다.

```
http://backend-service:8080
   ↓
EndpointSlices
   - 10.244.10.3
   - 10.244.10.4
   - 10.244.20.5
```

- **kube-proxy**가 API Server한테 물어봐서, etcd에 있는 EndpointSlices 정보를 가져온다.
- 리눅스 커널 기능 중 **iptables**를 이용해서, 네트워크 커널의 라우팅 테이블을 관리한다.

##### 4. 클러스터 내부와 외부 네트워크 간 통신

| 방향 | 명칭 | 예시 |
|---|---|---|
| 외부 → 내부 | **Ingress 통신** | 외부 사용자의 접속 |
| 내부 → 외부 | **Egress 통신** | apt 패키지 다운로드, OAuth 인증, Gen AI API 호출 |

- 외부 사용자가 쿠버네티스 내부로 통신하는 것을 **인그레스 트래픽**이라고 한다.
- **Ingress**는 HTTP/HTTPS 라우팅을 해주는 쿠버네티스 API 리소스(컴포넌트)다.

**인그레스 통신의 3가지 방식**

**1. NodePort를 통한 접근**

```
외부 사용자 → Worker Node IP:NodePort → Service → Pod
http://203.0.113.10:30080
```

**2. LoadBalancer Service를 통해서 접근**

```
외부 사용자 → 클라우드 Load Balancer → Service → Pod
```

> ⚠️ `kubeadm`으로 직접 구축한 클러스터는 클라우드 LB와 연동이 안 되어 있어서 쓰기 어렵다. **EKS**처럼 클라우드가 통합 관리하는 환경에서 사용한다.

**3. Ingress를 통해서 접근 (실무에서 대부분의 케이스)**

- 웹서비스가 여러 개이고, URL 도메인에 따라 요청을 구분해야 하는 경우 (`api.domain.com`, `api.domain.com/users`, `api.domain.com/auth` 등)
- URL, Path 기준으로 담당 Service에게 요청을 전달해야 하는데, 이건 **HTTP 레벨의 로드밸런서**가 필요하다는 의미다 → **L7 로드밸런서**

```
api.example.com → Backend Service
www.example.com → Frontend Service
```

**대표적인 Ingress Controller**
- Nginx Controller
- Traefik Controller
- Envoy
- API Gateway

#### | 쿠버네티스 네트워크 구성 요소 정리

| 구성 요소 | 역할 |
|---|---|
| **Pod IP** | 각 파드에 할당되는 IP. 결국 모든 요청은 다 Pod IP로 전달됨 |
| **Service IP** | 네트워크 인터페이스에 직접 연결되지 않는 **가상의 IP**. 여러 파드에 접근할 수 있는 엔드포인트 제공 |
| **Node IP** | 워커노드·마스터노드 IP (EC2 주소, 물리 장비 IP) |
| **CNI** | Pod 네트워크를 실제 구성하는 구현체. Pod 네트워크 인터페이스 구성, Pod IP 할당, Pod 네트워크 연결, 노드 간 Pod 통신 경로 제공 |
| **kube-proxy** | 서비스 트래픽을 특정 파드로 전달하는 네트워크 규칙을 구성하는 컴포넌트 |

##### 전체 통신 흐름

```
외부 사용자
   ↓
NodePort 또는 Ingress
   ↓
Frontend Service (kube-proxy)
   ↓
Frontend Pod
   ↓
Backend Service (kube-proxy)
   ↓
Backend Pod
   ↓
외부 또는 내부 Database
```

##### 구성요소별 역할 정리

| 구성 요소 | 역할 |
|---|---|
| **CNI** | 파드 네트워크 구성 / 패킷이 이동할 수 있게 네트워크 경로 만들기 |
| **Service** | 변경될 수 있는 파드 앞에서 접근 지점 제공 |
| **kube-proxy** | 서비스 트래픽이 실제 파드로 전달되도록 노드의 네트워크 규칙 구성 |
| **CoreDNS** | Service Domain → Service IP로 변환 |
| **Ingress, NodePort, LoadBalancer** | 외부 트래픽이 내부 클러스터로 전달되게 하는 역할 |

##### 실무에서 네트워크를 구성할 때

**1. 애플리케이션끼리는 대부분 Service로 통신한다.** Pod를 고정시키거나 노드를 특정할 수도 있긴 하지만, 실무에서는 거의 쓰지 않는다.

```yaml
# application.yml
backend_url = 10.244.10.30       # X (직접 IP 사용 지양)
backend_url = backend-service:8080  # O (서비스명 사용)
```

**2. 외부 공개가 필요 없으면 대부분 ClusterIP를 사용한다.** 백엔드 API나 프론트엔드도 외부에서 직접 연결이 필요 없을 때는 대부분 ClusterIP로 서비스를 구성한다.

**3. 외부 노출 지점을 최소화해야 한다.** FE, BE, MySQL을 전부 NodePort로 열면 노출 지점이 많아져서 좋지 않다.

```
외부 → Ingress → Backend Service
              → kube-proxy → EndpointSlice → iptables update
              → Backend Pod
```

**4. Namespace를 고려해야 한다.** 같은 네임스페이스에서는 서비스명만 호출하면 되지만, 다른 네임스페이스에서는 네임스페이스까지 명시해야 한다.

```
backend-service:8080
# 다른 네임스페이스
backend-service.prod.svc.cluster.local

# 통신 규칙을 설정하면 이것도 막을 수 있음
# NetworkPolicy를 설정하면 어디에서 어떤 트래픽이 올 수 있는지 결정할 수 있다
```

#### | kube-proxy

**각 노드에서 Service와 EndpointSlice 정보를 감시하고, Service로 들어온 트래픽이 실제 Pod로 전달되도록 노드의 네트워크 전달 규칙을 구성하는 컴포넌트**

- 서비스는 특정 네트워크 인터페이스에 직접 연결되어 있지 않은 **가상의 리소스**다 — 실제로 존재하지 않는다.
- 실제로 존재하는 건 **파드, 노드, 네트워크 인터페이스**(가상으로 있기도 함)뿐이다.
- 가상 리소스(Service)에게 요청을 보내려면, 실제 존재(Pod)에게 전달해줄 **중간 대리자**가 필요한데 이게 바로 **kube-proxy**다.

```
Pod → Service ClusterIP → kube-proxy가 구성한 네트워크 규칙 → Backend Pod
```

##### 이름이 "Proxy"라서 모든 요청이 kube-proxy를 통해 지나갈까?

→ **아니다.** kube-proxy는 **네트워크 규칙을 설정해두는 컴포넌트**일 뿐이다. 실제 트래픽은 리눅스 커널 안에 설정된 라우팅 규칙을 통해 전달된다.

- kube-proxy는 기본적으로 **모든 노드**에 배치된다 (워커노드, 마스터노드 모두).
- 파드의 위치를 예측하기 힘들기 때문에 — 어떤 파드가 어떤 서비스를 호출할지 모르므로 — 서비스에 대한 목적지를 어느 노드에서든 무조건 설정할 수 있어야 한다.

```
Control Plane Node  - kube-proxy Pod
Worker Node A        - kube-proxy Pod
Worker Node B        - kube-proxy Pod

kubectl get pods -n kube-system -o wide
```

##### kube-proxy 동작 방식

기본적으로 API Server에게 아래 정보를 감시한다.
- **Service** (Service IP, Service Port, Protocol)
- **EndpointSlices**

```yaml
apiVersion: v1
kind: Service
metadata:
  name: backend-service
spec:
  selector:
    app: backend
  ports:
    - port: 80
      targetPort: 8080
```

**동작 단계**

```
1. API Server에서 Service 정보를 감시한다.
2. API Server에서 EndpointSlice 정보를 감시한다.
3. 사용할 수 있는 Backend Pod 목록을 확인한다.
4. 노드에 패킷 전달 규칙을 구성한다.
5. Service 트래픽이 들어오면 커널이 규칙에 따라 Backend를 선택한다.
6. 패킷의 목적지를 Service IP에서 Pod IP로 변경한다.
```

> 💡 **4번까지는 kube-proxy가** 직접 규칙을 설치하는 작업이고, **5번부터는 리눅스 커널**이 그 규칙에 따라 실제로 동작한다.

**실제 통신 흐름**

```
kube-proxy → 커널에 규칙 설치

Frontend Pod → Linux 커널 네트워크 규칙 → Backend Pod
```

##### kube-proxy의 동작 모드

- **iptables** ← 가장 기본
- **IPVS**
- **nftables**

**iptables 동작 예시**

```
Service ClusterIP:Port
   ↓
iptables 규칙
   ↓
Endpoint PodIP:Port

# 예시
10.96.20.30:80 (Service IP:Port)로 온 요청
   ↓
10.244.1.10:8080 또는 10.244.2.15:8080 으로 DNAT
```

```bash
iptables -save | grep KUBE-SVC   # 규칙 확인 명령어
```

##### kube-proxy 역할 정리

- Service 감시
- EndpointSlice 감시
- ClusterIP 규칙 구성
- NodePort 규칙 구성
- Backend Endpoint 선택
- Service IP를 Pod IP로 DNAT

```
Frontend Pod → Backend Service 10.96.20.30:80
   → kube-proxy가 구성한 규칙
   → Backend Pod 10.244.2.15:8080
   → CNI가 구성한 네트워크 경로
   → 실제 Backend Pod 도착
```

- kube-proxy는 서비스의 주소를 실제 **EndpointSlice의 IP:Port**로 변환해서, 그걸 각 노드에 저장하는 역할을 해준다.
- 서비스를 호출해도 실제로는 **리눅스 커널의 iptables(라우팅 테이블)**를 거쳐 백엔드 Pod IP로 전달되는 것이다.

#### | CNI (Container Network Interface)

**Pod가 생성되거나 삭제될 때 네트워크 인터페이스, IP 주소, 라우팅 등을 설정하고 정리하기 위한 표준 규격**

- 컨테이너 네트워크를 설정하기 위한 **표준 규격**이다.
- **컨테이너 런타임 ↔ 네트워크 플러그인**이 서로 통신할 때 사용할 약속이다.

##### CNI에서 정의되어 있는 것들

- 네트워크 설정 파일의 형식
- 컨테이너 런타임이 플러그인을 호출하는 방법
- 플러그인을 순서대로 실행하는 방법
- 플러그인이 실행 결과를 반환하는 형식
- 여러 플러그인이 기능을 나누어 처리하는 방법

##### CNI와 CNI 플러그인 관계

| 구분 | 설명 | 예시 |
|---|---|---|
| **CNI** | 네트워크 플러그인이 따라야 하는 표준 규격 | CNI Specification |
| **CNI 플러그인** | CNI 규격에 따라 실제 네트워크를 구성하는 구현체 | Flannel, Calico, Cilium |
| **네트워크 기능** | 플러그인이 실제로 제공하는 기능 | IP 할당, 라우팅, 터널링, NetworkPolicy |

##### CNI를 사용하는 이유

| 이유 | 설명 |
|---|---|
| 네트워크 설정 자동화 | Pod 생성과 삭제에 맞춰 네트워크를 자동으로 연결하고 정리 |
| 표준화 | 컨테이너 런타임이 특정 네트워크 제품의 구현 방법을 알 필요가 없음 |
| 플러그인 선택 가능 | 환경에 따라 Flannel, Calico, Cilium 등 적합한 구현체를 선택할 수 있음 |
| Pod IP 관리 | 중복되지 않는 Pod IP를 할당하고 삭제 시 회수 |
| 노드 간 통신 구성 | 서로 다른 노드에 있는 Pod가 통신할 수 있도록 경로를 구성 |
| 환경 확장 | 클라우드, 온프레미스, 베어메탈 등 다양한 인프라에서 사용 가능 |

##### CNI는 어디에 있는가

- **모든 노드**에 CNI 플러그인이 동작하고 있다.
- Pod는 노드에서 만들어지고, 만들어진 파드는 네트워크에 등록되어야 한다. 이 등록을 CNI 플러그인이 담당한다 (IP 할당, 네트워크 인터페이스 연결).

```bash
kubectl get pods -A -o wide
```
→ 이 파드들 중 하나라도 노드가 `Ready`가 아니라면 통신이 안 된다.

##### CNI가 동작하는 과정

```
사용자
   │ kubectl apply
   ▼
API Server
   ▼
Scheduler       — Pod를 실행할 노드 결정
   ▼
해당 노드의 kubelet
   │ CRI 요청
   ▼
Container Runtime (containerd)
   │ CNI 플러그인 실행
   ▼
CNI Plugin
   ├── 네트워크 인터페이스 생성
   ├── Pod IP 할당
   ├── 라우팅 설정
   └── 실행 결과 반환
   ▼
Pod 네트워크 준비 완료
```

##### CNI 플러그인 종류

**1. Flannel — 단순한 구조, 파드 간 네트워크 연결에 집중**

| 구분 | 설명 |
|---|---|
| 주요 목적 | 서로 다른 노드에 있는 Pod 간 연결 |
| 대표 방식 | VXLAN 등 |
| 장점 | 설치와 구성이 비교적 단순함 |
| 주의점 | Flannel 단독 구성은 일반적으로 NetworkPolicy 집행 기능을 제공하지 않음 |

**2. Calico — 파드 네트워크 구성 + NetworkPolicy 지원**

| 구분 | 설명 |
|---|---|
| 주요 목적 | Pod 연결, 라우팅, 네트워크 정책 |
| 지원 방식 | BGP, IP-in-IP, VXLAN 등 구성에 따라 선택 |
| 장점 | 세밀한 네트워크 정책과 다양한 라우팅 구성 |
| 주의점 | 환경에 따라 라우팅과 네트워크 설정을 추가로 이해해야 함 |

- 기본적으로는 **네이티브 라우팅(BGP)**을, 필요에 따라 **오버레이 네트워크(IP in IP, VXLAN)**를 사용한다.
- 환경에 따라: **실습 → VXLAN**, **실무 → BGP**를 주로 사용한다.
- 실무적으로는 **칼리코(Calico)** 적용이 일반적이다.

**3. Cilium — eBPF 기반**

##### 현재 네트워크에서 통신하는 과정 (Flannel 예시)

```
Pod A 10.244.1.10
   ↓
worker-a의 Pod 네트워크
   ↓
Flannel VXLAN 캡슐화
   ↓
worker-a의 실제 네트워크 인터페이스  — 노드 IP 기반 전송
   ↓
worker-b의 실제 네트워크 인터페이스
   ↓
Flannel VXLAN 디캡슐화
   ↓
worker-b의 Pod 네트워크
   ↓
Pod B 10.244.2.20
```

- Pod A → Pod B로 가는 **실제 데이터 패킷은 노드끼리만 통신**한다.
- Control Plane을 거치지 않고 `worker-a → worker-b`로 바로 간다.

##### CNI와 kube-proxy는 뭐가 다를까?

| 구분 | CNI 플러그인 | kube-proxy |
|---|---|---|
| 주요 대상 | Pod 네트워크 | Service 네트워크 |
| 주요 역할 | Pod에 IP를 부여하고 통신 경로 구성 | Service IP로 온 요청을 실제 Pod Endpoint로 전달 |
| 동작 시점 | 주로 Pod 생성·삭제 시 네트워크 설정 | Service와 EndpointSlice 변경을 감지해 전달 규칙 구성 |
| 예시 | Flannel, Calico, Cilium | iptables, IPVS, nftables 등의 구현 모드 |

```
Service 요청
   ↓
kube-proxy가 만든 전달 규칙
   ↓
대상 Pod IP 선택
   ↓
CNI가 구성한 Pod 네트워크 경로로 전달
```

#### | VXLAN과 IP in IP

동작 자체는 거의 유사하다 — **어디에서 패킷을 감싸느냐(캡슐화 위치)가 다르다.**

##### IP in IP

```
┌────────────────────────────┐
│ Outer IP Header             │
│ Src: 192.168.10.11          │
│ Dst: 192.168.10.12          │
├────────────────────────────┤
│ Inner IP Header             │
│ Src: 10.244.1.10            │
│ Dst: 10.244.2.20             │
├────────────────────────────┤
│ TCP/UDP + Application Data  │
└────────────────────────────┘
```

##### VXLAN

```
┌────────────────────────────┐
│ Outer Ethernet Header       │
├────────────────────────────┤
│ Outer IP Header              │
│ Node A IP → Node B IP        │
├────────────────────────────┤
│ Outer UDP Header             │
├────────────────────────────┤
│ VXLAN Header                 │
├────────────────────────────┤
│ Inner Ethernet Header        │
├────────────────────────────┤
│ Inner IP Header              │
│ Pod A IP → Pod B IP          │
├────────────────────────────┤
│ TCP/UDP + Data                │
└────────────────────────────┘
```

→ VXLAN은 **UDP로 쏘기 때문에** 패킷 크기(오버헤드)는 더 크지만, 네트워크 장비 통과 측면에서는 더 유리할 수 있다.

##### 둘의 동작 방식

```
원래 Pod 패킷
   ↓
노드 간 전달용 외부 헤더 추가
   ↓
실제 노드 네트워크로 전송
   ↓
목적지 노드에서 외부 헤더 제거
```

##### 둘의 차이점

| 구분 | VXLAN | IP-in-IP |
|---|---|---|
| 캡슐화 대상 | 이더넷 프레임 | IP 패킷 |
| 계층 관점 | Layer 2 over Layer 3 | Layer 3 over Layer 3 |
| 외부 프로토콜 | UDP | IP Protocol 4 |
| 주요 헤더 | Outer IP + UDP + VXLAN | Outer IP |
| 터널 끝점 | VTEP | IPIP 터널 인터페이스 |
| 논리 네트워크 식별 | VNI 제공 | 별도 VNI 없음 |
| 오버헤드 | 더 큼 | 상대적으로 작음 |
| 네트워크 호환성 | UDP이므로 비교적 통과가 쉬운 편 | IP Protocol 4 허용 여부 확인 필요 |

#### | BGP (Border Gateway Protocol)

**어떤 IP 대역에 도달하려면 어느 이웃으로 보내야 하는지를 네트워크 장비 사이에서 교환하는 라우팅 프로토콜**

- 주변 노드들에게 광고할 때는 **prefix만 광고**한다. ("10.244.1.0/32로 오려면 나한테 와. 내 IP는 이거야.")

##### BGP 상세 동작

**1. BGP 이웃 관계 형성하기**
- Node A, Node B가 BGP 프로세스를 진행한다.
- **TCP 179번 포트**로 연결을 맺는다.

**2. 서로 보유한 경로를 광고**

```
# Node A → B
NLRI: 10.244.1.0/24
# "10.244.1.0 대역도 가려면 나를 거쳐야 해"
Next Hop: 192.168.10.11
```

**3. 수신한 경로를 라우팅 테이블에 반영**
- Node A가 Node B의 광고를 수신 → `10.244.2.0/24 via 192.168.10.12`
- Node B의 라우팅 테이블 → `10.244.1.0/24 via 192.168.10.11`

**4. 실제 파드 패킷 전달**

```
# Pod A → Pod B
10.244.1.10 → 10.244.2.20

# Node A 라우팅 테이블 확인
10.244.2.0/24 via 192.168.10.12

# Native Routing 사용
출발지: 10.244.1.10
목적지: 10.244.2.20
경유지: 192.168.10.12
```

##### BGP가 실제로 패킷 전달까지 할까?

→ **아니다.** BGP는 패킷을 직접 운반하지 않는다. **BGP는 라우팅 테이블을 만드는 경로 정보만 교환**한다.

```
BGP → 라우팅 정보 교환 → Linux Routing Table 갱신 → Linux Kernel이 실제 패킷 전달
```

> 💡 **BGP는 컨트롤 플레인, 실제 패킷 전달은 데이터 플레인**의 역할이다. BGP는 길을 알려주고, 리눅스 커널이 실제로 운반한다.

##### BGP 연결 방식

**1. Full Mesh**
- 모든 노드가 서로 BGP를 맺는다. 각 노드가 자신의 파드 대역대를 직접 광고한다.
- 노드 수가 많아지면 연결 수가 급격히 늘어나서 곤란해진다 → 노드가 많지 않을 때 사용.
- `BGP Peer 수 = N × (N-1) / 2` (N = 노드 수)

**2. BGP Reflector**
- 각 노드끼리 서로 Peer를 만들지 않고, 그 대신 **Route Reflector**를 만들어 거기에 경로를 저장하는 방식이다.

```
        Route Reflector
        /      |       \
   Node A   Node B    Node C
```

##### BGP 패킷 전달 2가지 방식

- **Native Routing**: `10.244.1.10 → 10.244.2.20`처럼 캡슐화 없이 바로 라우팅. 물리 하드웨어를 직접 핸들링할 수 있을 때 사용하기 좋다.
- **IP in IP와 함께 사용**: (VXLAN은 헤더 추가가 많아서 상대적으로 무겁다)

```
BGP: 10.244.2.0/24가 Node B 뒤에 있다는 경로를 학습
IP-in-IP: 실제 Pod 패킷을 Node B까지 캡슐화해서 운반

1. BGP가 경로를 알려줌      10.244.2.0/24 → Node B
2. Node A가 그 경로를 선택함
3. IP-in-IP로 패킷을 캡슐화함
     Outer: Node A → Node B
     Inner: Pod A → Pod B
4. Node B에서 디캡슐화함
```

##### 언제 어떤 걸 선택하는 게 좋을까?

**1. VXLAN**
- 언더레이 네트워크(물리 네트워크)를 수정하기 어려울 때
- 클라우드 환경에 적합함
- UDP 트래픽을 열어서 사용해도 괜찮을 때, 구현이 단순해도 괜찮을 때, 호환성이 좋아야 할 때
- 단점: 캡슐화 헤더가 크고, MTU를 조정해야 함

**2. IP in IP**
- Layer 3에서 단순한 캡슐화가 필요할 때
- 언더레이 네트워크가 IP 프로토콜 L4를 허용할 때

**3. BGP**
- 언더레이 네트워크 장비들에 Pod CIDR 경로를 설치할 수 있을 때 (안 될 경우 BGP 네이티브 대신 IP in IP 방식을 함께 사용할 수 있는데, 장비에 따라 허용하지 않을 수도 있음)
- 캡슐화 오버헤드를 줄이고 속도를 가져가야 하는 경우
- 대규모 환경에서 사용
- 네트워크 팀이 직접 BGP를 운영할 수 있을 때
- 단점: 네트워크 구성과 운영 난이도가 상당히 높아진다

##### 네트워크 구성 정리

| 방식 | 특징 |
|---|---|
| **VXLAN** | Ethernet Frame을 UDP 안에 캡슐화. Layer 2 over Layer 3. VTEP와 VNI 사용. 호환성이 좋지만 헤더 오버헤드가 큼 |
| **IP-in-IP** | IP Packet을 새로운 IP Packet 안에 캡슐화. Layer 3 over Layer 3. 구조가 단순하고 오버헤드가 상대적으로 작음 |
| **BGP** | 캡슐화 기술이 아님. 도달 가능한 IP Prefix와 Next Hop을 교환. 실제 패킷 전달은 Linux 라우팅이 수행 |

### 🗨️ 오늘의 회고

- 어제는 목표했던 것들을 모두 수행했다. 쉬는 시간도 활용하고, 자습 시간에 좀 더 집중해서 빠르게 처리하니 가능했던 것 같다. 오프라인 이후에 교육장까지 거리도 멀고 체력적으로도 벅차서 집에 가서 뭔가를 더 하는 게 잘 안 되는 편이라, 앞으로는 교육장에 있는 시간을 최대한 활용하는 방향으로 가야겠다는 생각이 들었다.
- 오늘까지 과제 수정과 피드백 반영을 마치고, 목요일부터 4일간 11주차 위클리 챌린지를 진행하는 것이 이번 주 목표다. 과제를 수행하면서 교재와 판서를 다시 복습하고, 이번 주에 배운 내용을 깊게 이해하려고 한다.
- **Job**을 배우면서 "왜 그냥 Pod를 apply하면 안 되는가"라는 질문에 대한 답이 명확해졌다. 실패했을 때 재시도, 로그 모니터링, 병렬 처리 같은 걸 사람이 직접 챙겨야 한다는 게 얼마나 번거로운 일인지 체감했고, 특히 `try-catch`에서 에러 로그만 찍고 넘어가면 종료 코드가 0으로 처리돼서 Job이 "성공"으로 오판할 수 있다는 주의사항은 실제 코드 작성 시 꼭 신경 써야겠다고 느꼈다.
- **CronJob**은 Job을 정해진 주기로 생성해주는 컨트롤러라는 게 단순하면서도 명확했다. `concurrencyPolicy`(Allow/Forbid/Replace)로 이전 작업이 끝나지 않았을 때의 처리 방식을 정할 수 있다는 부분은, 실제 배치 작업을 설계할 때 반드시 고려해야 할 옵션이라는 생각이 들었다.
- 오늘 가장 많은 시간을 들인 부분은 **Networking**이었다. Pod-Pod, Pod-Service, 내부-외부 통신을 계층별로 나눠서 정리하다 보니, 지난주에 배운 Service·Probe 개념과 오늘 배운 kube-proxy·CNI가 어떻게 맞물리는지 훨씬 선명해졌다. 특히 "Service는 가상의 리소스이기 때문에 실제 Pod에게 전달해줄 대리자가 필요하고, 그게 kube-proxy다"라는 설명과 "CNI는 Pod 네트워크, kube-proxy는 Service 네트워크를 담당한다"는 역할 구분이 오늘 배운 내용 중 가장 크게 정리된 부분이었다.
- VXLAN, IP-in-IP, BGP까지 들어가면서는 살짝 어려웠지만, "BGP는 컨트롤 플레인(길만 알려줌), 실제 패킷 전달은 리눅스 커널이 하는 데이터 플레인"이라는 비유 덕분에 개념이 헷갈리지 않게 정리됐다. 실습 환경에서는 VXLAN, 실무 대규모 환경에서는 BGP를 주로 쓴다는 실무 기준도 함께 기억해둬야겠다.
- 내일부터는 8, 9주차 피드백 반영과 과제 수정을 마무리하고, 목요일부터 시작하는 11주차 위클리 챌린지에서 오늘 배운 Job/CronJob/Networking 개념을 실습으로 직접 적용해봐야겠다.
