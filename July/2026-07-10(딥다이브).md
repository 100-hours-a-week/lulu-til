## 1. 무중단 배포 (Zero-Downtime Deployment)

### 무중단 배포란?

서비스 장애와 배포의 부담을 최소화하기 위해 **운영 중인 서비스를 중단하지 않고 신규 소프트웨어를 배포하는 기술**이다. 핵심은 로드밸런서(Load Balancer)를 통해 연결된 두 개 이상의(서로 다른 IP, 포트를 가진) 인스턴스에 트래픽을 제어하며 배포하는 것이다.

배포 작업이 서비스에 영향을 주지 않도록 하기 위해 고객의 이용량에 따라 인스턴스는 물론 로드밸런서도 다중화를 고려해야 한다. 즉, **무중단 배포를 위해서는 고가용성(High Availability)의 시스템 인프라가 구성되어 있어야 한다.**

### 무중단 배포가 특히 중요한 경우

- 트래픽이 많은 플랫폼 (전자상거래, SaaS, 금융)
- 24시간 내내 사용자가 있는 글로벌 애플리케이션
- 최대 가동 시간이 요구되는 규제 시스템

### 무중단 배포를 사용하는 이유

**1. 서비스 중단으로 인한 수익 손실 제거**

배포 중 다운타임은 수익 손실, 사용자 불편, 기업 이미지 손상 등을 유발할 수 있다. 무중단 배포는 업데이트 중에도 서비스 가용성을 유지한다.

**2. 배포 위험 감소 및 릴리스 주기 단축**

기존의 일괄 배포 방식은 문제 발생 시 모든 사용자에게 동시에 영향을 미친다. 무중단 전략을 통해 단계적 배포가 가능해지면서 영향 범위를 최소화하고, 문제를 조기에 발견하며, 안전하게 롤백할 수 있다.

**3. 사용자 경험 향상**

사용자들은 높은 가용성을 가진 서비스를 기대한다. 서비스 중단은 사용자들을 경쟁사로 이탈하게 만들고 신뢰를 손상시킬 수 있다.

### 무중단 배포의 종류 비교

| 구분 | 블루/그린 | 카나리아 | 롤링 |
|---|---|---|---|
| **환경 설정** | 거의 동일한 두 환경이 필요 | 거의 동일한 두 환경이 필요. 소수의 사용자 또는 서버로 시작 | 서버 하위 집합을 일괄적으로 업데이트 |
| **트래픽 전환** | 모든 트래픽을 한 번에 전환 | 새 버전으로의 트래픽을 점진적으로 증가 | 트래픽을 순차적으로 업데이트하고 전환 |
| **롤백 메커니즘** | 블루 환경으로 되돌아감 | 카나리 배포를 줄이거나 중단 | 일괄 처리를 되돌림. 더 복잡한 작업이 필요할 수도 있음 |

## 2. 블루-그린 배포 (Blue-Green Deployment)

### 개념

블루-그린 배포는 **두 개의 동일한 프로덕션 환경을 동시에 운영**하여 다운타임 없는 배포와 빠른 롤백을 가능하게 하는 소프트웨어 릴리스 전략이다.

- **블루(Blue)**: 현재 사용자가 이용 중인 라이브 버전 (기존 운영 환경)
- **그린(Green)**: 배포될 새 버전이 올라가는 환경 (테스트 및 스테이징 환경)

새 버전이 승인되면 로드밸런서가 트래픽을 블루 환경에서 그린 환경으로 **즉시** 전환한다.

### 동작 방식 — 단계별 상세

**1단계 — 동일한 환경 구성 및 애플리케이션 배포**

블루-그린 배포를 위한 동일한 환경 구축은 수동 또는 자동화 도구로 수행한다.

- **IaC(Infrastructure as Code)**: 기계가 읽을 수 있는 구성 파일을 사용해 미리 정의된 사양에 따라 IT 인프라를 자동으로 프로비저닝하고 구성한다. 개발자는 이미 운영 중인 블루 환경을 자동으로 복제하여 그린 환경으로 활용할 수 있다.
- **컨테이너화**: 애플리케이션 코드, 종속성, API를 모두 하나의 컨테이너 이미지로 패키징하여 두 환경이 완전히 동일함을 보장한다. Kubernetes가 이 목적으로 자주 사용된다.

**2단계 — 블루 환경으로 트래픽 라우팅**

로드밸런서가 클라우드 또는 Kubernetes 환경에서 애플리케이션 앞에 배치되어 운영 환경인 **블루 환경으로 트래픽을 전달**한다.

**3단계 — 그린 환경에 신버전 배포**

로드밸런서가 트래픽을 블루 환경으로 라우팅하는 동안, **그린 환경은 실험, 테스트, 구성을 위한 안전한 공간**이 된다.

**4단계 — 유효성 검사**

그린 환경 배포가 완료되면 필요한 모든 검증 단계를 수행한다.
- 보안 검사, 성능 검사, 버그 검사 등

**5단계 — 그린 환경으로 트래픽 전환**

모든 이해관계자의 검증 및 승인이 완료되면 로드밸런서를 업데이트하여 트래픽을 블루에서 그린으로 **즉시** 전환한다.

> 💡 **DNS 업데이트 대신 로드밸런서를 사용하는 이유**: DNS 업데이트는 캐시가 IP 주소를 새로 고치는 데 상당한 지연이 발생할 수 있다. 로드밸런싱 방식은 업데이트가 즉시 적용되므로 선호된다.

**6단계 — 모니터링**

새롭게 운영 중인 그린 환경을 지속적으로 모니터링한다. 문제가 발생할 경우 트래픽을 다시 블루 환경으로 즉시 리디렉션할 수 있다.

**7단계 — 블루 환경 폐기 여부 결정**

사용하지 않는 블루 환경 폐기는 **선택 사항**이다.
- 롤백을 위해 무기한 유지하는 조직도 있다.
- 비용 절감을 위해 특정 시점에 폐기하기도 한다.
- 차기 실험 환경 또는 다음 배포의 그린 환경으로 재활용할 수도 있다.

### CI/CD와 블루-그린 배포

CI/CD 파이프라인은 블루-그린 배포와 밀접하게 연관된 방식이다.

- **CI 파이프라인**: 새로운 코드가 중앙 저장소에 통합되기 전에 자동으로 테스트 및 통합하여 결함 있는 코드가 배포될 위험을 줄인다.
- **CD 파이프라인**: 블루-그린 배포 전략을 활용하여 사용자에게 영향을 주지 않고 신속한 릴리스를 최적화한다.

### 장단점

| 구분 | 내용 |
|---|---|
| ✅ **장점** | 롤백이 간단하다 (로드밸런서 전환만으로 즉시 복구) |
| ✅ **장점** | 완전히 격리된 환경에서 충분한 테스트 후 전환 가능 |
| ✅ **장점** | 전환이 즉각적으로 이루어져 배포 시간이 짧다 |
| ❌ **단점** | 동일한 환경을 두 배로 운영하므로 인프라 비용 증가 |
| ❌ **단점** | 데이터베이스 스키마 동기화가 복잡할 수 있다 |
| ❌ **단점** | 상태 저장 애플리케이션(쇼핑카트, 메시지 로그 등)에서 환경 전환 시 데이터 유실 위험 |
| ❌ **단점** | 두 환경을 동시에 운영하므로 보안 공격 표면이 증가 |

**사용 사례**: 대규모 모놀리식 아키텍처 또는 복잡한 상태 전환이 있는 서비스에 이상적

## 3. 롤링 배포 (Rolling Deployment)

### 개념

롤링 배포는 **여러 서버에 걸쳐 애플리케이션을 점진적으로 업데이트**하는 배포 전략이다. 소수의 서버를 오프라인으로 전환하여 업데이트한 뒤 다시 네트워크에 연결하는 과정을 모든 인스턴스가 업데이트될 때까지 반복한다. 인프라의 일부만 한 번에 변경되므로 광범위한 문제 발생 위험을 줄일 수 있다.

### 동작 방식 — 단계별 상세

**1단계 — 준비**

새 버전의 애플리케이션이 스테이징 환경에서 검증되고 프로덕션 배포 준비를 마친다.

**2단계 — 점진적 업데이트**

한 번에 하나 또는 소수의 인스턴스만 업데이트한다. 업데이트 중인 인스턴스는 로드밸런싱 풀에서 제외되어 트래픽을 받지 않는다.

**3단계 — 상태 점검 (Health Check)**

하위 집합 업데이트 후 해당 인스턴스가 올바르게 작동하는지 상태 점검을 실행한다. 특정 지표 모니터링 또는 자동화된 테스트가 포함될 수 있다.

**4단계 — 로드밸런서 재등록**

상태 점검을 통과한 인스턴스를 로드밸런서 풀에 다시 추가한다.

**5단계 — 반복**

모든 인스턴스에 신버전이 배포될 때까지 2~4단계를 반복한다.

### 롤링 배포 시작하기 — 체크리스트

1. **애플리케이션 아키텍처 점검**: 이전 버전과 새 버전이 동시에 공존할 수 있는지 확인한다. 이는 롤링 배포의 필수 조건이다.
2. **배포 파이프라인 설정**: 증분 업데이트(Incremental Update)를 지원하도록 CI/CD 도구를 구성한다.
3. **상태 점검 및 모니터링 구현**: 배포 진행 상황을 추적하고 문제를 신속하게 파악할 수 있는 모니터링 시스템을 구성한다.
4. **롤백 전략 수립**: 전체 롤백 없이 버전 간 전환이나 기능 비활성화를 위해 **기능 플래그(Feature Flag)** 사용을 고려한다.
5. **작게 시작해서 점차 확장**: 초기에는 인프라의 일부만 사용하고, 점진적으로 배포 규모를 늘려 나간다.

### 장단점

| 구분 | 내용 |
|---|---|
| ✅ **장점** | 인프라를 두 배로 늘릴 필요가 없어 리소스 효율적 |
| ✅ **장점** | 점진적 배포로 문제를 조기에 발견하고 영향 범위를 최소화 |
| ✅ **장점** | 배포 중 빈번한 상태 점검이 가능하여 모니터링이 쉬움 |
| ✅ **장점** | 빈번하고 작은 업데이트를 허용하는 CI/CD, 애자일 방법론과 잘 맞음 |
| ✅ **장점** | 로드밸런싱 없이도 구현 가능 |
| ❌ **단점** | 롤백이 복잡하다 (개별 인스턴스마다 되돌려야 함) |
| ❌ **단점** | 블루-그린 대비 전체 배포 시간이 더 오래 걸린다 |
| ❌ **단점** | 배포 중 구버전과 신버전이 공존하여 사용자마다 다른 버전을 볼 수 있다 |
| ❌ **단점** | 로드밸런서 구성을 지속적으로 업데이트해야 하므로 운영 오버헤드 증가 |
| ❌ **단점** | 구버전과 신버전 간 하위 호환성(Backward Compatibility)이 필수 |

**사용 사례**: Kubernetes 또는 컨테이너 오케스트레이션 환경에 이상적

## 4. 카나리 배포 (Canary Deployment)

### 개념

일부 사용자에게만 새 버전을 점진적으로 도입하는 방식이다. 제한된 노출로 업그레이드를 테스트하면서 다른 배포 시스템과 함께 사용할 수 있다.

### 장단점

| 구분 | 내용 |
|---|---|
| ✅ **장점** | 정식 공개 전 실제 사용자 피드백을 받을 수 있다 |
| ✅ **장점** | 문제 발생 시 영향 범위가 일부 사용자에게만 국한된다 |
| ❌ **단점** | 기능 플래그 지정 및 관찰 도구(모니터링)가 반드시 필요하다 |

**사용 사례**: 배포 빈도가 높고 사용자 기반이 큰 SaaS 제품

## 5. 데이터베이스 마이그레이션 (Database Migration)

### 개념

데이터베이스를 한 환경에서 다른 환경으로 이동하는 프로세스다. 기업에서는 애플리케이션을 현대화할 때 관련 데이터베이스를 온프레미스 서버에서 클라우드로 마이그레이션하며, 이를 통해 서비스 안정성, 확장성, 성능 향상 및 인프라 비용 절감을 기대할 수 있다.

### 마이그레이션 유형

| 유형 | 설명 |
|---|---|
| **동종 전송** | 유사한 데이터베이스 엔진 간의 이동 (예: MySQL → MySQL). 추가 변환 없이 직접 이동 가능하여 비교적 단순하다. |
| **이기종 전송** | 서로 다른 플랫폼이나 엔진 간의 이동 (예: Oracle → PostgreSQL). 스키마와 데이터 타입을 새 시스템에 맞게 변환해야 한다. |
| **부분 마이그레이션** | 전체 데이터셋의 일부만 이동한다. |
| **전체 마이그레이션** | 모든 데이터를 소스에서 대상으로 전송하고, 전송 후 소스 데이터베이스와 애플리케이션 연결을 끊는다. |
| **지속적 마이그레이션(복제)** | 소스의 데이터를 대상 데이터베이스로 무기한 복제한다. |

### 마이그레이션 시 고려사항

- **예산**: 데이터 감사, 매핑, 테스트, 마이그레이션 후 모니터링, 애플리케이션 수정 등에 필요한 시간·비용·리소스를 사전에 산정해야 한다.
- **가동 중지 시간**: 마이그레이션 중 소스-대상 전환 시 다운타임을 최소화하기 위한 계획이 필요하다. 전환 전 클라우드 데이터베이스를 충분히 테스트하여 애플리케이션 안정성과 데이터 정확성을 사전에 보장한다.
- **소스 및 타겟 시스템**: 동종/이기종 여부에 따라 마이그레이션 복잡도가 달라진다.

### 마이그레이션 생명 주기

```
1단계 — 평가 및 설계
현재 데이터베이스의 구조와 요구사항 분석
        ↓
2단계 — 스키마 및 데이터 이동
스키마 재생성 및 소스 → 대상 데이터베이스로 데이터 복제
        ↓
3단계 — 전환 및 검증
소스 데이터베이스를 읽기 전용으로 변경
애플리케이션 연결을 대상 데이터베이스로 리디렉션
        ↓
4단계 — 완료
대상 시스템이 안정화되면 소스 데이터베이스 폐기
```

## 6. 세션 동기화 (Session Synchronization)

### 세션이란?

'세션'은 웹사이트와 사용자의 **연결 상태**다. 사용자가 웹사이트에 로그인하면 서버는 '인증된 사용자'라는 기록을 만든다. 이 기록을 세션이라고 부른다.

### 세션 동기화란?

여러 서버가 사용자의 로그인 상태(세션)를 공유하고 유지하는 기술이다. 어느 서버에 접속하든 마치 한 대의 서버에 접속한 것처럼 끊김 없는 서비스를 제공한다.

### 왜 필요한가?

큰 웹사이트는 접속자가 많아 여러 대의 서버를 운영한다. 사용자가 A 서버에서 로그인한 뒤 다음 요청이 B 서버로 라우팅되면, B 서버는 해당 사용자의 로그인 정보를 알지 못한다. 결국 사용자는 로그아웃 처리되어 다시 로그인해야 하는 불편이 생긴다.

### 세션 동기화의 3가지 방식

**1. 세션 클러스터링 (Session Clustering)**

모든 서버가 서로 세션 데이터를 실시간으로 복사하고 공유하는 방식이다. 한 서버에 장애가 발생해도 다른 서버가 즉시 대신할 수 있다.
- 장점: 고가용성 보장
- 단점: 서버 수가 늘어날수록 복제 비용이 증가

**2. 세션 스토어 (Session Store — 중앙 집중형)**

서버들이 세션 정보를 각자 보관하지 않고, 외부의 전용 저장소(Redis, Memcached 등)에만 저장하는 방식이다. 서버가 늘어나도 관리가 편리하고, 서버 재시작 시 세션이 유지된다.
- 장점: 확장성이 뛰어남, 관리가 단순
- 단점: 세션 스토어 자체가 단일 장애점(SPOF)이 될 수 있으므로 이중화 필요

**3. 스티키 세션 (Sticky Session)**

특정 사용자의 모든 요청을 동일한 서버로 고정하는 방식이다. 로드밸런서가 사용자를 처음 처리한 서버로 계속 라우팅한다.
- 장점: 구현이 간단하고 기존 코드 변경이 불필요
- 단점: 서버 장애 시 해당 서버에 고정된 사용자의 세션이 모두 소실됨. 서버 간 부하 불균형 가능성 있음

## 7. 결론 : 주제 정리

### 블루-그린 vs 롤링 배포 핵심 비교

| 구분 | 블루-그린 배포 | 롤링 배포 |
|---|---|---|
| **방식** | 두 환경을 동시에 운영 후 즉시 전환 | 서버를 순차적으로 점진적 교체 |
| **다운타임** | 없음 (즉시 전환) | 없음 (항상 일부 인스턴스 운영) |
| **롤백 속도** | 빠름 (로드밸런서 전환) | 느림 (인스턴스별 개별 롤백 필요) |
| **인프라 비용** | 높음 (두 배의 자원 필요) | 낮음 (기존 인프라 활용) |
| **배포 중 버전 공존** | 없음 (순간 전환) | 있음 (구버전·신버전 동시 존재) |
| **적합 환경** | 모놀리식, 즉각 전환이 필요한 서비스 | Kubernetes, 컨테이너 기반 환경 |

### 문제 1. 데이터베이스 마이그레이션 및 스키마 호환성 문제

**문제 원인**

배포 과정에서 구버전과 신버전 애플리케이션이 동시에 단일 데이터베이스를 참조할 때 발생한다. 신버전에서 요구하는 스키마 변경이 구버전의 로직과 충돌하거나, 롤백 시 데이터가 손상될 수 있다.

**해결 방안 — Expand-Contract(팽창-축소) 패턴 적용**

```
1단계 — Expand (팽창): 하위 호환 스키마 변경 먼저 반영
        기존 코드가 손상되지 않도록 새 컬럼 추가만 먼저 수행
        ↓
2단계 — 신버전 애플리케이션 배포
        새로운 스키마를 활용하는 신버전을 배포
        ↓
3단계 — Contract (축소): 구버전 로직이 더 이상 사용되지 않으면
        사용하지 않는 컬럼, 인덱스 등을 다음 배포 주기에 삭제
```

- 블루-그린 배포 시에는 **AWS DMS(Database Migration Service)** 등 양방향 동기화 도구를 활용할 수 있다.
- GitHub Actions 같은 CI/CD 도구에 포함된 데이터베이스 배포 프로세스로 스키마 업데이트를 자동화할 수 있다.

> ⚠️ 스키마 변경이 너무 복잡하여 하위 호환성을 유지하기 어렵다면, 블루-그린 배포의 이점이 상쇄될 수 있다. 이 경우 두 환경에 각각 별도의 데이터베이스를 구축하고 실시간으로 동기화해야 할 수 있으며, 이는 데이터 불일치 및 성능 저하 위험을 수반한다.

### 문제 2. 세션 동기화 및 데이터 유실 문제

**문제 원인**

사용자의 로그인 세션 상태가 개별 서버의 메모리(In-Memory)에만 저장되어 있을 때 발생한다. 롤링이나 블루-그린 전환 중 요청이 다른 서버로 라우팅되면, 세션이 만료되어 사용자가 로그아웃되거나 장바구니 데이터 등이 유실될 수 있다.

**해결 방안**

1. **외부 세션 스토어 적용 (권장)**: 서버의 세션 데이터를 애플리케이션 외부의 중앙 저장소(예: Redis)에 분리하여 저장한다. 어떤 서버가 요청을 처리하더라도 동일한 세션 정보에 접근할 수 있다.

2. **스티키 세션(Sticky Session) 적용**: 로드밸런서의 세션 고정 기능을 사용하여 특정 사용자의 요청을 기존에 처리하던 서버로 일정 시간 유지한다. 다만 해당 서버 장애 시 세션이 소실될 수 있어 임시방편에 가깝다.

3. **JWT(JSON Web Token) 방식 적용**: 클라이언트 브라우저의 쿠키 또는 로컬 스토리지에 인증 상태를 저장하는 방식이다. 서버가 세션을 유지할 필요가 없는 **Stateless 구조**로 개선되어 근본적으로 세션 동기화 문제를 해결한다.

### 문제 3. 롤링 배포 선택 시 — 처리 용량 부족 및 버전 공존 문제

**문제 원인**

롤링 배포 중에는 일부 서버가 빠지고 신버전으로 교체되는 과정에서 **전체 서비스 처리 용량이 일시적으로 감소**한다. 트래픽이 집중되는 시간대에 배포가 진행되면 과부하가 발생할 수 있다. 또한 구버전과 신버전이 공존하는 동안 API 요청·응답의 데이터 포맷이 맞지 않아 오류가 발생할 수 있다.

**해결 방안**

1. **오토스케일링(Auto Scaling) 활용**: 배포 전 여분의 인스턴스를 미리 확보한 후 배포를 진행하여 용량 부족을 방지한다.

2. **롤링 비율 세밀하게 설정**: 동시에 교체하는 서버의 비율(예: 전체의 10%)을 낮게 설정하여 처리 용량 감소를 최소화한다.

3. **API 버전 관리 정책 엄격 적용**: 구버전과 신버전이 공존하는 동안 발생하는 데이터 포맷 오류를 방지하기 위해 API 하위 호환성을 보장하는 버전 관리 정책을 적용한다. (예: `/api/v1/...`, `/api/v2/...` 분리)
