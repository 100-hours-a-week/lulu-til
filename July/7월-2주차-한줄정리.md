# 한 줄 정리

---

## 2026-07-06

| 개념 | 정리 |
|------|------|
| **CI** | 개발자가 작성한 코드를 자동으로 통합 및 테스트하여 코드의 품질을 지속적으로 유지하는 개발 프로세스로, 코드의 통합을 자동화시키고 더 작은 단위로 문제를 해결하기 위해 사용한다. |
| **Test Code** | 기대하는 동작이 실제로 실행되는가를 검증하는 코드이다. |
| **Test Coverage** | 작성한 소프트웨어 테스트가 프로덕션 코드의 어느 정도 범위를 실행하고 검증하는지 나타내는 지표로, 테스트가 시스템의 요구사항과 코드를 얼마나 충분히 다루고 있는지 평가하는 척도로 사용한다. |
| **TDD** | 구현 코드를 작성하기 전에 테스트 코드를 먼저 작성하고, 테스트 코드를 통과하는 구현 코드를 작성하는 개발 방법론으로, 요구 사항을 먼저 코드로 작성하고 작은 단위의 개발을 위해 사용한다. |
| **Given-When-Then** | BDD(Behavior-Driven Development, 행위 주도 개발)에서 사용하는 테스트 시나리오 작성 패턴으로, 테스트 코드를 준비, 실행, 검증의 세 단계로 나누어 가독성과 유지보수성을 높이기 위해 사용한다. |
| **CD** | 코드가 자동화된 테스트를 통과하면 운영 환경까지 자동으로 배포되는 소프트웨어 배포 프로세스로, 예측 가능한 배포를 위해 사용한다. |
| **Github Actions** | GitHub 내에서 코드 빌드, 테스트, 배포 등의 워크플로우를 자동화하는 CI/CD 도구이다. |
| **Github Actions Workflows** | 자동화 전체 흐름을 정의한 YAML 파일이다. |
| **Github Actions Runner** | Job이 실제로 실행되는 컴퓨터 또는 가상 머신이다. |
| **Github Actions Jobs** | 워크플로우 안의 독립 실행 작업 단위이다. |
| **Github Actions Artifact** | CI를 통과하고 나서 실제로 배포할 수 있는 결과물이다. |

## 2026-07-07

| 개념 | 정리 |
|------|------|
| **Zero-downtime Deployment** | 사용자 입장에서 서비스 중단 없이 애플리케이션을 업데이트하거나 배포하는 방식으로, 안정적인 전환, 빠른 복구, 지속적인 관측 등을 위해 사용한다. |
| **Blue/Green Deployment** | 두 개의 독립된 환경을 동시에 운영하며, 트래픽을 새 버전으로 한 번에 전환해 서비스 중단 없이 배포와 롤백이 가능한 방식으로, 빠르고 쉬운 롤백, 장애 영향 최소화 등을 위해 사용한다. |
| **Rolling Deployment** | 기존 인스턴스를 하나씩 새로운 버전으로 순차 교체하며 전체 서비스를 점진적으로 배포하는 방식으로, 인프라 비용 절감, 점진적 배포 등을 위해 사용한다. |
| **Canary Deployment** | 새 버전을 전체에 적용하기 전에, 일부 사용자나 트래픽에만 점진적으로 배포하여 안정성을 검증한 후 전면 배포하는 방식이다. |

## 2026-07-08

| 개념 | 정리 |
|------|------|
| **AWS ECR** | AWS에서 제공하는 완전 관리형 컨테이너 이미지 저장소 서비스로, 이미지 저장, 버전 관리, 접근 제어, 보안 스캔까지 자동으로 처리되며, 컨테이너 이미지를 AWS 환경 안에서 안전하게 저장하고 AWS 서비스가 빠르게 가져와 실행할 수 있도록 하기 위해 사용한다. |
| **Registry** | 이미지를 저장하는 저장소이다. |
| **Repository** | 특정 이미지를 이름을 기준으로 버전을 모아두는 공간이다. |
| **AWS CodeDeploy** | AWS에서 제공하는 자동화된 애플리케이션 배포 서비스로, 배포를 표준화하고 AWS 리소스 대상 안정적인 배포를 위해 사용한다. |
