#### | CI (Continuous Integration)

**개발자가 작성한 코드를 자동으로 통합 및 테스트하여 코드의 품질을 지속적으로 유지하는 개발 프로세스**

여러 개발자의 코드 변경을 자주, 하나의 기준 코드에 통합하는 것을 의미한다. 개발 방법론에 가까운 개념이다.

```
매 통합 시점마다:
자동 빌드 + 테스트
→ 변경 사항을 너무 오랫동안 로컬 브랜치에 쌓아두지 않고
→ 자주 공유하는 코드베이스(GitHub)에 통합
→ 통합 결과를 자동으로 검증
→ 실패했을 때 빠르게 수정
```

##### CI가 없을 때 발생하는 문제

각자 기능을 오랫동안 개발하다 마지막에 한 번에 합치면:

- 개발자 A: 결제 / 개발자 B: 주문 / 개발자 C: 회원 정책 / 개발자 D: 쿠폰 정책
- 각자 로컬에서는 잘 동작하는데, 며칠 뒤 한 번에 합치면 다음 문제가 발생함
  - 같은 파일의 같은 부분을 수정했거나
  - A가 바꾼 DTO가 다른 영역에 문제를 일으키거나
  - 의존성 문제가 생기거나
  - C가 DB 스키마를 변경했거나
- 문제를 해결하는 것도 어렵고, **언제 누구의 커밋 때문에 어떻게 문제가 발생했는지 추적하기도 어렵다.**

| 문제 | CI가 해결하는 방식 |
|---|---|
| 개발자가 로컬에서만 테스트함 | 공통된 Runner 환경에서 자동 검증 |
| 병합 후 빌드 실패 발견 | PR 또는 main 병합 직후 자동 빌드 |
| 테스트를 빼먹음 | 테스트 실행을 병합 조건으로 설정 |
| 코드 리뷰만으로 오류를 찾으려 함 | 자동 테스트와 정적 분석을 함께 실행 |
| 누가 언제 깨뜨렸는지 모름 | 커밋·PR 단위의 실행 로그와 결과 제공 |
| 배포 직전에 여러 문제를 한꺼번에 발견 | 작은 변경 단위로 빠르게 실패 발견 |

> 💡 CI가 있다면, 개발자가 PR을 올리는 순간 자동으로 검증이 시작된다. **운영 서버에 문제가 반영되기 전에 GitHub 같은 저장소 안에서 먼저 실패하게 만드는 것**이 CI의 핵심 역할이다.

##### CI 동작 흐름

```
개발자 A, B, C → 각자 feature 브랜치에서 작업
→ Pull Request 생성
→ CI 자동 검증 (컴파일, 테스트, 코드 품질, 의존성, 빌드)
→ 성공: 코드 리뷰 → main 브랜치 병합 → main 브랜치 CI 재검증 → CD 단계로
→ 실패: 문제 확인 및 수정
```

##### CI를 구성하는 원칙

**① 하나의 코드 기준 베이스가 필요하다**

브랜치 전략으로 나누는데, 보통 아래처럼 운영된다.
- `main` → 항상 배포가 가능한 상태를 목표로 하는 기준 브랜치
- `dev` → 개발자들이 작업을 수행하는 브랜치
- `feature` → 특정 기능을 개발하는 브랜치

```
main → dev 로 브랜치를 하나 만듭니다.

dev 에서
  a → feature/login
  b → feature/payment
  c → feature/coupon

dev 에 합치는 과정으로 수행한다.
```

**② 변경 사항을 자주 통합해야 한다**

- 지속적으로 코드가 계속 통합되어야 한다.
- 개발자가 너무 오랫동안 다른 사람의 코드와 분리된 상태로 작업하지 말아야 한다.
- **큰 기능을 만들고 한 번에 합치려면 CI 과정이 복잡해진다.**

변경 단위 기준 예시:
- 로그인 API 형식이 수정됐다 → 통합
- 특정 예외 처리 추가 → 통합
- Dockerfile 개선 → 통합

> ⚠️ 인증, 결제, 주문, 쿠폰 기능을 하나의 PR에 다 때려놓거나, 리팩토링 + 기능 추가를 같이 PR에 넣는 것은 좋지 않다. 리팩토링은 전역에 영향을 미치는데, 기능 추가는 특정 도메인에만 영역을 가지기 때문에 많은 파일을 수정하고 한꺼번에 변경을 통합하는 건 좋지 않다.

**③ 재현 가능한 빌드**

개발자 노트북에서는 빌드가 되는데 CI 서버에서는 안 되는 경우가 상당히 흔하다. 코드에 다음이 포함되어야 한다.

| 항목 | 역할 |
|---|---|
| `pom.xml`, `build.gradle` | Java 의존성과 빌드 설정 |
| `package.json`, `package-lock.json` | Node.js 의존성과 버전 고정 |
| `Dockerfile` | 컨테이너 실행 환경 정의 |
| `docker-compose.yml` | 여러 컨테이너 구성 정의 |
| `.github/workflows/*.yml` | GitHub Actions 자동화 정의 |
| 테스트 코드 | 기대 동작 정의 |
| 환경 예시 파일 | `.env.example` 등 설정 구조 공유 |

**④ 자동으로 테스트를 돌려야 한다**

CI는 빌드만 된다고 끝이 아니다. CI에서 기대하는 바는:
- 빌드가 당연히 돌아야 하고
- 테스트 코드도 동작해야 하고
- 의존성 테스트도 동작해야 한다
- 이것들이 **자동으로** 이뤄져야 한다.

#### | 빌드와 테스트

##### 빌드 (Build)

소스코드를 실행 가능한 결과물로 만드는 과정. 보통 다음 작업이 포함된다.
- 자바 코드 컴파일
- 의존성 다운로드
- JAR 파일 생성

##### 테스트 (Test)

**기대하는 동작이 실제로 실행되는가?** 를 코드로 검증하는 과정.

```java
@Test
void 중복된_이메일로_회원가입하면_예외가_발생한다() {
    // given
    String email = "test@example.com";
    userRepository.save(User.create(email, "encoded-password"));

    // when & then
    assertThatThrownBy(() -> userService.signUp(email, "password"))
        .isInstanceOf(DuplicateEmailException.class);
}
```

> 💡 테스트 코드는 영어보다 **한글로 명시하는 경우**가 읽기 좋다.

테스트의 종류와 CI 적용 우선순위:

| 종류 | 확인 대상 | 특징 | CI 적용 우선순위 |
|---|---|---|---|
| **단위 테스트 (Unit Test)** | 함수, 클래스, 서비스 로직 | 빠르고 원인 추적이 쉬움 | 매우 높음 |
| **통합 테스트 (Integration Test)** | DB, Redis, API, 여러 모듈의 연결 | 실제 환경 문제를 더 잘 발견 | 높음 |
| **E2E 테스트 (End-to-End Test)** | 사용자의 전체 흐름 | 실제 사용자 경험과 가까움 | 선택적·핵심 시나리오 |
| **정적 분석 (Static Analysis)** | 코드 규칙, 잠재 오류 | 실행 전 코드 품질 검사 | 높음 |

##### 테스트 코드의 역할

| 역할 | 설명 |
|---|---|
| **기능 검증** | 현재 기능이 요구사항대로 동작하는지 확인 |
| **회귀 방지** | 새 기능을 추가하거나 리팩터링한 뒤, 기존 기능이 깨졌는지 확인 |
| **요구사항 문서** | "이 기능은 어떤 입력에서 어떤 결과를 내야 하는가"를 코드로 남김 |
| **변경 안전성 확보** | 코드를 수정할 때 어디까지 영향이 있는지 빠르게 확인 |
| **설계 피드백** | 테스트하기 너무 어려운 코드라면, 책임이 너무 많거나 의존성이 과도하게 얽혀 있을 가능성을 발견 |

##### 좋은 테스트 코드 작성 방법

**1. 동작을 검증한다**

```java
@Test
void 정상적인_이메일과_비밀번호로_회원가입하면_회원이_저장된다() {
    // given
    String email = "test@example.com";
    String password = "password123!";

    // when
    UserResponse response = userService.signUp(email, password);

    // then
    assertThat(response.email()).isEqualTo(email);
    assertThat(userRepository.existsByEmail(email)).isTrue();
}
```

**2. 하나의 중요한 동작만 검증한다**

테스트 하나에는 **하나의 기능**만 테스트하는 게 좋다.

**3. 테스트는 서로 독립적이어야 한다**

좋은 테스트는 **실행 순서와 상관없이 항상 같은 결과**를 만들어야 한다.

##### 테스트 코드 작성 방법 — Given-When-Then

테스트 코드를 읽기 쉽게 하기 위해 구분해놓은 패턴.

| 단계 | 설명 | 질문 |
|---|---|---|
| **Given** | 테스트 준비 | 어떤 상황인가? |
| **When** | 실행 | 무엇을 실행했는가? |
| **Then** | 검증 | 무엇이 일어나야 하는가? |

```java
@Test
void 주문_금액이_0원이하면_예외가_발생한다() {
    // given: 사용자가 상품을 주문하려고 한다
    Long userId = 1L;
    Long productId = 10L;
    int quantity = 0;

    // when & then: 주문 수량을 0개로 입력하면, 주문 수량 예외가 발생해야 한다
    assertThatThrownBy(
            () -> orderService.createOrder(userId, productId, quantity)
    ).isInstanceOf(InvalidOrderQuantityException.class);
}
```

##### 테스트 대상 선정 우선순위

모든 코드를 다 테스트할 수는 없다. 중요한 코드 우선으로 작성하되, 우선순위가 매우 높은 것만 짜도 충분하기는 하다.

| 우선순위 | 대상 | 예시 |
|---|---|---|
| **매우 높음** | 핵심 비즈니스 규칙 | 주문 가능 여부, 결제 금액 계산, 권한 검증 |
| **매우 높음** | 장애 시 피해가 큰 로직 | 중복 결제 방지, 재고 차감, 포인트 차감 |
| **높음** | 입력 검증과 예외 처리 | 잘못된 요청, 권한 없음, 중복 데이터 |
| **높음** | 복잡한 조건문과 계산 로직 | 할인 정책, 등급 정책, 상태 전이 |
| **중간** | 외부 시스템 연동 | Redis, DB, 메시지 큐, 외부 API |
| **낮음** | 단순 DTO, Getter, Setter | 단순 데이터 전달 객체 |
| **낮음** | 프레임워크 자동 설정 | Spring Boot 기본 설정, 단순 Configuration |

##### TDD(Test Driven Development)

**구현 코드를 작성하기 전에, 테스트 코드를 먼저 작성**하고 테스트 코드를 통과하는 구현 코드를 작성하는 개발 방법론.

| 구분 | 내용 |
|---|---|
| **장점** | 요구사항을 먼저 코드로 작성, 작은 단위 개발 |
| **단점** | 내가 실패할 것 같은 사항을 미리 예측할 수 있냐는 문제, 코드가 안전하다고 착각하게 됨, 테스트를 통과만 하는 코드를 작성하게 될 수 있음 |

TDD 흐름: 실패를 먼저 한다 → 그 다음에 바로 고친다 → 해당 부분에 대한 테스트 코드를 상당히 자세하게 작성한다 → 반복하면 테스트 코드 커버리지가 자연스럽게 넓어진다.

##### 테스트 코드 정리

> 테스트 코드는 커버리지를 올리기 위한 숫자가 아니라, **앞으로 변경이 되어도 고장나지 않아야 하는 기능에 대한 검증을 하는 코드**다.

좋은 테스트 코드:
- 중요한 비즈니스에 대한 동작을 검증
- 테스트끼리 독립적
- 같은 코드면 항상 같은 결과가 나와야 한다
- Given-When-Then 패턴으로 작성

##### CI 품질 게이트

어디까지 통과해야 CI가 성공했냐는 기준이 필요하다. 초기 프로젝트 기준:

| 단계 | 필수 여부 | 이유 |
|---|---|---|
| **컴파일·빌드 성공** | 필수 | 실행 불가능한 코드는 병합하면 안 됨 |
| **단위 테스트 통과** | 필수 | 핵심 로직 회귀 방지 |
| **통합 테스트 통과** | 가능하면 필수 | DB·Redis·외부 연동 오류 확인 |
| **PR 리뷰 승인** | 팀 상황에 따라 | 설계·가독성·의도 검토 |
| **정적 분석** | 점진적 도입 | 코드 규칙과 잠재 문제 탐지 |
| **보안 검사** | 점진적 도입 | 알려진 취약점 탐지 |
| **E2E 테스트** | 핵심 기능만 | 느린 테스트의 비용 관리 |

##### CI를 왜 PR에서도 하고, Main에서도 하나?

| 시점 | 목적 |
|---|---|
| **Pull Request 생성·수정 시** | 병합 전에 위험한 변경을 발견 |
| **main 병합 후** | 실제로 통합된 최종 코드 상태를 다시 검증 |
| **배포 직전** | 배포할 결과물이 정상인지 최종 확인 |

##### 실무적으로 CI 잘하기

**① CI는 빠르고 신뢰할 수 있어야 한다**

CI 결과까지 30분이 걸리면, 실패를 늦게 발견하는 것뿐 아니라 다음 문제가 생긴다.
- 이미 실패한 CI 위에 다른 개발자가 커밋을 쌓는다
- PR 병합이 느려지고, 해당 CI를 무시하거나 우회하게 된다

CI는 검사 속도에 따라 이벤트별로 분리해서 설정하는 게 좋다.

| 구분 | 실행 시점 | 예시 |
|---|---|---|
| **빠른 검사** | 모든 PR Push | 컴파일, 단위 테스트, 포맷 검사 |
| **중간 검사** | PR 생성·병합 | 통합 테스트, Docker 이미지 빌드 |
| **느린 검사** | main 병합 후 또는 야간 | 전체 E2E, 부하 테스트, 전체 보안 스캔 |

**② 깨진 테스트를 방치하지 않는다**

코드는 그대로인데 외부 환경에 따라 테스트 결과가 달라지는 테스트들이 있다.

원인 예시:
- 테스트 실행 시 이미 어떤 데이터가 있다고 가정하는 경우
- 현재 시간에 의존하는 경우
- 외부 API에 의존하는 경우
- 테스트가 병렬적으로 실행될 때 데이터가 충돌되는 경우

**③ Secret + 권한을 최소화한다**

CI 러너는 CI를 실행하는 서버로 보통 외부 환경에 있다(GitHub Actions Runner → GitHub에 있다). 내 환경에 있는 비밀 정보를 저기에 넣어야 하는데, 권한 + Secret을 과하게 주면 위험하다.

##### CI에 대한 오해

**"테스트가 있으면 CI다"** → 아니다. CI는 코드베이스, 개발팀이 같이 공유하는 "자주 통합하는 개발 규칙", 자동 빌드, 자동 테스트가 필요하다.

**"CI 통과했는데, 왜 오류가 나죠?"** → CI를 통과해도 버그는 사라지지 않는다. 테스트 코드는 내가 예상한 실패를 막는 코드다. 내가 예상 못 한 것은 막을 수 없다.

> **CI 실패 연쇄**: CI 실패 → CD 실패 → 운영 불안정 → 옵저버빌리티 저하 → 서비스 품질 저하

---

#### | CD (Continuous Deployment)

**코드가 자동화된 테스트를 통과하면 운영 환경까지 자동으로 배포되는 소프트웨어 배포 프로세스**

- CI ← 이 코드가 정상적으로 작동하는가?
- CD ← 확인한 코드를 실제 서버나 사용자에게 **어떻게 전달할거냐**

##### 배포 과정 (수동일 때)

```
운영 서버 접속 → 새 버전 준비 → 기존 서비스 교체 → 환경 변수 설정
→ 새 버전이 동작하는지 확인 → 문제가 있으면 이전 버전 복구
```

원래는 매번 사람이 직접 수행했다. 문제는 **느리고, 실수하기가 상당히 좋은 구조**라는 것이다.

##### CD가 없으면 이런 문제가 생긴다

| 문제 | 실제 상황 |
|---|---|
| 배포 절차가 사람마다 다름 | 어떤 개발자는 `git pull` 후 실행, 다른 개발자는 이미지만 교체 |
| 운영 서버를 잘못 조작 | 개발 서버가 아닌 운영 서버에서 테스트 명령 실행 |
| 테스트한 코드와 배포한 코드가 다름 | 테스트 이후 다른 커밋을 받아서 배포 |
| 환경 설정 누락 | `.env`, 포트, DB 주소, API Key 등을 빠뜨림 |
| 배포 이력 불명확 | 현재 운영 서버에 어떤 버전이 배포되었는지 모름 |
| 장애 발생 시 복구가 느림 | 이전 버전 이미지나 배포 방법을 다시 찾아야 함 |
| 배포 중 서비스가 중단될 수 있음 | 기존 컨테이너를 종료한 뒤 새 컨테이너를 실행 |

> 수동 배포의 핵심 문제점: **명령어를 치는 시간이 걸린다가 아니라, 진행하는 사람마다 배포 결과가 달라질 가능성이 존재한다.** CD를 통해서 배포하는 과정을 코드로 만들어서 자동화하면 **누가 배포하더라도 같은 결과**가 나온다.

##### CD를 통해 해결하는 것들

> CD는 단순히 배포를 빠르게 하는 게 아니라, **배포를 예측 가능하게 하는 것**이 목적이다.

| 해결하려는 문제 | CD가 제공하는 방식 |
|---|---|
| 반복적인 배포 작업 | 정해진 파이프라인으로 자동 실행 |
| 사람이 하는 실수 | 테스트 성공한 경우에만 다음 단계 진행 |
| 배포 버전 추적 어려움 | 이미지 태그, Commit SHA, 배포 이력 관리 |
| 환경별 설정 혼동 | 개발·스테이징·운영 환경 분리 |
| 배포 후 장애 발견 | Health Check, 로그, 모니터링 |
| 장애 시 복구 지연 | 이전 안정 버전으로 롤백 |
| 동시 배포로 인한 충돌 | 배포 순서와 실행 권한 제어 |

##### CD 파이프라인 단계

| 단계 | 하는 일 | 예시 |
|---|---|---|
| **CI** | 코드 품질 검증 | 빌드, 테스트, 정적 분석 |
| **Artifact 생성** | 배포 가능한 결과물 생성 | `.jar`, Docker 이미지, `dist/` |
| **Registry 저장** | 결과물을 저장하고 배포 대상에 전달 | Docker Hub, Amazon ECR |
| **Deploy** | 서버 또는 플랫폼에 새 버전 반영 | EC2, ECS, Kubernetes |
| **Health Check** | 새 버전이 실제로 정상 동작하는지 확인 | `/health` API 호출 |
| **Monitoring** | 배포 이후 장애 여부 확인 | 로그, 에러율, CPU, 응답 시간 |
| **Rollback** | 문제가 발생하면 이전 안정 버전으로 복구 | 이전 이미지 재배포, 트래픽 전환 |

##### Artifact

CI를 통과하고 나서 실제로 배포할 수 있는 결과물.
- 스프링 → `.jar`
- 도커 → `image`
- React, Vue → `dist/`

```
소스 코드 → CI Build → Docker Image → Docker Hub / Amazon ECR → EC2 / ECS / Kubernetes
```

> 왜 굳이 Artifact를 별도로 보관하냐? **운영 서버에서 빌드하지 않기 위해서다.** 운영 서버는 환경이 다를 수 있으므로, CI Runner에서 동일하게 빌드하고 결과물을 배포하는 방식을 사용한다. **build once → deploy many times.**

##### Health Check와 Rollback이 필요한 이유

"그냥 도커 컨테이너만 실행되면 된 거 아님?"

- 컨테이너가 실행되어도 **내부 프로세스(Spring Boot)가 아직 준비 중일 수 있다.**
- Health Check는 "컨테이너 실행 여부"가 아니라 **"서비스가 실제로 정상 동작하는지"** 를 확인한다.
- Rollback은 자동화되어야 한다. 새 버전 장애 시 사람이 수동으로 롤백하면 그 사이에도 서비스 중단이 계속된다.

##### CD의 두 가지 의미

| 구분 | Continuous Delivery (지속적 전달) | Continuous Deployment (지속적 배포) |
|---|---|---|
| 테스트·빌드 | 자동 | 자동 |
| 이미지 생성·저장 | 자동 | 자동 |
| 스테이징 배포 | 자동 | 자동 |
| **운영 배포** | **승인 후 실행** | **자동 실행** |
| 핵심 가치 | 안정성, 통제 | 속도, 자동화 |
| 적합한 환경 | 금융, 공공, 대기업, 승인 절차가 필요한 조직 | 빠른 배포와 피드백이 중요한 서비스 |

**Continuous Delivery 흐름:**
```
테스트 성공 → 이미지 생성 → 스테이징 배포(운영 전, 마지막 테스트 환경) → 담당자 승인
```
(게임 서버에서 많이 사용)

**Continuous Deployment 흐름:**
```
테스트 성공 → 이미지 만들고 → 운영 서버 배포 → 헬스 체크 → 성공이면 고 → 실패하면 롤백
```

---

#### | GitHub Actions

**GitHub 내에서 코드 빌드, 테스트, 배포 등의 워크플로우를 자동화하는 CI/CD 도구**

EC2 또는 ECS 같은 VM, Container 기반의 배포에서는 일반적으로 GitHub Actions를 사용하는 경우가 많다.

##### 왜 사용하는가

배포 과정을 사람이 하면:
1. GitHub에 코드 Push
2. 로컬 또는 서버에서 최신 코드 Pull
3. 의존성 설치
4. 테스트 실행
5. 빌드
6. Docker 이미지 생성
7. Docker Hub 또는 ECR에 Push
8. EC2 또는 ECS에 배포
9. 배포 결과 확인

→ 작업 누락, 환경 차이, 속도 저하가 발생한다.

GitHub Actions는 저장소 안에서 특정 이벤트(코드 변경, PR, 이슈, 릴리즈)가 발생하면, 저장소의 이벤트를 트리거로 CI/CD 작업을 자동 실행할 수 있다.

##### CI/CD + GitHub Actions 비교

| 구분 | CI에서 하는 일 | CD에서 하는 일 | GitHub Actions에서의 구현 |
|---|---|---|---|
| 목적 | 코드가 정상인지 검증 | 검증된 코드를 배포 | YAML 워크플로우 작성 |
| 대표 작업 | 테스트, 린트, 빌드 | Docker 이미지 Push, EC2/ECS 배포 | Job과 Step으로 단계 구성 |
| 실행 시점 | PR 생성, Push | main 병합, 태그 Push | `on:` 이벤트 설정 |
| 실패 시 | PR 병합 차단 가능 | 배포 중단 또는 롤백 | Job 실패 처리 |

##### GitHub Actions 핵심 구성

| 구성 요소 | 한 줄 설명 | 예시 |
|---|---|---|
| **Event** | 워크플로우를 시작시키는 GitHub 이벤트 | `push`, `pull_request`, `workflow_dispatch` |
| **Workflow** | 자동화 전체 흐름을 정의한 YAML 파일 | `.github/workflows/ci.yml` |
| **Job** | 워크플로우 안의 독립 실행 작업 단위 | `test`, `build`, `deploy` |
| **Step** | Job 안에서 순서대로 실행되는 한 단계 | 코드 내려받기, 의존성 설치, 테스트 |
| **Action** | 특정 기능을 재사용하기 위해 만든 자동화 코드 조각 | `actions/checkout`, `actions/setup-node` |
| **Runner** | Job이 실제로 실행되는 컴퓨터 또는 가상 머신 | `ubuntu-latest`, Self-hosted Runner |

**1. Event — 어떤 일이 발생했을 때 자동화를 시작하나?**

```yaml
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - dev
```

| Event | 사용 사례 |
|---|---|
| `push` | main 브랜치 병합 후 배포 |
| `pull_request` | PR 생성 시 테스트, 린트, 빌드 검증 |
| `workflow_dispatch` | GitHub 화면에서 수동 실행 |
| `schedule` | 매일 새벽 보안 검사, 정기 배치 작업 |
| `release` | 릴리즈 생성 시 패키지 배포 |
| `issues` | 이슈 생성 시 Slack 또는 Discord 알림 |

보통 이벤트를 다 분리해서 사용한다.
- PR → 테스트, 린트, 빌드 검증
- `main` push → 이미지 생성, 운영 배포

**2. Workflow — 깃헙 액션즈가 수행할 전체 자동화 절차**

```
my-project/
├── src
├── package.json
└── .github/
    └── workflows/
        ├── ci.yml
        └── cd.yml
```

워크플로우는 코드로 작성한다.

**3. Runner — Job이 실제로 실행되는 환경**

```yaml
runs-on: ubuntu-latest
```

이 워크플로우는 GitHub에서 제공하는 우분투 가상머신에서 잡을 실행한다.

| Runner 유형 | 설명 | 적합한 상황 |
|---|---|---|
| **GitHub-hosted Runner** | GitHub가 관리하는 임시 실행 환경 | 일반적인 CI, 테스트, 빌드 |
| **Self-hosted Runner** | 회사 또는 개인이 직접 설치·운영하는 Runner | 내부망 접근, 사내 서버 배포, 특수 장비 사용 |

> ⚠️ GitHub-hosted Runner는 내 EC2에서 실행되는 게 아니다. 깃헙에서 관리하는 서버에서 가상화시켜서 실행한다. 깃헙 러너가 EC2에 접근하려면:
> 1. GitHub Actions Runner → SSH EC2에 접속할 수 있게 하기 (SSH 포트 오픈, IP 대역 제한 — 권장하지 않음)
> 2. EC2에 Self-hosted Runner 설치
> 3. CodeDeploy 같은 CD 도구 사용

**4. Job — 워크플로우 안에서 실행되는 독립 작업**

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: npm test

  deploy:
    needs: test       # test Job이 성공해야 deploy Job이 실행됨
    runs-on: ubuntu-latest
    steps:
      - run: ./deploy.sh
```

| 특징 | 설명 |
|---|---|
| 기본 실행 방식 | Job들은 기본적으로 병렬 실행 가능 |
| 실행 환경 | 각 Job은 별도 Runner에서 실행 |
| 파일 공유 | Job 간 파일은 자동 공유되지 않음 |
| 실행 순서 제어 | `needs:`를 사용 |
| 실패 처리 | 선행 Job이 실패하면 후속 Job은 기본적으로 실행되지 않음 |

> ⚠️ Job을 너무 잘게 쪼개지 말 것. Job은 각각 병렬로 다른 Runner에서 실행되기 때문에, 서로 결과 파일을 공유하려면 별도 설정이 필요하다.

**5. Step — Job 내부에서 순차적으로 실행되는 한 단계**

```yaml
steps:
  - name: Checkout repository
    uses: actions/checkout@v6

  - name: Install dependencies
    run: npm ci

  - name: Run tests
    run: npm test
```

**6. Action — 반복 기능을 재사용하는 자동화 코드**

GitHub Actions에서 자주 사용되는 것들을 미리 정의해두었다. 외부 집단들도 Actions를 만들어 공유한다.

| Action | 역할 |
|---|---|
| `actions/checkout` | GitHub 저장소 코드를 Runner로 내려받음 |
| `actions/setup-node` | Node.js 설치 및 설정 |
| `actions/setup-java` | Java 설치 및 설정 |
| `actions/cache` | 의존성 캐시 저장 및 복구 |
| `docker/login-action` | Docker Hub 또는 Container Registry 로그인 |
| `docker/build-push-action` | Docker 이미지 빌드 및 Push |
| `actions/upload-artifact` | 빌드 결과물 업로드 |
| `actions/download-artifact` | 다른 Job에서 빌드 결과물 다운로드 |

##### Job 간 데이터 공유

Job은 독립적인 실행 환경이다. Job 간 데이터를 공유하려면 **Cache** 또는 **Artifact**를 사용해야 한다.

| 구분 | Cache | Artifact |
|---|---|---|
| 목적 | 속도 향상 | 결과물 전달 및 보관 |
| 대표 대상 | npm, Gradle, Maven 의존성 | `.jar`, `dist/`, 테스트 리포트 |
| 다음 Job 전달 | 가능하지만 목적이 다름 | 가능 |
| 예시 | npm 캐시 | 빌드된 프론트엔드 결과물 |

```yaml
# Artifact 업로드
- name: Upload build result
  uses: actions/upload-artifact@v4
  with:
    name: frontend-build
    path: dist/

# Artifact 다운로드 (다른 Job에서)
- name: Download build result
  uses: actions/download-artifact@v4
  with:
    name: frontend-build

# 캐시
- uses: actions/setup-node@v4
  with:
    node-version: 22
    cache: npm
```

##### CI/CD를 분리하는 구조

```
Pull Request  → CI Workflow
                  ├─ Lint
                  ├─ Test
                  └─ Build

main Branch Merge → CD Workflow
                      ├─ Docker Image Build
                      ├─ Container Registry Push
                      ├─ Deploy
                      ├─ Health Check
                      └─ Monitoring / Alert
```

##### GitHub Actions 정리

| 개념 | 핵심 |
|---|---|
| **GitHub Actions** | GitHub 이벤트를 기준으로 개발·운영 작업을 자동화하는 CI/CD 플랫폼 |
| **Workflow** | `.github/workflows/*.yml`에 정의하는 자동화 전체 흐름 |
| **Event** | 워크플로우 실행 조건 |
| **Job** | 독립된 실행 작업 단위 |
| **Step** | Job 안에서 순서대로 실행되는 단계 |
| **Action** | 재사용 가능한 자동화 코드 |
| **Runner** | Job이 실제로 실행되는 환경 |
| `run` | 셸 명령어 직접 실행 |
| `uses` | Action 호출 |
| `needs` | Job 실행 순서 제어 |
| **Cache** | 의존성 다운로드 속도를 높이는 기능 |
| **Artifact** | 빌드 결과물 또는 리포트를 전달·보관하는 기능 |

### 🗨️ 오늘의 회고

- CI/CD는 이름은 익숙했지만, 오늘 CI와 CD를 각각 "왜 필요한가"부터 "어떻게 구성하는가"까지 논리적으로 풀어내 주니 이제야 전체 그림이 잡혔다. 특히 "CI는 코드가 정상인지 검증하는 것, CD는 검증된 코드를 어떻게 전달하는가"라는 구분이 명확해졌다.
- **테스트 코드** 파트가 생각보다 많은 분량을 차지했는데, "커버리지를 올리기 위한 숫자가 아니라, 앞으로 변경이 되어도 고장나지 않아야 하는 기능에 대한 검증을 하는 코드"라는 정의가 인상 깊었다. 테스트를 모든 코드에 다 짜는 게 아니라 핵심 비즈니스 규칙과 장애 시 피해가 큰 로직에 먼저 집중한다는 우선순위 관점이 실용적이었다.
- `depends_on`과 비슷하게, CI도 "PR에서 성공했다고 main 병합 후에도 문제없다고 볼 수 없다"는 점이 흥미로웠다. PR 시점과 main 병합 후 시점의 CI 역할이 다르다는 것을 이제 이해했다.
- **Continuous Delivery와 Continuous Deployment의 차이**는 "사람이 중간에 개입하냐 마냐"라는 한 줄로 정리되는데, 조직의 성격(금융/공공 vs 스타트업)에 따라 어느 쪽을 선택하느냐가 달라진다는 관점이 실무와 맞닿아 있어서 좋았다.
- GitHub Actions의 Event → Workflow → Job → Step → Action → Runner 구조가 처음에는 복잡해 보였는데, "어떤 사건이 → 어떤 절차를 → 어떤 환경에서 → 어떤 단계로 실행하는가"라는 흐름으로 이해하니 자연스럽게 연결됐다. Job이 각각 별도 Runner에서 실행되기 때문에 파일을 Artifact로 공유해야 한다는 부분은 실습할 때 꼭 챙겨봐야겠다.
