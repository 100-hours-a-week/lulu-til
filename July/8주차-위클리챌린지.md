### 1. 프로젝트를 도커 이미지로 빌드, 프로젝트 배포하기

#### 1-1. 컨테이너 배치에 대한 고민

**Docker Container는 몇 개가 필요할까?**

처음 이 과제를 시작할 때 가장 먼저 부딪힌 질문은 "컨테이너를 몇 개로 나눠야 하는가"였다. 

- **하나의 컨테이너에 여러 앱을 넣으면 안 되는 이유**
    - 도커의 핵심 철학은 "1 컨테이너 = 1 프로세스"다. 만약 프론트와 백엔드를 하나의 컨테이너에 우겨넣으면, 프론트만 수정했는데도 백엔드까지 통째로 재빌드/재배포해야 하는 비효율이 생긴다.
    - 하나가 죽으면 둘 다 죽는다. 백엔드 프로세스에서 메모리 누수가 나서 컨테이너가 재시작되면, 멀쩡한 프론트까지 같이 내려가 버린다.
    - 장애 지점을 특정하기가 어려워진다. 로그가 한 컨테이너에 뒤섞여서 어디서 문제가 생겼는지 구분하기 힘들다.
- **DB는 어떻게 할 것인가: RDS vs 도커 컨테이너 MySQL**
    - 처음에는 실습에서처럼 `docker run mysql:8.4`로 컨테이너 DB를 띄우는 것도 고려했다. 하지만 실제로 배포를 진행하면서 이미 `lulu-db.cfkso8qu2sal.ap-northeast-2.rds.amazonaws.com`라는 RDS 엔드포인트를 쓰고 있었다.
    - **도커 기반 MySQL 장점**: 개발 환경 재현이 쉽고, 로컬-서버 간 동일한 이미지를 쓸 수 있어 일관성이 좋다.
    - **도커 기반 MySQL 단점**: 컨테이너가 삭제되면 데이터가 날아갈 위험이 있고(볼륨 마운트를 놓치면 특히), 백업/복구/패치를 전부 내가 수동으로 챙겨야 한다.
    - **AWS RDS 장점**: 백업, 패치, 장애 복구가 자동화되어 있다. 실제로 과제를 진행하면서 EC2 리소스(디스크 용량, 메모리)가 이미 빠듯했기 때문에, DB까지 EC2 안에서 직접 운영했다면 `No space left on device` 에러가 더 자주 발생했을 것이다.
    - **결론**: RDS를 사용하기로 결정. 예산 안에서 감당 가능한 수준이었고, 무엇보다 인프라 관리 부담을 줄이고 애플리케이션 배포 자체에 집중하는 것이 이번 과제의 본질이라고 판단했다.
- **파일 저장은 어떻게 할 것인가: S3**
    - 현재 구조는 S3 버킷을 사용하는 구조이기 때문에 컨테이너에 **Volume을 별도로 마운트할 필요가 없어졌다.** 컨테이너가 삭제되거나 재배포되어도 사용자가 올린 사진, 데이터는 RDS와 S3에 안전하게 남아있기 때문이다. "컨테이너는 언제든 죽었다 살아나도 되는 휘발성 자원"이라는 도커의 기본 철학을 지킬 수 있었던 부분이다.
- **컨테이너를 EC2에 어떻게 분배했는가**
    - 최종적으로 아래와 같이 역할을 분리했다.
        1. 백엔드 서버 2개(프라이빗 서브넷) — `lulu-backend` 컨테이너 2개
        2. 프론트엔드 서버 2개(프라이빗 서브넷) — `lulu-frontend` 컨테이너 2개
        3. 배스천 서버(퍼블릭) — Portainer 컨테이너
        4. RDS/S3 — AWS 관리형 서비스로 대체하여 별도 컨테이너 불필요
    - 이렇게 나눈 이유는, 특정 백엔드 서버 한 대가 죽더라도 다른 서버가 트래픽을 받아줄 수 있게 하기 위함이었다.

#### 1-2. 도커 파일에 대한 고민

이 부분에서 정말 많은 시행착오를 겪었다. 아래는 실제로 겪은 문제와 그 과정에서 배운 것들이다.

**초기 Dockerfile 작성 및 자바 버전 이슈**

처음 작성한 Dockerfile은 아래와 같았다.

```docker
FROM gradle:8-jdk24 AS build
WORKDIR /app
COPY . .
RUN chmod +x gradlew && ./gradlew clean build -x test

FROM eclipse-temurin:21-jre-jammy
WORKDIR /app
COPY --from=build /app/build/libs/*.jar app.jar
CMD ["java", "-jar", "app.jar"]
```

여기서 첫 번째로 마주친 에러가 `UnsupportedClassVersionError`였다.

```
class file version 68.0, this version of the Java Runtime only recognizes class file versions up to 65.0
```

- **원인 분석**: 빌드 스테이지는 JDK 24(class version 68)로 컴파일했는데, 실행 스테이지는 JRE 21(class version 65)이었다. 빌드 환경과 실행 환경의 자바 버전이 어긋나면, 컴파일된 바이트코드를 더 낮은 버전의 JVM이 읽지 못한다는 걸 직접 겪으면서 배웠다.
- **왜 JDK 24를 써야 했나**: 로컬 개발 환경이 이미 Java 24 기준으로 세팅되어 있었기 때문에, 서버의 런타임도 JDK 24와 짝을 맞춰야 했다.
- **해결**: 실행 스테이지 이미지를 `eclipse-temurin:21-jre-jammy` → `eclipse-temurin:24-jre-alpine`으로 교체. 빌드 스테이지와 실행 스테이지의 자바 메이저 버전을 반드시 일치시켜야 한다는 교훈을 얻었다.

**이미지 경량화: Alpine 선택**

디스크 공간을 고려하여 이미지 경량화도 함께 고민했다.

| 옵션 | 장점 | 단점 |
| --- | --- | --- |
| `-jammy` (Ubuntu 기반) | 패키지가 풍부하고 안정적 | 이미지 용량이 큼(200MB↑) |
| `-alpine` | 매우 가볍다(50MB 미만) | musl libc 기반이라 일부 네이티브 라이브러리 호환성 문제 가능 |

Spring Boot는 순수 JVM 위에서 도는 애플리케이션이라 alpine과의 호환성 문제가 거의 없을 거라 판단하고 `eclipse-temurin:24-jre-alpine`을 선택했다. 실제로 이 선택 덕분에 이후 서버 디스크 공간 부족 문제(`No space left on device`)를 완화하는 데도 도움이 되었다.

**빌드 도중 디스크 공간 부족 문제**

서버(EC2)에서 직접 `docker build`를 수행하다가 다음 에러를 만났다.

```
Attempt 1/1 failed. Reason: No space left on device
```

- **원인**: `gradle:8-jdk24` 빌드 이미지 자체가 1GB가 넘는 무거운 이미지였고, EC2의 기본 EBS 볼륨 용량이 빠듯했다.
- **해결 과정**:
    1. 우선 `docker system prune -a -f`로 사용하지 않는 이미지/캐시를 정리해 임시 공간을 확보했다.
    2. 근본적으로는 멀티 스테이지 빌드 구조 자체가 "빌드 환경에서는 무겁게 쓰고, 실행 환경만 가볍게 남긴다"는 원칙을 따르고 있었으므로, 최종 이미지 자체는 가벼워질 것이라 판단하고 재시도했다.
    3. 이 경험을 통해 "서버에서 직접 빌드하는 방식"의 한계를 체감했고, 추후 개선 방향(CI에서 빌드 후 이미지만 배포)의 필요성을 느꼈다.

**COPY 명령어 문제**

Gradle 빌드 결과물을 실행 스테이지로 복사하는 과정에서 반복적으로 아래 에러를 만났다.

```
When using COPY with more than one source file, the destination must be a directory and end with a /
```

- **원인 분석**: `COPY --from=build /app/build/libs/*.jar /app/app.jar`처럼 와일드카드(`.jar`)를 쓰면, Gradle이 빌드 시 생성하는 `xxx.jar`와 `xxx-plain.jar`(실행 불가능한 원본 jar) **두 개의 파일이 동시에 매칭**되어, 도커가 "여러 개의 소스 파일을 하나의 파일 이름으로 복사할 수 없다"고 거부하는 것이었다.
- **해결**: 빌드 스테이지 안에서 `find` 명령어로 `plain.jar`가 아닌 진짜 실행 파일만 찾아 이름을 `app.jar`로 통일시킨 뒤 복사하도록 변경했다.

```docker
FROM gradle:8-jdk24 AS build
WORKDIR /app
COPY . .
RUN chmod +x gradlew && ./gradlew clean build -x test \
    && find build/libs -name "*.jar" -not -name "*-plain.jar" -exec mv {} /app/app.jar \;

FROM eclipse-temurin:24-jre-alpine
WORKDIR /app
COPY --from=build /app/app.jar /app/app.jar
ENTRYPOINT ["java", "-jar", "/app/app.jar"]
```

Gradle의 `-plain.jar` 생성 관행을 모르고 있었다면 계속 헤맸을 부분이다.

**레이어 캐싱을 어떻게 적용했는가**

- 과제 가이드라인에서 제시한 대로, "라이브러리 정의 파일을 먼저 COPY하고 RUN한 뒤, 소스 코드를 나중에 COPY"하는 원칙을 지키려고 했으나, 실제로는 시간에 쫓겨 `COPY . .`로 전체를 한 번에 복사하는 방식을 사용했다. 이 부분은 아래 추후 보완할 예정이다..

#### 1-3. 레지스트리는 어떤 걸 어떻게 사용해야 할까

**Portainer 자체 레지스트리 vs AWS ECR vs Docker Hub**

- **Portainer로 레지스트리를 직접 구축하는 방법**: 처음에는 실습 자료처럼 Portainer 안에서 레지스트리까지 다 해결하려고 했다. 하지만 Portainer는 애초에 레지스트리 서버 자체를 만드는 도구가 아니라, 이미 존재하는 레지스트리(Docker Hub, ECR, GHCR 등)를 **등록해서 이미지를 pull/관리하는 대시보드**라는 걸 알게 되었다.
- **AWS ECR을 선택한 이유**: 프로젝트가 이미 AWS 인프라(RDS, S3, EC2, ELB, ACM)로 완전히 통합되어 있었기 때문에, 이미지 저장소도 ECR로 맞추는 것이 IAM 기반 권한 관리, 네트워크 통합(같은 VPC 안에서 처리) 측면에서 훨씬 자연스러웠다.
- **결론**: ECR에 이미지를 push하고, Portainer에서는 ECR을 Registry로 등록해서 이미지를 pull/관리하는 구조를 선택했다.

### 2. Portainer 프라이빗 레지스트리 구축 및 빌드 이미지 관리

#### 2-1. SSL 적용한 채로 Docker Image Push/Pull은 어떻게 했는가

**ECR 인증 문제 트러블슈팅**

Portainer의 Registries 메뉴에서 ECR을 등록하고 Pull을 시도했는데, 연달아 두 가지 에러를 만났다.

1. `invalid reference format`
    - **원인**: Portainer UI에서 Registry를 선택한 상태로 Image 칸에 전체 ECR 주소(`536697237610.dkr.ecr.../lulu-backend:latest`)를 또 입력해서, 주소가 중복으로 합쳐져 발생한 문제였다.
    - **해결**: Registry를 지정했다면 Image 칸에는 `lulu-backend:latest`만 입력하거나, 아예 Registry를 `None`으로 두고 전체 주소를 한 번에 입력하는 방식으로 정리했다.
2. `Unable to locate credentials`
    - 이 과정에서 `Unable to locate credentials` 에러도 만났는데, 이는 EC2 인스턴스에 ECR 접근 권한이 있는 IAM Role이 연결되어 있지 않았기 때문이었다. `AmazonEC2ContainerRegistryReadOnly` 정책을 가진 IAM Role을 EC2에 연결하고 나서야 해결되었다.

**S3 연동 시 자격 증명 문제**

DB 연결(`allowPublicKeyRetrieval=true` 옵션 추가로 MySQL 8 인증 문제 해결)까지 마친 뒤, 이번엔 S3 관련 빈 생성 에러를 만났다.

```
No qualifying bean of type 'com.amazonaws.services.s3.AmazonS3' available
```

- **원인**: `application.yml`에 S3 `bucket`과 `region`만 설정되어 있었고, `access-key`/`secret-key` 같은 자격 증명 정보와 이를 이용해 `AmazonS3` 빈을 생성해주는 `@Configuration` 클래스가 없었다.
- **해결**: `S3Config.java`를 새로 작성해 환경변수(`CLOUD_AWS_CREDENTIALS_ACCESS_KEY`, `CLOUD_AWS_CREDENTIALS_SECRET_KEY`)를 `@Value`로 주입받아 `AmazonS3` 빈을 명시적으로 생성했다. 자격 증명을 코드나 yml에 하드코딩하지 않고 `docker run -e` 옵션으로 주입하는 방식을 택해, 최소한의 보안 원칙을 지키려 했다.

```java
@Configuration
public class S3Config {

    @Value("${cloud.aws.credentials.access-key}")
    private String accessKey;

    @Value("${cloud.aws.credentials.secret-key}")
    private String secretKey;

    @Bean
    public AmazonS3 amazonS3() {
        AWSCredentials credentials = new BasicAWSCredentials(accessKey, secretKey);

        return AmazonS3ClientBuilder.standard()
                .withCredentials(new AWSStaticCredentialsProvider(credentials))
                .withRegion(Regions.AP_NORTHEAST_2)
                .build();
    }
}
```

- 이 과정에서 로컬-서버 간 `git pull` 충돌(`divergent branches`, merge conflict)도 겪었는데, `git fetch --all && git reset --hard origin/master`로 서버 상태를 원격 최신본으로 강제 동기화하는 방법을 택했다. 이후로는 "서버에서 직접 코드/설정 파일을 고치지 않고, 로컬에서 수정 → push → 서버 pull"이라는 규칙을 지키기로 했다.

#### 2-2. 보안 정책은 어떻게 설정했는가

- Portainer 접속 자체에 SSL(HTTPS)을 적용하기 위해, 처음에는 Let's Encrypt(Certbot)를 고려했으나 도메인 인증서 발급 절차가 복잡하고, 이미 서비스 전체에 **CloudFront(ACM, us-east-1 리전) + ALB(ACM, ap-northeast-2 리전)** 조합으로 SSL을 적용하고 있었기 때문에, Portainer 관리자 페이지만큼은 자체 서명 인증서(Self-signed Certificate)를 별도로 적용하는 방향으로 결정했다.

```bash
# 1. 인증서 생성
sudo openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout /var/lib/docker/volumes/portainer_data/_data/certs/portainer.key \
  -out /var/lib/docker/volumes/portainer_data/_data/certs/portainer.crt \
  -subj "/CN=localhost"

# 2. SSL 적용하여 Portainer 재실행
sudo docker run -d -p 9443:9443 --name portainer --restart=always \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v /var/lib/docker/volumes/portainer_data/_data:/data \
  -v /var/lib/docker/volumes/portainer_data/_data/certs:/certs \
  portainer/portainer-ce:latest \
  --tlscert /certs/portainer.crt --tlskey /certs/portainer.key
```

- 왜 서비스 SSL과 Portainer SSL을 이중으로 적용했는지가 이번 과제의 중요한 고민 포인트였다. **"End-to-End 암호화" 관점**에서, CloudFront/ALB가 처리하는 것은 "일반 사용자를 위한 서비스 트래픽"이고, Portainer는 "인프라 관리자만 접근하는 별도의 관리 평면(Management Plane)"이기 때문에 별도로 암호화해야 한다고 판단했다.
- 자체 서명 인증서를 쓰면 브라우저가 "연결이 비공개로 설정되어 있지 않습니다"라는 경고를 띄운다. 처음엔 이게 SSL 설정이 실패한 것으로 오해했으나, 자물쇠 아이콘을 눌러 인증서 상세 정보(`CN=localhost`, 발급일/만료일 등)를 확인하는 방법으로, **암호화 자체는 정상 동작하고 있고 단지 '신뢰 기관 미인증'으로 인한 경고임**을 확인했다.
- 명령어 옵션도 한 번 시행착오를 겪었다. 처음엔 `-ssl --sslcert --sslkey`를 사용했는데, 최신 Portainer 버전에서는 해당 플래그가 deprecated 되어 `-tlscert`, `-tlskey`로 대체해야 한다는 경고 로그를 보고 옵션을 교체했다.

#### 2-3. Portainer로 어떻게 이미지 관리를 했는가

- 최종적으로 Portainer의 Registries 메뉴에 ECR을 등록하고, Images/Containers 메뉴에서 배포된 컨테이너의 상태를 모니터링하는 방식으로 사용했다.
- 여러 대의 프라이빗 서버(백엔드/프론트엔드 각각)를 한 화면에서 관리하기 위해 **Portainer Agent** 방식을 도입했다.
    - 처음엔 Edge Agent로 설치했는데, 이는 포트를 열지 않아도 되는 대신, 에이전트가 먼저 포테이너로 접속을 시도하는 방식이라 배스천-프라이빗 서버가 이미 같은 VPC로 통신 가능한 나의 환경에는 맞지 않았다.
    
    **Agent vs Edge Agent 비교**
    
    | 구분 | Agent | Edge Agent |
    | --- | --- | --- |
    | 통신 방향 | Portainer → 대상 서버로 직접 접속 | 대상 서버 → Portainer로 먼저 접속(Poll) |
    | 적합한 환경 | 같은 네트워크(VPC) 내부 | 방화벽 뒤/NAT 뒤의 원격 서버 |
    | 실시간성 | 높음 | 상대적으로 낮음 |
    - 같은 VPC 안에서 사설 IP로 직접 통신이 가능한 환경이었으므로 **표준 Agent 방식**으로 재설치했다.
    - 이 과정에서 `context deadline exceeded`(타임아웃), `connection refused` 에러를 순서대로 만났는데, 각각 보안 그룹에 9001 포트 인바운드 허용이 안 되어 있던 문제, 그리고 Edge Agent 컨테이너가 포트를 열지 않은 채로 떠 있던 문제였다. 보안 그룹 룰 추가 + Agent 이미지를 포트 매핑(`p 9001:9001`)과 함께 재설치하여 최종적으로 해결했다.

```bash
docker run -d \
  --name portainer_agent \
  --restart=always \
  -p 9001:9001 \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v /var/lib/docker/volumes:/var/lib/docker/volumes \
  portainer/agent:latest
```

## 추후 보완 계획

1. **Docker Compose 도입**: 지금은 백엔드/프론트엔드/Portainer를 서버마다 `docker run` 명령어로 하나씩 개별 실행하고 있다. 서비스가 늘어나거나 설정이 복잡해지면 이 방식은 유지보수가 힘들어진다. `compose.yaml`로 네트워크(app_net, management_net 등 역할별 분리), 볼륨, 재시작 정책을 코드로 관리하여 "한 번의 명령으로 전체 스택을 재현 가능하게" 만들 계획이다.
2. **레이어 캐싱 원칙 재정비**: 이번엔 시간에 쫓겨 `COPY . .`를 먼저 하는 구조로 타협했지만, 다음에는 `COPY build.gradle settings.gradle ./` 후 의존성만 먼저 받고, 소스 코드는 마지막에 복사하는 원래 원칙을 지켜서 빌드 속도를 최적화할 것이다.
3. **정식 도메인 기반 SSL로 전환**: 지금은 Portainer에 자체 서명 인증서를 쓰고 있어 브라우저 경고가 뜬다. 서브도메인(`portainer.도메인.com`)을 발급받아 Let's Encrypt/ACM 기반의 신뢰된 인증서로 교체하여 적용하고 싶다.
4. **디스크 부족 문제 근본 해결**: 서버에서 직접 빌드하는 방식 대신, 로컬/CI에서 빌드한 이미지를 ECR에 올리고 서버는 pull만 하는 구조로 바꿔서 EC2의 디스크 부담 자체를 줄일 계획이다.
5. **Portainer Registry 관점 재정리**: 지금은 Portainer가 ECR을 참조하는 형태로만 쓰고 있는데, Push 권한(개발자용 토큰)과 Pull 권한(서버용 최소 권한)을 IAM 정책 수준에서 분리하여 최소 권한 원칙을 더 엄격히 적용하고 싶다.
6. AWS Secrets Manager 를 찾아보고, 사용을 고려해볼 생각이다.
