#### | Docker Compose

**여러 컨테이너로 구성된 애플리케이션을 YAML 형식의 파일로 명확하게 정의하고, 단일 명령어로 동시에 실행 및 관리할 수 있도록 지원하는 Docker의 공식 오픈소스 도구**

Docker Compose는 `docker-compose.yml` 이라는 Config YAML 파일을 읽어서, 이미지 빌드(Build) → 컨테이너 실행(Run) → 볼륨·네트워크 구성까지 한 번에 처리해준다.

##### Docker 명령어로만 구성했을 때의 어려움

서비스가 여러 개라면 네트워크, 볼륨, 컨테이너를 모두 수동으로 명령어를 입력해야 한다.

```bash
docker network create app-network
docker volume create mysql-data

docker run -d \
  --name mysql \
  --network app-network \
  -e MYSQL_ROOT_PASSWORD=password \
  -e MYSQL_DATABASE=demo \
  -v mysql-data:/var/lib/mysql \
  mysql:8.0

docker run -d \
  --name backend \
  --network app-network \
  -e DB_HOST=mysql \
  -e DB_PORT=3306 \
  -p 8080:8080 \
  my-backend:1.0.0

docker run -d \
  --name nginx \
  --network app-network \
  -p 80:80 \
  nginx:stable-alpine
```

처음에는 괜찮다. 하지만 다음과 같은 상황이 생기면 어떻게 될까?

- 서비스가 더 늘어나거나, 운영 중에 수시로 껐다 켜거나, 명령어가 수정되거나, 다른 개발자가 이 명령어를 쳐야 한다면?
- **실행 순서도 기억해야 한다** (백엔드는 DB가 만들어지고 난 뒤 실행해야 하는데, DB 컨테이너는 만드는 데 시간이 더 걸린다)

> 💡 도커 컴포즈는 "컨테이너 여러 개를 하나의 애플리케이션으로 관리하기 어려우니, 그걸 해결하기 위해" 나왔다.

##### YAML 파일 예시 (`./docker-compose.yml`)

도커 컴포즈 파일 안에는 애플리케이션 구성에 필요한 모든 것을 선언한다.

```yaml
services:
  # Dockerfile로 직접 빌드한 서비스
  app-build:
    build: .
    ports:
      - "3000:3000"
    container_name: your-container-name

  # Docker Hub의 미리 빌드된 이미지를 사용한 서비스
  app-image:
    image: your-username/your-image-name:latest
    ports:
      - "3001:3000"
    container_name: your-container-name
```

도커 컴포즈 파일에 선언하는 것들:
- 어떤 이미지로 실행할지
- 어떤 Dockerfile로 빌드할지
- 어떤 포트를 열지
- 어떤 환경 변수를 넣을지
- 어떤 네트워크에 연결할지
- 어떤 볼륨을 연결할지
- 어떤 서비스에 의존하는지
- 장애 시 재시작할지

##### Docker Compose의 핵심 개념

**예시 프로젝트 구조**

```
memo-app/
├── compose.yaml
├── backend/
│   └── Dockerfile
├── frontend/
│   └── Dockerfile
└── nginx/
    └── default.conf
```

`docker compose up -d` 명령 하나로 이 구조를 실행하면, `docker ps`에서 다음처럼 결과가 나온다.

```
memo-app-backend-1
memo-app-mysql-1
memo-app-nginx-1
memo-app_default       ← 네트워크
memo-app_mysql-data    ← 볼륨
```

> 이 단위를 **Stack** 또는 **Project**라고 부른다.

---

**1. Service**

Docker Compose에서 `Service`는 **실행해야 하는 컨테이너의 역할**이다.

```yaml
services:
  backend:
    image: my-backend:1.0.0

  mysql:
    image: mysql:8.0

  nginx:
    image: nginx:stable-alpine
```

- `backend`, `mysql`, `nginx`는 **서비스의 이름**이다.
- 서비스 이름은 컨테이너의 이름이 아니라, **애플리케이션 내부에서 사용하는 논리적인 역할 이름**이다.
- 백엔드에서 DB 주소를 바라볼 때 `DB_HOST: mysql`처럼 **서비스 이름을 그대로 DNS로 사용**할 수 있다.

**스케일 아웃**

```bash
docker compose up -d --scale backend=3
# → memo-app-backend-1
# → memo-app-backend-2
# → memo-app-backend-3
```

- `docker compose`에서 컨테이너 이름을 고정(`container_name:`)하면, 이름이 유일해야 하기 때문에 스케일 시 오류가 발생한다.
- 스케일이 필요하다면 **포트 공개(`ports:`)도 하면 안 된다.** 같은 포트를 3개가 쓸 수 없기 때문이다.
- 대신 Nginx가 내부 네트워크에서 `backend-1:8080`, `backend-2:8080`, `backend-3:8080`으로 분산하면 된다.

```
사용자 → 80/443
       → Nginx (Docker Compose Network)
       → backend-1:8080 / backend-2:8080 / backend-3:8080
```

---

**2. Network**

`compose`를 사용하면 별도로 네트워크를 만들지 않아도, **기본 네트워크를 자동으로 생성**한다.

```yaml
services:
  backend:
    image: my-backend:1.0.0
  mysql:
    image: mysql:8.0
  nginx:
    image: nginx:sts
```

```
Compose 기본 브리지 네트워크:
사용자 → Nginx(80/443) → backend(8080) → mysql(3306)
```

이 세 서비스가 자동으로 같은 네트워크에 묶이고, 서비스 이름으로 DNS가 자동 등록된다.

---

**3. Volume**

컨테이너가 삭제되면 내부 데이터가 사라진다. 볼륨으로 **호스트 저장소를 사용**하면 데이터를 컨테이너 외부에 영속적으로 보관할 수 있다.

```yaml
services:
  mysql:
    image: mysql:8.0
    volumes:
      - mysql-data:/var/lib/mysql

volumes:
  mysql-data:
```

---

**4. 환경 변수**

보통 환경 변수는 컨테이너마다 다른 값을 넣는다. 추가되거나 변경되는 경우도 있다.

> ⚠️ API Key, PW 같은 민감한 정보는 `docker-compose.yaml`에 직접 적지 않는다. `docker-compose.yaml`은 깃허브로 올라가는 파일, 즉 **공유되어야 하는 파일**이기 때문이다.

```yaml
services:
  mysql:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD}
      MYSQL_DATABASE: ${MYSQL_DATABASE}
```

> `.env`나 Secret 관리 서비스(AWS Secrets Manager, Vault 등)를 통해 외부에서 주입해준다.

---

**5. depends_on**

서비스 간 생성·시작 순서와 의존성을 선언하는 것.

```yaml
services:
  backend:
    image: my-backend:1.0.0
    depends_on:
      - mysql

  mysql:
    image: mysql:8.0
```

→ "mysql을 먼저 만들고, backend를 만들어야겠구나"

> ⚠️ 단, `depends_on`은 **컨테이너의 생성·시작 순서만 조정**할 뿐, **실제 요청 준비 순서를 보장하지 않는다.** 컨테이너가 시작되는 순서와, 내부에서 실제 프로세스(MySQL 서버)가 준비되는 순서는 일치하지 않기 때문이다.

**의존성 순서를 확실하게 보장하려면 `healthcheck`를 함께 사용해야 한다.**

```yaml
services:
  backend:
    image: my-backend:1.0.0
    depends_on:
      mysql:
        condition: service_healthy   # MySQL이 healthy 상태일 때만 backend 시작

  mysql:
    image: mysql:8.0
    environment:
      MYSQL_DATABASE: memo
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD}
    healthcheck:
      test: ["CMD-SHELL", "mysqladmin ping -h localhost -p$$MYSQL_ROOT_PASSWORD"]
      interval: 10s
      timeout: 5s
      retries: 10
```

**동작 흐름:**

```
Docker Compose → MySQL 컨테이너 시작
              → 초기화 및 DB 생성
              → healthcheck 반복 확인
              → healthy 상태 확인
              → Backend 컨테이너 시작
              → mysql:3306으로 연결
```

##### 실무 예시

도커 컨테이너를 별개로 사용하는 경우는 극히 드물다. **거의 도커 컴포즈로 사용**한다.

**① Nginx + WAS + DB 구성**

가장 기본적이고 흔한 구성이다.

**② 특히 개발 서버를 만들 때 많이 사용**

- 운영 서버는 보통 컨테이너들이 분리되어 있는 구성이 많다.
  - BE EC2 → BE Container 모아 두기
  - FE EC2 → FE Container 모아 두기
- **개발 서버는 하나의 서버를 운영하는 경우가 많다.**
  - 하나의 EC2에 여러 가지 컨테이너를 둬서 저렴하게 구성해야 하기 때문이다.

---

##### 주의해야 하는 사항

**① `latest` 태그를 피해야 한다**

- 실제로 어떤 버전을 배포하는지 알 수 없다.
- 태그에 버전을 명시하는 게 좋다.
- SHA 태그를 붙여도 된다. (이미지나 GitHub 커밋에서 만들어지는 태그)

**② `depends_on`을 신뢰하지 않는다**

의존성이 꼭 필요한 경우(DB 연결 등)는 **`healthcheck`를 반드시 함께 설정**해야 한다.

**③ 로그 회전 설정**

기본 도커 로그는 **서버 디스크에 계속 쌓인다.** 운영 환경에서는 로그가 회전할 수 있도록 설정해야 한다.

```yaml
services:
  backend:
    image: my-company/backend:1.0.0
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "5"
```

#### | IaC (Infrastructure as Code)

**인프라 자체를 코드로 만드는 것**

콘솔에서 클릭으로 인프라를 구성하는 것은 직관적이다. 하지만 운영이 시작되면 같은 작업을 다시 해야 하는 상황이 많아지고, 정말 다양한 문제와 어려움이 발생한다.

> "인프라에도 Git이 필요하지 않을까?" → **그래서 IaC**

**문서 대신 코드, 기억 대신 코드**

IaC는 실제 리소스를 어떤 문법으로 표현하느냐에 따라 **선언형**과 **코드형**으로 나뉜다.

---

##### 1. 선언형 IaC

**원하는 최종 상태를 적는다.** 도구가 순서와 의존성을 알아서 계산한다.

대표 도구: **Terraform**, **CloudFormation**

```hcl
resource "aws_instance" "web" {
  ami                    = "ami-xxxx"
  instance_type          = "t3.micro"
  vpc_security_group_ids = [aws_security_group.web.id]
}
```

핵심 질문: **"무엇이 있어야 하는가?"**

| 구분 | 내용 |
|---|---|
| **장점** | 인프라 구조가 눈에 잘 보임, 코드 리뷰와 버전 관리가 쉬움, 작은 실습과 기본 인프라에 적합 |
| **단점** | 환경이 많아지면 파일이 길어짐, 조건/반복/재사용 표현이 제한적, 복잡한 조직 표준을 매번 직접 적기 어려움, 모듈 설계가 없으면 복붙 인프라가 됨 |

---

##### 2. 코드형 IaC

**일반 프로그래밍 언어로 작성**하며, 조건문·반복문·클래스를 사용할 수 있다.

대표 도구: **AWS CDK**, **Pulumi**

```typescript
const web = new ec2.Instance(this, "Web", {
  instanceType: ec2.InstanceType.of(...),
  machineImage: ec2.MachineImage.latestAmazonLinux2023(),
  vpc,
});
```

코드형 IaC가 필요한 순간:
- 모든 서비스에 같은 태그 정책 적용
- 서비스마다 ALB + Target Group + SG 생성 등 반복 작업

| 구분 | 내용 |
|---|---|
| **장점** | 조건문/반복문으로 환경별 구성 표현, 함수·클래스·패키지로 표준 재사용, 테스트 코드와 정적 분석 적용 가능 |
| **단점** | 러닝커브가 있다, 실제 생성되는 리소스가 코드 뒤에 숨겨질 수 있음 |

---

##### 대표 IaC 도구 비교

| 도구 | 특징 |
|---|---|
| **CloudFormation** | AWS 공식 IaC. Stack / Template 기반 |
| **Terraform** | 멀티 클라우드 IaC. Provider / State / Plan 개념 사용 |
| **AWS CDK** | 프로그래밍 언어로 작성, 최종적으로 CloudFormation으로 배포 |

> 💡 **IaC의 State와 Plan은 매우 중요**하므로 잘 관리하고 잘 확인해야 한다. (특히 Terraform에서 State 파일이 실제 인프라와 어긋나면 큰 문제가 생길 수 있다.)

##### Git과 IaC

인프라 변경도 **코드 리뷰**의 대상이 된다. 즉, 인프라 수정이 곧 PR 리뷰 → 머지 → 반영의 흐름을 타게 된다. 이 점이 IaC의 가장 강력한 장점 중 하나다.

### 🗨️ 오늘의 회고

- 복습과 블로그 작성이 생각보다 너무 많은 시간을 차지하고 있다. 오늘은 위클리 챌린지 설계 완료를 목표로 잡고 집중했다.
- Docker Compose가 단순히 "여러 컨테이너를 편하게 실행하는 도구"가 아니라, **서비스 구성 전체를 코드로 선언하는 설계 도구**라는 관점이 생겼다. 특히 `services`, `networks`, `volumes`가 각각 컨테이너·네트워크·스토리지를 코드로 추상화한 개념이라는 걸 깨달으니, YAML 파일을 어떻게 읽어야 하는지가 명확해졌다.
- `depends_on`이 컨테이너 생성 순서만 보장하고 프로세스 준비 순서는 보장하지 않는다는 점이 중요한 함정이었다. MySQL 컨테이너가 시작됐어도 DB 초기화가 끝나지 않은 상태에서 backend가 연결을 시도하면 실패한다는 게 직관적으로는 헷갈리는 부분인데, `healthcheck` + `condition: service_healthy`로 해결하는 패턴을 알게 돼서 좋았다.
- **로그 회전 설정**은 놓치기 쉬운 운영 항목이었다. 기본 도커 로그가 디스크에 계속 쌓이면 결국 서버가 디스크 용량 부족으로 멈추는 상황이 생길 수 있다는 걸 강조해서 기억에 남았다.
- IaC는 "코드로 인프라를 찍어낸다"는 개념 자체보다, **"인프라 변경도 Git으로 관리하고 코드 리뷰를 받는다"**는 운영 문화 측면이 더 인상 깊었다. 선언형(Terraform)과 코드형(AWS CDK)의 차이도 "무엇이 있어야 하는가 vs. 어떻게 만들 것인가"의 관점 차이로 이해하니 명확해졌다.
- `latest` 태그 금지, `depends_on` 불신, 로그 회전 — 이 세 가지는 Docker Compose를 운영 환경에서 쓸 때 꼭 지켜야 할 원칙으로 따로 기억해둬야겠다.
