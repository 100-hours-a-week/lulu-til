#### | Namespace

**하나의 쿠버네티스 클러스터 안에서 리소스를 목적이나 팀별로 구분하여 관리하는 논리적인 공간**

> ⚠️ 네임스페이스는 **클러스터를 나누는 것이 아니다.** 어디까지나 하나의 클러스터 안에서의 논리적인 구분이다.

```
K8s Cluster
├── default namespace
│     ├── App1 (svc, pod, deploy)
│     └── App2 (svc, pod, deploy)
├── dev namespace
│     ├── App1 (svc, pod, deploy)
│     └── App2 (svc, pod, deploy)
└── qa namespace
      ├── App1 (svc, pod, deploy)
      └── App2 (svc, pod, deploy)
```

##### 네임스페이스가 왜 필요할까?

클러스터에 리소스가 적을 때는 사실 네임스페이스 하나에 이것저것 다 올려도 괜찮다. 하지만 쿠버네티스를 도입할 정도라면 보통 **팀의 규모가 커졌거나, 서비스 자체가 여러 개의 애플리케이션으로 나뉘는 상황**일 가능성이 크다.

**네임스페이스가 없다면?**

하나의 클러스터에 개발 환경, 스테이징 환경(운영 배포 전 마지막으로 실제 운영 환경과 유사하게 만든 환경), 운영 환경이 있고, 각 환경마다 애플리케이션(WAS)의 API Server를 띄운다고 가정해보자.

```
클러스터
├── api-server  ← 개발 환경
├── api-server  ← 스테이징 환경
└── api-server  ← 운영 환경
```

**쿠버네티스에서는 동일한 종류의 리소스가 같은 이름을 가질 수 없다.** 위 예시처럼 리소스 이름이 겹치면 리소스 구분이 힘들어지고("이 api-server는 개발용인지 운영용인지…"), 사람이 실수할 가능성이 높아지며, 전반적으로 관리가 복잡해진다.

> 💡 네임스페이스를 통해서 리소스들을 논리적으로 구분하면 이 문제가 해결된다.

```
Kubernetes Cluster
├── dev Namespace
├── staging Namespace
└── production Namespace
```

##### 네임스페이스는 사무실의 부서와 비슷하다

회사 건물 안에 개발팀, 디자인팀, 기획팀처럼 여러 부서가 있다고 해보자. 만약 자리가 다 섞여 있다면 혼란스러울 테니, 개발팀 공간·디자인팀 공간·기획팀 공간으로 나눈다.

모든 팀에 "회의실"이라는 개별 공간이 부여되었다면, "회의실로 모여"라고 했을 때 개발팀은 개발팀 회의실로, 디자인팀은 디자인팀 회의실로 모인다. **"회의실"이라는 이름은 같지만, 실제로는 서로 다른 공간(리소스)을 가리킨다.** 네임스페이스도 이와 같다 — 같은 이름의 리소스(`api-server` 등)라도 네임스페이스가 다르면 완전히 다른 리소스를 가리킨다.

##### 핵심 역할

**1. 리소스를 논리적으로 구분**

네임스페이스는 관련된 리소스를 하나의 그룹으로 묶어서 관리한다. 환경별로 나누기도 하고, 팀마다 방식이 다를 수 있다. **네임스페이스를 나누는 기준은 클러스터의 규모에 따라 달라진다.**

| 팀 규모 | 네임스페이스 사용 방식 |
|---|---|
| **규모가 작은 팀** | 환경 구분을 위해 사용 (예: `dev`, `staging`, `production`) |
| **규모가 큰 팀** | 클러스터 자체를 `dev`/`staging`/`prod`로 나누고, 네임스페이스는 팀 단위나 기능 단위(`frontend`, `backend`, `data`)로 사용 |

**2. 리소스의 이름 범위 제공** — 같은 네임스페이스 안에서는 같은 종류의 리소스 이름이 중복될 수 없다.

**3. 명령어 범위를 구분하는 역할** — 네임스페이스를 사용하면 특정 공간의 리소스만 조회하거나 명령할 수 있다.

```bash
kubectl get pods           # default namespace를 사용 중
kubectl get pods -n dev    # dev 네임스페이스만 조회
```

**4. 정책을 적용하는 단위가 된다** — 단순히 리소스를 나누는 것에서 그치지 않고, 네임스페이스 단위로 운영 정책을 적용할 수 있다.

| 정책 | 역할 |
|---|---|
| **ResourceQuota** | Namespace 전체가 사용할 수 있는 CPU, 메모리, Pod 개수 등의 총량 제한 |
| **LimitRange** | Namespace 안의 개별 컨테이너나 Pod가 사용할 수 있는 리소스 범위 설정 |
| **접근 권한** | 특정 사용자나 팀이 특정 Namespace만 관리하도록 구성 |
| **NetworkPolicy** | Namespace 또는 Pod 사이의 네트워크 통신을 제한 |

##### 오해하는 것들

**1. 클러스터를 나누지 않는다**

네임스페이스는 어디까지나 논리적인 구분일 뿐, 클러스터 자체는 그대로 함께 사용된다 — 클러스터의 노드 자원(CPU, 메모리, 디스크 등)을 실제로 공유한다. 노드에 문제가 생기거나 노드 리소스에 문제가 생기면, **네임스페이스만으로는 그 문제들이 격리되지 않는다.**

**2. 네임스페이스만으로 네트워크가 차단되지 않는다**

네임스페이스가 다르다고 해서 통신이 자동으로 차단되는 것이 아니다 (AWS VPC Subnet과 헷갈리기 쉬운 부분이다). 기본적으로 **Pod 간 통신은 네임스페이스와 무관하게 가능**하다.

> ⚠️ 쿠버네티스에서 가장 흔하게 보안 문제가 생기는 패턴 중 하나는, 특정 파드가 과하게 노출되는 경우다 (예: 포트가 열린 컨테이너 관리 도구, 사설 registry, bastion host, NAT instance, self-hosted runner 역할을 하는 파드 등). 이런 파드에 침투한 공격자는 **다른 네임스페이스로도 curl 통신이 가능**하다.
>
> 네임스페이스 간 통신을 제한하고 싶다면, **CNI에서 NetworkPolicy를 별도로 설정**해야 한다.

**3. 네임스페이스만으로는 접근 권한을 나눌 수 없다**

쿠버네티스는 다수의 개발자가 접근할 수 있는 환경이다. FE, BE, DB 또는 Team-A, Team-B처럼 네임스페이스를 쪼개도, 별도의 권한 설정 없이는 **Team A 사람들이 Team B의 네임스페이스에 접근할 수 있다.**

##### 기본 네임스페이스

클러스터를 만들면 기본으로 제공되는 네임스페이스들이 있다.

```bash
kubectl get namespaces
```

```
NAME                STATUS   AGE
default             Active   10d
kube-node-lease     Active   10d
kube-public         Active   10d
kube-system         Active   10d
```

| Namespace | 역할 |
|---|---|
| **default** | Namespace를 지정하지 않은 일반 리소스가 기본적으로 생성되는 공간 |
| **kube-system** | DNS, kube-proxy 등 쿠버네티스 시스템 컴포넌트가 실행되는 공간 |
| **kube-public** | 클러스터 전체에서 공개적으로 읽을 수 있는 정보에 사용되는 공간 |
| **kube-node-lease** | 각 노드의 상태 확인을 위한 Lease 오브젝트가 저장되는 공간 |

##### 구조적인 특징

1. **네임스페이스는 중첩할 수 없다** — 구조가 폴더처럼 보여서 중첩이 가능할 것 같지만, `production ns` 안에 `backend ns`를 두는 방식은 불가능하다.
2. **하나의 리소스는 하나의 네임스페이스에만 속한다** — 예를 들어 `api-server`라는 리소스는 `dev` 네임스페이스 또는 `prod` 네임스페이스 중 하나에만 속할 수 있다.

##### 사용 방법

**1. YAML**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: api-server
  namespace: dev
spec:
  containers:
    - name: nginx
      image: nginx
```

**2. 명령어**

```bash
kubectl get pods -n dev
kubectl apply -f pod.yaml -n dev
```

> ⚠️ **네임스페이스를 삭제할 때는 조심해야 한다.** 네임스페이스를 삭제하면 그 안의 리소스(Pod, Secret, Service 등)가 전부 함께 삭제된다.
>
> ```bash
> kubectl delete ns dev   # pod, secret, service 등이 함께 사라짐
> ```

##### 실무 사용 기준

1. **네임스페이스는 너무 세세하게 나누지 않는다** — 개별 리소스를 기준으로 나누기보다는, 정책이나 리소스 그룹을 기준으로 나누는 것이 좋다 (팀 단위: `team-a`, `team-b` / 서비스 단위: `payment-ns`, `auth-ns` / 환경별: `dev`, `staging`, `prod`).
2. **명명 규칙을 회사 단위로 공유해야 한다** — 네임스페이스는 클러스터 단위로 나뉘는 가상의 폴더이므로, 이름만 보고도 목적을 알 수 있어야 한다 (예: `payment-dev-ns`, `team-a-ns`).
3. **default 네임스페이스는 사용하지 않는다** — 애플리케이션을 default에 배치하면 다른 리소스들은 이미 별도 네임스페이스에서 네트워크 정책·환경 분리 정책 등으로 관리되고 있는데, default만 그 관리 체계에서 벗어나 있어 관리하기 상당히 까다로워진다.
4. **명령어로 관리할 때는 현재 네임스페이스를 잘 확인해야 한다** — 예를 들어 "test 네임스페이스를 정리해야지"라고 생각하고 `kubectl delete service backend-service`를 실행했는데, 알고 보니 default(운영) 네임스페이스의 리소스를 지워버리는 사고가 날 수 있다. 항상 `-n test`처럼 네임스페이스를 명시하는 습관이 필요하다.

> 💡 네임스페이스끼리 통신할 때는 보통 Service를 통해 통신한다.
> ```
> http://backend-service:8080              # 같은 네임스페이스
> http://backend-service.dev-ns:8080        # 다른 네임스페이스
> ```

##### 정리

| 항목 | 내용 |
|---|---|
| 한 줄 정리 | 하나의 클러스터 안에서 리소스를 논리적으로 구분하는 공간 |
| 주요 목적 | 팀, 프로젝트, 환경별 리소스 관리 및 정책 적용 |
| 이름 범위 | 같은 Namespace에서는 같은 종류의 리소스 이름이 중복될 수 없음 |
| 다른 Namespace | 동일한 이름의 리소스를 생성할 수 있음 |
| 기본 Namespace | `default` |
| 정책 적용 | ResourceQuota, LimitRange, NetworkPolicy 등의 기준으로 사용 |

#### | ResourceQuota

**하나의 네임스페이스에서 요청할 수 있는 CPU, 메모리와 생성할 수 있는 쿠버네티스 리소스의 총량을 제한하는 정책**

한마디로 **네임스페이스 자원량의 상한선을 설정하는 리소스**다.

##### 왜 필요할까?

네임스페이스를 사용하면 클러스터를 논리적으로 분리할 수 있지만, 앞서 정리했듯 이는 **논리적인 분리일 뿐 클러스터의 실제 자원은 함께 사용**한다. 워커 노드가 다음과 같다고 가정해보자.

```
Worker Node
CPU: 8 Core (Small x 4)
Memory: 16Gi (Small x 4)

├── team-a의 Pod
├── team-b의 Pod
└── team-c의 Pod
```

**ResourceQuota가 없으면?** team-a의 파드가 7코어를 요청하고, team-b의 파드가 1코어를 요청해버리면, team-c의 파드는 코어를 할당받지 못하는 상황이 생길 수 있다. **네임스페이스만으로는 자원량을 제한하지 못한다.**

ResourceQuota는 네임스페이스마다 사용할 수 있는 총량을 지정하는 방식으로 이 문제를 해결한다.

```
team-a
├── CPU requests 합계: 최대 2 Core
├── Memory requests 합계: 최대 4Gi
├── Pod 개수: 최대 10개
└── Service 개수: 최대 5개
```

##### 제한할 수 있는 대상

- **컴퓨팅 자원**: CPU, Memory
- **오브젝트**: Pod 개수, Service 개수, ConfigMap 개수, 스토리지 등

##### 비유

- 네임스페이스 = 부서
- 리소스쿼터 = 팀의 예산

예산을 팀별로 나누지 않으면 개발팀이 인건비로 50억을 다 써버릴 수도 있다. 그래서 "개발팀 최대 예산 10억, 디자인팀 3억…" 처럼 팀별 상한을 정해두는 것이 ResourceQuota의 역할이다.

##### ResourceQuota, Pod의 Limits/Requests의 관계

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: resource-demo
spec:
  containers:
    - name: app
      image: nginx
      resources:
        requests:
          cpu: "500m"
          memory: "256Mi"
        limits:
          cpu: "1"
          memory: "512Mi"
```

| 설정 | 값 | 의미 |
|---|---|---|
| `requests.cpu` | 500m | 스케줄링할 때 CPU 0.5개가 필요하다고 판단 |
| `requests.memory` | 256Mi | 스케줄링할 때 메모리 256Mi가 필요하다고 판단 |
| `limits.cpu` | 1 | 컨테이너가 사용할 수 있는 CPU 상한 |
| `limits.memory` | 512Mi | 컨테이너가 사용할 수 있는 메모리 상한 |

1. **requests / limits** — 각 컨테이너가 얼마나 요청하고 사용할 것인가를 결정한다.
2. **ResourceQuota** — Namespace 전체에서 requests/limits의 합계를 어디까지 허용할 것인가를 결정한다.

| 구분 | Pod의 requests · limits | ResourceQuota |
|---|---|---|
| 적용 범위 | 개별 컨테이너 또는 Pod | Namespace 전체 |
| 작성 위치 | Pod의 `resources` | 별도의 ResourceQuota 오브젝트 |
| 주요 목적 | 스케줄링 기준과 실행 상한 설정 | Namespace의 누적 요청량과 오브젝트 수 제한 |
| 적용 대상 | 특정 워크로드 | 해당 Namespace 안의 모든 워크로드 |
| 초과 시점 | 실행 중 제한 또는 스케줄링 실패 가능 | 리소스 생성·수정 요청 단계에서 거부 |
| 실제 자원 배분 | 개별 컨테이너에 적용 | 개별 컨테이너에 자원을 배분하지 않음 |

##### ResourceQuota YAML

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-a-quota
  namespace: team-a
spec:
  hard:
    requests.cpu: "2"
    requests.memory: "2Gi"
    limits.cpu: "4"
    limits.memory: "4Gi"
```

```
team-a Namespace 전체
requests.cpu 합계    ≤ 2 Core
requests.memory 합계 ≤ 2Gi
limits.cpu 합계      ≤ 4 Core
limits.memory 합계   ≤ 4Gi
```

만약 Pod가 이 ResourceQuota를 초과하면 **생성 요청 자체가 거절**된다.

##### ResourceQuota는 실제 사용량을 측정하지 않는다

ResourceQuota는 CPU·Memory 할당량에 대해, **현재 실제로 사용하고 있는 양을 기준으로 하지 않는다.** 각 파드에 **선언되어 있는** `limits`, `requests` 값을 기준으로 작동한다.

> ⚠️ Pod에 requests·limits를 작성하지 않으면 어떻게 될까? ResourceQuota가 걸려 있는 네임스페이스에 배치하려고 하면, 리소스 쿼터는 실제 사용량이 아니라 파드가 선언한 requests/limits 값을 합산하는 방식이기 때문에 **해당 값이 없으면 계산할 수 없어 그냥 생성 요청이 거절**된다. 그렇다고 매번 모든 파드에 일일이 값을 적기도 번거로운데, 이 문제는 뒤에 나오는 **LimitRange**로 해결한다.

##### 사용 방법

```bash
kubectl get resourcequota -n team-a
```

##### 언제 사용하냐

1. **여러 팀이 하나의 클러스터를 공유할 때** (클러스터 규모가 클 때) — 특정 팀이 자원을 다 가져다 쓰지 못하게 방지.
2. **환경별로 자원을 제한할 때** (클러스터 규모가 작을 때) — "dev 환경은 자원을 적게 써라" 같은 정책.
3. **오브젝트의 무한한 생성을 제한해야 할 때** — 예를 들어 트래픽이 많아지거나 요청 처리가 밀릴 때, 자동 확장 로직이 잘못 설계되어 있으면 다음과 같은 악순환이 생길 수 있다.
   - DB 레벨에서 데드락 발생
   - 처리 속도가 느려짐
   - (문제 해결을 위해) WAS Pod를 더 생성
   - DB 커넥션이 더 소진됨
   - 데드락이 더 안 풀림
   - WAS Pod를 계속 더 생성 → **무한 반복**

   ResourceQuota는 이럴 때 **최종 방어선**처럼 자원의 총 개수를 정해두는 역할을 한다.

##### 정리

| 항목 | 내용 |
|---|---|
| 한 줄 정리 | Namespace에서 사용할 수 있는 리소스의 총량을 제한하는 정책 |
| 적용 범위 | 특정 Namespace 전체 |
| 컴퓨팅 리소스 | CPU·메모리의 requests와 limits 합계 |
| 오브젝트 리소스 | Pod, Service, ConfigMap, Secret, PVC 등의 개수 |
| 초과 시 동작 | 새로운 생성 또는 수정 요청이 거부됨 |
| 계산 기준 | 일반적으로 선언된 requests와 limits의 합계 |

---

#### | LimitRange

**특정 네임스페이스에서 생성되는 개별 컨테이너나 Pod의 requests와 limits에 기본값, 최솟값, 최댓값을 적용하는 정책**

- **ResourceQuota** → NS 전체 총량 제한
- **LimitRange** → 개별 컨테이너의 리소스 범위 제한 (기본값, 최솟값, 최댓값)

##### ResourceQuota만 있는 경우 발생하는 문제

CPU request 총량이 4로 설정된 네임스페이스에, 아래와 같은 파드가 하나 생성되었다고 가정해보자.

```yaml
type: pod
resources:
  requests:
    cpu: "4"
```

이 경우 **네임스페이스 하나에 CPU 코어 개수가 4인 컨테이너 하나가 만들어지고, 다른 파드는 더 이상 배치할 수 없게 된다.** 즉 하나의 컨테이너가 네임스페이스 전체 총량을 다 사용해버릴 수 있다. ResourceQuota 관점에서는 정상적인 동작이지만, **운영 측면에서는 좋은 상황이 아니다.** 이 문제를 해결하기 위해 LimitRange를 함께 사용한다.

##### LimitRange가 설정할 수 있는 항목

```
LimitRange
├── 기본 request (defaultRequest)
├── 기본 limit (default)
├── 최소 request (min)
├── 최대 limit (max)
├── request와 limit의 최대 비율 (Ratio)
└── PVC의 최소·최대 요청 용량
```

| 기능 | 설명 |
|---|---|
| **기본값 적용** | requests나 limits를 생략한 컨테이너에 기본값 적용 |
| **최소값 제한** | 지나치게 작은 request를 선언하지 못하도록 제한 |
| **최대값 제한** | 하나의 컨테이너가 지나치게 큰 limit을 선언하지 못하도록 제한 |
| **비율 제한** | request와 limit의 차이가 지나치게 커지지 않도록 제한 |
| **PVC 제한** | 개별 PVC가 요청할 수 있는 스토리지 용량의 범위 제한 |

##### 비유

- 네임스페이스 = 부서
- ResourceQuota = 팀 전체 예산
- LimitRange = 개인별 결제 한도

팀 전체 예산이 천만 원(ResourceQuota)인데, 신입 인턴이 800만 원짜리 결제를 해버리면 문제가 생긴다. 그래서 "최소 결제 만 원, 최대 결제 200만 원"처럼 **개별 결제 규칙**을 만드는 것이 LimitRange다.

##### ResourceQuota vs LimitRange

| 구분 | ResourceQuota | LimitRange |
|---|---|---|
| 적용 범위 | Namespace 전체 | 개별 Container, Pod 또는 PVC |
| 주요 목적 | 전체 리소스 총량 제한 | 개별 리소스의 범위 제한 |
| CPU 예시 | 모든 CPU request 합계 최대 4 | 컨테이너 하나의 CPU request 최대 1 |
| Pod 개수 제한 | 가능 | 주된 역할이 아님 |
| 기본값 적용 | 하지 않음 | 가능 |
| 최소값 설정 | 전체 총량의 최소값을 정하지 않음 | 개별 request의 최소값 설정 가능 |
| 최대값 설정 | Namespace 전체 합계의 최대값 | 개별 오브젝트의 최대값 |
| 초과 시 | 생성 또는 수정 요청 거부 | 생성 또는 수정 요청 거부 |

> 💡 일반적으로 ResourceQuota와 LimitRange는 **함께 사용**한다.

##### 네임스페이스가 갖고 있는 특징 다시 짚기

- 클러스터를 물리적으로 분할하지 않는다.
- 다른 네임스페이스에서 생긴 에러나 활동이 현재 네임스페이스에 영향을 줄 수 있다 (리소스 제한으로 인한 노드 장애 포함).
- 해결법: **ResourceQuota**로 네임스페이스 자원량을 할당하지만, 컨테이너 하나에 대한 제한은 두지 못한다. → **LimitRange**로 컨테이너에 최소값·최대값·기본값을 걸어준다.

##### LimitRange 설정 예시

```yaml
spec:
  limits:
    - type: Container
      min:
        cpu: "100m"
        memory: "128Mi"
      max:
        cpu: "1"
        memory: "1Gi"
      defaultRequest:
        cpu: "250m"
        memory: "256Mi"
      default:
        cpu: "500m"
        memory: "512Mi"
      maxLimitRequestRatio:
        cpu: "2"
        memory: "2"
```

| 설정 | 역할 |
|---|---|
| `min` | 개별 리소스가 요청해야 하는 최소 기준 |
| `max` | 개별 리소스에 설정할 수 있는 최대 기준 |
| `defaultRequest` | request를 생략했을 때 적용할 기본값 |
| `default` | limit을 생략했을 때 적용할 기본값 |
| `maxLimitRequestRatio` | limit이 request의 몇 배까지 가능한지 제한 |
| `type` | 규칙을 적용할 오브젝트 종류 (Container, Pod, PersistentVolumeClaim) |

##### LimitRange가 실제 사용량을 제한하는가?

LimitRange는 `min`, `max` 값을 기준으로 **request, limit을 스케줄링할 때 함께 검사**한다. 하지만 **스케줄링 이후**의 상황은 다르다 — 예를 들어 LimitRange의 Max CPU가 4코어로 설정되어 있고, 요청이 스파이크 쳐서 CPU 사용량이 그 값을 살짝 넘어갔다고 해도 **갑자기 파드를 종료시키지는 않는다.**

즉 LimitRange는 **범위를 검사하는 용도로만 사용**된다.
- 스케줄링할 때: 새로 생성되는 리소스가 이 범위를 검사한다.
- 이미 배포된 파드: LimitRange를 나중에 수정해도 기존 파드에는 자동으로 적용되지 않는다. 최댓값을 내려야 한다면 파드의 limit도 함께 수정해야 한다.

##### 주의: 네임스페이스에 여러 개의 LimitRange를 만들면 좋지 않다

기술적으로는 하나의 네임스페이스에 여러 개의 LimitRange를 만드는 것이 가능하다. 하지만 이 경우 **어떤 값이 기본으로 적용되는지 결정하기가 어려워진다.** 그래서 네임스페이스별로 **하나씩만 만들어서 사용하는 것이 좋다.**

##### 정리

| 항목 | 내용 |
|---|---|
| 한 줄 정리 | Namespace 안의 개별 컨테이너나 Pod에 리소스 기본값과 허용 범위를 적용하는 정책 |
| 적용 범위 | 특정 Namespace |
| 주요 대상 | Container, Pod, PVC |
| `min` | 개별 request의 최소 기준 |
| `max` | 개별 limit의 최대 기준 |
| `defaultRequest` | request를 생략했을 때 적용되는 기본값 |
| `default` | limit을 생략했을 때 적용되는 기본값 |
| `maxLimitRequestRatio` | limit과 request의 최대 비율 |
| 위반 시 | 생성 또는 수정 요청 거부 |
| 기존 Pod | LimitRange 생성·변경 시 자동으로 수정되지 않음 |
| ResourceQuota와의 관계 | ResourceQuota는 전체 총량, LimitRange는 개별 범위를 담당 |

#### | ConfigMap

**애플리케이션이 필요한 구성 데이터를 키-값 형태로 저장하여, 이를 Pod에 동적으로 주입하거나 환경 변수로 전달하는 쿠버네티스의 리소스**

##### 왜 필요한가

쿠버네티스는 동적으로 생성되는 파드들에게 설정 값을 전달해야 하는데, 그때 사용하는 리소스가 ConfigMap이다. `.env`나 `application.yml` 같은 설정 파일과 같은 맥락이지만, **쿠버네티스 리소스로 관리된다는 점**이 다르다.

쿠버네티스에서는 여러 서비스가 동작한다 (`payment-service`를 쓰는 Pod, `auth-service`를 쓰는 Pod 등). 두 서비스가 필요로 하는 env 값이 다르고, 게다가 **해당 파드가 어떤 EC2에 뜰지는 미리 알 수 없다.** 서비스가 30가지라면, 30가지의 env 파일이 있는데 모든 EC2에 모든 env 파일을 미리 뿌려둬야 하는 상황이 생긴다. → **쿠버네티스 리소스로 K-V Store 형식의 환경 변수를 담는 공간이 필요하고, 이것이 ConfigMap이다.**

```yaml
data:
  LOG_LEVEL: "info"
  SERVER_PORT: "3000"
  API_ENDPOINT: "https://api.example.com"
```

##### ConfigMap을 사용하는 이유

**1. 애플리케이션 코드와 실행 환경을 분리하기 위해서**
이건 ConfigMap 고유의 이유라기보다는, 코드와 환경 변수를 분리하는 일반적인 원칙에 가깝다.

**2. 설정을 중앙에서 관리할 수 있다**
쿠버네티스에서는 파드가 여러 노드에 흩어져서 배포된다. 원래는 설정 파일도 각 노드에 함께 퍼져야 하는데, 그러면 관리가 매우 힘들어진다. ConfigMap을 만들면 다음과 같은 흐름으로 동작한다.

```
ConfigMap 생성
   ↓ (YAML 해석)
Master Node → etcd에 key:value 형식으로 데이터 저장

Worker Node → Pod 생성 시
   kubelet → Kube API Server → etcd
```

워커 노드가 어느 시점에 어떤 파드를 띄우든, Master Node의 etcd에 쿼리만 하면 설정 값을 가져올 수 있다.

**3. 쿠버네티스 리소스라는 점**
파라미터 스토어나 GitHub Secrets 등으로 환경 변수를 관리하면 보통 **새롭게 배포할 때** 그 값이 적용되는데, 배포라는 프로세스 자체가 시간이 꽤 걸리고 실패 가능성과 롤백 대처까지 고려해야 하는 단순하지 않은 과정이다. ConfigMap의 **파일 마운트 방식**을 사용하면, 변경된 리소스를 애플리케이션이 감지하고 자동으로 reload하는 방식으로 처리할 수 있다.

##### ConfigMap에는 어떤 데이터를 저장할까?

- `LOG_LEVEL` — 로그 레벨 설정
- `SERVER_PORT`
- `SPRING_ACTIVE_PROFILE`
- `API_ENDPOINT`
- Nginx Conf 파일 — `default.conf` 전체를 저장하기도 함
- Redis Conf
- MySQL 설정 값

> 💡 공통점은 **민감하지 않은 데이터**를 저장한다는 것이다. `kubectl get configmap -o yaml`로 조회하면 값이 평문 그대로 보인다. 민감한 값은 ConfigMap이 아니라 **Secret**에 저장해야 한다.

##### ConfigMap 데이터 구조

데이터를 저장할 때는 `data`와 `binaryData` 두 가지 필드를 사용할 수 있다.

**data**

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  LOG_LEVEL: "info"
  SERVER_PORT: "3000"
```

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: nginx-config
data:
  default.conf: |
    server {
      listen 80;
      location / {
        proxy_pass http://backend;
      }
    }
```

> 💡 Conf 파일 전체를 통째로 가져오기보다는, 변경되어야 하는 부분만 쪼개서 관리하는 경우가 많다.

**binaryData**

보통 쓸 일이 많지 않다. UTF-8은 글자를 입출력하는 표준 양식인데, 이 문자열로 표현하기 힘든 값(아주 작은 이미지, UTF-8이 아닌 다른 문자열, 임의의 바이트가 들어있는 설정 등)을 **Base64 형식으로 저장**할 때 사용한다.

```
89 50 4E 47 ...
   ↓ Base64 인코딩
특정 문자열로 치환하여 저장
```

> ⚠️ ConfigMap의 최대 크기는 **1MiB**다. 이보다 커지면 볼륨(PV/PVC)이나 S3 같은 외부 저장소를 사용해야 한다.

##### ConfigMap 사용 방법

**1. 환경 변수 (일반적인 사용 방식)**

ConfigMap:
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  MESSAGE: "Hello from ConfigMap"
```

Pod (필요한 Key만 선택 — `env`):
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: configmap-pod
spec:
  containers:
    - name: express
      image: adapterz/k8s-configmap:latest
      env:
        - name: APP_MESSAGE
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: MESSAGE
```

Pod (ConfigMap 전체를 사용 — `envFrom`):
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: configmap-pod
spec:
  containers:
    - name: express
      image: adapterz/k8s-configmap:latest
      envFrom:
        - configMapRef:
            name: app-config
```

| 구분 | env | envFrom |
|---|---|---|
| 전달 범위 | 필요한 Key만 선택 | 모든 Key 전달 |
| 환경 변수 이름 | ConfigMap Key와 다르게 지정 가능 | 기본적으로 ConfigMap Key 사용 |
| 설정 가독성 | 어떤 값을 사용하는지 명확함 | 설정이 많을 때 간결함 |
| 권장 상황 | 일부 값만 필요한 경우 | ConfigMap 전체를 사용하는 경우 |

**2. 파일 마운트**

환경 변수 vs 파일 마운트

| 구분 | 환경 변수 | 파일 마운트 |
|---|---|---|
| 적합한 설정 | 짧고 단순한 값 | 여러 줄의 설정 파일 |
| 사용 예 | 포트, 로그 레벨, API 주소 | Nginx, Redis, 애플리케이션 설정 파일 |
| 읽는 방식 | `process.env`, `System.getenv()` | 파일 읽기 |
| 변경 반영 | 실행 중인 컨테이너에는 자동 반영되지 않음 | 마운트된 파일은 일정 시간이 지나면 갱신될 수 있음 |
| 애플리케이션 처리 | 재시작 후 새 값을 읽음 | 파일을 다시 읽는 기능이 필요할 수 있음 |

> ⚠️ 파일 마운트로 연결된 볼륨은 실제로 갱신되긴 하지만, **볼륨이 갱신되었다고 해서 파드 내부에서 실행 중인 값이 즉시 적용되는 것은 아니다.** 애플리케이션이 명시적으로 reload를 호출하거나, 해당 볼륨 내용을 다시 읽는 로직이 있어야 새 값이 반영된다.
>
> 이 감지는 **kubelet이** 담당한다. ConfigMap이 수정되면 etcd에 저장되고, kubelet이 API Server에 질의해서 자신의 노드에 있는 파드들이 사용하는 ConfigMap과 대조하는 **Polling(특정 시간마다 질의하는 행위)** 방식으로 값을 반영한다.

ConfigMap도 Namespace에 속해 있으므로, **환경별로 서로 다른 ConfigMap을 사용**할 수 있다.

##### ConfigMap 말고 다른 방법이 없나요?

1. **Pod YAML에 직접 기입** — 관리하기 힘들다.
2. **도커 컨테이너 이미지에 값을 붙이기** — 설정값이 바뀌면 도커 이미지를 재빌드해야 하고, 배포도 다시 해야 한다.
3. **외부 볼륨에 다 관리** — 대용량 설정 파일이 필요할 때만 사용하는 경향이 있다.
4. **외부 설정 관리 시스템 사용** (Spring Cloud Config, AWS Systems Manager Parameter Store 등) — 단, 쿠버네티스는 특정 CSP에 종속되지 않으려는 방향성이 있는데, 이런 시스템은 그 방향성과는 다소 부딪히는 지점이 있다.

##### 정리

- ConfigMap은 일반 설정값에 사용한다.
- 환경 변수 또는 파일 형태로 Pod에 전달할 수 있다.
- 동일한 컨테이너 이미지를 여러 환경에서 재사용할 수 있게 해준다.
- Kubernetes API 오브젝트이므로 일반적으로 API Server를 통해 etcd에 저장된다.
- Pod가 etcd에 직접 접근하는 구조는 아니다.
- 환경 변수로 전달된 값은 실행 중 자동으로 변경되지 않는다.
- 볼륨으로 전달한 파일은 일정 시간이 지나면 갱신될 수 있지만, 애플리케이션이 다시 읽어야 실제 동작에 반영된다.
- 비밀번호, API Key, 토큰과 같은 민감한 정보는 ConfigMap이 아니라 Secret을 사용해야 한다.
- 간단한 설정은 Pod YAML에 직접 작성할 수 있지만, 공유·환경 분리·중앙 관리가 필요하면 ConfigMap이 적합하다.

---

#### | Secret

**민감한 데이터를 안전하게 저장하고 Pod에 전달하기 위해 설계된 키-값 쌍 기반의 리소스**

애플리케이션이 사용하는 데이터는 크게 둘로 나뉜다.
- 일반 설정 값 → ConfigMap
- 민감한 설정 값 → Secret

예를 들어 데이터베이스에 접근한다고 하면, 데이터베이스 주소·포트·사용자 이름은 ConfigMap에, **비밀번호는 Secret**에 저장한다.

> 🤔 "어차피 ConfigMap에 있는 것도 다 Secret으로 가면 되는 거 아닌가요?" — LOG_LEVEL, PORT_NUMBER, DB_PASSWORD를 전부 다 Secret에 넣어버리면, 운영자가 이 목록만 보고 어떤 정보가 실제로 민감한 정보인지, 어떻게 다뤄야 하는지를 제대로 구분하지 못할 가능성이 커진다. ConfigMap과 Secret을 나누고, 접근 권한·ServiceAccount·RBAC 등을 다르게 설정함으로써 **정보를 다루는 방식 자체를 다르게** 만드는 것이 목적이다.

##### Secret을 사용하는 이유

민감한 값을 애플리케이션 코드와 배포 설정으로부터 분리하여, 필요한 Pod에만 전달하기 위해서다.
- 데이터베이스 비밀번호
- 외부 API Key
- JWT 서명 키
- TLS 인증서와 개인 키

##### Secret 기본 구조

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: database-secret
type: Opaque
data:
  DB_USERNAME: YWRtaW4=
  DB_PASSWORD: cGFzc3dvcmQxMjM0
```

| 필드 | 설명 |
|---|---|
| `metadata.name` | Secret 이름 |
| `type` | Secret의 용도와 데이터 형식 |
| `data` | Base64로 표현된 Key-Value 데이터 |
| `stringData` | 평문 문자열을 편리하게 입력하기 위한 필드 |

##### data vs stringData

- **data** — 사용자가 값을 직접 Base64로 인코딩해서 넣는 경우.
- **stringData** — 평문으로 값을 작성하면, 실제로 저장될 때 Base64로 변환되어 저장되는 방식.

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: database-secret
type: Opaque
stringData:
  DB_USERNAME: admin
  DB_PASSWORD: password1234
```

##### Secret은 사실 완전한 "비밀"이 아니다..?

Secret의 저장소도 결국 **etcd**다. 그런데 etcd는 기본적으로 **암호화 저장 기능이 기본값(디폴트)으로 켜져 있지 않다.** 즉 Secret 값을 만들면 → Base64로 인코딩 → etcd에 **그냥 저장**되는데, 이 자체로는 암호화가 되지 않는다.

> ⚠️ 진짜 "시크릿"답게 사용하려면 **Encryption at Rest**(저장 시 암호화) 같은 암호화 설정을 별도로 걸어줘야 한다.

##### Secret의 타입

**1. Opaque** — 일반적인 시크릿 타입. 특별한 형식 없이 K-V 데이터를 저장한다 (데이터베이스 비밀번호, API Key, JWT 서명 키 등).

**2. kubernetes.io/tls** — TLS 인증서와 개인 키를 저장할 때 사용한다. 보통 Ingress나 Reverse Proxy 리소스 쪽에서 사용된다.

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: tls-secret
type: kubernetes.io/tls
data:
  tls.crt: <Base64 인증서>
  tls.key: <Base64 개인 키>
```

**3. kubernetes.io/dockerconfigjson** — Private Registry에 접근할 때 필요한 인증 정보를 담는 타입이다.

```yaml
type: kubernetes.io/dockerconfigjson

# Pod에서 사용할 때는
spec:
  imagePullSecrets:
    - name: registry-secret
```

##### 전달하는 방법

1. 환경 변수 방식
2. 파일 마운트 방식

(ConfigMap과 동일한 방식으로 Pod에 전달할 수 있다.)

##### Secret의 한계 지점

1. **Base64는 암호화가 아니다** — 인코딩/디코딩 기능일 뿐, 보안 기능이 아니다.
2. **etcd 보안 설정을 추가로 해야 한다** — 저장 시 암호화를 별도로 설정해야만 데이터가 실질적으로 보호된다.
3. **쿠버네티스 접근 권한 관리가 중요해진다** — Secret을 조회할 수 있는 권한만 있으면 `kubectl get secret database-secret -o json | base64 --decode`처럼 평문 값이 그대로 노출된다.
4. **비밀번호 교체(Rotation) 자동화가 어렵다** — 현대 웹 환경에서는 민감한 보안 정보를 특정 주기마다 교체(Rotation)하는 것이 권장되는데(예: AWS KMS를 사용하면 추가 비용이 들지만 지원됨), 순수한 쿠버네티스 Secret만으로는 자동 교체가 되지 않는다.

##### 실무에서는 Secret을 어떻게 사용할까?

보통은 **Vault**를 사용한다. Secret 리소스 자체를 그대로 사용하는 경우도 있긴 하지만, 다소 "굳이"에 가까운 선택이다. 대신 **External Secrets Operator**처럼, 내부의 Secret을 그 자체로 관리하기보다는 **외부의 시크릿 관리 시스템과 연동하는 오퍼레이터**를 사용하는 경우가 많다.

```
AWS Secrets Manager / Vault
        ↓
External Secrets Operator
        ↓
Kubernetes Secret
        ↓
      Pod
```

```yaml
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: database-external-secret
spec:
  secretStoreRef:
    name: aws-secret-store
    kind: SecretStore
  target:
    name: database-secret
  data:
    - secretKey: DB_PASSWORD
      remoteRef:
        key: production/database
        property: password
```

##### 쿠버네티스 생태계에 대한 짧은 메모

쿠버네티스는 자체 생태계가 상당히 크고 발전되어 있어서 실무적으로 많이 활용된다. 지금 배우는 Namespace, ResourceQuota, LimitRange, ConfigMap, Secret 등은 어디까지나 **기초 도구**에 해당한다. 쿠버네티스는 API가 인터페이스로 공개되어 있어서(예: CNI도 하나의 인터페이스다) 기본적으로 **써드파티 생태계가 매우 활성화**되어 있다.

| 영역 | 대표 도구 |
|---|---|
| 배포 | k3s, kubespray |
| Key Management | Vault (KMS는 특정 CSP에 종속되기 때문에 다소 꺼려지기도 함) |
| 테스트 | 카오스 엔지니어링(Chaos Engineering)용 도구들 |
| Networking | Mesh 관련 도구들 |

##### 정리

- ConfigMap은 일반 설정값에 사용한다.
- Secret은 비밀번호, API Key, 인증서처럼 민감한 값에 사용한다.
- **Secret의 Base64는 암호화가 아니라 인코딩이다.**
- Secret도 API Server를 통해 etcd에 저장된다.
- etcd 저장 시 암호화는 클러스터에서 별도로 설정해야 한다.
- 환경 변수 또는 파일 형태로 Pod에 전달할 수 있다.
- Secret을 변경해도, 이미 환경 변수로 사용 중인 Pod가 자동으로 갱신되지는 않는다.
- Secret YAML을 Base64 상태로 Git에 커밋해서는 안 된다.
- RBAC, 저장 시 암호화, 로그 관리, 정기 교체가 함께 필요하다.
- 실무에서는 Vault, AWS Secrets Manager와 같은 외부 관리 시스템을 사용할 수 있다.
- External Secrets Operator는 외부 값을 Kubernetes Secret으로 동기화해주는 역할을 한다.

### 🗨️ 오늘의 회고

- 오늘은 Namespace를 시작으로, 그 위에서 동작하는 정책들(ResourceQuota, LimitRange)과 설정 관리 리소스들(ConfigMap, Secret)까지 이어서 배웠는데, 전체적으로 "여러 팀·여러 환경이 하나의 클러스터를 안전하게 나눠 쓰는 방법"이라는 하나의 큰 흐름으로 연결됐다.
- Namespace가 "클러스터를 나누는 것이 아니다"라는 전제를 명확히 하고 시작한 게 오늘 이해에 큰 도움이 됐다. 특히 **"네임스페이스만으로는 자원도, 네트워크도, 접근 권한도 완전히 격리되지 않는다"**는 세 가지 오해를 짚어준 부분이 인상 깊었다 — 논리적 구분과 실제 격리는 다르다는 걸 명확히 인지하게 됐다.
- ResourceQuota와 LimitRange의 관계를 "회사 전체 예산 vs 개인별 결제 한도"로 비유한 부분이 특히 이해가 쉬웠다. ResourceQuota만 있으면 파드 하나가 네임스페이스 전체 자원을 독점할 수 있다는 예시를 보고, 왜 이 둘을 항상 함께 쓰는지 납득이 갔다. 그리고 두 리소스 모두 "실제 사용량"이 아니라 "선언된 requests/limits 값"을 기준으로 동작한다는 공통점도 기억해둬야겠다.
- ConfigMap과 Secret 파트에서는 **"Secret의 Base64는 암호화가 아니라 인코딩"**이라는 사실이 가장 충격적이면서도 중요한 포인트였다. etcd 자체가 기본적으로 암호화 저장이 켜져 있지 않다는 것도 처음 알았다. 실무에서는 순수 Secret보다 Vault나 External Secrets Operator 같은 외부 시스템과 연동해서 쓰는 경우가 많다는 정리를 통해, "쿠버네티스 기본 리소스 = 완전한 보안 솔루션"이 아니라는 걸 명확히 인식하게 됐다.
- 오늘 배운 개념들이 결국 "장애 격리"와 "자원 오남용 방지"라는 실무적인 문제의식에서 출발한다는 걸 느꼈다. WAS Pod가 무한 증식하며 DB 데드락을 악화시키는 예시처럼, 단순히 이론이 아니라 실제로 벌어질 수 있는 장애 시나리오와 함께 설명해준 덕분에 각 리소스의 존재 이유가 훨씬 잘 와닿았다.
