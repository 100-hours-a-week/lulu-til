## 1. Docker란?

도커는 애플리케이션을 신속하게 구축, 테스트 및 배포할 수 있는 소프트웨어 플랫폼이다. 소프트웨어를 **컨테이너**라는 표준화된 유닛으로 패키징하며, 이 컨테이너에는 라이브러리, 시스템 도구, 코드, 런타임 등 소프트웨어를 실행하는 데 필요한 모든 것이 포함되어 있다. 도커를 사용하면 환경에 구애받지 않고 애플리케이션을 신속하게 배포 및 확장할 수 있다.

### Docker를 사용해야 하는 이유

| 이점 | 설명 |
|---|---|
| **더 빠른 소프트웨어 제공** | Docker 사용자는 평균적으로 비사용자보다 7배 더 많은 소프트웨어를 제공한다 |
| **운영 표준화** | 작은 컨테이너식 애플리케이션으로 손쉽게 배포하고, 문제 파악 및 롤백이 가능하다 |
| **원활한 이전** | 로컬 개발 환경에서 AWS 프로덕션 배포로 원활하게 이전할 수 있다 |
| **비용 절감** | 서버당 더 많은 코드를 실행하여 사용률을 높이고 비용을 절감할 수 있다 |

### Docker를 사용하는 경우

- **마이크로서비스**: 표준화된 코드 배포를 활용하여 분산 애플리케이션 아키텍처를 구축하고 확장한다.
- **CI/CD**: 환경을 표준화하고 언어 스택 및 버전 간의 충돌을 제거하여 더 빠르게 배포한다.
- **데이터 처리**: 빅 데이터 처리를 서비스로서 제공하고, 이동식 컨테이너로 패키징한다.
- **서비스형 컨테이너**: 안전한 IT 관리형 인프라와 콘텐츠로 분산 애플리케이션을 구축 및 제공한다.

### Docker vs 가상 머신

가상 머신은 서버 **하드웨어**를 가상화하는 반면, 컨테이너는 서버의 **운영 체제**를 가상화한다. Docker는 컨테이너를 위한 운영체제로, 각 서버에 Docker Engine을 설치하여 컨테이너를 구축, 시작 또는 중단하는 간단한 명령 세트를 제공한다.

## 2. Docker Image

### Docker 이미지란?

Docker 이미지는 **컨테이너 생성 지침이 포함된 읽기 전용 템플릿**이다. 컨테이너 내에서 애플리케이션을 실행하는 데 필요한 라이브러리 및 종속성의 스냅샷으로, 공유 및 이동이 가능하다. 레지스트리(예: Docker Hub)에 이미지를 저장하여 복잡한 소프트웨어 아키텍처와 버전을 추적할 수 있다.

### Docker 이미지 vs Docker 컨테이너

| 구분 | Docker 이미지 | Docker 컨테이너 |
|---|---|---|
| **정의** | 컨테이너 실행을 위한 읽기 전용 템플릿 | 독립적이고 실행 가능한 소프트웨어 애플리케이션 |
| **소스** | Dockerfile로부터 생성 | Docker 이미지로부터 생성 |
| **구성** | 읽기 전용 이미지 계층으로 구성 | 이미지 계층 + 최상위 쓰기 가능한 컨테이너 계층 |
| **가변성** | 변경 불가 (수정 시 새 이미지 생성 필요) | 변경 가능 (런타임 중 수정 가능) |
| **수명** | 공유/재사용을 위해 저장 | 애플리케이션 수명 주기 동안 생성 및 폐기 |

## 3. Docker 이미지 크기 문제

### 이미지 크기란?

Docker 이미지 크기는 이미지를 저장하고 배포하는 데 필요한 **총 저장 공간량**이다. 기본 이미지의 크기, 빌드 과정에서 추가된 레이어, 추가 파일 또는 종속성이 모두 포함된다.

### 이미지가 커지면 발생하는 문제

| 문제 | 설명 |
|---|---|
| **빌드 시간 증가** | 레이어가 많고 무거울수록 빌드 속도가 느려진다 |
| **배포 속도 저하** | 이미지를 레지스트리에서 Pull하는 시간이 길어진다 |
| **저장 비용 증가** | 저장 공간과 네트워크 대역폭 사용이 늘어난다 |
| **보안 취약점 증가** | 불필요한 패키지가 많을수록 공격 표면(attack surface)이 넓어진다 |
| **성능 저하** | 애플리케이션 시작 시간 및 응답 속도에 부정적 영향을 미친다 |

### 이미지 크기에 영향을 미치는 요소

- **기본 이미지**: `ubuntu`, `alpine`, `scratch` 등 선택에 따라 크기가 크게 달라진다.
- **설치된 패키지**: 패키지의 수와 크기가 전체 크기를 증가시킨다.
- **불필요한 파일**: 실행에 필요하지 않은 파일이나 디렉토리가 이미지를 부풀린다.
- **이미지 레이어**: 레이어 수와 각 레이어의 변경량이 전체 크기에 영향을 미친다.
- **캐싱 비효율**: 캐싱을 효율적으로 사용하지 않으면 이미지 크기가 커질 수 있다.

## 4. Docker 이미지 최적화 전략

### 전략 1. 더 작은 베이스 이미지 선택

베이스 이미지 선택은 이미지 크기에 가장 큰 영향을 미치는 결정이다. `alpine`이나 `scratch` 같은 경량 이미지를 사용하면 크기를 크게 줄일 수 있다.

```dockerfile
## ❌ 더 큰 기본 이미지 사용
FROM ubuntu:22.04
RUN apt-get update && apt-get install -y python3

## ✅ 더 작은 기본 이미지 사용
FROM alpine:3.16
RUN apk add --no-cache python3
```

### 전략 2. 레이어 수 최소화

Dockerfile의 `RUN`, `COPY`, `ADD` 명령어는 각각 새로운 레이어를 생성한다. 명령어를 하나로 합치면 레이어 수를 줄일 수 있다.

```dockerfile
## ❌ 여러 레이어 생성
RUN apt-get update
RUN apt-get install -y python3
RUN pip3 install flask

## ✅ 하나의 레이어로 합치기
RUN apt-get update \
    && apt-get install -y python3 \
    && pip3 install flask
```

### 전략 3. 멀티스테이지 빌드 활용

멀티스테이지 빌드는 단일 Dockerfile에서 여러 `FROM` 문을 사용하여 **빌드 환경과 런타임 환경을 분리**한다. 빌드에만 필요한 도구(컴파일러, 빌드 패키지 등)가 최종 이미지에 포함되지 않으므로 이미지 크기를 크게 줄일 수 있다.

```dockerfile
## ✅ 멀티스테이지 빌드
# 1단계: 빌드 스테이지
FROM ubuntu:22.04 AS builder
RUN apt-get update && apt-get install -y build-essential
COPY . /app
RUN cd /app && make

# 2단계: 프로덕션 스테이지 (빌드 산출물만 복사)
FROM ubuntu:22.04
COPY --from=builder /app/bin /app/bin
CMD ["/app/bin/myapp"]
```

### 전략 4. 불필요한 파일 제거

패키지 설치 후 남은 캐시와 임시 파일을 같은 `RUN` 명령어 내에서 제거해야 한다. (별도 레이어로 제거하면 이전 레이어에 파일이 남아 효과가 없다.)

```dockerfile
## ✅ 패키지 캐시 및 임시 파일 제거
FROM ubuntu:22.04
RUN apt-get update \
    && apt-get install -y python3 \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*
```

### 전략 5. 캐시 효율적으로 활용

변경이 잦은 파일(소스 코드)은 뒤에, 변경이 드문 파일(의존성 목록)은 앞에 배치하여 캐시 히트율을 높인다.

```dockerfile
## ✅ 캐시를 고려한 Dockerfile 명령어 순서
FROM ubuntu:22.04

# 의존성 파일만 먼저 복사 → 의존성이 바뀌지 않으면 캐시 재사용
COPY requirements.txt .
RUN pip3 install -r requirements.txt

# 소스 코드는 나중에 복사 (자주 바뀌는 파일)
COPY . /app
CMD ["python3", "/app/app.py"]
```

### 전략 6. .dockerignore 파일 활용

불필요한 파일이 빌드 컨텍스트에 포함되지 않도록 `.dockerignore` 파일을 작성한다.

```
# 대용량 데이터셋 제외
data/

# 가상 환경 제외
venv/

# 캐시 및 임시 파일 제외
__pycache__/
*.log
*.tmp
*.pyc
*.pyo
.pytest_cache

# Git 관련 파일 제외
.git
.gitignore
README.md

# 모델 학습 체크포인트 및 로그
checkpoints/
runs/
```

## 5. 고급 최적화 기법

### Scratch 이미지

운영체제가 포함되지 않은 **완전히 빈 이미지**다. 필요한 파일을 직접 추가해야 하므로 가장 도전적이지만, 가장 작은 이미지 크기를 만들 수 있다.

**유용한 경우:**
- 독자적인 리눅스 배포판을 만들 때
- PyInstaller 등으로 컴파일된 독립 실행형 애플리케이션을 배포할 때

```dockerfile
# syntax=docker/dockerfile:1
FROM scratch
ADD myapp /
CMD ["/myapp"]
```

### Distroless 이미지

Google이 제공하는 이미지로, Scratch와 전체 OS 이미지의 **중간 형태**다. 필수 구성 요소만 포함하여 불필요한 부분을 제거하면서도 SSL 인증서, 타임존 데이터 등은 포함하고 있어 실용적이다.

```dockerfile
FROM gcr.io/distroless/python3-debian10

COPY --from=builder /app/dist/main /app/main
COPY --from=builder /app/model /app/model
COPY --from=builder /app/config.yml /app/config.yml

ENTRYPOINT ["/app/main"]
```

### Docker BuildKit 활성화

Docker BuildKit을 활성화하면 더 빠르고 안전한 빌드가 가능하다.

```bash
DOCKER_BUILDKIT=1 docker build -t myapp .
```

### 이미지 분석 도구 활용

- **Dive**: 이미지 내부 레이어를 시각적으로 분석하여 불필요한 파일을 찾는다.
- **Docker Slim**: 이미지를 자동으로 분석하고 최소화해준다.
- **Trivy**: 이미지 내 보안 취약점을 스캔한다.

```bash
# Dive 사용
dive my-docker-image

# Docker Slim 사용
docker-slim build my-docker-image

# 보안 취약점 스캔 (Trivy)
docker scan my-image:tag
```

### 보안 강화 모범 사례

```dockerfile
# 루트 사용자 대신 일반 사용자로 실행
RUN adduser --disabled-password --gecos "" appuser
USER appuser

# 네트워크 노출 최소화 (localhost에만 바인딩)
# docker run -p 127.0.0.1:8080:8080 myimage
```

## 6. 실제 최적화 사례 — 1.2GB → 8MB

### 대상

Python 기반 머신러닝 애플리케이션 (초기 이미지 크기: **1.2GB**)

### 기존 Dockerfile (비효율적)

```dockerfile
FROM python:3.9
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "main.py"]
```

**문제점:**
- 전체 Python 이미지 사용 (불필요한 도구 다수 포함)
- 빌드 도구 및 종속성이 최종 이미지에 그대로 포함
- 비효율적인 레이어 캐싱
- 불필요한 파일이 포함될 가능성

### 최적화된 Dockerfile (멀티스테이지 빌드 적용)

**1단계 — 빌드 스테이지**: 빌드에 필요한 모든 도구를 설치하고, PyInstaller로 독립 실행 파일 생성

```dockerfile
FROM python:3.9-slim AS builder

# 필수 빌드 종속성 설치
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential \
    gcc \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app

# 의존성 설치 (캐시 활용을 위해 먼저 복사)
COPY requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt

# 소스 코드 복사
COPY . .

# 모델 컴파일
RUN python compile_model.py

# PyInstaller로 독립 실행 파일 생성
RUN pip install pyinstaller
RUN pyinstaller --onefile inference.py
```

**2단계 — 프로덕션 스테이지**: scratch 이미지에 필요한 파일만 복사

```dockerfile
FROM scratch

WORKDIR /app

# 빌드 스테이지에서 필요한 파일만 복사
COPY --from=builder /app/dist/inference /app/inference
COPY --from=builder /app/model /app/model

ENTRYPOINT ["/app/inference"]
```

### 레이어 최적화 추가 적용

```dockerfile
## ❌ 비효율적인 예제 (각각 별도 레이어 생성)
RUN apt-get update
RUN apt-get install -y python3-pip python3-dev
RUN pip install numpy pandas
RUN apt-get clean

## ✅ 최적화된 예제 (하나의 레이어로 합산)
RUN apt-get update && apt-get install -y python3-pip python3-dev && \
    pip install numpy pandas && \
    apt-get clean && rm -rf /var/lib/apt/lists/*
```

### 최적화 결과

| 항목 | 최적화 전 | 최적화 후 | 개선율 |
|---|---|---|---|
| **이미지 크기** | 1.2 GB | 8 MB | 99.33% 감소 |
| **배포 시간** | 10초 | 1.5초 | 85% 감소 |
| **클라우드 비용** | 100% | 40% | 60% 절감 |

## 7. 결론 : 이미지 최적화 전략 정리

> 도커 이미지가 커지면 빌드 시간 증가, 배포 속도 저하, 저장 비용 증가, 보안 취약점 확대 등 복합적인 문제가 발생한다. 이를 해결하기 위한 핵심 전략은 **불필요한 것을 최대한 제거하고, 빌드 환경과 런타임 환경을 분리**하는 것이다.

| 전략 | 핵심 내용 | 효과 |
|---|---|---|
| **최소 베이스 이미지** | alpine, slim, distroless, scratch 사용 | 기본 용량 대폭 감소 |
| **멀티스테이지 빌드** | 빌드 환경과 런타임 환경 분리 | 빌드 도구가 최종 이미지에 미포함 |
| **레이어 최소화** | RUN 명령어를 하나로 합치기 | 불필요한 레이어 생성 방지 |
| **캐시 활용** | 변경 빈도 낮은 파일을 먼저 복사 | 재빌드 속도 향상 |
