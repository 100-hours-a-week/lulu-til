#### | Kubernetes (k8s)

**컨테이너화된 애플리케이션을 어디서나 배포, 확장, 관리할 수 있는 오픈소스 시스템**

여러 서버에서 실행되는 많은 컨테이너를 자동으로 배치하고, 감시하고, 복구하고, 확장하기 위한 **컨테이너 오케스트레이션 플랫폼**이다.

##### 쿠버네티스가 하는 일

- 컨테이너를 어느 서버에 배치할지 결정
- 필요한 수만큼 컨테이너 실행
- 장애가 발생한 컨테이너 복구
- 서버와 컨테이너 상태 감시
- 트래픽이 증가하면 컨테이너(파드)를 확장
- 애플리케이션의 실행 상태를 유지
- 클러스터 내에서 컨테이너화된 애플리케이션이 서로를 자동으로 발견하고 통신할 수 있도록 **서비스 디스커버리 및 로드밸런싱** 기능 제공

> 💡 **k8s**는 kubernetes의 줄임말이다 (k와 s 사이에 문자가 8개라서 k8s). 쿠버네티스는 명령어가 긴 편이라 실무에서는 `kgp`, `kgn`, `kap` 처럼 축약해서 쓰는 경우가 많다.

#### | 쿠버네티스는 무슨 문제를 해결하나?

그동안 애플리케이션이 어떻게 실행되어 왔는지를 단계별로 살펴보면 쿠버네티스가 왜 필요한지 이해할 수 있다.

##### 1단계 — 물리 서버

하나의 물리 서버에 프론트엔드, 백엔드, DB를 모두 설치하고 웹서버를 붙여 프로그램을 동작시켰다.

**문제점**
- 하나의 애플리케이션이 느려지면 같은 서버의 다른 애플리케이션도 함께 느려진다.
- 애플리케이션마다 필요한 라이브러리가 달라 **충돌 가능성**이 있다.
- 서버 한 대에 장애가 나면 전체 서비스가 죽어 **가용성이 떨어진다**.
- 물리 서버는 확장하기도 어렵다.

##### 2단계 — 가상 머신 (VM)

물리 서버의 자원을 가상 머신 단위로 나눴다. 이걸 클라우드에 띄운 것이 AWS 같은 서비스다.

```
물리 서버
└── Hypervisor
    ├── VM 1
    │     ├── Guest OS
    │     └── Application
    ├── VM 2
    │     ├── Guest OS
    │     └── Application
    └── VM 3
          ├── Guest OS
          └── Application
```

가상 머신을 사용하면 애플리케이션마다 독립적인 운영체제를 쓸 수 있어 **애플리케이션 간 격리**, **자원 사용량 분리**가 가능해지고, 물리 서버 하나를 여러 VM으로 쪼갤 수 있어 **확장성**도 보장된다.

**아직 남은 문제**
- VM마다 운영체제가 필요하고, 운영체제는 자원을 많이 필요로 해 **무겁다**.
- VM 이미지 크기가 크고, 운영체제를 함께 실행시키기 때문에 **초기 실행 시간이 느리다**.
- 운영체제 자체도 관리 포인트가 된다 (예: 보안 패치 — 윈도우 서버가 특히 이 문제가 심함).

##### 3단계 — 컨테이너

컨테이너는 VM처럼 운영체제를 따로 만들지 않고, **호스트 운영체제의 커널을 공유**해서 사용한다.

```
서버 또는 VM (EC2)
└── Host OS (ubuntu)
      └── Container Runtime (Docker)
            ├── Frontend Container
            ├── Backend Container
            └── Batch Container
```

**컨테이너의 장점**
- 운영체제가 빠져서 VM보다 실행 속도가 빠르다.
- 이미지가 가볍다 — 실행 환경 자체를 하나의 작은 이미지로 만들고, 그 이미지를 통째로 옮겨 사용할 수 있다 (CI/CD에서 Registry에 이미지를 올리고 그대로 받아 쓰는 방식).

**그런데 도커(컨테이너)로도 해결되지 않는 문제가 있다.** 도커는 컨테이너를 쉽게 실행해주지만, **많은 컨테이너를 여러 서버에서 "운영"하는 문제는 해결하지 못한다.**

개발자 로컬 PC에서는 FE, BE, DB 정도를 Docker Compose로 편하게 띄울 수 있다. 하지만 서비스가 커져서 FE 30개, BE 50개, 인증 서버 10개처럼 **100개가 넘는 컨테이너**가 돌아간다면, 사람이 수동으로 관리할 때 다음 문제들이 생긴다.

**문제 1. 컨테이너가 죽으면 어떻게 다시 실행하지?**
컨테이너는 애플리케이션 에러, 메모리 부족, 네트워크 장애, 잘못된 명령어 입력 등 다양한 이유로 종료될 수 있다. 컨테이너 수가 많아지면 **어떤 컨테이너가 어디서 왜 죽었는지 파악하기조차 힘들다.** 그래서 다음 기능이 필요하다.
- 컨테이너 상태를 지속적으로 확인
- 컨테이너가 종료되면 자동으로 재시작
- 서버가 비정상이면 다른 서버에서 실행
- 항상 필요한 수의 컨테이너를 유지

**문제 2. 어느 서버에서 컨테이너를 실행하지?**
35대의 서버에서 173개의 컨테이너가 돌아가는 상황에서 백엔드 요청이 급증해 컨테이너를 늘려야 한다면, 어디에 배치해야 할까?

| 서버 | CPU 사용률 | 메모리 사용률 |
|---|---|---|
| Server A | 85% | 80% |
| Server B | 30% | 40% |
| Server C | 55% | 60% |

단순히 "B가 낫지 않나?"라고 판단하면 안 된다. **B의 사용률이 낮은 이유가 네트워크 장애 때문이라면?** 컨테이너를 새 서버에 넣는 결정은 여러 조건을 고려해야 한다.
- 해당 서버의 유휴 자원이 얼마나 있는지
- 컨테이너가 필요한 자원이 얼마인지
- 특정 서버에서 실행하면 안 되는 이유(네트워크 이슈, 가용성 이슈)가 있는지

필요한 기능: 서버 자원 실시간 확인, 컨테이너 자원 요구사항 확인, 조건에 맞는 서버 자동 선택, 서버별 컨테이너 적절히 분산.

**문제 3. 특정 서버가 장애를 일으키면?**
A 서버(FE, BE, DB), B 서버(FE, BE, DB), C 서버(BE) 구성에서 B 서버가 죽으면, B에 있던 컨테이너들에는 더 이상 요청이 릴레이되지 않고 응답도 오지 않는다. 하지만 서비스는 여전히 **똑같은 처리량**을 처리해야 한다. 이를 위해서는 해당 서버가 처리하던 컨테이너의 수와 종류, 현재 자원 가용량을 파악해서 **재배치**해야 하며, **사용할 수 없는 서버를 감지**하고 **전체 서비스의 워크로드를 분산**하는 기능이 필요하다.

##### 쿠버네티스가 해결하는 것들

1. 컨테이너를 자동으로 실행·배치
2. 클러스터 내 서버의 자원 사용량을 실시간 모니터링
3. 컨테이너가 사용 중인 자원량도 실시간 모니터링
4. 서버 자원 상태에 맞춰 필요한 컨테이너를 자동 배치
5. 특정 서버에 장애가 나면 필요한 컨테이너를 자동 재배치
6. 필요한 처리량에 따라 컨테이너를 자동으로 늘리거나 줄임
7. 롤링 업데이트도 지원
8. 여러 서버(35대든 그 이상이든)에 일일이 접속하지 않고 **중앙에서 관리** 가능

→ 이것이 바로 **컨테이너 오케스트레이션**이다.

> 💡 **컨테이너 오케스트레이션**이란 여러 서버에서 실행되는 많은 컨테이너를 배치하고, 연결하고, 감시하고, 복구하는 작업을 자동화하는 것이다. ("오케스트라"에서 유래한 단어로, 여러 악기를 편성하고 지휘하는 것에 비유된다.) **Docker는 컨테이너를 만들고 실행하는 도구**이고, **쿠버네티스는 여러 서버에서 많은 컨테이너를 관리하는 도구**라는 차이가 있다.

#### | 쿠버네티스 핵심 개념 3가지

##### 1. 선언적 관리 (Declarative Management)

사용자가 작업 방법을 하나씩 명령하는 것이 아니라, **최종적으로 원하는 상태를 정의**하는 방식이다.

| 구분 | 방식 |
|---|---|
| **명령형 관리** | 운영자가 서버에 접속해서 필요한 명령을 순서대로 수행 (서버 접속 → 컨테이너 실행 → 상태 확인 → 실패 시 대응…) |
| **선언형 관리** | 운영자는 "나는 백엔드 애플리케이션이 3개 실행되면 좋겠어"처럼 원하는 상태만 정의. 쿠버네티스가 알아서 2개면 1개를 늘리고, 4개면 1개를 줄인다 |

##### 2. 원하는 상태 (Desired State)

사용자가 쿠버네티스에게 "나는 이 상태를 원해"라고 선언하는 것. 예: 애플리케이션 3개 실행, 각 앱은 500MB 메모리 사용, 지정한 컨테이너 이미지 사용.

이에 대응하는 개념이 **현재 상태**다 — 실제로 쿠버네티스 내에서 실행되고 있는 상태(예: 애플리케이션 2개 실행 중, 한 서버가 장애, 컨테이너 하나가 죽음). 쿠버네티스는 **원하는 상태와 현재 상태를 지속적으로 비교**한다.

##### 3. 지속적인 상태 조정 (Reconciliation)

쿠버네티스는 현재 상태를 관찰하고 원하는 상태와 다르면 작업을 수행하는데, 이 사이클을 계속 반복한다.

```
1. 원하는 상태 확인
2. 현재 상태 확인
3. 두 상태 비교
4. 차이가 있으면 수정
5. 다시 상태 확인
   (1번으로 반복)
```

이 반복적인 조정(Reconciliation Loop)이 **쿠버네티스 자동화의 핵심**이다.

#### | 쿠버네티스 요소

##### 1. 클러스터 (Cluster)

쿠버네티스가 관리하는 여러 서버, 컨테이너, 리소스의 집합. 쿠버네티스는 여러 서버를 하나로 묶어서 관리한다.

```
Kubernetes Cluster
├── Control Plane
├── Worker Node 1
├── Worker Node 2
└── Worker Node 3
```

클러스터를 구성하는 각 서버를 **Node**라고 부른다. 노드는 물리 서버, 가상 머신, EC2, 온프레미스 서버, 다른 클라우드의 VM 등이 될 수 있으며, 한 클러스터 안에 이런 노드 유형들이 혼합될 수도 있다(EC2로만 구성된 클러스터도 있고, 물리서버 + EC2 + GCP CE가 섞인 클러스터도 있다).

> 💡 일반적으로는 운영 복잡도를 낮추기 위해 하나의 노드 구성을 유지하는 경향이 있다. 다만 글로벌급 서비스를 제공하는 곳이라면 여러 구성의 노드로 클러스터를 만들기도 한다 (예: GPU 작업용 물리서버, WAS용 EC2+GCP, 비즈니스 로직용 온프레미스).

##### 2. Control Plane

**클러스터 전체의 상태를 관리하고, 파드 배치 같은 것을 판단하는 중앙 제어 영역**

역할: 사용자 요청 수신, 클러스터 상태 저장, 현재 상태와 원하는 상태 비교, 실행할 노드 선택, 장애 감지, 필요한 작업 조정.

> 💡 흔히 **Master Node = Control Plane**이라고도 부르는데, "Master/Slave"라는 용어는 해외에서는 민감하게 받아들여져 최근에는 Control Plane / Worker Node라는 표현을 더 많이 사용한다.

```
Control Plane
├── kube-apiserver
├── etcd
├── kube-scheduler
└── kube-controller-manager
```

**① kube-apiserver**
쿠버네티스 클러스터로 들어오는 **모든 관리 요청의 중앙 출입구**다. 사용자가 `kubectl` 명령을 내리면 워커 노드에 직접 전달되지 않고, 반드시 API Server를 거친다.

```
사용자 → kubectl → Control Plane → api-server → 클러스터 내부 구성요소
```

API Server의 역할: 사용자 요청 수신, 사용자 인증, 권한 확인, 요청 내용 검증, 내부 구성요소 간 통신 중계, 클러스터 상태 변경, 상태 조회.

> ⚠️ 다만 예외가 있다 — **노드 안에 있는 Pod들 간의 요청은 API Server를 거치지 않는다.**

**② etcd**
클러스터의 설정과 상태를 저장하는 **"분산" key-value 저장소**다. 어떤 노드가 있는지, 어떤 애플리케이션이 실행되어야 하는지, 몇 개가 실행돼야 하는지, 현재 상태는 어떤지, 어떤 설정·정책이 적용되는지 등 **쿠버네티스가 하려는 모든 정보**가 여기 저장된다.

> ⚠️ etcd 데이터가 손실되면 클러스터는 정상적으로 작동할 수 없다. 그만큼 중요하기 때문에 운영 환경에서는 etcd 백업, 다중화(고가용성), 복구 작업 사전 준비, 접근 제한이 필수적으로 요구된다.

**③ kube-scheduler**
**새로운 실행 단위(파드)를 어느 워커 노드에 배치할지 결정하는 구성 요소**다. 아직 실행할 노드가 정해지지 않은 파드를 적절한 워커 노드에 배정한다.

예를 들어 새로 배치되어야 하는 애플리케이션이 CPU 1코어, 메모리 512MB를 필요로 하는 상황에서, 위의 A/B/C 서버 사용률 표를 참고하면 CPU 여유가 부족한 A, C보다 **B에 배치**하는 식으로 판단한다.

스케줄러가 고려하는 정보: 노드의 CPU·메모리 자원, 애플리케이션이 요청한 자원, 노드 상태, 배치 조건(예: "이 컨테이너는 BE 컨테이너와 떨어져 있어야 한다", "이 컨테이너는 항상 Nginx Container와 같이 있어야 한다"), 스토리지 조건.

> 💡 스케줄러는 **컨테이너를 실행하지 않는다.** 어디에서 실행할지 "결정"만 한다.

**④ kube-controller-manager**
**현재 상태를 원하는 상태로 맞추는 여러 제어 기능을 실행하는 구성 요소**다. 클러스터 상태를 계속 확인하다가 원하는 상태와 다르면 조정 명령을 내린다. 예를 들어 원하는 상태가 3개인데 현재 2개라면, kube-controller-manager가 먼저 이를 감지하고 부족한 실행 단위(파드)를 생성 요청한다.

##### 3. Worker Node

**실제 애플리케이션(파드)이 실행되는 서버**다. Control Plane이 "두뇌" 역할을 한다면, Worker Node는 **실제 작업을 수행하는 손발**에 해당한다.

```
Worker Node
├── kubelet
├── Container Runtime
├── kube-proxy
└── 애플리케이션 실행 단위 (Pod)
```

워커 노드가 수행하는 역할: 컨테이너 이미지 다운로드, 컨테이너 실행, 컨테이너 상태 확인, 실행 상태를 Control Plane에 보고, 네트워크 통신 처리, 장애 발생 시 컨테이너 재시작.

**① kubelet**
**각 워커 노드에서 애플리케이션 실행 상태를 관리하고, Control Plane에 보고하는 에이전트**다. Worker Node마다 하나씩 붙어 있다.

```
API Server
   ↓ 실행해야 할 내용 전달
kubelet
   ↓
Container Runtime
   ↓
Container 실행
```

kubelet의 역할: API Server와 통신, 실행해야 하는 작업 확인, Container Runtime에 특정 작업 요청, 컨테이너 상태 확인, 애플리케이션 상태 점검, 노드 상태 보고, 문제가 발생한 컨테이너 재시작. 즉 **Worker Node에서 실제 실행 상태를 책임지는 에이전트**다.

**② Container Runtime**
**컨테이너 이미지를 가져오고, 실제 컨테이너 프로세스를 실행하는 소프트웨어**다. 쿠버네티스는 관리 소프트웨어이기 때문에 컨테이너 프로세스를 직접 실행하지 않고, Container Runtime에 실행을 요청한다.

```
kubelet
   ↓ 실행 요청
Container Runtime
   ↓
Image 다운로드
   ↓
Container 생성 및 실행
```

대표적인 Container Runtime으로는 **containerd**(표준으로 많이 사용), **CRI-O**가 있다. **Docker는 더 이상 쿠버네티스의 기본 런타임으로 쓰이지 않는데**, k8s가 요구하는 Container Runtime 필수 요소(CRI 규격)를 지원하지 않기 때문이다.

> 💡 쿠버네티스에서 도커 이미지를 사용한다고 해서 도커를 별도로 설치해야 하는 건 아니다. 도커는 "컨테이너를 관리하기 쉽게 만든 도구"일 뿐이며, `container != docker`다. 컨테이너를 실행할 수 있는 런타임만 있으면 쿠버네티스는 정상 동작한다.

**③ kube-proxy**
**Worker Node에서 네트워크 트래픽 전달을 위한 규칙을 관리하는 구성 요소**다. 클러스터 내에는 여러 노드가 분산되어 있고 그 안에서 여러 컨테이너가 실행되는데, 같은 노드 안의 통신은 문제가 없지만 **노드 간 통신**은 별도로 처리되어야 한다. kube-proxy는 워커 노드의 네트워크 규칙을 관리해 트래픽이 적절한 목적지로 전달되도록 지원한다.

> 💡 쿠버네티스에서 네트워킹은 상당히 까다로운 문제라서 kube-proxy만으로는 해결되지 않고, **CNI** 같은 별도의 구성요소가 함께 필요하다. (자세한 내용은 아래 CNI/Flannel 섹션 참고)

**④ Pod**
**쿠버네티스가 컨테이너를 실행하고 관리하는 가장 작은 배포 단위**다. 쿠버네티스는 컨테이너를 직접적인 실행 단위로 다루지 않고, **Pod**라는 단위로 관리한다.

```
Pod
├── Main Container
└── Helper Container
```

하나의 파드에는 하나 이상의 컨테이너가 들어갈 수 있으며, **같은 파드 안의 컨테이너들은 네트워크, IP 주소, localhost, 스토리지, 생명주기를 공유**한다.

```
Cluster
└── Worker Node
      └── Pod
            └── Container
```

##### 쿠버네티스 동작 흐름 (예: Deployment 생성 요청)

```
사용자 → kubectl → API Server → etcd → Controller

1. 사용자가 Deployment 생성 요청
2. kubectl이 Kubernetes API 호출
3. API Server가 인증·인가 및 요청 검증
4. API Server가 etcd에 "원하는 상태" 저장
5. Controller가 etcd에서 상태 조회
6. Controller가 필요한 리소스 생성을 API Server에 요청
```

#### | 쿠버네티스를 왜 사용하는가

**애플리케이션의 배포, 확장, 감시와 장애 복구를 자동화하여 운영 복잡성을 줄이기 위해 사용한다.**

##### 장점

**1. 자동 복구 (Self-Healing)**
컨테이너가 종료되거나 애플리케이션에 문제가 생기면 자동으로 다시 실행하도록 구성된다.

```
정상 실행 → 컨테이너 장애 → 장애 감지 → 컨테이너 재시작
```

이를 통해 장애 대응 시간이 줄고 가용성이 올라간다.

> ⚠️ 주의할 점: 쿠버네티스가 **애플리케이션의 버그 자체를 수정하는 것은 아니다.** 그냥 "이 컨테이너가 3개 있어야 하는데 2개네, 그럼 1개 더 만들자"는 식으로 **원하는 상태를 유지**해줄 뿐이다.
>
> 💡 이런 관점을 흔히 **"가축 vs 애완동물" 비유**로 설명한다. 애완동물은 이름을 붙여주고 아프면 병원에 데려가지만, 가축은 이름 대신 번호를 붙이고 아프면 도태시킨다. 쿠버네티스는 컨테이너를 "가축"처럼 다룬다 — 특정 컨테이너를 소중히 살리는 게 아니라, 죽으면 똑같은 걸 새로 만든다.

**2. 배치가 자동으로 된다**
쿠버네티스는 etcd에 노드 상태를 기록해두고, 애플리케이션 요구사항을 보고 알아서 필요한 노드에 배치한다. 서버마다 직접 들어가서 배치할 필요가 없고, 자원을 고려한 배치와 워크로드 분산이 자동으로 이루어진다.

**3. 확장성**
애플리케이션이 더 필요하면 원하는 상태 숫자만 늘리면 된다(예: 3 → 10). 쿠버네티스가 알아서 만들어준다. 트래픽이 줄어들면 자동으로 줄어들 수도 있는데, 이를 위해서는 **HPA(Horizontal Pod Autoscaler)** 설정이 필요하다.

**4. 운영 환경을 일관되게 만들 수 있음**
실행 환경과 원하는 상태를 YAML 파일로 정의해두면, 이 파일 하나로 어디서든 동일하게 운영 환경을 재현할 수 있다. 또한 **CSP(Cloud Service Provider, 예: AWS/GCP/NCP)에 종속되지 않는다** — 만약 S3를 쓰다가 GCP로 옮겨야 한다면 파일도 옮기고 DB의 파일 경로도 GCP 경로로 바꿔야 하는 번거로움이 생기는데, 쿠버네티스 방식은 이런 종속을 줄여준다.

**5. 선언적 관리 가능**
원하는 상태를 정의만 하면 되므로 반복적인 운영 작업이 줄어든다.

##### 단점

**1. 학습 난이도가 지나치게 높다**
Linux, Container, OS, Network, 보안, Storage, Cloud, 분산 시스템, 중앙 집중식 로깅·모니터링 등 여러 영역의 지식이 필요하다. 개념들이 상당 부분 추상화되어 있어, 표면에 드러나는 부분과 드러나지 않는 부분을 함께 이해해야 한다.

**2. 운영 구조가 너무 복잡하다**
Control Plane, Worker Node, etcd, Container Runtime, CNI, CoreDNS, LocalDNS, Storage, Monitoring, Logging, 중앙 집중형 대시보드, 인증 등 관리해야 할 요소가 애플리케이션 관리와는 별개로 매우 많다.

**3. 장애 원인을 찾기 힘들다**
사용자 요청이 여러 레이어를 거쳐서 동작한다.

```
사용자 요청
→ 외부 네트워크
→ Controller
→ Kubernetes Network (CNI)
→ Worker Node
→ kube-proxy
→ LocalDNS
→ NI
→ Pod
→ Container
→ Application
→ Database
```

또한 **파드의 특징상 어떤 파드가 어떤 노드에 배치될지 사전에 예상할 수 없다.** 그래서 중앙 집중형 로깅·모니터링이 사실상 필수인데, 이 모니터링 시스템 자체도 관리 포인트와 비용을 추가로 발생시킨다.

**4. 비용이 증가한다**
최소 구성만 하더라도 상당한 인프라가 필요하다. 예를 들어 단순히 백엔드만 서빙한다고 해도:
- DB용 EC2 t3.medium 2대
- FE용 EC2 t3.medium 2대
- 여기에 Control Plane(t3.small 등) 2대

이조차 **완전한 고가용성 구성은 아니다** — etcd 고가용성 구성, master node 2개 이상(보통 3개 정도) 구성이 더해져야 한다. 현업에서는 최소한 EC2 9대 정도는 되어야 쿠버네티스를 쓰는 의미가 생긴다는 감각이 있다.

**5. 작은 서비스에서는 의미가 없다**
QPS(초당 요청 수) 기준 3000 정도까지는 EC2만으로 돌려도 충분한 경우가 많다.

#### | 실무에서는 쿠버네티스를 어떻게 사용하는지?

**1. 관리형 서비스를 많이 사용한다**
AWS EKS 같은 관리형 쿠버네티스 서비스를 많이 사용한다. (학습 과정에서는 `kubeadm`이라는 배포 패키지를 사용한다.) 쿠버네티스는 대규모 트래픽을 가진 서비스에서 주로 활용되며, 스타트업은 대부분 클라우드를 사용하는 반면 대기업은 클라우드 도입 비율이 상대적으로 낮은 편이다. 관리형 서비스만이 아니라 쿠버네티스 자체를 다룰 줄 아는 역량도 필요하다.

**2. CI/CD 환경과 함께 묶어서 사용한다**
Container Runtime이 실행될 때 이미지를 항상 Registry에서 가져온다. 워커 노드에는 이미 수많은 파드가 동작하고 있어 리소스가 넉넉하지 않기 때문에, **빌드를 워커 노드에서 직접 하기는 어렵다.**

**3. 서비스가 많아질 때(MSA) 도입한다**
예를 들어 커뮤니티 서비스의 게시판 기능이 강화되면서, 필요한 데이터베이스가 더 이상 RDBMS(MySQL) 하나로는 부족해질 수 있다(예: Doc DB가 유용한 상황). 이럴 때 게시판 서비스를 코드와 DB 단위로 통째로 분리하는 과정을 반복하면 자연스럽게 **MSA(Microservice Architecture)** 환경이 되고, 쿠버네티스는 이런 여러 서비스(컨테이너)들을 오케스트레이션하는 역할을 한다.

**4. 중앙 집중형 모니터링·로깅 시스템을 붙이는 것이 거의 필수다**
시스템이 분산되어 있는 만큼, 이런 시스템 없이는 문제 발생 원인을 파악하기가 매우 어렵다.

##### 오해하기 쉬운 것들

1. **"쿠버네티스 = 장애 내결함성이 생긴다"는 오해** — 장애는 여전히 발생한다(DB 네트워크 장애, 애플리케이션 에러, 자원 부족 등). 쿠버네티스는 **장애를 없애는 것이 아니라, 일부 장애를 자동으로 감지하고 복구하는 시스템**일 뿐이다.
2. **"컨테이너가 많으니까 쿠버네티스를 써야 한다"는 오해** — 단순히 컨테이너 수만으로 도입 여부를 결정하면 안 된다. 기존 운영 복잡성이 이미 높은지, 운영 자동화가 정말 필요한지, 자동화로 얻는 이득이 쿠버네티스 운영 부담을 넘어서는지, 개발팀 규모, 배포 빈도, 트래픽 변화, 그리고 **비용**(최소 월 100만 원 수준)까지 종합적으로 고려해야 한다.

##### 최종 정리

| 항목 | 핵심 내용 |
|---|---|
| 쿠버네티스란 | 여러 서버에서 실행되는 컨테이너를 자동으로 배치하고, 감시하고, 복구하고, 확장하는 컨테이너 오케스트레이션 플랫폼 |
| 왜 사용하는가 | 컨테이너가 많아질수록 사람이 관리하기 힘든 배치·장애복구·확장·상태관리·배포 작업을 자동화 |
| 핵심 개념 | 원하는 상태와 현재 상태를 비교해 원하는 상태로 만든다 |
| 장점 | 자동 복구, 자동 배치, 확장성, 선언적 관리, 일관된 운영 환경 |
| 단점 | 높은 학습 난이도, 복잡한 운영 구조, 높은 장애 분석 난이도, 비용 |

#### | CNI와 Flannel

##### Pod끼리는 어떻게 통신하는가?

쿠버네티스는 파드를 생성하고 어느 노드에서 실행할지 결정하며, 파드에 IP를 부여한다. 그런데 **파드들이 서로 통신할 수 있는 네트워크 자체는 쿠버네티스가 직접 구현하지 않는다.** 이 네트워크 기능을 담당하는 것이 **CNI 플러그인**(예: Flannel)이다.

```
   EC2                                EC2
Node A (192.168.10.10)        Node B (192.168.10.20)
┌─────────────────┐           ┌─────────────────┐
│ Pod A            │  ──?──→   │ Pod B            │
│ 10.244.1.5        │  어떻게 전달?  │ 10.244.2.5        │
└─────────────────┘           └─────────────────┘
```

| 구분 | 예시 | 의미 |
|---|---|---|
| **Node IP** | 192.168.10.10 | EC2 또는 물리 서버가 가진 실제 네트워크 IP |
| **Pod IP** | 10.244.1.5 | 쿠버네티스 내부에서 Pod에 할당된 IP |

Node A와 Node B는 서로 실제 네트워크로 연결되어 있고 서로의 노드 IP는 알고 있지만, **기존 네트워크 장비는 상대방 Pod의 IP를 알지 못한다.** 이 문제를 해결하기 위해 CNI와 CNI 플러그인을 사용한다.

##### CNI란?

**Container Network Interface** — 컨테이너 네트워크를 설정하고 제거하는 방법을 정의한 **표준 규격 인터페이스**다. "컨테이너끼리 통신하려면 이 규격을 지키자"는 약속이며, 그 약속을 실제로 구현한 것이 **CNI 플러그인**이다.

CNI 플러그인이 담당하는 것: 컨테이너/파드에 네트워크 인터페이스 연결, IP 주소 할당, 라우팅 설정, 컨테이너 삭제 시 네트워크 자원 정리.

대표적인 CNI 플러그인: **Flannel**, **Calico**, **Cilium**

##### 왜 CNI를 사용하는가?

CNI가 없다면 쿠버네티스가 AWS, GCP, NCP 같은 모든 네트워크 환경을 직접 지원해야 한다. CNI를 사용하면 **쿠버네티스는 규격(인터페이스)만 정의**하고, 그 규격을 구현한 플러그인을 상황에 맞게 선택해서 쓸 수 있다 — 결과적으로 **특정 CSP에 종속되지 않고** AWS든 GCP든 어디서든 사용할 수 있다.

##### Flannel이란?

**여러 노드에 있는 Pod들이 서로 통신할 수 있도록 Pod 네트워크를 구성하는 네트워크 플러그인**

| 역할 | 설명 |
|---|---|
| **노드별 Pod 대역 관리** | 각 노드가 사용할 Pod IP 대역을 구분 |
| **Pod 네트워크 구성** | Pod가 네트워크에 연결될 수 있도록 인터페이스와 경로 구성 |
| **노드 간 통신 경로 제공** | 서로 다른 노드의 Pod끼리 통신할 수 있도록 연결 |
| **캡슐화** | VXLAN 사용 시 Pod 패킷을 Node 간 전송 가능한 패킷으로 감쌈 |

##### 왜 Flannel이 필요한가?

**같은 노드 안에서의 통신**은 노드 내부의 가상 네트워크(`cni0`)를 통해 별문제 없이 처리된다.

```
Node A
├── Pod A (10.244.1.5) ─┐
│                        ├── 가상 네트워크 (cni0)
└── Pod B (10.244.1.6) ─┘
```

**다른 노드에 있는 Pod 간 통신**은 상황이 다르다. 기존 물리 네트워크는 Pod의 IP 경로 자체를 모를 수 있다.

```
Node A (192.168.10.10)              Node B (192.168.10.20)
Pod A (10.244.1.5)  ──기존 물리 네트워크는 Pod IP 경로를 모를 수 있음──▶  Pod B (10.244.2.5)
```

이 문제를 해결하기 위해 Flannel은 **VXLAN** 방식을 사용한다.

##### VXLAN (Virtual eXtensible LAN)

**Pod의 패킷을 노드 간에 전달할 때, 원래 패킷을 UDP 패킷으로 한 번 더 감싸는 터널링 기술**이다.

**VXLAN 패킷 전달 과정**

```
1. 패킷 생성   : Pod A가 Pod B를 향한 패킷 생성
2. 캡슐화      : Node A가 원본 패킷을 외부 UDP 패킷으로 한 번 더 감쌈
3. 물리 네트워크 전송 : 물리 네트워크는 Pod IP를 몰라도 됨 — 그냥 "Node B로 간다"는 것만 알면 됨
4. 캡슐 제거   : Node B가 UDP 패킷을 받아 외부 VXLAN 헤더를 벗기면 내부 원본 패킷이 드러남
5. 목적지 전달 : Node B가 내부 파드(Pod B)에게 최종 전달
```

캡슐화된 외부 패킷(Outer Packet)은 대략 다음과 같은 형태다.

```
Outer Packet
출발지 Node : 192.168.10.10
목적지 Node : 192.168.10.20
프로토콜    : UDP
목적지 포트 : 8472
```

> 💡 **택배 상자를 이중 포장하는 것에 비유할 수 있다.** Pod가 원래 보내려는 패킷은 작은 택배 상자다. 일반 배송망(물리 네트워크)은 이 작은 상자의 진짜 주소(Pod IP)를 알지 못하므로, Flannel VXLAN이라는 큰 상자에 한 번 더 넣어서 보낸다. Node B가 이 큰 상자를 받으면 포장을 뜯어서 내부 파드에게 원래 상자를 전달하는 구조다.

##### 왜 TCP가 아니라 UDP로 감싸는가? (TCP over TCP 문제)

핵심 포인트: **TCP 패킷을 다시 TCP 터널로 감싸면 오히려 문제가 생긴다.**

**1. 과잉 친절의 부작용 — TCP Meltdown**

우리가 원래 보내려는 데이터는 이미 TCP로 안전하게 포장되어 있다(신뢰성·순서 보장이 내장됨). 그런데 이걸 나르는 터널까지 굳이 TCP로 다시 감싸면 오히려 방해가 된다.

- 네트워크가 잠깐 느려져 패킷이 늦게 도착하는 상황을 가정해보자.
- 내부(Inner) TCP: "어? 데이터가 안 왔네? 재전송해!" → 다시 보냄
- 외부(Outer) TCP: "어? 터널링 패킷이 안 갔네? 나도 재전송해!" → 또 보냄
- 결과: 안 그래도 네트워크가 막혀 있는데, 내부·외부 TCP가 동시에 재전송을 시도하면서 데이터가 쏟아져 **네트워크가 완전히 멈춰버리는 현상**이 발생한다.

이런 현상을 **TCP Meltdown**이라고 부른다.

**2. 빠르고 단순한 배송(단순 운반)**

UDP는 연결 상태를 유지하지 않는 프로토콜이다. Flannel(VXLAN)은 터널 역할만 하면 되고, 내용물이 뭔지 검사할 필요가 없다 — 그냥 A 지점에서 B 지점으로 빠르게 전달하는 역할만 하면 된다. 신뢰성(패킷 손실, 순서 등)은 이미 **내부의 원본 TCP 패킷이 알아서 감지하고 재요청**하기 때문에, 터널을 담당하는 UDP까지 그 역할을 중복으로 떠맡을 필요가 없다.

**3. 헤더 오버헤드 감소**

TCP는 연결을 맺고, 순서를 맞추고, 잘 받았는지 확인하느라 헤더가 무겁다. 모든 패킷마다 이 과정을 거치면 속도가 크게 느려진다. 가벼운 UDP를 터널링 프로토콜로 사용하면 이 오버헤드를 줄일 수 있다.

### 🗨️ 오늘의 회고

- 오늘은 그동안 배웠던 컨테이너·CI/CD·무중단 배포 지식들이 왜 필요했는지가 쿠버네티스라는 큰 그림 안에서 자연스럽게 연결됐다. "물리 서버 → 가상 머신 → 컨테이너 → 오케스트레이션"으로 이어지는 흐름을 따라가니, 컨테이너까지는 알고 있었지만 **"컨테이너를 여러 서버에서 운영하는 문제"는 도커만으로는 해결되지 않는다**는 지점이 명확해졌다.
- Control Plane과 Worker Node의 구성 요소(kube-apiserver, etcd, kube-scheduler, kube-controller-manager, kubelet, Container Runtime, kube-proxy)를 각각의 역할로 쪼개서 보니, 단순히 "쿠버네티스가 알아서 해준다"가 아니라 **원하는 상태 vs 현재 상태를 계속 비교하는 루프**가 시스템 전체를 관통하는 핵심 원리라는 게 인상 깊었다.
- "쿠버네티스는 컨테이너를 가축으로 본다"는 비유가 오늘 가장 기억에 남는다. 장애가 나도 그 컨테이너를 살리는 게 목적이 아니라, **원하는 개수를 유지하는 것**이 목적이라는 관점이 자동 복구의 본질을 잘 설명해줬다.
- 쿠버네티스의 장점만큼 단점(학습 난이도, 운영 복잡도, 비용)도 꽤 비중 있게 다뤄졌는데, 특히 "최소 EC2 9대는 되어야 쓰는 의미가 있다", "QPS 3000까지는 EC2로만 돌려도 된다"는 구체적인 기준이 실무 감각을 잡는 데 도움이 됐다. 쿠버네티스 도입 여부를 "컨테이너 개수"가 아니라 비용·팀 규모·운영 자동화 필요성으로 판단해야 한다는 정리도 실용적이었다.
- CNI/Flannel 파트는 오늘 배운 것 중 가장 네트워크 이론에 가까운 부분이었다. 특히 **VXLAN에서 왜 굳이 UDP로 감싸는지(TCP Meltdown 문제)**를 이해하고 나니, "이미 신뢰성이 보장된 패킷을 또 신뢰성 있는 터널로 감싸면 오히려 손해"라는 원리가 명확해졌다. 택배 이중 포장 비유가 개념을 잡는 데 특히 도움이 됐다.
