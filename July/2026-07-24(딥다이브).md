## 1. ReplicaSet

### ReplicaSet이란?

Pod만으로는 노드 장애나 삭제 시 자동 복구가 되지 않고, 여러 개를 유지하려면 수동으로 생성해야 한다. ReplicaSet은 **"이 Pod를 N개 유지하라"는 선언을 하고, Controller가 desired(원하는 수)와 actual(실제 수)를 맞추는 워크로드**이다.

- Pod를 직접 관리하며, 정확히 몇 개를 유지할지 지정할 수 있다.
- 이를 통해 서비스의 안정성과 고가용성을 확보한다.
- **Label**을 통해 어떤 Pod를 관리할지 식별한다.

### 구성 요소

| 필드 | 설명 |
|---|---|
| `replicas` | 유지할 Pod의 개수 |
| `selector` | 어떤 Pod를 관리할지 선택하는 조건 |
| `template` | 실제로 생성할 Pod의 정의 |

### ReplicaSet을 직접 사용하는 시기

ReplicaSet은 지정된 수의 Pod 레플리카가 항상 실행되도록 보장한다. 그러나 **Deployment는 ReplicaSet을 관리하고 Pod에 대한 선언적 업데이트, 롤아웃, 롤백 등 더 많은 기능을 제공하는 상위 개념**이다.

따라서 별도의 사용자 정의(custom) 업데이트 오케스트레이션이 필요한 경우, 또는 업데이트가 전혀 필요 없는 경우가 아니라면 ReplicaSet을 직접 사용하기보다는 **Deployment 사용을 권장**한다.

## 2. Deployment

### Deployment란?

쿠버네티스의 리소스로, **ReplicaSet을 관리하여 Pod의 개수뿐 아니라 애플리케이션의 배포, 업데이트, 롤백까지 자동으로 관리하는 상위 Controller**이다.

- 보통 **Stateless(스테이트리스) 애플리케이션**을 대상으로 한다.
- Pod와 ReplicaSet에 대한 **선언적 업데이트**를 제공한다.
- Deployment에서 의도하는 상태를 설명하면, Deployment Controller가 현재 상태에서 의도하는 상태로 비율을 조정하며 변경한다.

> 💡 **Stateless Application이란?** 클라이언트와 서버의 연결 구조에서 불필요한 상태 반영을 위한 데이터나 리소스가 사용되지 않는 형태의 서비스 구조를 말한다. 각 요청이 독립적으로 처리되며 이전 요청의 상태를 서버가 저장하지 않는다.

### Deployment의 주요 유스케이스

- **롤아웃**: ReplicaSet을 롤아웃할 Deployment를 생성하여 백그라운드에서 Pod를 생성한다.
- **업데이트**: `PodTemplateSpec`을 업데이트하여 새로운 ReplicaSet으로 Pod를 속도를 제어하며 이동시킨다.
- **롤백**: 현재 상태가 안정적이지 않은 경우 이전 revision으로 롤백한다.
- **스케일 업**: 더 많은 로드를 위해 Deployment를 스케일 업한다.
- **일시 중지 및 재개**: 롤아웃을 일시 중지하여 여러 수정 사항을 적용한 뒤 재개한다.
- **상태 확인**: 롤아웃이 막혀있는지를 나타내는 Deployment 상태를 확인한다.
- **정리**: 더 이상 필요 없는 이전 ReplicaSet을 정리한다.

### Deployment 생성 예시

3개의 nginx Pod를 유지하는 Deployment를 생성하는 예시이다.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.14.2
        ports:
        - containerPort: 80
```

**각 필드 설명:**

- `.metadata.name`: `nginx-deployment` 이름을 가진 Deployment가 생성된다. 이 이름은 이후 생성되는 ReplicaSet과 Pod의 기준이 된다.
- `.spec.replicas`: 복제된 Pod 3개를 만드는 ReplicaSet을 생성한다.
- `.spec.selector`: 생성된 ReplicaSet이 관리할 Pod를 찾아내는 방법을 정의한다. 이 예시에서는 `app: nginx` 레이블을 선택한다.
- `.spec.template`: Pod 템플릿을 정의한다. Pod에 `app: nginx` 레이블을 붙이고, nginx 1.14.2 이미지를 실행하는 컨테이너 1개를 생성한다.

> 💡 **참고 - selector.matchLabels**: `matchLabels` 필드는 `{key, value}` 쌍으로 매핑된다. 매칭을 위해서는 `matchLabels`와 `matchExpressions`의 모든 요건이 충족되어야 한다.

```bash
# Deployment 생성 명령어
kubectl apply -f nginx-deployment.yaml
```

### 롤아웃 (Rollout)

Deployment는 **template 변경 시 새 ReplicaSet을 만들고 기존 ReplicaSet을 0으로 줄이면서 Pod를 교체**한다. 이 "교체하는 과정"이 **롤아웃(rollout)**이다. 롤아웃이 얼마나 빠르게, 얼마나 안전하게 진행될지는 `strategy`와 그 안의 `maxSurge`, `maxUnavailable`로 제어한다.

> ⚠️ **주의**: Deployment의 Pod 템플릿(`.spec.template`)이 변경된 경우에만 rollout이 트리거된다. 예를 들어 템플릿의 레이블이나 컨테이너 이미지가 업데이트된 경우가 해당한다. `replicas` 수 변경 등은 rollout을 트리거하지 않는다.

### 롤백 (Rollback)

롤아웃 후 새 버전에 문제가 있으면 이전 버전으로 되돌리는 **롤백(rollback)**이 필요하다. Deployment는 template 변경 이력을 **revision**으로 저장하고, `kubectl rollout undo` 명령어로 롤백할 수 있다.

```bash
# 이전 revision으로 롤백
kubectl rollout undo deployment/nginx-deployment

# 특정 revision으로 롤백
kubectl rollout undo deployment/nginx-deployment --to-revision=2
```

## 3. 쿠버네티스 배포 전략 비교

쿠버네티스에서 애플리케이션을 배포할 때 두 가지 주요 전략을 사용할 수 있다.

### 전략 1. RollingUpdate (롤링 업데이트)

새로운 버전의 Pod를 점진적으로 배포하는 방식이다. 기존 Pod는 서서히 종료되고, 새로운 Pod는 차례로 추가된다.

| 구분 | 내용 |
|---|---|
| ✅ **장점** | 무중단 배포 가능 — 새 버전 실행 중에도 기존 버전이 계속 운영 |
| ✅ **장점** | 문제 발생 시 빠르게 이전 버전으로 롤백 가능 |
| ❌ **단점** | Pod 간 의존성이 높은 경우 예상치 못한 문제가 발생할 수 있다 |
| ❌ **단점** | 여러 버전이 동시에 실행되므로 추가 리소스를 소모한다 |

**적합한 상황**: 고객에게 지속적인 서비스를 제공해야 하는 웹 애플리케이션, 대규모 시스템 — 안정성과 가용성이 최우선인 경우

### 전략 2. Recreate (재생성)

기존의 모든 Pod를 먼저 종료한 후 새로운 Pod를 생성하는 방식이다.

| 구분 | 내용 |
|---|---|
| ✅ **장점** | 구현이 단순하고 두 버전 간 호환성 문제가 발생하지 않는다 |
| ✅ **장점** | Stateless 애플리케이션에서 효과적 |
| ❌ **단점** | 배포 중 서비스가 완전히 중단된다 (다운타임 발생) |
| ❌ **단점** | 실시간 서비스에서는 치명적인 문제가 될 수 있다 |
| ❌ **단점** | 대규모 서비스에서 모든 Pod를 동시에 종료하는 것은 위험 |

**적합한 상황**: 빠른 배포가 필요하거나 버전 간 호환성이 보장되는 경우, 시스템 다운타임이 큰 문제가 되지 않는 경우

### RollingUpdate vs Recreate 비교

| 구분 | RollingUpdate | Recreate |
|---|---|---|
| **다운타임** | 없음 | 있음 (전체 Pod 교체 동안) |
| **리소스 소모** | 배포 중 일시적으로 증가 | 적음 (동시 실행 Pod 없음) |
| **버전 공존** | 있음 (배포 중 구버전·신버전 공존) | 없음 |
| **롤백** | 빠름 | 다시 전체 교체 필요 |
| **복잡도** | 높음 (하위 호환성 필요) | 낮음 |

## 4. Rolling Update 상세 — maxSurge & maxUnavailable

### 설정 위치

`.spec.strategy.type == RollingUpdate`로 설정 시 롤링 업데이트 방식으로 업데이트하며, `maxUnavailable`과 `maxSurge`를 명시하여 프로세스를 제어한다.

```yaml
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.14.2
        ports:
        - containerPort: 80
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 1
```

### maxUnavailable (최대 불가 파드 수)

업데이트 진행 중 **동시에 사용 불가능(Terminating 또는 미기동) 상태가 허용되는 최대 Pod 수 또는 비율**이다.

- **기본값**: 25%
- 절대값(예: `1`) 또는 백분율(예: `25%`)로 지정 가능
- 퍼센트 계산 시 **내림(Round Down)** 처리
- `maxSurge`가 0이면 `maxUnavailable`은 0이 될 수 없다
- 값이 **크면** 업데이트가 빨리 끝나지만, 가용 서비스 용량이 줄어든다

예를 들어 이 값을 30%로 설정하면, 롤링 업데이트 시작 시 즉각 기존 ReplicaSet의 크기를 의도한 Pod의 70%로 스케일 다운할 수 있다. 업데이트 중 항상 사용 가능한 전체 Pod 수는 의도한 Pod 수의 **70% 이상**이 되도록 유지된다.

### maxSurge (최대 서지 파드 수)

Desired(원하는) 복제본 수보다 **일시적으로 더 생성할 수 있는 최대 Pod 수 또는 비율**이다.

- **기본값**: 25%
- 절대값(예: `1`) 또는 백분율(예: `25%`)로 지정 가능
- 퍼센트 계산 시 **올림(Round Up)** 처리
- `maxUnavailable`이 0이면 `maxSurge`는 0이 될 수 없다
- 값이 **크면** 기존 용량을 유지하거나 초과하면서 새 버전을 미리 띄울 수 있어 무중단에 유리하다

예를 들어 이 값을 30%로 설정하면, 롤링 업데이트 시작 시 새 ReplicaSet의 크기를 즉시 조정하여 기존 및 새 Pod의 전체 개수가 의도한 Pod의 **130%를 넘지 않도록** 한다.

> 💡 **참고**: 쿠버네티스가 `availableReplicas` 수를 계산할 때 종료 중인(Terminating) Pod는 포함하지 않는다. 이 수는 `replicas - maxUnavailable`와 `replicas + maxSurge` 사이에 존재한다. 롤아웃 중에는 Pod의 수가 예상보다 많을 수 있으며, 종료 중인 Pod의 `terminationGracePeriodSeconds`가 만료될 때까지 총 리소스가 `replicas + maxSurge` 이상일 수 있다.

## 5. 결론 : 주제 정리

### Rolling Update의 동작 과정 — 단계별 상세

```
[초기 상태]
Deployment → ReplicaSet v1 → Pod v1 × 3개

        ↓ template 변경 (예: 이미지 버전 업)

1단계: v2 ReplicaSet 생성
  - v2 ReplicaSet 생성, maxSurge만큼 신규 Pod 생성
  - 전체 Pod 수 = replicas + maxSurge 이하로 제한

2단계: v1 ReplicaSet 축소
  - v1 ReplicaSet의 replicas를 (replicas - maxUnavailable)로 감소
  - 해당 수치만큼 기존 Pod 제거

3단계: 트래픽 분산
  - 배포가 진행되는 동안 v1과 v2 Pod에 트래픽이 분산됨

4단계: 점진적 교체 반복
  - v2 replicas를 1씩 증가, v1 replicas를 1씩 감소
  - v2 replicas가 템플릿에 정의된 replicas와 일치할 때까지 반복

5단계: v1 ReplicaSet 제거
  - v1 replicas를 0으로 변경 후 남은 Pod 삭제
  - 배포 완료

[최종 상태]
Deployment → ReplicaSet v2 → Pod v2 × 3개
```

### maxSurge & maxUnavailable 설정 유형별 영향 분석

#### 유형 A. 기본값 설정 (maxSurge: 25%, maxUnavailable: 25%)

**영향**: 안정성과 배포 속도의 균형을 맞춘 타협안이다.

**동작 예시** (replicas: 4 기준):
- maxSurge로 인해 최대 **5개**(4 + 1)까지 Pod가 동시에 뜰 수 있다.
- maxUnavailable로 인해 최소 **3개**(4 - 1)의 Pod가 항상 유지된다.
- 기존 Pod를 일부 종료하고 새 Pod를 올리는 과정을 반복하므로 리소스 변화 폭이 적당하다.

```
replicas: 4 기준
최소 가용 Pod: 4 - 1 = 3개
최대 총 Pod:  4 + 1 = 5개
```

#### 유형 B. 완전 무중단·안정성 우선 (maxSurge: 25% 이상, maxUnavailable: 0)

**영향**: 가장 안전한 무중단 배포. 트래픽 처리 용량이 배포 중 절대 줄어들지 않는다.

**동작**:
1. 기존 Pod를 먼저 종료하지 않고(`maxUnavailable: 0`), 새 버전 Pod를 먼저 `maxSurge`만큼 생성한다.
2. 새 Pod가 **Ready 상태**가 되어 트래픽을 받을 수 있게 된 후에야 기존 Pod를 하나씩 종료한다.

**단점**: 일시적으로 노드 리소스(CPU, 메모리)를 평소보다 많이 소모하므로 클러스터 여유 자원이 충분해야 한다.

> ⚠️ `maxUnavailable: 0`이 제대로 동작하려면 애플리케이션에 **`readinessProbe`가 올바르게 설정**되어 있어야 한다. 새 Pod가 실제로 트래픽을 받을 준비가 됐을 때만 구형 Pod가 내려간다.

#### 유형 C. 리소스 절약·빠른 속도형 (maxSurge: 0, maxUnavailable: 1 이상)

**영향**: 새 Pod를 띄울 여유 자원이 부족한(클러스터 리소스가 꽉 찬) 환경에서 유용하지만, 배포 중 가용성이 일시적으로 떨어진다.

**동작**:
1. 초과 Pod를 만들지 않고(`maxSurge: 0`), 기존 Pod를 먼저 종료(`maxUnavailable`)한다.
2. 종료로 생긴 빈자리에 새 Pod를 올린다.

**단점**: 업데이트 도중 가용 Pod 수가 줄어들므로 트래픽이 몰릴 경우 에러가 발생하거나 응답 속도가 느려질 수 있다.

---

### 설정 유형 한눈에 비교

| 유형 | maxSurge | maxUnavailable | 가용성 | 리소스 소모 | 배포 속도 |
|---|---|---|---|---|---|
| **기본값** | 25% | 25% | 보통 | 보통 | 보통 |
| **무중단 우선** | 25%+ | 0 | 최고 | 높음 | 느림 |
| **리소스 절약** | 0 | 1 이상 | 낮음 | 낮음 | 빠름 |

### Production 환경 권장 가이드

**서비스 가동률과 무중단이 최우선인 경우**

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 25%        # 또는 1~2 (절대값)
    maxUnavailable: 0
```

트래픽을 처리하는 용량이 배포 중에 절대 줄어들지 않도록 보장한다. 단, `readinessProbe`가 올바르게 설정되어 있어야 새 Pod가 완전히 준비된 뒤 구형 Pod가 내려간다.

**리소스가 부족한 소규모 클러스터인 경우**

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 0
    maxUnavailable: 1
```

리소스를 초과해서 쓰지 않도록 막는다. 단, 배포 중 서비스 성능 저하 가능성을 인지하고 **트래픽이 적은 시간대에 배포**하는 것을 권장한다.
