### 1.1 GitHub Actions Workflow YAML

#### 1.1.1 백엔드 워크플로 (`be-cicd.yml`)

```yaml
name: Backend CI/CD

on:
  push:
    branches: [ "main" ]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      # 소스 코드 가져오기
      - name: Checkout Source Code
        uses: actions/checkout@v3

      # AWS 자격 증명 설정
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v2
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ap-northeast-2

      # AWS ECR 로그인
      - name: Login to Amazon ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v1

      # Docker 이미지 빌드 및 ECR 푸시 (Dockerfile 적용)
      # JDK 설치와 Gradle 빌드는 Dockerfile 내부에서 알아서 수행됩니다
      - name: Build, tag, and push image to Amazon ECR
        env:
          ECR_REGISTRY: ${{ steps.login-ecr.outputs.registry }}
          ECR_REPOSITORY: lulu-backend
          IMAGE_TAG: latest
        run: |
          docker build -t $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG .
          docker push $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG

      # 배스천을 경유하여 프라이빗 EC2들에 배포 (CD)
      - name: Deploy to Multiple Private EC2s via Bastion
        uses: appleboy/ssh-action@master
        with:
          host: ${{ secrets.BE_PRIVATE_IPS }} # 콤마(,)로 구분해놓음
          username: ubuntu
          key: ${{ secrets.EC2_SSH_KEY }}
          proxy_host: ${{ secrets.BASTION_IP }}
          proxy_username: ubuntu
          proxy_key: ${{ secrets.EC2_SSH_KEY }}
          script: |
            aws ecr get-login-password --region ap-northeast-2 | docker login --username AWS --password-stdin ${{ steps.login-ecr.outputs.registry }}
            cd ~/backend
            docker compose pull
            docker compose up -d
```

**단계별 설명**

| 항목 | 설명 |
| --- | --- |
| `name: Backend CI/CD` | 워크플로 이름. Actions 탭에서 실행 내역을 이 이름으로 확인 가능 |
| `on: push / branches: [ "main" ]` | `main` 브랜치에 push(직접 push 또는 merge)될 때만 파이프라인 실행 |
| `jobs: build-and-deploy` | 빌드, 이미지 푸시, 서버 배포를 하나의 Job으로 단순화 |
| `runs-on: ubuntu-latest` | 최신 Ubuntu가 설치된 GitHub 가상 러너에서 작업 실행 |
| `Checkout Source Code` (`actions/checkout@v3`) | 현재 레포지토리 소스 코드를 워크플로 가상 환경으로 가져옴 |
| `Configure AWS credentials` (`aws-actions/configure-aws-credentials@v2`) | GitHub Secrets에 저장된 IAM 계정 Access/Secret Key로 AWS 인증 수행 |
| `Login to Amazon ECR` (`aws-actions/amazon-ecr-login@v1`) | 인증된 자격 증명으로 프라이빗 ECR 레지스트리에 로그인 |
| `Build, tag, and push image to Amazon ECR` | 백엔드 코드를 Docker 이미지로 빌드 후 `latest` 태그로 ECR에 푸시 |
| `Deploy to Multiple Private EC2s via Bastion` (`appleboy/ssh-action@master`) | 배스천을 경유해 타겟 서버에 SSH 접속, ECR pull 및 `docker compose up -d`로 재배포 |

**설계 의도**

- **AWS 전용 인증 단계 도입 이유**: 퍼블릭 Docker Hub 대신 프라이빗 ECR을 사용해 이미지 외부 유출을 막고, AWS 백본망을 통한 배포 속도를 높이기 위함.
- **다중 서버 배포**: `host: ${{ secrets.BE_PRIVATE_IPS }}`에 IP 2개를 콤마로 입력하여, 반복문이나 스크립트 중복 없이 클러스터 내 모든 서버가 동시에 최신 상태로 배포되도록 함.
- **플러그인 + Proxy 사용 이유**: 인프라 보안을 위해 타겟 서버가 프라이빗 서브넷에 존재. 직접 셸 스크립트로 구현하면 복잡도가 매우 높아지므로 `appleboy/ssh-action`의 `proxy_host` 기능으로 배스천을 경유해 내부망에 안전하게 명령을 전달하도록 설계.

#### 1.2.2 프론트엔드 워크플로 (`fe-cicd.yml`)

```yaml
name: Frontend CI/CD

on:
  push:
    branches: [ "main" ] 

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Source Code
        uses: actions/checkout@v3

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v2
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ap-northeast-2

      - name: Login to Amazon ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v1

      # CI: Node.js 세팅이나 npm install 없이 바로 도커 빌드를 수행
      # Dockerfile 안에서 npm ci 가 실행됨
      - name: Build, tag, and push image to Amazon ECR
        env:
          ECR_REGISTRY: ${{ steps.login-ecr.outputs.registry }}
          ECR_REPOSITORY: lulu-frontend
          IMAGE_TAG: latest
        run: |
          docker build -t $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG .
          docker push $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG

      # CD: 배스천을 경유하여 프라이빗 프론트엔드 EC2들에 동시 배포
      - name: Deploy to Multiple Private EC2s via Bastion
        uses: appleboy/ssh-action@master
        with:
          host: ${{ secrets.FE_PRIVATE_IPS }} # 프론트엔드 1번, 2번 IP (콤마로 구분)
          username: ubuntu
          key: ${{ secrets.EC2_SSH_KEY }}
          proxy_host: ${{ secrets.BASTION_IP }}
          proxy_username: ubuntu
          proxy_key: ${{ secrets.EC2_SSH_KEY }}
          script: |
            aws ecr get-login-password --region ap-northeast-2 | docker login --username AWS --password-stdin ${{ steps.login-ecr.outputs.registry }}
            cd ~/frontend
            docker compose pull
            docker compose up -d
```

**단계별 설명**

| 항목 | 설명 |
| --- | --- |
| `name: Frontend CI/CD` | 워크플로 이름 |
| `on: push / branches: [ "main" ]` | `main` 브랜치 push 시 파이프라인 실행 |
| `runs-on: ubuntu-latest` | 최신 Ubuntu 환경에서 작업 실행 |
| `Checkout Source Code` | 레포지토리의 프론트엔드 코드를 워크플로 환경으로 가져옴 |
| `Configure AWS credentials & Login to Amazon ECR` | 백엔드와 동일하게 AWS 인증 및 ECR 로그인 수행 |
| `Build, tag, and push image to Amazon ECR` | 최신 프론트엔드 코드로 Docker 이미지 빌드 후 ECR 푸시 |
| `Deploy to Multiple Private EC2s via Bastion` | `FE_PRIVATE_IPS`를 대상으로 2대의 프론트엔드 서버에 동시 접속, `~/frontend`에서 `docker compose pull` → `up -d` |

**설계 의도 (환경 분리)**

일반적인 Node 파이프라인에 있는 `actions/setup-node`나 `npm install` 과정을 과감히 생략했다. `Dockerfile` 내부에 `npm ci --omit=dev` 실행 과정을 포함시켜 이미지가 자체적으로 패키지 의존성을 해결하도록 설계했기 때문. 이로써 외부 환경 설정 요소를 제거해 파이프라인 유지보수성을 극대화했다.

## 2. 회고

### 2.1 컨테이너 실행 방식: `docker run` vs `Docker Compose`

**문제 상황**

지금까지는 컨테이너마다 `docker run`으로 개별 옵션(포트, 환경변수, 재시작 정책)을 직접 입력해왔다. CI/CD 파이프라인이 서버에 접속해 배포 명령을 내려야 하는데, 매번 긴 `docker run` 커맨드를 스크립트에 박아두면 유지보수가 어렵다.

**선택지 비교**

| 선택지 | 설명 | 장점 | 단점 |
| --- | --- | --- | --- |
| `docker run` 개별 실행 | 컨테이너마다 CLI 옵션을 직접 나열 | 별도 파일 불필요, 단순 케이스에 빠름 | 옵션이 길어지면 가독성↓, 배포 스크립트에 중복 로직 발생, 버전 관리 어려움 |
| **Docker Compose** | YAML로 서비스 정의(이미지, 포트, env, 재시작 정책 등)를 선언 | 선언적 관리, `docker compose pull && up -d` 두 줄로 배포 완결, 인프라를 코드(IaC)로 관리 가능 | 컴포즈 파일 자체를 서버에 미리 배치해둬야 함, 멀티 컨테이너 오케스트레이션(스케일링 등)에는 한계 |

**결론**

**Docker Compose를 채택.** CI/CD 스크립트 안에서는 `docker compose pull` → `docker compose up -d` 두 명령만 실행하면 되도록 만들어, 배포 로직을 서버 쪽 `compose.yaml`과 파이프라인 쪽 스크립트로 관심사를 분리했다. `.env` 파일로 이미지 태그·시크릿을 주입해 `docker run -e` 방식보다 관리 포인트를 줄였다.

### 2.2 CI 도구 선택: GitHub Actions vs Jenkins

과제 내용 자체가 GitHub Actions를 사용하는 것이었지만, Jenkins와 장단점을 비교한 뒤 선택 이유를 따로 정리했다.

| 선택지 | 장점 | 단점 |
| --- | --- | --- |
| Jenkins | 커스터마이징 자유도 높음, 플러그인 생태계 풍부, 온프레미스 통제 용이 | 별도 서버(EC2) 운영·관리 비용 발생, 초기 설정 복잡, 강의에서 다루지 않음 |
| **GitHub Actions** | 레포지토리와 완전히 통합(별도 서버 불필요), `Actions` 탭에서 바로 실행 이력 확인, YAML로 선언적 워크플로 작성, 마켓플레이스 액션 재사용 | GitHub 종속적, 대규모 조직에서는 실행 시간 과금·러너 리소스 제약 존재 |

**결론**

**GitHub Actions 채택.** 현재 커리큘럼에서 다룬 도구이기도 하고, 별도 CI 서버를 운영할 필요 없이 코드 관리부터 빌드·배포까지 하나의 플랫폼에서 처리할 수 있다는 점이 실질적인 이유였다.

### 2.3 워크플로 권한 설정: Read/Write vs Read Only

(+ 개별 permissions)

**문제 상황**

교재에는 Settings → Actions → General에서 `Read and write permissions`를 선택하라고 안내되어 있었다.

**선택지 비교**

| 선택지 | 설명 | 장점 | 단점 |
| --- | --- | --- | --- |
| Read and write | 모든 워크플로가 기본적으로 저장소 쓰기 권한 보유 | 별도 설정 없이 코드 푸시·릴리즈 생성 등 어떤 작업도 바로 가능 | 워크플로가 탈취되거나 악성 서드파티 Action을 실수로 사용할 경우, 코드 임의 수정·삭제까지 가능해짐 (공격 표면 확대) |
| **Read only (기본값) + 필요 시 워크플로별 `permissions:`** | 저장소 기본 권한은 read-only로 최소화, 쓰기 권한이 필요한 워크플로만 `.yml` 내부에 개별 명시 | 최소 권한 원칙 준수, 사고 발생 시 피해 범위 제한 | 워크플로마다 필요한 권한을 직접 파악해서 명시해야 하는 번거로움 |

**결론**

**기본값(Read only)을 유지**했다. 지금 만든 CI/CD 워크플로는 코드를 수정·푸시하거나 릴리즈를 생성하지 않고 "빌드 → ECR 푸시 → SSH 배포"만 수행하므로 쓰기 권한이 필요 없었다. 나중에 워크플로가 코드를 커밋하거나 릴리즈를 만들어야 하는 경우가 생기면, 저장소 전체 설정이 아니라 **해당 워크플로 파일 안에서만** `permissions: contents: write` 처럼 필요한 범위만 부여하기로 했다.

### 2.4 AWS 인증 방식: Root/관리자 키 vs IAM 최소 권한 사용자

**문제 상황**

GitHub Actions가 ECR에 이미지를 푸시하려면 AWS 자격 증명이 필요하다. 가장 빠른 방법은 Root 계정이나 AdministratorAccess가 붙은 키를 그대로 쓰는 것이지만, 이 키가 GitHub Secrets에서 유출되면 인프라 전체가 위험해진다.

**선택지 비교**

| 선택지 | 장점 | 단점 |
| --- | --- | --- |
| Root/관리자 권한 키 | 설정이 빠르고, 어떤 AWS 작업이든 실패할 일이 없음 | 키 유출 시 EC2, S3, IAM 등 전 리소스가 탈취 대상이 됨. 사고 범위 예측 불가 |
| **전용 IAM 사용자 + `AmazonEC2ContainerRegistryPowerUser` 정책** | ECR 접근만 가능하도록 권한 범위가 한정됨, 키가 유출돼도 피해가 컨테이너 레지스트리로 국한됨 | 필요한 정책을 매번 찾아서 최소 단위로 부여해야 하는 설계 비용 |

> 📸 IAM 사용자 및 정책 설정 스크린샷 (원본 문서 참고: `image.png`)

**결론**

**GitHub Actions 전용 IAM 사용자(`github-actions-deploy`)를 별도 생성**하고 ECR 관련 권한만 가진 정책을 연결했다.

> 💡 **다음 개선 과제**: 지금은 장기 발급된 Access Key를 GitHub Secrets에 저장하는 방식이지만, **GitHub OIDC(OpenID Connect) + AWS IAM Role**을 사용하면 아예 장기 키를 저장하지 않고 워크플로 실행 시점에만 단기 토큰을 발급받아 인증할 수 있다. 지금 방식보다 보안이 한 단계 더 높은 대안이라, 다음 개선 과제로 남겨둔다.

### 2.5 민감 정보 관리: 하드코딩 vs GitHub Secrets

**문제 상황**

AWS 키, EC2 접속용 PEM 키, DB 비밀번호, JWT Secret 등 민감 정보를 어디에 둘 것인가.

**선택지 비교**

| 선택지 | 장점 | 단점 |
| --- | --- | --- |
| 소스코드/`.env` 파일을 레포에 직접 커밋 | 별도 설정 없이 바로 사용 가능 | 퍼블릭/프라이빗 여부와 무관하게 이력에 남아 유출 위험, 과금 폭탄·서버 해킹으로 직결 |
| **GitHub Secrets** | 레포지토리 설정에 암호화 저장, 워크플로 로그에도 마스킹되어 노출되지 않음, `${{ secrets.X }}`로 안전하게 주입 | 로컬 개발 환경과는 별도로 관리해야 함(동기화 필요) |

**결론**

**GitHub Secrets 채택.** `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `EC2_SSH_KEY`, `BASTION_IP`, `FE_PRIVATE_IPS`, `BE_PRIVATE_IPS`를 Settings → Secrets and variables → Actions에 등록했고, 워크플로에서는 참조만 하도록 했다.

### 2.6 다중 서버 배포 및 프라이빗 네트워크 접근

#### 2.6.1 다중 EC2 배포 전략

FE, BE 모두 EC2를 각각 2대씩 운영하고 있다. 반복문 없이 `appleboy/ssh-action`의 `host:` 값에 IP를 콤마로 나열(`FE_PRIVATE_IPS`, `BE_PRIVATE_IPS`)해서, 별도 매트릭스 전략(`strategy: matrix`) 없이도 한 스텝에서 모든 서버에 동시 접속해 배포되도록 구성했다.

> ⚠️ **주의**: 다만 이 방식은 두 서버 모두 **정확히 같은 스크립트를 병렬로 실행**한다는 뜻이라, 진짜 "무중단 배포"라고 하려면 두 서버가 동시에 내려갔다 올라오는 순간(재시작 텀)에는 순간적으로 무중단이 아닐 수 있다. 완전한 무중단을 원한다면 두 서버를 순차적으로(하나씩) 배포하거나, 로드밸런서 헬스체크와 연동해 한 대씩 트래픽에서 빼고 배포하는 **롤링 배포** 전략이 더 정확한 표현이라는 점을 짚어두고 싶었다.

#### 2.6.2 프라이빗 서브넷 접근: 직접 스크립트 구현 vs Bastion + `ssh-action` proxy 옵션

| 선택지 | 장점 | 단점 |
| --- | --- | --- |
| GitHub Actions에서 SSH 스크립트를 직접 작성해 이중 홉 처리 | 외부 플러그인 의존성 없음 | ProxyJump 설정, known_hosts 처리 등을 셸로 직접 구현해야 해서 복잡도가 매우 높음 |
| **`appleboy/ssh-action`의 `proxy_host`/`proxy_key` 옵션** | 배스천 경유 접속을 옵션 몇 줄로 처리, 유지보수 쉬움 | 서드파티 액션에 대한 신뢰가 전제되어야 함(공급망 리스크) |

**결론**

보안을 위해 FE/BE EC2를 프라이빗 서브넷에 두고, 배스천 서버(퍼블릭 서브넷)를 경유하도록 설계했다. 직접 구현하면 복잡도가 급격히 올라가므로 `appleboy/ssh-action`의 proxy 기능을 활용했다.

> ⚠️ **서드파티 액션 리스크**: 현재 `uses: appleboy/ssh-action@master`처럼 브랜치를 그대로 참조하고 있는데, 이는 액션 관리자가 `master` 브랜치에 어떤 코드를 올려도 다음 실행부터 그대로 반영된다는 뜻이라 공급망 공격에 취약할 수 있다. `@v0.1.10`처럼 **특정 버전으로 고정**하는 것이 안전해서 추후 특정 버전으로 고정할 예정이다.

### 2.7 GitHub Actions 러너 방식

GitHub Actions 러너 방식은 크게 **GitHub 호스트 러너(GitHub-hosted runner)**와 **셀프 호스팅 러너(Self-hosted runner)** 두 가지로 나뉜다.

**1. GitHub 호스트 러너 (GitHub-hosted runners)**

GitHub가 관리하는 가상 머신(VM)에서 작업을 실행한다. 초기 설정이 필요 없어 아주 편리하다.

- **장점**
  - 편리함: 별도의 서버 구축이나 OS 업데이트가 필요 없음
  - 깨끗한 환경: 작업이 끝날 때마다 가상 환경이 완전히 초기화되어 보안상 안전
  - 다양한 OS 지원: Windows, macOS, Ubuntu 등 손쉬운 선택
- **단점**
  - 비용 발생: 무료 시간 초과 시 분당 요금 청구
  - 커스텀 제한: 사내 네트워크망 접근이나 특수 소프트웨어 사전 설치가 어려움

**2. 셀프 호스팅 러너 (Self-hosted runners)**

내가 직접 관리하는 서버, 클라우드(AWS, GCP 등), 또는 쿠버네티스 환경에 러너를 설치한다.

- **장점**
  - 비용 절약: 러너 이용 요금 자체는 무료 (대규모 작업일수록 유리)
  - 높은 자유도: 원하는 사양의 컴퓨터 사용, 사내망 접근 및 필요한 도구 자유 설치
  - 캐싱 효과: 이전 작업 데이터를 남겨 빌드 속도 단축 가능
- **단점**
  - 유지보수 필요: 서버 보안, OS 업데이트, 러너 프로그램 관리를 직접 수행
  - 보안 주의: 작업 후에도 환경이 초기화되지 않아, 퍼블릭 저장소에서 사용 시 악의적 코드 실행 위험

**결론**

GitHub 호스트 러너 사용 시 관리 주체가 GitHub이고, 현재 무료 플랜으로 매달 2,000분이 초기화되어 제공되므로 추가 비용이 발생하지 않는다. 셀프 호스팅 러너는 서버 보안·OS 업데이트·러너 프로그램 관리를 직접 수행해야 하고 AWS 인스턴스 추가로 비용이 들기 때문에, **GitHub 호스트 러너 방식을 채택**했다.

### 2.8 트러블슈팅: 배스천 접속 타임아웃

**증상**

```
2026/07/12 14:05:19 dial tcp ***:22: i/o timeout
Error: Process completed with exit code 1.
```

**원인 분석**

배스천의 보안 그룹(Security Group)이 특정 IP 대역만 허용하고 있었기 때문에, GitHub 러너의 IP가 차단되어 22번 포트 연결 자체가 성립하지 않은 것이었다.

GitHub Actions 러너의 IP는 사용 방식에 따라 다르다.

1. **GitHub 호스트 러너 (클라우드)**: 마이크로소프트 클라우드 인프라(Azure) IP를 공유하므로 IP가 매번 바뀐다. 고정된 IP가 아니며, 현재 사용 중인 전체 IP 주소 범위는 GitHub Meta API 문서의 `actions` 키에서 확인 가능.
2. **셀프 호스트 러너 (로컬)**: 서버나 가상 머신(AWS EC2 등)에 러너를 직접 설치하는 방식. 서버의 IP가 곧 러너의 IP가 되어 고정되므로 방화벽 화이트리스트 관리가 쉬움.

**해결**

현재는 GitHub 호스트 러너를 사용하고 있기 때문에 배스천 보안 그룹의 SSH(22) 인바운드 규칙 소스를 `0.0.0.0/0`으로 변경해 GitHub Actions 러너가 어떤 IP에서 접속하든 허용되도록 조정했다.

| GitHub Actions 러너 | GitHub-hosted runners | 관리의 주체가 GitHub이고, 현재 무료 플랜 이용 가능 |

### 2.10 남은 과제 / 다음 개선 방향

1. AWS 인증을 Access Key 대신 **OIDC + IAM Role**로 전환해 장기 키 자체를 없애기
2. FE/BE 2대 배포를 병렬이 아닌 **순차(롤링) 배포**로 바꿔 진짜 무중단 배포 구현
3. `appleboy/ssh-action@master` → 특정 버전(가능하면 커밋 SHA)으로 고정
4. 배스천 보안 그룹을 `0.0.0.0/0` 대신 GitHub Actions IP 대역 또는 SSM으로 대체
5. (선택 항목) BE 테스트 코드 작성 및 워크플로에 테스트 단계 추가 — 현재 미포함
