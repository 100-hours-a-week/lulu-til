#### | Zero-downtime Deployment (무중단 배포)

**사용자 입장에서 서비스 중단 없이 애플리케이션을 업데이트하거나 배포하는 방식**

##### 중단 배포 vs 무중단 배포

**중단 배포 (기본 배포 방식)**

```
기존 서버 종료 → 새 코드 배포 → 새 서버 실행
```

문제는 **기존 컨테이너가 종료되고 새 컨테이너가 실행되기까지, 요청을 처리할 서버가 존재하지 않는 시간(다운타임)** 이 생긴다는 것이다.

**무중단 배포**

> ⚠️ 오해하기 쉬운 부분: 무중단 배포는 "서버(컨테이너)가 절대 멈추지 않는다"는 뜻이 아니다. 배포 전략에 따라 서버나 컨테이너 자체는 얼마든지 중단될 수 있다. 핵심은 **그 중단이 사용자의 요청 처리를 중단시키지 않는 형태**라는 것이다.

```
기존 서버가 종료되기 전에 새 서버가 먼저 준비된다
      ↓
새 버전을 먼저 준비 → 정상 동작 확인 → 트래픽 전환 → 기존 버전 종료
```

```
                 ┌──→ 기존 버전 V1
사용자 요청 → 로드밸런서 / 리버스 프록시
                 └──→ 새 버전 V2
```

##### 무중단 배포가 꼭 필요할까?

현대 웹 서비스는 대부분 "항상 동작하는 것"을 기본 전제로 한다. 또한 국내(KR) 트래픽만이 아니라 EN, JP, CN 등 글로벌 트래픽을 받도록 서비스가 확장되는 경우가 많아, 서비스가 지속적으로 돌아가야 할 필요성이 커졌다.

| 구분 | 특징 | 예시 |
|---|---|---|
| **중단 배포가 적합한 서비스** | 한 번에 크게 변경이 필요하거나, 변경에 대한 철저한 검증이 필요한 서비스 | 게임 서버, 은행 (메이저 버전 변경 시) |
| **무중단 배포가 필요한 서비스** | 지속적으로 유저가 들어와야 하는 서비스 | 커머스, SaaS, API 서비스, 실시간 서비스 |

> 💡 무조건 무중단 배포가 정답은 아니다. 서비스 성격에 따라 **의도적으로 중단 배포를 선택하는 것도 전략**이다.

**왜 무중단을 하는가** → 안정적인 전환, 빠른 복구, 지속적인 관측(모니터링)을 위해서다.

##### 좋은 배포를 위해 필요한 요구사항

| 요구사항 | 설명 |
|---|---|
| **가용성** | 배포 중에도 사용자가 서비스를 이용할 수 있어야 함 |
| **안정성** | 새 버전이 정상인지 확인한 뒤 트래픽을 받아야 함 |
| **롤백 가능성** | 문제가 생기면 이전 버전으로 빠르게 돌아갈 수 있어야 함 |
| **관측 가능성** | 오류율, 응답 시간, CPU, 메모리 등을 확인할 수 있어야 함 |
| **자동화** | 사람의 수동 작업과 실수를 줄여야 함 |
| **재현성** | 누가 배포해도 동일한 절차와 결과가 나와야 함 |

##### 무중단 배포의 구성요소

| 구성 요소 | 역할 |
|---|---|
| **다중 인스턴스 / 다중 컨테이너** | 기존 버전과 새 버전을 동시에 실행 |
| **로드밸런서 / 리버스 프록시** | 사용자 요청을 정상 서버로 분산 |
| **헬스 체크** | 새 버전이 실제 요청을 받을 준비가 되었는지 확인 |
| **트래픽 전환 정책** | 기존 버전과 새 버전 사이의 요청 비율을 조절 |
| **롤백 전략** | 새 버전에 문제가 생기면 이전 버전으로 복귀 |
| **모니터링** | 배포 후 오류율·응답 시간·리소스 상태 확인 |
| **CI/CD 파이프라인** | 빌드·테스트·배포 과정을 자동화 |

##### 기존 요청은 어떻게 처리되는가 — Connection Draining / Graceful Shutdown

새 버전으로 트래픽을 전환한다고 해서 기존 서버를 바로 종료하면 안 된다. 만약 기존 서버가 파일 업로드, 대용량 다운로드, 결제 요청처럼 **오래 걸리는 요청을 처리 중**이라면 문제가 생긴다.

이때 필요한 개념이 **Connection Draining (또는 Graceful Shutdown)** 이다.
- 기존 서버는 **새로운 요청은 더 이상 받지 않는다**
- 하지만 **이미 처리 중인 요청은 끝까지 마무리할 시간을 준다**

```
1. 새 버전 실행
2. 새 버전 헬스 체크 통과
3. 새 버전에 트래픽 전달 시작
4. 기존 버전에는 새 요청 전달 중단
5. 기존 요청 완료 대기 (Draining)
6. 기존 버전 종료
```

> 💡 ECS, EKS 같은 관리형 서비스는 기본적으로 Graceful Shutdown 타임(예: 300초)이 설정되어 있는 경우가 많다. 단순히 "새 서버가 떴네?" 하고 바로 기존 서버를 죽이는 방식은 무중단 배포가 아니다.

##### 무중단 배포 전략 한눈에 비교

| 전략 | 핵심 개념 | 특징 |
|---|---|---|
| **Blue/Green** | 두 환경을 모두 준비하고 트래픽을 한 번에 전환 | 빠른 전환·빠른 롤백 |
| **Rolling** | 기존 인스턴스를 순차적으로 새 버전으로 교체 | 구조가 비교적 단순, 추가 자원 적음 |
| **Canary** | 일부 사용자에게만 새 버전을 먼저 제공 | 실사용 환경에서 위험 최소화 |

#### | Blue/Green 배포

**두 개의 독립된 환경(Blue, Green)을 동시에 운영하며, 트래픽을 새 버전으로 한 번에 전환해 서비스 중단 없이 배포와 롤백이 가능한 방식**

```
로드밸런서 ──점선(대기/종료)──→ Blue  (v1, 종료 또는 대기)
        └──실선(트래픽 수신)──→ Green (v2, 운영 중)
```

> 💡 Blue가 항상 기존 버전, Green이 항상 새 버전인 것은 아니다. 두 이름은 고정된 게 아니라 **왔다갔다 하는 역할**에 가깝다. 트래픽을 받고 있는 환경과 대기 중인 환경을 항상 분리해서 운영하는 것이 핵심이다.

```
# 현재
app-blue  # 1.0.1  ← 트래픽 수신 중

# 여기서 배포 시작
app-green # 1.0.2  ← 새 버전 준비, 대기 상태로 실행

# 검증 완료 후 트래픽 전환
app-blue  # 1.0.3  ← 다음 배포 때는 blue가 새 버전이 됨
```

##### 장점

| 장점 | 설명 |
|---|---|
| **롤백이 쉽다** | 대기 중이던 이전 버전으로 리버스 프록시 경로만 되돌리면 끝 (예: `nginx -s reload`) |
| **배포 구조가 단순** | 현재 환경과 신규 환경이 완전히 분리되어 있어 과정이 직관적 |
| **장애 영향 최소화** | 사전 검증 + 빠른 롤백으로 서비스 영향을 최소화 |

블루/그린이 유리한 상황: 결제·주문처럼 장애 영향이 큰 서비스, API 변경 위험이 큰 배포, 신규 기능 릴리즈, 대규모 리팩터링, 인프라 설정 변경 등 **롤백이 필요할 수 있고, 필요하다면 즉시 롤백해야 하는 상황**에서 강점을 가진다.

##### 단점

1. **리소스 비용이 크다** — 기존 구성을 그대로 이중화해야 한다 (Blue EC2 10대라면 Green도 10대 준비).
2. **이중 운영 위험** — 안정화가 끝났다고 판단해 Blue를 정리하지 않으면 리소스가 이중으로 계속 사용된다.
3. **DB 변경이 까다롭다** — 두 환경이 동일한 스키마를 바라봐야 하기 때문에 스키마 변경 시 별도 전략이 필요하다.

##### 실무에서 자주 하는 실수

1. **Green이 실행되자마자 트래픽을 전환** → "실행됨"과 "정상 작동함"은 다르다. 반드시 헬스 체크를 거쳐야 한다.
2. **트래픽 전환 직후 Blue를 바로 종료** → 헬스체크 통과가 곧 안정성을 의미하지 않는다. 일정 시간 지표를 관찰한 뒤 종료해야 한다.
3. **롤백 기준을 배포 이후에 정함** → 어떤 지표가 어떤 수치에 도달하면 롤백할지 **미리 정의하고 코드로 작성**해야 한다 (예: HTTP 5xx 비율 5% 초과 시 롤백).
4. **Blue/Green 환경 설정이 서로 다름** → 환경 변수, 리소스 설정, 도커 네트워크, CPU, IAM/Role 등 설정이 다르면 예측 불가능한 문제가 생긴다.

#### | Rolling Deployment

**기존 인스턴스를 하나씩 새로운 버전으로 순차 교체하며 전체 서비스를 점진적으로 배포하는 방식**

인프라를 두 배로 준비하지 않아도 배포 중 서비스의 처리 용량을 어느 정도 유지할 수 있어서 사용한다.

```
Blue/Green:  Blue 10대 + Green 10대  (동시에 20대 필요)
Rolling:     v1 10대 → 9대(v1) + 1대(v2) → ... → 0대(v1) + 10대(v2)
```

##### 진행 과정

```
1. V1 인스턴스 중 하나를 리버스 프록시에서 제거
2. 해당 V1 인스턴스 종료, V2 인스턴스 실행
3. V2 정상 동작(헬스 체크) 확인
4. 리버스 프록시에 V2 등록
5. 위 과정을 나머지 인스턴스에 반복
```

> 💡 직접 구현하기엔 난이도가 있어 AWS ECS, Kubernetes(k8s), EKS 같은 관리형 서비스의 기능을 활용하는 경우가 일반적이다.

##### 방식 비교: In-place vs Side-by-side

| 비교 항목 | In-place 방식 | Side-by-side 방식 |
|---|---|---|
| 핵심 | 기존 인스턴스를 하나씩 종료한 뒤 같은 자리에서 새 버전 실행 | 새 인스턴스를 먼저 추가한 뒤 기존 인스턴스를 제거 |
| 새 버전 실행 시점 | 기존 서버 제거 후 실행 | 새 서버 먼저 추가 |
| 추가 리소스 | 거의 필요 없음 | 일시적으로 필요 |
| 배포 중 처리 용량 | 감소할 수 있음 | 비교적 유지 |
| 배포 안정성 | 상대적으로 낮음 | 상대적으로 높음 |
| 롤백 속도 | 이전 버전을 재배포해야 함 | 기존 버전이 남아있을 수 있음 |
| 적합한 환경 | 소규모·리소스 제한 환경 | 트래픽이 많고 안정성이 중요한 환경 |

선택 기준: 트래픽 유형(스파이크형 vs 스트레스형), 서비스 상황, 업데이트 소요 시간 등을 종합적으로 고려한다.

##### 장점

- **인프라 비용 절감**: Blue/Green 대비 동시에 유지해야 하는 리소스가 적다.
- **점진적 배포**: 문제가 생겨도 일부 인스턴스에만 영향이 국한된다.

##### 단점

1. **배포 시간이 길다** — Blue/Green처럼 "준비되면 한 번에 전환"이 아니라 순차적으로 진행된다.
2. **V1/V2가 동시에 혼재될 수 있다** — 같은 시점에 두 버전이 함께 요청을 처리하므로, 클라이언트가 V2 응답을 기대해도 V1이 응답할 수 있다. 이를 대비해 API 버저닝(`/api/v1/users/me`, `/api/v2/users/me`)을 적용한다.
3. **롤백 속도가 상대적으로 느리다** — 예를 들어 V2가 이미 10대 중 8대까지 교체된 상태에서 문제가 발견되면, 이미 종료된 8대의 V1을 다시 만들어야 한다.

#### | Blue/Green vs Rolling Deployment

| 비교 항목 | Blue/Green | Rolling Deployment |
|---|---|---|
| 배포 방식 | 환경을 두 개 준비 후 트래픽 전환 | 기존 인스턴스를 순차 교체 |
| 트래픽 전환 | 보통 한 번에 전환 | 인스턴스 교체 과정에서 자연스럽게 분산 |
| 추가 리소스 | 배포 중 상당히 필요 | 설정에 따라 일부만 필요 |
| 배포 속도 | 전환 자체는 빠름 | 전체 완료까지 시간이 걸릴 수 있음 |
| 롤백 속도 | 매우 빠름 | 이전 버전 재배포가 필요할 수 있음 |
| V1·V2 혼재 | 전환 전후로 비교적 분리 가능 | 배포 중 혼재 상태가 일반적 |

---

#### | Canary Deployment

**새 버전을 전체에 적용하기 전에, 일부 사용자나 트래픽에만 점진적으로 배포하여 안정성을 검증한 후 전면 배포하는 방식**

##### Canary vs Rolling — 목적이 다르다

| 구분 | Rolling Deployment | Canary Deployment |
|---|---|---|
| 핵심 목적 | 서버·Pod를 순차적으로 새 버전으로 교체 | 새 버전의 위험을 작은 사용자 집단에서 먼저 검증 |
| 기준 | 인스턴스 교체 순서 | 트래픽 비율 또는 사용자 그룹 |
| 새 버전 노출 | 새 인스턴스가 준비되면 자연스럽게 요청을 받음 | 의도적으로 일부 요청만 전달 |
| 배포 판단 | 새 인스턴스가 정상인지 확인 | 오류율·응답 시간·핵심 기능 지표를 보고 확산 여부 판단 |
| 대표 질문 | "몇 대씩 교체할까?" | "처음에 몇 %에게 노출할까?" |

> 정리하면, **롤링 배포는 인프라 교체 전략**이고 **카나리 배포는 사용자 영향 범위를 제어하는 전략**이다.

##### 트래픽/유저 제어 방식

- **트래픽 비율 제어**: 예) v1 90%, v2 10% → 요청이 확률적으로 v1 또는 v2로 분산된다.
- **유저 그룹 제어**: 클라이언트 트래킹 태그(디바이스 타입, OS 버전, 클라이언트 버전, 성별, 연령 등)를 기준으로 특정 유저 그룹만 고정적으로 새 버전을 사용하게 한다 → **A/B 테스트**와 연결되는 개념이다.

##### Bake Time

카나리 트래픽을 일정 시간 유지하면서, **새 버전의 실제 운영 지표를 관찰하는 시간**을 의미한다. 문제가 없다는 확신이 들 때까지 트래픽 비율을 점진적으로 늘려간다.

---

#### | 무중단 배포 시 고려해야 하는 사항

##### 1. 애플리케이션은 가능하면 Stateless하게 설계해야 한다

서버 메모리에 랭킹, 사용자 로그인, 쿼리 캐싱 같은 상태를 저장하면 서버가 교체될 때 그 상태가 사라져 문제가 생길 수 있다. 실무에서는 상태 값을 외부 저장소로 분리한다.

| 상태 데이터 | 일반적인 저장 위치 |
|---|---|
| 사용자 로그인 세션 | Redis |
| 사용자 데이터 | MySQL, PostgreSQL 등 DB |
| 업로드 파일 | S3, EFS, Object Storage |
| 캐시 | Redis, Memcached |
| 메시지 | Kafka, RabbitMQ, SQS |

##### 2. 데이터베이스 변경은 무중단이어도 문제가 많이 발생한다

DB 스키마 변경은 애플리케이션 배포보다 **훨씬 더 조심해야 한다**. 무중단 배포 중에는 V1과 V2가 동시에 실행되며, **둘 다 사용자의 요청을 정상적으로 처리해야 할 책임**이 있기 때문이다.

예시: `users` 테이블에서 `nickname` 컬럼을 제거하고 `display_name` 컬럼으로 바꾸는 경우
- V1: 기존 `nickname` 컬럼 사용
- V2: `nickname` 대신 `display_name` 컬럼 사용

```
1. display_name 컬럼 추가
2. V2가 nickname, display_name 모두 처리하도록 개발
3. 데이터 마이그레이션
4. 모든 서버를 V2로 전환 + V2가 안정적임을 확인
5. nickname 컬럼 제거
```

> ⚠️ `nickname` 컬럼을 미리(성급하게) 제거해버리면 V1이 아직 그 컬럼을 참조하고 있을 수 있어 대참사로 이어질 수 있다. **컬럼 제거는 반드시 가장 마지막 단계**여야 한다.

##### 3. 롤백 ≠ 배포 실패, 롤백 = 지표가 불안정할 때

새 버전이 실행되고 배포가 "성공"했다고 판단해 기존 버전 컨테이너와 DB를 곧바로 정리해버리면, 이후 v2에서 500 에러율이 갑자기 10%로 치솟아도 복구하기가 매우 어려워진다. **배포 직후 일정 기간은 반드시 관측(모니터링)해야 한다.**

| 관측 항목 | 확인 이유 |
|---|---|
| HTTP 5xx 비율 | 서버 오류 증가 여부 |
| 응답 시간 p95, p99 | 일부 사용자 요청 지연 여부 |
| CPU / 메모리 | 리소스 부족 여부 |
| DB 커넥션 수 | DB 과부하 여부 |
| 컨테이너 재시작 횟수 | 애플리케이션 비정상 종료 여부 |
| 핵심 비즈니스 지표 | 결제, 주문, 로그인 등의 성공률 확인 |

##### 4. 무중단 배포는 실행보다 롤백 계획이 더 중요하다

무중단 배포 자체는 생각보다 어렵지 않다. 진짜 어려운 부분은 **롤백을 얼마나 안정적이고 빠르게 할 수 있느냐**이다.

#### | 무중단 배포 정리

| 항목 | 핵심 내용 |
|---|---|
| 무중단 배포 | 사용자가 서비스를 이용하는 동안 새 버전으로 전환하는 배포 방식 |
| 해결하려는 문제 | 배포 중 발생하는 다운타임, 요청 실패, 사용자 불편 |
| 핵심 원칙 | 새 버전을 먼저 준비하고, 정상 확인 후 트래픽을 전환 |
| 필수 요소 | 다중 인스턴스, 로드밸런서·프록시, 헬스 체크, 롤백, 모니터링 |
| 기존 요청 처리 | Connection Draining, Graceful Shutdown으로 완료 대기 |
| 대표 전략 | Blue/Green, Rolling, Canary |

### 🗨️ 오늘의 회고

- 어제는 일요일에 밤을 새운 여파로 회복 겸 쉬어갔고, 오늘부터 다시 학습 흐름을 이어갔다. 수요일까지 이전 과제들을 수정·보완하고, 목요일부터 9주차 위클리 챌린지를 진행할 예정이다.
- 오늘은 "무중단 배포"라는 용어 자체는 익숙했지만, **"서버가 멈추지 않는 것"이 아니라 "서비스 요청이 중단되지 않는 것"** 이라는 정의 차이를 명확히 잡고 가니 이후 개념들이 훨씬 잘 연결됐다. 특히 Connection Draining/Graceful Shutdown이 왜 필요한지 이해하고 나니, 단순히 "새 서버 띄우고 헬스체크 통과하면 끝"이 아니라는 걸 깨달았다.
- Blue/Green, Rolling, Canary 세 전략의 차이를 "롤백 속도 vs 리소스 비용 vs 위험 범위 제어"라는 축으로 비교하니 각 전략이 왜 존재하는지가 훨씬 명확해졌다. 특히 롤링과 카나리를 헷갈렸었는데, **롤링은 인프라 교체 전략, 카나리는 사용자 영향 범위 제어 전략**이라는 구분이 인상 깊었다.
- DB 스키마 변경 파트에서 "컬럼을 미리 제거하면 대참사"라는 표현이 와 닿았다. V1과 V2가 동시에 운영되는 동안 두 버전 모두 정상적으로 요청을 처리해야 한다는 책임을 간과하기 쉬운데, 확장 → 마이그레이션 → 전환 → 정리 순서를 반드시 지켜야겠다는 걸 다시 새겼다.
- "무중단 배포는 실행보다 롤백 계획이 더 중요하다"는 말이 오늘 배운 내용 중 가장 실무적으로 느껴졌다. 배포 자체보다 **어떤 지표가 어떤 기준을 넘으면 롤백할지 미리 코드로 정의**해두는 습관을 들여야겠다.
- 별개로, 현재 Portainer에 자체 서명 인증서를 사용 중이라 브라우저 경고가 뜨는 문제가 있어 서브도메인(`portainer.도메인.com`)을 발급받아 Let's Encrypt/ACM 기반 인증서로 교체하는 것을 고려해야겠다. 더불어 AWS Secrets Manager 도입도 검토해볼 예정이다.
