#### | Volume

**컨테이너의 파일 시스템 외부에 별도의 저장 공간을 연결하여, 데이터를 보존하거나 Pod 내부 컨테이너끼리 공유할 수 있게 하는 데이터 저장소**

```
                POD
┌───────────┐  ┌───────────┐  ┌───────────┐
│ Container │  │ Container │  │ Container │
│ /var/logs │  │ /var/logs │  │ /var/data │
└─────┬─────┘  │ /var/data │  └─────┬─────┘
      │        └──┬─────┬──┘        │
      ▼           ▼     ▼           ▼
   [logs VOLUME]     [data VOLUME]
```

##### 볼륨이 필요한 이유

- 컨테이너는 자신의 파일 시스템에 데이터를 저장할 수 있다 (예: WAS Container 안의 `/data/logs/access.log`).
- 하지만 **컨테이너 안에 있는 파일 시스템은 기본적으로 컨테이너 생명 주기에 종속적**이다. 컨테이너가 종료되면 파일 시스템도 같이 사라진다.
- 하나의 파드 안에 여러 개의 컨테이너가 있을 수 있는데, **컨테이너 내부에 있는 파일 시스템은 같은 파드 내의 다른 컨테이너가 접근할 수 없다.**

##### 볼륨이 해결하는 문제

**1. 컨테이너가 재시작될 때 데이터를 유지**

볼륨은 동일한 파드 내에서 컨테이너가 재시작되더라도 데이터를 유지한다. 컨테이너가 재시작되더라도 볼륨은 **파드에 종속**되기 때문에, 컨테이너가 재시작해도 데이터가 유지된다.

**2. 파드 내부 컨테이너 간 데이터 공유**

```
WAS Pod
 ├── WAS container
 ├── Shared Volume
 │      └── access.log
 └── Logging Container
```

- 하나의 파드 안에서 여러 컨테이너가 동일한 볼륨을 서로 마운트해서, 파일을 통해 데이터를 공유할 수 있다.
- 예: 애플리케이션 컨테이너가 로그를 남기면, 로그 수집기 컨테이너가 이를 읽어서 중앙 로그 처리 장치로 전달한다.
- 볼륨을 사용하면 Pod 내 컨테이너들이 공유 파일 시스템을 사용하는 것처럼 데이터를 공유할 수 있다.

**3. 설정, Secret 파일로 전달**

볼륨을 데이터 저장 용도로만 사용하는 건 아니다. ConfigMap, Secret 볼륨을 연결하면, 쿠버네티스에 저장된 값을 컨테이너 내부의 파일로 제공할 때도 사용한다.

##### 볼륨을 사용하는 경우

- 임시 데이터
- 컨테이너 간 파일 공유
- 설정, 시크릿 값 전달할 때

##### Volume YAML

```yaml
spec:
  containers:
    - name: app
      image: nginx
      volumeMounts:
        - name: app-data
          mountPath: /data
  volumes:
    - name: app-data
      emptyDir: {}
```

| 필드 | 설명 |
|---|---|
| **volumes** | Pod가 사용할 공간을 정의하는 것 (위 코드: `app-data`라는 이름의 `emptyDir` Volume을 생성) |
| **volumeMounts** | 사용하는 볼륨을 컨테이너의 어느 경로에서 사용할지 정의 (위 코드: 컨테이너 내부에서는 볼륨을 `/data` 디렉토리로 접근) |

---

#### | Volume 유형: emptyDir / hostPath

##### emptyDir

**파드가 생성될 때 만들어지고, 파드가 삭제되면 함께 사라지는 임시 저장 공간**

```yaml
volumes:
  - name: temp-data
    emptyDir: {}
```

- 초기 상태는 비어 있고, 파드가 노드에 배치될 때 생성된다.
- 컨테이너가 재시작되면 → 데이터 유지
- 파드가 삭제되면 → 데이터 삭제

**주 용도**: 캐시, 임시 파일, 컨테이너 간 파일 공유

##### hostPath

**Pod가 실행되는 노드의 특정 파일이나 디렉토리를 컨테이너에 직접 연결하는 볼륨**

```yaml
volumes:
  - name: host-data
    hostPath:
      path: /data
      type: DirectoryOrCreate
```

```
Worker Node
  └ /data
     ↓
    Pod
     └ Container /mnt/data
```

- 노드의 `/data` 디렉토리를 파드에서 사용할 수 있게 한다.
- 저장되는 위치가 **파드가 실행되는 노드의 파일 시스템**이기 때문에, 파드가 삭제돼도 노드의 데이터는 남아 있다.
- 하지만 파드는 생성·삭제가 자유롭고 특정 노드에서만 만들어지지 않으므로, **다른 노드에서는 이전 노드의 데이터에 접근할 수 없다.**

**주 용도**: 노드의 로그, 시스템 파일, 런타임 소켓 접근

##### emptyDir vs hostPath

| 구분 | emptyDir | hostPath |
|---|---|---|
| 저장 위치 | Pod가 배치된 노드의 임시 저장 공간 | 노드의 지정된 경로 |
| 최초 상태 | 항상 빈 상태 | 기존 파일이 존재할 수 있음 |
| 컨테이너 재시작 | 유지 | 유지 |
| Pod 삭제 | 삭제 | 노드에는 남을 수 있음 |
| Pod가 다른 노드로 이동 | 기존 데이터 사용 불가 | 이전 노드의 데이터 사용 불가 |
| 주요 목적 | 임시 데이터, 컨테이너 간 공유 | 노드 파일 및 디렉토리 접근 |
| 일반 애플리케이션 데이터 저장 | 부적합 | 부적합 |

##### Volume의 한계점

**1. Volume은 영구 저장소가 아니다.** Volume에 저장되는 데이터가 영구적으로 저장되어 사용되기는 어렵다.

> ⚠️ "hostPath는 노드에 붙어 있으니까 영구히 저장되는 거 아닌가요?" → 그렇지 않다. **노드도 스케일링의 대상**이다. 워커노드가 부족하면 Cluster Autoscaler가 워커노드를 추가하고, 반대로 제거하기도 한다. 따라서 반드시 보존해야 하는 데이터에는 적합하지 않다.

**2. 노드에 종속적이다.** hostPath는 노드의 호스트 파일 시스템에, emptyDir는 파드(결국 노드)에 종속된다. 파드가 다른 노드에서 만들어지면 기존 볼륨을 사용할 수 없다.

**3. 스토리지 운영을 직접 해야 한다 (크리티컬!).** 볼륨은 노드에 종속되므로, 필요한 저장 용량, 여러 파드가 노드 스토리지를 나눠 쓰는 문제, 외부 스토리지 연결, 파드 삭제 후 데이터 처리 방식까지 전부 노드 단위로 직접 관리해야 한다.

→ 그래서 **Volume은 실무적으로 중요한 데이터를 처리할 때 사용하지 않는다.** 이 문제를 해결하기 위해 **PV/PVC + StorageClass**를 사용한다.

##### 실무에서 볼륨(emptyDir/hostPath)을 사용하는 경우

- 임시 캐시
- 임시 작업 공간
- 파드 내부 컨테이너끼리 데이터 공유
- ConfigMap, Secret 마운트
- 노드 로그 수집
- 모니터링 시스템 에이전트 역할

##### Volume 정리

| 구분 | 내용 |
|---|---|
| Volume | Pod가 사용할 저장 공간을 정의하는 구조 |
| 해결하는 문제 | 컨테이너 재시작 시 데이터 유지, 컨테이너 간 파일 공유 |
| emptyDir | Pod 수명 동안 사용하는 임시 저장소 |
| hostPath | 노드의 특정 경로를 Pod에 연결 |
| 공통 한계 | 노드 이동과 Pod 삭제 상황에서 영구성을 보장하기 어려움 |
| 영구 저장 필요 | PV, PVC, StorageClass 사용 |

#### | PV / PVC

##### 왜 필요한가

앞서 배운 볼륨의 한계를 다시 보면:
- **emptyDir**: 파드가 살아있는 동안만 유지되고, 파드가 사라지면 같이 삭제된다.
- **hostPath**: 특정 노드의 파일 시스템에만 연결되고, 다른 노드에서는 보이지 않는다. 파드는 여러 노드에서 사용될 수 있으므로 영구히 보존되어야 하는 데이터에는 적합하지 않다.

→ 어떤 파드, 어떤 노드에서든 접근 가능하고, **파드·노드의 생명주기와 상관없이 계속 보관되어야 하는 저장 공간**이 필요하다.

**PV/PVC는 파드와 저장 공간을 직접 연결하지 않는다.** 즉, 파드·노드가 죽어도 저장 공간은 그대로 살아 있도록 **생명 주기를 분리**한다. 저장소를 파드와 별개의 리소스로 만들어서 사용하는 방식이다.

```
기존 방식
Pod → 저장소의 위치, 종류를 직접 정의

PV/PVC
Pod → PVC ("영구 저장소를 갖고 싶어" 라고 원하는 상태를 정의)
PVC ↔ PV 를 연결해준다.
PV = Persistence Volume (클러스터에서 관리되고 있는 영구 저장소)
```

##### PV란?

**PersistentVolume — 클러스터에서 파드와 독립적으로 관리되는 저장 공간**

- 애플리케이션이 사용할 수 있도록 미리 준비된(혹은 동적으로 준비되는) 스토리지를 **쿠버네티스 리소스로 표현한 것.**
- 실제 저장소는 NFS, AWS EBS, 로컬 디스크 공간 등이며, 이를 쿠버네티스 리소스로 추상화한 것이 PV다.
- PVC를 통해 요청이 들어오면, 조건에 부합하는 PV를 할당해서 바인딩하는 형식으로 동작한다.

**PV의 역할**

| 역할 | 설명 |
|---|---|
| 저장 공간 제공 | Pod가 사용할 실제 저장소를 쿠버네티스 리소스로 표현 |
| Pod와 수명 분리 | Pod가 삭제되거나 교체되어도 저장소를 별도로 유지 가능 |
| 저장소 정보 캡슐화 | 저장소의 종류, 용량, 접근 방식 등을 PV에 정의 |
| 중앙 관리 | 클러스터 관리자가 저장 공간을 준비하고 관리 |

**PV YAML**

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: app-pv
spec:
  capacity:
    storage: 5Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: manual
  hostPath:
    path: /data/app
```

| 항목 | 설명 |
|---|---|
| **capacity** | 제공하는 저장 공간의 크기 |
| **accessModes** | 저장소에 접근할 수 있는 방식 |
| **storageClassName** | PV가 속한 스토리지 종류 또는 정책 |
| **persistentVolumeReclaimPolicy** | PVC 사용 종료 후 PV를 어떻게 처리할지 결정 |
| 스토리지 종류 | NFS, CSI, local 등 실제 저장소 연결 정보 |

##### PVC란?

**PersistentVolumeClaim — 파드가 필요한 저장 공간의 조건을 작성한 요청서**

- PVC는 **저장소 자체가 아니다.** "다음과 같은 조건의 저장 공간이 필요합니다"라고 요청하는 리소스다.
- 요청하는 조건: 필요한 용량, 읽기·쓰기 방식, 사용할 스토리지 종류

**PVC 주요 역할**

| 역할 | 설명 |
|---|---|
| 저장 공간 요청 | 애플리케이션이 필요한 용량과 접근 방식을 선언 |
| 저장소 추상화 | 실제 저장소 구현을 몰라도 스토리지 사용 가능 |
| PV와 연결 | 요청 조건에 맞는 PV와 바인딩 |
| Pod에 제공 | Pod가 PVC를 Volume으로 참조하여 사용 |

**PVC YAML**

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: app-pvc
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: manual
  resources:
    requests:
      storage: 3Gi
```

| 설정 | 의미 |
|---|---|
| `ReadWriteOnce` | 하나의 노드에서 읽기·쓰기가 가능한 저장소 요청 |
| `manual` | `manual` StorageClass에 속한 PV 요청 |
| `storage: 3Gi` | 최소 3Gi 이상의 저장 공간 요청 |

> 💡 PVC가 요청한 것을 꼭 그대로 다 들어주지는 않는다. 예를 들어 PVC가 3Gi를 요청했는데 준비된 3Gi짜리 PV가 없고, 5Gi PV만 남아 있으면 그냥 그 5Gi PV가 배정된다 (요청 용량보다 크면 배정, 작으면 배정 안 함). 남은 PV가 없으면 **Pending** 상태로 대기한다 — 너무 큰 PVC는 파드 생성에 제약 조건이 될 수 있다.

##### PV/PVC 비유 — 창고와 창고 신청서

```
PV : "5평짜리 냉장 창고가 준비되어 있습니다."
PVC: "최소 3평 이상의 냉장 창고가 필요합니다."
Kubernetes: "조건이 맞으므로 두 리소스를 연결하겠습니다."
```

##### PV-PVC의 바인딩

**PV ↔ PVC가 연결되는 과정을 바인딩이라고 한다.**

```
PV: app-pv    (용량 5Gi, 접근 모드 RWO, StorageClass: manual)
        │ 조건 비교
        ▼
PVC: app-pvc  (요청 용량 3Gi, 접근 모드 RWO, StorageClass: manual)
        │
        ▼
    Status: Bound
```

**바인딩 조건 (모두 만족해야 함)**

| 조건 | 설명 |
|---|---|
| 용량 | PV 용량이 PVC 요청 용량 이상이어야 함 |
| 접근 모드 | PVC가 요청한 접근 모드를 PV가 지원해야 함 |
| StorageClass | `storageClassName`이 서로 일치해야 함 |
| VolumeMode | 파일 시스템 또는 블록 장치 모드가 호환되어야 함 |
| Selector | PVC에 Selector가 있다면 PV 라벨이 조건을 만족해야 함 |

- 바인딩 조건이 맞지 않으면 **Pending** 상태가 된다.
- 바인딩은 기본적으로 **일대일 관계**다. 다만 스토리지 시스템이 NFS이거나 Access Mode에 따라서는 여러 파드가 접근할 수도 있다.

##### Pod에서 PVC 사용하기

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
spec:
  containers:
    - name: app
      image: nginx
      volumeMounts:
        - name: app-storage
          mountPath: /usr/share/nginx/html
  volumes:
    - name: app-storage
      persistentVolumeClaim:
        claimName: app-pvc
```

##### 접근 모드 (AccessMode)

| 접근 모드 | 약어 | 설명 |
|---|---|---|
| ReadWriteOnce | RWO | 하나의 노드에서 읽기·쓰기로 마운트 |
| ReadOnlyMany | ROX | 여러 노드에서 읽기 전용으로 마운트 |
| ReadWriteMany | RWX | 여러 노드에서 읽기·쓰기로 마운트 |
| ReadWriteOncePod | RWOP | 하나의 Pod만 읽기·쓰기로 마운트 |

> 💡 모든 스토리지가 모든 접근 모드를 지원하지는 않는다. 일반적인 블록 스토리지(EBS)는 주로 **RWO**를, NFS는 **RWX**를 지원한다. 필요한 접근 방식에 따라 스토리지의 지원 여부를 미리 확인해야 한다.

##### PV의 회수 정책 (Reclaim Policy)

PVC가 삭제되거나 파드에서 사용이 끝나면, PV(클러스터에서 관리되는 리소스)에 남아있는 데이터를 어떻게 처리할지를 결정한다.

**1. Retain**
- PVC가 삭제되더라도 PV 안의 데이터를 **자동으로 삭제하지 않는다.**
- PVC가 삭제되면 → PV 상태가 `Released`로 바뀐 채 그냥 대기 → 관리자가 데이터를 확인하고 수동으로 삭제한다.

**2. Delete**
- PVC가 삭제되면 PV와 연결된 스토리지를 삭제할 수 있다.
- 주로 StorageClass의 동적 프로비저닝에서 쓰인다.
- ⚠️ 주의: **PV에 Delete를 사용하겠다는 것은, 사실상 영구적인 저장에 대해서는 고려하지 않겠다는 뜻**이므로 신중해야 한다.

##### Volume vs PV/PVC

| 구분 | 일반 Volume | PV/PVC |
|---|---|---|
| 주요 목적 | Pod에 저장 공간 연결 | Pod와 독립적인 저장소 관리 |
| Pod와의 관계 | Pod YAML에서 Volume 종류를 직접 정의 | Pod는 PVC만 참조 |
| 수명 | Volume 종류에 따라 Pod와 함께 종료 가능 | PV는 Pod와 독립적인 수명 주기 |
| 노드 이동 | 로컬 Volume은 기존 데이터 접근이 어려움 | 외부 스토리지 사용 시 다른 노드에서도 재연결 가능 |
| 저장소 추상화 | 낮음 | 높음 |
| 일반적인 영구 데이터 | 부적합한 유형이 많음 | 적합 |
| 운영 주체 분리 | 어려움 | 관리자와 개발자의 역할 분리 가능 |

##### 실무에서 PV/PVC 사용 사례

- PV를 정적으로 만들어서 사용하지 않는 편이다 (하루에 쌓이는 로그가 100GB라고 해서 무조건 PV를 3TB로 만들 필요는 없다 — 100GB는 어디까지나 평균치이지 항상 그런 건 아니기 때문).
- **동적 프로비저닝**이 일반적이다: 운영자가 PVC를 만들고, PV가 없으면 StorageClass를 통해 동적으로 PV를 생성·할당한다.
- 데이터베이스 저장소로 많이 사용한다 (다만 "데이터베이스는 k8s로 관리하기 적절하지 않다"는 의견도 있다는 점은 참고할 만하다).
- 상태 기반 애플리케이션: 큐를 사용하는 WAS 중 큐가 컨테이너별로 만들어져도 괜찮은 경우, 작은 예매 시스템 등
- 사용자 파일: 이미지, 문서 보존

##### 실무에서 주의해야 하는 것

**1. PVC 삭제 전, 회수 정책을 봐야 한다.** `kubectl get pv`로 확인하고, 중요한 데이터는 미리 백업한다.

**2. PV는 영구적 저장소이지만 자동으로 백업되지 않는다.** 클러스터가 특정 리전에만 있다면(예: 서울 리전), 그 리전에 장애가 생기면 데이터도 함께 위험해진다. 중요도가 높은 데이터는 별도로 S3에 넣거나, 로컬 스토리지에 주기적으로 백업해두는 게 좋다.

**3. PV는 자동으로 줄어들지 않는다.** 스토리지 드라이버에 따라 자동으로 용량을 줄여주는 기능을 지원하는 경우도 있지만, 기본적으로는 PVC와 PV를 무분별하게 만들면 그만큼 비용이 늘어난다는 점을 감안해야 한다.

**4. PV를 정적으로 만들겠다면** 로컬 디스크로도 만들 수 있지만, 로컬 디스크가 가득 차지 않는지 확인해야 한다. 다만 동적 프로비저닝과는 자연스럽게 맞지 않아서 보통 로컬 디스크로 PV를 잘 만들지는 않는다.

**5. 네임스페이스 구분에 유의해야 한다.**

| 리소스 | 범위 |
|---|---|
| PV | 클러스터 전체 |
| PVC | 네임스페이스 |
| Pod | 네임스페이스 |

내가 쓰려고 만든 PV를 다른 네임스페이스에서 가져갈 수도 있으므로, **라벨링을 잘 달아놓아야 한다.**

##### 정리

| 개념 | 한 줄 정리 |
|---|---|
| Volume | Pod가 사용할 저장 공간을 연결하는 기본 구조 |
| PV | 클러스터에 준비된 실제 저장 공간을 표현하는 리소스 |
| PVC | 애플리케이션이 필요한 저장 공간을 요청하는 리소스 |
| Binding | 조건에 맞는 PV와 PVC를 연결하는 과정 |
| AccessMode | 저장소를 어느 범위에서 읽고 쓸 수 있는지 정의 |
| ReclaimPolicy | PVC 사용 종료 후 PV와 데이터를 어떻게 처리할지 정의 |

---

#### | Storage Class

**PVC 요청에 따라 어떤 종류의 저장소를 어떤 정책으로 자동 생성할지 정의하는 스토리지 템플릿**

```
PVC: "10Gi 저장 공간이 필요합니다."
   ↓
StorageClass: "이 방식으로 저장소를 만들겠습니다."
   ↓
Provisioner / CSI Driver: "실제 저장소와 PV를 생성합니다."
```

- 스토리지 클래스 자체가 데이터를 저장하거나 생성하지는 않는다.
- 스토리지 클래스는 실제 저장소를 생성할 때 사용할 **프로비저너, 스토리지 종류, 생성 시점, 삭제 정책** 등의 규칙을 정의하는 것이다.

##### PV/PVC의 한계 — 정적 프로비저닝

```
1. 관리자가 실제 저장소 준비 (EBS, Node Storage)
2. 관리자가 PV 생성
3. 사용자가 PVC 생성
4. 조건에 맞는 PV와 PVC 바인딩
5. Pod가 PVC 사용
```

- 이 방식을 **정적 프로비저닝**이라고 한다.
- 사용하기 힘들고, 환경이 다르면 스토리지도 정책을 따로 분리해야 하며, **운영자가 실수할 가능성이 높아진다.**

##### 스토리지 클래스가 해결하는 문제 — 동적 프로비저닝

- 요청이 오면 그 요청에 맞춰 PV를 만들어서 사용한다 — **관리자가 미리 저장소·PV를 준비하지 않아도 된다.**
- PVC가 생기면, StorageClass에 정의한 정책을 기반으로 실제 저장소 + PV를 만들어서 바인딩한다.

```
1. PVC 생성
   ↓
2. StorageClass (어떻게 스토리지를 만들어야 하는지에 대한 정의서)
   ↓
3. Provisioner
   - 실제 저장소 자동 생성
   - PV 자동 생성
   - PVC와 자동 바인딩
```

##### StorageClass의 동작 구조

```
Pod
 │ PVC 참조
 ▼
PersistentVolumeClaim
  - storage: 10Gi
  - storageClassName: fast
 │ StorageClass 선택
 ▼
StorageClass: fast
  - provisioner
  - parameters
  - reclaimPolicy
  - volumeBindingMode
 │ 실제 저장소 생성 요청
 ▼
Provisioner / CSI Driver
  ├─ 실제 Volume 생성
  └─ PersistentVolume 생성
 │
 ▼
PVC와 PV 바인딩
 │
 ▼
Pod에서 Volume 사용
```

##### StorageClass YAML

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: app-pvc
spec:
  storageClassName: fast-storage
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
```

```yaml
# 스토리지 클래스
storageClassName: fast-storage
provisioner: ebs.csi.aws.com
```

- AWS EBS API를 호출해서, PVC가 필요하다고 한 EBS 블록을 만들고 → PV를 생성 → PVC-PV를 바인딩한다.

##### Provisioner

**PVC 요청을 보고 실제 저장소, PV를 생성하는 담당자**

- 스토리지 클래스가 정책을 정의하고, 실제 저장소를 생성하는 기능은 **프로비저너**나 **CSI Driver**가 수행한다.
- 프로비저너가 스토리지 시스템에 생성 요청을 보내는 역할을 한다.

##### CSI (Container Storage Interface)

**쿠버네티스와 외부 스토리지 시스템을 연결할 때 사용하는 인터페이스**

- 스토리지 공급자마다 저장소 생성·수정·삭제 방식이 다르다 (AWS EBS, Azure Disk, GCP Persistent Disk, NFS 등).
- **CSI가 쿠버네티스와 스토리지 드라이버 사이에서 표준 인터페이스 역할**을 한다 — CSI만 부르면 알아서 스토리지별로 생성·수정·삭제를 처리한다.

##### StorageClass 속성

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-storage
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  encrypted: "true"
reclaimPolicy: Delete
allowVolumeExpansion: true
volumeBindingMode: WaitForFirstConsumer
```

**1. provisioner** — 어떤 스토리지 드라이버가 실제 볼륨을 생성할지 결정. 프로비저너 이름과 사용 설정은 CSI 드라이버 문서에서 확인한다.

| Provisioner | 역할 |
|---|---|
| `ebs.csi.aws.com` | AWS EBS Volume 생성 |
| `efs.csi.aws.com` | AWS EFS 파일 시스템 연결 |
| `pd.csi.storage.gke.io` | Google Persistent Disk 생성 |
| `disk.csi.azure.com` | Azure Disk 생성 |
| `rook-ceph.rbd.csi.ceph.com` | Ceph 블록 Volume 생성 |

**2. parameters** — 생성할 스토리지의 세부 속성(디스크 종류, 암호화 여부, 파일 시스템, IOPS, 복제 방식 등)을 정의. CSI 드라이버마다 다르게 정의된다.

**3. reclaimPolicy** — PVC 삭제 이후 동적으로 생성된 PV와 실제 저장소를 어떻게 처리할지 결정. `Retain`이 일반적이며, `Delete`는 데이터가 정말 지워져도 괜찮은 경우에만 사용해야 한다.

**4. allowVolumeExpansion** — PVC 요청 용량 증가를 허용할지 여부. `true`로 설정하면 PVC YAML의 request 용량을 수정해 실제 볼륨을 늘릴 수 있다 (단 CSI Driver가 볼륨 확장을 지원해야 하고, 실제 스토리지 시스템·파일 시스템도 확장이 가능해야 한다). 일반적으로 확장은 가능하지만, **축소는 대부분 지원되지 않는다.**

**5. volumeBindingMode** — PV와 PVC가 바인딩되는 시점, 실제 볼륨이 생성되는 시점을 결정한다. 두 가지 방식이 있다.

- **Immediate (기본값)**

```
PVC 생성 → 즉시 Volume 생성 → PV와 PVC 바인딩 → 이후 Pod 스케줄링
```

  PVC가 생성되면 즉시 볼륨을 만든다. 문제는 **파드가 어느 노드, 어떤 가용영역에 배치될지 결정되기 전에** 스토리지를 먼저 만든다는 점이다.

- **WaitForFirstConsumer**

```
PVC 생성 → 아직 Volume 생성하지 않음 → PVC를 사용하는 Pod 생성 → Pod가 실행될 노드와 위치 고려 → 적절한 위치에 Volume 생성
```

  PV가 사용될 파드의 노드 위치·가용 영역이 결정되고 나서 스토리지를 만든다. **클라우드 환경(Multi AZ)에서 특히 유리하다.** 단점은 볼륨 생성을 대기해야 하고, 볼륨 생성이 안 되는 경우엔 Pod가 Pending 상태가 될 수 있다는 점이다.

| 구분 | Immediate | WaitForFirstConsumer |
|---|---|---|
| Volume 생성 시점 | PVC 생성 직후 | PVC를 사용하는 Pod 스케줄링 시점 |
| Pod 위치 고려 | 어려움 | 가능 |
| 가용 영역 고려 | Volume이 먼저 결정될 수 있음 | Pod와 Volume 위치를 함께 결정 |
| 로컬 스토리지 | 부적합할 수 있음 | 주로 사용 |
| 클라우드 스토리지 | 사용 가능 | 일반적으로 더 안전한 선택 |

##### 실무에서 StorageClass 사용하기

실무에서는 용도에 따라 스토리지 클래스를 분리해두고 사용한다.

```
StorageClass: fast     — 고성능 SSD, 데이터베이스, 비용 높음
StorageClass: standard — 일반 SSD, 일반 애플리케이션, 기본 StorageClass
StorageClass: shared   — NFS 또는 파일 스토리지, 여러 Pod의 파일 공유
StorageClass: retain   — 중요한 데이터, reclaimPolicy: Retain
```

##### 서드파티 스토리지 솔루션

**1. OpenEBS** — 쿠버네티스에서 로컬 디스크 + 분산 스토리지를 관리할 수 있도록 제공하는 클라우드 네이티브 스토리지 플랫폼. Local PV HostPath를 쉽게 사용할 수 있고, 복제 스토리지 엔진도 제공한다. 보통 온프레미스 환경, 고성능 로컬 디스크를 쓰면서 스토리지 구성을 직접 통제하고 싶을 때, 베어메탈 쿠버네티스 환경에서 사용한다.

**2. Ceph** — 분산 스토리지를 쿠버네티스 Operator 방식으로 배포·관리하는 솔루션. 여러 스토리지 노드(디스크)를 묶어 노드 간 데이터 복제, 고가용성 구성이 가능하다. 단점은 구성이 많이 복잡해진다는 점이다.

|---|---|
| PV | 실제 저장 공간을 Kubernetes 리소스로 표현 |
| PVC | 필요한 저장 공간을 요청 |
| StorageClass | 저장소를 생성하는 종류와 정책을 정의 |
| Provisioner | 실제 저장소와 PV를 생성 |
| CSI Driver | Kubernetes와 외부 스토리지를 연결하는 표준 드라이버 |
| 동적 프로비저닝 | PVC 요청에 따라 저장소와 PV를 자동 생성 |

#### | Controller - StatefulSet

**각 Pod의 고유한 이름, 네트워크 주소, 스토리지를 유지하면서 상태가 있는 애플리케이션을 관리하는 컨트롤러**

- 동일한 파드를 만드는 것만 하는 게 아니라, **파드를 서로 구분**하고, 파드가 재생성되어도 **기존의 이름 + 스토리지를 다시 연결**할 수 있도록 관리한다.

##### Stateless vs Stateful — 상태가 있냐 없냐

**Stateless**
- WAS 애플리케이션, API Server처럼 요청을 처리하는 파드가 바뀌어도 문제가 없을 때 → **Deployment Controller**로 관리

**Stateful**
- 각 파드가 **자신의 데이터, 자신의 역할**을 가지고 있는 애플리케이션. 예를 들어 데이터베이스 클러스터를 보면:

```
database-0 : 첫 번째 데이터베이스 인스턴스 (Master)
database-1 : 두 번째 데이터베이스 인스턴스 (Replica)
database-2 : 세 번째 데이터베이스 인스턴스 (Standby)
```

- 만약 `database-1`이 삭제됐다면, **단순히 아무 파드나 추가해서는 안 된다.** `database-1`이라는 이름으로 다시 생성되고, `database-1`이 사용하던 스토리지에 다시 연결되어야 한다.
- 파드의 이름과 데이터가 파드가 재생성되어도 지속적으로 이어져야 하는 이런 파드들을 관리하는 컨트롤러가 **StatefulSet**이다.

| 구분 | Stateless | Stateful |
|---|---|---|
| Pod 구분 | 각 Pod가 서로 대체 가능 | 각 Pod를 구분해야 함 |
| Pod 이름 | 중요하지 않음 | 고유한 이름이 중요할 수 있음 |
| 데이터 | 외부 저장소를 사용하거나 상태를 가지지 않음 | Pod별 데이터가 필요할 수 있음 |
| 생성 순서 | 대부분 중요하지 않음 | 순서가 중요할 수 있음 |
| 대표 컨트롤러 | Deployment | StatefulSet |
| 대표 사례 | 웹 서버, API 서버 | 데이터베이스, 메시징·분산 시스템 |

##### StatefulSet이 해결하는 문제

**1. 파드 재생성 시, 이름 변경 문제**

- Deployment가 생성한 파드는 일반적으로 이름 뒤에 랜덤 텍스트가 붙고, 파드가 재생성되면 이름도 새롭게 바뀐다.
- 모든 파드가 똑같은 역할을 하면 문제가 없지만, 데이터베이스 클러스터처럼 특정 인스턴스를 구분해야 하는 경우 **파드의 이름이 변경되지 않아야 한다** → StatefulSet

**2. 파드마다 다른 스토리지가 필요**

```
mysql-0 → mysql-0 전용 PVC
mysql-1 → mysql-1 전용 PVC
mysql-2 → mysql-2 전용 PVC
```

- 모든 파드가 하나의 PV를 공유하는 게 아니라, 각 파드가 독립적인 스토리지를 유지해야 하는 경우가 있다.
- **volumeClaimTemplates**를 사용하면 각 파드에 고유한 PVC를 자동으로 만들어준다. 파드가 삭제돼도 기본적으로 PVC는 남아 있고, 같은 순번의 파드가 재생성되면 기존 PVC(및 그 PVC가 쓰던 PV)를 그대로 사용한다.

**3. 특정 파드를 네트워크에서 찾아야 함**

- 일반적으로는 서비스(ClusterIP)를 사용해서 `be-pod A`, `be-pod B` 중 아무 파드나 트래픽을 받아도 무방하다.
- 하지만 데이터베이스처럼 `db-0`은 master(쓰기 요청), `db-1`은 slave(읽기 요청)로 역할이 나뉜 경우, **특정 파드로 직접 연결해야 하는 요구사항**이 생긴다.

```
db-0에 직접 연결
db-1에 직접 연결
db-2에 직접 연결

# Headless Service 라는걸 사용함
db-0.db-service.default.svc.cluster.local
db-1.db-service.default.svc.cluster.local
db-2.db-service.default.svc.cluster.local
```

- **Headless Service**는 프록시나 로드밸런싱이 적용되지 않고, DNS 조회 시 서비스가 선택한 **개별 파드의 IP 주소**를 그대로 리턴한다.

**4. 파드의 생성과 삭제 순서가 중요**

- 분산 디비, 분산 메시징 시스템 같은 분산 애플리케이션은 **모든 인스턴스가 동시에 시작되면 안 된다** (마스터 노드가 초기화되고 나서 워커 노드가 등록되는 식으로, 역할별 초기화 순서가 고정되어야 함).
- StatefulSet의 기본 정책은 **OrderedReady**다.

##### Deployment vs StatefulSet

| 구분 | Deployment | StatefulSet |
|---|---|---|
| 주요 목적 | Stateless 애플리케이션 관리 | Stateful 애플리케이션 관리 |
| Pod 관계 | 서로 대체 가능 | 각 Pod의 정체성 유지 |
| Pod 이름 | 임의 문자열 포함 | 순번 기반 고정 이름 |
| 예시 | `api-7b8d9c-x2k4p` | `mysql-0` |
| 생성 순서 | 일반적으로 병렬 | 기본적으로 순차 생성 |
| 삭제 순서 | 개별 순서 보장 없음 | 기본적으로 역순 삭제 |
| 스토리지 | 필요에 따라 PVC 연결 | Pod별 PVC 생성에 적합 |
| 개별 Pod DNS | 일반적으로 필요 없음 | Headless Service와 함께 제공 가능 |
| 대표 용도 | 웹 서버, API 서버 | 데이터베이스, 분산 시스템 |

##### StatefulSet YAML

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: mysql-service
  replicas: 3

  selector:
    matchLabels:
      app: mysql

  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
        - name: mysql
          image: mysql:8.4
          ports:
            - name: mysql
              containerPort: 3306
          volumeMounts:
            - name: data
              mountPath: /var/lib/mysql

  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes:
          - ReadWriteOnce
        storageClassName: standard
        resources:
          requests:
            storage: 10Gi
```

**1. serviceName** — 파드의 네트워크 도메인을 관리하는 서비스 이름. 일반적으로 Headless Service를 만들 때 여기 지정한 이름을 사용한다.

```yaml
# Headless Service YAML
apiVersion: v1
kind: Service
metadata:
  name: mysql-service
spec:
  clusterIP: None
  # 단일 클러스터 IP 를 안만들고
  # 개별 파드의 IP 주소를 DNS 로 조회 할 수 있음
  selector:
    app: mysql
  ports:
    - name: mysql
      port: 3306
      targetPort: 3306
```

**2. replicas** — 생성할 파드 수

**3. selector, template**

```yaml
selector:
  matchLabels:
    app: mysql
    # 컨트롤러가 상태 관리할 파드를 결정하는 라벨

template:
  metadata:
    labels:
      app: mysql
    # 여기에 있는 라벨은 셀렉터 라벨과 일치
```

**4. volumeClaimTemplates** — 각 파드에 PVC를 자동 생성하는 템플릿

##### StatefulSet, 실무에서 어디에 사용하나

**1. 데이터베이스 클러스터 구성**
- 구성 비용이 생각보다 저렴하다. RDS Multi-AZ는 AZ 간 데이터 통신, AZ 간 가용성 확보를 위한 추가 비용이 상당히 크게 나온다.
- Primary, Replica, Standby, Sharding 구조를 구성할 수 있다.
- StatefulSet이 관리하는 범위: 파드 개수(`replicas`), PVC/PV, 이름 고정, 파드 생성 순서 — 그 외의 것들은 직접 고민해야 한다.
  - 복제 구조, Primary 선출, 데이터 동기화, 자동 백업, 데이터 복원, 애플리케이션 수준의 장애 조치

**2. 메시징 — 메시지를 받아서 다른 쪽에 전달해주는 메시지 브로커**
- Apache Kafka, ZooKeeper, RabbitMQ, Elasticsearch(또는 OpenSearch)

##### 실무에서 주의해야 할 것

**1. Liveness Probe를 과하게 설정하면 안 된다.** 데이터베이스는 초기화 시간이 오래 걸리고, 커넥션 풀이 가득 차면 일시적으로 요청을 처리하지 못하는 지연이 잦은 편이다. 라이브니스 프로브를 너무 예민하게 설정하면 컨테이너가 계속 재시작되고, StatefulSet은 파드의 역할이 고정되어 있기 때문에 이런 상황이 더 위험하다. **모든 프로브를 설정할 때 StatefulSet은 예민도를 낮춰야 한다.**

**2. StatefulSet이 데이터를 복제해주지는 않는다.** StatefulSet이 제공하는 것은 고정된 Pod 이름, 고정된 네트워크 식별자, Pod별 스토리지 연결, 생성 및 삭제 순서뿐이다. 실제 클러스터 구성에 필요한 **데이터베이스 복제, Primary 선출, 데이터 동기화, 자동 백업, 데이터 복원, 애플리케이션 수준의 장애 조치**는 별도로 구현해야 한다.

**3. PVC, PV는 함부로 삭제하지 않는다.** Stateless 애플리케이션에서 사용하는 PVC는 삭제해도 보통 상관없지만, **StatefulSet에서 사용하는 PVC는 절대로 삭제해서는 안 된다** — 데이터베이스 데이터가 날아갈 수 있다 (백업이 있으면 다행이지만, StatefulSet을 사용할 때는 보통 백업 설정을 넉넉하게 해두는 게 좋다). PV는 클러스터 전체가 관리 주체라서 다른 네임스페이스의 유저가 실수로 PV를 삭제할 수도 있으므로, **RBAC 권한 설정을 빡빡하게 해야 한다.** PVC를 삭제해야 하는 경우엔 해당 PVC가 바인딩된 PV가 남아있지 않은지, 이전에 사용하던 PV에 데이터가 남아있지 않은지 항상 확인해야 한다.

**4. 순차 생성은 연쇄 에러의 원인이 될 수 있다.** `OrderedReady` 정책을 통해 앞 순번 파드가 Ready 상태가 되어야 다음 파드를 만든다. 실제로 필요한 순서와 내가 설정한 순서가 다르면 무한히 대기할 수 있다 (예: MySQL 애플리케이션은 실제로 1번이 main인데, 설정은 0번이 먼저 초기화되게 되어 있는 경우). 디버깅할 때는 숫자가 높은 순서대로 한 번씩 접속해보는 걸 추천한다. 순서 보장이 필요 없다면 `Parallel` 정책을 선택할 수도 있다.

#### | Node Scheduling

**새로 생성된 Pod를 실행할 수 있는 노드들을 찾고, 그중 가장 적합한 노드를 선택하여 배치하는 과정**

- 쿠버네티스 기본 스케줄러인 **kube-scheduler**가 아직 노드가 정해지지 않은 파드들을 감시한다.
- 두 단계를 거쳐 파드가 실행될 노드를 결정한다.
  1. **Filtering** — Pod를 실행할 수 없는 노드를 제외
  2. **Scoring** — 실행 가능한 노드에서 점수를 계산해서 가장 적합한 노드를 선택
- 최종적으로 선택된 노드에 파드를 배치한다.

```
Pod 생성 요청
   ↓
kube-scheduler가 Pod 확인
   ↓
실행 가능한 노드 필터링
   ↓
후보 노드 점수 계산
   ↓
가장 적합한 노드 선택
   ↓
해당 노드의 kubelet이 Pod 실행
```

##### 왜 노드 스케줄링이 필요할까?

노드마다 사용할 수 있는 자원과 특성이 다르기 때문이다.

| 노드 | CPU | 메모리 | 특징 |
|---|---|---|---|
| worker-1 | 2 Core | 4Gi | 일반 노드 |
| worker-2 | 4 Core | 8Gi | SSD 장착 |
| worker-3 | 8 Core | 16Gi | GPU 장착 |

파드의 특성마다 필요한 환경이 다르다 (일반 웹 서버 Pod / 많은 메모리가 필요한 데이터 처리 Pod / SSD가 필요한 데이터베이스 Pod). 스케줄링 기능이 없으면 사용자가 파드마다 매번 적절한 노드를 직접 선택해줘야 한다.

| 사용 이유 | 설명 |
|---|---|
| 실행 가능한 노드 선택 | Pod가 요청한 CPU와 메모리(Requests)를 제공할 수 있는 노드를 찾음 |
| 노드 특성 반영 | SSD, GPU, 운영체제, 가용 영역 등 노드의 특성에 따라 배치할 수 있음 |
| 특정 노드의 과부하 방지 | 실행할 수 없는 노드를 제외하고, 후보 노드들의 상태를 비교함 |
| 워크로드 분리 | 운영용, 개발용, GPU용 노드처럼 목적에 따라 워크로드를 분리할 수 있음 |
| 안정적인 운영 | Pod가 자신의 실행 조건을 충족하는 노드에서 동작하도록 제어함 |

##### 기본적인 스케줄링 동작

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
spec:
  containers:
    - name: nginx
      image: nginx:1.27
      resources:
        requests:
          cpu: 200m
          memory: 128Mi
```

배치 조건을 별도로 작성하지 않으면 kube-scheduler가 **리소스(Requests)를 기반**으로 자동으로 노드를 선택한다.

**1. Filtering** — 파드를 실행할 수 없는 노드를 후보에서 제외하는 단계

확인하는 조건:
- 노드가 정상 상태인가?
- Pod가 요청한 CPU와 메모리가 남아 있는가?
- Pod가 요구한 노드 라벨을 만족하는가?
- 노드에 Pod가 허용하지 못하는 Taint가 있는가?
- 사용하는 스토리지를 해당 노드에서 연결할 수 있는가?

→ 필터링 후 후보 노드가 하나도 없으면 파드는 **Pending** 상태로 대기한다.

**2. Scoring** — 필터링을 통과한 후보 노드들의 점수를 매기는 과정. 가장 높은 점수를 받은 노드가 선택된다.

##### 노드 스케줄링을 제어하는 방법

| 방법 | 기준 | 강제성 | 특징 |
|---|---|---|---|
| 기본 스케줄링 | 리소스와 클러스터 상태 | 자동 | 일반적인 Pod 배치 |
| `nodeName` | 노드 이름 | 매우 강함 | 스케줄러를 우회하여 직접 지정 |
| `nodeSelector` | 노드 라벨 | 필수 | 단순한 `key=value` 조건 |
| Node Affinity `required` | 노드 라벨 조건식 | 필수 | 복잡한 필수 조건 |
| Node Affinity `preferred` | 노드 라벨 조건식 | 선호 | 가능하면 조건에 맞게 배치 |
| Taint/Toleration | 노드의 배치 거부 조건 | 허용 여부 | 특정 Pod만 노드에 들어오도록 제한 |

**1. NodeName** — 파드에 지정한 이름의 노드와 직접 연결하는 방식

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: node-name-pod
spec:
  nodeName: worker-2
  containers:
    - name: nginx
      image: nginx:1.27
```

| 특징 | 설명 |
|---|---|
| 노드 이름 직접 지정 | 정확한 노드 이름을 작성해야 함 |
| 스케줄러 우회 | 일반적인 kube-scheduler의 노드 선택 과정을 거치지 않음 |
| 유연성 부족 | 노드 이름이 변경되거나 해당 노드가 사라지면 다른 노드를 자동 선택하지 못함 |
| 일반 배포에는 비권장 | 특정 노드에 강하게 종속되므로 운영 유연성이 떨어짐 |

**2. nodeSelector** — 지정한 라벨을 모두 가진 노드에 파드를 배치하는 방법

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: node-selector-pod
spec:
  nodeSelector:
    disktype: ssd
  containers:
    - name: nginx
      image: nginx:1.27
```

- 필수 조건을 지정하며, 조건을 여러 개 붙일 수 있다 (AND 조건).
- 조건을 만족하는 노드가 없으면 **Pending** 상태가 된다.
- 언제 사용하나: SSD 노드에서만 데이터베이스 실행, 운영용 노드와 개발용 노드 분리, GPU가 설치된 노드에 특정 Pod 배치, 특정 CPU 아키텍처의 노드 선택, 특정 가용 영역의 노드 선택 등.

**3. Node Affinity** — 노드 라벨을 기반으로 필수 조건과 선호 조건을 세밀하게 표현하는 스케줄링 방식. **실무에서는 대부분 이 방식을 사용**한다.

복잡한 조건 표현이 가능하다: 여러 값 중 하나 선택, 특정 값 제외, 특정 라벨의 존재 여부 확인, 반드시 지켜야 하는 조건, 가능하면 지키는 선호 조건, 선호 조건별 가중치 부여 등.

Node Affinity에는 두 가지 조건식이 있다.

| 종류 | 의미 |
|---|---|
| `requiredDuringSchedulingIgnoredDuringExecution` | 스케줄링할 때 반드시 만족해야 하는 조건 |
| `preferredDuringSchedulingIgnoredDuringExecution` | 가능하면 만족하는 노드를 선호하지만 필수는 아닌 조건 |

| 단어 | 의미 |
|---|---|
| `required` | 반드시 지켜야 함 |
| `preferred` | 가능하면 선호함 |
| `DuringScheduling` | 스케줄링하는 시점에 확인 |
| `IgnoredDuringExecution` | 실행된 이후 노드 라벨이 바뀌어도 계속 실행 |

**Required 예시**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: required-affinity-pod
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: disktype
                operator: In
                values:
                  - ssd
  containers:
    - name: nginx
      image: nginx:1.27
```

→ `disktype = ssd`인 노드에만 배치하고, 없으면 대기한다.

**Preferred 예시**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: preferred-affinity-pod
spec:
  affinity:
    nodeAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 80
          preference:
            matchExpressions:
              - key: disktype
                operator: In
                values:
                  - ssd
  containers:
    - name: nginx
      image: nginx:1.27
```

→ 가능하면 `disktype = ssd`인 걸 선호하되, SSD 노드가 없으면 그냥 다른 노드에도 배치된다. `weight`는 1~100 사이의 가중치이며, 높을수록 조건을 만족하는 노드가 더 높은 점수를 받지만 **가중치가 100이라고 반드시 그 노드에 배치되는 건 아니다.**

**Node Affinity 연산자**

| 연산자 | 설명 |
|---|---|
| `In` | 노드 라벨 값이 지정된 목록에 포함되어야 함 |
| `NotIn` | 노드 라벨 값이 지정된 목록에 포함되지 않아야 함 |
| `Exists` | 해당 라벨 키가 존재해야 함 |
| `DoesNotExist` | 해당 라벨 키가 없어야 함 |
| `Gt` | 라벨 값이 지정한 숫자보다 커야 함 |
| `Lt` | 라벨 값이 지정한 숫자보다 작아야 함 |

**여러 조건을 결합하는 방식** — Node Affinity에서는 조건이 결합되는 위치에 따라 AND 연산이 되기도 하고 OR 연산이 되기도 한다.

```yaml
# AND 연산 (하나의 matchExpressions 안에 여러 key)
nodeSelectorTerms:
  - matchExpressions:
      - key: env
        operator: In
        values: [production]
      - key: disktype
        operator: In
        values: [ssd]
```

```yaml
# OR 연산 (matchExpressions를 별도 항목으로 분리)
nodeSelectorTerms:
  - matchExpressions:
      - key: env
        operator: In
        values: [production]
  - matchExpressions:
      - key: disktype
        operator: In
        values: [ssd]
```

**4. Taint ↔ Toleration**

- **Taint**는 노드가 일반 Pod를 밀어내는 제한 조건이고, **Toleration**은 Pod가 그 제한을 견딜 수 있도록 허용하는 설정이다.
- Node Affinity는 **파드 입장에서 특정 노드를 선택**하는 방식이고, Taint는 **노드 입장에서 특정 파드를 거부**하는 방식이다.

```
Node Affinity
  Pod → "나는 SSD 노드로 가고 싶다."

Taint
  Node → "허용된 Pod가 아니면 들어오지 마라."

Toleration
  Pod → "나는 이 노드의 제한을 허용할 수 있다."
```

```bash
kubectl taint nodes worker-2 workload=gpu:NoSchedule
```

→ 워커 노드에 `workload=gpu` Taint를 걸면, 이 라벨을 허용하는 Toleration이 없는 파드는 접근할 수 없다.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: gpu-pod
spec:
  tolerations:
    - key: workload
      operator: Equal
      value: gpu
      effect: NoSchedule
  containers:
    - name: nginx
      image: nginx:1.27
```

→ 이 파드는 `worker-2`에 배치될 수 있는 **자격**을 얻은 것이다. 단, Toleration으로 Taint를 무시할 수 있다는 것이 곧 반드시 그 노드에 배치된다는 뜻은 아니다 — Taint는 **필터링 단계의 조건**이라서, 1) 필터링에서는 taint를 무시하고 후보에 남지만, 2) 스코어링 단계에서 점수가 낮으면 결국 배치되지 않을 수도 있다.

**Taint의 Effect**

| Effect | 신규 Pod 스케줄링 | 기존 실행 Pod | 의미 |
|---|---|---|---|
| `NoSchedule` | 제한 | 계속 실행 | 새 Pod를 받지 않음 |
| `PreferNoSchedule` | 가능하면 제한 | 계속 실행 | 가능하면 배치를 피함 |
| `NoExecute` | 제한 | Toleration이 없으면 축출 가능 | 새 Pod 제한과 기존 Pod 축출 |

##### 실무에서 노드 스케줄링을 어떻게 활용할까?

**1. 보통은 그냥 기본 스케줄링을 사용한다.** 쿠버네티스는 기본적으로 Stateless 애플리케이션을 타겟으로 만들어졌다. GPU가 필요하거나 Stateful 애플리케이션은 외부에서 별도로 관리하는 경우가 많다.

**2. 하드웨어가 필요한 경우** — `Label + nodeSelector + Node Affinity`를 조합해서 GPU 연산, SSD 배치 처리, 특정 가용영역에 파드를 배치한다.

**3. 전용 노드를 만드는 경우** — GPU 노드, 데이터베이스 노드처럼 전용 노드를 만들 때는 `Taint/Toleration`을 함께 사용하고, 여기에 `Node Affinity`도 함께 써서 원하는 파드만 정확히 그 노드로 유도한다.

### 🗨️ 오늘의 회고

- 어제는 일찍 잠에 들어서 목표했던 과제 피드백 반영을 진행하지 못했다. 오늘부터는 반드시 위클리 챌린지를 시작해야 하기 때문에, 주말에 피드백 반영을 진행할 예정이다. 어제 잠을 많이 자서인지 컨디션은 괜찮은 것 같다. 위클리 챌린지를 빠르게 수행하게 된다면 블로그 작성까지 해보려고 한다.
- 오늘은 **Volume**부터 시작해서 **PV/PVC → StorageClass → StatefulSet → Node Scheduling**까지, 저장소와 배치 관련 개념을 쭉 이어서 정리한 하루였다. 개별 개념이 아니라 "왜 이게 필요해졌는가"의 흐름으로 따라가니 이해가 훨씬 쉬웠다.
- **Volume**은 결국 "컨테이너의 파일 시스템은 컨테이너 생명주기에 종속된다"는 문제에서 출발한다는 걸 명확히 짚었다. emptyDir와 hostPath 모두 결국 파드나 노드에 종속되기 때문에 영구 저장에는 부적합하다는 한계가, 다음 개념인 **PV/PVC**로 자연스럽게 이어지는 구조가 인상 깊었다.
- 특히 "hostPath는 노드에 붙어 있으니까 영구적인 거 아닌가요?"라는 질문에 대한 답 — **노드도 스케일링의 대상**이라서 언제든 사라질 수 있다는 설명 — 이 오늘 가장 크게 와닿았던 부분이다. PV/PVC가 "파드·노드와 저장 공간의 생명주기를 분리한다"는 컨셉도 이 맥락에서 훨씬 명확하게 이해됐다.
- **StorageClass**는 "PV를 미리 준비해두는 정적 프로비저닝은 운영자가 실수할 가능성이 높다"는 문제의식에서 동적 프로비저닝으로 넘어가는 흐름이 논리적으로 잘 이어졌다. `volumeBindingMode`의 `Immediate` vs `WaitForFirstConsumer` 차이 — 특히 클라우드 Multi-AZ 환경에서는 파드의 배치 위치가 결정된 뒤에 볼륨을 만드는 `WaitForFirstConsumer`가 유리하다는 점 — 은 실제 EKS 환경을 다룰 때 꼭 기억해둬야겠다.
- **StatefulSet**은 지난주에 배운 Deployment와 대비해서 이해하니 훨씬 수월했다. "Stateless는 Pod가 서로 대체 가능하지만, Stateful은 각 Pod가 자신의 역할과 데이터를 가진다"는 구분과, 그로 인해 필요해지는 고정 이름·전용 PVC·Headless Service·생성 순서(OrderedReady)까지 하나씩 왜 필요한지 이해하며 따라갈 수 있었다. 특히 "StatefulSet은 데이터를 복제해주지 않는다"는 주의사항과 "PVC/PV는 절대 함부로 삭제하지 않는다"는 실무 팁은 실제 운영에서 사고로 이어지기 쉬운 부분이라 각별히 기억해야겠다.
- **Node Scheduling**은 Filtering-Scoring 2단계 구조가 명확했고, 특히 Node Affinity와 Taint/Toleration의 관계 — "Node Affinity는 파드가 노드를 선택하는 것, Taint는 노드가 파드를 거부하는 것"이라는 대비 — 가 오늘 배운 것 중 가장 깔끔하게 정리된 개념이었다.
- 내용이 많아서 하루에 소화하기는 벅찼지만, Volume부터 Node Scheduling까지 하나의 스토리로 이어지는 걸 느끼면서 정리하니 훨씬 오래 기억에 남을 것 같다. 내일부터 시작하는 11주차 위클리 챌린지에서 오늘 배운 PV/PVC, StatefulSet 개념을 실제로 적용해보면서 부족한 부분을 채워나가야겠다.
