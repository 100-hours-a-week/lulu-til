### 🚀 Today I Learned

#### | Portainer

**Docker와 Kubernetes 같은 컨테이너 환경을 손쉽게 관리할 수 있도록 해주는 웹 기반 GUI 관리 도구**

`도커 엔진 + API` 위에서, **Portainer Agent**를 통해 도커의 연결 상태를 읽고 도커 명령어를 대신 실행해주는 구조다.

> 💡 GUI에서 버튼을 클릭하면, 내부적으로 **도커 명령어로 전환되어 전달**된다. Portainer가 Docker를 대체하는 것이 아니라, **Docker를 GUI 기반으로 쉽게 사용할 수 있게 해주는 도구**라는 점이 핵심이다.

##### 왜 필요한가

기본적으로 도커를 사용할 때는 명령어로 동작한다.

```bash
docker ps
docker logs
docker restart ...
```

CLI 명령 자체는 잘 동작하고 자동화하기도 편하다. 하지만 **운영자가 컨테이너를 여러 개 관리해야 한다면** 상황이 달라진다.

| 문제 | 상황 | 결과 |
|---|---|---|
| **상태 파악이 어려움** | 컨테이너가 많아지면 `docker ps`, `docker logs`를 반복 실행 | 현재 서비스 상태를 한눈에 보기 어려움 |
| **명령어 실수** | 컨테이너 이름, 포트, 볼륨, 네트워크를 잘못 지정 | 잘못된 컨테이너 삭제·재시작 가능 |
| **운영 접근성이 낮음** | CLI 사용 경험이 적은 팀원도 운영 작업 필요 | 운영 업무가 특정 인력에게 집중 |
| **여러 서버 관리가 불편함** | EC2가 여러 대고 각각 Docker가 설치됨 | 서버마다 SSH 접속 후 별도 명령 실행 |
| **설정이 눈에 보이지 않음** | 볼륨·네트워크·환경 변수·포트 설정이 분산 | 장애 원인 파악이 느려짐 |

> 이런 문제들을 해결하기 위해 등장한 것이 **Portainer 사용**이다.

##### Portainer 동작 흐름

```
운영자 → Browser (컨테이너 상태 확인 요청)
       → Browser → Portainer (웹 요청)
       → Portainer → Docker Engine (Docker API 요청)
       → Docker Engine → Portainer (컨테이너 상태 반환)
       → Portainer → Browser (상태·로그·자원 사용량 표시)

운영자 → Browser (컨테이너 재시작 요청, Restart 클릭)
       → Browser → Portainer
       → Portainer → Docker Engine (restart 명령 요청)
       → Docker Engine → App Container (컨테이너 재시작)
```

> 즉 운영자는 Portainer라는 웹 UI만 조작할 뿐이고, 그 뒤에서 Portainer가 Docker Engine API를 호출해 실제 명령을 수행한다.

##### Portainer 기능

| 기능 | 설명 | Docker CLI 기능 |
|---|---|---|
| 컨테이너 조회 | 실행·중지·종료 상태 확인 | `docker ps -a` |
| 컨테이너 시작·중지·재시작 | 버튼으로 컨테이너 제어 | `docker start`, `stop`, `restart` |
| 로그 확인 | 브라우저에서 로그 조회 | `docker logs -f` |
| 콘솔 접속 | 컨테이너 내부 셸 접속 | `docker exec -it` |
| 이미지 관리 | 이미지 조회·Pull·삭제 | `docker images`, `pull`, `rmi` |
| 볼륨 관리 | 볼륨 생성·연결·조회 | `docker volume ...` |
| 네트워크 관리 | 컨테이너 네트워크 연결 상태 확인 | `docker network ...` |
| Stack 관리 | Docker Compose 기반 서비스를 화면에서 배포 | `docker compose up -d` |
| 자원 상태 확인 | CPU·메모리·네트워크·I/O 사용량 확인 | `docker stats`, `top` 등 |

##### Docker Desktop vs Portainer

둘 다 도커를 GUI로 관리하는 툴이지만, **사용 환경과 목적이 다르다.**

- Docker Desktop은 **로컬**에서 관리
- Portainer는 **서버 환경**에서 관리

| 구분 | Docker Desktop | Portainer |
|---|---|---|
| 주 목적 | 개인 로컬 개발 환경 | 서버·운영 환경 관리 |
| 주 설치 위치 | 개발자 개인 PC | EC2, 온프레미스 서버, 클라우드 VM |
| 관리 범위 | 보통 내 PC의 Docker | 여러 Docker 서버, Swarm, Kubernetes 환경 |
| 주요 사용자 | 개인 개발자 | 개발팀, 운영팀, DevOps 담당자 |
| 핵심 가치 | 개발 편의성 | 운영 가시성, 원격 관리, 표준화 |
| 대표 사용 상황 | 로컬에서 앱 실행·테스트 | EC2의 컨테이너 상태 확인·재시작·로그 분석 |

##### Portainer의 역할

- 서비스가 제대로 동작하는지 확인
- 장애 컨테이너의 로그 확인
- 컨테이너 재시작
- 이미지 풀(Pull)하고 재시작
- 볼륨, 네트워크 확인

> 한마디로 정리하면 **"도커의 운영 관리 도구"**다.

##### Portainer 구조

```
운영자 브라우저 → Portainer Server
                    ├─ Local Docker Engine (같은 서버)
                    ├─ Portainer Agent → 원격 Docker Server 1
                    │                  → 원격 Docker Server 2
                    └─ Edge Agent      → 사설망 또는 원격 Docker Server
```

##### 구성 요소

| 구성 요소 | 역할 |
|---|---|
| **Portainer Server** | 웹 UI와 관리 기능을 제공하는 중심 서버 |
| **Docker Engine** | 실제 컨테이너·이미지·볼륨·네트워크를 실행하고 관리하는 엔진 |
| **Portainer Agent** | 원격 Docker 서버의 정보를 Portainer Server에 제공하는 에이전트 |
| **Edge Agent** | 외부에서 직접 접근하기 어려운 원격 환경을 관리할 때 사용하는 방식 |

##### Portainer 설치 전에 알아야 하는 개념 — Docker Socket

리눅스에서 Docker CLI는 Docker 데몬과 통신해야 한다. 이 통신 경로가 바로 다음 소켓이다.

```
/var/run/docker.sock
```

> Docker API와 통신하기 위한 소켓.

Portainer 컨테이너에서 **EC2 A의 Docker Socket을 연결하면**, Portainer는 해당 도커 엔진을 완전히 제어할 수 있게 된다.

> ⚠️ 이건 **보안적으로 상당히 위험할 수 있다.** 도커 소켓으로 연결되어 있는 인스턴스라면, Portainer 컨테이너가 그 호스트의 도커 엔진을 그대로 제어할 수 있게 되는 것이기 때문이다.

도커 소켓으로 할 수 있는 작업:
- 컨테이너 생성, 중지, 삭제
- 호스트 디렉터리 마운트
- 민감한 환경 변수 조회 가능성

> ⚠️⚠️⚠️ **중요** ⚠️⚠️⚠️
> **Portainer 컨테이너는 절대로 외부 인터넷에 노출하면 안 된다!** 항상 네트워크 접속 가능한 영역을 줄여놓아야 한다.
> - Private Subnet에 배치
> - VPN을 통해서만 접근
> - 특정 IP 대역에서만 접근 허용

##### 실무에서 Portainer를 사용하는 시점

**① 운영 가시성 확보**

실무에서 장애가 발생하면 가장 먼저 필요한 정보는 **"지금 어떤 컴포넌트, 컨테이너가 죽어있느냐"** 다. Portainer로 다음을 한눈에 확인할 수 있다.
- 어떤 컨테이너가 실행 중이고, 중지되었고, 삭제되었고, 최근에 재시작되었는지
- 문제가 생기는 컨테이너는 보통 재시작 시도 횟수가 비정상적으로 높음
- 컨테이너가 어떤 이미지 버전으로 러닝 중인지
- CPU, RAM 자원 사용량

**② 개발자-DevOps 협업**

Portainer는 도커를 보기 쉽게 만들어주는 역할을 하면서, 다음과 같은 협업 구조를 매끄럽게 한다.

| 역할 | 관심사 |
|---|---|
| 개발자 | 코드를 만들고, CI/CD 파이프라인을 돌리고, 테스트가 정상 통과하면 OK |
| DevOps 팀 | 실제로 배포가 안정적으로 되는지 확인 |

**③ 소규모 운영**

대형 중앙 집중 로그·메트릭 수집 툴(예: Grafana 풀스택)은 무겁고 설치·관리도 까다롭다. EC2를 심플하게 운영하는 소규모 환경에서는 Portainer 정도가 적당한 선택이다.

##### 간단하게 Portainer 설정하기 — 주의사항

**① GUI에서 바꾸면, 실제로 변경된다**

Portainer는 단순 조회 도구가 아니라, GUI 조작이 실제 Docker Engine에 명령으로 전달된다는 점을 항상 기억해야 한다.

**② RBAC(역할 기반 접근 제어)를 적용해야 한다**

사용자 권한을 세분화해서 관리해야 한다.
- 예: 헤일리에게는 조회 권한만, 맥스에게는 로그 조회 권한만 부여

> ⚠️ "누가 책임지냐?" — **삭제한 사람**이 책임지는 게 아니라, **삭제할 수 있는 권한을 부여한(환경을 만든) 사람**이 책임진다는 관점이 중요하다.

**③ Portainer 관련 보안 사항을 준수해야 한다**
- 꼭 Private Subnet에 배치
- VPN 또는 Bastion Host를 통해서만 접근
- Portainer Admin 계정을 공유하면 안 됨

##### Portainer 정리

| 항목 | 내용 |
|---|---|
| **역할** | 도커를 관리하는 GUI |
| **관리 대상** | 컨테이너, 이미지, 볼륨, 네트워크 |
| **장점** | 가시성 확보, 분산된 로그를 집중화해서 볼 수 있음, 간단한 제어를 버튼으로 편하게 수행 가능 |
| **주의사항** | 도커 소켓의 권한이 아주 강함, 인터넷에 노출하면 안 됨 |
| **실무 사용 시점** | 장애 확인, 컨테이너 재시작, 소규모 서버에서의 운영 도구로 선택 |

#### | Volume

**컨테이너 간 데이터 공유 및 영속성을 제공하기 위해 호스트 파일 시스템에 생성되는 독립적인 저장소**

##### 컨테이너 안에 파일을 저장하면 어떻게 될까

```bash
# 데이터의 영속성이 필요한 컨테이너를 만들었다.
docker run -d --name mysql-test mysql:8.0

# 컨테이너 내부의 아래 경로에
/var/lib/mysql
```

겉보기에는 잘 동작한다. 컨테이너를 중지했다 다시 시작해도 데이터가 살아있고, 테이블도 만들어지고 로우도 저장된다.

하지만:

```bash
docker rm mysql-test
# 컨테이너 내부에 있는 저장 공간도 같이 삭제가 된다.
```

> ⚠️ **컨테이너는 비영속적이다.** 컨테이너를 삭제하면 내부 데이터도 함께 사라진다.

##### 왜 이렇게 되는가

컨테이너는 이미지 위에 **쓰기 가능한 임시 레이어**를 얹어서 동작한다.

- 컨테이너가 실행되면서 파일을 만들거나 수정해도, 그 변경 사항은 **이미지에 저장되지 않는다.**
- 변경 사항은 컨테이너의 **읽기·쓰기(Read/Write) 레이어**에 저장된다.
- 이 쓰기 레이어가 **컨테이너의 생명 주기에 묶여 있기 때문에**, 컨테이너가 삭제되면 함께 사라진다.

| 상황 | 데이터 |
|---|---|
| 컨테이너 실행 중 | 컨테이너 내부 변경 사항 유지 |
| 컨테이너 stop | 컨테이너가 남아 있으면 데이터도 남아 있음 |
| 컨테이너 start | 기존 컨테이너라면 데이터 유지 |
| 컨테이너 rm | 컨테이너 쓰기 레이어도 함께 삭제 |
| 새 컨테이너 생성 | 이전 컨테이너 내부 데이터는 이어지지 않음 |

##### 왜 이런 구조로 만들었는가

애초에 컨테이너의 설계 목적 자체가 다음과 같았기 때문이다.

| 목적 | 설명 |
|---|---|
| **빠른 생성** | 컨테이너를 빠르게 만들 수 있어야 함 |
| **빠른 삭제** | 필요 없으면 쉽게 제거할 수 있어야 함 |
| **동일한 실행 환경** | 이미지로부터 언제든 같은 환경을 재현할 수 있어야 함 |
| **일회성 실행 가능** | 테스트, 배치, 임시 작업처럼 쓰고 버리는 실행에 적합해야 함 |
| **배포 단순화** | 서버마다 직접 설치하지 않고 이미지로 실행해야 함 |

이 설계 목적은 **애플리케이션 실행 환경**에서는 매우 좋다. 예를 들어 백엔드 컨테이너에 문제가 생기면 다음처럼 가볍게 교체할 수 있다.

```bash
docker rm -f backend
docker run -d --name=backend backend:1.3.1
```

> 💡 하지만 이 가벼움이 **DB처럼 영속성이 필요한 데이터**에는 그대로 적용되면 안 된다. 그래서 등장한 게 Volume이다.

##### Volume이 해결하는 방법

컨테이너 내부에 데이터를 쓰지 않고, **컨테이너 외부**에 데이터를 작성하도록 만든다.

> - `/var/lib/mysql` ← 여기에 저장하는 것처럼 *보이지만*
> - 실제 데이터는 **Host OS → Docker Volume**에 저장된다.
> - 그렇기 때문에 컨테이너를 삭제하고 다시 컨테이너를 만들어도, **볼륨은 어차피 Host OS에 저장되어 있어서** 데이터가 그대로 유지된다.

##### 기본적인 사용 방법

```bash
docker run \
  -v mysql-data:/var/lib/mysql \
  mysql:8.3
```

| 부분 | 의미 |
|---|---|
| `mysql-data` | Docker Volume 이름 |
| `/var/lib/mysql` | 컨테이너 내부 경로 |
| `:` | 왼쪽 저장소를 오른쪽 경로에 연결한다는 의미 |

##### Volume 사용 이유

| 사용 이유 | 설명 |
|---|---|
| **데이터 영속성** | 컨테이너를 삭제하거나 재생성해도 데이터를 유지 |
| **컨테이너 재배포 안전성** | 새 컨테이너를 띄워도 기존 데이터를 재사용 가능 |
| **데이터 공유** | 여러 컨테이너가 같은 데이터를 공유 가능 |
| **Docker가 저장 위치 관리** | 호스트 경로를 직접 지정하지 않아도 Docker가 볼륨 위치를 관리 |
| **운영 안정성** | DB, 업로드 파일, 설정 데이터 등 중요한 데이터를 보호 |

##### 실제 명령어 흐름

```bash
# 볼륨 생성
docker volume create mysql-data

# 볼륨 목록
docker volumes
docker volume ls

# 볼륨 내부 정보
docker volume inspect mysql-data

# 볼륨 컨테이너 연결
docker run -d \
  --name mysql-test \
  -e MYSQL_ROOT_PASSWORD=1234 \
  -v mysql-data:/var/lib/mysql \
  mysql:8.0
```

##### 현업에서 Volume을 언제 사용하는가

**① 데이터베이스 컨테이너 (압도적으로 가장 많이 사용)**
- 영속성이 반드시 필요하기 때문에 거의 모든 DB 컨테이너에 Volume을 연결한다.

**② 파일 업로드 임시 폴더**
- 예: EC2 내부에 WAS 컨테이너가 3개 들어있을 때, 이 컨테이너들이 **공용으로 보는** 업로드 파일 임시 저장 폴더를 Volume으로 공유한다.

**③ 설정 파일 마운트**
- Nginx 설정 파일
- Spring 환경변수(properties/yml) 파일

**④ 로그 보관**
- 컨테이너 로그는 기본적으로 **컨테이너 내부**에 생성된다.
- 컨테이너가 삭제되면 로그도 함께 지워진다.

> ⚠️ 오류가 발생했을 때 컨테이너가 삭제되는 경우도 있는데, 이 경우 정작 원인 분석에 필요한 로그까지 같이 사라질 수 있다. 그래서 로그도 Volume(또는 외부 로그 수집기)으로 별도 보관해야 안전하다.

### 🗨️ 오늘의 회고

- Portainer를 배우면서 "GUI 도구 = 명령어를 대체하는 게 아니라 명령어를 더 쉽게 호출해주는 창구"라는 관점이 명확해졌다. 결국 내부적으로는 똑같이 Docker API를 호출하는 것뿐이라서, CLI를 모르고 Portainer만 쓰는 건 위험할 수 있겠다는 생각이 들었다.
- Docker Socket을 외부에 노출하면 안 된다는 부분이 가장 인상 깊었다. 단순히 "포트를 막아야 한다" 수준이 아니라, 이 소켓 하나로 호스트의 도커 엔진 전체를 통제할 수 있다는 걸 알고 나니, Portainer 같은 관리 도구일수록 보안 설계(Private Subnet, VPN/Bastion)가 더 중요하다는 게 와닿았다.
- "삭제한 사람이 아니라, 삭제할 수 있게 환경을 만든 사람이 책임진다"는 RBAC 관련 문장이 기억에 남는다. 권한 설계 자체가 운영 책임의 시작점이라는 인식을 갖게 됐다.
- Volume 파트는 지난주에 배운 "컨테이너는 비영속적이다"라는 명제를 실습 차원에서 다시 확인한 느낌이었다. 컨테이너의 쓰기 레이어가 컨테이너 생명 주기에 묶여 있다는 구조적 이유를 알고 나니, "왜 DB는 항상 Volume이랑 같이 써야 하는지"가 단순 암기가 아니라 이해로 남았다.
- 로그도 기본적으로 컨테이너 내부에 쌓인다는 점, 그리고 오류로 컨테이너가 삭제될 때 정작 디버깅에 필요한 로그까지 같이 날아갈 수 있다는 점은 실무에서 꼭 신경 써야 할 부분이라고 느꼈다. 다음 실습에서는 로그도 Volume이나 외부 로깅 도구로 분리해보고 싶다.
