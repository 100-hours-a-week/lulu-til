#### | 프록시 (Proxy)

**클라이언트와 서버 사이에서 중계자 역할을 하는 네트워크 장치**

보통 네트워크 통신은 `Client → Server`로 직접 이뤄지지만, 실무에서는 중간에 다른 서버를 하나 더 두는 경우가 많다.

```
Client → Proxy → Server
```

중간에 있는 저 서버를 **프록시 서버**라고 부른다. Proxy = 대리인이라는 뜻으로, 클라이언트와 서버 사이에서 요청과 응답을 중계하는 장치 또는 소프트웨어다.

| 구조 | 장점 | 단점 |
|---|---|---|
| `Client → Server` | 구조가 단순 | 보안, 로깅, 캐싱, 접근 제어 등을 중간에서 처리하기 힘듦 |

##### 프록시가 있을 때의 흐름

```
1. 클라이언트가 프록시에게 요청
2. 프록시가 서버에게 대신 요청
3. 서버가 프록시에게 응답
4. 프록시가 클라이언트에게 응답 전달
```

##### 프록시를 사용하는 이유

1. **보안 강화**
2. **트래픽 최적화** : 캐싱, 압축, 로드 밸런싱 등으로 성능 개선
3. **접근 제어** : 허용된 사용자나 IP만 특정 서비스에 접근하게 함
4. **로깅/감시** : 누가 어떤 요청을 했는지 기록
5. **익명성** : 클라이언트의 실제 IP나 서버의 실제 IP를 숨김
6. **정책 적용** : 회사나 학교 네트워크에서 접근 가능한 사이트 제한

> 💡 **"이 프록시는 누구를 대신하느냐"** 에 따라 종류가 나뉜다.

##### Forward Proxy

**클라이언트 대신 외부 서버에 요청을 보내는 프록시. 클라이언트의 대리자.**

```
클라이언트 → Forward Proxy → 서버
응답도 Forward Proxy가 받아서 클라이언트에 전달
```

- 목적 : 클라이언트 보호, 우회, 익명성, 사내 정책 적용
- 사용 예시 : 회사 인터넷 프록시, 학교 네트워크 프록시, 캐싱 프록시, 익명 프록시, 제한 우회

##### Reverse Proxy

**서버 대신 클라이언트의 요청을 먼저 받아 처리하는 프록시. 서버의 대리자.**

클라이언트는 실제 서버에 접근할 수도, 알 수도 없다. 대신 리버스 프록시에게 요청을 한다.

- 목적 : 서버 보호, 로드 밸런싱, SSL 처리, 캐싱

| 역할 | 설명 |
|---|---|
| **요청 라우팅** | `/api`는 API 서버, `/static`은 정적 파일 서버로 전달 |
| **로드 밸런싱** | 여러 서버 중 하나로 요청 분산 |
| **SSL/TLS 처리** | HTTPS 인증서 처리, 백엔드 서버 부담 감소 |
| **서버 보호** | 실제 서버 IP와 구조를 외부에 숨김 |
| **캐싱** | 자주 요청되는 응답을 저장해 서버 부하 감소 |
| **압축** | 응답 데이터를 압축해 전송량 감소 |
| **접근 제어** | 특정 IP, 경로, 헤더 조건으로 허용/차단 |
| **로그 기록** | 요청 IP, 경로, 응답 코드, 처리 시간 기록 |

**Reverse Proxy 대표 예시**

| 예시 | 설명 |
|---|---|
| **NginX** | Spring, Node.js 같은 애플리케이션 서버 앞에서 요청 중계 |
| **Apache HTTPD** | 웹 서버 겸 Reverse Proxy |
| **Envoy** | 마이크로서비스, 서비스 메시에서 자주 사용 |
| **HAProxy** | 고성능 로드밸런싱 프록시 |
| **CloudFront** | CDN 역할과 Reverse Proxy 성격 |
| **API Gateway** | API 요청을 받아 백엔드 서비스로 라우팅 |

##### Forward Proxy vs Reverse Proxy

| 구분 | Forward Proxy | Reverse Proxy |
|---|---|---|
| 누구의 대리자냐? | 클라이언트 | 서버 |
| 위치 | 클라이언트 쪽 | 서버 앞쪽 |
| 클라이언트가 아는 대상 | 프록시를 명시적으로 설정하는 경우 많음 | 보통 서비스 주소만 알고 프록시는 의식하지 못함 |
| 서버가 보는 요청자 | 프록시 | 프록시 |
| 주요 목적 | 클라이언트 보호, 접근 제어, 우회, 익명성 | 서버 보호, 로드밸런싱, SSL 처리, 캐싱 |
| 대표 예시 | 회사 인터넷 프록시, 캐싱 프록시 | Nginx, ALB, API Gateway, CDN |

> 💡 클라우드/백엔드 개발자 입장에서는 **Reverse Proxy**가 훨씬 중요하다.

##### 현대 웹의 구조

```
Client
  ↓
CloudFront / WAF
  ↓
ALB / Nginx / API Gateway
  ↓
Spring Boot / Node.js Server
```

##### 헷갈리는 것들

**Q1. 프록시는 보안 장비인가?**
→ 아니다. 프록시는 요청과 응답의 중계자이며, 보안 기능을 *수행하도록* 만들 수 있는 것이다.

**Q2. Forward Proxy, Reverse Proxy는 같은 장비인가?**
→ 기술적으로는 같은 소프트웨어로 둘 다 구현할 수 있다. 배치 위치와 역할이 다를 뿐이다.

**Q3. Reverse Proxy와 Load Balancer는 같은 것인가?**
→ 완전히 같은 개념은 아니다.
- 리버스 프록시 : 서버 앞에서 요청을 대신 받고 처리하는 중간자
- 로드 밸런서 : 여러 서버로 트래픽을 분산하는 기능

실무에서는 하나의 서버가 두 역할을 같이 수행하는 경우가 많다 (예: Nginx, ALB).

#### | VPN (Virtual Private Network)

**공용 인터넷 위에 암호화된 가상 터널을 만들어, 멀리 떨어진 사용자나 네트워크를 사설망처럼 안전하게 연결하는 기술**

우리는 여러 네트워크 장비를 거쳐서 통신을 하고, 기본적으로 공용 네트워크를 거친다. 특히 카페, 공항, 호텔 와이파이는 보안성이 약할 수 있다.

```
내 노트북 (VPN Client) → [암호화된 VPN 터널] → VPN Server → 회사 내부망/인터넷
```

##### VPN을 사용하는 이유

**1. 원격지에서 안전하게 내부망을 사용해야 할 때**
- 회사 내부에 있는 서비스들(DB, Session Store, WAS)은 외부에서 접근이 불가능하다
- 회사 네트워크 = 내부 네트워크 → 접근 가능
- 재택근무, 출장 = 외부 네트워크 → VPN 사용 필요

**2. 보안성 강화**
```
클라이언트 입장 : "나는 지금 카카오 서버에게 채팅을 보낼거야"
  domain : kakao.com
  payload : (채팅 내용)

VPN 터널링으로 이 내용을 통째로 감싸고 암호화

외부에서 보이는 HTTPS 요청에는:
  목적지 : VPN 서버
  내용 : 암호화된 데이터
```
- 익명성이 생긴다 → 내가 어떤 도메인에 요청하는지 외부에서 알 수 없다
- 외국 사이트 접근이 차단되는 경우에도 사용

**3. 떨어진 사설망들을 하나의 사설망처럼 묶을 때**

##### VPN 핵심 구성 요소

| 구성 요소 | 설명 |
|---|---|
| **VPN Client** | 사용자의 PC나 스마트폰에서 VPN 연결을 시작하는 프로그램 |
| **VPN Server** | VPN 접속을 받아 인증하고 터널을 만들어주는 서버 |
| **VPN Tunnel** | 클라이언트와 서버 사이의 암호화된 가상 통로 |
| **인증 정보** | 계정, 비밀번호, 인증서, MFA 등 사용자 확인 수단 |
| **암호화 키** | 데이터를 암호화하고 복호화하기 위한 키 |
| **가상 IP** | VPN 연결 후 클라이언트에게 할당되는 내부망용 IP |
| **라우팅 정책** | 어떤 트래픽을 VPN으로 보낼지 결정하는 설정 |

##### VPN 동작 흐름

```
1. VPN Client 실행
2. VPN Server로 연결 요청
3. 사용자 인증
4. 암호화 키 협상
5. VPN 터널 생성
6. 가상 IP 할당
7. 라우팅 테이블 변경
8. 트래픽 암호화 후 터널로 전송
9. VPN Server가 복호화 후 목적지로 전달
10. 응답도 다시 암호화되어 Client로 반환
```

##### 터널링이란?

**원본 패킷을 다른 프로토콜로 감싸서 보내는 방식**

```
원래 데이터 → VPN 프로토콜로 감싸기 → 암호화된 터널로 전송 → VPN 서버에서 꺼내서 사용
```

원래의 네트워크 통로 위에 가상의 터널을 하나 만들어서, 그 안에서만 통신이 이뤄지도록 하는 것이다.

##### VPN 연결 방식

| 방식 | 설명 |
|---|---|
| **Full Tunnel** | 모든 트래픽을 항상 VPN으로 보내는 방식 |
| **Split Tunnel** | 사내망에 가야 하는 트래픽만 VPN으로 보내고, 나머지는 일반 트래픽으로 내보내는 방식 |

##### 실무적인 관점

- **Bastion Server / Jump Server** : 운영 DB는 private network에 있어 외부망과 연결이 안 되어 있다. VPN을 통해 안전하게 접근.
- 사내 API 테스트
- 어드민 페이지를 외부에서 접속해야 할 때
- IDC ↔ Public Cloud Server(AWS) 간 LAN-LAN 연결

##### 헷갈리는 것들

**Q1. VPN을 쓰면 완전히 익명성을 가지나요?**
→ 아니다. VPN 사업자는 사용자의 통신 내역을 다 볼 수 있다.

**Q2. VPN과 HTTPS는 같은 것인가?**
→ 아니다.

| 구분 | VPN | HTTPS |
|---|---|---|
| 보호 대상 | 장치/네트워크 수준 트래픽 | 웹 브라우저와 웹 서버 사이의 HTTP 트래픽 |
| 사용 위치 | 네트워크 터널 | 웹 서비스 연결 |
| 예시 | 사내망 접속 | `https://example.com` 접속 |
| 관계 | VPN 안에서 HTTPS를 사용할 수도 있음 | VPN 없이도 HTTPS 사용 가능 |

#### | NAT (Network Address Translation)

**사설 IP와 공인 IP 주소를 변환하는 기술**

##### IPv4 주소의 한계

- IPv4 주소 범위 : `0.0.0.0 ~ 255.255.255.255` (약 40억 개)
- 폰, 스마트워치, IoT 등 인터넷 연결 장치가 점점 늘어나면서 부족해짐

##### 사설 IP와 NAT의 등장

| 구분 | 공인 IP | 사설 IP |
|---|---|---|
| 의미 | 인터넷에 등록되어야 하는 IP | 공개되지 않는 IP |
| 개수 | 40억 개로 제한됨 | 제한 거의 없음 |
| 비용 | 비쌈 | 비용 없음 |

→ 사설 IP를 쓰는 내부 네트워크가 외부와 통신할 때, NAT가 사설 IP ↔ 공인 IP를 변환해준다.

##### NAT 주소 변환 방식

###### 1. Symmetric NAT

**동일한 내부 IP/Port를 사용해도, 외부 대상에 따라 매핑 IP/Port를 별도로 생성**

```
출발지(내부)        NAT 변환             목적지
192.168.0.1:80  →  52.8.0.3:50001  →  8.8.8.8:80      (구글)
192.168.0.1:80  →  52.8.0.3:50002  →  168.126.36.1:80 (네이버)
192.168.0.1:80  →  52.8.0.3:50003  →  8.8.8.8:3000    (구글 채팅)
```

| 특징 | 설명 |
|---|---|
| 매핑 독립성 | 내부/목적지 별로 독립적인 매핑 테이블 |
| 보안 강화 | 외부에서 내부로 임의 접근 어려움 |
| P2P 연결 제약 | 비대칭 통신 제한 |
| 리소스 사용 | 목적지별 매핑 생성으로 NAT 테이블 크기 증가 |

- 사용처 : 외부 네트워크로부터 내부 네트워크 보호, 네트워크 관리 효율성, 비대칭 통신 차단이 필요한 기업 보안망, 정부기관, 보안 강화 내부망

###### 2. Cone NAT

**특정 내부 IP와 포트에 대한 NAT 매핑을 고정하고, 외부 IP에서 그 매핑에 접근 가능하도록 하는 방식**

```
출발지(내부)        NAT 변환             목적지
192.168.0.1:80  →  52.8.0.3:50001  →  8.8.8.8:80       (구글)
192.168.0.1:80  →  52.8.0.3:50001  →  168.126.6.1:80   (네이버)
```
→ Symmetric NAT와 달리 **같은 내부 IP/Port는 항상 같은 매핑(52.8.0.3:50001)**을 사용한다.

Cone NAT는 개방성에 따라 3가지로 나뉜다.

**① Full Cone NAT**
- 내부 클라이언트가 아웃바운드 요청을 한 번이라도 수행하면 매핑이 생성됨
- 이후 **어떤 외부 IP, Port든** 매핑된 정보로 클라이언트에게 패킷 전송 가능 (가장 개방적)
- 용도 : P2P 통신, 온라인 게임, 화상 통화 등
- 보안 : 허용 범위가 넓어 보안에 취약. NAT 장치가 외부 요청을 별도로 필터링하지 않음

**② Restricted Cone NAT**
- 내부 클라이언트가 아웃바운드 요청을 보낸 **외부 IP에만** 접근 허용 (Full Cone의 보안 문제 해결)
- 특징 : 송신 기록이 없는 외부 장치의 요청은 차단. 동일한 외부 IP라면 포트 번호와 무관하게 통신 가능
- 보안 : Full Cone NAT보다 안전하지만, 동일 외부 IP라면 여전히 접근 가능한 보안 이슈가 남아있음

**③ Port Restricted Cone NAT**
- Restricted Cone NAT의 접근 조건을 더욱 엄격히 제한
- **외부 IP + 포트 번호까지** 매칭 조건으로 추가
- 내부 클라이언트가 아웃바운드 요청을 보낸 외부 IP와 포트가 **정확히 일치**해야 접근 허용
- Cone NAT 방식 중 가장 보안성이 높음

| 특징 | 설명 |
|---|---|
| 매칭 조건 | 외부 IP + Port까지 일치해야 함 (동일 IP라도 포트가 다르면 접근 불가) |
| 보안 수준 | Cone NAT 중 가장 엄격함, 보안을 중시할 때 적합 |
| 사용처 | 금융 거래 서버, 기업 네트워크 등 |
| 한계 | 외부 무작위 접근 가능성은 최소화되지만, P2P 통신·일부 실시간 통신에서는 추가 설정 필요 |

##### NAT 방식 비교 정리

| 방식 | 매핑 조건 | 개방성 | 보안 |
|---|---|---|---|
| Symmetric NAT | 내부 + 목적지(IP+Port)별 독립 매핑 | 낮음 | 높음 |
| Full Cone NAT | 송신 1회 후 고정 매핑 | 매우 높음 | 낮음 |
| Restricted Cone NAT | 외부 IP 일치 | 중간 | 중간 |
| Port Restricted Cone NAT | 외부 IP + Port 일치 | 낮음 | 높음 |

#### | 네트워크 프로토콜

**네트워크 상에서 데이터를 주고 받는 규칙과 절차를 정의하는 체계**

데이터를 안전하고 신뢰성 있게 전송하기 위해, 모든 장치가 통신의 약속을 지키고 통신이 원활하게 이뤄지도록 하는 규칙이다.

##### 주요 네트워크 프로토콜

| 프로토콜 | 특징 |
|---|---|
| **TCP** | 신뢰성 있는 데이터 전송 보장, 패킷 전송/오류 검출 및 복구/흐름 제어, 연결 기반 통신, 데이터 순차 처리 |
| **UDP** | 비연결성 프로토콜, 데이터 전송 시 신뢰성 보장 안 함, 빠른 전송 속도 제공, 스트리밍/온라인 게임 등에 사용 |
| **ICMP** | 네트워크 진단, 오류 메시지 전송 용도, 네트워크 상태 확인, 오류 발생 시 알림, `ping` 명령어 등으로 사용 |
| **ARP** | IP 주소 기반으로 MAC 주소를 찾는 프로토콜, 네트워크 내 IP 주소를 가진 장치에게 MAC 주소를 얻기 위해 사용 |

##### 주요 응용계층 프로토콜

| 프로토콜 | 특징 |
|---|---|
| **HTTP, HTTPS** | 웹브라우저-서버 간 사용, 웹페이지 텍스트/이미지/비디오 전송, HTTPS는 보안 기능 추가 |
| **FTP** | 서버-클라이언트 간 파일 전송 프로토콜, 파일 업로드/다운로드 기능, 파일 서버에서 사용 |
| **SSH** | 안전한 원격 접속 제공 프로토콜, 원격 서버에 안전하게 접속, 암호화를 통한 기밀성 보장 |
| **SMTP** | 이메일 전송을 위한 프로토콜, 이메일 서버 간 메시지 전달, 클라이언트-서버 메일 전송 시에도 사용 |

#### | 패킷 (Packet)

**보내야 할 데이터를 잘게 쪼개서 통신하는 단위**

데이터를 한 번에 스트림으로 보내면 다른 요청이 계속 기다려야 할 수도 있다. 패킷 단위로 쪼개면 대기 없이 많은 장비에서 동시에 통신이 가능해진다.

> ⚠️ 패킷으로 잘게 나눠지면서 네트워크 데이터가 복잡해지는 문제가 발생했고, 이를 체계적으로 관리하기 위해 **TCP/IP**가 등장했다.

#### | TCP/IP

**(Transmission Control Protocol / Internet Protocol) — 인터넷에서 컴퓨터 간 통신을 위한 표준 프로토콜 스택**

대부분의 현대 네트워크 통신에 사용되며, TCP/IP를 중심으로 **4계층 구조**를 사용한다.

##### TCP/IP를 사용하는 이유

- **통신 표준화** : 서로 다른 장치가 데이터를 주고받는 규칙을 통일
- **확장성** : 다양한 네트워크 기술과 호환
- **신뢰성** : 신뢰성 있는 데이터 전송
- **인터넷 기반** : 인터넷의 근간 프로토콜, 전 세계 통용

##### 계층화하는 이유

- 복잡한 네트워크 과정에서 특정 역할을 담당하도록 분리
- 체계적인 데이터 전송 보장
- 네트워크 통신의 복잡성을 나눠서 관리
- 각 계층은 독립적으로 작동하면서, 다른 계층과 상호작용으로 통신을 완성

##### HTTP로 이미지 전송하기 — 계층별 캡슐화 과정

**1. 응용 계층 (HTTP 등)**
- 애플리케이션(Spring 등)에서 이미지 데이터 생성
- 데이터를 생성하고, 이를 "메시지"라고 부름
- 메시지에 데이터를 담아 전송 계층으로 전달

**2. 전송 계층 (TCP 등)**
- TCP : 신뢰성 있는 전송 담당
- 전달받은 메시지 데이터를 **세그먼트**로 자름
- MSS 단위 : 한 번에 보낼 수 있는 패킷 양
- 자른 뒤 각 데이터에 전송에 필요한 헤더를 붙임

```
[ 전송 헤더 | 세그먼트 데이터 ]
```

**3. 인터넷 계층 (IP)**
- 전송 계층에서 전달받은 세그먼트에 IP 헤더 추가
- 네트워크 간 경로 설정에 필요한 정보 포함
- 데이터의 목적지 IP, 출발지 IP 같은 정보를 헤더에 추가

```
[ IP 헤더 | 전송 헤더 | 세그먼트 데이터 ]
```

**4. 네트워크 인터페이스 계층**
- 물리적 네트워크 상 전송을 위한 계층
- 이더넷, 와이파이 등으로 전송되기 위한 **프레임** 생성
- MAC 주소 등이 헤더에 추가됨

```
[ IP 헤더 | 전송 헤더 | 세그먼트 데이터 ] ← 이 전체가 프레임으로 감싸짐
```

##### 캡슐화 / 디캡슐화

```
[송신측]                          [수신측]
응용 계층            ↓              응용 계층        ↑
전송 계층            ↓              전송 계층        ↑
인터넷 계층          ↓              인터넷 계층      ↑
네트워크 인터페이스   ↓              네트워크 인터페이스 ↑
```

- 보내는 쪽 : 각 계층을 지날 때마다 헤더를 덧붙이는 과정 = **캡슐화(Encapsulation)**
- 받는 쪽 : 전달받은 신호를 다시 응용 계층까지 올라가며 헤더를 하나씩 벗겨내는 과정 = **디캡슐화(Decapsulation)**

#### | TCP vs UDP

##### TCP 전송 (연결 기반)

```
클라이언트                서버
   ──── SYN ────→
   ←── SYN/ACK ──        (3-way handshake : 연결 수립)
   ──── ACK ────→
   - - - - - - - - - -   (연결 완료)
   ──── DATA ───→
   ──── DATA ───→
   ←──── ACK ────        (수신 확인)
```

- 연결을 먼저 수립(3-way handshake)하고 나서야 데이터를 전송
- 데이터마다 ACK(확인 응답)를 받아 신뢰성을 보장

##### UDP 전송 (비연결성)

```
클라이언트                서버
   ──── DATA ───→
   ──── DATA ───→
   ──── DATA ───→
   ── DATA/ERROR →       (손실되어도 확인하지 않음)
   ──── DATA ───→
   ──── DATA ───→
```

- 연결 수립 과정 없이 바로 데이터 전송
- 중간에 데이터가 손실돼도 재전송하거나 확인하지 않음 (신뢰성 X, 속도 우선)

| 구분 | TCP | UDP |
|---|---|---|
| 연결 방식 | 연결 기반 (3-way handshake) | 비연결성 |
| 신뢰성 | 보장 (ACK, 재전송) | 보장 안 함 |
| 속도 | 상대적으로 느림 | 빠름 |
| 사용 예 | 웹 요청, 파일 전송, 채팅 | 스트리밍, 온라인 게임, 실시간 음성/영상 |

#### | CIDR (Classless Inter-Domain Routing)

**라우팅할 때 트래픽 경로를 명확하게 지정하기 위해 네트워크 범위를 구분하는 주소 표준**

##### IP 주소의 구성

```
192.168.1.10
```
- 겉으로 보면 숫자 4개로 구성
- 각 숫자(옥텟)는 0~255 값을 가질 수 있음

IP 주소는 두 가지 정보로 구성된다.

| 영역 | 의미 | 예시 |
|---|---|---|
| 네트워크 영역 | 이 IP가 어느 네트워크에 속해 있는지 | `192.168.1` |
| 호스트 영역 | 그 네트워크 안에서 몇 번 장치인지 | `10` |

> ⚠️ 그런데 "어디까지가 네트워크 영역인가?"는 모호하다. `192`까지일 수도, `192.168`까지일 수도, `192.168.1`까지일 수도 있다. 그래서 필요한 게 **서브넷 마스크(또는 프리픽스 길이)**다.

```
192.168.1.10/24
→ 24 = 앞의 24비트가 네트워크 영역, 나머지 8비트가 호스트 영역
```

##### 네트워크를 나누는 방식 — 클래스 방식 (초기)

초기 IP 주소 체계는 네트워크를 A, B, C 클래스라는 **고정된 크기**로 나눴다.

| 클래스 | 기본 네트워크 크기 | 서브넷 마스크 | 설명 |
|---|---|---|---|
| A Class | /8 | 255.0.0.0 | 매우 큰 네트워크 |
| B Class | /16 | 255.255.0.0 | 중간 규모 네트워크 |
| C Class | /24 | 255.255.255.0 | 작은 네트워크 |

- 네트워크가 `/24`라면 호스트는 8비트 → 256개 중 **254개**의 호스트 사용 가능 (0, 255 제외)
  - `0` : 네트워크 자체를 나타내는 주소
  - `255` : 해당 네트워크 전체에게 보내는 브로드캐스트 주소
- 네트워크가 `/16`이면 호스트는 65,534개, `/8`이면 더 많은 호스트를 가짐

> ⚠️ 클래스를 고정된 크기로 나누다 보니 네트워크가 너무 크거나 너무 작은 문제가 발생했다. 이를 해결하기 위해 **CIDR 방식**이 등장했다.

##### CIDR 방식

서브넷 마스크를 클래스에 고정하지 않고, **필요한 크기만큼 자유롭게** 네트워크 영역을 정하는 방식.

IP 주소 뒤에 `/숫자`를 붙여, 앞에서부터 몇 비트를 네트워크 주소로 할당했는지를 표현한다.

| CIDR | 네트워크 비트 | 호스트 비트 | 사용 가능 호스트 수 |
|---|---|---|---|
| /16 | 16 | 16 | 65,534개 |
| /24 | 24 | 8 | 254개 |
| /26 | 26 | 6 | 62개 |
| /30 | 30 | 2 | 2개 |
| /32 | 32 | 0 | 단일 IP 1개 |

##### AWS 서브넷팅 예시

```
VPC: 10.0.0.0/16
  ├── Public Subnet A  : 10.0.1.0/24  → Load Balancer
  ├── Public Subnet B  : 10.0.2.0/24  → Load Balancer
  ├── Private Subnet A : 10.0.11.0/24 → App Server
  └── Private Subnet B : 10.0.12.0/24 → App Server
```

큰 VPC 네트워크(`/16`)를 CIDR로 더 작은 서브넷(`/24`)들로 나누고, Public/Private 용도에 맞게 분리해서 사용하는 구조다.

### 🗨️ 오늘의 회고

- 오늘 배운 내용이 **프록시 → VPN → NAT → 프로토콜 → TCP/IP → CIDR** 순으로 이어지는데, 전부 "어떻게 안전하고 효율적으로 네트워크 경로를 통제하느냐"라는 하나의 큰 질문에 대한 답이라는 걸 느꼈다.
- Reverse Proxy가 백엔드 개발자에게 왜 중요한지 명확해졌다. Nginx, ALB, API Gateway 같은 것들이 결국 다 Reverse Proxy의 구현체였다.
- VPN과 HTTPS를 헷갈렸었는데, 오늘 확실히 구분했다. VPN은 네트워크 수준의 터널, HTTPS는 애플리케이션(웹) 수준의 암호화. 둘은 함께 쓰일 수 있는 별개의 개념이다.
- NAT의 Cone 방식 3단계(Full → Restricted → Port Restricted)가 보안성이 강해질수록 P2P 연결성이 떨어지는 트레이드오프 관계라는 게 인상적이었다. 보안과 편의성은 항상 맞교환 관계다.
- TCP/IP의 캡슐화/디캡슐화 과정을 그림으로 보니, "헤더를 계속 덧붙이고 벗겨내는 과정"이라는 게 직관적으로 이해됐다. 계층마다 책임이 명확히 분리되어 있다.
- CIDR 표기법(`/24` 등)은 AWS VPC 설계할 때 실제로 계속 마주칠 개념이라 오늘 제대로 정리해둬서 다행이다.
- 5주차 과제 피드백 반영을 오늘 반드시 마무리하자.
