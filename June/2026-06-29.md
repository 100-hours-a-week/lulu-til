##### 2단계 — Docker + Nginx 배포 흐름

```
개발자 → GitHub (소스코드 push)
       → Docker 환경 (Dockerfile 작성 및 이미지 빌드)

#### | DevOps

**개발과 운영의 협업을 통해 소프트웨어 배포 속도와 안정성을 높이는 문제 해결 방식 및 방법론**

크게 **개발**과 **운영**, 두 영역으로 구성된다.

##### DevOps가 등장한 이유 — 기존 방식의 문제

개발팀과 운영팀이 분리되어 있을 때 추구하는 목표 자체가 서로 다르다.

| 팀 | 목표 |
|---|---|
| 개발 팀 | 기능을 빠르게 만들고, 빨리 배포하고 싶음 |
| 운영 팀 | 서비스를 안정적으로 서빙하고 싶음 (장애 위험 때문에 배포는 최대한 적고 안전하게) |

이 목표 차이에서 다음과 같은 **문제**가 발생한다.

**① 배포 지연**
- 개발팀은 빨리 배포하고 싶지만, 운영팀은 정해진 배포 일정에 맞춰 몇 가지 기능을 한꺼번에 릴리즈
- 배포의 덩어리가 커지고 → 테스트하고 안정화시키기 어려워짐
- 운영팀 입장에서는 배포가 잦을수록 서비스를 안정적으로 운영하기 어려움

**② 수동 작업**
- 개발팀은 원칙적으로 운영 서버에 직접 접속할 수 없음 (운영 서버 접속 자체가 위험한 행동)
- 그래서 운영팀이 직접 다음 과정을 수동으로 처리

```
서버 접속 → 파일 복사 → 명령어 실행 → 설정 수정 → 배포 실행
```

**③ 환경 차이**
- 개발 서버 ↔ 운영 서버 간 자바 버전, 라이브러리 버전 등이 다를 수 있음

**④ 책임 차이**
- 배포 후 장애가 발생했을 때 "코드 문제인지 서버 문제인지"를 가리기 애매해짐

> ⚠️ 결국 개발과 운영이 분리되어 있으면 **소통 지연 → 배포 지연 → 수작업 문제**로 이어진다. DevOps는 이 문제를 줄이기 위해 등장했다.

##### DevOps의 핵심 목표

> 더 자주 배포하는 것 자체가 목표는 아니다.
> 작은 변경을 더 빠르게 검증하고, 안전하게 배포하고, 문제가 생기면 빠르게 복구할 수 있는 환경을 만드는 것이 목표다.
> → 결국 **서비스의 품질을 높이는 것**이 본질적인 목표.

| DevOps가 추구하는 것 | 의미 |
|---|---|
| **빠른 전달** | 기능 수정이 사용자에게 도달하는 시간을 줄임 |
| **안정적인 배포** | 배포가 장애의 큰 원인이 되지 않도록 만듦 |
| **자동화** | 반복되는 빌드·테스트·배포 작업을 줄임 |
| **공동 책임** | 개발과 운영이 서비스 품질을 함께 책임짐 |
| **빠른 피드백** | 장애와 성능 문제를 관찰하고 개선에 반영 |
| **지속적 개선** | 한 번 구축하고 끝내지 않고 계속 개선 |

#### | DevSecOps

**개발, 보안, 운영을 통합해 소프트웨어의 개발부터 배포까지 모든 단계에서 보안을 적용하는 방법론**

##### DevOps의 허점

DevOps 흐름을 그대로 따라가다 보면 보안 공백이 생긴다.

```
개발 → GitHub 업로드 (네트워크를 탄다)
     → GitHub Actions로 빌드 (네트워크를 탄다, 키가 노출될 수 있다)
     → EC2에 배포
```

> ⚠️ 이 과정 어딘가에서 보안 문제가 발생할 수 있다. 즉, DevOps는 "빠르고 안정적인 배포"에 집중했을 뿐, **보안을 명시적으로 다루지 않는다.**

##### 보안 영역의 역할

소프트웨어 개발 및 배포 과정에서 발생 가능한 보안 위협을 **미리 탐지하고 예방**하며, 운영 과정에서 지속적으로 관리하여 보안 리스크를 최소화하는 역할.

| 구분 | 설명 |
|---|---|
| **보안 점검 자동화** | CI/CD 파이프라인 내 정적 코드 분석(SAST), 동적 애플리케이션 테스트(DAST) 등 자동화된 보안 점검 수행 |
| **취약점 및 의존성 관리** | 애플리케이션과 사용하는 라이브러리의 보안 취약점 주기적 점검 및 관리 |
| **보안 모니터링 및 대응** | 운영 중 발생 가능한 보안 위협 실시간 모니터링 및 즉각적인 대응 체계 구축 |
| **보안 컴플라이언스 관리** | 보안 정책 및 표준 준수를 위한 지속적인 관리와 점검 수행 |

##### DevOps와 DevSecOps의 차이점

핵심은 **보안 점검이 이뤄지는 시점과 접근 방식**이다.

| 구분 | DevOps (이후 별도 보안 적용) | DevSecOps |
|---|---|---|
| **보안 점검 시점** | 개발과 운영 단계 완료 후 별도로 진행 | 개발 단계부터 운영까지 지속적으로 통합하여 수행 |
| **보안 책임 주체** | 보안 전담 부서가 주도적으로 관리 | 개발자, 운영자, 보안 담당자가 공동으로 관리 |
| **보안 대응 속도** | 문제 발견 후 대응까지 시간이 오래 걸림 | 문제 발생 즉시 발견하고 대응 가능 |
| **리스크 관리** | 보안 문제를 뒤늦게 발견하여 비용과 리스크가 높음 | 보안 문제를 초기에 발견하여 비용과 리스크를 최소화 |

##### 사용 이유

- 배포 속도와 안정성을 유지하면서, 보안 문제를 초기에 발견하고 빠르게 해결하기 위해서
- 결국 **서비스의 퀄리티 보장**이라는 목표는 DevOps와 동일하지만, 보안이라는 축이 추가된 것

##### 사용 방법

CI/CD 파이프라인에 보안 점검 및 자동화를 통합하고, 개발 초기부터 배포 후 운영까지 지속적으로 보안을 관리한다.

| 순서 | 단계 | 설명 | 주요 기술 및 방법 |
|---|---|---|---|
| 1 | **개발 단계** | 소스 코드 수준의 보안 취약점 점검 및 의존성 관리 | 정적 코드 분석(SAST), 의존성 점검(Dependency Check) |
| 2 | **테스트 및 빌드 단계** | 실행 환경에서 보안 취약점 점검 | 동적 애플리케이션 테스트(DAST), 취약점 스캔 |
| 3 | **배포 단계** | 보안 설정 검증 및 정책 준수 확인 | 보안 컴플라이언스(Compliance) |
| 4 | **운영 단계** | 보안 문제의 실시간 대응 및 관리 | 모니터링 및 로깅, 보안 이슈의 즉각적 피드백 및 수정 |

##### DevSecOps 주요 보안 활동

| 용어 | 설명 |
|---|---|
| **정적 코드 분석 (SAST)** | 소스 코드를 실행하지 않고 코드 자체를 분석하여 잠재적인 보안 취약점을 미리 발견하는 기법 |
| **의존성 점검 (Dependency Check)** | 애플리케이션에서 사용하는 오픈소스 라이브러리나 외부 패키지의 보안 취약점 여부를 검사하여 안전한 버전 사용을 권장하는 과정 |
| **동적 애플리케이션 테스트 (DAST)** | 애플리케이션을 실제 실행한 상태에서 보안 취약점을 찾아내는 기법으로, 실행 중 발생 가능한 보안 문제를 점검 |
| **보안 컴플라이언스 (Compliance)** | 소프트웨어 및 인프라가 조직의 보안 표준이나 법적 요구사항을 준수하는지 확인하고 관리하는 활동 |

#### | Docker

**애플리케이션이 실행되는 데 필요한 모든 요소(코드, 라이브러리, 설정 등)를 하나로 묶어, 어떤 컴퓨터에서든 동일하게 실행되도록 도와주는 도구**

##### 도커 이전에 발생한 문제들

코드는 Git으로 공유했지만, **실행하는 사람마다 환경이 달랐다.**

예를 들어 Spring 서버를 만들었다면:
- 자바 버전, Gradle 버전, 운영체제 패키지, 데이터베이스 주소 등이 필요
- 이 중 하나라도 안 맞으면 → "로컬에서는 됐는데 서버에 올리면 안 된다"

| 문제 원인 | 구체적 내용 |
|---|---|
| 디폴트 버전 차이 | 개발자 컴�터는 이미 세팅이 되어 있지만, 운영 서버의 디폴트 자바/DB 버전은 다를 수 있음 |
| 쿼리 미지원 | 버전에 따라 지원하지 않는 쿼리가 있을 수 있음 |
| 패키지 매니저 차이 | `apt-get install` 같은 패키지 매니저는 기본 다운로드 버전을 자체적으로 관리하는데, 맥북의 `brew`와 서버의 `apt`가 기본으로 설치하는 버전이 다를 수 있음 |
| 포트 환경 차이 | 로컬에서는 모든 포트가 열려있지만, 운영 환경은 방화벽 → 보안 그룹 → 프로세스 순으로 통제됨 |

> ⚠️ 직접 수동으로 환경을 설정하면 문제가 발생하기 쉽다. 서버가 30대라면…

##### 기존 해결 방식 — 가상 머신(VM)

VM은 하이퍼바이저를 통해 **서비스마다 독립적인 OS를 제공**하는 방식이다.

```
물리 서버 또는 EC2
    └─ 하이퍼바이저
        ├─ 가상 머신 1 → Guest OS → 애플리케이션 A
        └─ 가상 머신 2 → Guest OS → 애플리케이션 B
```

| 구분 | 내용 |
|---|---|
| 장점 | 서비스별로 강하게 분리되어 운영체제를 다르게 사용할 수 있음. 하나의 VM 장애가 다른 VM으로 잘 전파되지 않음 |
| 단점 | 애플리케이션 하나를 실행하기 위해 항상 **전체 OS**를 실행해야 함 (운영체제, 시스템 프로세스, 커널 자원, 부팅 과정이 매번 필요) |

> 💡 안정적인 격리에는 좋지만, 서비스가 많아질수록 자원 사용량과 관리 비용이 커진다는 게 VM의 한계다.

##### Docker — 앱만 분리하자

도커는 운영체제 전체를 여러 개 복제하는 대신, **하나의 운영체제 위에서 애플리케이션 실행 환경만 분리**한다.

> 운영체제를 복사하지 말고, 앱 + 파일 + 라이브러리만 묶어서 실행하자.

| 비교 | 가상 머신 | Docker 컨테이너 |
|---|---|---|
| 격리 대상 | 운영체제 전체 | 애플리케이션 프로세스 |
| Guest OS | VM마다 필요 | 컨테이너마다 없음 |
| 커널 | VM마다 독립 | 호스트 Linux 커널 공유 |
| 실행 속도 | OS 부팅이 필요 | 프로세스 실행에 가까움 |
| 자원 사용량 | 상대적으로 큼 | 상대적으로 작음 |
| 주 용도 | 서버 단위 격리 | 애플리케이션 단위 배포·실행 |

##### Docker가 만들어내는 결과물 3단계

```
Dockerfile (실행 환경 설명서) → Docker Image (배포 패키지) → Docker Container (실행 상태)
```

| 개념 | 정의 | 비유 |
|---|---|---|
| **Dockerfile** | 애플리케이션을 실행하기 위해 필요한 작업을 코드로 작성한 파일 | 붕어빵 **레시피** |
| **Docker Image** | 애플리케이션을 실행할 수 있도록 준비된 결과물 (실행 환경까지 포함된 배포 패키지) | 붕어빵 **틀** |
| **Docker Container** | 이미지를 실제로 실행한 상태 | 구워진 **붕어빵** |

도커파일 예시:

```dockerfile
FROM eclipse-temurin:21-jre
WORKDIR /app
COPY app.jar app.jar
CMD ["java", "-jar", "app.jar"]
```

스프링 부트 이미지 안에는 보통 다음이 포함된다.
- 자바 런타임
- 애플리케이션 jar 파일
- 필요한 라이브러리
- 기본 실행 명령어
- 설정

> 💡 도커 이미지만 있으면 언제 어디서든 똑같은 환경으로 재사용할 수 있다.

##### 격리된 프로세스 — 컨테이너는 어떻게 분리되는가

EC2를 쓰는 것도 실제로는 하드웨어 장비를 공유받는 것처럼, 하나의 호스트 OS를 나눠 쓰면서도 서로 격리된 것처럼 보이게 만드는 게 핵심이다.

> 컨테이너는 별도의 컴퓨터가 아니라, **하나의 컴퓨터에서 보이는 환경과 사용 가능한 자원을 제한한 프로세스**다.

| 구성 요소 | 역할 |
|---|---|
| **커널 (Kernel)** | 애플리케이션이 CPU, 메모리, 디스크, 네트워크를 쓰도록 실제로 처리해주는 운영체제의 핵심 |
| **Namespace** | 컨테이너마다 서로 다른 프로세스·네트워크·파일시스템이 있는 것처럼 보이게 함 |
| **cgroups** | 컨테이너별로 CPU·메모리·프로세스 수 등을 얼마나 사용할지 제한하고 측정함 |

> ⚠️ **도커 컨테이너에는 커널이 없다.** 도커 이미지 안에는 자바 런타임, 라이브러리 등은 있지만, 커널은 호스트 OS의 것을 공유해서 쓴다. 그래서 Guest OS를 전체 부팅해야 하는 VM보다 도커 컨테이너 방식이 훨씬 가볍다.

**Namespace** — 컨테이너마다 프로세스, 네트워크, 파일시스템이 독립적으로 존재하는 것처럼 보이게 만드는 리눅스 커널 기능. 분리 레벨은 다음과 같다.

| 종류 | 설명 |
|---|---|
| **PID Namespace** | 프로세스 목록을 분리. 컨테이너가 자기 컨테이너 안의 프로세스만 보도록 만듦 → 장애 영향이 훨씬 덜 퍼짐 |
| **Network Namespace** | 컨테이너마다 독립적인 IP, 포트, 라우팅 테이블, 네트워크 인터페이스(eth0), DNS 설정 등을 분리 |
| **Mount Namespace** | 파일시스템을 분리 |

```
Docker Host
 ├─ MySQL Container   : 3306 / 172.18.0.3 / eth0
 └─ Spring Boot Container : 8080 / 172.18.0.2 / eth0
        └─ docker0 (가상 브리지)로 서로 통신
```

**cgroups** — 여러 프로세스를 하나의 그룹으로 묶고, 그 그룹의 CPU·메모리·I/O·프로세스 수 사용량을 제한하고 측정하는 리눅스 커널 기능.

> ⚠️ Namespace로 다른 컨테이너의 프로세스/파일/네트워크는 안 보이게 했어도, **같은 호스트 OS의 실제 자원은 공유**하고 있다. 만약 MySQL 컨테이너가 메모리 7GB를 쓰면, Spring Boot 컨테이너가 쓸 메모리가 부족해져서 서버 전체 성능 저하나 OOM이 발생할 수 있다. 그래서 cgroups로 **컨테이너별 메모리/CPU 최대치를 미리 제한**해두는 것이 중요하다.

**Namespace vs cgroups**

| 구분 | Namespace | cgroups |
|---|---|---|
| 핵심 질문 | 무엇이 보이는가? | 얼마나 사용할 수 있는가? |
| 주요 목적 | 격리 | 자원 제어 |
| 프로세스 목록 | 다른 컨테이너 프로세스를 안 보이게 함 | 프로세스 개수 제한 가능 |
| 네트워크 | 컨테이너별 IP·포트·라우팅 분리 | 네트워크 사용량 측정·일부 제어 가능 |
| 파일시스템 | 컨테이너별 루트 파일시스템처럼 보이게 함 | 직접적인 파일시스템 분리는 하지 않음 |
| CPU·메모리 | 분리하지 않음 | CPU·메모리 사용량 제한 |
| 대표 비유 | 각 팀에 다른 사무실을 보여줌 | 각 팀의 전기·회의실 사용량을 제한 |

##### 실무에서의 Docker 활용

```
애플리케이션 → Dockerfile → Docker Image → Registry에 저장 → 서버에서 Pull → 컨테이너로 실행
```

**① 로컬 개발 환경 통일할 때**
- 예: 6명이 한 팀으로 백엔드 서비스(MySQL, Redis, Spring, Java …)를 개발하는데, 각자 컴퓨터 설정이 다 다르면 곤란함 → 컨테이너로 환경을 통일시킨다.

**② 초기 프로젝트에서 단일 EC2 서버에 사용**
- EC2 하나에 다양한 서비스를 띄울 때, 컨테이너로 쪼개서 운영한다.

**③ 배포 과정에서 컨테이너 사용 (실무에서 가장 흔함)**

```
사용자 → EC2(Docker Host)
            └─ Nginx Container (외부 요청 수신)
                  └─ Spring Boot Container (비즈니스 로직)
                        ├─ Redis Container
                        └─ MySQL Container → Docker Volume (MySQL 데이터)
```

배포 흐름:

```
개발자 → GitHub (코드 Push)
       → GitHub Actions (CI/CD 실행, 테스트 및 Image 빌드)
       → Registry (Image Push)
       → EC2 (최신 Image Pull → 새 Container 실행)
       → App Container
```

> 💡 요즘은 EC2에 소스코드를 직접 복사하지 않는다. 코드는 GitHub에서만 관리하고, **빌드된 Docker 이미지**를 서버에 전달해서 사용하는 방식이 일반적이다.

**④ 컨테이너를 여러 개 사용할 때 — Docker Compose**

애플리케이션이 하나면 `docker run` 한 줄로 충분하지만, Nginx + Redis + MySQL + Spring처럼 여러 개를 동시에 띄워야 한다면 매번 환경변수·옵션을 따로 입력하기가 번거롭다. 이때 **Docker Compose**로 한 번에 정의해서 관리한다.

```
docker-compose.yml
 ├─ Nginx Container
 ├─ Spring Boot Container
 ├─ Redis Container
 ├─ MySQL Container
 ├─ Network
 └─ Volume
```

#### | Docker 사용 시 꼭 지켜야 하는 5가지 원칙

##### ① 이미지에는 비밀번호를 넣지 않는다

- Dockerfile에 DB 비밀번호를 직접 작성하면, 이미지 내부에 그 비밀번호가 그대로 남는다.
- 이미지는 암호화된 파일이 아니라서, **열어보면 그대로 보인다.**
- 즉 도커 이미지만 습득할 수 있으면 보안이 완전히 뚫릴 수 있는데, 이미지는 생각보다 습득하기 쉽고 보안을 신경 쓰지 않는 경우도 많다. 특히 **레지스트리(이미지 보관소) 자체의 보안이 소홀한 경우**가 많아 더 위험하다.
- 비밀번호, API Key, `.env` 파일 등은 다음과 같은 방식으로 분리 관리한다.
  - AWS Secrets Manager에 직접 저장하고, `${내계정_시크릿_APIKEY}` 형태의 변수로 주입
  - HashiCorp Vault 같은 비밀 관리 서비스 활용

##### ② 태그 중 `latest`만 사용하지 않는다

- 도커 이미지는 버저닝이 가능하다 (예: `spring-app:v1.0`).
- 그런데 실무에서는 종종 가장 최신 버전만 가져와서 배포하려고 `latest` 태그를 사용하는 경우가 있다.
- 이렇게 하면 **실제로 어떤 버전이 운영 중인지 디버깅 시점에 알 수 없다.** 운영 배포에서는 버전 명시가 필수다.

##### ③ 컨테이너 안에 데이터를 영구 저장하지 않는다

- 컨테이너는 데이터의 영속성을 보장하지 않는다. 기본 설정상 **컨테이너 종료 → 내부 데이터 삭제**가 원칙이다.
- 데이터의 영속성이 필요하다면 **Docker Volume**을 함께 사용해야 한다.

##### ④ 하나의 컨테이너에는 하나의 핵심 역할만 둔다

- Nginx + Spring + MySQL을 한 컨테이너에 다 띄우는 것도 기술적으로는 문제가 없긴 하다.
- 하지만 배포할 때 확장하거나 장애를 분석하기가 매우 어려워진다.
- 관리와 책임 분리 원칙을 위해, 역할별로 컨테이너를 분리하는 것이 좋다.

##### ⑤ 운영 환경에서는 로그를 컨테이너 밖에서도 볼 수 있게 해야 한다

- 로그는 컨테이너 내부 프로세스에서 발생하기 때문에, 기본적으로는 **컨테이너에 직접 접속해야만** 로그를 볼 수 있다.
- 컨테이너를 100개쯔음 운영한다면, 에러가 날 때마다 하나하나 접속해서 확인하는 건 현실적으로 불가능하다.
- 그래서 **옵저버빌리티(Observability)** 차원에서, CloudWatch · Loki · Datadog 같은 도구로 외부에 로그를 모아주는 설정이 필요하다.

#### | Dockerfile

**도커 이미지를 생성하기 위한 명령어 및 설정이 담긴 텍스트 기반의 스크립트 파일**

> "내 컴퓨터에서는 되는데 여기선 왜 안 되지?" → 환경 차이를 줄이기 위한 도구가 Docker이고, **그 환경을 코드로 정의한 설계서**가 Dockerfile이다.

도커파일에는 다음 내용을 코드로 작성한다.
- 이 애플리케이션은 **어떤 기반 환경**에서 실행되는가? (예: `node:20-alpine`)
- **어떤 파일**을 가지고 있는가? (예: `COPY src src`)
- **어떤 명령**으로 실행되는가?

> 💡 "도커만 쓴다고 환경 차이가 완전히 없어지는 건 아니다." 도커만 사용한다면, 컨테이너를 만들 때마다 환경 변수를 매번 수동으로 직접 입력해야 한다. 이 수동 과정 자체를 **파일로 코드화**하는 것이 Dockerfile의 역할이다.

##### Docker 실행 과정

```
Dockerfile (실행 환경 정의) → docker build → Docker Image (배포 패키지) → docker run → Container (실제 실행)
```

도커파일은 보통 **프로젝트 최상위 경로**에 둔다.

##### 핵심 명령어

| 명령어 | 질문 | 역할 |
|---|---|---|
| `FROM` | 어디서 시작하는가? | 기반 이미지 선택 |
| `WORKDIR` | 어디에서 작업하는가? | 컨테이너 내부 기본 작업 폴더 설정 |
| `COPY` | 어떤 파일을 넣는가? | 로컬 파일을 이미지 내부로 복사 |
| `RUN` | 이미지를 만들 때 무엇을 실행하는가? | 패키지 설치, 빌드 등 수행 |
| `EXPOSE` | 앱은 어느 포트를 사용하는가? | 컨테이너가 사용하는 포트를 문서화 |
| `CMD` / `ENTRYPOINT` | 컨테이너가 시작되면 무엇을 실행하는가? | 기본 실행 명령 지정 |

**1. FROM** — 어떤 OS, 어떤 런타임 환경에서 시작할지 정하는 명령어. 보통 다른 사람이 이미 만들어둔 이미지(예: `ubuntu`, `nginx`)에서 시작한다.

```dockerfile
FROM ubuntu        # 우분투에서부터 시작
FROM nginx         # nginx 기능을 그대로 사용
```

> `FROM`은 1번만 쓰이지 않을 수도 있다. **멀티 스테이지 빌드**라는 기법에서는 빌드용 이미지와 실행용 이미지를 분리해서 사용한다.
> - 빌드 단계 → `FROM jdk` (빌드에 필요한 도구 포함)
> - 실행 단계 → `FROM jre` (실행에만 필요한 최소 구성)
>
> 이렇게 분리하는 이유는 최종 이미지의 **경량화**를 위해서다.

**2. WORKDIR** — 컨테이너 안에서 작업할 기본 폴더를 지정한다.

```dockerfile
FROM jre21
WORKDIR /app
# 이후 명령어는 기본적으로 /app 폴더를 기준으로 실행한다.

COPY app.jar app.jar
# 로컬에 있는 app.jar → 이미지 내부의 /app/app.jar 로 복사한다.
```

**3. COPY** — 내 프로젝트 파일을 이미지 안으로 넣는다.

```dockerfile
COPY /build/libs/my-app.jar app.jar
# 로컬에 있는 my-app.jar
# → 컨테이너 내부 경로에 app.jar 라는 이름으로 복사한다.
```

**4. RUN** — 이미지를 만드는 도중에 실행되는 명령어. **컨테이너가 실행될 때** 실행되는 게 아니라, **이미지를 빌드하는 과정**에서 실행된다는 점을 구분해야 한다.

```dockerfile
RUN chmod +x gradlew
RUN ./gradlew bootJar --no-daemon
```

| 구분 | 실행 시점 |
|---|---|
| `RUN` | `docker build` 중 |
| `CMD` / `ENTRYPOINT` | `docker run`으로 컨테이너 시작 시 |

**5. EXPOSE** — 이 애플리케이션이 사용하는 포트를 문서화한다.

```dockerfile
EXPOSE 8080
# 이 컨테이너는 내부 애플리케이션이 8080 포트를 사용한다는 "표시"일 뿐
```

> ⚠️ `EXPOSE 8080`은 **호스트의 8080 포트를 자동으로 열어주는 게 아니다.** 단순한 문서화 목적의 명령어다.

**6. CMD + ENTRYPOINT** — 컨테이너가 시작되면 무엇을 실행할지 정의한다.

```dockerfile
ENTRYPOINT ["java", "-jar", "app.jar"]
# 컨테이너가 시작되면 java -jar app.jar 를 실행한다.
```

| 구분 | 의미 |
|---|---|
| **CMD** | 기본 명령 또는 기본 인자. `docker run` 실행 시점에 쉽게 덮어쓸 수 있다. ("기본 동작은 이렇게 하는데, 필요하면 실행할 때 바꿔서 써") |
| **ENTRYPOINT** | 컨테이너가 반드시 실행할 주 프로그램. 컨테이너 내부 프로그램의 실행 명령어 그 자체 |

**둘 다 꼭 같이 써야 할까?** 그렇지 않다. 무엇을 **고정**하고, 무엇을 **실행 시 바꿀지**에 따라 다음과 같이 구분해서 사용한다.

- **하나만 쓰는 경우**: 실행 명령어가 단순할 때. `CMD`만 쓰는 경우는 흔치 않은데, 보통 "항상 실행할 명령어를 고정하지 않고, 완전히 다른 명령어를 실행하고 싶을 때" 쓴다.
- **병행해서 쓰는 경우 (일반적)**: 실행할 프로그램은 고정하고, 실행 옵션만 쉽게 바꾸고 싶을 때.

```dockerfile
ENTRYPOINT ["java", "-jar", "app.jar"]
CMD ["--spring.profiles.active=prod"]

# docker run my-app 실행 시:
# java -jar app.jar --spring.profiles.active=prod

# 이미지는 한 번 만들어서 재사용하는데, profile만 dev로 바꾸고 싶다면:
# docker run my-app --spring.profiles.active=dev
# → java -jar app.jar --spring.profiles.active=dev
```

##### 완성된 Dockerfile 예시

```dockerfile
FROM eclipse-temurin:21-jre
# 이 이미지는 JRE 21 버전으로 시작

WORKDIR /app
# 이후 명령어는 /app 폴더에서 시작합니다.

COPY build/libs/*.jar app.jar
# build 결과물을 다 app.jar 로 컨테이너 내부에 복사

EXPOSE 8080
# 포트는 8080을 사용합니다.

ENTRYPOINT ["java", "-jar", "app.jar"]
# 컨테이너가 시작되면 java -jar app.jar 를 실행하세요.
```

빌드 및 실행:

```bash
# 프로젝트 최상단에서 도커 빌드
docker build -t my-spring-app:1.0 .

docker run -d \
  -p 8080:8080 \
  --name my-spring-app \
  my-spring-app:1.0
```

##### Dockerfile과 이미지 레이어 (★ 중요)

도커 이미지는 **레이어**로 구성되어 있고, 도커는 **이전 빌드에서 변경되지 않은 레이어는 그대로 재사용**한다.

```
Base Layer (Java Runtime)
   ↓
Dependency Layer (라이브러리 설치)
   ↓
Application Layer (JAR 또는 소스 코드)
   ↓
최종 Image
```

예를 들어 자바 버전은 잘 바뀌지 않으므로, 재사용을 위해 변경 빈도에 따라 레이어를 나눠서 구성한다.

```
각 단계를 나누고
// 자바 레이어
// 라이브러리 레이어
// jar 레이어
// 환경변수 레이어

매번 이미지를 새롭게 만들 때마다
변경되지 않은 단계는 기존의 결과를 그대로 재사용한다.
```

> ⚠️ 만약 다음처럼 작성하면, 소스 코드 중 단 하나만 바뀌어도 캐시가 전부 깨진다.
> ```dockerfile
> COPY . .          # 전체 소스코드를 복사
> RUN npm install    # 패키지를 설치한다.
> ```
> 전체 소스 코드 중 하나라도 수정되면 `COPY` 단계부터 다시 실행되어 캐시 효과가 사라진다.

**레이어 캐싱 흐름 예시 (Node.js 기준)**

```
Node.js 기반 Layer (Base) → 변경 없음 → 캐시 재사용
package.json Layer        → 변경 없음 → 캐시 재사용
npm install Layer         → 변경 없음 → 캐시 재사용
index.js Layer            → 코드 변경 → 새 Layer 생성
```

> 💡 이런 특징 때문에 **Dockerfile 내 명령어의 순서가 중요해진다.** 자주 변경되는 레이어(소스 코드 COPY 등)는 가능한 한 나중에 배치해야 한다. 빌드 속도 차이는 생각보다 크게 나며, 특히 스프링처럼 무거운 빌드라면 체감이 크다.

##### `.dockerignore`

`docker build` 시 빌드 컨텍스트에 **전달하지 않을 파일들**을 지정하는 파일.

```
.git           # git 커밋 이력 (컨테이너에 필요 없음)
.gradle        # 내부 의존성 파일, 의존성 설치 시 다시 생성됨
build
node_modules
*.log          # 과거 로그가 들어간다
.env           # 키/비밀번호가 들어갈 수 있음 (보안)
.env.*
```

##### Dockerfile 정리

| 질문 | 답 |
|---|---|
| 왜 필요한가? | 실행 환경을 수동 작업이 아니라 **코드**로 정의하기 위해 |
| 무엇을 만드는가? | Docker Image |
| 장점 | 실행 환경이 코드로 표준화되어 **재현성**이 높아지고, 모든 개발자가 같은 환경을 쓰게 되어 **협업성**도 높아짐 |

---

#### | Docker Image

**애플리케이션 실행에 필요한 파일을 여러 개의 읽기 전용 레이어로 쌓아 만든 실행 환경의 스냅샷**

```
Dockerfile (설계도, COPY/RUN/CMD 등) → 여러 레이어가 쌓인 하나의 이미지 생성 → 컨테이너는 이 이미지를 기반으로 실행
```

이미지를 레이어로 만드는 이유는, **전체 파일을 매번 새로 복사하지 않고, 이전 상태에서 바뀐 파일만 새로운 레이어에 기록**하기 위해서다.

##### 레이어의 본질

레이어는 **전체 파일 시스템의 복사본이 아니라**, 이전 레이어와 비교해 **추가·수정·삭제된 파일의 차이(diff)** 를 저장하는 단위다. 결과적으로 이전 빌드의 결과물 레이어를 캐싱하기 좋은 구조가 된다.

```
Layer 1 — 기본 Linux 파일
Layer 2 — Java Runtime 추가
Layer 3 — 라이브러리 추가
Layer 4 — my-api.jar 추가
```

> ⚠️ 레이어는 **폴더가 아니다.** 레이어마다 `/app`, `/etc`, `/usr` 같은 폴더가 통째로 존재하는 게 아니라, 그냥 이전 상태에서 새롭게 추가되거나 바뀐 파일만 기록된다.

```
Layer 1: /usr/bin/sh, /etc/passwd, /bin/ls
Layer 2: /usr/bin/java, /usr/lib/jvm/...
Layer 3: /app/my-api.jar
```

##### OverlayFS

여러 개의 읽기 전용 이미지 레이어를 겹쳐서, 애플리케이션이 **하나의 완성된 파일시스템**처럼 사용할 수 있게 하는 리눅스 파일 시스템 기능.

```
Image Layer 1 (기본 Linux 파일)
Image Layer 2 (Java Runtime)        →  merged (하나로 합쳐 보이는 파일시스템)
Image Layer 3 (애플리케이션 JAR)
```

기본 명령어:

```bash
# 이미지 다운로드
docker pull [이미지이름]:[태그]

# 컨테이너 생성 및 실행
docker run -d -p [호스트포트]:[컨테이너포트] --name [컨테이너명] [이미지이름]:[태그]
```

---

#### | Container

**도커 이미지를 바탕으로 실제 애플리케이션 프로세스를 실행한, 격리된 실행 단위**

| 개념 | 역할 |
|---|---|
| **Dockerfile** | 이미지를 어떻게 만들지 정의 |
| **Docker Image** | 애플리케이션 + 실행환경이 담긴 **읽기 전용** 결과물 |
| **Container** | 이미지를 기반으로 실제로 실행된 **격리 실행 단위** (호스트 OS 커널 위에서 실행되는 격리된 프로세스) |

```
Container A (App + Libraries + FS Layer)   Container B (App + Libraries + FS Layer)
              └──────────── 공유 ────────────┘
                  Host OS (Kernel, Linux 또는 Linux VM)
                  Hardware (CPU · Memory · Disk · NIC)
```

##### 컨테이너가 필요한 이유

이미지는 **공통 실행환경**만 제공할 뿐이고, 실제 서비스를 돌리려면 환경에 맞는 **이름 / 포트 / 환경변수**를 적용해서 실행해야 한다. 그 역할을 하는 게 컨테이너다.

```
개발 환경  → 개발 DB 주소 사용 → 8080 포트 사용
테스트 환경 → 테스트 DB 주소 사용 → 8081 포트 사용
운영 환경  → 운영 DB 주소 사용 → 80 또는 443 포트로 서비스
```

> 💡 스프링 앱의 도커 이미지는 동일하더라도, 실행되는 환경(컨테이너)에서 어떤 변수를 주입하느냐에 따라 개발/테스트/운영이 나뉜다.

##### 컨테이너 실행 흐름

```bash
docker run nginx
```

내부적으로는 다음 순서로 진행된다.

1. **레지스트리**에서 해당 이미지가 로컬에 있는지 확인
2. 없으면 외부 저장소에서 이미지를 가져오는 것을 시도 (`docker pull`)
3. 컨테이너를 만든다 (`docker create`)
4. 실제 실행 변수를 적용한다
5. 메인 프로세스를 실행한다 (`docker start`)
6. `-d` 옵션으로 백그라운드 실행 상태를 유지한다
7. 프로세스가 종료되면
8. 컨테이너도 함께 **"중지"** 된다

> ⚠️ 컨테이너는 **메인 프로세스가 끝나면 중지**된다. 내부에 메인 프로세스가 살아있는 동안만 컨테이너가 유지된다. 그래서 서버 프로세스는 보통 `-d` (detached, 백그라운드) 옵션으로 실행한다.

##### 컨테이너의 생명 주기

```
        docker create / docker run
                  ↓
              Created
           ↙ docker start        ↖ docker start
        Running ←──────────── Exited
                                  ↑
                  프로세스 종료 / docker stop
           ↓ docker rm -f                 ↓ docker rm
                          Removed
```

> 💡 도커는 컨테이너가 종료되어도 **자동으로 삭제되지 않는다.** 컨테이너 + 볼륨 + 이미지가 그대로 남는다. EC2에서 배포를 100번 반복했다면, 그동안 만들어진 이미지들이 디스크 용량을 계속 차지하게 된다. **중지(Stop)와 삭제(Remove)는 다른 개념**이라는 점을 꼭 구분해야 한다.

| 명령어 | 결과 |
|---|---|
| `docker stop` | Container를 멈춘다 |
| `docker start` | 멈춘 기존 Container를 다시 실행한다 |
| `docker rm` | Container 자체를 삭제한다 |
| `docker run` | 새 Container를 생성하고 실행한다 |
| `docker rmi` | 기존 이미지를 삭제한다 |

##### Image vs Container

| 구분 | Image | Container |
|---|---|---|
| 역할 | 실행 환경을 담은 템플릿 | Image를 실제로 실행한 단위 |
| 상태 | 읽기 전용 | 실행·중지·삭제 가능 |
| 생성 방법 | `docker build`, `docker pull` | `docker run`, `docker create` |
| 변경 | 일반적으로 직접 수정하지 않음 | 실행 중 파일 생성·수정 가능 |
| 예시 | `nginx:1.27` | `nginx-web` |
| 개수 관계 | 하나의 Image로부터 | 여러 Container 생성 가능 |

> 💡 우분투 계열 컨테이너 내부에는 `sh`는 기본 제공되지만 `bash`는 없을 수도 있고, 네트워크 명령어도 안 깔려있을 수 있다. 컨테이너는 기본적으로 **경량화**를 지향하기 때문이며, 필요하면 직접 설치해야 한다.

##### 자주 쓰는 명령어 모음

| 명령어 | 설명 |
|---|---|
| `docker ps` | 현재 실행 중인 Container 조회 |
| `docker ps -a` | 종료된 Container까지 모두 조회 |
| `docker run ...` | 새 Container를 만들고 실행 |
| `docker start <이름>` | 중지된 기존 Container 재실행 |
| `docker stop <이름>` | 정상 종료 절차를 거쳐 중지 |
| `docker restart <이름>` | 중지 후 다시 실행 |
| `docker logs <이름>` | 표준 출력 로그 확인 |
| `docker logs -f <이름>` | 새 로그를 계속 확인 (실시간) |
| `docker exec -it <이름> sh` | Container 내부 디버깅 |
| `docker stats` | CPU·메모리·네트워크 I/O 확인 |
| `docker rm <이름>` | 중지된 Container 제거 |

##### 실무적으로 컨테이너 다루기

**① 컨테이너는 이름을 붙여야 한다**
- 관리하는 서버가 많거나, 서버 내부에 컨테이너가 많을 수 있기 때문에 식별이 쉬워야 한다.

**② `docker run`을 반복해서 쓰지 말자**
- `docker run`은 기존 컨테이너를 찾아서 실행하는 명령어가 **아니다.** 매번 새로운 컨테이너를 만든다.
- 항상 `docker ps -a`로 기존 컨테이너 존재 여부를 먼저 확인할 것.
- 없으면 `docker create` + `docker start`로 명령어를 분리해서, 만들고 → 확인하고 → 시작하는 방식이 더 안전하다. (`create & start` = `run`)

**③ 운영 컨테이너는 직접 수정하지 않는다**
- 변경이 필요하면 **이미지를 수정**해서, 새로운 이미지를 빌드하고 그걸로 새 컨테이너를 만들어야 한다.
- 컨테이너 내부를 직접 고치는 건 휘발성이 강한 **Thin R/W 레이어**에만 수정을 가하는 것이라 위험하다.

**④ 컨테이너가 실행 중이라고 해서 정상 상태인 건 아니다**
- `docker ps`의 `RUNNING` 상태는 "프로세스가 살아있다"는 뜻일 뿐, **"서비스가 정상이다"와는 다르다.**
- `docker logs <이름>`으로 로그를 확인하고, `docker stats`로 자원 사용량을 함께 살펴봐야 한다.

**⑤ 운영 재시작 정책을 고려해야 한다**
- 서버는 언제든 재부팅될 수 있다. 도커도 서비스로 등록되어 있어 자동 재실행되는데, 이때 컨테이너가 어떤 정책으로 재시작될지 미리 정의해두는 것이 좋다.

#### | Registry

**컨테이너 이미지를 저장하고 배포하기 위한 중앙 저장소**

필요한 이미지를 언제든지 다운로드(`pull`)하거나 업로드(`push`)할 수 있고, **private** 또는 **public**으로 설정해서 조직 내부 또는 외부 사용자와 이미지를 공유할 수 있다.

| 레지스트리 서비스 | 설명 |
|---|---|
| **Docker Hub** | 가장 널리 쓰이는 공개 레지스트리 |
| **Amazon ECR** | AWS에서 제공하는 프라이빗/퍼블릭 레지스트리 |
| **GCR (Google Container Registry)** | GCP에서 제공하는 레지스트리 |

### 🗨️ 오늘의 회고

- 오늘은 "빅뱅 배포 → Docker/Nginx 배포 → (이후 단계)"로 이어지는 배포 파이프라인 실습의 큰 그림을 먼저 잡고, 그 배경이 되는 **DevOps → DevSecOps → Docker**의 흐름을 깊게 파고든 날이었다.
- DevOps가 "더 자주 배포하는 것"이 목표가 아니라 "안전하게 검증하고 빠르게 복구할 수 있는 환경"이 목표라는 점이 인상적이었다. 배포 빈도는 결과일 뿐, 본질은 **품질**이라는 걸 다시 새겼다.
- DevSecOps는 DevOps의 연장선이라기보다, DevOps 파이프라인 자체(GitHub → Actions → EC2)에 숨어 있는 **보안 공백을 메우는 관점**으로 이해하니 훨씬 명확해졌다. "왜 또 별도 개념이 필요하지?"라는 의문이 풀렸다.
- Docker 부분이 분량도 가장 많고 핵심이었다. 특히 **Namespace(무엇이 보이는가)와 cgroups(얼마나 쓸 수 있는가)**를 구분해서 이해하니, "왜 컨테이너는 가벼운데도 서로 격리될 수 있는지"가 명확해졌다. 지난주 배운 가상화/하이퍼바이저 개념이 VM과 Docker를 비교하는 데 그대로 연결됐다.
- Dockerfile의 레이어 캐싱 원리(자주 바뀌는 레이어를 뒤로 보내야 빌드가 빨라진다)는 단순히 "그렇게 하라"가 아니라 **이미지가 레이어의 누적이라는 구조**를 이해해야 왜 그런지 납득이 갔다. `COPY . .`를 맨 위에 두면 캐시가 통째로 무효화된다는 점은 직접 Dockerfile을 짤 때 꼭 기억해야겠다.
- "이미지에 비밀번호를 넣지 않는다", "latest 태그만 쓰지 않는다" 같은 원칙들은 당장 실습에서는 편하다고 대충 넘어가기 쉬운데, 운영 단계에서 사고로 이어지는 지점이라는 걸 강조한 게 좋았다. 다음 실습(GitHub Actions 자동화 배포) 단계에서 직접 신경 써봐야겠다.
