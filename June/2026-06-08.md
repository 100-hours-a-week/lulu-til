#### | GPT 동작 방식 이해

**지금까지 나온 문장을 보고, 다음에 올 단어의 확률을 계산해 가장 높은 확률의 단어를 선택해 생성하는 AI**

##### 동작 과정

```
1. 질문 이해
2. 단어 확률 계산 ← 핵심
   "다음에 어떤 말이 오는게 적절한가?" 를 확률로 계산
3. 문장 생성
   확률로 선택된 단어를 이어 붙여 자연스러운 문장 완성
```

##### LLM에 대한 오해

| 오해 | 실제 |
|---|---|
| 정답을 알고 있다 | 정답이 아니라 가장 가능성이 높은 단어를 생성한다 |
| 생각을 한다 | 확률을 계산하는 것이다 |
| 인터넷을 검색한다 | 기본적으로 이미 학습된 데이터를 기반으로 생성한다 |

> 💡 GPT는 **생각하고 정답을 아는 AI가 아니라, 말을 이어붙이는 확률 계산기**다.

##### 한계와 극복 방법

**한계**
- 학습 데이터의 출처 문제 (인터넷 글은 항상 신뢰할 수 없다)
- 학습된 정보는 최신 정보가 아니다
- 100% 사실이 아닌 높은 확률의 말을 만들어낼 뿐이다

**한계 극복 방법**

| 방법 | 설명 |
|---|---|
| **RAG (검색 증강)** | 외부 기억 장치를 사용해 기억력 한계 극복. 예) `commit_role.md`를 RAG로 찾아 쓸 수 있게 한다 |
| **프롬프트 엔지니어링** | 후카츠 프롬프트, 생각의 트리(Tree of Thoughts), 프로파일 기법 등으로 더 정확한 결과 유도 |

#### | 운영체제 (Operating System)

**하드웨어와 소프트웨어를 관리하며, 사용자와 컴퓨터 간 상호작용을 가능하게 하는 시스템 소프트웨어**

##### 왜 필요한가?

| 역할 | 설명 |
|---|---|
| **자원 관리** | CPU, 메모리 등 하드웨어 자원을 분배해 여러 프로그램이 충돌 없이 실행되도록 |
| **사용자 편의성** | 복잡한 기계 동작을 추상화해 사용자가 쉽게 명령을 내릴 수 있는 환경 제공 |
| **멀티태스킹** | 여러 작업을 동시에 실행할 때 간섭을 방지 |
| **보안 및 안정성** | 데이터와 시스템을 보호하고, 오류 발생 시 복구 절차 진행 |

##### 운영체제의 주요 계층

```
사용자 인터페이스  ← GUI(Mac, Windows) / CLI(Linux)
        ↓
시스템 콜 (System Call)  ← 응용 프로그램이 핵심 서비스에 접근하는 표준 인터페이스
        ↓
커널 (Kernel)  ← 하드웨어 자원 관리. 하드웨어 ↔ 프로그램 사이 중재자
        ↓
하드웨어
```

##### 리눅스를 서버에 쓰는 이유

서버는 24시간 7일 내내 돌아야 하고, 서버 자원(RAM, CPU)은 비싸다.

- **GUI** : 사용자가 편하지만 자원을 많이 쓴다
- **CLI** : 사용자가 불편하지만 자원을 적게 쓰고 제어가 훨씬 강력하다
  - 텍스트 명령어 → 코드로 작성 가능 → 서버 300대도 자동화 스크립트 하나로 세팅

> 💡 `매출 - 비용 = 이익` 구조에서 서버 비용을 줄이는 게 곧 이익. CLI 기반 리눅스가 선택되는 이유다.

#### | 유저 모드와 커널 모드

운영체제에서 프로세스 실행 환경을 구분하는 두 가지 실행 모드

> 만약 자바 프로그램이 직접 메모리에 접근한다면? → 문제 발생
> 일반 프로그램의 중요한 하드웨어 작업은 운영체제에서만 하도록 권한을 분리한 것

| 구분 | 유저 모드 | 커널 모드 |
|---|---|---|
| **실행 주체** | 우리가 만든 앱 / 자바 / 쉘 | OS 커널 |
| **접근 권한** | 제한적 (하드웨어 접근 불가) | 무제한 (하드웨어 제어 가능) |
| **예시** | `System.out.println()`, `cat`, `ls` | `read()`, `write()`, `fork()`, `exec()` |
| **전환 방법** | 시스템 콜(System Call) | 인터럽트 발생 후 진입 |

##### 시스템 콜 (System Call)

유저 모드 프로그램이 운영체제의 커널 기능을 요청하는 **공식 통로(인터페이스)**

```
커널아, 나 파일 좀 읽고 싶어
커널아, 메모리 더줘
커널아, 네트워크 데이터 보내줘
```

**실제 시스템 콜 흐름 예시 — 자바에서 파일 읽기**

```java
FileReader reader = new FileReader("test.txt");
int c = reader.read();
```

```
1. 유저 모드
   자바에서 read 메서드 → OS한테 파일 읽기 요청

2. JVM → OS 커널에 요청 전달
   JVM 내부에서 C언어 함수 read → 리눅스 시스템 콜로 호출

3. 커널 모드 전환
   시스템 호출 인터럽트 → 커널 모드 전환
   커널이 test.txt를 실제 파일 디스크에서 읽어온다

4. 결과를 유저 모드에게 전달
   읽은 데이터를 커널 → JVM 전달
   CPU가 유저 모드로 복귀
```

> 💡 터미널에서 `cat test.txt`를 실행해도 동일하다.
> `cat`은 유저 모드에서 실행되는 프로그램이지만, 내부적으로는 `open`, `read`, `write` 시스템 콜을 호출하고, 커널이 실제로 읽어서 cat 프로그램에 반환한다.

#### | 메모리 구조

**프로그램이 빠르고 안정적으로 실행될 수 있도록 RAM을 용도에 맞게 4가지 구역으로 나눠 관리하는 방식**

> 주방, 침실, 화장실처럼 용도에 맞게 공간을 분리하는 것과 같다.

| 영역 | 역할 | 저장되는 데이터 | 특징 |
|---|---|---|---|
| **코드(Code) 영역** | CPU가 참조할 명령어가 저장되는 공간 | 소스 코드, 제어문, 함수 | 읽기 전용 (실행 중 코드 변경 방지) |
| **데이터(Data) 영역** | 프로그램 시작부터 끝까지 유지되는 데이터 공간 | 전역 변수, 정적 변수(Static) | 프로그램 실행과 동시에 할당, 종료 시 소멸 |
| **힙(Heap) 영역** | 필요할 때마다 동적으로 빌려 쓰고 반납하는 자유 공간 | 파일 내용, 사용자가 입력한 텍스트, 자바의 객체 | 런타임에 크기가 결정됨 |
| **스택(Stack) 영역** | 잠깐 사용하고 버릴 임시 데이터가 쌓이는 작업 공간 | 지역 변수, 매개 변수, 함수 호출 내역 | 함수가 끝나면 자동 삭제 |

##### 왜 굳이 4개로 나누나?

1. **안정성 확보** : 코드 영역은 데이터가 실수로 덮어쓰이지 않도록 분리
2. **자원 관리 효율화** : 프로그램 종료까지 필요한 데이터 vs 바로 버릴 데이터를 분리
3. **유연성 제공** : 실행 전까지 얼마나 많은 메모리가 필요한지 모르므로(파일 1MB vs 100GB), 동적으로 쓸 수 있는 힙 공간이 별도로 필요

#### | 메모리 관리자 (Memory Manager)

**운영체제의 핵심 구성요소. 디스크에서 필요한 데이터를 메모리에 올리고, 다 쓴 데이터를 치우는 역할**

```
CPU(셰프) ↔ Memory(도마) ↔ Disk(냉장고)
```

- CPU는 메모리에 있는 데이터만 처리할 수 있다
- 디스크에 있는 데이터를 메모리에 올려서 CPU에 공급
- 주문이 50개인데 CPU가 동시에 10개만 처리 가능 → 10개의 메모리 공간 할당 + 다 쓴 데이터 치우기 = 메모리 관리자의 역할

##### 왜 사용하나?

1. **제한된 자원 최적 활용** : 메모리는 비싸고 한정적, 빈 공간 없이 최대한 활용
2. **프로세스 간 충돌 방지** : 카카오톡이 점유한 메모리 공간에 다른 프로그램이 접근하면 안 됨
3. **물리적 메모리 한계 극복** : 가상 메모리 기법으로 실제 RAM보다 큰 프로그램 실행

#### | 메모리 주소 체계

**CPU가 메모리에 있는 데이터에 접근하기 위한 고유 주소 체계**

메모리를 1byte 단위로 나누고 각각에 고유한 주소를 부여해 관리한다.

##### 1. 논리 주소 (Virtual Address)

- **프로그램이 바라보는 가상의 메모리 주소**
- 실제 메모리 주소가 아님
- 자바에서 객체의 메모리 주소를 찍으면 → 이 가상의 논리 주소를 보는 것
- 프로그램은 자기가 0번지부터 메모리를 독점해서 사용한다고 생각함

> 💡 프로그램을 짜면서 "30번 주소는 내 거고 31번은 다른 프로그램이 쓸 수 있으니 쓰지 말자"는 게 가능할 리 없다. 논리 주소 덕분에 각 프로그램이 독립된 공간을 가진다고 착각(?)하며 개발할 수 있다.

##### 2. 물리 주소 (Physical Address)

- **실제 RAM에 납땜되어 있는 물리적인 전기 주소**

##### 3. MMU (메모리 관리 장치)

- **논리 주소 → 물리 주소 변환 담당**

```
CPU: "메모장 프로그램에서 논리 주소 10번에 있는 데이터 가져와"
MMU: "메모장 프로그램의 가상 주소 10번 → 물리 주소 변환 → 실제 데이터 반환"
```

#### | 메모리 오버레이 vs 메모리 스왑

##### 메모리 오버레이

**제한된 메모리 환경에서 프로그램의 필요한 부분만 메모리에 로드해 실행하는 기법**

- 개발자가 직접 코드를 모듈로 쪼개고, 지금 필요한 조각만 메모리에 올려 사용
- 요즘은 잘 사용하지 않음 → 개발자가 메모리 공간을 고려하며 코드를 짜야 해서 개발 난이도가 극도로 높아짐

##### 메모리 스왑 (Memory Swap)

**사용하지 않는 데이터를 디스크의 스왑 공간으로 이동시켜 RAM을 확보하고, 필요할 때 다시 메모리로 복구하는 방식**

```
메모리 (RAM)                    보조기억장치 (Disk)
  실행할 데이터  ←── 스왑 인 ──    스왑 영역
  사용 안 하는 데이터 ── 스왑 아웃 ──→
```

**문제점**
- 속도가 상당히 느려짐 (RAM ↔ Disk I/O 발생)
- **스레싱(Thrashing)** : 메모리가 극도로 부족할 때 스왑 인/아웃이 폭주 → 스왑 처리 자체가 프로세스를 점유 → 컴퓨터가 멈춘 것처럼 보임

> ⚠️ **쿠버네티스, 도커 환경에서는 스왑이 불가능하다.**

| 구분 | 오버레이 | 스왑 |
|---|---|---|
| **관리 주체** | 프로그래머 | 운영체제 |
| **관리 단위** | 하나의 프로그램 내부의 모듈 단위 | 실행 중인 프로세스 단위 |
| **목적** | 단일 프로그램의 크기 제한 극복 | 여러 프로그램을 동시에 실행하기 위한 공간 확보 |

#### | 메모리 관리 단위 (메모리 분할)

여러 프로세스가 물리 메모리를 효율적으로 공유하기 위해 메모리를 블록으로 나누는 기법

##### 4가지 방식 비교

| 항목 | 고정 분할 | 가변 분할 | 페이징 | 세그멘테이션 |
|---|---|---|---|---|
| **분할 기준** | 고정 크기 블록 | 프로세스 크기 | 고정 크기 페이지 | 논리적 단위 (코드, 데이터, 스택 등) |
| **단편화 문제** | 내부 단편화 발생 | 외부 단편화 발생 | 외부 단편화 없음 | 외부 단편화 발생 가능 |
| **주소 변환 복잡성** | 단순 | 단순 | 페이지 테이블 필요 | 세그먼트 테이블 필요 |
| **적용 사례** | 단순한 환경 (임베디드 등) | 동적 메모리 환경 | 현대 운영체제 | 논리적 구조 관리가 필요한 경우 |

**1. 고정 분할 방식**
- 메모리를 사전에 고정된 크기의 블록으로 나눠두고 프로세스를 하나씩 넣음
- ✅ 관리 쉬움, 할당 속도 빠름, 충돌 가능성 낮음
- ❌ **내부 단편화** : 할당된 블록보다 프로세스가 필요한 공간이 작아서 블록 안에 빈 공간 낭비

**2. 가변 분할 방식**
- 프로세스의 실제 크기에 딱 맞게 메모리를 동적으로 할당
- ✅ 메모리 효율적
- ❌ **외부 단편화** : 프로세스들이 들어갔다 나왔다 반복하며 메모리 중간중간에 작은 빈 공간들이 듬성듬성 남아, 전체 공간은 충분한데 연속된 공간이 없어서 새로운 큰 프로세스를 못 넣는 상황
  - 해결책 : 메모리 압축(조각 모음) → But 이 과정에서 메모리 주소를 바꿔야 해서 **컴퓨터가 멈춘다**

**3. 페이징 방식**
- 메모리와 프로그램 모두를 동일한 크기의 페이지로 나눠 할당
- ✅ 메모리를 연속적으로 비워둘 필요 없음
- ❌ 페이지 테이블을 위한 추가 메모리 사용, MMU를 통한 주소 변환 시간 소요

**4. 세그멘테이션 방식**
- 프로그램을 논리적 단위(코드, 데이터, 스택 등)로 나누어 연속된 공간에 할당
- 컴파일러-링커가 논리적 의미 단위로 코드를 잘라냄
- ✅ 프로그램의 구조를 그대로 반영, 데이터 보호·공유 편리
- ❌ 세그먼트 크기가 제각각 → 외부 단편화 발생 가능

**현대 OS의 방식 : 세그멘테이션 + 페이징 혼합**

```
프로그램
  ↓ 세그멘테이션으로 먼저 나눔 (논리적 단위)
  ↓ 다시 페이징으로 나눔 (고정 크기)
세그멘테이션의 장점 + 페이징의 장점 결합
```

---

#### | 가상 메모리 (Virtual Memory)

**물리적인 RAM보다 큰 프로그램을 실행하면서, 동시에 여러 개의 프로세스를 안전하게 실행하려는 추상적 개념**

페이징과 세그멘테이션은 가상 메모리를 구현하기 위한 기법이다.

---

#### | ⭐ 프로세스와 스레드

##### 프로세스 (Process)

**실행 중인 프로그램의 흐름. 운영체제에게 자원을 할당받아 독립적으로 실행되는 단위.**

하나의 프로그램이라고 이해해도 무방하다.

##### 스레드 (Thread)

**프로세스 내에서 병렬 처리를 위해 생성되는 실행 흐름의 최소 단위.**

##### 왜 알아야 하나?

우리는 그냥 코드를 작성하는 게 아니다.
**성능을 최적화하면서, 가장 저렴하게 서빙하는 시스템을 개발해야 한다.**

- 프로세스 → 멀티태스킹이 어떻게 수행되는지 이해, 시스템 자원의 효율적 관리
- 스레드 → 동시성 + 병렬성을 활용해 프로그램 실행 속도 + 반응성 극대화

##### 프로세스 vs 스레드 비교

| 구분 | 프로세스 | 스레드 |
|---|---|---|
| **메모리** | 독립된 공간 사용 (Code, Data, Heap, Stack 각자) | 프로세스 내 메모리 공유 (Code, Data, Heap 공유, Stack은 각자) |
| **통신** | 복잡함 (프로세스 간 통신 필요 → IPC) | 용이함 (같은 메모리 공유) |
| **안정성** | 서로 영향을 주지 않아 높음 | 하나가 죽으면 전체 프로세스에 영향 |
| **속도** | 전환(Context Switching) 비용 큼 | 전환 비용이 작아 빠름 |

> 💡 **멀티 스레드 대표 주자 → 자바 Web Spring** (요청마다 스레드가 만들어진다)
> **싱글 스레드 대표 주자 → Node.js** (싱글 스레드 기반의 이벤트 루프)
>
> **스레드가 많다고 성능이 좋아지는 건 아니다. 스레드를 어떻게 쓰냐가 핵심.**

---

#### | IPC (Inter-Process Communication)

**프로세스 간 통신. 독립적인 메모리 공간을 가진 프로세스들이 데이터를 주고받기 위한 운영체제 제공 메커니즘.**

기본적으로 프로세스는 독립적인 메모리 공간을 가지므로 다른 프로세스의 메모리에 직접 접근 불가 → IPC 필요

##### IPC 주요 방식

**1. 메시지 전달 (Message Passing)**
```
프로세스 A → 커널 → 프로세스 B
```
- 운영체제 커널을 통해 메시지를 주고받는 방식
- 직접적인 메모리 공유 없이 **안전하게** 통신
- 단, 커널을 거치므로 상대적으로 느림

**2. 공유 메모리 (Shared Memory)**
```
프로세스 A ↔ [공유 메모리 영역] ↔ 프로세스 B
```
- 프로세스들이 공유할 수 있는 메모리 공간을 직접 생성
- 통신 속도가 빠름
- ❗ **동기화 필수** : 동시 접근 시 충돌 위험 → 접근 프로세스가 많을수록 동기화 문제 빈번

---

#### | PCB (Process Control Block, 프로세스 제어 블록)

**운영체제가 각 프로세스를 효율적으로 관리하기 위해 프로세스의 상태, CPU 레지스터 값, 메모리 정보 등 모든 실행 정보를 담아두는 핵심 자료 구조**

##### 왜 필요한가?

CPU는 여러 프로세스를 아주 빠른 속도로 번갈아가며 실행하는데, **CPU 자체는 기억하지 않는다.**

```
유튜브 프로세스에서 14번 명령어 실행 → CPU에게 "나 잠깐 끝났어"
카카오톡 프로세스 실행 → "이전에 어디까지 실행했지?" ← 이걸 기억하는 게 PCB
```

##### 컨텍스트 스위칭 (Context Switching)

**프로세스를 전환할 때 현재까지 작업 사항을 PCB에 저장하고, 돌아올 때 PCB를 보고 작업을 이어서 시작하는 과정**

```
Process P0 실행 중
    ↓ 인터럽트 / 시스템 콜 발생
P0의 상태 → PCB0에 저장
    ↓
Process P1의 상태 → PCB1에서 복원
    ↓ P1 실행
    ↓ 인터럽트 / 시스템 콜 발생
P1의 상태 → PCB1에 저장
    ↓
P0의 상태 → PCB0에서 복원 → P0 재실행

컨텍스트 스위칭 비용 = 약 5ms
```

> 💡 CPU가 작업을 중단했다가 돌아온 것을 우리는 모른다. PCB 덕분에 이음새 없이 이어진 것처럼 느껴지는 것이다.

##### PCB의 역할

1. **프로세스 생성** : 새 프로세스가 생성되면 운영체제가 PCB를 생성하고 프로세스 ID와 초기 상태를 기록
2. **실행 상태 기록** : 현재 CPU 레지스터 값, 프로그램 카운터 등 실행 상태 저장
3. **문맥 교환 관리** : Context Switching 시 이전 상태 보존 및 복원
4. **자원 정보 관리** : 프로세스가 사용 중인 파일, 메모리 등 자원 목록 관리
5. **프로세스 종료** : 종료 시 PCB 삭제 및 자원 반환

### 🗨️ 오늘의 회고

- GPT가 "확률 계산기"라는 관점이 신선했다. LLM의 한계를 이해하면 RAG나 프롬프트 엔지니어링을 왜 쓰는지 자연스럽게 연결된다.
- 운영체제는 평소에 당연하게 쓰고 있었는데, 유저 모드 / 커널 모드 / 시스템 콜의 흐름을 보니 자바에서 파일 하나 읽는 것도 꽤 많은 단계를 거친다는 걸 알게 됐다.
- 메모리 분할 4가지 방식(고정, 가변, 페이징, 세그멘테이션)을 비교하면서 현대 OS가 두 가지를 혼합하는 이유가 이해됐다. 각각의 단점을 서로의 장점으로 보완하는 구조.
- 프로세스 vs 스레드는 Spring(멀티스레드)과 Node.js(싱글스레드)의 차이로 직결되는 개념이라 특히 중요하게 느껴졌다. "스레드가 많다고 성능이 좋아지는 게 아니다"는 말이 인상적이었다.
- PCB와 컨텍스트 스위칭은 이전에 멀티스레드 부분에서 80%만 이해했던 부분인데, 오늘 운영체제 관점에서 다시 보니 훨씬 명확해졌다.
