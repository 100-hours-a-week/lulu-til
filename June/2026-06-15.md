#### | 네트워크 (Network)

**데이터를 송수신하는 기기들 간의 연결을 통해 정보와 자원을 공유할 수 있게 하는 시스템**

##### 네트워크를 사용하는 이유

- **정보 공유** : 문서, 이미지, 영상, DB 정보
- **자원 공유** : 프린터, 파일 서버, 인터넷 회선
- **원격 접속** : 멀리 떨어진 서버나 서비스에 접속 가능
- **협업** : 여러 사람이 같은 시스템이나 데이터를 기반으로 함께 작업 가능
- **서비스 제공** : 웹 서버, API 서버, 데이터베이스 서버 등이 사용자에게 서비스 제공 가능

##### 네트워크 요소 - 우편 배송 시스템 비유

| 네트워크 요소 | 우편 비유 | 역할 |
|---|---|---|
| 데이터 | 편지 내용 | 전달하고 싶은 실제 내용 |
| 패킷 | 편지 또는 소포 | 데이터를 잘게 나누어 보내는 단위 |
| IP 주소 | 도로명 주소 | 목적지 네트워크나 장치를 찾기 위한 주소 |
| MAC 주소 | 집 호수 또는 현관 번호 | 같은 네트워크 안에서 특정 장치를 찾기 위한 주소 |
| 스위치 | 아파트 관리실 또는 지역 분류소 | 같은 네트워크 안에서 어느 장치로 보낼지 결정 |
| 라우터 | 물류센터 | 다른 네트워크로 가는 경로를 결정 |
| 인터넷 | 전국·전세계 배송망 | 수많은 네트워크가 연결된 거대한 네트워크 |

##### 우리 컴퓨터에서 외부 서버로의 흐름

```
내 컴퓨터
  → 공유기 (스위치 + 라우터)
  → 통신사 네트워크
  → 여러 개의 지역 라우터
  → 목적지 라우터
  → 목적지 서버
  → 다시 내 컴퓨터로 응답
```

중간에 **프로토콜(규약)** 이 통신 방식을 정의한다. HTTP는 TCP 위에서 동작한다.

##### 네트워크 사용에 필요한 구성 요소

| 구성 요소 | 설명 |
|---|---|
| 장치 (Node) | PC, 스마트폰, 서버, 프린터 등 |
| 네트워크 인터페이스 | 유선 LAN 카드, Wi-Fi 모듈 등 |
| 연결 매체 | 랜 케이블, Wi-Fi, 광케이블 등 |
| 주소 | IP 주소, MAC 주소 등 |
| 네트워크 장비 | 스위치, 라우터, 공유기 등 |
| 통신 규칙 | TCP/IP, HTTP, DNS 같은 프로토콜 |

##### 인터넷과 네트워크의 차이

- **네트워크** : 장치들이 서로 연결되어 데이터를 주고받는 구조. 와이파이, 회사 내부망, 학교 전산망
- **인터넷** : 전 세계의 네트워크들이 서로 연결된 거대한 네트워크. 구글 접속, 유튜브 시청, 게임

##### 백엔드 개발자 관점 - 네트워크 개념이 필요한 이유

| 상황 | 네트워크 개념이 필요한 이유 |
|---|---|
| API 서버가 응답하지 않음 | 서버 문제인지, 포트 문제인지, 방화벽 문제인지 구분해야 함 |
| DB 연결 실패 | DB 주소, 포트, 네트워크 접근 권한을 확인해야 함 |
| CORS 문제 | 브라우저와 서버 간 요청 흐름을 이해해야 함 |
| 로그인은 되는데 특정 API만 실패 | 라우팅, 인증 서버, 게이트웨이, 프록시 문제일 수 있음 |
| 배포 후 외부 접속 불가 | 보안 그룹, 포트, NAT, 로드밸런서 설정을 확인해야 함 |

##### 인프라 엔지니어 관점

| 상황 | 네트워크 개념이 필요한 이유 |
|---|---|
| 서버를 외부에 공개 | 공인 IP, 포트, 라우팅, 방화벽 설정 필요 |
| 사내망과 외부망 분리 | 보안과 접근 제어를 위해 네트워크 구조 설계 필요 |
| 장애 대응 | 어느 구간에서 패킷이 막히는지 추적해야 함 |
| 클라우드 구성 | VPC, Subnet, NAT Gateway, Load Balancer 이해 필요 |
| 보안 설계 | 내부망, 외부망, VPN, WAF, 보안 그룹 등을 설계해야 함 |

#### | 짧은 틈새 강의 — 실무에서의 네트워크 디버깅

##### 네트워크 문제 해결 순서

1. 애플리케이션이 살아 있는지?
2. 서버 OS 단의 문제가 없나?
3. 포트 레벨 — 애플리케이션까지 도달하는 트래픽이 열려 있는 포트들을 타고 들어오나?
4. 방화벽 설정을 잘못했나? — 보안적인 측면에서 모든 요청을 다 열지는 않고, 특정 요청만 허용
5. DNS가 잘못됐나? — 내 로컬에서는 잘 되는데, 배포하면 안 된다
6. 라우팅 경로? — GCP, AWS 안에서 라우팅?
7. 로드밸런서, 프록시 문제가 있나?

> 💡 **물리 → IP → 경로 → 포트 → 앱** 순으로 좁혀 들어가면 문제를 빠르게 찾을 수 있다.

##### 개발자가 알면 좋은 명령어

```bash
ping 8.8.8.8             # 목적지까지 네트워크 연결이 되는지 확인
curl https://localhost   # HTTP 응답을 주는지 확인
nslookup google.com      # 도메인이 어떤 IP로 해석되는지 확인
netstat -an | grep 8080  # 특정 포트가 열려 있는지 확인
traceroute google.com    # 구글까지 어떻게 트래픽이 도달하는지, 몇 개의 라우터를 거치는지 확인
```

#### | 노드 (Node)

**네트워크에서 데이터를 생성, 수신 또는 전송하는 장치나 지점**

네트워크는 단순히 케이블이나 와이파이만 있다해서 완성되는 게 아니다. 그 안에서 데이터를 주고 받는 장치들이 있어야 한다. 단순 케이블은 노드가 아니다.

##### 카카오톡 메시지 전송 흐름 (노드 예시)

| 구간 | 노드 |
|---|---|
| 메시지를 보내는 장치 | 내 스마트폰 |
| 집 안에서 데이터를 넘겨주는 장치 | Wi-Fi 공유기 |
| 중간 네트워크 장비 | 통신사 라우터 등 |
| 메시지를 처리하는 장치 | 카카오 서버 |
| 메시지를 받는 장치 | 친구 스마트폰 |

##### 노드 vs 호스트

| 구분 | 설명 | 예시 |
|---|---|---|
| **노드** | 네트워크에서 데이터를 송수신하거나 전달하는 모든 장치 또는 지점 | PC, 서버, 라우터, 스위치, 프린터 |
| **호스트** | 네트워크에 연결되어 서비스를 사용하거나 제공하는 최종 장치 | PC, 스마트폰, 웹 서버, DB 서버 |

> 💡 **노드가 더 넓은 개념**이다. 호스트는 통신의 출발지/목적지가 되는 장치, 노드는 출발지/목적지 + 중간에서 데이터를 전달하는 장치까지 모두 포함한다.

##### 노드의 특징 — 주소를 가진다

| 주소 종류 | 설명 | 비유 |
|---|---|---|
| **MAC 주소** | 네트워크 인터페이스 카드(NIC)에 부여된 물리적 식별자 | 기기의 고유 일련번호 |
| **IP 주소** | 네트워크 상에서 위치를 나타내는 논리적 주소 | 도로명 주소 |

**왜 두 가지 주소를 같이 쓰는가?**
- MAC 주소는 같은 네트워크 안에서만 사용 가능
- 다른 네트워크와 통신할 때는 IP 주소 사용
- IP = 목적지 네트워크를 찾는 용도, MAC = 네트워크 안에서 실제 장치를 찾는 용도

#### | LAN (Local Area Network)

**집, 사무실, 학교처럼 제한된 지역 내에서 컴퓨터나 기기들이 연결된 네트워크**

| 공간 | LAN 예시 |
|---|---|
| 집 | 공유기에 연결된 노트북, 스마트폰, TV, 프린터 |
| 사무실 | 직원 PC, 사내 서버, 복합기, NAS |
| 학교 | 강의실 PC, 교수실 PC, 프린터, 내부 서버 |
| PC방 | 여러 대의 PC가 같은 내부망에 연결된 구조 |

> ⚠️ **LAN ≠ 인터넷.** 인터넷에 연결이 없더라도 같은 로컬 네트워크 안에서 서로 통신하면 LAN이다. 군대, 특정 기업 내부망처럼 외부 인터넷과 분리된 환경도 LAN이다.

##### LAN을 사용하는 이유

| 사용 이유 | 설명 |
|---|---|
| **빠른 내부 통신** | 가까운 거리의 장치끼리 데이터를 빠르게 주고받을 수 있음 |
| **자원 공유** | 프린터, 파일 서버, NAS, 인터넷 회선 등을 함께 사용할 수 있음 |
| **관리 편의성** | 같은 공간의 장치들을 하나의 네트워크로 묶어 관리할 수 있음 |
| **비용 절감** | 모든 장치가 각각 별도 회선을 가질 필요 없이 하나의 네트워크를 공유 가능 |

> 💡 **LAN 네트워크를 사용할 때 인터넷을 거치지 않는다.** 이게 굉장히 큰 장점이다. 내부 통신이 빠르고, 외부 영향을 받지 않는다.

##### LAN 구성 요소

| 구성 요소 | 역할 |
|---|---|
| PC, 서버, 프린터 등 | 네트워크에 참여하는 장치 |
| NIC | 장치가 네트워크에 연결되기 위한 네트워크 인터페이스 |
| 스위치 | 같은 LAN 안의 장치들을 연결하고 데이터를 적절한 장치로 전달 |
| 라우터/공유기 | LAN과 외부 인터넷을 연결 |
| 이더넷 케이블 또는 Wi-Fi | 장치들이 데이터를 주고받는 물리적/무선 연결 방식 |
| IP 주소 | LAN 안에서 장치를 구분하기 위한 논리적 주소 |
| MAC 주소 | 같은 네트워크 구간에서 장치를 식별하기 위한 물리적 주소 |

##### LAN 통신에서 알아야 하는 포인트

**1. 같은 네트워크 대역대를 가진다**

```
집 공유기에 연결된 장치들의 IP 예시
  노트북 : 192.168.0.10
  스마트폰 : 192.168.0.11
  공유기 : 192.168.0.1
```

**2. LAN 내부 통신 vs 인터넷 통신**

| 구분 | 설명 | 라우터 필요 여부 |
|---|---|---|
| LAN 내부 통신 | 같은 네트워크 안에서 노드들끼리 통신 | ❌ 불필요 |
| 인터넷 통신 | LAN을 벗어나서 외부에 있는 노드랑 통신 | ✅ 필요 |

```
내 컴퓨터 → LAN 내 프린터 사용 = LAN 통신
내 컴퓨터 → 구글 접속 = 인터넷 통신
```

#### | WLAN (Wireless Local Area Network)

**LAN 케이블 대신, 무선 신호(와이파이)로 연결한 네트워크. 무선 랜.**

##### WLAN 구성 요소

| 구성 요소 | 역할 |
|---|---|
| 무선 단말 | 노트북, 스마트폰, 태블릿, IoT 장치 |
| 무선 NIC | 장치가 Wi-Fi에 연결되기 위한 네트워크 인터페이스 |
| AP (Access Point) | 무선 장치들을 네트워크에 연결해주는 장비 |
| SSID | 사용자가 보는 Wi-Fi 이름 |
| 인증/암호화 | 비밀번호, WPA2/WPA3 같은 무선 보안 |

##### 공유기는 무슨 장비인가?

> 💡 **공유기 = 라우터 + 스위치 + 무선 AP + NAT + 게이트웨이를 합친 장비**

서버 측 개념에서는 이것들을 모두 분리해서 사용한다. 이유는 장애 전파 지점에 한계를 짓고, 속도 처리와 관심사를 분리하기 위해서다.

##### WLAN 주의사항

| 주의점 | 설명 |
|---|---|
| **신호 간섭** | 벽, 거리, 다른 무선 장비 때문에 품질이 떨어질 수 있음 |
| **보안 문제** | 무선 신호는 공기 중으로 퍼지므로 인증과 암호화가 중요함 |
| **속도 변동** | 유선보다 환경 영향을 많이 받음 |
| **AP 과부하** | 하나의 AP에 너무 많은 사용자가 연결되면 느려질 수 있음 |
| **음영 지역** | AP 신호가 닿지 않는 구역에서는 접속 품질이 낮아짐 |

#### | VLAN (Virtual Local Area Network)

**하나의 물리적인 네트워크를 여러 개의 독립된 네트워크처럼 나누는 기술**

##### LAN / WLAN / VLAN 비교

| 구분 | LAN | WLAN | VLAN |
|---|---|---|---|
| 한 줄 정리 | 가까운 공간의 내부 네트워크 | 무선으로 연결하는 LAN | 물리 네트워크를 논리적으로 나누는 기술 |
| 핵심 질문 | 어디까지가 같은 내부망인가? | 랜선 없이 연결할 수 있는가? | 같은 장비를 쓰면서 네트워크를 나눌 수 있는가? |
| 연결 방식 | 주로 유선 이더넷 | Wi-Fi 같은 무선 신호 | 물리 연결과 별개로 논리 분리 |
| 주요 장비 | 스위치, 라우터, 케이블 | AP, 무선 공유기, 무선 NIC | VLAN 지원 스위치, L3 스위치, 라우터 |
| 목적 | 내부 장치 간 통신과 자원 공유 | 이동성과 무선 접속 제공 | 보안, 트래픽 분리, 관리 편의성 |
| 예시 | 사무실 PC와 프린터 연결 | 강의실 Wi-Fi, 카페 Wi-Fi | 학생망/교직원망/게스트망 분리 |

##### VLAN을 사용하는 이유

| 사용 이유 | 설명 |
|---|---|
| **보안 강화** | 부서별, 사용자 유형별로 네트워크 접근 범위를 제한 |
| **트래픽 감소** | 브로드캐스트 도메인을 나누어 불필요한 트래픽 확산 방지 |
| **관리 편의성** | 물리적 위치와 관계없이 논리적으로 네트워크 그룹 구성 |
| **비용 절감** | 네트워크를 나누기 위해 장비를 물리적으로 모두 분리하지 않아도 됨 |

##### 브로드캐스트란?

같은 네트워크 안에서 모든 장치에게 동시에 메시지를 보내는 방식.

새로운 장치가 네트워크에 연결되면, 처음에는 아무것도 모르는 상태다. IP 주소가 뭔지, 프린터가 어디 있는지 모르기 때문에 브로드캐스트로 전체 네트워크에 물어본다.

```
새 컴퓨터: "저 새로 등록되었어요! 제 MAC 주소는 이거입니다. 혹시 프린터 서버 계신가요?"
A: "어, 나 아님"
B: "어, 나 아님"
C: "내가 프린터임 ㅋㅋ"
```

> ⚠️ **브로드캐스트 스톰**: 브로드캐스트가 너무 많아서 통신이 제대로 안 되는 현상. 그래서 VLAN으로 브로드캐스트 도메인(브로드캐스트 전파 영역)을 제한한다.

| 개념 | 설명 |
|---|---|
| 브로드캐스트 | 같은 네트워크 안의 모든 장치에게 보내는 통신 |
| 브로드캐스트 도메인 | 브로드캐스트가 전달되는 범위 |
| VLAN의 역할 | 브로드캐스트 도메인을 논리적으로 분리 |

##### VLAN 주의 사항

- VLAN만으로 완전한 보안은 아님 → 방화벽, ACL, 라우팅 정책이 함께 필요
- VLAN 간 통신은 별도 설정 필요 → 서로 다른 VLAN은 라우터나 L3 스위치 필요
- 설정 실수 위험 → 포트 할당, 트렁크 허용 VLAN 설정을 잘못하면 장애 발생
- 게스트망 분리는 필수

---

#### | 네트워크 범위에 따른 분류

```
PAN < LAN < MAN < WAN
```

| 구분 | 범위 | 대표 예시 |
|---|---|---|
| **PAN** (Personal Area Network) | 개인 주변, 매우 짧은 거리 | 스마트폰-이어폰, 스마트워치, 무선 마우스 |
| **LAN** (Local Area Network) | 집, 사무실, 학교, 건물 | 가정 공유기, 사무실 내부망 |
| **MAN** (Metropolitan Area Network) | 캠퍼스, 도시, 여러 건물 | 대학 캠퍼스망, 도시 공공망 |
| **WAN** (Wide Area Network) | 도시, 국가, 대륙 | 인터넷, 본사-해외지사 연결 |

- **PAN** : 블루투스, 지그비, NFC 같은 무선 기술로 많이 통신
- **MAN** : 대학교 내 네트워크 통합, 도시 내 여러 지점을 가진 회사
- **WAN** : 인터넷 = WAN

---

#### | 네트워크 장비

데이터가 노드에서 출발해 목적지까지 도달하는 흐름:

```
[내 컴퓨터]
  └ NIC : 네트워크에 연결되는 입구
       └ MAC 주소 : 같은 네트워크 안에서 나를 구분하는 주소
            └ 스위치 : 같은 LAN 안에서 목적지 장치로 전달
                 └ 라우터 : 다른 네트워크로 가는 경로 결정
                      └ 게이트웨이 : 내부망과 외부망을 연결하는 관문
                           └ 인터넷
```

| 장비 | 주로 보는 정보 | 핵심 역할 |
|---|---|---|
| 리피터 | 전기/무선 신호 | 신호를 증폭하거나 재생 |
| 허브 | 신호 | 모든 포트로 전달 |
| 브릿지 | MAC 주소 | LAN 세그먼트 연결 |
| 스위치 | MAC 주소 | 같은 네트워크 안에서 목적지로 전달 |
| 라우터 | IP 주소 | 다른 네트워크로 경로 결정 |
| 게이트웨이 | IP 이상, 경우에 따라 응용 계층 정보 | 서로 다른 네트워크/프로토콜 연결 |

---

#### | NIC (Network Interface Card)

**컴퓨터, 서버 또는 기타 네트워크 장치가 네트워크에 연결되도록 해주는 하드웨어 구성요소**

호스트(PC, Mobile)와 네트워크 사이에서 데이터를 주고받는 역할을 한다.

##### NIC 종류

| 종류 | 특징 | 사용 장치 |
|---|---|---|
| **유선 NIC** | 이더넷 케이블, 랜선 연결. 안정적이고 빠름 | 컴퓨터, 서버 |
| **무선 NIC** | 와이파이 연결. 이동성 있는 장비에 사용 | 노트북, 스마트폰 |

##### MAC Address

- NIC에 부여된 고유한 물리적 식별자 (예: `00:1A:2B:3C:4D:5E`)
- NIC 제조될 때 하드웨어에 내장됨
- 와이파이가 바뀌어도 NIC에 있는 MAC Address는 그대로 유지

| 인터페이스 | 예시 |
|---|---|
| 유선 LAN | 이더넷 MAC 주소 |
| Wi-Fi | 무선 MAC 주소 |
| Bluetooth | Bluetooth MAC 주소 |
| 가상 NIC | Docker, VM, VPN에서 생성된 MAC 주소 |

> 💡 호스트에 NIC가 하나면 MAC 주소 1개, 여러 개면 여러 개의 MAC 주소를 갖는다.

---

#### | MAC 주소 vs IP 주소

| 구분 | MAC 주소 | IP 주소 |
|---|---|---|
| 의미 | 네트워크 인터페이스의 물리적 식별자 | 네트워크 상의 논리적 위치 주소 |
| 사용 계층 | 데이터 링크 계층, L2 | 네트워크 계층, L3 |
| 주 사용 범위 | 같은 LAN 내부 전달 | 다른 네트워크까지의 경로 결정 |
| 변경 가능성 | 기본적으로 하드웨어에 부여됨, 단 스푸핑 가능 | 네트워크 환경에 따라 변경 가능 |
| 예시 | `00:1A:2B:3C:4D:5E` | `192.168.0.10` |
| 주로 보는 장비 | 스위치, 브릿지, AP | 라우터, L3 스위치 |

##### ARP (Address Resolution Protocol)

LAN 안에서 실제로 데이터를 보내려면 상대방의 MAC 주소를 알아야 한다. IP 주소에 해당하는 MAC 주소를 알아내는 프로토콜이 ARP다.

```
내 노트북: ARP Request → "192.168.0.20의 MAC 주소가 누구인가요?" (LAN 전체에 브로드캐스트)
프린터: ARP Reply → "제 MAC 주소는 BB:BB:BB:BB:BB:BB 입니다"
이후 MAC 주소를 사용해 데이터 전송
```

> ⚠️ **MAC 주소는 같은 네트워크 구간에서 전달할 때만 사용한다.** 라우터를 넘어가는 순간 MAC 주소는 바뀐다.

##### MAC Spoofing

노드의 원래 MAC 주소를 다른 MAC 주소처럼 보이게 하는 기술

| 사용 사례 | 설명 |
|---|---|
| 테스트/디버깅 | 특정 MAC 주소 조건을 테스트할 때 |
| 장치 교체 | 기존 장비의 MAC 주소를 새 장비에서 임시로 사용 |
| 접근 우회 | MAC 기반 접근 제어를 우회하려는 악용 가능성 |
| 추적 회피 | 원래 장치 식별을 숨기려는 목적 |

---

#### | 리피터 (Repeater)

**네트워크에서 신호를 증폭하거나 재생하여 더 먼 거리로 전달하는 역할**

- 동작 계층 : OSI 1계층 (물리 계층)
- 다루는 대상 : 비트, 전기 신호, 무선 신호, 광 신호
- 모르는 정보 : MAC 주소, IP 주소, TCP 포트

##### 리피터의 한계

- 트래픽을 구분하지 못함 (불필요한 신호도 전달)
- 보안 제어 불가
- 네트워크 충돌 문제 해결 불가
- 현대 LAN에서는 사용 빈도가 낮음

---

#### | 게이트웨이 (Gateway)

**서로 다른 네트워크를 연결하여 통신을 가능하게 하는 장치. 입구이자 출입구.**

```
내부 네트워크 → 게이트웨이 → 외부 네트워크
```

공유기가 가정에서 게이트웨이 역할을 한다.

##### 게이트웨이의 주요 기능

| 기능 | 설명 |
|---|---|
| **IP 주소 변환** | 내부 사설 IP와 외부 공인 IP 사이의 변환 (NAT) |
| **프로토콜 변환** | 서로 다른 통신 방식 사이의 변환 |
| **데이터 필터링** | 허용할 트래픽과 차단할 트래픽 판단 |
| **접근 제어** | 내부망과 외부망 사이의 보안 정책 적용 |
| **중계** | 내부 요청을 외부로 전달하고, 외부 응답을 내부로 다시 전달 |

##### 게이트웨이의 종류

- **기본 게이트웨이** : 내부에서 외부로 나가는 모든 요청이 통과하는 출입구
- **API 게이트웨이** : 여러 API 요청을 하나의 진입점에서 처리하는 서버 측 게이트웨이

##### 라우터 vs 게이트웨이

| 구분 | 게이트웨이 | 라우터 |
|---|---|---|
| 핵심 의미 | 서로 다른 네트워크의 출입구 | IP 기반 경로 선택 장비 |
| 주요 기능 | 네트워크 연결, 변환, 필터링, 중계 | 패킷의 다음 경로 결정 |
| 동작 범위 | L3 이상, 경우에 따라 L4~L7 기능 포함 | 주로 L3, 네트워크 계층 |
| 관계 | 라우터가 게이트웨이 역할을 할 수 있음 | 게이트웨이 기능을 포함할 수 있음 |

---

#### | 라우터 (Router)

**서로 다른 네트워크를 연결하고, IP 주소를 보고 패킷이 갈 다음 경로를 결정하는 장비**

##### 핵심 개념

**1. 라우터는 다른 네트워크로 보내는 장비다**
- 스위치 : 같은 LAN 네트워크 안에서 장치들을 연결
- 스위치는 내부 정보만 알고 있어서 외부로 연결하기에 부족
- 외부 연결할 때 사용하는 게 라우터

**2. 라우터는 IP를 보고 판단한다**
- L3 (IP 레이어)에서 동작
- IP 주소를 보고 패킷을 어디로 라우팅할지 결정
- 스위치 : MAC 주소를 보고 같은 LAN 안에서 전달
- 라우터 : IP 주소를 보고 다른 네트워크로 전달

> 💡 **인터넷은 하나의 네트워크가 아니다.** 엄청나게 쪼개져 있는 다양하고 많은 네트워크의 집합이다. 그래서 라우터가 필요하다.

> ⚠️ **라우터는 최종 목적지까지 한 번에 보내는 장비가 아니다.** "내가 아는 범위까지 전달하고, 그 다음 라우터에게 넘긴다."

##### 라우터 기본 구조

| 포트 | 연결 대상 | 의미 |
|---|---|---|
| **WAN 포트** | 통신사 모뎀, 외부 인터넷 | 외부 네트워크 방향 |
| **LAN 포트** | PC, 스위치, 내부 장치 | 내부 네트워크 방향 |
| **Wi-Fi** | 스마트폰, 노트북 | 내부 무선 네트워크 방향 |

##### 라우터 동작 방식

```
1. 패킷을 받는다 (패킷 안에는 목적지 IP, 출발지 IP가 적혀있다)
2. 패킷을 열어서 목적지 IP 확인
3. 라우팅 테이블 조회 → 어느 네트워크로 보내야 하지?
4. 가장 구체적인 경로로 다음 라우터에 전달
5. 못 찾으면 기본 경로(0.0.0.0/0)로 ISP 방향으로 전달
```

##### 라우팅 테이블

라우터가 참고하는 길 안내표

| 항목 | 의미 |
|---|---|
| 목적지 네트워크 | 어느 IP 대역으로 가는 패킷인가 |
| 다음 홉 | 다음으로 보낼 라우터 또는 인터페이스 |
| 인터페이스 | 어느 포트/네트워크로 내보낼 것인가 |
| 메트릭 | 경로 선택 우선순위 |

##### 라우팅 테이블이 만들어지는 방법

1. **직접 연결 통로** : 라우터 인터페이스에 IP를 설정하면 자동 등록
2. **정적 라우팅** : 관리자가 직접 경로 입력 (예: "특정 사내망은 VPN으로 보내라")
3. **동적 라우팅** : 라우터끼리 경로 정보를 교환 (OSPF, BGP 등)

| 프로토콜 | 주 사용 위치 | 특징 |
|---|---|---|
| **OSPF** | 회사 내부망, 데이터센터 내부 | 내부 라우터끼리 빠르게 경로 공유 |
| **RIP** | 교육용, 소규모/구형 환경 | 홉 수 기반, 단순하지만 한계 큼 |
| **BGP** | 인터넷 사업자, 클라우드, 대규모 네트워크 간 | 조직과 조직 사이의 경로 교환 |
| **IS-IS** | 대형 ISP, 통신망 | 대규모 내부망에서 사용 |

---

#### | 허브 (Hub)

**네트워크에서 데이터를 여러 장치로 전달하는 장치. 들어온 데이터를 모든 장치에 뿌린다.**

- 동작 계층 : 물리 계층 (L1)
- 보는 정보 : 전기 신호, 비트 신호
- 보지 않는 정보 : MAC 주소, IP 주소, 포트 번호, HTTP 내용

| 장점 | 단점 |
|---|---|
| 싸다 | 불필요한 트래픽이 많아짐 |
| 구성이 단순하다 | 보안에 불리 (모두가 같은 데이터를 수신) |
| | 충돌이 발생할 가능성이 높다 |

---

#### | 스위치 (Switch)

**같은 네트워크 안에서 MAC 주소를 보고 데이터를 필요한 장치에게만 전달하는 장비**

허브의 문제(모든 포트에 전달)를 해결한 장비다.

##### 스위치의 계층

| 구분 | 내용 |
|---|---|
| 동작 계층 | 데이터 링크 계층, Layer 2 |
| 주로 보는 정보 | MAC 주소 |
| 다루는 데이터 단위 | 프레임 |
| 주 역할 | 같은 LAN 안에서 목적지 장치로 프레임 전달 |
| 기본적으로 하지 않는 일 | IP 기반 라우팅, 외부 네트워크 경로 결정 |

##### 스위치의 동작 방식

**1. 목적지 MAC 주소를 알 때**
- MAC Address Table 확인 → 해당 포트로만 전달 (유니캐스트)

**2. 목적지 MAC 주소를 모를 때 (Flooding)**
- MAC Address Table에 없음 → 모든 포트로 전달 → Unknown Unicast Flooding
- 목적지 PC가 응답하면 → 이때 MAC Address Table을 업데이트한다

> 💡 **브로드캐스트 vs Flooding 차이**
> - 브로드캐스트 : 목적지가 "모든 장치"를 의미
> - Flooding : "내가 모르는 특정 MAC 주소를 찾기 위해" 모든 포트에 물어보는 것

##### 허브 vs 스위치

| 구분 | 허브 | 스위치 |
|---|---|---|
| 동작 계층 | L1 물리 계층 | L2 데이터 링크 계층 |
| 보는 정보 | 신호 | MAC 주소 |
| 전달 방식 | 모든 포트로 전달 | 목적지 포트로만 전달 |
| 목적지 판단 | 못함 | 가능 |
| MAC 주소 학습 | 안 함 | 함 |
| 트래픽 효율 | 낮음 | 높음 |
| 보안성 | 낮음 | 상대적으로 높음 |
| 사용성 | 거의 사용하지 않음 | LAN의 기본 장비 |

##### 스위치 vs 라우터

| 구분 | 스위치 | 라우터 |
|---|---|---|
| 주 역할 | 같은 네트워크 안에서 장치 연결 | 서로 다른 네트워크 연결 |
| 주로 보는 주소 | MAC 주소 | IP 주소 |
| 동작 계층 | L2 데이터 링크 계층 | L3 네트워크 계층 |
| 사용 범위 | LAN 내부 | LAN ↔ WAN, 네트워크 간 연결 |
| 비유 | 건물 내부 우편 분류함 | 도시 간 물류센터 |

---

#### | 브릿지 (Bridge)

**네트워크에서 두 개 이상의 LAN을 연결하는 장치**

- L2 장비로 MAC 주소를 읽어보고, 같은 LAN이면 내부에서 동작, 반대편 LAN이면 옆 LAN으로 이동시킨다.

##### 브릿지를 사용하는 이유

1. LAN을 연결 : `LAN A ↔ Bridge ↔ LAN B`
2. 불필요한 트래픽을 줄일 때
3. 충돌 범위도 적어짐

> 💡 LAN이 점점 규모가 커지면 VLAN으로 쪼개는 것을 먼저 생각하지만, 스위치 하나에서 VLAN이 동작하면 스위치가 부하를 받는다. 그래서 물리적으로 LAN을 여러 개로 나누고, 브릿지로 연결하는 방식을 쓰기도 한다.

##### 브릿지 vs 스위치

| 구분 | 브릿지 | 스위치 |
|---|---|---|
| 주 역할 | LAN 세그먼트 연결 | 여러 장치를 LAN 안에서 연결 |
| 동작 계층 | L2 | L2 |
| 기준 주소 | MAC 주소 | MAC 주소 |
| 포트 수 | 상대적으로 적음 | 많음 |
| 처리 방식 | 소프트웨어 기반이 많았음 | 하드웨어 기반 고속 처리 |
| 현대 사용 | 독립 장비로는 줄어듦 | 현대 LAN의 기본 장비 |

### 🗨️ 오늘의 회고

- 네트워크 → 노드 → LAN/WLAN/VLAN → 네트워크 장비 순으로 배웠는데, 개념들이 계층적으로 연결되어 있다는 걸 느꼈다. 하나를 이해해야 다음 개념이 자연스럽게 붙는다.
- 우편 배송 비유가 네트워크 요소를 이해하는 데 정말 직관적이었다. IP = 도로명 주소, MAC = 집 호수, 라우터 = 물류센터. 이 비유로 머릿속에 지도가 그려졌다.
- **허브 → 스위치 → 라우터 → 게이트웨이** 순으로 장비가 발전하고 책임이 나뉜다는 흐름이 인상적이었다. 허브의 비효율(모두에게 전달)을 스위치가 해결하고, 스위치의 한계(외부 연결 불가)를 라우터가 해결한다.
- MAC 주소와 IP 주소를 함께 쓰는 이유, ARP가 왜 필요한지를 오늘 확실히 이해했다. "IP = 어디로 갈지, MAC = 실제로 누구한테 줄지"로 기억해 두자.
- VLAN의 핵심은 브로드캐스트 도메인 분리. 네트워크가 커질수록 브로드캐스트가 문제가 되고, 그걸 논리적으로 잘라내는 게 VLAN이다.
- 실무 디버깅 순서(물리 → IP → 경로 → 포트 → 앱)는 지금 당장 외워둬야 할 것 같다. 배포 후 네트워크 문제가 생겼을 때 이 순서로 접근하면 훨씬 빠르게 원인을 좁힐 수 있다.
