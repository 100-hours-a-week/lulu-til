#### | 데이터센터 (Data Center)

**서버 컴퓨터들을 안전하게 운영하기 위해 모아둔 전용 시설**

##### 과거의 문제 — 사무실에 서버를 두던 시절

회사 사무실 한쪽에 서버를 그냥 세워두던 시절이 있었다. 이 방식은 여러 문제를 일으켰다.

- 청소하다가 전원선을 뽑아버림
- 먼지가 쌓여서 냉각 팬이 막힘
- 커피 같은 음료를 쏟음

서버용 컴퓨터는 일반 PC와 요구사항이 완전히 다르기 때문에, 이런 사고는 단순한 불편이 아니라 서비스 장애로 직결된다.

| 일반 컴퓨터 | 서버용 컴퓨터 |
|---|---|
| 꺼져도 그때그때 대응 | **365일 24시간 무중단**이 기본 |
| 오류가 나면 재부팅하면 됨 | **데이터 오류가 없어야 함**, 오류 발생 시 즉시 복구 |
| 속도가 우선 | **속도보다 안전이 우선** |
| 발열·먼지 관리 느슨 | **발열 관리**, **먼지 차단**이 필수 |

##### 데이터센터의 등장

서버용 컴퓨터를 전용으로 관리하기 위해 데이터센터라는 시설이 등장했다. 데이터센터는 아래 기능을 표준으로 갖춘다.

1. 습도 조절 시스템
2. 온도 조절 시스템
3. 공기 청정 시스템
4. 비상 전력 공급 시스템 (UPS, 발전기)
5. 상시 관리 직원 상주
6. 건물 전력 승압 지원

##### 물리적 거리와 지연 시간의 한계

우리가 통신할 때 쓰는 랜선 내부에는 광섬유가 들어있고, 데이터는 빛의 형태로 전달된다.

> 💡 빛의 속도는 초당 약 30만 km. 이 이상의 전송 속도는 물리적으로 존재할 수 없다.

거리가 멀어질수록 지연 시간(Latency)이 늘어난다. 그래서 지연 시간을 줄이려면 서버를 전 세계 여러 곳에 분산 배치해야 한다.

##### 그런데 또 다른 고민이 시작된다

- 제대로 된 서버 한 대를 구축하려면 약 **8,000만원** 수준의 비용이 든다
- 데이터센터를 직접 운영하는 건 관리 측면에서는 좋지만, **확장이 어렵고** 글로벌 서비스를 위해 여러 리전에 두기엔 **너무 비싸다**

##### 그래서 등장한 것 — AWS (클라우드)

> "서버가 트래픽에 비해 너무 많이 남는데, 이걸 가상화해서 빌려주면 어떨까?"

이 아이디어가 클라우드 컴퓨팅, 그중에서도 AWS의 출발점이다.

**왜 AWS를 많이 쓸까?**
- 클라우드 서비스에는 **안정성**이 무엇보다 중요한데, AWS의 안정성이 매우 높다
- 제공하는 **기능적 지원 범위**가 압도적으로 넓다
- 다른 클라우드 사업자들보다 **약 7년 먼저 시작**했기 때문에 축적된 노하우와 생태계가 크다
- 최근에는 한 회사가 하나의 클라우드만 쓰지 않고 **멀티 클라우드**(예: AWS + GCP)를 함께 사용하는 추세

#### | VPC (Virtual Private Cloud)

**공용 클라우드 환경에서 사용자에게 논리적으로 분리된 독립 가상 네트워크를 제공하는 서비스**

IP 대역, 서브넷, 라우팅, 보안 규칙을 직접 조립해서 사용하는 네트워크 설계 단위다.

##### VPC를 왜 사용하는가

서버와 데이터베이스를 무작정 인터넷에 그대로 연결하지 않고, **서비스별로 통신 경로와 접근 범위를 통제**하기 위해 사용한다.

> ⚠️ VPC가 없으면 웹서버, DB, 캐시 서버가 전부 외부 인터넷에 그대로 노출된다.

VPC를 사용하면:
- 서비스를 네트워크 단위로 나눌 수 있다
- 트래픽이 지나가는 경로를 제어할 수 있다

##### VPC 구성 요소

**1. VPC**

AWS 리소스를 배치하는 전체 가상 네트워크 공간. 회사 전체의 사내망에 비유할 수 있다.

- VPC를 만들 때 가장 먼저 지정하는 값은 **IPv4 CIDR 블록** (예: `10.0.0.0/16` → 약 6.5만 개의 IP)
- 이 대역을 서브넷 단위로 분할해서 사용
- CIDR 블록은 보통 **`/16` ~ `/28`** 범위로 설정

**2. 서브넷 (Subnet)**

VPC의 IP 주소 범위를 더 작은 네트워크 단위로 나눈 공간.

| 구분 | 예시 |
|---|---|
| VPC 전체 | `10.0.0.0/16` |
| Public Subnet A | `10.0.1.0/24` |
| Public Subnet B | `10.0.2.0/24` |
| Private Subnet A | `10.0.11.0/24` |
| Private Subnet B | `10.0.12.0/24` |

용도에 맞게 서브넷을 분리하고, 필요하면 더 작게 쪼개서 쓸 수 있다.

**3. AZ (Availability Zone)**

AWS는 전 세계에 흩어진 데이터센터를 가상화해서 컴퓨팅 자원으로 대여하는 서비스다. 한국 리전(서울)의 AZ는 다음과 같이 구성되어 있다.

- 2a zone
- 2b zone
- 2c zone
- 2d zone

> 💡 서브넷을 나눌 때는 보통 **최소 두 개 이상의 AZ**에 분산되도록 구성한다. 고가용성이 필요한 운영 환경에서는 AWS도 이를 권장한다. 다만 AZ를 분리해서 구성하면 그만큼 **비용이 늘어난다**는 트레이드오프가 있다.

**Public Subnet vs Private Subnet**

해당 서브넷의 **라우팅 테이블에 인터넷 게이트웨이(IGW)로 가는 경로가 있는지 없는지**로 구분한다. 즉, 인터넷에 연결되어 있는지 여부가 기준이다.

```
Internet → Internet Gateway → [Public Subnet: Web/ALB] → [Private Subnet: Application] → Database
```

| 구분 | 라우팅 테이블 기준 | 일반 배치 대상 |
|---|---|---|
| **Public Subnet** | IGW로 가는 경로가 **있음** | ALB, Bastion Host, NAT Gateway |
| **Private Subnet** | IGW로 가는 경로가 **없음** | App Server, RDS, ElastiCache |

서브넷 내부 구성 요소가 인터넷에 직접 연결되어 있으면 보안적으로 좋지 않기 때문에, 꼭 필요한 자원만 Public에 두고 나머지는 Private에 둔다.

**4. 라우팅 테이블 (Route Table)**

이 목적지로 가는 트래픽을 어디로 보낼지 정의하는 네트워크 경로표.

```
Destination          Target
10.0.0.0/16     →    local
0.0.0.0/0       →    igw-xxxxxxxx
```

**5. 인터넷 게이트웨이 (Internet Gateway, IGW)**

VPC와 인터넷을 연결하는 출입구.

- VPC에 IGW를 연결한 뒤, **퍼블릭 서브넷의 라우팅 테이블**에 `0.0.0.0/0 → igw-id` 경로를 추가하면 사용 가능해진다
- "퍼블릭 서브넷을 만들었다" = "라우팅 테이블에 IGW로 가는 통로가 설정되어 있다"
- EC2가 퍼블릭 서브넷에 있으면 인터넷과 직접 통신 가능, 프라이빗 서브넷에 있으면 인터넷 통신 불가

**6. NAT Gateway**

프라이빗 서브넷은 기본적으로 인터넷 통신이 안 된다. 그런데 프라이빗 서브넷의 서버도 다음과 같은 작업이 필요하다.

- `apt-get install`, `ping` 같은 패키지 설치
- `git clone`
- OAuth 인증
- SSH를 통한 외부 접속

> NAT Gateway는 **프라이빗 서브넷의 서버가 외부 인터넷으로 나갈 수 있게 하되, 인터넷에서 먼저 들어오는 연결은 차단**하는 장치다.

```
Private Subnet App Server → NAT Gateway → Internet Gateway → Internet
                ↑                                                 ↓
                └──────────── 응답 전달 ──────────────────────────┘
```

사용 예시: 스프링 서버의 외부 API 호출, Maven/Gradle 의존성 다운로드, 보안 패치 적용 등.

```
프라이빗 서브넷 → NAT Gateway → 인터넷
```

##### VPC 구성 시 주의해야 할 것들

**① 퍼블릭 IP 할당은 조심해야 한다**
- 퍼블릭 IP는 **개당 과금**된다
- 인스턴스를 **Stop**해도 퍼블릭 IP는 과금된다
- 퍼블릭 IP는 할당받은 인스턴스가 **동작 중일 때는 과금되지 않는다**
- 인스턴스가 동작하지 않을 때(중지 상태)는 오히려 퍼블릭 IP에 과금이 발생한다

**② 내부 서비스(RDS, ElastiCache)에 퍼블릭 IP를 함부로 할당하지 말 것**
- "SSH 접속도 안 되고 DB GUI, Redis CLI 접근도 안 되니까" 하면서 RDS에 퍼블릭 IP를 부여하고 퍼블릭 서브넷에 올리는 경우가 있는데, 이는 보안상 위험한 지름길이다. 원칙적으로는 VPN/Bastion Host를 통해 접근해야 한다.

**③ SSH 포트를 `0.0.0.0/0`으로 열어두지 말 것**
- "pem 키는 내가 갖고 있으니까 그냥 열어도 된다"는 생각은 위험하다. 전 세계에서 22번 포트로 무차별 대입 공격이 들어온다.

**④ NAT Gateway는 비용에 주의해야 한다**
- NAT Gateway의 과금 구조는 **네트워크 통신량 기반**이라 생각보다 비싸다
- 예: 프라이빗 서브넷에서 컨테이너 이미지를 다운로드하거나, 깃헙 클론(저장소 변경 내역이 대량으로 들어오는 경우), S3 데이터를 NAT Gateway로 보내는 작업이 누적되면 비용이 급격히 늘어난다

##### VPC 정리

VPC는 결국 **AWS 리소스를 사용하기 위한 네트워크 설계도**다. 설계할 때 고민해야 하는 것:

- 어떤 IP 대역을 사용할 것인지
- 어떤 서버를 외부에 공개하고, 어떤 서버를 내부로 숨길 것인지
- 인터넷 트래픽을 어디로 보낼 것인지
- 서버-DB 간 통신은 어디까지 허용할 것인지
- 장애가 발생하면 다른 AZ로 어떻게 전달할 것인지

#### | IAM (Identity and Access Management)

**AWS 리소스에 대한 사용자 인증과 권한 관리를 제공하는 서비스**

##### IAM이 다루는 두 가지

| 영역 | 내용 |
|---|---|
| **인증 (Authentication)** | 사용자, 역할, 로그인 방식, MFA 확인 |
| **인가 (Authorization)** | AWS 서비스/리소스/API별 권한 확인 |

예시 질문:
- 찰리라는 사용자는 EC2를 생성할 수 있는가?
- 개발자는 운영 RDS를 삭제할 수 있는가?
- GitHub Actions는 ECR에 도커 이미지를 올릴 수 있는가?

##### IAM 구성 요소

| 구성 요소 | 역할 |
|---|---|
| **Root User** | AWS 계정을 처음 만들 때 생성되는 최고 관리자 |
| **User** | AWS에 접근하는 개별 사용자 |
| **Group** | 여러 사용자에게 공통 권한을 부여하는 묶음 |
| **Role** | 특정 사용자나 AWS 서비스가 임시로 맡는 권한 |
| **Policy** | 허용·거부할 작업을 작성한 권한 규칙 |
| **MFA** | 로그인 시 비밀번호 외에 추가 인증 요구 |

##### IAM을 사용하는 이유

- **Root 계정은 정말 모든 걸 다 할 수 있는 계정**이며, 보통 결제 카드까지 연결되어 있다
- 실무에서는 다양한 역할의 작업자들이 함께 일한다
  - 개발자가 네트워크 설정에 굳이 관여할 필요는 없다
  - 재무팀이 EC2에 접근할 필요는 없다
- **모두가 모든 권한을 가지고 있으면 그 자체로 큰 보안 위험**이 된다

##### 최소 권한 원칙 (Principle of Least Privilege)

**필요한 작업을 수행할 때, 그 작업에 필요한 최소한의 권한만 부여하는 보안 원칙**

예를 들어 이미지 업로드 서버를 만들었다고 하면:

> S3 전체 권한 + EC2 전체 권한 + RDS 전체 권한을 다 주는 것은 위험하다. **특정 버킷의 특정 경로에 대한 권한만 허용**하고 나머지는 차단해야 한다.

```
이미지 업로드 서버
 ├─ 허용 → s3:PutObject (my-service-image/image/*)
 ├─ 차단 → s3:DeleteBucket
 ├─ 차단 → iam:CreateUser
 └─ 차단 → ec2:TerminateInstances
```

권한은 **처음엔 최소한으로 할당**하고, 나중에 필요성이 확인되면 그때 검토해서 확장해주는 방식이 안전하다.

##### Group — 권한을 묶어서 여러 유저에게 부여

여러 IAM 유저에게 동일한 권한을 한 번에 부여할 때 사용한다. (예: 개발자 3명에게 똑같은 권한이 필요한 경우)

현업에서 흔히 쓰는 그룹 예시:

| 그룹 | 권한 예시 |
|---|---|
| Developers | 개발 계정의 EC2, CloudWatch, S3 일부 접근 |
| Operators | 배포·모니터링·장애 대응 권한 |
| ReadOnlyAuditors | 리소스 조회만 허용 |
| BillingAdmins | 비용·결제 관련 설정 접근 |

##### Policy — 허용/거부 규칙

AWS에서 어떤 작업을 허용할지, 거부할지 정하는 규칙. 보통 4가지 요소로 정의한다.

| 요소 | 의미 |
|---|---|
| **Effect** | 허용(`Allow`)할지 거부(`Deny`)할지 |
| **Action** | 어떤 AWS API 작업을 허용/거부할지 |
| **Resource** | 어떤 AWS 리소스에 적용할지 |
| **Condition** | 어떤 조건에서만 허용/거부할지 |

예시 — 특정 버킷에 파일 업로드만 허용하는 정책:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-service-image/image/*"
    }
  ]
}
```

##### Role — 임시로 빌려 쓰는 권한 묶음

IAM 유저나 AWS 서비스가 **특정 권한을 임시로 빌려서 사용**할 수 있게 하는 권한 묶음. Policy를 미리 묶어두고, 누가 이 권한을 갖고 사용하느냐를 정의하는 개념이다.

예시:
```
BackendFileUploader (Role)
 └─ Policy 연결
     ├─ S3 업로드 권한
     └─ S3 조회 권한
```

##### MFA (Multi-Factor Authentication)

비밀번호 외에 추가 인증 수단(OTP, 인증 앱 등)을 요구하는 다중 인증 방식.

#### | 인프라 구성을 어떻게 잘할 수 있을까?

서비스를 만들 때 흔히 마주치는 질문들:
- VPC 구성을 어떻게 할까?
- 퍼블릭/프라이빗 서브넷 구성을 어떻게 나눌까?
- 데이터베이스를 어떻게 구성해야 할까?
- 내 서비스에는 어느 정도의 크기와 가용성이 필요할까?

##### 인프라 설계 6단계

**1. 제공하는 서비스/기능 명세를 확인**

어떤 서비스인지 알아야 인프라를 구성할 수 있다.
- 비즈니스 로직에서 어디가 부하가 걸리는지, 어디에 중요한 데이터가 있는지 파악
- 제공 형태 (웹 / 앱 / API / 임베디드)
- 보관해야 하는 정보 종류 (데이터 / 파일 / 이미지)
- API Docs, 기획서 참고

**2. 필요한 구성 요소를 고려**

- 데이터베이스가 필요한가? 어떤 DB가 왜 필요한가? 그 DB가 아니면 안 되는 이유는?
- 세션 스토어가 필요한가?
- 웹 서버, WAS가 필요한가?
- 이미지 저장, 컨텐츠 캐싱이 필요한가?
- 로깅이 필요한가? (간단히 에러만 추적한다면 CloudWatch만으로 충분할 수 있음)
- 메트릭 트래킹이 필요한가?

**3. 구성 요소를 확정**

예시:
- 데이터베이스: **MariaDB** (READ 비율이 압도적으로 높은 서비스라면)
- 세션 스토어: 필요 없음
- 웹 서버: Nginx, Apache, H2 등
  - 웹 서버의 역할: Reverse Proxy / Load Balancer / Content Caching
- 파일 컨텐츠 서빙: S3, EFS, CDN(CloudFront) 도입 — 필요하면 CDN 적용 범위를 `images/profile`처럼 좁게 한정

**4. 구성 요소에 대한 노출도를 설정**

| 노출 수준 | 대상 |
|---|---|
| 공개가 필요한 것 | S3, CloudFront |
| 공개가 일부 적용되어야 하는 것 | 웹 서버 |
| 비공개해야 하는 것 | 데이터베이스, WAS |

**5. 목표를 설정**

이 아키텍처가 얼마나 많은 트래픽을 감당해야 하는지 결정한다. 모든 아키텍처가 엄청난 트래픽을 견딜 필요는 없다는 점이 중요하다.

- 제품의 매출 상태, 트래픽 강도와 유형(Spike형인지 예측 가능한 형태인지)을 고려
- 예: 100만 MAU 커뮤니티 → RPS/QPS 기준으로 환산하면 보통 **100~1000 RPS** 수준
- 트래픽을 시스템 어디에서 받아낼지 결정 (WS에서 받을지, DNS 단계에서 로드밸런싱할지, WAS에 큐잉을 적용할지)

**6. 실행**

VPC를 그리고 → 네트워킹 설정 → 데이터베이스 구성 → 인스턴스 배치 순으로 진행한다.

> 💡 실시간 가상화폐 거래소와 네이버 검색 서비스가 같은 기술 구조로 되어 있을까? **아니다.** 비즈니스 특성에 따라 내부 아키텍처는 완전히 달라진다.

#### | 보안 그룹 (Security Group)

**클라우드 환경에서 네트워크 트래픽을 제어하는 가상의 방화벽**

AWS 리소스에 들어오고 나가는 네트워크 트래픽을 허용 규칙으로 제어하는 **상태 저장형(Stateful)** 가상 방화벽이다. "누가, 어떤 포트로, 이 리소스에 접근할 수 있는지"를 정의한다.

##### 보안 그룹의 적용 범위

> ⚠️ 보안 그룹은 **EC2 인스턴스 자체에 붙는 것이 아니다.** AWS에서 EC2는 이미 가상화되어 있는 컴퓨터이기 때문에, 보안 그룹은 그 컴퓨터의 **네트워크 인터페이스(ENI)**에 연결된다.

```
Internet → Security Group → Network Interface(ENI) → EC2 Instance
```

##### 보안 그룹을 사용하는 이유

서버를 인터넷에 무작정 노출하지 않고, 필요한 대상에게만 필요한 통신만 허용하기 위해서다. 보안 그룹이 제대로 설정되지 않으면 다음과 같은 문제가 생긴다.

| 문제 | 예시 |
|---|---|
| SSH 무차별 대입 공격 | 전 세계에서 22번 포트 로그인 시도 |
| DB 외부 노출 | MySQL 3306 포트가 인터넷에 공개 |
| Redis 외부 노출 | Redis 6379 포트에 누구나 접근 |
| 내부 관리 포트 노출 | Spring Boot 8080, 관리자 페이지 노출 |
| 침해 확산 | 하나의 서버가 뚫린 뒤 DB·캐시까지 접근당함 |

> 💡 기본적으로 인바운드 규칙이 없으면, 모든 인바운드 트래픽은 **허용되지 않는다** (기본 거부).

##### 인바운드 / 아웃바운드 트래픽

- **인바운드 트래픽**: 외부나 다른 리소스에서 내 리소스로 **들어오는** 트래픽 (예: 외부 인터넷 → ALB, ALB → WAS)
- **아웃바운드 트래픽**: 내 리소스에서 외부로 **나가는** 트래픽 (예: WAS → OAuth, 3rd Party API)

##### 보안 그룹의 특징

**① 상태 저장형(Stateful) 방화벽**

한 번 허용된 요청의 응답은, 반대 방향 규칙을 별도로 작성하지 않아도 **자동으로 허용**된다.

```
사용자 → ALB 보안그룹 → ALB    (HTTPS 443 요청)
사용자 ← ALB 보안그룹 ← ALB    (응답: 별도 규칙 없이 자동 허용)
```

**② Allow는 가능하지만, 명시적 Deny는 불가능**

보안 그룹은 명시적으로 허용한 트래픽만 통과시키고, 허용 규칙이 없으면 기본적으로 차단한다. 즉 "명시적 거부(Deny)"라는 개념 자체가 없다.

특정 트래픽을 명시적으로 거절하고 싶다면 (특정 IP 대역, 특정 국가 단위 차단 등):
- **Network ACL**
- **WAF**
- **AWS Network Firewall**
- 스프링이나 Nginx 레벨에서 직접 차단

**③ 여러 개의 보안 그룹을 하나의 리소스에 연결 가능**

보안 그룹을 여러 개 연결한다고 보안이 강화되는 것은 아니다. 오히려 **Allow의 범위가 합쳐져서 늘어나는** 결과가 된다. 현업에서는 역할별로 보안 그룹을 쪼개서 필요한 조합만 붙여 쓴다.

```
sg-https              (443)
sg-ssh-for-admin       (22)
sg-tcp-for-file-transfer (4132)
```

##### 보안 그룹 구성 요소

| 항목 | 의미 | 예시 |
|---|---|---|
| Type | 서비스 유형 | SSH, HTTP, HTTPS, MySQL |
| Protocol | 통신 규약 | TCP, UDP, ICMP |
| Port Range | 허용할 포트 | 22, 443, 3306 |
| Source | 인바운드 출발지 | `203.0.113.10/32`, `sg-alb` |
| Destination | 아웃바운드 목적지 | `0.0.0.0/0`, `sg-db` |
| Description | 규칙 목적 설명 | "Allow HTTPS from Internet" |

##### 보안 그룹 참조 (Security Group Reference)

보안 그룹의 Source/Destination을 **IP 대역**으로 설정할 수도 있지만, **보안 그룹 자체**를 소스로 지정할 수도 있다.

> ⚠️ IP 대역대로만 설정하면 문제가 생긴다. 내부망에서 WAS가 트래픽에 따라 늘어나고 줄어들면서 EC2가 생성/삭제되고, 그때마다 새로운 사설 IP를 할당받기 때문이다.

예시 구성:

| 보안 그룹 | 인바운드 허용 | Source |
|---|---|---|
| `sg-alb` | HTTPS 443 | `0.0.0.0/0` |
| `sg-app` | TCP 8080 | `sg-alb` |
| `sg-db` | MySQL 3306 | `sg-app` |

보안 그룹 참조의 장점:
- 서버 수가 늘어나거나 줄어도 **관리가 편함** (IP가 아니라 그룹 단위로 관리)
- "어떤 역할의 서비스가 접근하는지"를 **명시적으로 이해하기 편함**

#### | Route 53

**AWS에서 제공하는 확장 가능한 클라우드 DNS(Domain Name System) 서비스**

사람이 읽을 수 있는 도메인 주소를 서버나 AWS 서비스의 실제 주소로 연결하고, 상황에 따라 어느 서비스로 트래픽을 보낼지 결정하는 DNS 서비스다.

##### Route 53을 왜 사용하는가

- 일반적인 도메인 서비스(가비아 등)는 AWS 내부 서비스와 유기적으로 연결되어 있지 않다
- **글로벌 위치 기반 호스팅**이 가능 (예: 서울 리전 vs 버지니아 북부 리전 — 클라이언트 위치에 따라 가까운 리전으로 호스팅)

##### 도메인 등록 ≠ DNS 관리

도메인을 **등록**하는 기관과, 도메인의 **DNS를 관리/호스팅**하는 곳은 서로 다른 개념이다.

| 구분 | 예시 |
|---|---|
| 도메인 등록 기관 | 가비아, 후이즈, SpaceSquare, Cloudflare 등 |
| DNS 호스팅 서비스 | Route 53, 가비아 DNS 등 |

> 💡 도메인을 가비아에서 구매했더라도, 그 도메인의 **DNS는 Route 53에서 관리**하도록 위임할 수 있다.

##### Hosted Zone

특정 도메인(과 그 서브도메인)에 대한 DNS 레코드를 보관하고 관리하는 공간.

```
example.com
 ├─ www.example.com   (301 Redirect → example.com)
 ├─ api.example.com
 ├─ admin.example.com
 └─ dev.example.com
```

**Public Hosted Zone**
- 인터넷 전체에서 조회 가능한 호스트 영역
- 공개 웹사이트, 공개 API 등
- 외부 DNS 조회가 가능 (예: `api.example.com`, `www.example.com`)

**Private Hosted Zone**
- **VPC 내부에서만** 조회가 가능
- 내부 DB, 사내 API 등에 사용 (예: `db.internal.example.com`)
- 외부 DNS에서는 보이지 않음
- DB 인스턴스도 늘었다 줄었다 하면서 IP가 바뀌기 때문에, `application.yml` 등에서 DB host로 IP 대신 **도메인**을 사용하면 안정적으로 관리할 수 있다

##### NS 레코드 / SOA 레코드

| 레코드 | 질문에 대한 답 |
|---|---|
| **NS (Name Server)** | "이 도메인의 DNS 설정은 어디에서 관리하는가?" |
| **SOA (Start of Authority)** | "이 DNS 영역의 대표 네임서버와 관리/갱신 기준 정보는 무엇인가?" |

##### 자주 사용하는 DNS 레코드

| 레코드 | 역할 | 예시 |
|---|---|---|
| **A** | 도메인 → IPv4 주소 | `example.com → 203.0.113.10` |
| **AAAA** | 도메인 → IPv6 주소 | `example.com → 2001:db8::1` |
| **CNAME** | 도메인 → 다른 도메인 | `www → example.com` |
| **Alias** | Route 53 전용 AWS 리소스 연결 방식 | `example.com → ALB` |
| **MX** | 메일 서버 지정 | `example.com → mail.example.com` |
| **TXT** | 텍스트 기반 인증·정책 정보 | SPF, DKIM, ACM 검증 |
| **NS** | 도메인의 DNS 담당 네임서버 | Route 53 네임서버 |
| **SRV** | 특정 서비스의 호스트·포트 정보 | 일부 서비스 디스커버리 |

**A 레코드**: 도메인을 IPv4 주소로 직접 연결. AWS 내부에서는 **고정 IP(Elastic IP)**를 쓰는 경우에만 사용한다.

**CNAME**: 한 도메인의 이름을 다른 도메인의 이름에 연결.
> ⚠️ DNS 규칙상 **최상위 도메인(루트 도메인) 영역에서는 CNAME을 사용할 수 없다.**
> - `www.domain.com → domain.com` ✅ 가능
> - `domain.com → otherdomain.com` ❌ 불가능

**Alias 레코드**: Route 53에서 AWS 리소스(CloudFront, ALB, NLB, API Gateway, S3 엔드포인트 등)를 도메인에 **직접** 연결하기 위한 AWS 전용 기능. CNAME의 루트 도메인 제약을 우회할 수 있다.

**TXT 레코드**: 일종의 "메모장"으로, 도메인 소유권 확인이나 이메일 보안 정책(SPF, DKIM) 등 텍스트 정보를 저장한다.
> DNS 레코드는 그 도메인의 소유자만 수정할 수 있기 때문에, "내가 이 도메인의 진짜 소유자다"라는 걸 증명할 때 TXT 레코드에 외부 서비스가 요구하는 값을 넣어 인증한다.

##### Route 53 라우팅 정책

DNS의 본질적인 역할은 **A 레코드의 IP를 리턴해주는 것**이지만, Route 53은 이를 활용해 부하 분산, 장애 조치, 카나리 배포 등을 지원한다.

1. **Simple Routing** — 하나의 도메인에 하나의 IP/리소스를 연결 (예: 사용자 → Route 53 → ALB)
2. **Weighted Routing** — 여러 대상에 비율대로 트래픽을 분산 (예: 기존 v1에 가중치 90, 신규 v2에 가중치 10)
3. **Failover Routing** — 주 서버가 정상이면 주 서버로, 비정상이면 대기 중인 부 서버로 전환
4. **Latency Routing** — 사용자와 AWS 리전 사이 **지연 시간이 가장 낮은** 리전으로 라우팅
5. **Geolocation Routing** — 사용자의 IP를 기준으로 **가장 가까운 대륙/지역**의 리전으로 라우팅

##### 실무에서 사용하는 방식

**① 도메인 → ALB 연결**

ALB는 고정 IP를 갖지 않기 때문에 A 레코드로 직접 연결할 수 없고, **Alias 레코드**로 연결한다.

```
사용자 → api.example.com → Route 53 → (Alias) → ALB → Spring Boot App 1/2
```

**② CloudFront + S3 정적 웹사이트**

프론트엔드와 백엔드를 분리할 때 자주 쓰는 구성.

```
사용자 → www.example.com → Route 53 → (Alias) → CloudFront → S3 Bucket
```

- `www.example.com` → CloudFront + S3 (또는 프론트엔드 서비스)
- `api.example.com` → ALB + Spring Boot
- `admin.example.com` → 관리자 서비스

#### | EC2 (Elastic Compute Cloud)

**AWS에서 제공하는 클라우드 가상 서버 서비스**

##### 사전 개념 — 하이퍼바이저(Hypervisor)와 가상화

**하이퍼바이저**: 물리 서버 한 대의 CPU, 메모리, 디스크, 네트워크 등의 자원을 여러 개의 가상 컴퓨터가 나눠 쓰면서도 서로 격리되도록 해주는 소프트웨어 계층.

> 물리 서버 한 대를 고객 한 명에게 통째로 빌려주면, CPU나 다른 자원이 남아돌아도 다른 사용자에게는 빌려줄 수 없다. 그래서 서버를 잘게 쪼개고, 하이퍼바이저 소프트웨어로 OS를 분리해서 여러 개를 동시에 띄운다.

**하이퍼바이저의 역할**
- **CPU 분배**: 어떤 가상 서버가 언제 CPU를 사용할지 (CPU 실행 시간 분배)
- **메모리 분리**: 각 가상 서버가 자기 메모리만 쓰는 것처럼 보이게 함
- **가상 장치 제공**
- **격리**: 가상 서버들이 서로 영향을 주지 않도록 분리

```
물리 서버
 └─ 하이퍼바이저
     ├─ 가상머신 1: Ubuntu + Spring Boot
     ├─ 가상머신 2: Amazon Linux + Node.js
     └─ 가상머신 3: Windows + IIS/.NET
```

**가상화**: 하나의 물리 컴퓨터로 여러 개의 독립된 가상 컴퓨터를 실행하는 기술. 각 가상머신은 서로 독립되어 있고, 자기만의 컴퓨터인 것처럼 동작한다. **EC2의 "인스턴스"라는 단위도 결국 하이퍼바이저가 분리해놓은 단위**다.

##### EC2를 왜 배우는가

- AWS에서 가장 **기초적인 컴퓨팅 자원**
- 백엔드 서버, 게임 서버, 크롤러, VPN 서버 등 매우 범용적이고 자유롭게 활용 가능
- 운영체제 자체를 빌려서 격리시켜 쓰는 것이기 때문에, 다음을 직접 다뤄야 한다: 애플리케이션, 네트워크, 운영체제, 보안, 스토리지, 모니터링

**AWS 서비스가 파생되는 방식**

EC2에 이런저런 소프트웨어를 설치하는 과정은 대체로 반복적이고 비슷하다.

```
OS를 빌린다 → 컨테이너 시스템(Docker)을 설치 → 컨테이너를 실행 → 서비스로 만든다 (ECS, EKS 등)
```

> 💡 "서비스"란 결국 **남이 하기 싫은 걸 대신 해주고 돈을 받는 것**이다. 그런 관점에서 가장 저렴한 서비스가 바로 EC2다 (동일한 컴퓨팅 실행 시간을 기준으로 비교했을 때).

##### EC2가 생성되는 흐름

```
사용자 → AWS 제어 영역 → (AMI, 인스턴스 타입, VPC, 보안 그룹 확인)
       → 하이퍼바이저/Nitro System → 적절한 물리 서버 선택
       → CPU/메모리/네트워크 자원 할당 요청
       → 가상 CPU, 메모리, 가상 NIC, 가상 디스크 제공
       → EC2 인스턴스에 AMI 기반 운영체제 부팅
       → 사용자에게 EC2 Running 상태 반환
```

##### 온프레미스 vs 클라우드

| 구분 | 온프레미스 | 클라우드(EC2) |
|---|---|---|
| 초기 비용 | 서버 구매 비용이 **서비스 판매 전에 먼저 발생** | 사용한 만큼만 과금 |
| 자산 처리 | 구매한 서버는 **자산**으로 통계 | 빌린 것이므로 그대로 **비용**으로 통계 |
| 증설 | 증설하려면 비용과 시간이 크게 듦 | 즉시 확장 가능 |
| 장애 대응 | 직접 고쳐야 함 | AWS가 인프라 레벨을 관리 |
| 비용 (장기·대규모) | 상대적으로 **저렴**할 수 있음 (예: 게임회사처럼 서버가 대규모로 항상 필요한 경우 직접 구매) | 소규모/가변적 트래픽에 유리 |

> 내 상품이 얼마나 잘 팔릴지 모르는 상태에서 서버를 먼저 구매하는 건 부담이 크다. 이것이 클라우드가 해결한 핵심 문제 중 하나다.

##### EC2 핵심 요소

**1. AMI (Amazon Machine Image)**

서버를 만드는 템플릿. 운영체제 + 초기 소프트웨어 구성이 들어있는 서버 이미지.

- EC2 인스턴스를 만들 때 Ubuntu, Windows 등 다양한 **기본 제공 AMI** 중 선택 가능
- 실무에서는 운영 서버를 매번 새로 만들 때마다 JDK 설치 → 소켓 서버 설치 → 도커 설치 → git clone & 빌드 → ... 항상 SSH로 들어가서 반복하는 대신, **미리 구워놓은 Custom AMI**를 사용한다
- Custom AMI는 반복 배포와 오토 스케일링에 필수적이다

**2. 인스턴스 타입**

CPU + 메모리 조합으로 정의되는 성능 사양. 예: `t3.medium`

- `t` : 버스터블 클래스 (Burstable — 인스턴스 계열)
- `3` : 세대 (몇 세대 CPU인지)
- `medium` : CPU/메모리 조합 크기

| 계열 | 특징 | 대표 사용 사례 |
|---|---|---|
| **T 계열** | 평소엔 낮은 CPU 사용량, 순간적으로 CPU를 더 사용할 수 있음(버스터블) | 개발 서버, 작은 웹 서버, 개인 프로젝트 |
| **M 계열** | CPU·메모리 균형형 | 일반적인 백엔드 서버, API 서버 |
| **C 계열** | CPU 중심 | 배치 작업, 인코딩, 계산량 많은 서비스 |
| **R 계열** | 메모리 중심 | 캐시, 대용량 JVM, 메모리 사용량 큰 DB |
| **G/P 계열** | GPU 중심 | AI 추론, 머신러닝, 그래픽 처리 |
| **I/D 계열** | 고성능 로컬 스토리지 중심 | 고성능 데이터 처리, 임시 대용량 작업 |

> 💡 어떤 계열을 써야 할지는 결국 **부하 테스트**로 결정한다. 우리 서비스의 트래픽 패턴을 분석해서 적합한 타입을 골라야 한다.

**3. EBS (Elastic Block Store)**

EC2에 연결되는 **영구 블록 스토리지**. 가상의 하드디스크 같은 개념이다.

> EC2는 하이퍼바이저로 가상화된 컴퓨팅 자원이다. 그렇다면 서버(인스턴스)를 끄면 내 데이터는 어떻게 될까? — 이 데이터를 보관하는 별도 저장소가 바로 **EBS**다.

EC2와 EBS는 **마운트되는 별개의 관계**다. 하나의 EBS를 다른 EC2에 옮겨 붙여도 동일하게 사용할 수 있다.

| 상황 | EC2 | EBS |
|---|---|---|
| 인스턴스 실행 | 서버 실행 | 디스크 사용 |
| 인스턴스 중지(Stop) | CPU·메모리 사용 중단 | **EBS는 계속 존재 가능** |
| 인스턴스 시작 | 다시 실행 | 기존 EBS 연결 |
| 인스턴스 종료(Terminate) | 서버 삭제 | 삭제 정책에 따라 EBS도 삭제될 수 있음 |
| 스냅샷 생성 | 서버와 별개로 동작 | 특정 시점 백업 생성 |

**4. 네트워크**

AWS에 존재하는 대부분의 자원은 네트워크를 통해 접근한다 (클라우드이므로 당연하다). **EC2는 반드시 VPC 내 서브넷 안에 배치**된다.

```
Internet → Internet Gateway → Public Subnet(10.0.1.0/24)
                                  └─ EC2 (Public IP + Private IP)
                                       ↓
                              Private Subnet(10.0.2.0/24)
                                  └─ EC2 (Private IP만 보유)
                                       ↓
                                      RDS
```

**5. 보안 그룹**

앞서 정리한 대로, EC2(정확히는 ENI)에 연결되어 인바운드/아웃바운드 트래픽을 제어한다.

##### 실무에서 EC2를 잘 사용하는 방법

**① 태그를 강제한다**

| 태그 | 예시 |
|---|---|
| Name | `prod-api-01` |
| Environment | `dev`, `staging`, `prod` |
| Service | `community-api` |
| Owner | `backend-team` |
| CostCenter | `startup-core` |
| ManagedBy | `terraform` |

**② EC2에는 중요한 파일을 저장하지 않는다**

EC2는 언제든 삭제될 수 있는 자원이다. 중요한 파일은 **S3** 같은 별도 스토리지에 저장하고 관리하는 것이 안전하다.

**③ 서버에서 직접 SSH로 파일을 수정하지 않는다**

서버가 한 대일 때는 가능하지만, 서버가 30대라면 일일이 들어가서 고칠 수 없다. 보통은 **수동 배포 트리거**를 누르면 배포 스크립트가 `.env` 파일을 참조해서 최종 실행 파일을 만들고, 이를 서버들에 일괄 전달하는 방식으로 처리한다.

**④ 장애를 전제로 구성한다 (★ 중요)**

EC2는 특히 장애가 발생할 수 있는 자원이다. 장애가 발생했을 때 **빠르게 알아차릴 수 있는 구조**(모니터링, 알람)를 처음부터 함께 설계해야 한다.

### 🗨️ 오늘의 회고

- 오늘 강의는 데이터센터의 한계 → AWS의 등장 → VPC → IAM → 보안그룹 → Route 53 → EC2로 이어지는 흐름이었는데, 결국 AWS에 내 서비스를 어떻게 안전하고 효율적으로 배치할 것인가라는 하나의 질문에 대한 답이라는 걸 느꼈다.
- 지난주에 배운 Reverse Proxy, NAT, CIDR 개념이 오늘 VPC/EC2 내용과 그대로 이어졌다. NAT Gateway는 결국 지난주에 배운 NAT의 AWS 구현체였고, CIDR은 VPC와 서브넷을 나누는 데 그대로 쓰였다.
- IAM의 최소 권한 원칙이 인상적이었다. 이미지 업로드 서버에 S3/EC2/RDS 전체 권한을 다 주는 건 당장은 편하지만, 침해가 한 번이라도 발생하면 피해 범위가 전체로 번진다는 걸 깨달았다.
- 보안 그룹이 Allow만 가능하고 Deny는 불가능하다는 점, 그리고 여러 개를 붙일수록 보안이 강화되는 게 아니라 Allow 범위가 합쳐져서 늘어난다는 점은 처음 들었을 때 직관과 반대라 헷갈렸는데, 그림으로 보니 이해가 됐다.
