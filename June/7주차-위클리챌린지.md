#7주차 위클리 챌린지

# 1. 트래픽 가정 및 인프라 규격 도출 이유

## 회원 수 가정

- **전체 회원 수**: 10,000명
- **MAU (월간 활성 사용자 수)**: 10,000명 (모든 회원이 월 1회 이상 접속)
- **DAU (일간 활성 사용자 수)**: 1,000명 (MAU의 10%)
- **사용자당 일일 세션 수**: 1.5회
- **세션당 API 호출 수**: 10번

## 트래픽 계산

- 하루 평균 API 요청 수: 1,000명 X 1.5회 X 10번 = 15,000번
- 평균 API RPS: 15,000번 / 86,400초 = 약 0.17 RPS
- 피크 RPS: 평시 트래픽의 약 10배 가정
- 트래픽 폭증의 경우(이벤트, 사용자 수 급증)을 대비하기 위해 **최대 20 RPS**를 안정적으로 처리하는 것을 목표로 설정합니다.

## 인프라 규격 도출 이유

- **t3.micro(vCPU 2, Mem 1GiB)** 성능으로도 시스템 자원이 여유롭습니다.
- 전면에 **CloudFront가** 배치되어 프론트엔드의 정적 소스 및 S3 내 이미지 요청을 최전방 Edge Location에서 대부분 캐싱 처리하므로 EC2가 받는 실질적인 부하는 더욱 낮아집니다.
- 백엔드 EC2에서 S3로 이미지를 업로드할 때 발생하는 대용량 데이터 트래픽은 **S3 Gateway Endpoint**가 무료로 AWS 내부망 라우팅을 수행하므로, NAT Gateway 유료 데이터 처리 비용을 전혀 발생시키지 않습니다.

# 2. 클라우드 아키텍처 설계

- **VPC 대역**: `10.0.0.0/16`
- **가용 구역 (Availability Zone)**: `ap-northeast-2a` (AZ 1), `ap-northeast-2c` (AZ 2)
- **네트워크 및 스토리지 구성**:
    - **DNS 망**: 가비아 공식 도메인(`lulu-roh.xyz`)을 AWS 최전방 인프라와 결합
    - **Edge Location**: 최전방 **CloudFront(CDN)** 배치. 무료 HTTPS 통신 암호화 및 정적 자원 캐싱 전담
    - **Public Subnet 1 & 2**: 외부 인터넷과 접하는 **ALB** 배치
    - **Private Subnet 1 & 2**: **프론트엔드 EC2 (2대)**, **백엔드 EC2 (2대)** 배치
    - **VPC Endpoint**: **S3용 Gateway Endpoint** 배치. 백엔드에서 S3 버킷으로 업로드하는 파일 트래픽을 인터넷망을 거치지 않고 AWS 내부망으로 다이렉트 전송
    - **Privte Subnet 3 & 4**: **RDS (MySQL)** 배치
    - **AWS S3**: 이미지 파일 저장 전용 스토리지

# 3. 기술 선택 이유

**1. 네트워크 및 스토리지: 로컬 파일 저장 vs AWS S3**

- **고민 과정:**
    - **방안 A (로컬 파일 시스템):** 개발이 간편하고 추가 비용이 없으나, 서버가 여러 대일 경우 파일 공유가 불가능하며 서버 인스턴스 종료 시 데이터가 휘발됨
    - **방안 B (AWS S3):** 외부 스토리지로 파일 분리 가능, 무제한 확장성, 안정성 확보.
- **최종 선택 (S3) 및 이유:** 트래픽 증가에 따른 EC2 스케일 아웃 시, 파일 공유 문제 없이 모든 서버가 동일한 데이터에 접근해야 하므로 S3를 사용하였습니다
    - **Stateless(무상태) 설계:** 서버 인스턴스가 언제든 교체되어도 데이터는 안전하게 보존되는 현대적인 클라우드 설계 원칙을 준수하기 위해 선택했습니다.

**2. 네트워크 아키텍처: NAT Gateway vs S3 Gateway Endpoint**

- **고민 과정:**
    - **방안 A (NAT Gateway):** 프라이빗 서버의 인터넷 통신을 위한 표준 방법이지만, 월 $30 이상의 고정 비용 발생.
    - **방안 B (S3 Gateway Endpoint):** 데이터 처리 비용 없이 내부망 통신 가능.
- **최종 선택 (S3 Gateway Endpoint) 및 이유:** 학습 환경에서 NAT Gateway의 고정 비용은 비효율적이라 판단했습니다.
    - **보안:** S3 통신을 인터넷 구간이 아닌 AWS 으로 격리함으로써 보안 사고를 차단하고, 업로드 속도를 극대화할 수 있다는 점에 선택했습니다.

#### 3. 인프라 구조: 단일 서버 vs 다중 인스턴스 (Multi-AZ)

- **고민 과정:**
    - **방안 A (단일 인스턴스):** 구축이 빠르고 비용이 저렴함.
    - **방안 B (Multi-AZ):** 초기 설정 비용과 복잡도는 증가하지만, 고가용성 확보 가능.
- **최종 선택 (Multi-AZ 이중화) 및 이유:** 단일 서버는 장애 발생 시 서비스가 완전히 중단됩니다. 서비스의 가용성을 확보하기 위해 리전 내 2개의 AZ로 트래픽을 분산하는 구조를 채택했습니다.
