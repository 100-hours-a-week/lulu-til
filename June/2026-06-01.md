#### | Spring & Spring Boot 복습

##### Spring

자바를 자바답게, **대규모 애플리케이션** 개발을 편하게 하기 위한 프레임워크

| 핵심 특징 | 설명 |
|---|---|
| **AOP** | 로깅, 트랜잭션 같은 공통 관심사와 비즈니스 로직을 분리 |
| **IoC / DI** | 객체의 생명주기를 개발자가 아닌 프레임워크 컨테이너가 담당 → 결합도 낮아짐 |
| **POJO** | 순수 자바로만 이루어진 프레임워크, 외부 라이브러리 의존성 없음 |
| **Bean** | 스프링이 관리하는 객체. 빈 컨테이너에 등록해 필요할 때 꺼내 사용 |

스프링을 더 쉽게 시작할 수 있게 만든 프레임워크. 스프링을 대체하는 게 아니라 **스프링 위에 얹어진 것**

| 핵심 기능 | 설명 |
|---|---|
| **Auto-Configuration** | 클래스패스의 라이브러리를 감지해 자동으로 스프링 빈 설정 (`@SpringBootApplication`) |
| **Starter Dependencies** | 웹·DB·보안 등 특성별로 필요한 라이브러리를 묶음으로 제공 (`spring-boot-starter-*`) |
| **Embedded Server** | 외부 WAS 없이 내장 Tomcat으로 동작 → JAR 파일 하나로 실행 가능 |

#### | Annotation

**자바 코드에 메타데이터를 추가해, 컴파일러나 런타임에 특정 동작을 지시하거나 정보를 제공하는 표기 방법**

- 사람이 읽는 주석 역할 + 프레임워크/컴파일러가 읽는 기계용 주석 역할을 동시에 함
- 프로그램의 로직을 스스로 가지지 않고, **어노테이션을 보고 프레임워크가 행동을 결정**

##### 왜 써야 할까?

1. **설정이 응집된다** : 기존에는 XML에 설정을 따로 작성했지만, `@Entity` `@Id` 처럼 코드와 설정이 한 곳에 모임
2. **정적·동적 검증 자동화** : `@NotNull` `@Size` 같은 어노테이션으로 원래 if문으로 작성해야 할 검증을 프레임워크가 런타임에 자동 처리

##### 자주 쓰는 어노테이션

| 어노테이션 | 역할 |
|---|---|
| `@Override` | 부모 메서드를 정확히 구현했는지 컴파일러가 검증 |
| `@Deprecated` | 더 이상 사용하지 않는 구형 코드임을 표기 |
| `@SuppressWarnings` | 의도한 경고이므로 컴파일러에게 특정 경고 메시지를 숨기라고 지시 |

#### | ORM (Object-Relational Mapping)

**자바 객체와 데이터베이스 테이블을 자동으로 연결해, 객체 지향 코드로 데이터를 조작할 수 있게 해주는 기술**

##### 왜 ORM이 필요한가?

SQL 중심으로 개발하면 자바 개발자가 비즈니스 로직보다 SQL 매핑과 최적화에 더 많은 시간을 쓰게 된다.
ORM은 중간에서 객체와 테이블을 자동으로 연결해준다.

##### 객체 지향 vs 관계형 DB의 차이

| 항목 | 자바 (객체) | RDB |
|---|---|---|
| **상속** | 클래스 상속 지원 | 상속 개념 없음 (유사하게 구현 시 복잡) |
| **연관관계** | 참조 (단방향) | 외래키 + JOIN (양방향) |
| **데이터 탐색** | 그래프 탐색 (A → B → C) | SQL에 따라 넓은 범위 탐색 가능 |

> 💡 ORM은 이 구조적 차이를 중간 다리 역할로 연결해준다.

#### | JPA (Java Persistence API)

**자바 객체와 관계형 데이터베이스를 매핑하고, 객체 단위로 데이터를 저장/조회할 수 있게 해주는 표준 ORM 인터페이스**

##### 왜 JPA까지 왔나? — 기술 발전 흐름

```
데이터의 체계적인 저장 필요          → 데이터베이스
중복 제거, 관계로 구조화              → 관계형 데이터베이스(RDB)
체계적인 질문 필요                   → SQL
자바에서 RDB SQL 실행                → JDBC
객체-테이블 불일치 + 반복 SQL 문제   → ORM
자바 표준 ORM 인터페이스             → JPA
```

##### JPA 동작 방식

```
개발자 → 엔티티 전달 → JPA가 구조 분석 → SQL 자동 생성 → JDBC로 실행
```

개발자가 직접 JDBC를 다루지 않고, JPA가 대리인(ORM) 역할을 수행한다.

> 💡 JPA는 **표준 인터페이스**이고, 실제 구현체는 **Hibernate**가 담당한다.

#### | Entity

**데이터베이스 테이블과 1:1로 매핑되어 데이터를 객체 형태로 표현하는 자바 클래스** (`@Entity`로 선언)

##### 엔티티를 쓰는 이유

1. **데이터 규칙 응집** : 테이블은 틀만 제공하지만, 엔티티에는 `@Size` `@NotNull` 같은 제약 조건과 비즈니스 로직을 함께 담을 수 있다
2. **수정 용이** : 데이터와 규칙이 한 곳에 있어 엔티티 클래스 한 곳만 수정하면 됨

##### 엔티티 작성 규칙

- 매개변수 없는 **기본 생성자** 필요
- `final` 클래스 불가 (JPA가 프록시 객체를 만들 때가 있음)
- `enum`, `interface`는 JPA 엔티티로 사용 불가

#### | DTO (Data Transfer Object)

**계층 간 데이터 전달을 위해 사용하는, 값만 담는 전용 객체**

##### 왜 엔티티를 그냥 쓰면 안 될까?

엔티티에는 외부에 노출되면 위험한 데이터(비밀번호, 권한 등)도 있어 직접 전달하면 보안상 좋지 않다.

##### DTO의 핵심 원칙

- **비즈니스 로직 배제** — 데이터 전달에만 집중 (필드 + Getter)
- **소통 규격화** — 프론트엔드와 API 스펙으로 통신할 때 기준점 역할

**현업 사용 패턴**

```
Repository → Entity 조회
Service    → Entity를 필요한 DTO로 변환
Controller → Request/Response DTO로 클라이언트와 통신
```

#### | @Id & 기본 키 생성 전략

**`@Id`** : 엔티티의 식별자(PK)를 지정하는 어노테이션. 영속성 컨텍스트에서 엔티티를 구분하는 유일한 키

##### 기본 키 생성 전략 (`@GeneratedValue`)

| 전략 | 방식 | 특징 |
|---|---|---|
| **IDENTITY** | DB의 AUTO_INCREMENT에 위임 | DB에 실제 삽입 후 ID를 받아옴 → **쓰기 지연 불가**, 배치 처리에 불리 |
| **SEQUENCE** | 별도 시퀀스 테이블에서 번호 관리 | 쓰기 지연 가능, `allocationSize`로 미리 번호 묶음을 가져와 성능 최적화 |
| **TABLE** | 키 생성 전용 테이블을 만들어 시퀀스 흉내 | DB 종류 무관하나 속도 느림. 잘 안 씀 |
| **AUTO** | DBMS에 맞게 자동 선택 | MySQL → IDENTITY, Oracle/PostgreSQL → SEQUENCE |

```java
// IDENTITY 예시
@Id @GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;

// SEQUENCE 예시
@SequenceGenerator(name = "user_seq_gen", sequenceName = "user_sequence", allocationSize = 50)
@Id @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "user_seq_gen")
private Long id;
```

#### | 주요 어노테이션 정리

##### @Column

엔티티 필드가 DB에 어떤 이름과 형식으로 저장될지 지정

| 속성 | 설명 | 기본값 |
|---|---|---|
| `name` | 매핑할 컬럼명 | 필드명과 동일 |
| `nullable` | null 허용 여부 (`false` = NOT NULL) | `true` |
| `unique` | 유니크 제약 조건 설정 | - |
| `length` | 문자열 컬럼 길이 | 255 |
| `insertable` | INSERT 시 포함 여부 | `true` |
| `updatable` | UPDATE 시 포함 여부 | `true` |

##### @Enumerated

Enum 값을 DB에 저장할 때 사용

| 전략 | 설명 |
|---|---|
| `ORDINAL` | Enum 순서(숫자)로 저장 → Enum 순서 변경 시 데이터 깨질 위험 ⚠️ |
| `STRING` | Enum 이름(문자열)로 저장 → 권장 |

##### @Transient

비즈니스 로직에만 필요한 필드를 엔티티에 포함하되 **DB에는 저장하지 않을 때** 사용

#### | 영속성 컨텍스트 (Persistence Context)

**JPA가 엔티티 객체를 관리하며, 동일 트랜잭션 내에서 변경 사항을 추적하고 자동으로 DB와 동기화하는 메모리 상의 저장소**

WAS(메모리)와 DB(하드디스크) 사이의 속도 차이를 완충하고, 엔티티 변경을 놓치지 않고 DB에 반영하기 위해 존재한다.

```
클라이언트 → Entity Manager ↔ 영속성 컨텍스트 (1차 캐시 + 쓰기 지연 저장소) ↔ DB
```

##### 영속성 컨텍스트의 3가지 핵심 특징

**1. 1차 캐시**
- 엔티티와 식별자가 Map 형태로 저장
- 같은 ID로 두 번 조회해도 두 번째는 DB에 접근하지 않고 캐시에서 바로 반환

```java
entityManager.persist(user1);              // 영속성 컨텍스트 등록
entityManager.find(User.class, user1.getId()); // 1차 캐시에서 조회 (DB 접근 없음)
entityManager.find(User.class, user1.getId()); // 동일, DB 접근 없음
```

**2. 쓰기 지연 (Write-behind)**
- 엔티티 변경 시 쿼리를 즉시 DB에 보내지 않고 **쓰기 지연 저장소에 누적**
- `flush()` 또는 트랜잭션 커밋 시 한 번에 DB에 반영

```java
entityManager.persist(user1); // 쓰기 지연 저장소에 쌓임
entityManager.persist(user2); // 쓰기 지연 저장소에 쌓임
// 트랜잭션 커밋 → flush → 한 번에 INSERT 2건
```

**3. 변경 감지 (Dirty Checking)**
- 엔티티가 영속성 컨텍스트에 처음 들어올 때 **스냅샷** 저장
- `flush()` 전, 현재 엔티티와 스냅샷을 비교
- 차이가 있으면 자동으로 UPDATE 쿼리를 생성해 DB에 반영

##### 영속성 컨텍스트 내 명령어

| 메서드 | 역할 |
|---|---|
| `persist(entity)` | 새 엔티티를 영속성 컨텍스트에 등록 |
| `flush()` | 영속성 컨텍스트의 변경 내용을 DB에 반영 |
| `remove(entity)` | 엔티티를 삭제 예약 상태로 전환 |
| `clear()` | 영속성 컨텍스트 전체 초기화 (엔티티 준영속 전환) |
| `detach(entity)` | 특정 엔티티만 영속성 컨텍스트에서 분리 |
| `close()` | 영속성 컨텍스트 종료 |
| `merge()` | 준영속/비영속 엔티티를 다시 영속성 컨텍스트에 등록 |

##### 영속성 컨텍스트와 트랜잭션의 생명주기

트랜잭션과 영속성 컨텍스트는 생명주기를 함께한다.

```
① 트랜잭션 시작
② 영속성 컨텍스트 생성 (저장소 생성)
③ 엔티티 변화 감지
④ 작업 완료 → 트랜잭션 커밋
⑤ 영속성 컨텍스트 flush (DB 반영)
⑥ 트랜잭션 종료
⑦ 영속성 컨텍스트 close
```

> ⚠️ **같은 트랜잭션이면 다른 EntityManager라도 동일한 영속성 컨텍스트를 공유한다.**
> 반대로 같은 EntityManager라도 다른 트랜잭션이면 별도의 영속성 컨텍스트를 사용 → **Lost Update 발생 가능**

##### 엔티티의 4가지 상태

| 상태 | 설명 |
|---|---|
| **비영속** | 객체는 생성됐지만 영속성 컨텍스트가 관리하지 않는 상태 |
| **영속** | 영속성 컨텍스트에 저장·관리되는 상태. 1차 캐시·쓰기 지연·변경 감지 대상 |
| **준영속** | 영속 상태였으나 영속성 컨텍스트에서 분리된 상태 (영속성 컨텍스트가 닫히면 자동 전환) |
| **삭제** | DB에서 삭제 예약된 상태 |

#### | 연관관계 매핑

**엔티티 객체 간의 관계를 설정하여, 객체지향적으로 연결된 데이터 구조를 정의하고 관리하는 매핑**

##### 1. 단방향 연관관계

한 엔티티에서만 다른 엔티티를 참조하는 방식

- 참조를 가진 쪽 → 반대편 접근 가능
- 반대편 → 접근 불가
- 주요 어노테이션 : `@ManyToOne`, `@OneToOne`

**사용 이유** : 설계 단순화, 불필요한 의존성 제거
양방향으로 모두 참조를 만들면 생성·수정 시 양쪽을 항상 일치시켜야 해 코드가 복잡해진다.

##### 2. 양방향 연관관계

두 엔티티가 서로를 참조해 양쪽 모두에서 탐색 가능한 방식

- 주요 어노테이션 : `@OneToMany`, `@ManyToMany`
- **연관관계의 주인** : 외래키를 실제로 관리하는 주체 (보통 `@ManyToOne` 쪽)

**사용 이유** : 양쪽 참조가 모두 필요한 경우
예) 게시글 상세에서 작성자 조회 + 사용자 프로필에서 작성한 글 목록 조회

> 💡 **실무 팁** : 초기에는 단방향으로 설계하고, 필요한 경우 이후 양방향을 추가하는 방식으로 개발하는 것이 권장된다.

#### | 프록시와 로딩 전략

##### 프록시 (Proxy)

JPA에서 지연 로딩을 사용할 때, 실제 엔티티 대신 먼저 생성되어 전달되는 **가짜 객체**
실제 데이터가 필요한 시점에 DB를 조회해 채워진다.

##### 즉시 로딩 (Eager Loading)

연관 엔티티를 **LEFT OUTER JOIN으로 한 번에** 조회

- ❌ 불필요한 JOIN 포함 가능, 성능 최적화 어려움

```java
@ManyToOne(fetch = FetchType.EAGER) // 즉시 로딩
```

##### 지연 로딩 (Lazy Loading)

연관 엔티티를 **실제 접근하는 시점에** DB 조회

- 처음에는 프록시 객체를 넣어두고, 실제 데이터가 필요할 때 DB 조회
- ✅ 불필요한 쿼리를 줄일 수 있어 성능 최적화에 유리

```java
@OneToMany(mappedBy = "user", fetch = FetchType.LAZY) // 지연 로딩
```

##### 기본 로딩 전략

| 어노테이션 | 기본값 |
|---|---|
| `@ManyToOne` | 즉시 로딩 (EAGER) |
| `@OneToOne` | 즉시 로딩 (EAGER) |
| `@OneToMany` | 지연 로딩 (LAZY) |
| `@ManyToMany` | 지연 로딩 (LAZY) |

> ⚠️ `@ManyToOne`, `@OneToOne`의 기본값이 EAGER이므로, **실무에서는 모두 LAZY로 명시적으로 설정**하고 필요한 경우에만 페치 조인으로 즉시 로딩하는 것이 권장된다.

---

### 🗨️ 오늘의 회고

- Spring → Spring Boot → JPA → 영속성 컨텍스트 순서로 흐름이 이어지면서, 각 기술이 왜 등장했는지 맥락이 자연스럽게 연결됐다.
- 영속성 컨텍스트의 1차 캐시, 쓰기 지연, 변경 감지가 각각 "DB 접근 줄이기", "I/O 묶어서 처리", "자동 UPDATE"라는 목적으로 연결된다는 점이 인상적이었다.
- 기본 키 생성 전략은 IDENTITY가 쓰기 지연을 무효화한다는 점이 중요한 트레이드오프였다.
- 연관관계 매핑은 단방향부터 시작해 필요할 때 양방향을 추가하는 방식이 실무 권장 패턴이라는 점을 기억해야겠다.
- `@ManyToOne`이 기본값이 EAGER라는 것, 실무에서는 전부 LAZY로 명시하는 게 낫다는 점도 중요한 포인트.
