## 1. FetchType 개요

### FetchType이란?

JPA에서 엔티티를 조회할 때, **연관된 엔티티를 어떤 방식으로 조회할지 결정하는 전략**이다.
`javax.persistence.FetchType` 패키지 내에 열거형(Enum) 형태로 제공된다.

```java
public enum FetchType {
    EAGER,
    LAZY
}
```

### 언제 사용하는가?

A 엔티티와 B 엔티티가 연관 관계에 있을 때, A를 조회하면서 B를 **즉시 함께 가져올지(EAGER)**, **필요할 때만 가져올지(LAZY)** 를 결정하는 데 사용한다.

예를 들어, 부모 엔티티에 자식 엔티티가 무수히 많다면, 부모 목록을 가져올 때 자식 전체를 동시에 불러오면 심각한 성능 저하가 발생한다. 이럴 때 LAZY 로딩을 사용하면 자식 엔티티는 필요할 때만 접근할 수 있다.

### 관계 매핑 어노테이션

| 어노테이션 | 관계 | 기본 FetchType |
|---|---|---|
| `@ManyToOne` | N:1 | `EAGER` |
| `@OneToOne` | 1:1 | `EAGER` |
| `@OneToMany` | 1:N | `LAZY` |
| `@ManyToMany` | N:N | `LAZY` |

### 기본값이 다른 이유

- **`@ManyToOne`, `@OneToOne`**: 단일 객체 참조 형태이다. 부모 엔티티 조회 시 참조 엔티티도 함께 필요한 경우가 많고, 성능에 미치는 영향이 비교적 적기 때문에 기본값이 `EAGER`이다.
- **`@OneToMany`, `@ManyToMany`**: 컬렉션(Collection) 또는 Set 형태이다. 즉시 로딩으로 설정하면 한 번에 너무 많은 데이터를 가져와 심각한 성능 저하를 일으킬 수 있기 때문에 기본값이 `LAZY`이다.

### 어떤 전략을 선택해야 할까?

- **대용량 컬렉션에는 지연 로딩(LAZY)**: 연관된 엔티티의 수가 많을 수 있는 관계에서는 LAZY가 성능상 유리하다.
- **단일 관계에는 즉시 로딩(EAGER) 고려 가능**: 연관 엔티티가 항상 함께 필요하거나 성능에 미치는 영향이 미미한 경우 EAGER가 편리할 수 있다.
- **실무에서는 가급적 모든 연관 관계를 LAZY로 설정**하고, 필요한 경우에만 Fetch Join을 적용하는 것이 권장된다.

## 2. FetchType.EAGER (즉시 로딩)

### 개념

엔티티를 조회할 때 **연관된 엔티티도 즉시 함께 로드**하는 방식이다.
한 번의 쿼리로 모든 데이터를 가져올 수 있으나, 필요하지 않은 데이터까지 조회될 수 있어 성능 이슈가 발생할 수 있다.

### 엔티티 코드 예시

```java
// Member 클래스 (다대일에서 '다'에 해당)
@Entity
public class Member {
    @Id
    @Column(name = "MEMBER_ID")
    private Long id;

    @Column(name = "USERNAME")
    private String username;

    // @ManyToOne의 fetch 기본값은 FetchType.EAGER
    @ManyToOne
    @JoinColumn(name = "TEAM_ID")
    private Team team;

    // Getter, Setter, Constructor...
}

// Team 클래스 (다대일에서 '일'에 해당)
@Entity
public class Team {
    @Id
    @Column(name = "TEAM_ID")
    private Long id;

    @Column(name = "NAME")
    private String name;

    // Getter, Setter, Constructor
}
```

### 실행 코드 및 발생 쿼리

```java
Member member = entityManager.find(Member.class, 0L);
Team team = member.getTeam();
System.out.println("회원 이름 : " + member.getUsername());
System.out.println("팀 이름 : " + team.getName());
```

```sql
-- 즉시 로딩이므로 한 번의 JOIN 쿼리로 처리
SELECT m.MEMBER_ID, m.TEAM_ID, m.USERNAME, t.TEAM_ID, t.NAME
FROM Member m
LEFT OUTER JOIN Team t ON m.TEAM_ID = t.TEAM_ID
WHERE m.MEMBER_ID = 0;
```

대부분의 JPA 구현체(Hibernate)는 즉시 로딩을 최적화하기 위해 가능하면 **JOIN 쿼리**를 사용한다.
단, 이는 `em.find()` 등을 통한 단건 조회 시의 동작이며, **JPQL을 사용할 경우 N+1 문제가 발생할 수 있다.**

## 3. FetchType.LAZY (지연 로딩)

### 개념

연관된 엔티티를 **실제로 사용하는 시점(필요에 따라 요청될 때)** 에 비로소 데이터베이스에서 로드하는 방식이다.
처음에는 프록시 객체를 반환하고, 실제 필드에 접근하는 순간 DB 쿼리를 실행한다.

### 지연 로딩의 내부 원리

1. 엔티티 매니저가 엔티티를 로드할 때, JPA는 **프록시 객체(HibernateProxy)** 를 실제 엔티티 대신 반환한다.
2. 프록시 객체는 실제 엔티티와 동일한 인터페이스를 가진 껍데기로, **사용 시점까지는 실제 데이터를 로드하지 않는다.**
3. 프록시 객체의 메서드(getter 등)가 호출되는 시점에 데이터베이스에서 실제 데이터를 조회한다.
4. 한 번 초기화된 프록시 객체는 이후부터 실제 데이터를 사용한다. (초기화는 한 번만 발생)

### 엔티티 코드 예시

```java
// Member 클래스 (다대일에서 '다'에 해당)
@Entity
public class Member {
    @Id
    @Column(name = "MEMBER_ID")
    private Long id;

    @Column(name = "USERNAME")
    private String username;

    @ManyToOne(fetch = FetchType.LAZY)  // 명시적으로 LAZY 설정
    @JoinColumn(name = "TEAM_ID")
    private Team team;

    // Getter, Setter, Constructor...
}
```

### 실행 코드 및 발생 쿼리

```java
System.out.println("1. 회원 객체 조회");
Member member = entityManager.find(Member.class, 0L);

System.out.println("2. 회원 이름 조회");
System.out.println("회원 이름 : " + member.getUsername());

System.out.println("3. 팀 객체 조회");
Team team = member.getTeam(); // 프록시 객체 반환 (DB 조회 없음)
System.out.println(team.getClass()); // → HibernateProxy$...

System.out.println("4. 팀 이름 조회");
System.out.println("팀 이름 : " + team.getName()); // 이 시점에 DB 조회 발생 (프록시 초기화)
System.out.println(team.getClass()); // → 여전히 HibernateProxy$... (프록시 래퍼는 유지됨)
```

**출력 로그:**

```
1. 회원 객체 조회
-- 쿼리 실행 --
SELECT m.MEMBER_ID, m.TEAM_ID, m.USERNAME
FROM Member m
WHERE m.MEMBER_ID = 0;

2. 회원 이름 조회
회원 이름 : 회원1

3. 팀 객체 조회
class TIL.jpa.Domain.Team$HibernateProxy$hbJa4wRu  ← 프록시 객체

4. 팀 이름 조회
-- 쿼리 실행 (프록시 초기화) --
SELECT t.TEAM_ID, t.NAME
FROM Team t
WHERE t.TEAM_ID = 0;

팀 이름 : 팀1
class TIL.jpa.Domain.Team$HibernateProxy$hbJa4wRu  ← 프록시 래퍼는 유지
```

> **참고**: 프록시 초기화 이후에도 `getClass()`는 여전히 HibernateProxy를 반환한다. 프록시 래퍼 자체가 교체되는 것이 아니라, 내부적으로 실제 엔티티의 데이터를 참조하게 되는 방식으로 동작하기 때문이다.

## 4. JPQL

### JPQL이란?

JPA는 SQL을 추상화한 **JPQL(Java Persistence Query Language)** 이라는 객체 지향 쿼리 언어를 제공한다.
테이블을 대상으로 쿼리하는 SQL과 달리, **엔티티 객체를 대상으로 쿼리**하기 때문에 특정 데이터베이스(MySQL, PostgreSQL 등)에 종속되지 않고 이식성 높은 코드를 작성할 수 있다. JPQL은 최종적으로 SQL로 변환되어 실행된다.

### SQL vs JPQL 비교

```sql
-- SQL: 테이블과 컬럼 기준
SELECT * FROM user WHERE name = 'John';
```

```java
// JPQL: 엔티티 클래스와 필드 기준
SELECT u FROM User u WHERE u.name = 'John'
```

### 기본 JPQL 문법

**전체 엔티티 조회 (Select All)**

```java
@Query("SELECT u FROM User u")
List<User> findAllUsers();
```

**특정 필드만 조회**

```java
// 결과는 Object[] 배열의 리스트로 반환
@Query("SELECT u.name, u.email FROM User u")
List<Object[]> findUserNamesAndEmails();
```

**WHERE 절과 파라미터 바인딩**

```java
@Query("SELECT u FROM User u WHERE u.name = :name")
List<User> findUsersByName(@Param("name") String name);
```

**정렬 (ORDER BY)**

```java
// 단일 필드 정렬
@Query("SELECT u FROM User u ORDER BY u.name ASC")
List<User> findUsersSortedByNameAsc();

// 다중 필드 정렬
@Query("SELECT u FROM User u ORDER BY u.name DESC, u.email ASC")
List<User> findUsersSortedByNameDescAndEmailAsc();
```

**LIKE / BETWEEN / IN 절**

```java
// LIKE: 패턴 매칭
@Query("SELECT u FROM User u WHERE u.name LIKE :namePattern")
List<User> findUsersByNameLike(@Param("namePattern") String namePattern);

// BETWEEN: 범위 검색 (양 끝값 포함)
@Query("SELECT u FROM User u WHERE u.age BETWEEN :startAge AND :endAge")
List<User> findUsersByAgeRange(@Param("startAge") int startAge, @Param("endAge") int endAge);

// IN: 리스트 내 값 매칭
@Query("SELECT u FROM User u WHERE u.name IN :names")
List<User> findUsersByNames(@Param("names") List<String> names);
```

**집계 함수 및 GROUP BY**

```java
// 개수
@Query("SELECT COUNT(u) FROM User u")
long countUsers();

// 평균
@Query("SELECT AVG(u.age) FROM User u")
double findAverageAge();

// 그룹화
@Query("SELECT u.age, COUNT(u) FROM User u GROUP BY u.age")
List<Object[]> countUsersByAge();
```

### JPQL 성능 고려사항

- **페치 전략(Fetch Strategy)**: 연관 관계가 있는 데이터 조회 시 무분별한 JOIN은 대량의 데이터 로딩을 유발한다. N+1 문제를 방지하려면 **`JOIN FETCH`** 문법을 적극 활용해야 한다.
- **페이징 처리(Pagination)**: 대용량 데이터를 한 번에 메모리에 올리면 과부하가 발생할 수 있다. JPA의 `Pageable` 객체를 사용해 페이징 처리를 해야 한다.
- **인덱스(Index)**: WHERE 절이나 ORDER BY에 자주 사용되는 컬럼에는 인덱스가 생성되어 있는지 확인해야 한다.

## 5. 프록시 객체 (Proxy Object)

### 프록시 객체란?

프록시(Proxy)는 JPA와 Hibernate에서 사용되는 개념으로, **실제 엔티티 객체를 대리하여 동작하는 가짜 객체**이다.
프록시를 통해 데이터베이스 조회를 지연시키고 성능을 최적화할 수 있다.

지연 로딩을 이용하기 위해서는 실제 객체 대신 DB 조회를 지연할 수 있는 가짜 객체가 필요한데, 이것이 바로 프록시 객체이다. 진짜 객체를 즉시 가져오는 것이 아니라, 데이터베이스 접근 권한을 위임받은 가짜 객체를 자리에 두는 방식이다.

### 프록시 객체의 주요 특징

**1. 데이터베이스 조회 지연**

- `em.find()`: 데이터베이스를 즉시 조회하여 **실제 엔티티 객체**를 반환한다.
- `em.getReference()`: 데이터베이스 조회를 미루고 **프록시 객체**를 반환한다. 실제 데이터는 프록시 객체의 메서드가 처음 호출될 때 조회된다.

**2. 프록시 객체의 구조**

- 프록시 객체는 **실제 엔티티 클래스를 상속받아** 만들어진다.
- 프록시 객체는 실제 엔티티 객체를 참조하는 방식으로 동작한다.
- 프록시 객체는 **실제 객체의 식별자(primary key) 값**을 가지고 있다. 이를 통해 연관 관계를 설정하며, 초기화 시 해당 식별자로 DB 조회를 수행한다.

**3. 프록시 객체의 초기화**

- 프록시 객체는 **처음 사용할 때 한 번만 초기화**된다.
- 초기화 시 프록시 객체는 실제 엔티티 객체의 데이터에 접근할 수 있게 된다.
- 단, 일대다 관계에서는 프록시 객체가 여러 개이므로, 프록시 객체별로 한 번씩 초기화가 일어나면 결국 **여러 번의 DB 접근**이 발생하게 된다. → N+1 문제의 원인

**4. 프록시 객체와 타입 비교**

- 프록시 객체는 실제 엔티티와 동일한 인터페이스를 구현하므로 대부분의 경우 구분할 필요가 없다.
- 단, 타입 비교 시 `==` 연산자는 실패할 수 있으므로 **`instanceof` 연산자를 사용**해야 한다.

```java
// 잘못된 타입 비교
member.getTeam() == anotherTeam  // ❌ 프록시 객체이면 false가 나올 수 있음

// 올바른 타입 비교
member.getTeam() instanceof Team  // ✅
```

**5. 영속성 컨텍스트와 프록시**

- 영속성 컨텍스트에 이미 로드된 엔티티가 있다면, `em.getReference()`는 프록시가 아닌 **실제 엔티티 객체**를 반환한다.
- **준영속 상태**에서 프록시를 초기화하면 `org.hibernate.LazyInitializationException` 예외가 발생한다.

### 프록시 객체 작동 방식 예시

```java
EntityManager em = emf.createEntityManager();
EntityTransaction tx = em.getTransaction();

tx.begin();

// 1단계: 프록시 객체 생성
Member member = em.find(Member.class, 1L);
Team team = member.getTeam(); // 프록시 객체 반환 (DB 조회 없음)

// 2단계: 초기화 전 상태
// team은 프록시 객체 상태. 아직 실제 데이터는 없음.

// 3단계: 실제 데이터 접근 → 프록시 초기화
String teamName = team.getName(); // 이 시점에 DB 쿼리 실행
System.out.println("Team Name: " + teamName);

// 4단계: 초기화 후 상태
// 이후 team에 대한 접근은 이미 로드된 데이터를 사용

tx.commit();
em.close();
```

### 프록시 초기화 확인 및 강제 초기화

```java
// 초기화 여부 확인
PersistenceUnitUtil persistenceUnitUtil = emf.getPersistenceUnitUtil();
boolean isInitialized = persistenceUnitUtil.isLoaded(member.getTeam());

// 강제 초기화
org.hibernate.Hibernate.initialize(member.getTeam());
```

### 프록시의 장단점

| | 내용 |
|---|---|
| **장점** | 데이터베이스 조회를 지연시켜 성능 최적화 가능 |
| **장점** | 초기에는 프록시 객체만 메모리에 로드되므로 메모리 사용량 절감 |
| **단점** | 프록시 초기화 시 성능 저하 발생 가능 |
| **단점** | 잘못된 사용으로 `LazyInitializationException` 예외 발생 가능 |
| **단점** | 프록시 객체와 실제 객체를 혼동할 수 있어 코드 작성 시 주의 필요 |

## 6. N+1 문제

### N+1 문제란?

N+1 문제는 JPA에서 자주 발생하는 성능 저하 이슈로, **하나의 쿼리를 실행했을 때 예상치 못하게 추가적인 N개의 쿼리가 실행되는 현상**이다.

- 1: 대상 엔티티 목록을 조회하는 최초 쿼리 1회
- N: 조회된 엔티티 수만큼 연관 엔티티를 개별 조회하는 N회

### 발생 예시

```java
@Entity
public class Member {
    @Id @GeneratedValue
    private Long id;
    private String name;

    @OneToMany(mappedBy = "member", fetch = FetchType.EAGER)
    private List<Order> orders = new ArrayList<>();
}

@Entity
public class Order {
    @Id @GeneratedValue
    private Long id;

    @ManyToOne
    private Member member;
    private String productName;
}

// 서비스 코드
List<Member> members = memberRepository.findAll();
// → Member 조회 쿼리 1회 + 각 Member의 Order 조회 쿼리 N회 실행
```

### EAGER에서의 N+1 발생 원인

`FetchType.EAGER`로 설정된 연관 관계는 **조회 시점에 함께 로딩**해야 한다.
그런데 JPQL은 작성된 쿼리를 그대로 SQL로 변환하여 실행하기 때문에, 처음에는 `SELECT * FROM member`만 실행된다.
이후 JPA가 EAGER 설정을 감지하고 각 Member의 연관 엔티티(Order)를 **별도의 쿼리로 N번 추가 조회**하게 된다.

> 더 나아가, 다대다(`@ManyToMany`) 관계에서 EAGER 로딩을 하게 되면 무한 루프에 빠져 프로세스가 무한으로 돌아가는 경우도 있다.

### LAZY에서도 N+1은 발생한다

LAZY로 설정하면 자동으로 연관 엔티티를 모두 불러오지 않아 위 문제를 방지할 수 있다.
그러나 **LAZY도 완전히 안전한 것은 아니다.**

```java
List<Member> members = memberRepository.findAll(); // Member 조회 쿼리 1회
for (Member member : members) {
    System.out.println(member.getOrders().size()); // 각 Member마다 Order 조회 쿼리 실행 → N회
}
```

반복문 내에서 프록시 객체를 초기화(연관 데이터에 접근)하면 그 수만큼 쿼리가 발생하여 N+1 문제가 동일하게 나타난다.
또한, 트랜잭션 범위 밖에서 접근하면 `LazyInitializationException`이 발생한다.

### 해결 방안

**① Fetch Join 사용 (가장 권장)**

```java
@Query("SELECT m FROM Member m JOIN FETCH m.orders")
List<Member> findAllWithOrder();
```

SQL 조인을 사용해 연관 엔티티를 **한 번의 쿼리로** 함께 조회한다.
중복 데이터가 발생할 경우 `DISTINCT`를 함께 사용하면 DB 탐색 성능이 향상된다.

```java
@Query("SELECT DISTINCT m FROM Member m JOIN FETCH m.orders")
List<Member> findAllWithOrderDistinct();
```

**② EntityGraph 어노테이션 활용**

```java
@EntityGraph(attributePaths = {"orders"})
@Query("SELECT m FROM Member m")
List<Member> findAllWithOrder();
```

JPA 2.1부터 지원되며, 쿼리 수행 시 함께 로딩할 엔티티를 지정하는 방식이다. Fetch Join과 유사하지만 어노테이션으로 선언적으로 처리할 수 있다.

**③ Batch Size 설정**

```properties
# application.properties
spring.jpa.properties.hibernate.default_batch_fetch_size=100
```

지정한 크기만큼만 한 번에 조회한다. N+1 문제를 **1+1 문제로 완화**시킬 수 있다.
네트워크 장애 시 Retry 메커니즘을 도입하기도 쉬워진다.

> **각 방법 비교 요약**
> - Fetch Join: 쿼리 1번으로 해결. 페이징과 함께 쓰면 메모리에서 페이징 처리되어 주의 필요.
> - EntityGraph: 선언적이고 간결하지만 복잡한 쿼리에서는 한계가 있음.
> - Batch Size: 완전한 해결은 아니지만 기존 코드 변경 없이 적용 가능하며 페이징과 함께 사용 가능.

## 7. 결론: 주제 정리

### FetchType.EAGER에서의 N+1 발생 과정 및 프록시 역할

**동작 과정:**

1. JPQL이 SQL로 번역되어 데이터베이스로 전달된다.
2. 조회 대상 엔티티 목록을 가져오는 **1개의 쿼리**가 실행된다.
3. 즉시 로딩 전략으로 인해 영속성 컨텍스트는 가져온 엔티티 수(N)만큼 연관된 엔티티를 추가로 조회하는 **N개의 쿼리**를 연달아 실행한다.
4. 결과적으로 총 **1 + N번**의 쿼리가 발생한다.

**프록시 객체의 역할:**

EAGER 환경에서는 **조회 시점부터 실제 엔티티를 직접 가져오기 때문에 프록시 객체를 사용하지 않는다.**
대신 JPA가 연관 엔티티를 즉시 가져오기 위해 별도의 SELECT 쿼리를 N번 실행한다. 이로 인해 연관 관계가 복잡해질수록 불필요한 조인이 발생하며, 예측하기 어려운 쿼리가 쏟아지게 된다.

---

### FetchType.LAZY에서의 N+1 발생 과정 및 프록시 역할

**동작 과정:**

1. JPQL을 실행하면 조회 대상 엔티티만 가져오는 **1개의 쿼리**가 실행된다.
2. 이때 연관된 엔티티는 로드되지 않고, **프록시 객체가 자리를 채운다.**
3. 이후 코드에서 연관 데이터를 실제로 호출(getter 호출 등)하면, 해당 연관 엔티티를 가져오는 쿼리가 추가로 실행된다.
4. 조회된 데이터가 N개일 때 반복문 내에서 개별적으로 프록시를 초기화하면 총 **N개의 추가 쿼리**가 발생하여 N+1 문제가 나타난다.

**프록시 객체의 역할:**

최초 JPQL 조회 시 연관 필드에는 **프록시 객체가 채워진다.**
클라이언트가 연관 데이터를 실제 사용하는 시점(초기화 시점)에 프록시 객체가 영속성 컨텍스트에 실제 엔티티 조회를 요청한다.

이 프록시 메커니즘 덕분에 **불필요한 초기 조회를 방지**할 수 있다. 그러나 리스트 순회 등을 통해 여러 개의 연관 프록시를 초기화할 때, 결국 쿼리가 개별적으로 N번 나가게 되어 N+1 문제가 발생한다.

---

### EAGER vs LAZY N+1 비교 요약

| 구분 | FetchType.EAGER | FetchType.LAZY |
|---|---|---|
| 연관 엔티티 로딩 시점 | 부모 엔티티 조회 즉시 | 연관 데이터 실제 접근 시 |
| 프록시 사용 여부 | 사용 안 함 (실제 엔티티 반환) | 사용 (HibernateProxy 반환) |
| N+1 발생 시점 | JPQL 실행 직후 (자동으로 N번 추가 쿼리) | 프록시 초기화 시 (getter 호출 등) |
| N+1 발생 인지 난이도 | 어려움 (자동으로 쿼리 발생) | 비교적 명확 (접근 시점에 발생) |
| 실무 권장 여부 | ❌ 비권장 | ✅ 권장 (단, Fetch Join 병행) |

---

### 실무 적용 권장 사항

1. **모든 연관 관계를 LAZY로 설정**: `@ManyToOne`, `@OneToOne`도 명시적으로 `fetch = FetchType.LAZY`로 설정한다.
2. **필요한 경우에만 Fetch Join 적용**: 실제로 연관 데이터가 필요한 쿼리에서만 `JOIN FETCH`를 사용한다.
3. **성능 테스트 필수**: 대량 데이터 환경에서 반드시 테스트한다.
4. **쿼리 로그 모니터링**: `spring.jpa.show-sql=true` 설정으로 실행 쿼리를 확인한다.
5. **페이징 처리 주의**: Fetch Join과 페이징(`Pageable`)을 함께 사용하면 메모리에서 페이징 처리되므로, 이 경우에는 Batch Size 설정을 고려한다.
