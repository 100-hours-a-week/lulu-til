**서버를 직접 관리하지 않고 코드만 실행할 수 있는 서버리스 컴퓨팅 서비스**

##### 서버리스란

> ⚠️ 서버가 없는 게 아니다. 서버가 사라진 게 아니라, **서버 운영의 상당 부분을 AWS가 책임지는 방식**이다.

| EC2를 만들어서 사용 | Lambda 방식 |
|---|---|
| 인스턴스 만들고 | 함수 코드 작성 |
| 운영체제 관리 | 이벤트 트리거 연결 |
| 서버 실행 | IAM 권한 설정 |
| 패치 | |
| 용량 조절 | |
| (모두 직접 관리) | (AWS가 알아서 관리) |

##### EC2 vs Lambda

**EC2 — 서버를 켜두는 방식**
- 사용자가 요청하면, 이미 켜져 있는 서버가 응답한다
- 사용자 요청이 없어도 서버는 계속 켜져 있다
- 보통 시간 단위로 과금된다 = 켜져 있는 동안 계속 돈을 낸다

| 장점 | 단점 |
|---|---|
| 서버가 이미 켜져 있으니 바로 응답 | 요청이 없어도 비용이 발생 |
| 서버 내부 상태(디스크, 파일)를 상대적으로 관리하기 쉬움 | |
| 필요한 라이브러리를 자유롭게 사용할 수 있음 | |

**Lambda — 이벤트가 발생할 때만 실행되는 방식**
- 지정한 특정 이벤트가 발생할 때만 람다가 실행된다
- 일정 시간이 지나면 사라진다

| 장점 | 단점 |
|---|---|
| 실행 시간만큼만 과금 (함수=코드가 돌아가는 시간만큼만 비용 지불) | Cold Start |
| 확장성이 좋음 (코드 단위의 실행) | 15분 실행 시간 제약 |
| | 동시성 문제 |

##### Lambda의 실행 방식

```
이벤트 발생 → Lambda → Handler 실행 → 함수 코드 → 필요한 작업 수행 (DB/S3/외부 API)
                                                    ← 결과 반환
                       ← 실행 완료
```

**이벤트 (Event)**

람다는 다양한 이벤트로 실행될 수 있다.

| 이벤트 소스 | Lambda 사용 예시 |
|---|---|
| API Gateway | HTTP API 요청 처리 |
| S3 | 이미지 업로드 후 썸네일 생성 |
| EventBridge | 매일 오전 9시 정기 작업 |
| SQS | 비동기 메시지 처리 |
| DynamoDB Streams | 데이터 변경 감지 후 후속 처리 |
| CloudWatch Alarm | 장애 발생 시 자동 복구 작업 |
| Cognito | 회원가입 전후 사용자 처리 |
| ALB | HTTP 요청을 Lambda로 전달 |

##### 람다 실행 순서

```
이벤트 소스 → AWS Lambda → Lambda Handler → DB/S3/외부 API
            (이벤트 전달)   (실행 환경 준비)
                          (Init 초기화)
                          (Handler 실행) → (필요한 작업 수행)
                                        ← (결과 반환)
            ← (실행 결과 반환)
← (응답 또는 처리 완료)
            (환경 재사용 또는 종료)
```

**1. 이벤트 발생**

API Gateway, S3 파일 업로드 등 다양한 트리거를 통해 발생.

**2. 람다가 실행 환경을 준비한다**

코드를 실행해야 하므로 컴퓨팅 자원이 필요하다. 람다가 실행할 **격리된 실행 공간**을 준비한다.

실행 환경에 포함되는 것들:
- 설정한 메모리, CPU 자원
- Runtime
- 함수 코드
- `/tmp` 임시 저장소 (EBS와 비슷한 역할)
- 환경 변수

**3. Init 단계**

초기화 코드들이 실행되는 단계.

a. 함수 코드 + Lambda Layer 다운로드
b. Runtime 초기화
c. 핸들러 밖에 작성한 코드들도 초기화

자바 코드 예시:

```java
public class Lambda {
    private String functionName;
    public Lambda(String name) {
        this.functionName = name;
    }
}
```

> 이 생성자처럼 핸들러 메서드 바깥에 작성된 코드는 **Init 단계**에서 실행된다.

**4. Invoke 단계**

초기화는 끝났고, 람다가 핸들러를 호출한다.

```java
public class GreetingHandler
        implements RequestHandler<Map<String, String>, Map<String, String>> {

    @Override
    public Map<String, String> handleRequest(
            Map<String, String> event,
            Context context
    ) {
        String name = event.getOrDefault("name", "Guest");

        return Map.of(
                "message", "Hello " + name
        );
    }
}
```

코드 블록 실행이 끝나면, 람다는 특정 대기 시간 동안 이 실행 환경을 유지한다.

##### Cold Start vs Warm Start

**Cold Start**
- 새 실행 환경을 만들고 Init 단계까지 수행한 뒤 요청을 처리하는 경우
- 실행 환경이 준비되어 있지 않아서, 환경을 준비하고 나서 핸들러를 호출
- 요청 처리 시간이 상대적으로 **길다**

**Warm Start**
- 기존에 만들어진 실행 환경을 재사용하는 경우
- Init 단계를 건너뛴다
- 상대적으로 요청 처리 시간이 **짧다**

> 💡 "그럼 그냥 실행 환경을 한 번 만들고 계속 켜두면 되는 거 아닌가?"
> → 그건 그냥 EC2다. 그만큼 과금이 된다. 그러지 않으려고 람다를 쓰는 것이다.

**Warmup 설정**

람다 안에 설정을 둘 수 있다. 특정 시간마다 헬스체크 요청을 보내서, 람다 입장에서는 요청이 계속 들어오는 것처럼 만들어 실행 환경이 꺼지지 않게 한다.

##### 람다는 언제 쓰는 게 좋을까?

**① 파일 업로드 처리**

파일 업로드는 S3에 저장되는데, S3 업로드 완료 트리거가 발생하면 람다가 실행된다.
- 프로필 사진 리사이징 (예: 10KB로)
- 이미지 형태 변환 (예: webp)
- 업로드 파일 바이러스 검사

**② 극초기 API**

> 맨 처음부터 유저도 없는데 굳이 EC2에 올려야 할까?

다만 사실 **API는 람다와 잘 맞지 않는 편**이다. 람다는 웜업 기능을 사용하지 않으면 자원을 잃고(Cold Start), API는 요청-응답 시간이 중요하기 때문에 이 둘은 트레이드오프 관계에 있다.

**③ 예약 작업 (배치 작업)**
- 매일 새벽, 임시 데이터 삭제
- 매주 월요일 리포트 만들어주기

**④ 비동기 작업 처리**

주문 생성 같은 경우, 주문 API에서는 응답을 바로 주고 큐에 주문 정보를 적재한다. 람다가 이 큐를 수신해서 이메일, 슬랙, 알림, 통계 갱신 같은 작업을 비동기로 처리한다.

##### 람다는 상태를 오래 저장하지 않는다

람다는 기본적으로 **짧은 작업**에 적합하다.

- 람다 함수가 실행되고 이벤트 처리가 끝나면, 특정 시간 대기 후 자원을 반납한다
- 저장 공간도 마찬가지다 — 실행 환경은 재사용될 수 있지만, 저장된 값은 언젠가 사라진다 (전역 변수, `/tmp` 폴더에 저장된 데이터 포함)
- 상태를 오래 저장해야 한다면 **외부 저장소**를 사용해야 한다

##### 람다 설정

**1. 메모리 설정**

람다에서는 **메모리가 늘어나면 CPU 할당도 자동으로 늘어난다.**

> 💡 "메모리를 낮게 설정하면 비용이 줄어드니까 좋은 거 아니야?"
> → 메모리를 낮추면 실행 시간이 길어지고, 결과적으로 비용이 더 나올 수도 있다.

**2. 타임아웃**

람다 함수가 최대 얼마 동안 실행될 수 있는지 정하는 설정.

- 디폴트: 3초
- 최대값: **900초(15분)**
- 람다 함수는 15분 이상 실행되지 않는다

작업에 따라 타임아웃을 잘 설정해야 한다.

| 작업 유형 | 권장 타임아웃 |
|---|---|
| 이미지 변환 작업 | 30초 |
| 외부 API 호출 | 10초 |
| 간단한 API | 3~10초 |

##### 실무에서 람다 잘 사용하기

**① 함수 하나에 하나의 책임만 둔다 (SRP)**

어떤 작업이 오래 걸려서 실패할지 모르기 때문에, 함수를 쪼개두는 것이 안전하다.

```
회원가입 Lambda
 → 사용자 저장

회원가입 이벤트
 → 이메일 발송 Lambda
 → 쿠폰 발급 Lambda
 → 통계 저장 Lambda
 → Slack 알림 Lambda
```

쪼개는 기준:
- 독립적인 실패가 가능해야 할 때
- 독립적인 처리가 필요한 경우

**② Cold Start 문제 대응하기**

- 애초에 실행 환경이 무거운 코드를 안 쓴다 (예: Spring Boot 기반 자바는 상대적으로 무거움)
- 의존 라이브러리를 줄인다 — 보통 Node.js나 Python은 상대적으로 실행 환경이 가볍다
- **프로비저닝(Provisioned Concurrency)**: 람다 실행 환경을 미리 N개 이상 준비해두는 기능
- **헬스체크 (Warmup)**: 프로비저닝보다 훨씬 원시적인 방식. EventBridge로 N초마다 람다 함수 안에 헬스체크 펑션을 만들어서 람다 환경이 죽지 않도록 유지

**③ 동시성 문제 해결**

람다는 한도까지 자동으로 확장된다.

```
A 요청 → 1번 람다 함수
B 요청 → 2번 람다 함수
C 요청 → 3번 람다 함수
```

계정 한도는 디폴트로 **1,000개**다.

> ⚠️ 람다가 RDS 연결을 요청한다고 가정해보자. 동시에 수많은 요청이 오면 DB 커넥션을 다 차지하게 되고, RDS는 커넥션을 전부 람다에게만 할당하면서 DB가 뻗는다. 그러면 WAS가 처리 불가능 상태가 된다.

**문제 해결 방식**: 중간에 **큐**를 둔다.
```
이벤트 → 현재 람다가 수행하는 함수 개수 확인
  ├─ 10개 미만이면 → 즉시 실행
  └─ 10개 이상이면 → 큐에 적재
```

**④ 장기간 연결해야 하는 요청 피하기**

람다는 실행 시간이 15분까지로 제한되어 있다.

예시 시나리오:
1. 최초 개발자가 람다로 실행할 수 있는 함수를 만든다 — "회원 탈퇴 30일 이전 사용자를 DB에서 삭제하는 함수"가 1만 명 처리 기준으로 15분 이내에 끝났다
2. 서비스가 성장하면서 탈퇴자가 많아진다
3. 람다 함수가 15분 제약에 걸려 **실행이 불가능해진다**

웹소켓, 파일 처리 용량이 큰 작업도 마찬가지로 람다와는 잘 맞지 않을 수 있다.

**⑤ 람다를 모든 백엔드 기능의 서버로 두면 안 된다**

람다가 편하긴 하지만 위에서 다룬 단점들(Cold Start, 15분 제약, 동시성 문제)이 있기 때문에, 모든 백엔드를 람다로만 구성하는 것은 위험하다.

#### | API Gateway

**클라이언트(웹, 모바일 등)와 백엔드 서비스(EC2 등) 사이에서 API 요청을 관리/변환/보호해주는 완전관리형 서비스**

##### 백엔드 서비스가 커지면 생기는 문제

처음에는 백엔드 하나가 모든 요청을 처리한다. 그런데 서비스가 커지면 백엔드가:
- API 입구 역할도 해야 하고
- 인증, CORS 같은 공통 문제도 해결해야 한다

즉, 백엔드는 본래 **비즈니스 로직을 처리**해야 하는데, **API의 입구 역할까지 함께 떠안게** 된다.

##### API Gateway란

```
웹/모바일 앱 → API Gateway → Lambda
                            → Spring Boot on EC2/ECS
                            → 외부 HTTP API
                            → 다른 AWS 서비스
```

API Gateway는 앞단에서 **CORS, 인증(JWT)** 등을 처리해주고, 설정된 규칙에 따라 **적절한 요청만 백엔드에 전달**한다.

```
클라이언트 → GET /products → API Gateway
                                ↓ 인증 / 요청 제한 / CORS 확인
                              GET /products 전달 → 백엔드
                                                  ← 상품 목록 응답
                              ← JSON 응답
클라이언트 ← JSON 응답
```

##### 굳이 API Gateway를 사용해야 하는 이유

**① API의 입구를 하나로 줄이기 위해서**

지금은 백엔드가 하나뿐이지만, 서비스가 커지면 하나의 서비스에 여러 개의 백엔드가 붙는다.

**② 인증·인가를 분리할 때**

여러 백엔드 코드가 각각 인증·인가 코드를 갖고 있으면, 인증·인가 정책이 바뀔 때마다 전부 수정해야 한다. API Gateway에 인증·인가 책임을 넘기면 이 부담을 줄일 수 있다.

**③ 요청 수 제한, API Key 컨트롤이 필요할 때**

외부에 API를 제공할 때, 사용자 등급별로 요청 수를 제한할 수 있다.
- 무료 사용자: 초당 5회
- 유료 사용자: 초당 50회

**④ CORS도 중앙 관리**

분리된 프로젝트마다 CORS 설정이 중복되는데, API Gateway에서 한 번에 관리할 수 있다.

**⑤ 람다 같은 AWS 서비스를 직접 호출할 수 있다**

##### API Gateway 유형

| API 유형 | 특징 | 대표 사용 사례 |
|---|---|---|
| **HTTP API** | 단순하고 비용 효율적인 HTTP API | 일반 REST API, Lambda 연동, 간단한 백엔드 |
| **REST API** | 기능이 많고 세밀한 제어 가능 | API Key, 요청 검증, Usage Plan, WAF, Private API |
| **WebSocket API** | 양방향 실시간 연결 | 채팅, 실시간 알림, 협업 서비스 |

> 💡 보통 **REST API** 유형을 사용한다.

##### API Gateway vs ELB

둘 다 요청을 받아서 백엔드로 전달하지만, **주된 목적이 다르다.**
- **API Gateway** ← API를 관리해주는 게 주 목적
- **ELB** ← 여러 서버에 트래픽을 분산시키는 게 주 목적

| 구분 | API Gateway | ALB |
|---|---|---|
| 핵심 목적 | API 관리·보안·제어 | 여러 서버에 트래픽 분산 |
| 주요 대상 | API 요청 | 웹·API 트래픽 전체 |
| 백엔드 연결 | Lambda, HTTP API, AWS 서비스 등 | EC2, ECS, IP 대상 등 |
| 인증·인가 | API 단위로 강력하게 지원 | 별도 구성 필요 |
| 요청 제한 | Throttling, Usage Plan 등 제공 | 핵심 기능 아님 |
| API Key | 지원 | 지원하지 않음 |
| CORS | API 수준에서 설정 가능 | 애플리케이션 또는 프록시에서 주로 처리 |
| 헬스 체크 | 백엔드 인스턴스 헬스 체크가 핵심은 아님 | Target Health Check가 핵심 |
| 서버 분산 | API 요청을 Integration으로 연결 | 여러 Target에 자동 분산 |
| 대표 사용 사례 | Lambda API, 공개 API, 외부 파트너 API | EC2/ECS 기반 웹 서비스, Spring Boot 다중 서버 |

##### 어떤 걸 사용할까?

| 상황 | 우선 검토 |
|---|---|
| Spring Boot 서버를 EC2 여러 대로 운영 | ALB |
| ECS/EKS에서 여러 컨테이너에 요청 분산 | ALB |
| Lambda 기반 서버리스 API | API Gateway |
| 외부 파트너에 API 제공 | API Gateway |
| API Key, Usage Plan, 클라이언트별 제한 필요 | API Gateway |
| API 요청을 세밀하게 인증·통제해야 함 | API Gateway |
| URL 경로 기준으로 여러 EC2 서비스에 분기 | ALB |
| 서버 상태를 검사해 장애 서버를 자동 제외해야 함 | ALB |
| API와 일반 웹 트래픽을 함께 다중 서버에 전달 | ALB |

##### 이 둘은 경쟁 관계인가?

> 그렇지는 않다. **함께 조합해서 사용**할 수 있다.

```
클라이언트 → API Gateway → Internal ALB → Spring Boot 1
                                         → Spring Boot 2
```

API Gateway가 인증·요청 관리를 담당하고, 내부적으로는 Internal ALB가 여러 Spring Boot 서버로 트래픽을 분산하는 구조다.

---

#### | CloudTrail

**AWS 계정에서 발생하는 모든 API 호출과 콘솔 활동을 자동으로 기록하고 추적하는 서비스**

쉽게 말해, **누가, 언제, 어떤 서비스에 어떤 요청을 했는지**에 대한 로그를 남겨주는 감사 및 추적 도구다. 계정 내에서 일어난 모든 활동의 행위 이력을 확인할 수 있다.

**사용 이유**: 보안 감사, 컴플라이언스, 문제 해결을 위해 AWS 계정의 활동을 추적하고 분석하기 위해 사용된다.

---

#### | CloudWatch

**AWS 리소스와 애플리케이션에서 발생하는 로그, 지표, 이벤트를 수집/모니터링하는 통합 관제 서비스**

##### 왜 필요한가

서비스를 운영하다 보면 다음과 같은 일들이 생긴다.

- EC2 CPU가 갑자기 높아짐
- 디스크가 가득 참
- ALB의 5xx 오류가 증가함
- RDS 연결 수가 급증함
- 특정 API만 느려짐
- 배포 이후 오류가 늘어남

##### 문제를 언제 인지하는가

CloudWatch가 없으면 보통 다음과 같은 흐름이 된다.

```
배포 → 사용자에게 전달 → 사용자 리포트("이거 안 되는데요?") → 개발자가 확인 → "어, 큰일났다.."
```

즉, **후행적으로 보고받는 형식**이 된다. CloudWatch는 이런 문제들을 사용자가 알아차리기 **전에** 먼저 감지하려는 목적으로 사용한다.

##### DevOps에서 모니터링이 왜 중요한가

```
개발 → 배포 → 운영 → 모니터링 → 개선 → (다시 개발로)
```

| 확인 질문 | CloudWatch 사용 |
|---|---|
| 지금 서비스가 정상인가? | 대시보드와 메트릭 확인 |
| 오류가 늘었는가? | ALB 5xx, 애플리케이션 ERROR 로그 확인 |
| 어느 서버가 느린가? | CPU, 메모리, 응답 시간 확인 |
| DB가 병목인가? | RDS CPU, 연결 수, 지연 시간 확인 |
| 배포 후 문제가 생겼는가? | 배포 전후 오류율·응답 시간 비교 |
| 운영자가 지금 대응해야 하는가? | Alarm 알림 |

##### CloudWatch의 핵심 기능

| 기능 | 역할 | 예시 |
|---|---|---|
| **Metrics** | 시간에 따른 숫자 지표 확인 | CPU, 요청 수, 오류 수 |
| **Logs** | 시스템과 애플리케이션 기록 저장 | Spring ERROR 로그, Nginx 로그 |
| **Alarms** | 조건 충족 시 알림 또는 자동 대응 | CPU 80% 초과 시 이메일 |
| **Dashboards** | 여러 지표를 한 화면에서 확인 | EC2·ALB·RDS 상태 한눈에 보기 |
| **Logs Insights** | 로그를 쿼리로 검색·분석 | 5xx 오류가 많은 API 찾기 |

**1. Metrics**

시간에 따라 변화하는 숫자 데이터. 예를 들어 EC2 CPU 사용률이 높아졌다면:
- 서버에 트래픽이 많아졌을 수도 있고
- 애플리케이션이 비효율적으로 짜여 있을 수도 있다 (N+1 쿼리, Full Table Scan 등)

자주 보는 메트릭:

| 서비스 | 메트릭 | 의미 |
|---|---|---|
| EC2 | `CPUUtilization` | CPU 사용률 |
| EC2 | `StatusCheckFailed` | 인스턴스 또는 네트워크 상태 문제 |
| ALB | `RequestCount` | 요청 수 |
| ALB | `HTTPCode_Target_5XX_Count` | 백엔드 서버 5xx 오류 |
| ALB | `TargetResponseTime` | 백엔드 응답 시간 |
| ALB | `HealthyHostCount` | 정상 Target 수 |
| RDS | `DatabaseConnections` | DB 연결 수 |
| RDS | `FreeStorageSpace` | 남은 DB 스토리지 |
| ElastiCache | `Evictions` | 메모리 부족으로 제거된 캐시 수 |
| ElastiCache | `CacheHits`, `CacheMisses` | 캐시 효율 |

**2. Logs**

로그는 "왜 이 일이 생겼는가"를 기록한다. 여러 서비스의 로그를 **중앙집중화**해서 한 곳에 모아두면 확인하기 좋다.

**3. Alarms**

특정 조건이 일정 시간 유지될 때, 운영자에게 알림을 보내는 기능.

##### 옵저빌리티(Observability)란

아래 질문들에 답할 수 있다면 **옵저빌리티가 높은** 시스템이다.

- 지금 서비스는 정상인가?
- 어떤 기능이 실패하고 있는가?
- 언제부터 문제가 시작됐는가?
- 어느 서버 또는 어느 API가 문제인가?
- 사용자는 얼마나 영향을 받고 있는가?
- DB, Redis, 외부 API 중 어디가 병목인가?
- 최근 배포와 관련이 있는가?

##### 실무에서 옵저빌리티를 높이는 방법

**① 로그를 구조화한다**

```json
{
  "timestamp": "2026-06-25T10:12:20+09:00",
  "level": "ERROR",
  "service": "community-api",
  "traceId": "ab12cd34",
  "path": "/api/posts",
  "status": 500,
  "message": "Database connection timeout"
}
```

단순 텍스트 로그 대신 **JSON 형태로 구조화**하면, `traceId`로 요청을 추적하거나 `service`/`status` 같은 필드로 검색·집계하기가 훨씬 쉬워진다.

**② 핵심 API 응답 시간, 오류율을 수집한다**

핵심 API의 응답 시간과 오류율을 지속적으로 수집해두면, 위에서 나열한 "옵저빌리티 질문"들에 빠르게 답할 수 있는 기반이 된다.

### 🗨️ 오늘의 회고

- 오늘은 컴퓨팅(Lambda), 게이트웨이(API Gateway), 관제(CloudTrail, CloudWatch) 영역을 배웠다. 어제 배운 ELB·CloudFront와 묶어서 보니, AWS 서비스들이 "요청을 어떻게 받고, 어디로 보내고, 무슨 일이 일어났는지 어떻게 알 수 있는가"라는 세 가지 축으로 정리되는 느낌이다.
- EC2와 Lambda를 "서버를 켜두는 방식 vs 이벤트가 있을 때만 실행되는 방식"으로 대비한 설명이 가장 명확하게 다가왔다. 결국 핵심은 **상시 가동 비용 vs 콜드 스타트 지연**이라는 트레이드오프였다.
- 람다의 15분 타임아웃 제약을 "처음엔 1만 명 처리가 15분 안에 끝났는데, 서비스가 성장하면서 탈퇴자가 늘어 실행이 불가능해진다"는 시나리오로 설명한 게 인상적이었다. 초기 설계가 영원히 맞는 게 아니라는 점을 다시 느꼈다.
- 람다의 동시성 문제(RDS 커넥션 고갈) 해결법으로 "중간에 큐를 둔다"는 패턴이 어제 배운 ElastiCache의 캐시 스탬피드 해법(TTL 지터)과 본질적으로 비슷하다고 느꼈다. 둘 다 "한꺼번에 몰리는 요청을 시간적으로 분산시킨다"는 원리였다.
- API Gateway와 ALB를 처음엔 경쟁 관계로 생각했는데, 둘의 목적이 다르다는 걸 이해하니(API 관리 vs 트래픽 분산) `API Gateway → Internal ALB → Spring Boot`처럼 함께 조합해서 쓰는 구조가 자연스럽게 이해됐다. 어제 CloudFront와 ALB의 관계와 똑같은 패턴이었다.
- CloudWatch 부분에서 "CloudWatch가 없으면 사용자 리포트를 받고 나서야 문제를 안다"는 설명이 가장 와닿았다. 모니터링은 있으면 좋은 것이 아니라, 운영 단계에서는 사실상 필수라는 걸 다시 깨달았다.
- 로그를 JSON으로 구조화하고 `traceId`를 남기는 습관이 작은 차이지만 옵저빌리티에 큰 영향을 준다는 걸 배웠다. 지금 진행 중인 커뮤니티 백엔드 과제에도 바로 적용해볼 만한 내용이다.
- 클라우드 설계가 계속 늦어지고 있다. VPC, EC2, S3, RDS, ElastiCache, ELB, CloudFront, Lambda, API Gateway, CloudWatch까지 배운 내용을 한 장의 아키텍처로 통합하는 작업을 오늘은 반드시 마무리해야겠다.
