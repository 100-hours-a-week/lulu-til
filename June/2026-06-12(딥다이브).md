## 🖥️ 프로세스 (Process)

> 운영체제로부터 자원을 할당받아 **독립적으로 실행 중인 프로그램의 인스턴스**

- 디스크에 저장된 프로그램(정적)이 메모리에 올라와 실행되는 순간 프로세스(동적)가 된다
- 각 프로세스는 **독립된 메모리 공간**(Code, Data, Heap, Stack)을 가진다
- 운영체제는 **PCB(Process Control Block)** 를 통해 각 프로세스의 상태를 관리한다

> 💡 **쉽게 이해하기** : 크롬 브라우저 아이콘을 클릭하는 순간, 디스크의 실행 파일이 메모리로 올라오며 '프로세스'가 된다. 크롬을 두 번 실행하면 두 개의 독립된 프로세스가 생긴다. 이 둘은 서로의 메모리를 전혀 볼 수 없다.

### 프로세스 메모리 구조

| 영역 | 설명 | 예시 |
|---|---|---|
| **Code (Text)** | 실행할 프로그램 코드 (기계어) | `main()` 함수, 각종 메서드 |
| **Data** | 전역 변수, 정적 변수 저장 | `static int count = 0;` |
| **Heap** | 동적 할당 메모리 (런타임 시 크기 결정) | `new ArrayList<>()`, `malloc()` |
| **Stack** | 함수 호출 정보, 지역 변수, 매개변수 저장 | 메서드 호출 시 생성, 리턴 시 소멸 |

> 💡 **Stack과 Heap의 차이** : `int x = 5;` 처럼 함수 안에서 선언한 변수는 Stack에 올라가 함수가 끝나면 자동으로 사라진다. `new User()` 처럼 객체를 동적으로 생성하면 Heap에 올라가고, GC(가비지 컬렉터)가 정리할 때까지 유지된다.

### PCB (Process Control Block)

> 운영체제가 프로세스를 관리하기 위해 유지하는 **자료 구조**. 프로세스의 모든 정보를 담은 '신분증' 같은 역할

- **PID** (프로세스 식별자) : 각 프로세스의 고유 번호
- **프로세스 상태** : New / Ready / Running / Waiting / Terminated
- **PC** (Program Counter) : 다음에 실행할 명령어의 주소 (어디까지 실행했는지 기록)
- **CPU 레지스터 값** : 컨텍스트 스위칭 시 저장/복원에 사용
- **메모리 관리 정보** : 해당 프로세스의 가상 주소 공간 정보
- **I/O 상태 정보** : 열려 있는 파일, 소켓 목록

> 💡 **PCB가 중요한 이유** : CPU는 한 번에 하나의 프로세스만 실행한다. A 프로세스를 잠시 멈추고 B 프로세스를 실행한 뒤 다시 A로 돌아올 때, "어디까지 했더라?"를 기억하기 위해 PCB에 저장해두는 것이다.

### 프로세스 상태 전이

```
     생성(New)
        ↓ 메모리 할당 완료
     준비(Ready) ←──────────────────┐
        ↓ 스케줄러가 선택           │
     실행(Running) ──I/O 발생──→ 대기(Waiting)
        ↓ 실행 완료                 ↑
     종료(Terminated)         I/O 완료
```

### 프로세스 간 통신 (IPC)

프로세스는 메모리를 공유하지 않으므로, 데이터를 주고받으려면 **IPC(Inter-Process Communication)** 메커니즘이 필요하다. 이 자체가 이미 비용이다.

| IPC 방식 | 설명 | 특징 |
|---|---|---|
| **파이프(Pipe)** | 단방향 데이터 흐름 | 부모-자식 프로세스 간 주로 사용 |
| **소켓(Socket)** | 네트워크를 통한 통신 | 원격 프로세스 간 가능, 가장 범용적 |
| **공유 메모리(Shared Memory)** | 메모리 영역을 공유 | 빠르지만 동기화 필수 |
| **메시지 큐(Message Queue)** | 큐에 메시지를 넣고 꺼내는 방식 | 비동기 통신 가능 |
| **세마포어(Semaphore)** | 공유 자원 접근 제어 | 동기화 목적, 데이터 전달보다는 신호 전달 |

## 🧵 스레드 (Thread)

> 프로세스 내에서 **실행의 흐름을 담당하는 가장 작은 단위**

- 프로세스 안에서 실제 작업을 수행하는 주체
- 같은 프로세스 내의 스레드들은 **Code, Data, Heap 영역을 공유**하며, **Stack만 개별적으로** 가진다
- TCB(Thread Control Block) 를 통해 스레드 상태를 관리한다

> 💡 **왜 Stack만 독립적일까?** : Stack은 함수 호출 기록과 지역 변수를 저장한다. 각 스레드는 독립적인 실행 흐름을 가지므로 자신만의 Stack이 필요하다. 반면 Heap과 Data는 "공유 자원"으로 여러 스레드가 같은 객체에 접근하고 수정할 수 있다. 이 때문에 동기화 문제(Race Condition)가 발생한다.

### 스레드 메모리 구조

```
[ 프로세스 ]
┌──────────────────────────────────────────┐
│  Code (공유)  │  Data (공유)  │ Heap (공유) │ ← 스레드 간 공유 영역
├──────────────────────────────────────────┤
│   Stack (T1) │  Stack (T2)  │ Stack (T3) │ ← 각 스레드의 독립 영역
└──────────────────────────────────────────┘
```

### TCB (Thread Control Block)

> 운영체제가 스레드를 관리하기 위해 유지하는 자료 구조. PCB보다 가볍다.

- Thread ID
- 스레드 상태
- PC (Program Counter)
- 스택 포인터
- 레지스터 값
- **소속된 PCB에 대한 포인터** ← 스레드는 반드시 프로세스에 속한다

## ⚖️ 프로세스 vs 스레드 비교

| 구분 | 프로세스 | 스레드 |
|---|---|---|
| **메모리** | 독립된 메모리 공간 | 프로세스 내 메모리 공유 |
| **생성 비용** | 높음 (메모리 복제 등) | 낮음 (Stack만 추가) |
| **컨텍스트 스위칭** | 비용 큼 (메모리 공간 전환 + TLB 플러시) | 비용 작음 (Stack 포인터만 전환) |
| **통신 방식** | IPC (복잡하고 시스템 콜 필요) | 공유 메모리 직접 접근 (빠름) |
| **안정성** | 한 프로세스 장애가 다른 프로세스에 영향 없음 | 한 스레드 장애가 프로세스 전체에 영향 가능 |
| **동기화** | 필요 없음 (메모리 독립) | 필수 (공유 자원 경쟁 발생) |

> 💡 **비유** : 프로세스는 독립된 집, 스레드는 같은 집 안에 사는 룸메이트다. 룸메이트끼리는 거실(Heap, Data)을 공유하지만 각자의 방(Stack)은 따로 쓴다. 독립된 집(프로세스)끼리는 서로의 집에 들어갈 수 없으므로 편지(IPC)를 주고받아야 한다.

### 실무 선택 기준

| 상황 | 선택 | 이유 |
|---|---|---|
| 탭 하나가 죽어도 브라우저 전체가 죽으면 안 될 때 | **프로세스** | 독립된 메모리로 장애 격리 (크롬) |
| 동일 데이터를 여러 작업이 빠르게 공유해야 할 때 | **스레드** | 공유 메모리 직접 접근 |
| 마이크로서비스처럼 기능이 완전히 분리될 때 | **프로세스** | 독립 배포, 독립 확장 |
| 웹 서버에서 요청마다 가벼운 처리를 할 때 | **스레드** | 생성 비용이 프로세스보다 훨씬 낮음 |

## 🔄 컨텍스트 스위칭 (Context Switching)

> CPU가 현재 실행 중인 프로세스/스레드의 상태를 저장하고, 다음 프로세스/스레드의 상태를 복원하는 과정

운영체제는 여러 프로세스를 "동시에" 실행하는 것처럼 보이게 하기 위해 매우 빠르게 프로세스를 교체한다. 이 교체 작업이 컨텍스트 스위칭이다.

### 컨텍스트 스위칭 절차

```
① 현재 프로세스의 레지스터, PC, 스택 포인터 값을 PCB에 저장
② 운영체제 스케줄러가 다음에 실행할 프로세스 선택
③ 다음 프로세스의 PCB에서 레지스터, PC 값 복원
④ 다음 프로세스 실행 재개
```

### 컨텍스트 스위칭 비용 — 숫자로 보기

실제 측정 결과(Intel 계열 CPU 기준):

| 전환 유형 | 소요 시간 | 비고 |
|---|---|---|
| **프로세스 간 전환** | ~1,300 ~ 3,000 ns | TLB 플러시 포함 |
| **스레드 간 전환** | ~1,100 ~ 1,500 ns | 같은 주소 공간 |
| **Go 고루틴 전환** | ~170 ns | 커널 전환 없음 |

> 💡 1,200ns = 1.2μs 처럼 작아 보이지만, **초당 10만 번의 컨텍스트 스위칭**이 발생하면 CPU의 12%가 스위칭에만 쓰인다. 실제 작업을 하나도 안 하는데 말이다.

### 비용이 발생하는 이유 — 3가지 핵심

**① CPU 캐시(L1, L2, L3) 무효화**

CPU는 자주 사용하는 데이터를 캐시에 올려두고 빠르게 접근한다. 다른 프로세스로 전환하면 캐시에 쌓인 데이터가 쓸모없어지고(캐시 미스), 새 프로세스의 데이터를 다시 메모리에서 가져와야 한다.

```
[캐시가 뜨거울 때]   : L1 캐시 접근 ~1 ns  ✅
[컨텍스트 스위칭 후] : 메인 메모리 접근 ~100 ns ❌ (100배 느림)
```

**② TLB(Translation Lookaside Buffer) 플러시**

프로그램은 가상 주소를 사용하지만, CPU는 실제 물리 주소가 필요하다. 이 변환을 캐싱하는 것이 TLB다. 프로세스가 바뀌면 TLB를 비워야 하고, 이후 모든 메모리 접근에서 주소 변환을 다시 해야 한다. (스레드 간 전환은 같은 주소 공간이므로 TLB 플러시가 덜 발생한다.)

**③ CPU 파이프라인 플러시**

컨텍스트 스위칭 후 CPU 명령어 파이프라인이 비워지고 다시 채워지는 데 10~50 사이클이 소요된다.

## ⚙️ 시스템 콜 (System Call)

> 사용자 프로그램이 운영체제 커널의 기능을 사용하기 위해 **커널에 서비스를 요청하는 인터페이스**

### 왜 사용자 모드와 커널 모드를 분리할까?

만약 모든 프로그램이 하드웨어에 직접 접근할 수 있다면, 악성 프로그램이 다른 프로세스의 메모리를 읽거나 하드 디스크를 마음대로 초기화할 수 있다. 이를 막기 위해 CPU는 두 가지 실행 모드를 제공한다.

| 모드 | 설명 | 접근 가능 자원 | 비유 |
|---|---|---|---|
| **사용자 모드 (Ring 3)** | 일반 애플리케이션 실행 | 제한된 자원만 접근 | 일반 직원 |
| **커널 모드 (Ring 0)** | OS 커널 실행 | 모든 하드웨어/메모리 접근 가능 | 시스템 관리자 |

> 💡 현대 OS(Linux 포함)는 대부분 Ring 0(커널)과 Ring 3(사용자)의 두 단계만 사용한다. Ring 1, 2는 이론상 존재하지만 거의 사용되지 않는다.

### 시스템 콜의 종류

| 분류 | 대표 시스템 콜 | 설명 |
|---|---|---|
| **프로세스 제어** | `fork()`, `exec()`, `exit()`, `wait()` | 프로세스 생성/종료 |
| **파일 관리** | `open()`, `read()`, `write()`, `close()` | 파일 입출력 |
| **장치 관리** | `ioctl()`, `read()`, `write()` | 하드웨어 장치 제어 |
| **정보 유지** | `getpid()`, `alarm()`, `sleep()` | 시스템/프로세스 정보 조회 |
| **통신** | `socket()`, `send()`, `recv()`, `bind()` | 네트워크 통신 |
| **메모리 관리** | `mmap()`, `brk()` | 메모리 영역 할당/해제 |

### 커널 모드 전환 절차 — 단계별 상세

```
① 애플리케이션: read("file.txt") 호출
   → 사용자 모드에서 시스템 콜 번호(예: read = 0)와 인자를 레지스터에 저장

② CPU: 소프트웨어 인터럽트(trap) 발생
   → 현재 사용자 모드 레지스터 상태 저장
   → 커널 스택으로 스택 포인터 전환
   → CPU 권한 레벨: Ring 3 → Ring 0

③ 커널: 시스템 콜 번호 확인 후 대응 루틴 실행
   → 실제 파일 I/O 수행
   → 드라이버를 통해 디스크에 접근

④ CPU: 커널 모드 → 사용자 모드 복귀
   → 저장했던 레지스터 상태 복원
   → CPU 권한 레벨: Ring 0 → Ring 3

⑤ 애플리케이션: read() 결과(읽은 바이트 수) 반환받고 실행 재개
```

### 커널 모드 전환 비용 항목

한 번의 시스템 콜에는 **수백 ~ 수천 CPU 사이클**이 소비된다.

| 비용 항목 | 설명 |
|---|---|
| **모드 전환 오버헤드** | 사용자 모드 ↔ 커널 모드 전환 시 레지스터 저장/복원 |
| **커널 스택 전환** | 사용자 스택 → 커널 전용 스택으로 교체 |
| **캐시 미스** | 커널 코드/데이터로 전환으로 CPU 캐시 효율 저하 |
| **TLB 미스 가능성** | 주소 공간 전환으로 인한 TLB 적중률 저하 |
| **스케줄링 개입 가능성** | I/O 대기 시 컨텍스트 스위칭으로 이어질 수 있음 |

> 💡 **왜 이게 중요한가?** : 파일 1바이트를 읽는 `read()` 시스템 콜 하나가 수백 ns ~ 수μs를 소비한다. 비즈니스 로직 자체는 1ns도 안 걸릴 수 있는데, 커널 전환 비용이 수백 배 더 클 수 있다.

## 🏭 실무 애플리케이션에서의 성능 영향 — 구체적 사례

### 사례 1 : 웹 서버 — Apache vs Nginx, I/O 처리 방식의 차이

**문제 상황 — Apache의 전통적 방식 (Blocking I/O)**

Apache는 전통적으로 요청이 올 때마다 스레드(또는 프로세스)를 생성하는 방식으로 동작했다. 각 스레드가 `read()` 시스템 콜을 호출하면 I/O가 완료될 때까지 블로킹 상태가 된다.

```
[Blocking I/O 흐름 — Apache 방식]

Thread 1: [read() 시스콜 호출] ──── [커널에서 I/O 대기 중...] ─── [응답 반환]
Thread 2:                           [read() 시스콜 호출] ─── [커널 대기...] ─ [응답]
Thread 3:                                                    [read() ...]...

→ 동시 요청 10,000건 = 10,000개 스레드 필요
→ Apache Prefork 방식에서 10,000 동시 접속 처리 시 약 50~100 GB의 RAM이 필요
→ 수만 번의 컨텍스트 스위칭이 지속적으로 발생
```

**해결 방식 — Nginx의 Non-Blocking I/O + 이벤트 루프**

Nginx는 `epoll`(Linux), `kqueue`(macOS/BSD) 같은 OS 수준의 비동기 I/O 시스템 콜을 사용한다. 단 하나의 워커 스레드가 이벤트 루프를 돌며, 소켓이 "읽을 준비가 됐다"는 알림이 올 때만 처리한다.

```
[Non-Blocking I/O 흐름 — Nginx 방식]

Single Worker Thread:
  [epoll_wait()] → [이벤트A 처리] → [이벤트B 처리] → [이벤트C 처리] → ...
        ↑
   커널이 "이 소켓은 지금 읽을 수 있어요"라고 알려줄 때만 처리
   → 불필요하게 블로킹하거나 컨텍스트 스위칭이 발생하지 않음
```

**실제 수치 비교**

| 항목 | Apache (Prefork) | Nginx |
|---|---|---|
| 10,000 동시 접속 RAM | ~50~100 GB | ~50~100 MB |
| 동시 처리 방식 | 스레드/프로세스 1:1 | 이벤트 루프 (싱글 스레드) |
| 컨텍스트 스위칭 | 요청 수만큼 증가 | 거의 없음 |

> 💡 Nginx가 Apache보다 고부하 환경에서 성능이 뛰어난 핵심 이유가 바로 커널 모드 전환 및 컨텍스트 스위칭 비용의 절감이다. Nginx는 프로세스나 스레드를 요청마다 생성하지 않으므로 create-destroy 패턴이 없고, CPU 사이클을 낭비하지 않는다.

### 사례 2 : Java의 스레드 모델 변화 — OS 스레드 vs 가상 스레드 (Project Loom)

**문제 상황 — 전통적인 Java 스레드의 한계**

Java의 전통적인 플랫폼 스레드는 **OS 스레드와 1:1 매핑**된다. 스레드가 `socket.read()` 같은 블로킹 I/O를 호출하면, OS 스레드 자체가 블로킹된다.

```
[전통 Java 스레드 — Blocking I/O]

스레드 풀 (최대 200개 가정)

Thread-1:   [HTTP 요청 처리 중] ── [DB 쿼리 대기 중... 20ms ──────] ── [응답]
Thread-2:   [HTTP 요청 처리 중] ── [외부 API 대기 중... 150ms ───────────────] ──
Thread-3:   [HTTP 요청 처리 중] ── [파일 읽기 대기... 5ms ──] ─ [응답]
...
Thread-200: ...

→ 201번째 요청이 오면? 스레드 풀 고갈 → 대기 큐에 적체 → 응답 지연
→ OS 스레드 생성 비용: 약 1MB 스택 메모리 + 커널 자료구조 할당
```

**해결 방식 — Java 21 가상 스레드 (Virtual Thread)**

Java 21(Project Loom)에서 도입된 가상 스레드는 JVM이 수백만 개의 가상 스레드를 **소수의 OS(캐리어) 스레드 위에서 스케줄링**한다. 마치 가상 메모리가 적은 RAM으로 큰 주소 공간을 흉내 내듯, 가상 스레드는 적은 OS 스레드로 무수히 많은 동시 작업을 처리한다.

```
[가상 스레드 동작 방식]

OS 스레드 (= 캐리어 스레드) — CPU 코어 수만큼만 존재
  │
  ├── 가상 스레드 V1 실행 중
  │       ↓ 블로킹 I/O 발생 (DB 쿼리)
  │   JVM: V1의 스택을 힙 메모리에 저장 → V1을 OS 스레드에서 분리(unmount)
  │       ↓
  ├── 가상 스레드 V2 실행 (OS 스레드 재활용)
  │       ↓ 블로킹 I/O 발생
  │   JVM: V2 unmount → V3 실행...
  │
  └── V1의 DB 응답 완료 시 → JVM이 다시 OS 스레드에 올림(mount) → 실행 재개
```

| 구분 | 플랫폼 스레드 | 가상 스레드 |
|---|---|---|
| **OS 매핑** | 1:1 (OS 스레드와 직결) | 다수 : 소수 OS 스레드 |
| **스택 크기** | 약 1MB | 수 KB ~ 동적 확장 |
| **블로킹 시 동작** | OS 스레드 블로킹 → 컨텍스트 스위칭 | JVM이 OS 스레드 재활용, 커널 전환 최소화 |
| **생성 가능 수** | 수천 개 (메모리 한계) | 수백만 개 |
| **스케줄링 주체** | OS | JVM |

```java
// 가상 스레드 생성 — Java 21
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    // 100만 개의 가상 스레드도 거뜬히 생성 가능
    for (int i = 0; i < 1_000_000; i++) {
        executor.submit(() -> {
            // 블로킹 I/O 호출이 발생해도 OS 스레드는 블로킹되지 않음
            // JVM이 알아서 unmount → 다른 가상 스레드 실행 → 완료 후 재개
            String result = httpClient.get("https://api.example.com/data");
        });
    }
}
```

> ⚠️ **주의** : 가상 스레드는 I/O 바운드 작업(DB 쿼리, 네트워크 호출)에 효과적이다. CPU를 계속 사용하는 CPU 바운드 작업(암호화, 이미지 처리 등)에는 오히려 기존 스레드 풀이 적합하다.

## 📝 결론 : 프로세스·스레드의 차이와 시스템 콜 비용이 실무 성능에 미치는 영향

> 프로세스와 스레드는 운영체제가 작업을 처리하는 단위로, 둘의 핵심 차이는 **메모리 공유 여부**에 있다. 시스템 콜을 통한 커널 모드 전환은 CPU 사이클을 소비하는 비용이며, 이를 얼마나 잘 제어하느냐가 실무 애플리케이션의 성능을 결정짓는 핵심 요소다.

---

### 1. 프로세스 vs 스레드 — 핵심은 메모리 격리 수준

프로세스는 완전히 독립된 메모리 공간을 가져 안정성이 높지만, 생성과 컨텍스트 스위칭 비용이 크고 IPC를 통한 통신이 필요하다. 스레드는 메모리를 공유하므로 생성이 가볍고 통신이 빠르지만, 공유 자원에 대한 동기화 문제와 한 스레드의 오류가 프로세스 전체를 위협하는 위험이 있다.

실무에서는 이 특성을 바탕으로 선택한다.

- **Chrome 브라우저** : 탭마다 별도 프로세스 → 한 탭이 죽어도 다른 탭에 영향 없음
- **웹 서버 요청 처리** : 요청마다 스레드 → 빠른 생성과 메모리 효율

---

### 2. 시스템 콜 비용 — 사소해 보이지만 누적되면 치명적

단일 시스템 콜의 비용은 수백 ns 수준으로 미미해 보인다. 하지만 초당 수만 건의 요청을 처리하는 서버에서 요청마다 수십 번의 시스템 콜이 발생하면, 그 비용의 합이 실제 비즈니스 로직 처리 시간을 초과하는 경우도 발생한다.

실무 대응 전략 요약:

| 문제 | 전략 | 적용 사례 |
|---|---|---|
| 잦은 블로킹으로 컨텍스트 스위칭 폭증 | Non-Blocking I/O + 이벤트 루프 | Nginx의 epoll |
| OS 스레드 한계로 처리량 병목 | 가상 스레드 / 코루틴 | Java 21 Virtual Thread |
| 매 요청마다 소켓 생성 시스템 콜 | 커넥션 풀링 | HikariCP |
| 잦은 read() 시스템 콜 | 버퍼드 I/O, sendfile() | BufferedReader, Nginx |

---

### 3. 성능 최적화의 핵심 원칙

**⚠️ 주의해야 할 점:**

- 멀티스레드 환경에서 공유 자원 동기화를 소홀히 하면 **경쟁 조건(Race Condition)** 과 **데드락(Deadlock)** 이 발생할 수 있다
- Non-Blocking I/O는 CPU 집약적인 작업(연산 위주)에는 오히려 비효율적이다
- 스레드 수를 무조건 늘리면 컨텍스트 스위칭 오버헤드가 증가해 **스레싱(Thrashing)** 현상이 발생한다
- 가상 스레드는 `synchronized` 블록 내부에서 블로킹 I/O를 호출하면 **Pinning** 현상이 발생해 OS 스레드가 블로킹될 수 있다

> ✅ 결국 프로세스, 스레드, 시스템 콜의 동작 원리를 이해하는 것은 단순한 이론 지식이 아니라, **언제 어떤 비용이 발생하는지를 파악해 실무에서 올바른 아키텍처 결정을 내리기 위한 기반**이다. "왜 이 라이브러리가 빠른가"에 대한 답이 항상 이 원리 안에 있다.
