## 1. JPA, QueryDSL 세팅

### 의존성 추가


```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-data-jpa'

    // QueryDSL
    implementation 'com.querydsl:querydsl-jpa:5.0.0:jakarta'
    annotationProcessor 'com.querydsl:querydsl-apt:5.0.0:jakarta'
    annotationProcessor 'jakarta.annotation:jakarta.annotation-api'
    annotationProcessor 'jakarta.persistence:jakarta.persistence-api'
}
```

QueryDSL은 JPA와 별도로 Q클래스를 생성하기 위한 어노테이션 프로세서 설정이 추가로 필요하다. Jakarta EE 기반(Spring Boot 3.x)이라 `jakarta` 클래시파이어가 붙은 버전을 사용했다.

### QueryDSL 설정

```java
@Configuration
public class QuerydslConfig {

    @PersistenceContext
    private EntityManager entityManager;

    @Bean
    public JPAQueryFactory jpaQueryFactory() {
        return new JPAQueryFactory(entityManager);
    }
}
```

`JPAQueryFactory`를 빈으로 등록해두면, 이후 Repository 구현체에서 주입받아 바로 QueryDSL 쿼리를 작성할 수 있다.

### Q클래스 생성 확인

빌드를 실행하면 `build/generated` 경로에 `QUser`, `QPost` 같은 Q클래스가 자동 생성된다. 이 클래스들이 자바 코드로 타입 안전한 쿼리를 작성할 수 있게 해주는 핵심이다. Q클래스가 안 보인다면 IDE에서 generated source 경로를 소스 루트로 잡아줘야 한다 (IntelliJ는 보통 자동으로 잡아준다).

## 2. 대표 도메인 엔티티 설계

이번 주에 작성한 엔티티 중 설계 의도가 뚜렷한 4개를 정리했다.

---

### 게시글 (Post)

```java
package com.example.Community.domain.entity;

import jakarta.persistence.*;
import lombok.Getter;
import lombok.RequiredArgsConstructor;
import org.springframework.data.annotation.CreatedDate;
import org.springframework.data.annotation.LastModifiedDate;
import org.springframework.data.jpa.domain.support.AuditingEntityListener;

import java.time.LocalDateTime;

@Entity
@Table(name = "posts")
@Getter
@EntityListeners(AuditingEntityListener.class)
@RequiredArgsConstructor
public class Post {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "post_id")
    private Long id;

    private String title;
    private String content;

    @CreatedDate
    @Column(name = "reg_date")
    private LocalDateTime createdDate;  // 자동으로 생성일자 입력

    @LastModifiedDate
    @Column(name = "upd_date")
    private LocalDateTime lastModifiedDate;  // 자동으로 수정일자 입력

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id")
    private User author;

    public Post(String title, String content, User author) {
        this.title = title;
        this.content = content;
        this.author = author;
    }

    public void changeTitle(String title) {
        this.title = title;
    }

    public void changeContent(String content) {
        this.content = content;
    }
}
```

**핵심 설계**

- `author`(`User`)와 `@ManyToOne(FetchType.LAZY)` 관계로 연결
- `@CreatedDate`, `@LastModifiedDate`로 생성/수정 시각 자동 기록
- `@Setter`를 배제하고 `changeTitle()`, `changeContent()` 메서드로만 수정 허용

**설계 이유**

DB 구조상 `posts` 테이블이 `user_id` 외래키(FK)를 가진다. JPA 표준에 따라 외래키를 가진 N 측, 즉 `Post` 엔티티를 연관관계의 주인으로 설정했다.

`@Setter`를 열어두면 외부에서 제목이나 내용을 아무렇게나 바꿀 수 있어 데이터 추적이 어렵고 무결성이 깨질 수 있다. `changeTitle()`, `changeContent()`처럼 명확한 의미를 가진 비즈니스 메서드만 열어두어, 엔티티가 스스로 상태 변화를 제어하도록 설계했다.

### 유저 (User)

```java
package com.example.Community.domain.entity;

import jakarta.persistence.*;
import lombok.Getter;
import lombok.RequiredArgsConstructor;
import org.springframework.data.annotation.CreatedDate;
import org.springframework.data.annotation.LastModifiedDate;
import org.springframework.data.jpa.domain.support.AuditingEntityListener;

import java.time.LocalDateTime;
import java.util.ArrayList;
import java.util.List;

@Entity
@Table(name = "users")
@Getter
@EntityListeners(AuditingEntityListener.class)
@RequiredArgsConstructor
public class User {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name="user_id")
    private Long id;

    @Column(name="user_email")
    private String email;

    @Column(name="user_pwd")
    private String password;

    @Column(name="user_name")
    private String name;

    @CreatedDate
    @Column(name = "reg_date")
    private LocalDateTime createdDate;

    @LastModifiedDate
    @Column(name = "upd_date")
    private LocalDateTime lastModifiedDate;

    @OneToMany(mappedBy = "author")
    List<Post> posts = new ArrayList<>();

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "file_id")
    private File profileImage;

    public User(String email, String password, String name, File profileImage) {
        this.email = email;
        this.password = password;
        this.name = name;
        this.profileImage = profileImage;
    }

    public void changePassword(String password) {
        this.password = password;
    }

    public void changeName(String name) {
        this.name = name;
    }

    public void updateProfileImage(File profileImage) {
        this.profileImage = profileImage;
    }
}
```

**핵심 설계**

- 프로필 이미지를 단순 텍스트가 아닌 `File` 엔티티와 `@ManyToOne(FetchType.LAZY)` 관계로 연결
- `posts` 목록은 `mappedBy = "author"`로 읽기 전용 양방향 매핑

**`@ManyToOne`을 채택한 이유**

일반적인 커뮤니티 사이트는 가입 시 모든 유저에게 기본 프로필 이미지(`/public/images/default.jpg`)를 부여한다. 이를 `@OneToOne`으로 잡으면 유저 수만큼 똑같은 기본 이미지 데이터가 DB에 중복 생성되어야 한다.

`@ManyToOne` 구조를 택하면 `files` 테이블에는 기본 이미지 레코드를 딱 1개만 만들어두고, 수많은 유저가 이를 참조할 수 있다. DB 공간을 절약할 수 있는 구조다.

**지연 로딩(LAZY)을 통한 N+1 차단**

유저 목록을 조회하거나 로그인을 처리할 때, 프로필 이미지의 상세 정보나 해당 유저가 쓴 게시글 목록까지 매번 함께 가져올 필요는 없다. 전부 `FetchType.LAZY`로 설정해, 실제로 해당 데이터가 화면에 그려지거나 호출되는 시점에만 쿼리가 나가도록 제어했다.

### 리프레시 토큰 (RefreshToken)

```java
package com.example.Community.domain.entity;

import jakarta.persistence.*;
import lombok.AccessLevel;
import lombok.Getter;
import lombok.NoArgsConstructor;

import java.time.LocalDateTime;

@Entity
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class RefreshToken {

    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "token_id")
    private Long id;

    @Column(nullable = false, unique = true)
    private String token;

    @Column(name = "user_id", nullable = false)
    private Long userId;

    @Column(name = "expires_at", nullable = false)
    private LocalDateTime expiresAt;

    public RefreshToken(String token, Long userId, LocalDateTime expiresAt) {
        this.token = token;
        this.userId = userId;
        this.expiresAt = expiresAt;
    }

    // 현재 시각을 기준으로 토큰 만료 여부를 판단
    public boolean isExpired() {
        return LocalDateTime.now().isAfter(this.expiresAt);
    }
}
```

**핵심 설계**

- 회원 엔티티(`User`)를 직접 참조하지 않고, 단순 식별자 값인 `userId(Long)` 컬럼만 보유
- 토큰 자체에 만료 여부를 판별하는 비즈니스 메서드 `isExpired()` 내장

`User`를 직접 참조(`@ManyToOne`)하지 않은 이유는, 토큰의 생명주기가 회원 엔티티의 영속성과 굳이 묶일 필요가 없기 때문이다. 인증 모듈은 회원 도메인과 느슨하게 연결되어 있는 편이 유지보수에 유리하다. `isExpired()`처럼 판단 로직을 엔티티 안에 두면, Service 레이어에서 매번 만료 시각을 비교하는 코드를 반복하지 않아도 된다.

### 파일 (File)

```java
package com.example.Community.domain.entity;

import jakarta.persistence.*;
import lombok.Getter;
import lombok.NoArgsConstructor;
import org.springframework.data.annotation.CreatedDate;

import java.time.LocalDateTime;

import static lombok.AccessLevel.PROTECTED;

@Entity
@Getter
@Table(name = "files")
@NoArgsConstructor(access = PROTECTED)
public class File {

    @Column(name="file_id")
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name="file_path", nullable = false)
    private String filePath = "/public/images/default.jpg";

    @Column(name="file_category", nullable = false)
    @Enumerated(EnumType.STRING)
    private FileCategory fileCategory;

    @Column(name="uploader_id")
    private Long uploaderId;  // 업로더 추적용 (nullable)

    @CreatedDate
    @Column(name = "reg_date")
    private LocalDateTime createdDate;

    public File(String filePath, FileCategory fileCategory, Long uploaderId) {
        this.filePath = filePath;
        this.fileCategory = fileCategory;
        this.uploaderId = uploaderId;
    }

    // 정적 팩토리 메서드
    public static File createProfileImage(String filePath, Long uploaderId) {
        return new File(filePath, FileCategory.PROFILE_IMAGE, uploaderId);
    }
}
```

**핵심 설계**

- 파일 카테고리를 `@Enumerated(EnumType.STRING)`으로 관리
- `uploaderId`는 `nullable`로 설정
- 기본 생성자를 `protected`로 제한하고, 정적 팩토리 메서드(`createProfileImage`) 사용

**Enum을 STRING으로 저장한 이유**

파일 카테고리(용도)를 숫자(`ORDINAL`)가 아닌 문자열 기반의 Enum으로 저장하도록 강제했다. `ORDINAL`은 enum 순서가 바뀌면 기존 데이터가 다른 의미로 읽히는 위험이 있어, 데이터 정합성을 위해 `STRING`을 채택했다.

**기본 생성자를 protected로 잠근 이유**

다른 개발자가 서비스 레이어에서 임의로 `new File()`을 호출해, 필수 값이 빠진 "유령 파일" 객체를 만드는 상황을 막기 위해서다. 대신 `createProfileImage()`라는 정적 팩토리 메서드를 제공해, 오직 유효한 프로필 이미지 파일 객체만 정해진 절차로 생성되도록 강제했다.

**`uploaderId`로 느슨하게 연결한 이유**

회원 테이블과 강하게 FK 관계를 맺지 않고 단순 `Long uploaderId` 컬럼만 두었다. 유저가 탈퇴해도 파일 정보는 유지되어야 하기 때문이다. 회원이 완전히 삭제되더라도 파일 데이터는 무너지지 않도록 설계한 것이다.

## 3. 이번 주 리팩터링 & 기능 추가

이번 주 작업은 두 번의 커밋으로 나뉜다. 첫 번째 커밋에서는 기존 코드의 응답 형식 통일, 예외 처리 구조 개선, 유효성 검증 강화를 리팩터링했고, 두 번째 커밋에서는 Spring Security 도입, 파일 업로드 기능, QueryDSL 연동 등 새로운 기능을 추가했다.

### 리팩터링 (커밋 1)

#### ① 공통 응답 형식 통일 — `ApiResponse<T>`

기존 코드는 컨트롤러마다 `Map<String, Object>`를 직접 조합해 응답을 만들었다. 그 결과 응답 키 이름이 제각각이었다(`"data"`, `"message"` 등이 혼용). 프론트엔드 입장에서는 매 API마다 응답 구조를 다시 파악해야 하는 번거로움이 있었다.

```java
// 변경 전
return ResponseEntity.ok(Map.of("data", Map.of("post_id", created.getId())));

// 변경 후
return ResponseEntity.status(HttpStatus.CREATED)
        .body(ApiResponse.of("POST_CREATED", result));
```

`ApiResponse<T>`를 도입해 모든 응답을 `{ code, data }` 구조로 통일했다. `ErrorResponseDto`도 동일한 구조를 따르도록 맞췄다. 성공/실패 응답 형태가 같으므로, 프론트에서는 단일 파싱 로직 하나로 모든 응답을 처리할 수 있게 됐다.

#### ② 예외 계층 분리 — `NotFoundException`

기존에는 존재하지 않는 리소스를 조회할 때 `IllegalArgumentException`을 던지거나, `BusinessException`에 직접 `HttpStatus.NOT_FOUND`를 넣어 사용했다. 이 방식은 코드만 봤을 때 "이게 404 응답을 의도한 예외인지"가 명확하게 드러나지 않는 문제가 있었다.

`BusinessException`을 상속하는 `NotFoundException`을 별도로 만들고 `HttpStatus.NOT_FOUND`를 고정했다. `GlobalExceptionHandler`에도 `NotFoundException` 핸들러를 우선순위 높게 등록해, 더 구체적인 예외가 먼저 처리되도록 했다. 덕분에 서비스 레이어에서는 `"USER_NOT_FOUND"`, `"POST_NOT_FOUND"` 같은 문자열 코드만 넘기면 되어 코드가 한결 간결해졌다.

#### ③ DTO 유효성 검증 강화

클라이언트 입력값 검증이 서비스 레이어 내부에서 임의로 처리되거나 아예 빠져 있었다. `@Valid`와 Bean Validation 제약 조건을 추가해, 컨트롤러에 진입하기 전 단계에서 검증이 끝나도록 개선했다.

#### ④ 응답 코드 문자열 정규화

기존에는 예외 메시지로 소문자 스네이크 케이스(`non_existent_user`, `invalid_refresh_token`)를 사용했다. 이를 대문자 스크리밍 스네이크 케이스로 통일했다.

- 성공 응답: `POST_CREATED`, `USER_CREATED`, `LOGIN_SUCCESS` 등
- 에러 응답: `INVALID_CREDENTIALS`, `UNAUTHORIZED`, `USER_NOT_FOUND` 등

클라이언트가 코드 값으로 메시지를 분기할 때 일관된 규칙이 있어야 에러 핸들링이 수월해지기 때문이다.

#### ⑤ 토큰 재발급 엔드포인트 경로 변경

기존 `/users/token/refresh` 경로는 유저 리소스 하위에 토큰 관련 경로가 포함된 구조라 REST 설계상 어색했다. `/token/refresh`로 변경해 인증 관련 엔드포인트를 유저 리소스와 분리했고, JWT 필터 화이트리스트에서도 해당 경로를 함께 수정했다.

#### ⑥ Post 엔티티에서 image 필드 제거

게시글 이미지를 `String` 컬럼으로 저장하던 방식에서, 별도 `File` 엔티티 관계로 분리하는 방향으로 설계를 바꿨다. 이번 커밋에서는 우선 `Post.image` 필드와 관련 DTO 매핑을 제거했다. 파일을 독립 엔티티로 관리하면, 동일 이미지를 여러 게시글에서 참조하거나 파일 메타데이터(카테고리, 업로더 등)를 추적하기 쉬워진다.

#### ⑦ DB 컬럼명 수정

`Post`와 `User` 엔티티의 날짜 컬럼이 `reg_dat`, `upd_dat`이었다. 무분별한 줄임말을 지양하기 위해 `reg_date`, `upd_date`로 수정했다.

### 기능 추가 (커밋 2)

#### ① Spring Security 도입 및 JWT 필터 재설계

기존에는 `JwtAuthenticationFilter`에서 직접 `response.setStatus(401)`을 반환하고, `request.setAttribute("userId", userId)`로 사용자 정보를 전달했다. 이 방식은 Spring의 인증 체계와 완전히 분리되어 있어 `@AuthenticationPrincipal` 같은 편의 기능을 쓸 수 없었다.

Spring Security를 도입하고, 인증 성공 시 `UsernamePasswordAuthenticationToken`을 생성해 `SecurityContextHolder`에 저장하도록 변경했다. 인증 실패(`authenticationEntryPoint`)와 인가 실패(`accessDeniedHandler`)는 각각 `ApiResponse` 형식의 JSON으로 응답하도록 커스텀했다. 덕분에 컨트롤러에서 `httpRequest.getAttribute("userId")` 대신 `@AuthenticationPrincipal Long userId`로 깔끔하게 주입받을 수 있게 됐다.

#### ② 비밀번호 BCrypt 암호화

기존 코드는 평문 비밀번호를 그대로 DB에 저장하고 `equals()`로 비교했다. DB가 노출되면 모든 사용자의 비밀번호가 즉시 탈취되는 치명적인 보안 취약점이었다.

`PasswordEncoder` 빈을 `BCryptPasswordEncoder`로 등록하고, 회원가입 시 `passwordEncoder.encode()`로 해시화해 저장하도록 변경했다. 로그인 시에는 `passwordEncoder.matches()`로 비교해, 원문을 복호화하지 않고도 인증할 수 있다. 시드 데이터(`SeedConfig`)도 암호화된 비밀번호를 사용하도록 함께 수정했다.

#### ③ File 엔티티 & 파일 업로드 기능 구현

프로필 이미지를 `User` 엔티티의 단순 `String` 컬럼으로 관리하는 방식은 확장성이 부족했다. 파일 카테고리, 업로더 추적, 경로 관리 같은 관심사를 분리하기 위해 위에서 다룬 `File` 엔티티를 별도로 설계했다.

- **File 엔티티**: `id`, `filePath`, `fileCategory`(`PROFILE_IMAGE` / `POST_ATTACHMENT`), `uploaderId`, `createdDate`
- `User` 엔티티에서 `image: String` 필드를 제거하고 `profileImage: File`(`@ManyToOne`) 관계로 교체
- `FileService.uploadProfileImage()`는 확장자 검증 → 디렉터리 생성 → 파일 저장 → DB 레코드 생성 순서로 처리
- 저장 경로는 프로젝트 루트의 `uploads/profile/`, 클라이언트 접근 URL은 `/public/profile/**`
- `WebConfig`에서 `/public/**` 정적 리소스 핸들러를 등록해 업로드된 파일을 HTTP로 서빙

#### ④ N+1 문제 해결

`User` 조회 시 `profileImage`(File)가 `LAZY` 로딩으로 설정되어 있어, 프로필 이미지에 접근할 때마다 추가 쿼리가 발생하는 N+1 문제가 있었다.

`UserQueryRepository` 인터페이스와 `UserQueryRepositoryImpl` 구현체를 만들어 `findByEmailWithProfileImage()`, `findByIdWithProfileImage()` 메서드를 제공했다. 두 메서드 모두 `left join fetch`로 `profileImage`를 한 번의 쿼리로 함께 가져온다. `AuthService`와 `UserService`에서 기존 `userRepository.findByXxx()` 대신 `userQueryRepository`를 사용하도록 변경했다.

```java
// UserQueryRepositoryImpl 예시
@Override
public Optional<User> findByIdWithProfileImage(Long id) {
    return Optional.ofNullable(
        queryFactory.selectFrom(user)
            .leftJoin(user.profileImage, file).fetchJoin()
            .where(user.id.eq(id))
            .fetchOne()
    );
}
```

여기서 앞서 세팅한 QueryDSL이 실제로 빛을 발한다. `leftJoin(...).fetchJoin()` 한 줄로 N+1을 해결할 수 있다.

#### ⑤ UserController REST 구조 개선

기존 `/users/{userId}` 패턴은 토큰에 사용자 ID가 이미 있음에도 URL에 ID를 그대로 노출시켜, 다른 사용자의 정보를 조회·수정할 수 있는 보안 취약점이 있었다. `/users/me`로 변경해 현재 인증된 사용자만 자신의 정보에 접근하도록 했다.

회원가입 요청/응답 DTO도 `UserRequestDto` 하나에서 `SignupRequest` / `SignupResponse`로 역할에 맞게 분리했고, `PATCH /users/me`는 이름 수정만 담당하도록 `UpdateUserRequest`를 별도로 만들었다.

#### ⑥ FileUtil 유틸리티 클래스

파일 경로와 URL 변환 로직이 여러 서비스에 흩어질 수 있어 `FileUtil`로 통합했다.

- `toFullUrl(relativePath)`: DB에 저장된 상대 경로(`/public/...`)를 현재 서버 도메인이 붙은 전체 URL로 변환
- `extractPathFromUrl(url)`: 클라이언트에서 받은 전체 URL에서 경로 부분만 추출해 DB 조회에 사용
- 인스턴스화 방지를 위해 `private` 생성자를 둠

---

## 4. 회고

### 잘 된 점

응답 형식을 `ApiResponse<T>`로 통일하니 컨트롤러 코드가 짧아지고 프론트와의 계약이 명확해졌다. `File` 엔티티를 별도로 분리하고 `@ManyToOne`으로 연결하는 경험을 통해, 단순 `String` 컬럼보다 관계형 모델링이 훨씬 유연하다는 점을 체감했다.

### 아쉬운 점 & 개선 과제

전체 API를 아직 구현하지 못해서, 다음 주 화요일까지 전체 구현을 마치는 것을 목표로 하고 있다. 가장 기초가 되는 부분들을 최대한 이해하고 완성도 높게 구현하려다 보니 시간이 많이 소요되는 것 같다. 시간을 더 투자해서 빠르게 구현을 마쳐야겠다.

### 정리

이번 주는 단순히 기능을 추가하는 데 그치지 않고, 다양한 부분을 고민하며 코드를 개선하는 데 집중한 한 주였다. 공통 응답 래퍼, 예외 계층, 보안(BCrypt / Spring Security), 파일 엔티티 분리, N+1 최적화까지 — 실무에서 자주 마주치는 문제들을 직접 다루면서, 각각의 결정이 어떤 문제를 해결하는지 맥락을 이해하며 구현할 수 있었다.
