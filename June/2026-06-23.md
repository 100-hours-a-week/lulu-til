#### | S3 (Simple Storage Service)

**이미지, 동영상, 백업파일, 로그처럼 많은 파일을 객체 단위로 저장하고, 인터넷 또는 AWS 서비스에서 접근할 수 있게 해주는 객체 스토리지 서비스**

##### 서버 디스크(EBS)에 파일을 저장하면 생기는 문제

EC2의 EBS에 이미지 파일을 그대로 저장하면 다음과 같은 상황에서 한계가 생긴다.

- "이 이미지 파일은 외부에서 바로 볼 수도 있어야 한다" → EBS는 EC2에 종속된 디스크라 외부 직접 노출에 적합하지 않다
- 그렇다고 EFS로 묶기에도 애매한 경우가 있다

**EFS (Elastic File System)**
- 공유 네트워크 파일 스토리지
- **VPC 내부에서만** 사용해야 한다 (외부 공개에는 적합하지 않음)
- WS, WAS의 정적 파일 서빙처럼 여러 서버가 같은 파일을 공유해야 할 때 사용

이런 한계를 해결하기 위해 **S3**를 사용한다.

##### S3란

파일을 객체 단위로 저장하는 **관리형 객체 스토리지 서비스**다.

> ⚠️ S3는 서버 디스크를 폴더처럼 마운트해서 쓰는 서비스가 아니다.

파일을 업로드하면 S3는 이를 **객체(Object)**라는 단위로 저장하고, **HTTP 또는 AWS SDK를 통해서만** 접근할 수 있다.

##### 객체 스토리지란

**객체**는 S3에 저장되는 실제 데이터 단위다. 이미지, 동영상, PDF 등 어떤 파일이든 객체로 저장될 수 있다.

**파일과 객체의 차이** — S3가 관리하는 객체는 단순히 파일 데이터만이 아니라 아래 정보를 함께 저장한다.

| 구성 요소 | 설명 |
|---|---|
| **Bucket** | 객체가 저장되는 버킷 |
| **Key** | 객체의 고유한 경로 이름 |
| **Value** | 실제 파일 데이터 |
| **Metadata** | Content-Type, 생성 시각, 사용자 정의 메타데이터 등 |
| **Version ID** | 버전 관리 활성화 시 객체 버전 식별자 |

> 💡 EFS, EBS는 **파일 시스템**이라서 마운트하거나 `mkdir` 같은 명령어를 쓸 수 있지만, S3는 **객체 스토리지**라서 이런 파일 시스템 명령어를 직접 쓸 수 없다.

##### S3 핵심 구성 요소

**1. Bucket**

S3 객체를 담는 최상위 저장 공간으로, 폴더와 비슷한 개념이다.

버킷 이름의 특징:
- **AWS 계정 내에서 고유**해야 한다
- 일반적으로 소문자, 숫자, 하이픈을 사용한다
- 한 번 정한 이름은 다른 사람이 이미 사용 중이면 변경하거나 만들 수 없다
- 버킷은 **특정 리전에 생성**된다 (다른 리전에 있는 버킷은 서로 별개)
- 버킷은 실제 폴더가 아니다 — 버킷이 없는 상태에서 `s3:PutObject`로 `charlie-bucket`을 지정하면 그냥 그 이름의 버킷이 만들어진다

**2. Object**

실제로 저장되는 데이터 단위.

```
s3://myserver-bucket/image/profile/charlie.png
HTTP GET https://mybukc~~.charlie.png
```

**3. Key**

버킷 안에서 객체를 구분하는 고유한 이름.

```
버킷: my-service-bucket
Key: image/profile/charlie-01.png

→ s3://my-service-bucket/image/profile/charlie-01.png
```

> ⚠️ 계층형 폴더처럼 보이지만, 실제로는 파일 시스템이 아니다. 그냥 **하나의 Key 문자열**로 저장되는 객체 스토리지일 뿐이다. (`image/profile/charlie-01.png` 전체가 하나의 Key)

##### EFS vs S3 비교

| 구분 | EFS | S3 |
|---|---|---|
| 저장 방식 | 파일 스토리지 | 객체 스토리지 |
| 접근 방식 | Linux 파일 시스템처럼 마운트 | SDK, CLI, REST API, HTTP |
| `cd`, `touch`, `chmod` 사용 | 가능 | 직접 사용 불가 |
| 여러 서버가 동시에 사용 | 가능 | 가능하지만 파일 시스템 방식은 아님 |
| 대표 용도 | 공유 폴더, CMS, 컨테이너 공유 볼륨 | 이미지, 동영상, 백업, 로그, 정적 콘텐츠 |
| 외부 브라우저 접근 | 직접 목적은 아님 | URL, CloudFront, Presigned URL로 가능 |

##### S3를 사용하는 이유

**① 제한 없이 확장 가능**
- EC2의 EBS는 용량 제한이 있어서 부족하면 직접 늘리는 작업이 필요하다
- S3는 저장할 객체가 늘어나도 별도로 디스크를 증설할 필요가 없다

**② 서버가 바뀌어도 파일이 유지된다**
- EC2는 언제든 종료될 수 있는 리소스다
- 새로운 EC2가 IAM Role을 통해 `s3:GetObject` 권한으로 S3에 접근하면, 기존 파일을 그대로 이어서 사용할 수 있다

**③ 내구성 + 가용성이 높다**
- S3 Standard는 연간 **내구성(객체 내구성) 99.999999%**, **가용성 99.99%**를 제공한다
- **내구성**: 저장한 데이터가 사라지지 않을 가능성
- **가용성**: 필요할 때 서비스를 사용할 수 있는 가능성

##### S3 ↔ EC2를 함께 사용하는 방식

**① 백엔드가 업로드를 처리하는 방식**

```
사용자 브라우저 → 백엔드 API : 이미지 업로드 요청
백엔드 API → S3 : PutObject
S3 → 백엔드 API : 객체 저장 완료
백엔드 API → 사용자 브라우저 : 객체 URL 또는 Key 반환
```

이 방식을 쓰는 이유:
- 사용자가 업로드한 파일을 **일관적으로 관리**해야 할 때 (포맷 맞추기, 파일 사이즈 제한, 리사이징/압축 등)
- S3의 관심사는 **파일을 저장하고 조회하게 하는 것뿐**이다. 그 안에 어떤 파일이 들어오든 S3는 신경 쓰지 않는다
- 보안적으로 중요한 파일을 다뤄야 할 때 백엔드가 검증 로직을 거치는 게 더 안전하다

**② Presigned URL 구조**

```
사용자 브라우저 → 백엔드 API : 업로드 권한 요청
백엔드 API → S3 : Presigned URL 생성
백엔드 API → 사용자 브라우저 : 시간 제한 업로드 URL 반환
사용자 브라우저 → S3 : 파일 직접 업로드
S3 → 사용자 브라우저 : 업로드 완료
사용자 브라우저 → 백엔드 API : 저장된 객체 Key 전달
```

**Presigned URL**: 일정 시간 동안 특정 S3 버킷·객체에 접근할 수 있게 만든 **서명된 URL**. 시간 제한을 두고 업로드 또는 다운로드 권한을 부여한다.

| 장점 | 단점 |
|---|---|
| 백엔드 서버가 큰 파일 데이터를 직접 받지 않아도 됨 → EC2의 네트워크 자원을 덜 사용 → 비용 절감 | 파일에 대한 검증을 클라이언트에게 맡겨야 함 |
| 업로드 가능한 시간, 파일 경로, 권한을 세밀하게 제한 가능 | 백엔드 입장에서 "Don't Trust Client" 원칙상 클라이언트는 변조하기 쉬움 |
| | 파일 압축·형 변환을 클라이언트가 수행해야 하는데, 모든 브라우저/클라이언트 자원이 일정하지 않아 결과가 일정하지 않음 |

> 💡 이런 단점을 보완하기 위해, 형 변환이나 압축 없이 원본 파일 그대로 업로드받고 이후 **배치 프로그램**으로 후처리하는 방식을 택할 수도 있다.

##### S3 주요 기능

**1. Versioning**

같은 Key로 파일을 다시 올려도 이전 버전을 보관하는 기능.
- 여러 버전을 동시에 보존 가능
- 실수로 삭제하거나 덮어쓴 경우에도 복구 가능
- 버저닝을 켜면 보관되는 데이터 양(과 비용)도 함께 늘어난다
- 보통 **Lifecycle Rule과 함께 설계**한다

**2. Lifecycle Rule**

오래된 객체를 더 저렴하지만(접근은 더 느린) 스토리지 클래스로 자동으로 옮기는 기능.

```
S3 Standard (자주 접근) → S3 Standard-IA (가끔 접근) → S3 Glacier (장기 보관) → 보관 기간 종료 후 삭제
```

**3. 스토리지 클래스 설정 (성능/비용 트레이드오프)**

접근 속도와 복구 속도 요구사항에 따라 적절한 스토리지 클래스를 선택한다.

| 스토리지 클래스 | 대표 사용처 |
|---|---|
| S3 Standard | 자주 사용하는 이미지, 애플리케이션 파일 |
| S3 Intelligent-Tiering | 접근 패턴이 예측하기 어려운 파일 |
| S3 Standard-IA | 가끔 읽는 백업 파일 |
| S3 One Zone-IA | AZ 장애를 감수할 수 있는, 재생성 가능한 데이터 |
| Glacier Instant Retrieval | 거의 안 쓰지만 빠른 조회가 필요한 데이터 |
| Glacier Flexible Retrieval | 장기 보관 백업 |
| Glacier Deep Archive | 매우 장기 보관, 가장 저렴한 수준의 아카이브 |

실무에서 흔히 쓰는 매핑:

```
서비스 이미지            → S3 Standard
접근 패턴이 불확실한 파일  → S3 Intelligent-Tiering
백업·감사 로그           → Lifecycle Rule로 IA 또는 Glacier 이동
```

##### S3 사용처

1. **사용자 이미지 처리** — 실제 파일은 S3에 저장하고, 그 객체의 **Key는 DB에 저장**하는 방식이 일반적
2. **정적 웹사이트 배포** — CloudFront + S3 조합으로 프론트엔드를 정적으로 호스팅

#### | RDS (Relational Database Service)

**클라우드에서 관계형 데이터베이스를 손쉽게 생성, 운영, 확장할 수 있도록 제공하는 완전관리형 서비스**

##### 예시 상황 — EC2에 DB를 직접 설치하면 생기는 일

애플리케이션은 보통 데이터를 저장할 때 데이터베이스를 사용한다. 가장 단순한 방법은 EC2에 MySQL을 직접 설치하는 것이다.

하지만 **운영 환경**(실제 사용자가 접속하는 환경)에서는, DB를 설치하는 일 자체보다 그것을 **365일 24시간 안정적으로 운영하는 일이 훨씬 더 어렵다.**

직접 설치하면 다음을 전부 직접 관리해야 한다.

- OS와 DB 엔진 설치
- MySQL / PostgreSQL 버전 관리
- 보안 패치
- 디스크 용량 관리
- 백업 및 복구
- 장애 감지와 복구
- 모니터링
- 고가용성 구성
- 읽기 성능 확장

> 💡 우리는 결국 **가치를 만들어 내야 한다.** 개발자는 비즈니스를 위해 개발이라는 도구로 가치를 만들어내야 하는데, DB 운영 자체에 너무 많은 시간을 쓰는 건 그 가치 창출과는 거리가 있다.

##### RDS란

**운영 환경을 AWS가 대신 처리하도록 만든 서비스**다.

> ⚠️ RDS 자체는 데이터베이스 프로그램이 아니다. RDS는 **데이터베이스를 운영하기 위한 AWS의 관리 서비스**다. (실제 DB 엔진은 MySQL, PostgreSQL 같은 기존 제품을 그대로 사용한다.)

##### RDS vs 직접 설치 비교

```
Amazon RDS                          EC2에 MySQL 직접 설치
 └─ RDS DB 인스턴스                   └─ EC2
     └─ 백업/패치/모니터링/장애조치          └─ MySQL 설치
        (AWS가 관리)                        └─ EBS
```

| 구분 | EC2에 DB 직접 설치 | RDS |
|---|---|---|
| DB 설치 | 직접 설치 | AWS가 프로비저닝 지원 |
| 운영체제 관리 | 직접 관리 | AWS가 관리 |
| DB 패치 | 직접 계획·수행 | 유지보수 설정 기반 관리 가능 |
| 백업 | 직접 스크립트·스케줄 구성 | 자동 백업 기능 제공 |
| 장애 복구 | 직접 구성 | Multi-AZ 등 관리형 옵션 제공 |
| 모니터링 | 직접 도구 설치·구성 | CloudWatch 등과 통합 |
| 디스크 관리 | EBS 직접 관리 | RDS 스토리지 설정으로 관리 |
| 유연성 | 매우 높음 | 엔진별 제약 존재 |
| 운영 부담 | 큼 | 상대적으로 작음 |

한마디로: **EC2 + MySQL = 모든 관리를 직접 한다 / RDS = AWS에게 관리를 맡긴다.**

##### "그럼 그냥 RDS만 쓰면 되는 거 아닌가?" — 비용 문제

EC2 기반으로 고가용성 + 읽기 속도 개선까지 직접 구성한다고 하면 (예: AZ 2a에 Master 1 + Read-only Replica 3, AZ 2b에 Standby 1) 월 비용이 대략 **20만원~30만원** 수준일 수 있는 구성을, RDS로 동일하게 맞추면 **60만원~80만원** 수준이 될 수도 있다.

> 그래서 "찰리는 그냥 EC2에 직접 구성한다"는 선택을 할 수도 있다. 다만 **초기 프로젝트라면 그냥 RDS로 세팅하는 게 더 나은 선택**일 때가 많다. 초기에는 **운영 효율보다 개발 속도**가 더 중요하기 때문이다.

##### 일반적인 RDS 구성

```
사용자 → ALB → EC2/ECS 애플리케이션 서버 A ─┐
              EC2/ECS 애플리케이션 서버 B ─┴─ MySQL 3306 → RDS
```

- **사용자는 RDS에 직접 접속하지 않는다.** (사용자 = 일반 유저, 개발자 포함)
- 개발자가 RDS에 직접 접속해야 할 일이 있다면 **Bastion Host**를 거쳐야 한다
- 평상시에는 **애플리케이션 서버만 RDS에 접속**한다
- 운영 환경에서는 RDS를 외부 인터넷에 노출시키지 않기 위해 **Private Subnet**에 둔다 → 모든 외부 인터넷 접속이 차단된다는 의미

##### RDS 구성 요소

**1. DB Engine**

실제 데이터를 저장하고 SQL을 처리하는 데이터베이스 엔진 자체. (MySQL, PostgreSQL, Oracle 등)

**2. DB Instance**

DB 엔진이 실제로 실행되는 서버 자원. EC2 인스턴스 타입을 고르듯, RDS에서도 CPU + 메모리 + 네트워크 성능을 조합한 **DB 인스턴스 클래스**를 선택한다.

**3. Storage**

RDS의 스토리지에는 단순히 테이블 로우(row)만 저장되는 게 아니다. **인덱스, 트랜잭션 로그, 자동 백업본** 등도 함께 저장해야 한다.

스토리지에서는 용량뿐 아니라 **성능 지표**도 함께 봐야 한다.

| 항목 | 의미 |
|---|---|
| Storage Size | DB 데이터를 저장할 수 있는 용량 |
| IOPS | 초당 디스크 읽기·쓰기 처리 횟수 |
| Throughput | 초당 읽고 쓸 수 있는 데이터 양 |

**IOPS (Input/Output Operation Per Second)**

디스크가 1초 동안 처리할 수 있는 작업 횟수. 데이터베이스의 데이터는 결국 디스크에 쓰여진다.

IOPS가 중요해지는 상황:
- PK, 인덱스 기반으로 조회가 많이 일어나는 경우
- 업데이트가 많은 데이터베이스 (주문, 결제, 실시간 로그)
- 랜덤 액세스가 많은 경우

> 💡 데이터베이스에 **캐싱**(MySQL InnoDB Buffer Pool, Query Caching 등)을 적용하면 Disk 접근 자체가 줄어들어 IOPS 부담이 낮아진다.

예시 — 커뮤니티 게시글 상세 조회 API가 초당 500번 호출되고, 한 요청당 평균 DB I/O가 8번 발생한다면:

```
500회/초 × 8회 = 4,000 IOPS
```

(버퍼 풀, 인덱스, 캐시 적중 여부에 따라 실제 디스크 접근량은 달라지며, 디스크에서 대기하는 시간이 길어질수록 응답이 느려진다.)

**Throughput**

디스크가 1초에 얼마나 많은 양의 데이터를 읽고 쓰는지 (단위: MiB/s, MB/s). 예를 들어 125MiB라면 1초 동안 디스크와 약 125MB의 데이터를 주고받을 수 있다는 뜻이다.

Throughput이 중요해지는 상황:
- 대량 조회/대량 생성이 발생하는 경우
- 리포트 생성 (CSV, 엑셀 추출)
- 대규모 배치 프로그램 (예: 유저 삭제 시 그 유저가 작성한 300만 건의 좋아요, 30만 개의 댓글, 3만 건의 게시글을 일괄 삭제하는 배치)
- 데이터 마이그레이션 (DB A → DB B로 옮기는 작업)

**IOPS vs Throughput 비교**

| 구분 | IOPS | Throughput |
|---|---|---|
| 의미 | 초당 I/O 작업 횟수 | 초당 전송 가능한 데이터 양 |
| 단위 | IOPS | MiB/s, MB/s |
| 병목이 자주 나는 작업 | 작은 랜덤 조회·쓰기 | 대용량 순차 읽기·쓰기 |
| 대표 예시 | 사용자 조회, 주문 처리, 재고 차감 | 배치, 백업, 리포트, 대량 데이터 처리 |
| 비유 | 택배기사가 1초에 처리하는 **상자 개수** | 1초에 옮기는 **전체 물량(무게)** |

**4. Endpoint**

애플리케이션이 RDS에 접속할 수 있는 주소(엔드포인트).

```
mydb.abc123.ap-northeast-2.rds.amazonaws.com
```

```yaml
spring:
  datasource:
    url: jdbc:mysql://mydb.abc123.ap-northeast-2.rds.amazonaws.com:3306/app_db
    username: app_user
    password: ${DB_PASSWORD}
```

**5. Security Group**

어떤 서버(어떤 보안 그룹)가 이 데이터베이스의 포트에 접근할 수 있는지를 제어한다.

##### 일반적인 RDS를 포함한 웹 서비스 구성

```
인터넷
  ↓ HTTPS 443
[VPC]
  ↓
Public Subnet
  └─ ALB
       ↓ HTTP 8080         ↓ HTTP 8080
Private Subnet - App
  ├─ EC2/ECS A (sg-app)    EC2/ECS B (sg-app)
       ↓ MySQL 3306             ↓ MySQL 3306
Private Subnet - DB
  └─ RDS MySQL (sg-db)
```

##### 자동 백업과 스냅샷

**① 자동 백업 (Automated Backup)**
- RDS가 **주기적으로 수행**하는 관리형 백업
- 정상적으로 운영되다가 문제가 있다고 판단되면 백업을 수행
- 자동 백업은 **지정된 백업 보관 기간 동안만** 유지된다
- 그 기간 안에서 특정 시점으로 복구(Point-in-Time Recovery)가 가능하다

**② Manual Snapshot**
- 사용자가 **원하는 시점에 직접** 만드는 DB 스냅샷
- 주로 다음과 같이 데이터베이스에 큰 변화를 주기 **직전**에 현재 상태를 보관하기 위해 찍는다:
  - 데이터베이스 스키마 변경
  - 데이터베이스 엔진 업데이트
  - 운영 데이터 정리
  - 마이그레이션
  - 대규모 배포

##### 실무 구성 방법

**① Multi-AZ 구성**

DB 장애 상황에서 서비스 중단을 줄이기 위해, **다른 AZ에 대기(Standby) DB를 유지**하는 구성 방식. 고가용성 확보가 목적이다.

**② Read Replica 구성**

보통 웹 서비스는 **쓰기보다 읽기가 훨씬 많다.** 커뮤니티 서비스를 예로 들면:

```
애플리케이션
 ├─ INSERT / UPDATE / DELETE → Primary DB
 │                                  ↓ 비동기 복제
 └─ SELECT (다수) ──────→ Read Replica A, Read Replica B
```

- 글 작성·수정·삭제 → **Primary DB**로
- 게시글 목록 조회 같은 읽기 트래픽 → **Read Replica**로 분산

##### 실무적으로 적용해야 할 원칙

**① RDS 백업 기능을 100% 신뢰하지 말 것**
- 자동 백업이 제공되는 건 맞지만, **실제 복구가 정상적으로 되는지는 별도로 검증**되지 않는다
- 에러 발생 시점의 트랜잭션, 그리고 그 이후 발생한 트랜잭션에 대한 조치를 따로 고민해야 한다
- 예: Redis(인메모리 DB)에 **실패한 트랜잭션을 일정 기간 모아두는** 방식으로 보완할 수 있다

**② DB 연결 수를 무한정 늘리지 말 것**
- DB에는 **커넥션 풀**이 있고, 동시 연결 수에는 제한이 있다
- 그 제한을 초과하는 연결은 받을 수 없다
- 커넥션 풀을 유지하는 데에도 자원(메모리 등)이 소모된다

##### RDS 정리

| 기억할 내용 | 설명 |
|---|---|
| RDS | AWS의 관리형 관계형 데이터베이스 서비스 |
| DB Engine | MySQL, PostgreSQL 같은 실제 DB 제품 |
| DB Instance | DB 엔진이 실행되는 관리형 서버 자원 |
| Endpoint | 애플리케이션이 DB에 연결하는 주소 |
| DB Subnet Group | RDS를 배치할 VPC Subnet 묶음 |
| Security Group | 어떤 서버가 DB 포트에 접근할지 제어 |
| Automated Backup | 자동 백업과 특정 시점 복구 지원 |
| Multi-AZ | 장애 대응과 고가용성 목적 |
| Read Replica | 읽기 성능 확장 목적 |

### 🗨️ 오늘의 회고

- S3가 "객체 스토리지"이고 "파일 시스템이 아니다"라는 점이 처음엔 헷갈렸는데, Key가 폴더 경로처럼 보여도 실제로는 그냥 하나의 문자열일 뿐이라는 설명을 듣고 EFS와의 차이가 명확해졌다.
- RDS와 EC2 직접 설치 비교에서 비용 차이(20~30만원 vs 60~80만원)가 꽤 크게 느껴졌는데, 초기에는 개발 속도가 더 중요하다라는 말이 와닿았다. 무조건 저렴한 선택이 정답은 아니라는 점.
- Read Replica 구성에서 "쓰기는 Primary, 읽기는 Replica로"라는 원칙이 어제 배운 보안 그룹 참조 방식과도 연결된다. `sg-app`이 `sg-db`에 접근하는 구조처럼, 결국 역할별로 트래픽 경로를 분리하는 게 AWS 설계의 일관된 패턴인 것 같다.
- "RDS 백업을 100% 신뢰하지 말라"는 말이 가장 기억에 남는다. 관리형 서비스를 쓴다고 해서 장애 대응을 AWS에 완전히 떠넘길 수 있는 게 아니라, 애플리케이션 레벨에서도 실패 트랜잭션을 다루는 보완 장치를 같이 고민해야 한다는 걸 배웠다.
