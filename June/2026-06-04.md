#### | 파일 처리

##### 1. 문제 정의

커뮤니티에서 이미지 파일 사용 시나리오
- 이미지 파일을 업로드한다
- 업로드된 이미지 파일에 접근한다

##### 2. 개략적 설계

**지원 파일 형태**
- `jpg`, `png`, `webp` 지원
- 고용량 파일 및 기타 이미지 형식은 미지원
- 파일 크기 제한 : 10MB

**압축 및 변환 전략**
- 지원 파일은 모두 `jpg`와 `webp` 두 가지 포맷으로 변환 후 저장
- 브라우저가 webp를 지원하면 webp 제공, 아니면 jpg 제공

| 유형 | 압축률 |
|---|---|
| 프로필 사진 | 60% |
| 게시글 원본 | 20% |
| 게시글 썸네일 | 60% |

**데이터 저장**
- 파일 자체 : 스토리지(로컬 or 클라우드)에 저장
- 파일 메타데이터 : RDB의 `images` 테이블에 저장

저장 항목 : 스토리지 접근 경로, 썸네일 접근 경로, 생성일, 활성 여부 (비활성 이미지는 배치 프로그램으로 주기적 삭제)

**개략적 설계 흐름**

```
클라이언트 → Web Server → WAS → Storage (파일 저장)
                              ↕
                              DB (파일 정보 저장)
```

##### 3. 코드 책임 설계

```
controller/
  └─ ImageController     : 이미지 전달받는 외부 API 컨트롤러
service/
  ├─ ImageService        : 비즈니스 로직 (DB + 파일서비스 호출)
  ├─ FileService         : 파일 저장/로드 책임 (인터페이스)
  └─ ImageProcessor      : 변환/압축 책임
repository/
  └─ ImageRepository     : DB 접근
domain/
  └─ ImageEntity         : 이미지 메타정보 (파일명, 타입, 경로 등)
```

> 💡 **클래스 단일 책임 원칙** : 각 클래스는 하나의 명확한 책임만 가진다.

**LocalFileService — 로컬 파일 저장 구현체**

```java
@Service
public class LocalFileService implements FileService {

    @Value("${file.upload-dir}")
    private String uploadDir;

    @Override
    public String uploadFile(MultipartFile file) {
        try {
            // 디렉토리 없으면 생성
            Path directory = Paths.get(uploadDir).toAbsolutePath().normalize();
            Files.createDirectories(directory);

            // 파일명 충돌 방지
            String filename = UUID.randomUUID() + "_" + file.getOriginalFilename();
            Path targetPath = directory.resolve(filename);

            // 파일 저장
            Files.copy(file.getInputStream(), targetPath);
            return filename;
        } catch (IOException e) {
            throw new RuntimeException("파일 저장 실패", e);
        }
    }
}
```

**ImageService — 비즈니스 로직**

```java
@Service
public class ImageService {

    @Autowired
    private FileService fileService;

    @Autowired
    private ImageProcessor imageProcessor;

    @Autowired
    private ImageRepository imageRepository;

    public ImageEntity handleImageUpload(MultipartFile file, String type) {
        // 1. 원본 이미지 변환 (JPG, WEBP)
        var processedFiles = imageProcessor.processImage(file, type);

        // 2. 변환된 파일 업로드
        String jpgPath = fileService.uploadFile(processedFiles.getJpgFile());
        String webpPath = fileService.uploadFile(processedFiles.getWebpFile());

        // 3. 메타데이터 DB 저장
        ImageEntity entity = new ImageEntity();
        entity.setOriginalName(file.getOriginalFilename());
        entity.setType(type);
        entity.setJpgPath(jpgPath);
        entity.setWebpPath(webpPath);

        return imageRepository.save(entity);
    }
}
```

#### | 스프링 예외처리

API는 항상 같은 요청/응답 규격을 유지해야 한다. 에러 처리에서도 동일한 규격이 필요하다.

##### try-catch의 한계 (백엔드에서)

- 서비스마다, 컨트롤러마다 중복되는 예외 처리 코드
- 가독성 저하, 유지보수 어려움
- 예외 처리 흐름 불명확

##### 스프링 기본 제공 — BasicErrorController

- 예외 발생 시 에러 요청을 WAS에 전달 → `BasicErrorController`가 처리
- 별도 설정 없으면 JSON으로 응답하지만, 클라이언트 입장에서 유용하지 않은 정보

```json
{
    "timestamp": "2021-12-31T03:35:44.675+00:00",
    "status": 404,
    "error": "Not Found",
    "path": "/product/5000"
}
```

##### 예외처리 방법 3가지

**1. @ResponseStatus**

예외 클래스에 HTTP 상태 코드를 직접 지정

```java
@ResponseStatus(value = HttpStatus.NOT_FOUND)
public class NoSuchElementFoundException extends RuntimeException {
    ...
}
```

- `@ExceptionHandler`와 함께 메서드에 사용 가능
- `@RestControllerAdvice`와 함께 클래스에 사용 가능

**2. ResponseStatusException**

스프링 5에서 추가. `HttpStatus`와 함께 reason, cause를 선택적으로 추가 가능

```java
@GetMapping("/product/{id}")
public ResponseEntity<Product> getProduct(@PathVariable String id) {
    try {
        return ResponseEntity.ok(productService.getProduct(id));
    } catch (NoSuchElementFoundException e) {
        throw new ResponseStatusException(HttpStatus.NOT_FOUND, "Item Not Found");
    }
}
```

- ✅ 예외 처리 빠르게 적용, HttpStatus 설정으로 예외 클래스 결합도 낮음
- ❌ 직접 예외 처리 프로그래밍, 일관된 처리 어려움, 코드 중복 발생

**3. @ExceptionHandler**

매우 유연하게 에러를 처리. Exception 클래스를 속성으로 받아 처리할 예외를 지정

```java
@RestController
@RequiredArgsConstructor
public class ProductController {

    private final ProductService productService;

    @GetMapping("/product/{id}")
    public Response getProduct(@PathVariable String id) {
        return productService.getProduct(id);
    }

    @ExceptionHandler(NoSuchElementFoundException.class)
    public ResponseEntity<String> handleNoSuchElementFoundException(
            NoSuchElementFoundException exception) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
                             .body(exception.getMessage());
    }
}
```

- ❌ 컨트롤러에 요청 처리 + 에러 처리 책임이 혼재, 예외 처리 코드 중복
- → **전역으로 처리하는 기술이 필요**

##### @ControllerAdvice / @RestControllerAdvice

스프링 전역에 `@ExceptionHandler`를 적용. 여러 컨트롤러에서 발생하는 예외를 한 곳에서 처리.

| 구분 | 설명 |
|---|---|
| `@ControllerAdvice` | 전역 예외 처리 + View 반환 가능 |
| `@RestControllerAdvice` | 전역 예외 처리 + `@ResponseBody` 포함 → JSON 응답 |

**이점**
- 하나의 클래스로 모든 컨트롤러 전역 예외 처리
- 직접 정의한 에러 응답을 일관성 있게 전달
- 별도의 try-catch 없이 코드 가독성 향상

**주의**
- 프로젝트당 하나의 ControllerAdvice 관리
- 직접 구현한 Exception 클래스는 한 곳에서 관리

##### 4. @RestControllerAdvice 구현

```java
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler extends ResponseEntityExceptionHandler {

    // 우리가 만든 RestApiException 처리
    @ExceptionHandler(RestApiException.class)
    public ResponseEntity<Object> handleCustomException(RestApiException e) {
        ErrorCode errorCode = e.getErrorCode();
        return handleExceptionInternal(errorCode);
    }

    // 스프링에서 만들어둔 IllegalArgumentException 처리
    @ExceptionHandler(IllegalArgumentException.class)
    public ResponseEntity<Object> handleIllegalArgument(IllegalArgumentException e) {
        log.warn("handleIllegalArgument", e);
        ErrorCode errorCode = CommonErrorCode.INVALID_PARAMETER;
        return handleExceptionInternal(errorCode, e.getMessage());
    }

    // 에러 응답 만드는 팩토리 메서드 (반복되는 코드 분리)
    private ErrorResponse makeErrorResponse(ErrorCode errorCode) {
        return ErrorResponse.builder()
                .code(errorCode.name())
                .message(errorCode.getMessage())
                .build();
    }

    private ErrorResponse makeErrorResponse(ErrorCode errorCode, String message) {
        return ErrorResponse.builder()
                .code(errorCode.name())
                .message(message)
                .build();
    }
}
```

#### | 필터와 인터셉터

##### 서블릿 컨테이너와 Spring MVC 구조

```
클라이언트 → HTTP Request
    ↓
서블릿 컨테이너 (Servlet Container)
    1. Servlet Req/Res 객체 생성
    2. 설정 파일로 매핑할 서블릿 확인 (없으면 생성)
    3. 컨테이너에 스레드 만들고 Req, Res 전달
    ↓
Dispatcher Servlet (Spring Container)
    ↓ Handler Mapping → Handler Adapter → Controller → Service → Repository
    ↓
응답 전달
```

- **서블릿 컨테이너** : 서블릿을 담아두고 관리하는 바구니 (라이프 사이클 관리 등)
- **Dispatcher Servlet** : 모든 요청을 받아 적절한 컨트롤러로 라우팅하는 프론트 컨트롤러 패턴

---

##### 1. 필터 (Filter)

서블릿 컨테이너가 Dispatcher Servlet 요청을 전달하기 **전/후**에 URL 패턴에 맞는 모든 요청에 대해 부가 처리

```
클라이언트 → [Servlet Context: Filter Chain] → Dispatcher Servlet → Controller
                스프링 컨테이너로 요청 전달 전 서블릿 컨테이너에서 필터를 먼저 거침
```

**필터 특징**
- 서블릿, 정적 콘텐츠에 대한 요청/응답에 필터링 작업
- **스프링 프레임워크와 무관하게 작동** 가능

**필터 주요 용도**
- 공통 보안, 인증/인가 작업
- 모든 요청에 대한 로깅, 감사
- 이미지 데이터 압축 및 문자열 인코딩
- 스프링과 분리되어야 하는 기능

**Filter 인터페이스 — 3개의 메서드**

```java
public interface Filter {
    public default void init(FilterConfig filterConfig) throws ServletException {}
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) throws IOException, ServletException;
    public default void destroy() {}
}
```

| 메서드 | 설명 |
|---|---|
| `init()` | 필터 객체 초기화. 웹 컨테이너가 1회 호출 |
| `doFilter()` | URL 패턴에 맞는 모든 요청이 Dispatcher Servlet 전달 전 실행. `FilterChain`으로 다음 필터에 요청 전달 |
| `destroy()` | 필터 객체를 서비스에서 제거. 사용한 자원 반환. 웹 컨테이너에 의해 1번 호출, 이후 doFilter 처리 없음 |

**Custom Filter 구현 예시**

```java
public class CustomFilter implements Filter {

    @Override
    public void init(FilterConfig filterConfig) throws ServletException {
        Filter.super.init(filterConfig);
    }

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) throws IOException, ServletException {
        System.out.println("log: filter doFilter Method");
        chain.doFilter(request, response); // 다음 필터로 요청 전달
    }

    @Override
    public void destroy() {
        Filter.super.destroy();
    }
}
```

**필터 체인 (Filter Chain)**

```
클라이언트 → Filter → Filter → Filter → Dispatcher Servlet
                서블릿 컨테이너가 관리하는 필터들의 연결고리
                필터 등록 순서에 따라 순차 처리
                각 필터에서 doFilter() 호출로 다음 필터 호출
```

##### 2. 인터셉터 (Interceptor)

Dispatcher Servlet이 **컨트롤러 호출 전/후**에 호출되는 기능

```
클라이언트 → Filter Chain → Dispatcher Servlet → Interceptor → Controller
                        디스패처 서블릿에서 컨트롤러 요청 전 인터셉터를 먼저 거침
```

**인터셉터 특징**
- **스프링이 제공하는 기술** (필터와 다르게 스프링 컨텍스트에서 동작)
- 요청과 응답을 참조, 가공하는 기능
- 스프링 빈에 등록된 객체들(`Repository`, `Service`)을 사용할 수 있다

**인터셉터 주요 용도**
- 세부적인 보안, 인증/인가 작업
- API 호출에 대한 로깅 또는 감사
- 컨트롤러로 넘겨주는 정보 가공

**HandlerInterceptor 인터페이스 — 3개의 메서드**

```java
public interface HandlerInterceptor {

    default boolean preHandle(HttpServletRequest request,
                               HttpServletResponse response,
                               Object handler) throws Exception {
        return true;
    }

    default void postHandle(HttpServletRequest request,
                             HttpServletResponse response,
                             Object handler,
                             @Nullable ModelAndView modelAndView) throws Exception {}

    default void afterCompletion(HttpServletRequest request,
                                  HttpServletResponse response,
                                  Object handler,
                                  @Nullable Exception ex) throws Exception {}
}
```

| 메서드 | 실행 시점 | 특징 |
|---|---|---|
| `preHandle()` | 컨트롤러 호출 **전** | 반환값 true → 다음으로 / false → 작업 중지 |
| `postHandle()` | 컨트롤러 호출 **후** | 컨트롤러 하위에서 예외 발생 시 호출 안 됨 |
| `afterCompletion()` | 모든 작업 **완료 후** | 예외 발생해도 항상 호출됨. 리소스 반환 등 |

---

##### 필터 vs 인터셉터 핵심 비교

| 구분 | 필터 | 인터셉터 |
|---|---|---|
| **관리 컨테이너** | 서블릿 컨테이너 | 스프링 컨테이너 |
| **스프링 컨텍스트 접근** | ❌ | ✅ |
| **Req, Res 객체 조작** | ✅ | ❌ |
| **스프링 빈 사용** | ❌ | ✅ |
| **주요 용도** | 공통 보안/인증·인가, 전역 로깅, 이미지 압축 및 인코딩 | 세부 보안/인증·인가, API 로깅·감사, 컨트롤러 정보 가공 |

##### AOP와 필터, 인터셉터의 역할 구분

| 기술 | 주 역할 |
|---|---|
| **필터** | 전역 로직 (인코딩, XSS 필터링 등) |
| **인터셉터** | 컨트롤러 관련 로직 (요청별 공통 데이터 주입 등) |
| **AOP** | 스프링 빈 메서드 관련 로직 (트랜잭션 등) |

---

##### 인증/인가를 어디서 구현해야 할까?

> **인증과 인가는 모두 필터에서 하는 것이 일반적으로 유리하다.**

요청 흐름: `유저 → 서블릿 → 필터 → 스프링 → Dispatcher Servlet → 인터셉터 → 컨트롤러 → 서비스`

**필터에서 인증/인가를 처리하는 이유**
- HTTP 요청을 제일 먼저 바라보는 곳이 서블릿 컨테이너
- `Req` 객체에서 `Authorization` 헤더(JWT, 세션, API Key)를 바로 확인 가능
- 인증/인가가 안 된 요청을 굳이 스프링까지 넘길 필요 없음 → **필터 단에서 401, 403으로 Reject**
- 인증 필터 1번 → 인가 필터 2번 순서로 처리
- 필터를 거친 모든 요청은 "인증과 인가 정보가 담긴 요청"으로 인식

**인터셉터에서 처리하는 경우**
- 건물 입장 게이트(필터)는 통과했지만, 특정 방(URL)에 대한 세부적인 인가가 필요한 경우
- 스프링 빈(`Repository`, `Service`)을 활용한 권한 검증이 필요한 경우

##### 실제 구현 예시 — JWT 인증 필터 + 인가 인터셉터 + AOP 로깅

**JWT 인증용 필터**

```java
public class JwtAuthFilter extends OncePerRequestFilter {

    private final JwtProvider jwtProvider;

    public JwtAuthFilter(JwtProvider jwtProvider) {
        this.jwtProvider = jwtProvider;
    }

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                     HttpServletResponse response,
                                     FilterChain filterChain)
            throws ServletException, IOException {

        String authHeader = request.getHeader("Authorization");

        if (authHeader != null && authHeader.startsWith("Bearer ")) {
            String token = authHeader.substring(7);

            // 토큰 검증
            if (jwtProvider.validateToken(token)) {
                String userId = jwtProvider.getUserId(token);

                // 유저 정보를 SecurityContext에 저장
                var auth = new UsernamePasswordAuthenticationToken(userId, null, null);
                SecurityContextHolder.getContext().setAuthentication(auth);
            } else {
                response.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
                return;
            }
        }

        filterChain.doFilter(request, response);
    }
}
```

**인가용 인터셉터**

```java
public class RoleCheckInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest request,
                              HttpServletResponse response,
                              Object handler) throws Exception {

        Authentication auth = SecurityContextHolder.getContext().getAuthentication();

        if (auth == null || auth.getPrincipal() == null) {
            response.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
            return false;
        }

        // 예: 관리자만 접근 가능한 엔드포인트
        if (request.getRequestURI().startsWith("/admin")) {
            String userId = (String) auth.getPrincipal();

            // DB에서 관리자 여부 확인 가능 (스프링 빈 사용 가능)
            if (!"1".equals(userId)) {
                response.setStatus(HttpServletResponse.SC_FORBIDDEN);
                return false;
            }
        }

        return true;
    }
}
```

**JWT 토큰 생성 및 검증 — JwtProvider**

```java
public class JwtProvider {

    private final Key key = Keys.hmacShaKeyFor(
        "my-secret-key-1234567890abcdef".getBytes());
    private final long expiration = 1000 * 60 * 60; // 1시간

    public String generateToken(Long userId) {
        return Jwts.builder()
                .setSubject(String.valueOf(userId))
                .setIssuedAt(new Date())
                .setExpiration(new Date(System.currentTimeMillis() + expiration))
                .signWith(key, SignatureAlgorithm.HS256)
                .compact();
    }

    public boolean validateToken(String token) {
        try {
            Jwts.parserBuilder().setSigningKey(key).build().parseClaimsJws(token);
            return true;
        } catch (JwtException e) {
            return false;
        }
    }

    public String getUserId(String token) {
        Claims claims = Jwts.parserBuilder()
                .setSigningKey(key).build()
                .parseClaimsJws(token)
                .getBody();
        return claims.getSubject();
    }
}
```

### 🗨️ 오늘의 회고

- 백엔드 설계 11단계 프로세스가 인상적이었다. "개발은 건축이다"라는 비유처럼, 기반 설계가 탄탄해야 이후 도메인 추가가 찍어내듯 빨라진다는 것.
- 파일 처리에서 `FileService`를 인터페이스로 추상화해 로컬/클라우드 구현체를 분리하는 구조가 OCP와 DIP를 실제로 적용한 좋은 예시였다.
- 예외 처리에서 `@RestControllerAdvice` + `ErrorCode Enum` + `RestApiException` 조합이 실무 표준 패턴이라는 것을 알게 됐다. 에러 코드를 enum으로 관리하면 도메인별로 분리도 깔끔하고 유지보수도 좋아진다.
- 필터 vs 인터셉터의 차이가 단순히 "위치 차이"가 아니라 "서블릿 컨텍스트 vs 스프링 컨텍스트"의 차이라는 것이 핵심이었다. 인증은 필터(JWT 검증), 세부 인가는 인터셉터(스프링 빈 접근 필요), 메서드 레벨 공통 로직은 AOP로 역할을 나누는 구조가 명확해졌다.
- 과제 구현에 속도를 내야 할 것 같다. 오늘 API 구현을 완료하는 게 목표.
