##### 도메인

이벤트 중인 가성비 도메인(`.xyz`)을 최소 비용으로 직접 구매했다. 직접 DNS A 레코드를 설정해 보면서, 실제 상용 서비스가 도메인과 인프라를 매핑하는 과정을 온전히 경험해 보고 싶었기 때문이다.

##### 인프라 아키텍처 — 단일 EC2 1대 제약 조건 활용

과제 요구사항인 **EC2 1대만 사용**하는 조건을 충족하기 위해, 한 대의 `t3.micro` 서버 안에서 프론트엔드 웹 서버와 자바 백엔드 서버를 동시에 구동하는 구조를 설계했다.

> 💡 가상 머신 내부의 포트를 쪼개고, **Nginx를 앞단에 세워 트래픽을 교통정리**하는 능력을 기르는 데 집중한 구조다.

```
사용자 → 80포트(Nginx) → ┬─ Spring Boot (8080)
                          └─ Frontend Express (3000)
```

##### 트러블슈팅 1 — 프론트엔드 배포 중 `Missing script: "build"` 에러

| 항목 | 내용 |
|---|---|
| **발생 문제** | 프론트엔드 레포지토리를 클론받고 습관적으로 `npm run build`를 실행했으나, 스크립트가 없다는 에러와 함께 컴파일이 실패함 |
| **원인 분석** | `package.json`을 열어보니 일반적인 React/Vite 같은 CSR 빌드 구조가 아니었음. HTML, CSS, JS 정적 리소스를 내장된 Express 서버(`app.js`)가 직접 띄워주는 **내장 웹 서버 형태의 FE 구조**였음 |
| **해결 조치** | 빌드 프로세스가 필요 없는 구조임을 인지하고, 터미널을 닫아도 서버가 죽지 않도록 프로세스 매니저인 **PM2**를 도입 → `pm2 start app.js --name "community-frontend"`로 무중단 구동에 성공 |

> **PM2란?**
> PM2는 **P**(Process) **M**(Manager) **2**의 약자로, Node.js 프로세스를 관리하기 위한 패키지다.
> - 서비스 도중 갑자기 서버가 중지되어도 **자동으로 다시 켜준다**
> - Node.js는 싱글 스레드 기반이지만, **멀티 코어/하이퍼 스레딩**을 활용할 수 있게 해준다
> - 클라이언트 요청이 오면 여러 노드 프로세스에 고르게 분배한다 (**로드 밸런싱**)
>
> 💡 리눅스나 맥에서 PM2 실행 시, 1024번 이하의 포트를 사용하려면 관리자 권한이 필요하다. 따라서 명령어 앞에 `sudo`를 붙여 실행한다. (`sudo npm start`, `sudo npx pm2 list`, `sudo npx pm2 kill`, `sudo npx pm2 monit` 등)
> 출처: [Inpa Dev — Node.js PM2 모듈 사용법](https://inpa.tistory.com/entry/node-📚-PM2-모듈-사용법-클러스터-무중단-서비스)

##### 트러블슈팅 2 — OS 패키지 제약으로 인한 Java 24 설치 실패와 수동 바이너리 매핑

| 항목 | 내용 |
|---|---|
| **발생 문제** | `sudo apt install openjdk-24-jdk` 실행 시 `Unable to locate package` 에러 발생. 외부 PPA 저장소를 등록해도 최신 우분투 버전 트리 충돌로 404 에러가 나며 설치가 완전히 막힘 |
| **원인 분석** | OS의 기본 패키지 매니저가 제공하는 인덱스 범위를 넘어서는 **최신 런타임 버전**이었기 때문에 자동 설치가 불가능한 환경적 제약 |
| **해결 조치** | 자동 설치를 포기하고, 오라클 정식 보관소에서 리눅스 x64 타깃 공식 바이너리 압축 파일(`jdk-24.0.1_linux-x64_bin.tar.gz`)을 `wget`으로 직접 서버에 다운로드 |

```bash
# /opt/ 폴더에 압축 해제 후, ~/.bashrc 에 환경 변수 직접 주입
export JAVA_HOME=/opt/jdk-24.0.1
export PATH=$JAVA_HOME/bin:$PATH
```

> **결과:** 자바 24를 정착시킨 후 `./gradlew build -x test`를 통해 `.jar` 빌드에 성공했다.

#### | 아키텍처 표준화 및 Nginx 최적화 의사결정

##### 초기 포트 뒤엉킴 문제

모든 서버를 켜고 도메인을 연결했으나, 화면에 프로젝트 대신 **Nginx 기본 웰컴 페이지**만 떴다.

원인을 확인해 보니:
- 프론트엔드가 비표준 포트인 `8080`을 사용
- 백엔드가 `8081`을 사용 (개발 과정에서 변경했었음)
- 전면의 Nginx는 프론트엔드가 `3000`번 포트에 있다고 가정하고 포워딩 설정을 해둔 상태

> ⚠️ 즉, 실제 구동 포트와 Nginx 설정상의 포트가 서로 어긋나 있던 것이 원인이었다.

##### 소스 코드 리팩토링을 통한 표준 포트 확립

전체 아키텍처의 복잡도를 낮추기 위해 **소스 코드를 수정**하기로 결정했다.

1. 프론트엔드 `app.js` 코드를 수정해 구동 포트를 공인 표준 관례인 **`3000`**으로 변경
2. 백엔드(Spring Boot)는 기본 포트 명세인 **`8080`**으로 원복하여 포트 충돌 가능성 배제
3. 주소창의 지저분한 `/v1` 식별자 프리픽스를 제거하여 깔끔한 URL 구조 채택 (백엔드 코드는 그대로 두고, **프론트만 수정**)
4. `/api` 같은 식별자도 동일한 방식으로 정리

| 구성 요소 | 변경 전 | 변경 후 |
|---|---|---|
| 프론트엔드 포트 | 8080 | **3000** |
| 백엔드 포트 | 8081 | **8080** |
| API 식별자 | `/v1/...` | 프리픽스 제거 |

##### 트러블슈팅 4 — EC2 서버 프리징(멈춤) 및 무한 로딩 현상

| 항목 | 내용 |
|---|---|
| **문제 상황** | `git pull` 이후 자바 백엔드를 다시 빌드하고 배포하는 과정에서 터미널이 완전히 멈추고, SSH 접속조차 불가능한 현상이 반복됨 |
| **원인 분석** | AWS `t2.micro` 인스턴스의 메모리가 **1GB**로 매우 협소하여, 무거운 Spring Boot 애플리케이션의 빌드·기동 과정을 버티지 못하고 **OOM(Out Of Memory)**으로 서버가 다운됨. 게다가 데이터베이스 미연결로 인해 PM2가 자바를 **무한 재부팅**하면서 CPU·RAM 자원을 추가로 고갈시킴 |

**해결 방법**

1. AWS EC2 콘솔에서 인스턴스를 강제 재부팅
2. 하드디스크 공간을 활용해 **2GB 크기의 Swap File**을 주입하여 총 가용 RAM을 3GB 수준으로 확보

```bash
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
```

3. PM2 구동 시 자바의 최대 메모리 사용량을 제한하는 옵션(`-Xmx400m`)을 부여하고, 무거운 백엔드보다 프론트엔드를 먼저 켜는 **시차 기동** 방식을 적용해 자원 과부하를 방지

> 💡 작은 인스턴스에서는 메모리 부족이 단순한 "느려짐"이 아니라 **SSH 접속 자체가 끊기는 전면적인 장애**로 이어질 수 있다는 걸 직접 경험했다. Swap과 메모리 상한 옵션 둘 다 필요한 이유를 체감했다.

##### Nginx 에러 인터셉트 기반 폴백(Fallback) 라우팅 구현

주소창에서 `/v1`, `/api` 같은 식별자가 사라지자, Nginx가 **화면 요청과 API 요청을 구별하지 못하는 문제**가 발생했다. 이를 해결하기 위해 **에러 인터셉트 기반의 폴백 구조**를 세웠다.

> 핵심 아이디어: 모든 요청을 일단 백엔드(8080)로 먼저 보내고, 백엔드가 모르는 경로라서 404가 나면 그 에러를 Nginx가 가로채서 프론트엔드(3000)로 다시 돌려보낸다.

```nginx
server {
    listen 80;
    server_name lulu-roh.xyz www.lulu-roh.xyz;

    location / {
        # 사용자의 모든 요청을 최우선으로 자바 백엔드(8080)로 전달
        proxy_pass http://localhost:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 백엔드에 없는 주소라 404 에러가 나면 Nginx가 중간에서 가로챔
        proxy_intercept_errors on;
        error_page 404 = @frontend;
    }

    # 가로챈 에러 트래픽을 프론트엔드(3000) 런타임 공간으로 리다이렉션
    location @frontend {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

| 동작 흐름 | 설명 |
|---|---|
| 1 | 사용자가 `lulu-roh.xyz/anything`으로 요청 |
| 2 | Nginx가 우선 백엔드(8080)로 요청을 전달 |
| 3 | 백엔드가 해당 경로를 모르면 404 응답 |
| 4 | `proxy_intercept_errors on` 설정으로 Nginx가 그 404를 가로챔 |
| 5 | `error_page 404 = @frontend`에 따라 프론트엔드(3000)로 재전달 |
| 6 | 프론트엔드가 화면을 렌더링해서 사용자에게 응답 |

> ⚠️ `proxy_intercept_errors on`이 핵심이다. 이 옵션이 없으면 Nginx는 백엔드가 보낸 404 응답을 그대로 사용자에게 전달해버려서, 폴백 라우팅 자체가 동작하지 않는다.

### 🗨️ 오늘의 회고

- EC2 1대라는 제약 조건이 오히려 Nginx의 역할(트래픽 분기, 에러 인터셉트, 리버스 프록시)을 제대로 체감하게 해준 계기가 됐다. 여러 대로 나눠서 배포했다면 이런 라우팅 고민을 깊게 할 일이 없었을 것 같다.
- 로컬에서 잘 되던 빌드 스크립트가 서버에서 안 통한 사례(`npm run build` 부재)는, 프로젝트 구조를 제대로 파악하지 않고 관습적으로 명령어를 치는 습관을 돌아보게 했다. `package.json`을 먼저 열어보는 습관이 필요하다.
- Java 24처럼 너무 최신 버전을 쓰면 OS 기본 패키지 매니저의 지원 범위를 벗어날 수 있다는 걸 직접 겪었다. 운영 환경에서는 안정적으로 패키지 매니저가 지원하는 LTS 버전을 쓰는 것도 고려해볼 만하다.
- `t2.micro`(1GB RAM)에서 Spring Boot를 직접 빌드하고 띄우는 건 메모리 측면에서 상당히 빠듯하다는 걸 몸으로 배웠다. Swap 메모리 추가 + JVM 힙 제한(`-Xmx`) + 기동 순서 조정까지, 작은 스펙의 서버를 다룰 때 필요한 운영 기법들을 한 번에 적용해본 경험이었다.
- 포트 뒤엉킴 문제는 결국 "설정값과 실제 값이 다를 수 있다"는 아주 기본적인 원인이었는데, 인프라 전체를 점검하게 만든 트리거였다. 다음부터는 포트 같은 핵심 설정값을 README나 별도 문서에 명시해두는 습관을 들여야겠다.
- `proxy_intercept_errors`를 이용한 폴백 라우팅은 처음 접해본 패턴이라 신기했다. API 서버와 프론트 서버를 분리해서 운영할 때, 굳이 URL 프리픽스로 구분하지 않고도 자연스럽게 트래픽을 분기시킬 수 있다는 아이디어가 인상 깊었다.
