#### | 값 타입과 임베디드 매핑

JPA에서 값 타입은 3가지로 나뉜다.

##### 1. 기본 값 타입

단일 값(이름, 나이, 금액)으로만 의미를 가지는 데이터를 엔티티 필드로 직접 정의해 컬럼에 저장하는 방식

| 분류 | 예시 |
|---|---|
| 자바 기본형 | `int`, `long`, `double` |
| 래퍼 클래스 | `Integer`, `Long`, `Double` |
| 문자열 | `String` |

> 💡 **`long` vs `Long` 차이**
> `long`은 null 불가, `Long`은 null 허용.
> DB 저장보다 객체 생성이 먼저 발생하기 때문에 ID 타입으로는 보통 `Long`을 사용한다.
> null 허용 여부는 `@Column(nullable = false)` 로 명시하는 것이 좋다.

##### 2. 임베디드 타입

두 개 이상의 값을 하나의 객체로 묶어 엔티티에 포함시켜 사용하는 타입.
**테이블 구조는 그대로, 엔티티 구조만 더 객체지향적으로** 만드는 것이 핵심.

```java
@Embeddable
public class Address {
    private String city;
    private String street;
    private String zipcode;

    public Address() {}
    public Address(String city, String street, String zipcode) {
        this.city = city;
        this.street = street;
        this.zipcode = zipcode;
    }
}

@Entity
public class User {
    @Id @GeneratedValue
    @Column(name = "user_id")
    private Long id;
    private String username;
    private int age;

    @Embedded
    private Address address; // city, street, zipcode 컬럼이 user 테이블에 그대로 포함됨
}
```

- **코드 재사용성 향상** : `Address` 타입을 다른 엔티티에서도 재사용 가능
- `@Embeddable` : 임베디드 타입 클래스에 선언
- `@Embedded` : 엔티티에서 임베디드 타입 필드에 선언

##### 3. 컬렉션 값 타입

엔티티에 직접 연관되지 않는 여러 개의 값을 List, Set 형태로 저장하기 위한 값 타입

```java
@Embeddable
public class Tag {
    private String name;
    protected Tag() {}
    public Tag(String name) { this.name = name; }
}

@Entity
public class Post {
    @Id @GeneratedValue
    private Long id;
    private String title;

    @ElementCollection
    @CollectionTable(name = "post_tags",
                     joinColumns = @JoinColumn(name = "post_id"))
    @Column(name = "tag_name")
    private Set<Tag> tags = new HashSet<>();
}
```

> ⚠️ 현업에서는 많이 쓰지 않는다. 컬렉션 값 타입은 변경 시 전체 삭제 후 재삽입하는 방식으로 동작해 성능 이슈가 있다. 실무에서는 별도 엔티티로 분리하는 것이 권장된다.

#### | JPQL (Java Persistence Query Language)

**엔티티 객체를 대상으로 데이터를 조회하거나 조작할 수 있도록 제공되는 객체 지향 쿼리 언어**

##### 왜 JPQL이 필요한가?

JPA만으로는 단일 엔티티 관리는 가능하지만, 복잡한 관계식·조인·조건 처리에는 한계가 있다.

**JPA만 사용할 때의 문제 — N+1 문제**

```java
// 1. 전체 게시글 조회 (쿼리 1번)
List<Post> posts = postRepository.findAll();

// 2. 루프를 돌며 작성자 닉네임 접근 → 추가 쿼리 N번 발생
for (Post post : posts) {
    // post.getAuthor()까지는 프록시, 실제 데이터 접근 순간 쿼리 발생
    System.out.println(post.getAuthor().getNickname());
}
// → 게시글 100개면 쿼리 1 + 100 = 101번 발생
```

**JPQL 사용 시**

```java
// 한 번의 쿼리로 조건 + 정렬 + Lazy 없이 한번에 처리
return em.createQuery(
    "SELECT p FROM Post p " +
    "WHERE p.user.id = :userId " +
    "AND p.deleted = false " +
    "ORDER BY p.createdAt DESC",
    Post.class)
    .setParameter("userId", userId)
    .getResultList();
```

##### JPQL 특징

- **엔티티 중심 쿼리** : DB 테이블이 아닌 Entity 기준으로 쿼리 작성
- **DB 종속 없음** : JPA 구현체가 사용하는 DB에 맞게 SQL을 자동 생성
- **객체 중심 개발** : SQL을 직접 작성하지 않아도 됨

##### JPQL 문법

```sql
-- 기본 구조 (테이블명이 아닌 엔티티명 사용!)
select 별칭
from 엔티티명 별칭
where 조건식
group by 그룹핑기준
having 그룹핑조건
order by 정렬기준

-- 예시
select u
from User u
where u.nickname = 'tester'
group by u.userRole
having count(u) > 1
order by u.createdAt desc
```

##### JPA에서 JPQL 사용하기

```java
// 반환 타입이 명확한 경우
TypedQuery<User> query = entityManager.createQuery("select u from User u", User.class);
List<User> users = query.getResultList();

// 반환 타입이 불명확한 경우
Query query = entityManager.createQuery("select u.nickname, u.createdAt from User u");
List<Object[]> result = query.getResultList();
```

**EntityManager 주요 메서드**

| 메서드 | 역할 |
|---|---|
| `createQuery` | 쿼리 생성 |
| `getResultList` | 여러 결과 가져오기 |
| `getSingleResult` | 결과 하나만 가져오기 |
| `executeUpdate` | 수정, 삭제 실행 |

##### 파라미터 바인딩

```java
// 이름 기반 (권장)
TypedQuery<User> query = em.createQuery(
    "select u from User u where u.nickname = :nickname", User.class);
query.setParameter("nickname", "tester");

// 위치 기반
TypedQuery<User> query = em.createQuery(
    "select u from User u where u.nickname = ?1", User.class);
query.setParameter(1, "tester");
```

##### 조건식

**NULLIF** : 두 값이 동일하면 null, 아니면 첫 번째 값 반환
```sql
select nullif(u.email, 'tester@adapterz.kr') from User u
```

**COALESCE** : 여러 값을 차례로 확인해 null이 아닌 첫 번째 값 반환
```sql
select coalesce(u.profileInfo.intro, u.email) from User u
```

**CASE** : 조건을 순서대로 검사해 해당 값 반환
```sql
select case
    when u.createdAt >= '2025-01-01' then '신규 회원'
    when u.createdAt <= '2020-01-01' then '장기 회원'
    else '일반 회원'
end
from User u
```

##### 경로 표현식 (조인)

| 방식 | 설명 | 권장 |
|---|---|---|
| **묵시적 조인** | join 키워드 없이 경로만 표현, 항상 내부 조인 | ❌ |
| **명시적 조인** | join 키워드로 직접 선언, 조인 구조 명확 | ✅ |

```sql
-- 묵시적 조인 (비권장)
select p.* from user u inner join post p on u.user_id = p.user_id

-- 명시적 조인 (권장)
select p.title
from user u
inner join post p on u.user_id = p.user_id
where u.username = 'kim'
```

##### Fetch Join

연관 엔티티를 한 번에 가져와 N+1 문제를 해결하는 방법

```sql
select p from Post p join fetch p.author
```

| 장점 | 단점 |
|---|---|
| N+1 해결, 필요한 데이터 미리 로딩 | 페이징 불가, 필요 이상 데이터 포함 가능 |

##### Projection (프로젝션)

엔티티 전체가 아닌 필요한 필드만 선택해 조회하는 방식

| 종류 | 설명 | 실무 빈도 |
|---|---|---|
| **엔티티 프로젝션** | SELECT에 엔티티 타입 지정, 전체 컬럼 조회 | 일반적 |
| **임베디드 타입 프로젝션** | SELECT에 임베디드 타입 지정 | 보통 |
| **스칼라 타입 프로젝션** | 숫자, 문자열 등 단순 값만 조회 | 보통 |
| **DTO 프로젝션** | 필요한 필드만 DTO 생성자로 매핑 | ✅ 실무 권장 |

#### | Spring Data JPA

**반복적인 CRUD 코드 없이 인터페이스만으로 리포지토리를 구현하는 방법**

```
Application
    ↓ Repository 사용
Spring Data JPA (Repository)
    ↓ 내부적으로
JPA → Hibernate → JDBC → Relational Database
```

실무에서 EntityManager를 직접 호출할 일은 많지 않다.
Spring Data JPA가 자주 쓰이는 JPA 동작들을 미리 구현해 제공한다.

##### 1. 기본 사용법

```java
@Repository
public interface PostRepository extends JpaRepository<Post, Long> {
    // JpaRepository 상속만으로 save(), findById(), findAll(), delete() 등 자동 제공
}
```

##### 2. Query Method

메서드 이름만으로 JPQL 쿼리를 자동 생성

```java
public interface UserRepository extends JpaRepository<User, Long> {

    List<User> findByNickname(String nickname);
    // → SELECT u FROM User u WHERE u.nickname = ?

    List<User> findByEmailAndNickname(String email, String nickname);
    // → WHERE u.email = ? AND u.nickname = ?

    List<User> findByNicknameOrderByIdDesc(String nickname);
    // → WHERE u.nickname = ? ORDER BY u.id DESC

    Optional<User> findByEmail(String email);
    // 단건 조회, 없으면 빈 Optional

    boolean existsByEmail(String email);
    // 존재 여부 확인

    long countByUserRole(UserRole role);
    // 개수 집계
}
```

> ⚠️ 메서드 이름이 길어지면 가독성이 떨어진다. 복잡한 조건은 `@Query` 또는 QueryDSL 사용.

##### 3. @Query

복잡한 조건, 조인, 집계 쿼리를 직접 작성

```java
public interface UserRepository extends JpaRepository<User, Long> {

    // 엔티티 조회
    @Query("select u from User u where u.email = :email")
    Optional<User> findByEmail(@Param("email") String email);

    // DTO 프로젝션
    @Query("select new com.example.dto.UserDto(u.email, u.nickname) " +
           "from User u where u.userRole = :role")
    List<UserDto> findUserDtos(@Param("role") UserRole role);

    // 페이징 + 정렬
    @Query("select u from User u where u.nickname like concat('%', :kw, '%')")
    Page<User> search(@Param("kw") String keyword, Pageable pageable);
}
```

##### 4. @EntityGraph

N+1 문제 방지. 특정 엔티티 조회 시 함께 로딩할 연관 엔티티를 지정

```java
@Repository
public interface PostRepository extends JpaRepository<Post, Long> {

    @EntityGraph(attributePaths = {"author"})
    Optional<Post> findById(Long id);
    // Post 조회 시 author를 LEFT JOIN FETCH로 한 번에 조회
}
```

##### 5. Pageable

페이징과 정렬을 손쉽게 처리하는 객체

```java
// Service
Pageable pageable = PageRequest.of(page, size, Sort.by("id").descending());
return postRepository.findByAuthorId(userId, pageable);

// Controller
@GetMapping("/user/{userId}")
public Page<Post> getPosts(
    @PathVariable Long userId,
    @RequestParam(defaultValue = "0") int page,
    @RequestParam(defaultValue = "10") int size) {
    return postService.getPostsByUser(userId, page, size);
}

// Page 사용
Page<User> page = userRepository.findAll(pageable);
page.getContent().forEach(u -> System.out.println(u.getNickname()));
```

##### 6. 사용자 정의 리포지토리

다른 방법으로 해결하기 어려운 경우 커스텀

```java
// 1. 커스텀 인터페이스
public interface UserRepositoryCustom {
    List<User> searchByCustom();
}

// 2. 구현 클래스 (리포지토리명 + "Impl" 필수 규칙)
public class UserRepositoryImpl implements UserRepositoryCustom {
    @Override
    public List<User> searchByCustom() { ... }
}

// 3. 레포지토리에 상속 추가
@Repository
public interface UserRepository extends JpaRepository<User, Long>, UserRepositoryCustom {
    // CRUD → JpaRepository / 커스텀 → UserRepositoryCustom
}
```

#### | QueryDSL

**자바 코드로 타입 안전한 쿼리를 작성할 수 있게 해주는 쿼리 라이브러리**

##### JPQL vs QueryDSL 비교

| 항목 | JPQL | QueryDSL |
|---|---|---|
| **쿼리 작성** | 문자열 | 자바 코드 (타입 안전) |
| **바인딩** | 수동 (`:name`) | 메서드 체인 (`u.name.eq`) |
| **동적 쿼리** | 문자열 이어붙이기 → 복잡 | 객체 조립하듯 간결 |
| **오류 발견** | 런타임 에러 | **컴파일 타임에 발견** |
| **IDE 자동완성** | 어려움 | 가능 |
| **성능** | 동일 (둘 다 JPQL → SQL 변환) | 동일 |

##### Q 클래스

QueryDSL의 동작 핵심. 빌드 시 엔티티를 기반으로 자동 생성되는 메타 모델 클래스

```java
QUser u = QUser.user;
QPost p = QPost.post;
```

##### 기본 설정

```java
@Configuration
public class QuerydslConfig {

    @PersistenceContext
    private EntityManager entityManager;

    @Bean
    public JPAQueryFactory jpaQueryFactory() {
        return new JPAQueryFactory(entityManager);
    }
}
```

##### 기본 사용법

```java
// 단순 조건 조회
User user = query.select(u)
                 .from(u)
                 .where(u.name.eq("Adapterz"))
                 .fetchOne();

// 조인 + 조건 + 정렬
List<Post> result = query
    .select(p)
    .from(p)
    .join(p.author, u)
    .where(u.name.eq("kim").and(p.createdAt.after(someDate)))
    .orderBy(p.createdAt.desc())
    .fetch();
```

##### 동적 쿼리 — BooleanBuilder

```java
BooleanBuilder builder = new BooleanBuilder();

if (value1 != null) {
    builder.and(entity.field1.eq(value1));
}
if (start != null && end != null) {
    builder.and(entity.field2.between(start, end));
}
if (keyword != null && !keyword.isBlank()) {
    builder.or(entity.field3.containsIgnoreCase(keyword));
}

List<Entity> results = queryFactory
    .selectFrom(entity)
    .where(builder) // 비어 있으면 where 절 생략
    .fetch();
```

##### 동적 쿼리 — Where 다중 파라미터 (더 권장)

조건을 메서드로 분리해 `null`이면 자동으로 무시하는 방식. **BooleanBuilder보다 가독성 및 재사용성이 좋다.**

```java
List<Post> searchPosts(String title, String content, Long minId, Long maxId) {
    return queryFactory
        .selectFrom(post)
        .where(
            titleContains(title),
            contentContains(content),
            idBetween(minId, maxId)
        )
        .fetch();
}

private BooleanExpression titleContains(String title) {
    return (title == null || title.isBlank()) ? null : post.title.containsIgnoreCase(title);
}

private BooleanExpression contentContains(String content) {
    return (content == null || content.isBlank()) ? null : post.content.containsIgnoreCase(content);
}

private BooleanExpression idBetween(Long minId, Long maxId) {
    return (minId == null || maxId == null) ? null : post.id.between(minId, maxId);
}
```

#### | 실무에서 뭘 쓰나? — 선택 기준

```
Spring Data JPA 쿼리 메서드   ← 기본. 대부분 여기서 해결
        ↓ 안 되면
사용자 정의 리포지토리 (커스텀 인터페이스)
        ↓ 그래도 복잡하면
QueryDSL  ← 동적 쿼리, 복잡한 조인

@Query  ← 배치 프로그램 등 단발성 복잡 쿼리에 따로 사용
```

| 방법 | 언제 쓰나 |
|---|---|
| **쿼리 메서드** | 단순 조건 조회 (기본값) |
| **@Query** | 단발성 복잡 쿼리, 배치 처리 |
| **커스텀 인터페이스** | 쿼리 메서드로 안 되는 경우 |
| **QueryDSL** | 동적 쿼리, 복잡한 조인, 타입 안전성이 필요한 경우 |

### 🗨️ 오늘의 회고

- 값 타입의 3가지(기본, 임베디드, 컬렉션)가 각각 어떤 상황에서 쓰이는지 정리됐다. 임베디드 타입은 "테이블은 그대로, 코드 구조만 객체지향적으로"라는 포인트가 핵심.
- N+1 문제가 왜 생기는지를 코드로 직접 보니 훨씬 와닿았다. Fetch Join과 `@EntityGraph`가 해결책이라는 것도 함께 이해됐다.
- JPQL vs QueryDSL 비교에서 "동일 조건이면 동일 성능(둘 다 SQL로 변환)"이라는 점이 흥미로웠다. QueryDSL을 선택하는 이유는 성능이 아니라 타입 안전성, 컴파일 타임 오류 감지, 동적 쿼리 편의성이라는 것.
- Spring Data JPA 선택 기준(쿼리 메서드 → 커스텀 → QueryDSL)을 실무 플로우로 정리해두니 앞으로 어떤 걸 먼저 시도해야 할지 기준이 생겼다.
- 오늘 수업 중 캡처를 많이 놓쳤는데 코드 예시들을 직접 보완하면서 오히려 더 잘 이해된 것 같다. 다음엔 놓치지 않도록 하자.
