#### | 공유 자원 (Shared Resource)

**여러 사용자나 프로세스, 스레드가 동시에 접근하거나 사용하는 시스템 내의 자원**

##### 왜 공유 자원이 필요한가?

CPU, 메모리, 파일, 네트워크 등 자원은 무한하지 않다.
모든 프로세스가 각각 별도의 자원을 요청하면 → 자원이 금방 소진된다.
→ 하나의 자원을 여러 작업이 함께 사용하는 **공유 자원** 제공

**공유 자원의 종류** : 메모리, 파일, 데이터베이스, 네트워크 소켓, 전역 변수

**공유 자원을 사용하는 이유**
- 메모리 절약
- 데이터 일관성 유지
- 프로세스 간 협업

##### 공유 자원의 문제점

동시에 공유 자원을 수정할 때 **경쟁 상태(Race Condition)** 가 발생한다.

```java
// 2개 이상의 스레드가 동시에 접근하면?
viewCount++;
// 1. 데이터 읽고
// 2. viewCount + 1 계산하고
// 3. 저장하고
// → 이 3단계 사이에 다른 스레드가 끼어들면 결과가 꼬인다
```

---

#### | 임계 영역 (Critical Section)

**공유 자원에 접근하는 프로그램 코드의 일부분. 동시에 여러 프로세스나 스레드가 접근하면 문제가 생길 수 있는 구간.**

```java
int balance = 1000;     // 공유 자원

// ↓↓ 임계 영역 시작 ↓↓
int temp = balance;     // 1. 공유 자원 읽기
temp += 100;            // 2. 값 수정
balance = temp;         // 3. 공유 자원에 다시 저장
// ↑↑ 임계 영역 끝 ↑↑
```

##### 임계 영역에서 발생하는 문제들

**1. 경합 조건 (Race Condition)**

여러 스레드가 동시에 공유 자원에 접근할 때 실행 순서에 따라 결과가 달라지는 문제

```
계좌 잔액 1000원, T1과 T2가 동시에 +100원 입금

기대 결과: 1200원
실제 결과: 1100원 (Lost Update 발생)

이유: T1이 1000원을 읽고 계산하는 사이에
      T2도 동일하게 1000원을 읽고 계산하면
      마지막에 저장되는 값이 1100원이 되어버린다
```

**2. 데이터 무결성 침해**

데이터가 논리적으로 맞지 않는 상태가 되는 문제

```
재고 1개, 두 개의 스레드가 동시에 주문

T1: "재고 있음" → 주문 생성
T2: "재고 있음" → 주문 생성 (동시에!)

결과: 재고 1개인데 주문 2개 생성 → 데이터 불일치
```

**실무에서 자주 발생하는 예시**

| 상황 | 문제 |
|---|---|
| 쿠폰 발급 | 한 명에게 쿠폰 2개 발급 |
| 결제 처리 | 결제가 2번 처리 |
| 메시지 큐 처리 | 같은 메시지를 여러 워커가 동시 처리 |
| 파일 쓰기 | 두 에디터가 동시에 같은 파일에 작성 |

> ⚠️ **왜 특히 위험한가?** 이런 문제들은 눈에 보이지 않는다. 가끔 이상 데이터가 쌓이는데, 재현하기도 어렵고 알아차리기도 힘들다.

**3. 교착 상태 (Deadlock)**

여러 스레드가 서로 가진 자원을 기다리며 영원히 진행하지 못하는 상태

```
스레드 A → 자원 1 점유 중, 자원 2를 기다리는 중
스레드 B → 자원 2 점유 중, 자원 1을 기다리는 중
→ 서로 기다리며 영원히 진행 불가
```

> ⚠️ 실무에서도 생각보다 자주 발생한다. 특정 임계 영역에 두 스레드가 특정 순서로 들어올 때만 발생하지만, 발생하면 무조건 시스템이 죽는다.

#### | 동기화 (Synchronization)

**여러 스레드가 데이터를 안전하게 읽고 쓸 수 있도록 접근 순서를 제어하는 기술**

임계 영역 문제를 해결하기 위한 3가지 조건:
1. **상호 배제 (Mutual Exclusion)** : 한 번에 하나의 스레드만 임계 영역 진입
2. **진행 (Progress)** : 임계 영역이 비어 있으면 대기 중인 스레드가 진입 가능해야 함
3. **한정된 대기 (Bounded Waiting)** : 특정 스레드가 무한정 기다리는 상황 방지

---

#### | 동기화 방법

##### Lock / Mutex

**공유 자원에 들어가기 전에 잠금을 획득하고, 작업 후 잠금을 해제하는 방식**

**Mutex (Mutual Exclusion) 동작 흐름**

```
① 스레드 1 → 락 획득
② 스레드 1 → 임계 영역 실행
③ 스레드 2 → 락이 풀릴 때까지 대기
④ 스레드 1 → 언락(해제)
⑤ 스레드 2 → 락 획득 후 임계 영역 실행
```

> ⚠️ **락은 반드시 `finally` 블록에서 해제해야 한다.** 예외 발생 시 락이 해제되지 않으면 다른 스레드가 영원히 대기한다.

```java
Lock lock = new ReentrantLock();

lock.lock();
try {
    // 임계 영역
    balance += 100;
} finally {
    lock.unlock(); // 반드시 finally에서 해제
}
```

##### 세마포어 (Semaphore)

**임계 영역에 들어갈 수 있는 입장권의 개수를 정해두는 방식**

- Mutex : 딱 1명만 입장
- Semaphore : N명까지 동시 입장 가능

```
입장권 개수 = 3 (최대 3개의 스레드 동시 접근 가능)
DB 커넥션 풀에서 많이 사용
```

##### Atomic 연산

**읽기, 수정, 저장을 하나의 쪼갤 수 없는 연산처럼 처리하는 방식**

```
일반 연산: 읽기 → 계산 → 저장  (중간에 다른 스레드 끼어들 수 있음)
Atomic 연산: [읽기 + 계산 + 저장]  (원자적으로 처리, 중간에 끼어들기 불가)
```

```java
// 일반 int → 스레드 안전하지 않음
int count = 0;
count++;

// AtomicInteger → 스레드 안전
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();
```

> ⚠️ **Atomic이 만능은 아니다.** 단순 카운터 증가는 해결 가능하지만, 여러 작업을 하나의 비즈니스 로직으로 묶는 건 한계가 있다.

##### CAS (Compare And Swap)

**내가 읽은 값이 아직 그대로일 때만 변경을 적용하는 방식**

```
"현재 값이 내가 읽었던 값과 같으면 → 새 값으로 교체"
"다르면 → 실패, 재시도"
```

```java
// stock.get(), decrementAndGet()을 따로 쓰는 게 아니라
// 읽은 값이 아직 그대로일 때만 차감을 보장
AtomicInteger stock = new AtomicInteger(10);
int current = stock.get();
boolean success = stock.compareAndSet(current, current - 1);
```

##### 트랜잭션 (Transaction)

- 여러 작업을 하나의 원자적 단위로 묶어 처리
- 임계 영역 문제의 DB 레벨 해결책

##### 임계 영역을 작게 유지해야 하는 이유

```
임계 영역이 넓으면
→ 하나의 스레드가 락을 오래 보유
→ 다른 스레드들의 대기 시간 증가
→ 처리량 감소 → 성능 저하

공유 자원에 접근하는 순간만 잠그고, 나머지는 락 밖에서 처리
```

##### 동기화의 실무 예시

**백엔드 개발자 관점**

| 상황 | 사용하는 동기화 방법 |
|---|---|
| 재고 차감 | DB 트랜잭션 (Row Lock) |
| 쿠폰 발급 | DB Unique 제약 + 트랜잭션 |
| 결제 처리 | DB 트랜잭션 + 비즈니스 로직 트랜잭션 |
| 포인트 적립 | AtomicInteger, 트랜잭션 |
| 메시지 큐 | Consumer Lock |
| 캐시 갱신 | CAS, TTL, 분산 락 |

> 💡 백엔드 개발할 때 OS 레벨 락을 직접 쓰진 않지만, **트랜잭션, Redis 락, 메시지 큐, 동시 요청 처리**에서 이미 동기화 개념을 사용하고 있다. 임계 영역이라고 드러나지 않을 뿐!

**인프라 엔지니어 관점**

| 상황 | 동기화 방법 |
|---|---|
| 커널 내부 자료 구조 | 커널 내부 락 |
| 파일 시스템 | MVCC, 파일 락 |
| 데이터베이스 운영 | 락 경합, 데드락, 트랜잭션 대기 분석 |
| 쿠버네티스 | 여러 컨트롤러의 리소스 상태 갱신 제어 |

##### 자주 헷갈리는 것들

| 질문 | 답변 |
|---|---|
| 공유 자원 = 임계 영역? | ❌ 다른 개념이다 |
| 공유 자원에 접근하면 다 임계 영역? | ❌ 읽기만 한다면 임계 영역이 아닐 수 있다 |
| 락을 많이 걸면 안전한가? | 안전하지만 성능이 나빠진다 |
| 락을 쓰면 다 해결되나? | ❌ 데드락, 성능 저하, 기아 상태, 락 누락, 락 범위 오류 등 새로운 문제가 생긴다 |

##### 작업 팁 ✅

- 임계 영역은 **작게** 잡아야 한다
- 락을 잡으면 **반드시 해제**해라 (`finally`)
- 여러 락이 필요하면 **순서를 통일**해라 (항상 A→B 순서로)
- DB의 `Unique`, `Transaction`을 적극 활용해라
- 분산 환경(서버 여러 대)에서는 로컬 락으로 부족 → **DB Lock, Redis 분산 락** 사용

#### | 락 (Lock)

**공유 자원 앞에 걸어두는 자물쇠. 임계 영역에 한 번에 하나의 스레드만 들어가게 하는 도구.**

```
공유 자원 : 여러 스레드가 함께 사용하는 대상
임계 영역 : 공유 자원에 접근하는 위험한 코드 구간
락       : 임계 영역에 한 번에 하나의 스레드만 들어가게 하는 도구
```

**락 동작 흐름**

```
락 요청 → 락 획득 → 임계 영역 실행 → (다른 스레드 대기) → 락 해제 → 다음 스레드 진입
```

##### 락의 종류

**1. 스핀락 (Spinlock)**

락이 풀릴 때까지 계속 확인하며 기다리는 락. 잠들지 않고 CPU를 계속 사용.

```
락 풀렸나? 락 풀렸나? 락 풀렸나? ...  (Busy-waiting)
```

| 장점 | 단점 |
|---|---|
| 락이 빨리 풀리면 빠르게 진입 | 기다리는 동안 CPU를 계속 소비 |
| 컨텍스트 스위칭 비용 없음 | 락 대기 시간이 길면 비효율적 |
| 커널 내부, 짧은 임계 영역에 적합 | 단일 CPU 환경에서 특히 비효율적 |

> 💡 백엔드 개발에서는 쓸 일이 거의 없다. 주로 운영체제 커널, 런타임, 저수준 라이브러리에서 사용.

**2. 낙관적 락 (Optimistic Lock)**

충돌이 거의 없다고 가정하고, 작업을 먼저 하고 마지막에 충돌 여부를 검사하는 방식

- 충돌 발생 시 롤백하거나 재시도
- 읽기가 많고 쓰기가 적은 시스템에 적합
- 주요 구현 방법 : **DB version 컬럼**, CAS, 분산 환경 상태 갱신

#### | Mutex (뮤텍스)

**한 번에 하나의 스레드만 자원에 접근하게 하는 대표적인 락**

Mutual Exclusion(상호 배제)의 약자

##### 핵심 3가지

1. **상호 배제** : 한 스레드만 락 획득 가능
2. **소유권** : 락을 획득한 스레드만 해제 가능
3. **해제 권한** : 락을 건 스레드가 반드시 직접 해제해야 함

##### 뮤텍스 내부 구조

| 구성 요소 | 설명 |
|---|---|
| **잠김 상태 (Lock State)** | 현재 락이 걸려 있는지 여부 |
| **소유 스레드 (Owner Thread)** | 현재 락을 보유한 스레드 |
| **대기 큐 (Wait Queue)** | 락을 얻지 못한 스레드들이 대기하는 줄 |
| **정책 (Policy)** | 대기 큐에서 깨어나는 순서/방식 |

##### 뮤텍스 장단점

| 장점 | 단점 |
|---|---|
| 데이터 무결성 보장 | 병렬성 감소 |
| 사용 방식이 직관적 | 컨텍스트 스위칭 오버헤드 |
| OS/언어 차원에서 지원 | 데드락 위험 (잘못된 순서로 여러 뮤텍스 잡으면) |
| CPU 낭비 감소 (기다릴 때 스레드를 잠들게 함) | 우선순위 역전 가능성 |

##### 조건 변수 (Condition Variable)

뮤텍스로는 "한 번에 하나의 접근"은 해결되지만, **특정 조건을 만족할 때까지 기다려야 하는 상황**은 처리하기 어렵다.

**조건 변수** : 스레드가 특정 조건이 충족될 때까지 대기하거나, 다른 스레드가 신호를 보내 대기 중인 스레드를 깨워 실행을 재개하도록 하는 동기화 메커니즘

| 구분 | 역할 |
|---|---|
| **Mutex** | 공유 자원 접근 권한 제어 |
| **조건 변수** | 특정 조건 충족 여부 관리 |

---

#### | 리드-라이트 락 (Read-Write Lock)

**읽기는 여러 명이 동시에 허용하고, 쓰기는 한 명만 허용하는 락**

```
Reader + Reader  = ✅ 가능 (동시 읽기 허용)
Reader + Writer  = ❌ 불가능
Writer + Writer  = ❌ 불가능
```

**읽기가 압도적으로 많은 시스템에서 성능 향상 효과가 크다.**

##### 장단점

| 장점 | 단점 |
|---|---|
| 읽기 요청이 많은 시스템에서 성능 향상 | 구현이 Mutex보다 복잡 |
| 리더는 동시 접근 가능 | 정책 설계 잘못하면 데드락 가능 |
| 라이터는 독점 접근으로 데이터 무결성 보장 | 라이터/리더 기아 가능성 |
| | 업그레이드 문제 (Read Lock → Write Lock 전환 시 데드락 위험) |

##### 주요 사용 예시

- **캐시 조회** : 읽기가 압도적으로 많음
- **설정 정보 조회** : 변경은 드물고 조회는 빈번
- **파일 시스템 메타데이터** : 조회 많고 변경 제한적
- **인메모리 자료구조** : 읽기 중심 자료구조 보호

##### 대기 순서 정책

| 정책 | 설명 |
|---|---|
| **라이터 우선** | 라이터가 들어오면 새 리더를 막고 라이터가 먼저 권한 획득 |
| **리더 우선** | 라이터가 대기 중이어도 계속 리더를 받아줌 |

### 🗨️ 오늘의 회고

- 공유 자원 → 임계 영역 → 동기화 → 락의 흐름이 한 줄기로 연결된다는 걸 오늘 확실히 이해했다. 각 개념이 이전 개념의 문제를 해결하기 위해 등장한 것.
- 백엔드 개발에서 이미 트랜잭션, Redis 락, 메시지 큐를 통해 동기화를 사용하고 있었다는 점이 인상적이었다. "임계 영역이라고 드러나지 않을 뿐"이라는 말이 특히 와닿았다.
- 락의 종류(스핀락, 낙관적 락, Mutex, 리드-라이트 락)마다 적합한 상황이 다르다는 것, 상황에 맞는 락을 선택하는 것이 중요하다는 것을 배웠다.
- `finally`에서 락 해제, 임계 영역 최소화, 락 획득 순서 통일 — 이 세 가지를 실제 코드 작성 시 반드시 기억해야겠다.
- 오늘 좋아요/댓글 API 구현에 집중해서 목요일 완료 목표를 반드시 지키자.
