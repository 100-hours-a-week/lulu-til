#### | 교착 상태 (DeadLock)

**두 개 이상의 프로세스나 스레드가 자원을 점유한 채, 서로가 필요로 하는 자원을 기다리며 무한히 대기하는 상태**

##### 왜 알아야 하나?

1. **시스템 안정성 확보** : 교착 상태가 발생하면 시스템 전체가 멈출 수 있다
2. **자원 효율성 확보** : 점유된 채로 방치된 자원은 다른 작업에 쓰일 수 없다
3. **장애 원인 분석** : 원인을 알아야 재현하고 해결할 수 있다
4. **동시성 코드 설계** : 처음부터 교착 상태를 고려한 구조를 설계할 수 있다

> ⚠️ 교착 상태는 **가시성이 낮은 에러**다. 특정 조건이 맞아떨어질 때만 발생하고, 발생하면 시스템이 조용히 멈춰버린다. 로그도 없고, 에러도 없이.

##### 실무에서 어떻게 보이는가

| 실무 상황 | 교착 상태 가능성 |
|---|---|
| DB 트랜잭션 | 트랜잭션 A는 row 1을 잡고 row 2를 기다림, 트랜잭션 B는 row 2를 잡고 row 1을 기다림 |
| 멀티스레드 코드 | Thread A는 Lock 1을 잡고 Lock 2를 기다림, Thread B는 Lock 2를 잡고 Lock 1을 기다림 |
| 커넥션 풀 | 요청들이 커넥션을 점유한 채 다른 커넥션이나 락을 기다림 |
| 파일 처리 | 여러 스레드가 파일, 버퍼, 메타데이터 락을 서로 다른 순서로 점유 |

##### 교착 상태 발생 조건 (4가지가 동시에 충족되어야 발생)

| 조건 | 의미 | 예시 |
|---|---|---|
| **상호 배제 (Mutual Exclusion)** | 하나의 자원을 한 번에 하나의 프로세스나 스레드만 사용할 수 있음 | 하나의 파일 핸들은 동시에 두 스레드가 쓸 수 없음 |
| **점유와 대기 (Hold and Wait)** | 이미 자원을 가진 상태에서 다른 자원을 추가로 기다림 | Lock A를 잡은 상태에서 Lock B를 기다림 |
| **비선점 (No Preemption)** | 다른 프로세스가 점유한 자원을 강제로 빼앗을 수 없음 | 락을 잡은 스레드가 직접 unlock 하기 전까지 다른 스레드가 가져갈 수 없음 |
| **순환 대기 (Circular Wait)** | 자원을 기다리는 관계가 원형 구조를 이룸 | A는 B를 기다리고, B는 C를 기다리고, C는 A를 기다림 |

> 💡 이 4가지 조건 중 **하나라도 깨뜨리면** 교착 상태는 발생하지 않는다. 해결 전략도 이 조건들을 무너뜨리는 방향으로 설계된다.

##### 해결 방법

| 방법 | 설명 |
|---|---|
| **점유와 대기 줄이기** | 필요한 자원을 한 번에 얻거나, 실패하면 기존 자원을 놓고 재시도 |
| **순환 대기 방지** | 모든 스레드가 자원을 같은 순서로 획득하도록 규칙화 |

##### 예방 방법 (실무)

| 방법 | 설명 | 실무 예시 |
|---|---|---|
| **락 획득 순서 통일** | 모든 스레드가 같은 순서로 락을 획득하게 함 | 항상 `userLock → orderLock → paymentLock` 순서로 획득 |
| **타임아웃 설정** | 일정 시간 안에 락을 얻지 못하면 포기하거나 재시도 | `tryLock(3, TimeUnit.SECONDS)` 사용 |
| **락 범위 최소화** | 락을 잡고 있는 시간을 최대한 줄임 | DB 조회나 외부 API 호출은 락 밖에서 수행 |
| **한 번에 여러 락 잡지 않기** | 중첩 락 구조를 줄임 | `synchronized` 안에서 다른 `synchronized` 진입 피하기 |
| **자원 선점 또는 롤백** | 교착 가능성이 있으면 작업을 중단하고 자원 반환 | DB deadlock 발생 시 한 트랜잭션 rollback |
| **감시와 복구** | 대기 시간이 긴 스레드나 트랜잭션을 탐지 | Thread dump, DB lock wait 모니터링 |

##### 교착 상태 예시 코드 (Java)

```java
// ❌ 교착 상태 발생 예시
public static void main(String[] args) {
    Thread thread1 = new Thread(() -> {
        synchronized (lockA) {
            System.out.println("Thread 1: lockA 획득");
            sleep(100);
            synchronized (lockB) {
                System.out.println("Thread 1: lockB 획득");
            }
        }
    });

    Thread thread2 = new Thread(() -> {
        synchronized (lockB) {
            System.out.println("Thread 2: lockB 획득");
            sleep(100);
            synchronized (lockA) {
                System.out.println("Thread 2: lockA 획득");
            }
        }
    });
}

// ✅ 교착 상태 방지 : 락 획득 순서 통일
Thread thread1 = new Thread(() -> {
    synchronized (lockA) {        // 항상 A → B 순서
        synchronized (lockB) { /* ... */ }
    }
});
Thread thread2 = new Thread(() -> {
    synchronized (lockA) {        // 항상 A → B 순서
        synchronized (lockB) { /* ... */ }
    }
});
```

#### | 기아 상태 (Starvation)

**특정 프로세스나 스레드가 자원을 계속 할당받지 못해 실행 기회를 잃는 상태**

교착 상태와 달리 시스템이 멈추지는 않지만, 특정 작업만 무한히 대기하게 된다.

##### 기아 상태 주요 원인

**1. 우선순위 기반 스케줄링**
- 우선순위가 높은 작업을 먼저 실행하는 방식
- 중요한 작업을 먼저 처리할 수 있다는 장점이 있지만
- 낮은 우선순위의 작업은 계속 밀릴 수 있다

**2. 불공정한 락 획득**
- 락 경쟁: 여러 스레드가 같은 락을 얻으려고 경쟁
- 특정 스레드가 반복적으로 락 획득에 실패
- 불공정 락을 사용하는 이유 → 저렴하게 운영 가능 (도착 순서를 별도 자료구조로 관리할 필요 없음)
- 공정 락은 "누가 언제 도착했는지" 알아야 하므로 자원 소비 증가

**3. Read-Write Lock에서 Write 기아**
- 리더가 계속 들어와서 라이터가 쓰기 락을 얻지 못하고 계속 대기

##### 해결 방법

| 방법 | 설명 |
|---|---|
| **에이징 (Aging)** | 오래 기다린 작업의 우선순위를 점진적으로 높이는 방식 |
| **라운드 로빈 (Round Robin)** | 모든 작업에게 일정한 시간 조각을 부여해 그 시간만큼 작업 수행. 공정성은 좋지만 컨텍스트 스위칭이 잦을 수 있다 |
| **최대 대기 시간 설정** | 일정 시간 이상 기다린 작업은 강제로 실행 기회를 부여 |
| **공정한 큐 사용하기** | 기아 상태는 "누가 먼저 기다렸냐"가 무시될 때 발생 |

**공정한 큐 방식 비교**

| 해결 방법 | 설명 |
|---|---|
| **FIFO 큐** | 먼저 기다린 작업을 먼저 처리 |
| **Fair Lock** | 오래 기다린 스레드가 락을 얻을 수 있도록 보장 |
| **대기열 기반 처리** | 요청 순서를 명시적으로 관리 |

##### 실무 예시 — DB 커넥션 풀의 기아 상태

```
API 요청 급증
  → DB 커넥션 풀 고갈
  → 일부 요청이 커넥션 대기
  → 응답 지연 발생
  → 스레드 풀도 점유
  → 전체 서버 지연 (장애 확산)
```

특히 DDoS/DoS 공격이나 트래픽 폭증 시 이 흐름이 빠르게 발생할 수 있다.

**DB 커넥션 풀 점검 항목**

| 점검 항목 | 설명 |
|---|---|
| `connection timeout` | 커넥션을 얼마나 기다릴 것인가 |
| `max pool size` | 동시에 몇 개의 DB 커넥션을 허용할 것인가 |
| `slow query` | 커넥션을 오래 점유하는 쿼리가 있는가 |
| `transaction 범위` | 트랜잭션을 불필요하게 오래 잡고 있지 않은가 |

##### 기아 상태 대응의 대원칙

> 💡 **모든 에러 대응의 대원칙 = 관측해야 수정한다 → 옵저빌리티(Observability)가 높아야 한다**

| 영역 | 봐야 할 지표 |
|---|---|
| 애플리케이션 | request latency, thread pool active count, queue size |
| DB | connection wait time, lock wait time, slow query |
| JVM | thread dump, blocked/waiting thread 수 |
| Kubernetes | CPU throttling, Pending Pod, Restart Count |
| 메시지 큐 | consumer lag, retry count, DLQ 증가 |

#### | 세마포어 (Semaphore)

**공유 자원에 접근할 수 있는 허가권(permit)의 개수를 관리해서, 동시에 접근 가능한 스레드 수를 제한하는 동기화 도구**

락(Lock)이 "잠금"의 개념이라면, 세마포어는 **"입장권"** 의 개념에 가깝다.

##### 직관적인 비유

```
놀이공원 입장 티켓 (최대 3명 동시 입장 가능)

→ A 입장 (티켓 1 사용, 남은 티켓: 2)
→ B 입장 (티켓 1 사용, 남은 티켓: 1)
→ C 입장 (티켓 1 사용, 남은 티켓: 0)
→ D 대기 (티켓 없음, 입장 불가)
→ B 나감 (티켓 1 반납, 남은 티켓: 1)
→ D 입장 가능
```

##### 왜 사용하는가

| 이유 | 설명 | 예시 |
|---|---|---|
| **자원 보호** | 한정된 자원을 동시에 너무 많이 사용하지 않도록 제한 | DB 커넥션 10개만 허용 |
| **동시성 제어** | 여러 스레드가 동시에 실행되더라도 안전하게 순서 조정 | 파일 쓰기, 네트워크 요청 제한 |
| **시스템 안정성** | 과도한 요청으로 서버, DB, 외부 API가 무너지지 않게 보호 | 외부 API 초당 호출 수 제한 |

**DB 커넥션 예시**
```
세마포어 없이 1000개 요청 → 1000개 커넥션 생성 → DB 뻥 💥
세마포어 사용 시 → 커넥션 풀에 10개만 생성 → 1000개 요청을 줄 세워서 순차 처리 ✅
```

##### 세마포어의 핵심 개념

**1. 세마포어 값 S**
- `S > 0` : 자원 사용 가능 (허가권 남아있음)
- `S = 0` : 모든 자원이 사용 중, 대기해야 함

**2. P 연산 (wait / acquire)**
- 허가권을 하나 가져가는 연산
- S가 0이면 허가권이 생길 때까지 대기

**3. V 연산 (signal / release)**
- 허가권을 하나 반납하는 연산
- 대기 중인 스레드가 있으면 하나를 깨워 자원 접근 허용

```java
Semaphore semaphore = new Semaphore(3); // 최대 3개 동시 접근

semaphore.acquire(); // P(wait): 허가권 획득
try {
    // 공유 자원 사용
} finally {
    semaphore.release(); // V(signal): 허가권 반납 (반드시 finally에서!)
}
```

##### 세마포어의 종류

| 종류 | 값 범위 | 사용 목적 | 사용 예시 |
|---|---|---|---|
| **이진 세마포어** | 0 또는 1 | 단일 자원 접근 제어 | 단일 장치 접근, 단일 쓰기 |
| **카운팅 세마포어** | 0 이상의 정수 | 다중 자원 접근 제어 | DB 커넥션 풀, 스레드 풀, 네트워크 포트 |

> ⚠️ **뮤텍스 vs 세마포어**: 뮤텍스는 소유권 개념이 있어 락을 획득한 스레드만 해제할 수 있다. 세마포어는 소유권이 없고 단순한 "허가권" 개념에 가깝다.

##### 백엔드 개발자가 세마포어를 활용하는 상황

**외부 API 호출 제한**
```
외부 API에 1000개 요청 동시 발송
→ 외부 API 입장에서 공격처럼 인식
→ 해당 IP를 일정 시간 차단 (Rate Limit)
→ 우리 서비스 전체가 외부 API 의존도로 인해 마비
→ 세마포어로 동시 요청 수를 제한해 안전하게 관리
```

**파일/이미지 처리 서버**
```
이미지 1개 처리 시 스레드에서 약 100MB 메모리 소비
서버 여유 힙 메모리: 400MB
→ 이미지 처리 스레드 최대 4개로 세마포어 제한
```

**인프라 관점**
- 로드 밸런서: WAS로 부하 분배 시 동시 요청 수 제한
- Nginx: 동시 HTTP 연결 수 제한
- Rate Limit: 클라이언트별 API 요청 횟수 제한

##### Crontab

특정 시간마다 작업을 반복 실행하게 하는 스케줄러. 배치 작업, 로그 정리, 백업, 주기적인 스크립트 실행에 사용.

| 항목 | 의미 | 범위 |
|---|---|---|
| 분 | 실행 시각(분) | 0-59 |
| 시 | 실행 시각(시) | 0-23 |
| 일 | 실행 날짜(일) | 1-31 |
| 월 | 실행 월 | 1-12 |
| 요일 | 실행 요일 (0=일요일) | 0-7 |
| 명령어 | 실행할 명령 또는 스크립트 | 문자열 |

```bash
# 매일 새벽 3시 0분에 backup.sh 실행
0 3 * * * /home/ubuntu/backup.sh
```

##### File Permission

```bash
ls -al                            # 파일 권한 확인
chmod +x deploy.sh                # 실행 권한 추가
chmod 755 deploy.sh               # rwxr-xr-x 설정
sudo chown ubuntu:ubuntu app.log  # 소유자 변경
```

**권한 숫자값**

| 권한 | 문자 | 값 | 의미 |
|---|---|---|---|
| Read | r | 4 | 파일 읽기, 디렉터리 목록 조회 |
| Write | w | 2 | 파일 수정, 디렉터리 내부 파일 생성/삭제 |
| Execute | x | 1 | 파일 실행, 디렉터리 접근 |

> 💡 `755` = 소유자(7=rwx) / 그룹(5=r-x) / 기타(5=r-x). 실무에서 `Permission Denied` 에러를 자주 만나므로 권한 숫자값은 꼭 외워두자.

##### Pipe (`|`)

명령어를 조합할 때 사용. **앞 명령어의 결과를 뒤 명령어의 입력으로 넘긴다.**

```bash
ps aux | grep java          # 실행 중인 java 프로세스 찾기
cat app.log | grep ERROR    # 로그에서 ERROR 줄만 필터링
```

##### Grep

텍스트 데이터 안에서 특정 패턴과 일치하는 줄을 찾아 출력하는 검색 전용 명령어

```bash
grep "ERROR" app.log        # app.log에서 ERROR가 포함된 줄 출력
grep -i "error" app.log     # 대소문자 구분 없이 검색
grep -n "Exception" app.log # 줄 번호 함께 출력
grep -r "DB_HOST" .         # 현재 디렉터리 전체에서 재귀 검색
```

##### Head / Tail

```bash
head -n 20 app.log          # 파일 앞쪽 20줄만 출력
tail -n 50 app.log          # 파일 뒷쪽 50줄만 출력
tail -f app.log             # 실시간으로 로그 추적 (서버 로그 모니터링 시 자주 사용)
```

##### Top / htop

```bash
top     # 실시간으로 시스템 프로세스 상태, 자원 사용량 모니터링
htop    # top을 더 시각적으로 보여주는 버전 (설치 필요)
```

##### PS

현재 실행 중인 프로세스 목록을 출력하는 명령어

```bash
ps aux | grep java     # java 프로세스 확인
ps aux | grep nginx    # nginx 프로세스 확인
```

##### Symbolic Link

Windows의 "바로가기"와 같은 개념. 파일의 실제 위치를 기록해 놓은 링크.

```bash
ln -s /var/log/app.log /home/ubuntu/app.log   # 심볼릭 링크 생성
```

##### DU / DF

디스크 문제 발생 시 디스크 상태를 확인하는 명령어

```bash
df -h               # 파일 시스템 전체 용량 확인 (디스크 몇 퍼센트 찼나?)
du -sh /var/log/*   # 특정 디렉터리/파일이 얼마나 차지하는지 확인 (누가 디스크를 많이 쓰나?)
```

**실무에서 디스크가 가득 차는 주요 원인**
- 로그 파일 누적
- 도커 이미지 쌓임
- 파일 업로드 후 temp 파일 미처리
- DB binlog 누적
- 백업 파일을 안 치운 경우

##### Dig

DNS 확인 명령어

| 상황 | 확인 대상 |
|---|---|
| 도메인이 서버 IP를 잘 바라보는지 | A, AAAA 레코드 |
| 도메인 소유권 인증이 되었는지 | TXT 레코드 |
| 메일 발송 도메인 설정이 맞는지 | MX, SPF, DKIM, DMARC |
| DNS 변경이 전파되었는지 | 특정 DNS 서버 조회 |
| CDN / 로드밸런서 연결이 맞는지 | CNAME |
| 장애 시 DNS 문제인지 서버 문제인지 구분 | `dig` + `curl` 조합 |

##### SS

어떤 프로세스가 몇 번 포트를 사용 중인지 확인

```bash
ss -tulnp | grep 8080    # 8080 포트 사용 중인 프로세스 확인
```

### 🗨️ 오늘의 회고

- 교착 상태, 기아 상태, 세마포어가 각각 독립된 개념이 아니라 **동시성 문제를 다루는 하나의 흐름** 안에 있다는 걸 이해했다. 공유 자원 → 임계 영역 → 동기화 → 락 → 교착/기아 상태로 이어지는 맥락이 이제 연결된다.
- 교착 상태는 4가지 조건이 동시에 충족될 때 발생하므로, 하나만 깨뜨려도 예방 가능하다. 특히 **락 획득 순서 통일**이 가장 현실적인 해결책임을 기억하자.
- 기아 상태 대응의 핵심은 **옵저빌리티(Observability)**. 관측하지 않으면 수정할 수 없다.
- 세마포어가 "락"이 아니라 "허가권"이라는 개념이 인상 깊었다. DB 커넥션 풀이 결국 세마포어 패턴이라는 사실도.
- 리눅스 명령어는 실무에서 서버를 직접 다룰 때 반드시 필요하다. `tail -f`, `grep`, `ps aux | grep`, `df -h` 조합은 장애 대응 시 기본 무기.
- 오늘 반드시 좋아요/댓글 API 구현을 완료하자. 더 이상 미루지 않는다.
