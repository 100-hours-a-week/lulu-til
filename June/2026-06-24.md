### 🚀 Today I Learned

#### | ElastiCache

**Redis나 Memcached 같은 인메모리 캐시 서버를 직접 EC2에 설치하고 운영하지 않아도, AWS에서 간편하게 생성하고 사용할 수 있도록 제공하는 관리형 캐시 서비스**

한마디로 **AWS에서 관리해주는 인메모리 캐시 서비스**다.

##### 데이터베이스만으로 서비스할 때 생기는 문제점

커뮤니티에서 게시글을 조회하는 상황을 가정해보자. 인기글, 게시글 목록 같은 데이터를 처음에는 RDS(MySQL)에서 조회해도 크게 문제가 되지 않는다.

> 그런데 1분 동안 만 명이 조회한다면?

MySQL은 결국 디스크를 조회하게 되는데, **디스크는 IOPS 제한량이 있어서 속도가 느려진다.** 보통 데이터베이스에 부하가 걸릴 때, 바로 이 디스크 I/O 구간이 병목이 되는 게 일반적이다.

##### 캐시란

**원본 데이터를 매번 다시 조회하지 않도록, 자주 사용하는 데이터를 빠르게 꺼낼 수 있게 임시로 저장하는 기술**

```
애플리케이션 → 먼저 조회 → ElastiCache
                              ↓ 없으면
                            RDS → 조회 결과 → ElastiCache
```

- 캐시에 데이터가 있으면 → **Cache Hit** → 바로 응답
- 캐시에 데이터가 없으면 → **Cache Miss** → RDS에서 조회하고, 응답하면서 **캐시에 저장**

```
사용자 요청 → 캐시에 데이터가 있는가?
  ├─ 있음(Cache Hit)  → 캐시 데이터 반환
  └─ 없음(Cache Miss) → RDS 조회 → 캐시에 저장 → 조회 결과 반환
```

##### TTL (Time To Live)

**캐시 데이터가 자동으로 만료되기까지의 시간**

**TTL을 설정하는 이유**
- 캐시에 데이터가 너무 오래 남아 있는 걸 방지하기 위해서
- 캐시는 메모리 자원을 사용하는데, 메모리는 제한적인 자원이다
- 캐시 데이터는 **정합성을 보장하지 않는다** — 캐시 데이터가 항상 최신이라는 보장이 없기 때문에, 결국 최신 데이터로 갱신해줘야 한다

비즈니스 로직상 자연스럽게 캐시가 갱신되는 경우도 있다. 예를 들어 누군가 좋아요를 누르거나 취소하면, 그 시점에 알아서 캐시도 갱신하는 로직을 만들면 된다. 다만 중간에 트랜잭션이 실패하거나, 메인 비즈니스 로직 흐름에서 벗어난 트랜잭션이 들어오는 경우도 있다. (예: 광고 목적으로 특정 게시글의 좋아요 수를 인위적으로 높게 설정해서 상단에 노출시켜야 하는 경우)

**TTL은 짧을수록 좋은가?** → 그렇지 않다. 너무 짧아도, 너무 길어도 문제다.

| TTL이 너무 짧으면 | TTL이 너무 길면 |
|---|---|
| 캐시 미스가 자주 발생 | 오래된 데이터를 계속 보여줄 수 있음 |
| 캐시 효과가 줄어듦 | 데이터 변경이 제때 반영되지 않을 수 있음 |
| (대신) 데이터 정합성은 높아짐 | |

> 💡 TTL은 **어떤 값을 캐싱하느냐에 따라** 다르게 설정해야 한다.

##### 캐시 무효화 전략

원본 데이터가 변경되었을 때, 기존 캐시 값을 삭제하거나 수정하는 작업.

##### ElastiCache란

**분산형 인메모리 데이터 저장소**

**캐시를 왜 메모리(RAM)에 저장할까?**
- 캐시 저장소는 보통 In-Memory인 경우가 흔하다
- 캐시는 **자주 접근되고, 자주 변경되고, 빠른 속도가 필요**하다 → 그래서 RAM에 저장한다

##### RDS vs ElastiCache

| 구분 | RDS | ElastiCache |
|---|---|---|
| 핵심 역할 | 영구 데이터 저장 | 빠른 반복 조회 |
| 저장 위치 | 디스크 중심 | 메모리 중심 |
| 데이터 성격 | 원본 데이터 | 임시 복사본 또는 빠른 상태 데이터 |
| 정합성 | 강하게 관리 | TTL·무효화 전략 필요 |
| 대표 사례 | 회원, 주문, 결제, 게시글 | 인기 목록, 세션, 랭킹, API 제한 |

> 💡 보통 RDS를 메인으로 사용하면서 **ElastiCache를 함께 사용**하는 구조로 설계한다.

##### 캐시 사용 패턴 — Cache-Aside

애플리케이션이 **먼저 캐시를 조회**하고, 없을 때만 DB를 조회한 뒤 캐시에 저장하는 방식.

```
사용자 요청
  ↓
캐시에 데이터가 있는가?
  ├─ 있음(Cache Hit)  → 캐시 데이터 반환
  └─ 없음(Cache Miss) → RDS 조회 → 캐시에 저장 → 조회 결과 반환
```

##### ElastiCache에서 사용할 수 있는 엔진

캐시 저장소를 하나 만들면, 그걸 한 가지 목적으로만 쓰지 않는다. 세션 스토어, 인기글, 랭킹처럼 여러 목적을 동시에 수행하는 형태로 쓰이는 게 보통이다. 그래서 단순한 키-값 형태만으로는 충분하지 않을 수 있다.

| 엔진 | 특징 | 대표 활용 |
|---|---|---|
| **Valkey** | Redis OSS 호환 계열의 인메모리 저장소 | 캐시, 세션, 랭킹, Pub/Sub |
| **Redis OSS** | 다양한 자료구조와 기능 제공 | 캐시, 세션, 분산 락, 실시간 기능 |
| **Memcached** | 단순하고 가벼운 분산 키-값 캐시 | 단순 조회 캐시 |

**Redis / Valkey가 제공하는 자료구조**
- String
- Hash
- List
- Set
- Sorted Set
- Stream
- Pub/Sub

**Memcached**는 단순한 캐시, Key-Value Store 형태에 집중한다.

##### ElastiCache 고가용성 구성

**① Replica**

```
애플리케이션 → 쓰기 → Primary Node ─ 복제 → Replica A
                                    ─ 복제 → Replica B
애플리케이션 → 읽기 → Replica A / Replica B
```

쓰기는 Primary Node로, 읽기는 Replica로 분산해서 처리한다.

**② Multi-AZ**

```
애플리케이션 → Primary Endpoint → [AZ-A] Primary Node ─ 복제 → [AZ-B] Replica Node
```

Primary Node가 있는 AZ에 장애가 발생해도, 다른 AZ의 Replica Node로 전환할 수 있게 구성한다.

**③ Sharding**

```
                    ┌→ Shard 1 (일부 Key)
ElastiCache Cluster ┼→ Shard 2 (다른 Key)
                    └→ Shard 3 (또 다른 Key)
```

데이터를 여러 샤드에 나눠서 저장 용량과 쓰기 처리량을 확장한다.

##### 캐시를 사용할 때 알아야 하는 문제

**① 캐시는 원본 데이터가 아니다**

- 캐시는 애초에 **휘발성 데이터**다 (인메모리이기 때문)
- 레디스가 꺼지면? → RAM은 전하로 데이터를 유지하는데, 전기 공급이 끊기면 그대로 사라진다
- TTL이 끝나거나, 장애로 제거될 수도 있다

> ⚠️ 비즈니스 로직에서 간혹 **캐시에서만 데이터를 조회**하도록, 즉 캐시 의존도가 지나치게 높은 시스템을 구성하는 경우가 있는데 이는 위험하다. 캐시는 **복구 정책**이 필요하다 — 캐시가 히트되지 않았을 때는 `finally`처럼 항상 **영구 저장소(DB)를 조회하는 로직**을 추가해야 한다.

**② 캐시 스탬피드 (Cache Stampede)**

게시글이 매일 자정 인기글로 지정되는 상황을 가정해보자.

```
만 개의 글이 인기글로 지정된다
  → 모두 동시에 캐시에 등록된다
  → 만 개의 키가 모두 동일한 TTL(86,400초)로 설정된다
  → 정확히 1초 사이에 만 개의 키가 동시에 만료(miss)된다
  → 갑자기 대량의 요청이 RDBMS로 폭증
  → 응답 지연
  → 다른 서비스들도 함께 지연
  → 결국 Timeout이 발생하며 서비스 안정성 저하
```

> 💡 실무에서는 캐시의 TTL을 설정할 때 **항상 임의 값(jitter)을 추가**해서 만료 시점을 분산시킨다.

```
기본 TTL: 300초
실제 TTL: 300초 + 0~30초 랜덤 값
```

**③ 메모리 부족 현상**

- 캐시는 메모리로 동작하고, 메모리는 한정된 자원이다
- 캐시 메모리가 가득 차면 새로운 캐시가 생성되지 않고, 새로운 쓰기가 계속 실패할 수 있다
- 이를 위해 **Eviction(축출) 정책**을 설정한다 — 캐시 메모리가 부족할 때 오래되었거나 덜 사용된 키를 자동으로 제거하는 정책이다

##### ElastiCache 정리

| 항목 | 설명 |
|---|---|
| ElastiCache | AWS 관리형 인메모리 캐시 서비스 |
| 주요 엔진 | Valkey, Redis OSS, Memcached |
| 핵심 목적 | 반복 조회를 빠르게 처리하고 RDS 부하 감소 |
| 저장 위치 | 메모리(RAM) |
| 대표 패턴 | Cache-Aside |
| Cache Hit | 캐시에 데이터가 있어 빠르게 응답 |
| Cache Miss | DB에서 조회한 후 캐시에 저장 |
| TTL | 캐시 데이터 자동 만료 시간 |
| 캐시 무효화 | 원본 변경 시 기존 캐시 삭제 또는 갱신 |
| 주요 활용 | 게시글 캐시, 세션, 인증번호, 랭킹, Rate Limiting |
| Multi-AZ | 캐시 노드 장애 시 Replica 승격으로 가용성 향상 |
| Replica | 고가용성과 읽기 확장 |
| Shard | 저장 용량·쓰기 처리량 확장 |

#### | ELB (Elastic Load Balancer)

**사용자의 요청을 여러 서버에 나누어 전달하고, 장애 난 서버는 자동으로 제외하며 서비스가 느려지거나 멈추는 일을 줄여주는 AWS 관리형 로드 밸런서 서비스**

##### 왜 로드 밸런서가 필요할까

웹 서비스는 처음 만들 때 보통 서버 한 대로 시작한다. 하지만 서버 한 대로 운영하면 다음과 같은 문제가 발생한다.

| 상황 | 발생하는 문제 |
|---|---|
| 사용자가 많아짐 | 한 서버에 요청이 몰려 응답 속도가 느려짐 |
| 서버 CPU·메모리 부족 | 요청 처리 실패, 5xx 오류 증가 |
| 서버 프로세스 종료 | 서비스 전체 접속 불가 |
| 배포를 위해 서버 재시작 | 배포 중 사용자 요청 실패 |
| 서버를 여러 대 추가 | 어느 서버로 요청을 보내야 하는지 결정 필요 |

> ⚠️ 서비스를 안정화하기 위해 서버를 여러 개 추가하더라도, 그 자체로 안정화되는 건 아니다. 여러 서버를 쓴다 해도 사용자 요청이 특정 서버에만 몰리거나, 장애 난 서버로 계속 요청이 전달되는 경우가 생길 수 있다.

##### ELB가 해결한다

```
사용자 → ELB → EC2 서버 1
              → EC2 서버 2
              → EC2 서버 3
```

사용자는 ELB로 접속을 요청하고, ELB는 연결된 리소스들 중 **건강하고(Healthy) 트래픽도 적은 서버**로 요청을 전달한다.

##### ELB가 하는 역할

| ELB의 역할 | 설명 |
|---|---|
| 요청 분산 | 여러 서버에 요청을 나누어 전달 |
| 장애 서버 제외 | 응답하지 않는 서버에는 새 요청을 보내지 않음 |
| 단일 진입점 제공 | 사용자는 하나의 도메인 또는 주소로 접속 |
| 서버 확장 지원 | 서버가 늘어나도 ELB가 새 서버에 요청 분산 |
| HTTPS 처리 | 인증서를 연결해 HTTPS 요청을 받을 수 있음 |
| 요청 분기 | URL 경로·도메인 등에 따라 다른 서버 그룹으로 전달 가능 |

##### ELB가 장애 대응을 하는 방식

서버를 여러 대 두는 이유는 **고가용성**과 **내결함성**을 확보해서 서비스가 안정적으로 유지되도록 하는 것이다.

```
사용자 → ELB → EC2 1 (Healthy)
              → EC2 2 (Healthy)
              ┄┄┄┄┄→ EC2 3 (Unhealthy) ── 요청 전달 제외
```

> 💡 이건 ELB만의 기능이 아니라 웹서버들이 보통 다 제공하는 기능이다. (예: Nginx도 헬스 체크 기능이 있다)

**헬스 체크**

스프링 서버에 아래와 같은 API를 추가한다.

```
GET /actuator/health

정상 응답: { "status": "UP" }
```

ELB는 이 경로를 **주기적으로** 요청해서, 정상 응답(`200 OK`)이 오는 동안 그 서버를 Healthy 상태로 유지한다.

```
ELB → GET /actuator/health → Spring Boot 서버
ELB ← 200 OK ← Spring Boot 서버
→ Healthy 상태 유지
```

##### ELB의 구조

```
사용자 → Load Balancer → Listener → Rule → Target Group → EC2 1
                                                          → EC2 2
```

**1. Load Balancer**

사용자가 접속하는 입구. 사용자는 LB의 경로(도메인)만 알면 요청할 수 있다.
- Route 53에 도메인을 등록할 때, `A api.domain.com → Alias`로 LB를 바라보게 설정한다

**2. Listener**

ELB가 **특정 포트**에서 요청을 기다리는 입구.

| Listener | 의미 |
|---|---|
| HTTP : 80 | 일반 웹 요청을 받는 입구 |
| HTTPS : 443 | 암호화된 웹 요청을 받는 입구 |

```
브라우저 --HTTPS 443--> ALB Listener --HTTP 8080--> Spring Boot
```

사용자는 요청을 HTTPS로 안전하게 전달하고, ALB가 그 요청을 받아 서버 내부에는 HTTP로 전달하는 구조다.

**3. Rule**

"이 요청을 어디로 보낼 것인가"를 정하는 규칙.

```
사용자 → ALB → /api/*    → API Target Group
              → /admin/*  → Admin Target Group
              → 기본 경로  → Web Target Group
```

**4. Target Group**

실제 요청을 처리할 서버들의 묶음.

```
ALB → backend-target-group → EC2 1
                            → EC2 2
                            → EC2 3
```

| 프로토콜 | HTTP |
| 포트 | 8080 |
| Health Check 경로 | `/actuator/health` |
| 성공 코드 | 200 |
| 등록 대상 | EC2 서버 1, 2, 3 |

> 타겟 그룹은 지정한 프로토콜 + 포트로 등록된 대상에게 요청을 전달한다.

##### ELB의 종류

지금 단계에서 모든 종류를 다 알 필요는 없고, 다음 기준만 알면 된다.

> - 웹 서비스 + REST API → **ALB**를 검토
> - TCP, UDP처럼 HTTP가 아닌 트래픽 → **NLB**를 검토

| 종류 | 주로 다루는 트래픽 | 대표 사용 사례 |
|---|---|---|
| **ALB** | HTTP, HTTPS | 웹 서비스, Spring Boot API, 마이크로서비스 |
| **NLB** | TCP, TLS, UDP, QUIC | 게임 서버, TCP 서비스, 초저지연 서비스 |
| **GWLB** | 네트워크 보안 장비 트래픽 | 방화벽, IDS/IPS, 보안 어플라이언스 |
| **CLB** | 이전 세대 로드 밸런서 | 기존 레거시 시스템 |

##### ALB (Application Load Balancer)

HTTP, HTTPS 요청을 이해하는 **L7(응용 계층)** 로드 밸런서다.

- URL 경로를 볼 수 있고
- 도메인을 볼 수 있고
- 헤더 조건을 볼 수 있다

> 이런 정보를 기준으로 요청을 분기할 수 있다.

```
사용자 → ALB → /api/*    → Spring Boot API 1
              → /api/*    → Spring Boot API 2
              → /admin/*  → 관리자 서버
              → /socket/* → WebSocket 서버
```

##### ALB 구성 방법

**① 보안 그룹 구성**

ELB를 구성할 때는 애플리케이션 서버를 인터넷에 직접 노출하지 않는다.

**ALB 보안 그룹**

| 방향 | 포트 | 허용 대상 |
|---|---|---|
| Inbound | 80 | 인터넷 사용자 |
| Inbound | 443 | 인터넷 사용자 |
| Outbound | 8080 | 애플리케이션 EC2 보안 그룹 |

**애플리케이션 보안 그룹**

| 방향 | 포트 | 허용 대상 |
|---|---|---|
| Inbound | 8080 | ALB 보안 그룹 |
| Inbound | 22 | 관리자 고정 IP 또는 Bastion |
| Outbound | 3306 | RDS 보안 그룹 |

> ⚠️ 애플리케이션의 8080 포트를 `0.0.0.0/0`으로 열어두면 안 된다. 반드시 **ALB 보안 그룹만 허용**하도록 좁혀야 한다.

**② HTTPS 처리**

현업에서는 다음 두 가지 작업을 한다.
- 요청에서 HTTPS-SSL을 해석/제거하는 처리 (SSL Termination)
- 응답에 HTTPS를 다시 붙이는 처리

> 💡 이 처리는 **애플리케이션 서버에서 직접 하지 않는다.** 대신 **ALB에서 처리**한다 — ALB에 인증서를 붙이고, 인증서는 **ACM(AWS Certificate Manager)** 서비스로 발급받으면 된다.

이렇게 ALB에서 HTTPS를 처리하면 얻는 장점:

| 장점 | 설명 |
|---|---|
| 인증서 중앙 관리 | EC2마다 인증서를 설치할 필요 없음 |
| 서버 확장 편리 | 서버가 늘어도 인증서를 복제할 필요 없음 |
| 자동 갱신 활용 | ACM 인증서와 연동 가능 |
| 애플리케이션 단순화 | Spring Boot의 TLS 설정 부담 감소 |

##### ELB 정리

| 개념 | 핵심 |
|---|---|
| ELB | 여러 서버에 요청을 분산하는 AWS 서비스 |
| ALB | HTTP/HTTPS 웹 서비스에 주로 사용하는 로드 밸런서 |
| Listener | ELB가 요청을 받는 포트와 프로토콜 |
| Rule | 요청을 어디로 보낼지 결정하는 규칙 |
| Target Group | 실제 요청을 처리하는 서버 묶음 |
| Health Check | 정상 서버에게만 요청을 보내기 위한 상태 확인 |

#### | CloudFront

**전 세계에 분산된 서버 네트워크(CDN)를 통해 사용자에게 콘텐츠를 빠르고 안전하게 전달하는 서비스**

##### CloudFront가 필요한 이유

웹 서비스를 운영하면 사용자가 요청하는 파일, API 응답은 모두 **원본 서버**에서 만들어진다.

- 정적 웹 페이지 파일 → S3
- 이미지, 영상 → S3
- 백엔드 API → ALB → EC2

사용자가 원본 서버를 바로 바라보는 구조도 처음에는 잘 동작한다. 하지만 사용자가 많아지거나, 사용자와 서버의 물리적 거리가 멀어지면 문제가 생긴다.

| 문제 | 왜 발생할까? |
|---|---|
| 페이지 로딩이 느림 | 사용자가 원본 서버와 멀리 떨어져 있으면 네트워크 왕복 시간이 길어짐 |
| 이미지 요청이 원본 서버에 집중 | 같은 이미지라도 모든 사용자가 원본 서버에 요청 |
| 갑작스러운 트래픽 증가 | 원본 서버가 정적 파일 요청까지 모두 처리해야 함 |
| 서버 비용 증가 | 파일 전송과 서버 처리량이 계속 증가 |
| 원본 S3 또는 서버 직접 노출 | 사용자가 CloudFront를 우회해 원본에 직접 접근할 가능성 |

##### CDN의 역할

CloudFront도 **CDN(Content Delivery Network)의 한 종류**다.

- 사용자 ↔ 원본 서버 사이에 **캐시 계층**을 둔다
- 전 세계에 분산된 CDN 서버를 둬서, 매번 원본을 요청하지 않아도 CDN에서 데이터를 가져올 수 있게 한다

```
서울의 원본 서버 → 미국 사용자
                 → 유럽 사용자
                 → 일본 사용자
```

위처럼 원본 서버가 모든 지역 사용자를 직접 상대하는 대신:

```
한국 사용자 → 서울과 가까운 엣지 ┐
미국 사용자 → 미국과 가까운 엣지 ├→ 원본 S3 / ALB
유럽 사용자 → 유럽과 가까운 엣지 ┘
```

**CDN(Content Delivery Network)**: 여러 지역에 콘텐츠 전달 서버를 분산 배치하고, 사용자와 **가까운 서버**에서 콘텐츠를 내려받도록 하는 방식. CloudFront는 AWS에서 제공하는 CDN이다.

##### 엣지 로케이션 (Edge Location)

AWS의 분산 네트워크 거점. 엣지 로케이션에는 원본 데이터가 영구적으로 저장되는 게 아니라, **자주 요청되는 콘텐츠의 복사본을 잠시 들고 있는** 캐시 서버 역할을 한다.

##### CloudFront 동작 방식

```
사용자 → Edge Location(캐시) → Origin(원본 서버)
```

엣지 로케이션이 캐시를 들고 있다가, 캐시가 없거나 만료됐을 때만 원본으로 요청을 보낸다.

```
사용자 → /images/logo.png 요청 → CloudFront Edge
                                    ↓ 캐시 확인
[Cache Hit]  → 캐시된 logo.png 응답
[Cache Miss] → Origin에 logo.png 요청 → logo.png 응답
              → 캐시에 저장
              → logo.png 응답
```

##### CloudFront의 구성 요소

| 구성 요소 | 역할 |
|---|---|
| **Distribution** | CloudFront 배포 단위. 어떤 원본과 캐시 정책을 사용할지 정의 |
| **Viewer** | CloudFront에 요청하는 사용자 또는 브라우저 |
| **Edge Location** | 사용자 가까이에서 캐시된 콘텐츠를 전달하는 위치 |
| **Origin** | 원본 콘텐츠가 실제 저장된 위치 |
| **Cache Behavior** | 어떤 경로를 어떤 원본으로 보내고, 어떻게 캐시할지 정하는 규칙 |
| **Cache Policy** | 캐시 키와 TTL 등 캐시 동작을 결정하는 정책 |
| **OAC** | S3 원본에 CloudFront만 접근하도록 제한하는 기능 |

##### Cache Behavior

URL 경로에 따라 어떤 원본으로 요청을 보내고, 캐시를 어떻게 적용할지 정하는 규칙.

```
사용자 → CloudFront → /assets/*   → S3 이미지 버킷
                     → /images/*  → S3 이미지 버킷
                     → /api/*     → ALB → Spring Boot
                     → 기본 경로   → S3 정적 웹 페이지
```

| 요청 경로 | Origin | 캐시 전략 |
|---|---|---|
| `/images/*` | S3 | 길게 캐시 |
| `/assets/*` | S3 | 길게 캐시 |
| `/api/*` | ALB | 짧게 캐시 또는 캐시 비활성화 |
| `/` | S3 | HTML 특성에 맞게 짧게 캐시 |

> 💡 HTML 같은 웹 정적 페이지 파일도 CloudFront 캐싱을 적용할 수 있다. 다만 HTML 파일은 변경되는 주기가 뚜렷하기 때문에, **배포할 때 배포 스크립트에 캐시 무효화(Invalidation) 코드를 함께 넣는** 방식으로 관리한다.

##### 어떤 콘텐츠를 캐시해야 할까

모든 요청을 똑같이 캐시하면 안 된다. **콘텐츠 특성에 따라 캐시 전략을 다르게 설정**해야 한다.

| 콘텐츠 종류 | 예시 | 권장 캐시 전략 |
|---|---|---|
| 변경이 거의 없는 정적 파일 | 로고, 폰트, 라이브러리 파일 | 길게 캐시 |
| 빌드 결과물 | `app.8f3a2.js`, `main.2c4d.css` | 매우 길게 캐시 |
| 일반 이미지 | 프로필 이미지, 게시글 이미지 | 변경 빈도에 맞춰 캐시 |
| HTML | `index.html` | 짧게 캐시 |
| 로그인·결제·개인 정보 API | `/api/me`, `/api/orders` | 기본적으로 캐시하지 않음 |
| 공개 조회 API | 상품 목록, 공지 목록 | 조건부로 짧게 캐시 가능 |

##### CloudFront vs ELB

둘 다 사용자 요청의 앞단에 위치하지만, **해결하려는 문제가 다르다.**

| 구분 | CloudFront | ALB |
|---|---|---|
| 핵심 목적 | 콘텐츠 캐싱과 글로벌 전송 가속 | 여러 서버에 요청 분산 |
| 주요 위치 | 사용자와 원본 사이의 글로벌 엣지 | AWS 리전 내부의 애플리케이션 앞단 |
| 대표 대상 | 이미지, CSS, JS, 영상, 정적 파일 | EC2, ECS, 컨테이너, API 서버 |
| 캐시 | 핵심 기능 | 핵심 기능 아님 |
| Health Check | Origin 장애 대응 기능은 가능 | Target 서버 상태 확인이 핵심 |
| 대표 구조 | CloudFront → S3 | ALB → EC2/ECS |
| 함께 사용 | CloudFront → ALB → EC2/ECS | ALB가 CloudFront의 Origin으로 사용 가능 |

- **CloudFront** ← CDN
- **ALB** ← 로드 밸런서(LB)

##### CloudFront + ALB 조합해서 사용하기

```
사용자
  ↓
Route 53
  ↓
CloudFront
  ├─ /assets/* → S3 (정적 파일·이미지)
  └─ /api/*    → ALB → Spring Boot 1
                      → Spring Boot 2
                           ↓
                          RDS
```

정적 자산은 CloudFront가 S3에서 바로 캐시해서 전달하고, API 요청은 ALB를 거쳐 백엔드 애플리케이션으로 전달되는 구조다.

##### 실무 구조 예시

**① 정적 웹 사이트**

```
사용자 → Route 53 → CloudFront → Private S3 Bucket (React / Vite Build Files)
```

- S3에 Vite, Vue, React 등 프론트엔드 **빌드 파일**을 저장
- CloudFront가 콘텐츠를 캐싱
- Route 53에서 CloudFront 배포를 가리키는 도메인을 연결

**② 이미지 업로드 서비스**

```
사용자 → Frontend → Spring Boot API → S3 이미지 저장
       ↑              ↓
       └── CloudFront ┘
```

1. 사용자가 이미지를 업로드한다
2. 백엔드가 이미지를 S3에 저장한다
3. DB에는 S3 원본 주소 대신 **CloudFront URL 또는 객체 Key**를 저장한다
4. 프론트엔드는 **CloudFront 주소로** 이미지를 요청한다
5. CloudFront가 이미지 캐시를 처리한다

장점:
- 백엔드에서 이미지 업로드를 일괄적으로 처리할 수 있음
- 이후 요청부터는 CloudFront에 캐싱되어 있는 이미지를 바로 사용할 수 있음

> 💡 CloudFront는 파일을 영구적으로 저장하는 서비스가 아니라, **원본 콘텐츠를 사용자 가까이에서 빠르게 전달**하기 위한 캐시 계층이다. 그래서 원본 서버에 대한 반복 요청을 줄이고 싶을 때 사용한다.

### 🗨️ 오늘의 회고

- 오늘은 캐시(ElastiCache), 로드밸런서(ELB), CDN(CloudFront)을 배웠는데, 셋 다 결국 **"원본 서버(RDS, EC2)에 가는 부하를 줄이거나 분산시키는" 같은 목적**을 서로 다른 위치에서 해결하고 있다는 공통점이 보였다.
- ElastiCache의 캐시 스탬피드 문제가 가장 인상적이었다. 만 개의 캐시 키가 정확히 같은 시각에 만료되면서 RDBMS로 트래픽이 폭증한다는 시나리오는, 단순히 "캐시를 쓴다"고 끝나는 게 아니라 TTL에 랜덤 값을 더하는 디테일까지 신경 써야 한다는 걸 깨닫게 해줬다.
- "캐시는 원본 데이터가 아니다"라는 원칙이 단순하지만 중요했다. 캐시 의존도가 너무 높은 시스템을 만들면 안 되고, 캐시 미스 시 항상 영구 저장소를 조회하는 fallback 로직이 필요하다는 점을 기억해야겠다.
- ELB의 헬스 체크가 어제 배운 RDS, EC2 구성과 자연스럽게 이어졌다. `/actuator/health` 같은 엔드포인트 하나로 장애 서버를 자동으로 트래픽에서 제외시킬 수 있다는 게 단순하면서도 효과적인 설계라고 느꼈다.
- ALB에서 HTTPS(SSL Termination)를 처리하고 애플리케이션 서버는 내부적으로 HTTP만 신경 쓰면 된다는 구조가 깔끔했다. 인증서를 EC2마다 설치할 필요가 없다는 장점이 특히 실무적으로 크게 다가왔다.
- CloudFront와 ALB를 처음에는 비슷한 개념으로 생각했는데, "CloudFront는 캐싱과 전송 가속, ALB는 트래픽 분산"이라는 역할 차이를 명확히 구분하게 됐다. 둘은 경쟁 관계가 아니라 `CloudFront → ALB → EC2`처럼 같이 쓰이는 조합이라는 게 핵심이었다.
- 이미지 업로드 서비스 구조에서 DB에 S3 원본 주소 대신 CloudFront URL(또는 Key)을 저장한다는 부분이 실무적으로 꼭 기억해둘 만했다. 원본을 직접 노출하지 않고 캐시 계층을 통해서만 접근하게 만드는 패턴이다.
- 클라우드 설계를 아직 시작 못 했다는 게 마음에 걸린다. 지금까지 배운 VPC, EC2, S3, RDS, ElastiCache, ELB, CloudFront, Route 53을 어떻게 하나의 아키텍처로 엮을지 목요일까지는 꼭 정리해야겠다.
