#### | 동시성 (Concurrency)

**하나의 CPU가 작업을 빠르게 전환하여 마치 여러 작업이 동시에 실행되는 것처럼 보이게 하는 방식**

운영체제가 작업 실행 순서를 조율하고 관리해, 시스템이 여러 작업을 효율적으로 처리할 수 있도록 지원한다.

##### 왜 알아야 하나?

많은 사용자 요청이 몰리는 **백엔드 서버**에서 적절히 자원을 관리하고, 병렬 작업 구조를 이해하고 최적화하기 위해 필수적이다.

##### 동작 방식

작업을 작은 단위로 나눠 짧은 실행 시간을 할당하고, **스케줄러**가 이를 관리하며 작업 간 전환(컨텍스트 스위칭)을 수행한다.

> 💡 **Node.js 이벤트 루프** : 싱글 스레드 기반의 이벤트 루프를 사용해 동시성을 구현. 비동기 작업 처리 방식으로 CPU가 대기 상태에 머물지 않도록 설계됨.

#### | 스케줄러 (Scheduler)

**CPU와 같은 시스템 자원을 적절히 분배하기 위해 프로세스나 스레드의 실행 순서를 관리하는 운영체제의 핵심 구성 요소**

##### 왜 필요한가?

| 문제 | 설명 |
|---|---|
| **자원 독점** | 무거운 연산 하나가 CPU를 독점 → 마우스 움직이기, 키보드 입력도 안 됨 |
| **기아 상태 (Starvation)** | 먼저 온 것 또는 가장 급한 것만 처리하면 → 특정 프로세스는 영원히 실행 안 됨 |
| **시스템 낭비** | CPU가 디스크 I/O를 기다리는 동안 Idle 상태 → 이 시간에 다른 작업을 처리하면 효율 향상 |

##### 스케줄러의 목표

1. **CPU 이용률 최대화** : CPU는 컴퓨터 자원 중 가장 비싼 자원
2. **처리량 증가** : 단위 시간 안에 처리할 수 있는 작업의 양 최대화 (백엔드 서버의 최대 동시 처리량)
3. **총 처리 시간 감소** : 작업 시작부터 완료까지 시간 단축
4. **대기 시간 최소화**
5. **응답 시간 최소화**

##### 스케줄러 동작 방식

```
① 작업 대기열 관리
② 우선순위 설정
③ 자원 분배
④ 작업 전환 (Context Switching)
⑤ 완료 및 자원 해제
```

##### 스케줄러 기능

**1. 작업 선택**
- 현재 실행 가능한 프로세스/스레드 중 다음에 CPU를 줄 대상 결정
- 알고리즘 사용 : 라운드 로빈, 우선순위 기반 등

**2. 자원 분배 (타임 슬라이스)**
- 선택한 작업에 CPU 권한 분배
- 한 작업이 계속 CPU를 독점하지 않도록 **타임 슬라이스** 단위로 할당

**3. 컨텍스트 스위칭 관리**
- 기존 작업 상태를 PCB에 저장
- 새 작업의 PCB를 불러와 이어서 실행

##### 스케줄러의 종류

| 종류 | 역할 | 특징 |
|---|---|---|
| **장기 스케줄러** | 디스크에 있는 프로그램 중 어떤 것을 RAM에 올릴지 결정 | 현대 OS에서는 RAM이 저렴해져 잘 사용 안 함 |
| **중기 스케줄러** (스왑 관리자) | 메모리 부족 시 활성 프로세스 일부를 디스크 스왑 영역으로 이동, 여유 생기면 복구 | 가상 메모리 시스템 안정성 담당 |
| **단기 스케줄러** (CPU 스케줄러) | 메모리에서 준비 상태인 프로세스 중 CPU를 줄 프로세스 결정 | 밀리초 단위로 빠르게 동작 |

#### | 스케줄링 (Scheduling)

**CPU와 같은 자원을 프로세스와 스레드들이 사용할 수 있도록 줄을 세우는 "과정". CPU 사용 순서를 결정하는 운영체제의 정책.**

##### 스케줄링의 목표

1. CPU 이용률 증가
2. 처리량 증가
3. 응답 시간 감소
4. 대기 시간 감소
5. **공정성 확보** ← 특정 프로세스의 독점 방지

##### 스케줄링 레벨

| 레벨 | 설명 | 목표 |
|---|---|---|
| **고수준 스케줄링** | 메모리에 올릴 프로세스 결정 | 시스템 전체 부하 조절 |
| **중간 스케줄링** | 메모리 부족 시 일부 프로세스를 잠시 내보내기 | 메모리 확보, 시스템 안정성 |
| **저수준 스케줄링** | CPU를 누구에게 할당할지 결정 | CPU 자원의 효율적 분배 |

##### 스케줄링 정책 (시스템 목표에 따라 선택)

| 환경 | 정책 |
|---|---|
| **서버 환경** | 처리량을 높이는 정책 → 가성비 우선 |
| **실시간 시스템** | 응답 시간 최소화, 시간 제약 충족 → 실시간성 우선 (비쌈) |
| **데스크톱 환경** | 사용자 응답성 최우선, 다중 작업 간 공정성 유지 |

#### | 스케줄링 알고리즘

##### 1. FCFS (First Come First Served)

먼저 온 작업을 먼저 처리

- ✅ 구현이 쉬움
- ❌ 먼저 온 프로세스의 작업 시간이 길면 뒤의 프로세스가 전부 기다려야 함 (Convoy Effect)

##### 2. SJF (Shortest Job First)

가장 짧은 작업을 먼저 처리

- ✅ 평균 대기 시간이 가장 적음. 운영체제에서 가장 효율적
- ❌ 실행 시간 예측이 어려움, **기아 현상** 발생 (긴 작업은 영원히 실행 안 될 수 있음)

##### 3. SRTF (Shortest Remaining Time First)

남은 시간이 가장 짧은 프로세스를 먼저 처리. **SJF의 선점형 버전**

```
선점형이란?
CPU가 프로세스를 실행 중이더라도 다른 실행 흐름으로 강제로 전환 가능
CPU가 할당될 때 실행 가능한 시간(타임 슬라이스)을 설정
```

- ✅ 응답성이 좋아짐
- ❌ 컨텍스트 스위칭 비용이 높아짐

##### 4. Priority (우선순위)

우선순위가 높은 프로세스를 먼저 처리

- ✅ 중요한 작업을 먼저 처리 가능
- ❌ **기아 상태** 발생 (낮은 우선순위 프로세스는 굶어 죽음)

##### 5. Round Robin

모든 프로세스에게 동일한 CPU 시간(Time Slice) 제공

- ✅ 공정함, 응답성 준수
- ❌ 타임 슬라이스 크기에 민감

| 타임 슬라이스 | 문제 |
|---|---|
| 너무 작으면 | 컨텍스트 스위칭 비용 폭증 |
| 너무 크면 | 응답성이 나빠짐 |

##### 선점형 vs 비선점형

| 구분 | 선점형 (Preemptive) | 비선점형 (Non-Preemptive) |
|---|---|---|
| **CPU 강제 회수** | 가능 | 불가능 |
| **대표 OS** | Linux, Windows, Mac | 일부 임베디드 OS |
| **특징** | 동시 작업 필수 환경에 적합 | 프로세스가 종료 안 되면 자원 대기 무한 |

> 💡 현대 운영체제가 선점형을 사용하는 이유 : 비선점형이면 프로세스가 비정상 종료되지 않을 경우 자원을 무한히 독점하기 때문.

#### | Multilevel Queue (다단계 큐)

**프로세스를 그룹별로 나눠 관리하고, 각 그룹 큐마다 다른 알고리즘을 적용하는 방식**

```
실시간 작업     → 우선순위 기반
일반 사용자 작업 → 라운드 로빈
배치 작업       → FCFS
```

**현대 OS는 여러 알고리즘을 혼합해서 사용한다**

| OS | 스케줄링 방식 |
|---|---|
| **Linux** | 우선순위 + 라운드 로빈 + Fair 스케줄링 + 로드 밸런싱 |
| **Windows** | 동적 우선순위 + 라운드 로빈 |

##### 현업에서 개발자가 스케줄링을 어떻게 사용하나?

| 역할 | 사용 방식 |
|---|---|
| **백엔드 개발자** | 기본적으로 직접 사용하지 않음 (OS가 결정). But 성능 최적화·트러블슈팅 시 이해 필요 |
| **인프라 엔지니어** | Linux의 우선순위 조절, CPU Affinity 설정, 쿠버네티스의 CPU 할당 정책 설정 |

#### | 병렬성 (Parallelism)

**여러 CPU 코어 또는 다중 프로세서를 사용하여 여러 작업을 실제로 동시에 실행하는 개념**

##### 동시성 vs 병렬성

```
동시성 (Concurrency)
  → 코어 1개, 여러 작업을 빠르게 전환하며 동시에 처리하는 것처럼 보임
  → 실제로는 번갈아 실행

병렬성 (Parallelism)
  → 코어 여러 개, 실제로 여러 작업을 물리적으로 동시에 실행
  → 진짜 동시 실행
```

##### 병렬성이 필요한 이유

- CPU 코어가 8개인데 1개만 사용하면 7개가 낭비
- 여러 코어를 활용해 작업을 동시 수행 → 시스템 성능 극대화

**주요 사용처**

| 분야 | 예시 |
|---|---|
| **게임** | 물리 연산, 렌더링, AI 동시 처리 |
| **영상 작업** | 인코딩, 렌더링 |
| **인공지능 학습** | 머신러닝, 빅데이터 분석 |

##### 병렬 동작 방식

```
① 작업 분할 : 하나의 큰 작업을 여러 개로 나눔
② 코어 할당 : 분할된 작업을 각 CPU 코어에 할당
③ 병렬 실행 : 각 코어가 독립적으로 병렬 실행
④ 결과 통합 : 각 코어의 결과를 하나로 합침
⑤ 오류 처리
```

##### 병렬성 사용 시 주의사항

**1. 병렬성이 항상 답은 아니다**

```
병렬 처리 총 비용 = 작업 분할 비용 + 스레드 생성 비용 + 결과 병합 비용
싱글 코어 동시성 비용 = y

병렬 처리 총 비용 < 싱글 코어 비용이 명백할 때만 병렬성 선택
```

**2. 공유 자원 접근 시 조심**
- 여러 코어가 동시에 같은 자원에 접근 → **레이스 컨디션(Race Condition)** 발생 가능
- 동기화 처리 필수

**3. 물리적 제한**
- 코어 수를 초과하는 스레드를 만들어도 의미 없음
- 실제 하드웨어 스펙 확인 필요

### 🗨️ 오늘의 회고

- 동시성과 병렬성의 차이가 "1개의 코어를 빠르게 전환 vs 여러 코어를 실제로 동시 사용"으로 정리되니 명확해졌다. Node.js(동시성)와 멀티코어 머신러닝(병렬성)이 각각 어느 개념에 해당하는지도 연결됐다.
- 스케줄링 알고리즘 5가지 중 FCFS는 단순하지만 Convoy Effect가, SJF는 효율적이지만 기아 현상이 있고, Round Robin은 공정하지만 타임 슬라이스 튜닝이 필요하다는 트레이드오프가 인상적이었다.
- "병렬성이 항상 빠른 게 아니다"는 포인트가 중요하다. 작업 분할 + 스레드 생성 + 결과 병합 비용이 생각보다 비쌀 수 있으니 병렬화가 이득인지 계산이 필요하다.
- 백엔드 코드 구현에 오늘 집중해서 목요일 완료 목표를 지켜야겠다.
