#### | 네트워크 트래픽 처리 방식 — Inline

**네트워크 상의 데이터 패킷이 반드시 해당 장치를 통과하도록 설계된 네트워크 구조**

네트워크 장비는 **어디에 놓이느냐**가 중요하다.

```
일반 구조 : 사용자 → 인터넷 → 서버

보안 장비 필요 시 :
사용자 → 인터넷 → [보안 장비] → 서버
                        ↑
              모든 요청이 반드시 여기를 거친다 = Inline 방식
```

Inline은 장비의 이름이 아니라 **장비를 배치하는 방식**이다. 같은 방화벽이라도 트래픽이 무조건 통과하면 Inline, 통과하지 않으면 Out of Path다.

##### Inline 방식의 특징

**1. 트래픽이 반드시 지나간다 (강제성)**
- 트래픽이 인라인 장비를 우회하지 못하고, 반드시 거쳐야만 목적지에 도달하는 구조
- `사용자 → 인라인 장비 → 서버`

**2. 트래픽을 직접 제어한다**
- 이 요청이 정상인가?
- 이 IP는 차단 대상인가?
- 이 포트는 허용되었나?
- 이 요청은 너무 많이 들어오고 있지 않나? (Rate Limit)
- 이 패턴은 악성 패턴인가?
- 결과에 따라 트래픽을 **통과 / 드롭 / 다른 곳으로 전달** 제어 가능

| 처리 방식 | 의미 |
|---|---|
| **허용** | 정상 트래픽을 통과시킴 |
| **차단** | 위험하거나 정책에 맞지 않는 트래픽을 버림 |
| **필터링** | 특정 조건에 따라 트래픽을 제한하거나 조정 |
| **수정** | 필요하면 헤더나 요청 정보를 변경 |
| **기록** | 트래픽 로그를 남김 |

> 💡 차단하는 트래픽은 국가 단위의 트래픽을 많이 막는다.

**3. 서버 보호 (= Reverse Proxy)**
- 인라인 장비가 서버 앞에 있으면, 모든 요청을 서버가 직접 처리하지 않아도 된다
- 악성 요청 → 인라인 장비에서 처리(차단)
- 정상 요청 → 인라인 장비 → 서버가 처리
- 이 구조가 백엔드 서버를 보호할 때 요긴하게 사용된다 = **Reverse Proxy**

##### Inline 방식 동작 흐름

```
트래픽 통과
  → 헤더 분석
  → 페이로드 분석
  → 규칙 검사
  → 판단 결과
      ├ 허용 → 목적지 전달
      ├ 차단 → Drop → 로그 기록
      └ 필터링 → 제한/조정 후 전달
```

##### Inline 방식을 사용하는 이유

1. **실시간 보안 적용** : 인라인 장비 자체가 보안 정책을 적용하기 좋다
2. **트래픽 제어** : IP 기반 차단, 포트 기반 제어, 프로토콜 제어(HTTPS/DNS), Rate Limit, 접근 제어(`/admin`은 사내망에서만)
3. **서버 보호** : 악성 요청을 서버까지 도달하기 전에 차단

##### 대표적인 Inline 장비

- **방화벽** : 대표적인 인라인 장비. 보안을 위해 항상 모든 트래픽이 거치도록 설계
- **로드 밸런서** : 들어오는 요청을 서버들에게 분배해주는 장비

##### 인라인 장단점

| 장점 | 단점 |
|---|---|
| 강제성 : 모든 트래픽이 반드시 지나감 | 병목 현상 발생 가능 |
| 실시간 차단 | 단일 장애 지점이 될 수 있음 (SPOF) |
| 정책 적용 확실함 | 잘못된 정책이면 모든 서비스가 죽는다 |
| 로그 확보 | 처리량이 트래픽을 감당 못하면 장애 |
| 트래픽 제어 / 서버 보호 | |

> ⚠️ **SPOF (Single Point of Failure)**: 인라인 장비가 죽으면 전체 서비스가 죽는다. 규칙을 잘못 만들면 모든 서비스가 무조건 죽는다.

##### 장애 극복 방법

| 방법 | 설명 |
|---|---|
| **이중화** | Inline 장비를 2대 이상 구성 |
| **Failover** | 장애 발생 시 대기 장비로 자동 전환 |
| **Bypass Mode** | 장비 장애 시 트래픽을 우회 통과시킴 |
| **로드 밸런싱** | 여러 장비로 검사 부하 분산 |
| **정책 테스트** | 운영 반영 전 탐지 모드나 테스트 환경에서 검증 |
| **모니터링** | 지연, 차단 수, 에러율, CPU/메모리 감시 |
| **규칙 최적화** | 불필요하거나 과도한 규칙 제거 |

##### 헷갈리기 쉬운 것들

**Q1. Inline은 장비 이름인가?**
→ 아니다. 장비를 **배치하는 방식**이다. 같은 방화벽이라도 트래픽이 무조건 통과하면 Inline, 그렇지 않으면 Out of Path다.

**Q2. Inline이면 무조건 좋은가?**
→ 아니다. 강력한 제어가 가능한 만큼 병목과 장애 가능성이 높다. 도입 전 아래 질문을 먼저 해야 한다.
- "반드시 통과시켜야만 하는 트래픽인가?"
- "인라인 장비가 장애 나면, 전체 서비스가 죽나?"
- "인라인 장비의 처리량이 지나가는 트래픽을 감당할 수 있나?"

#### | Out of Path

**실제 트래픽 경로 밖에서 트래픽 복사본을 받아 분석하는 방식**

트래픽이 지나가는 길은 그대로 두고, 그 트래픽의 **복사본**을 옆으로 보내는 방식이다.

```
[Inline 방식]
Client → Firewall/WAF (Inline) → Switch → Server

[Out of Path 방식]
Client → Switch → Server
                ↓ (트래픽 복사본)
           IDS/Sensor (Out-of-Path) → SIEM/Alert
```

##### Out of Path를 사용하는 이유

**1. 서비스 트래픽에 영향을 주지 않는다**
- 분석 장치가 고장나더라도 실제 처리 흐름에는 영향을 주지 않는다

**2. 모니터링을 안정적으로 할 수 있다**
- 트래픽을 직접 건드리지 않고 복사본을 보기 때문에

**3. 탐지와 분석에 집중**
- 장비도 단일 책임 원칙처럼 역할이 분리된다
- 이상 트래픽 발견 → 로그 저장 → 알림 발생 → 관리자 확인 → 필요한 조치 수행

> 💡 **Inline vs Out of Path 한 줄 차이**
> - Inline = "이 트래픽 이상하네. 바로 차단"
> - Out of Path = "이 트래픽 이상해. 너가 봐야해. 로그도 적을게. 이메일도 보낼게"

##### Out of Path 특징

| 특징 | 설명 |
|---|---|
| **경로 외부 동작** | 실제 트래픽 경로 밖에서 동작 |
| **비간섭성** | 원본 트래픽에 직접 개입하지 않음 |
| **탐지 중심** | 차단보다는 탐지, 분석, 알림, 기록에 적합 |
| **서비스 안정성** | 분석 장비 장애가 서비스 전체 장애로 이어질 가능성이 낮음 |
| **확장성** | 분석 장비를 추가하거나 교체하기 비교적 쉬움 |
| **직접 차단 한계** | 원본 트래픽을 이미 지나가게 두기 때문에 즉시 차단은 제한적 |

##### Inline vs Out of Path 비교

| 구분 | Inline | Out of Path |
|---|---|---|
| 트래픽 처리 | 직접 통과, 제어 가능 | 복사본만 수신, 원본 불간섭 |
| 차단 가능 여부 | ✅ 실시간 차단 가능 | ❌ 즉시 차단 제한적 |
| 서비스 영향 | 장비 장애 시 서비스 전체 영향 | 장비 장애가 서비스에 영향 없음 |
| 주 목적 | 제어, 차단, 보호 | 탐지, 분석, 모니터링 |
| 대표 장비 | 방화벽, 로드 밸런서, WAF, IPS | IDS, SIEM, 네트워크 분석 도구 |
| 병목 위험 | 높음 | 낮음 |

#### | WireShark

**내 컴퓨터나 네트워크 인터페이스를 지나가는 패킷을 캡처하고 분석하는 오픈 소스 네트워크 프로토콜 분석 도구**

네트워크 통신은 눈에 보이지 않는다. 브라우저 접속 하나에도 내부적으로 상당히 많은 일이 일어난다.

```
브라우저 접속 흐름:
브라우저 접속 → DNS 조회 → TCP 연결 → TLS Handshake → HTTP 요청 → 서버 응답
```

WireShark는 이 보이지 않는 네트워크의 흐름을 눈으로 볼 수 있게 해주는 도구다.

##### WireShark를 사용하는 경우

- API 서버 안됨 → 서버 로그에 요청 로그가 안 찍힘 → 요청 자체가 서버에 도달하고 있는지 확인
- DNS에서 우리 서버에 IP를 제대로 서빙하나?
- HTTPS 연결이 실패했나?
- 포트의 문제인가? TLS 키 교환의 문제인가?

##### WireShark로 확인할 수 있는 것들

| 확인할 수 있는 것 | 예시 |
|---|---|
| **DNS 조회** | 도메인이 어떤 IP로 해석되는지 |
| **TCP 연결** | 3-way handshake가 성공하는지 |
| **재전송** | 네트워크 손실로 재전송이 발생하는지 |
| **TLS Handshake** | HTTPS 연결이 어디서 실패하는지 |
| **HTTP 요청** | 평문 HTTP 요청/응답 내용 |
| **포트 통신** | 특정 포트로 패킷이 실제로 나가는지 |
| **응답 지연** | 요청 후 응답까지 시간이 얼마나 걸리는지 |

##### WireShark에서 네트워크 계층 구조

| Wireshark 표시 | 네트워크 개념 |
|---|---|
| **Frame** | 캡처된 프레임 전체 |
| **Ethernet II** | L2, MAC 주소 |
| **Internet Protocol** | L3, IP 주소 |
| **TCP/UDP** | L4, 포트 번호 |
| **HTTP/DNS/TLS** | L7, 애플리케이션 프로토콜 |

##### WireShark 필터 사용법

모든 네트워크 패킷을 다 보려고 하면 너무 많아서 원하는 패킷을 보기 힘들다. 필터를 통해 내가 보고 싶은 패킷만 골라보는 기능.

**Capture Filter** : 패킷을 캡처하기 전에 적용. 어떤 패킷을 저장할지 결정.
```
host 8.8.8.8   # 특정 호스트만 캡처
port 22        # 특정 포트만 캡처
tcp            # TCP만 캡처
```

**Display Filter** : 캡처한 다음에 어떤 걸 보여줄지 결정. 필터를 나중에 수정해서 캡처된 것들을 다시 볼 수 있다.

| 보고 싶은 것 | Display Filter |
|---|---|
| DNS만 보기 | `dns` |
| HTTP만 보기 | `http` |
| TLS만 보기 | `tls` |
| TCP만 보기 | `tcp` |
| UDP만 보기 | `udp` |
| 특정 IP 보기 | `ip.addr == 8.8.8.8` |
| 특정 출발지 IP | `ip.src == 192.168.0.10` |
| 특정 목적지 IP | `ip.dst == 8.8.8.8` |
| 특정 포트 | `tcp.port == 443` |
| TCP 연결 시작 | `tcp.flags.syn == 1` |
| TCP 재전송 | `tcp.analysis.retransmission` |

##### 미니퀘스트 실습 — DNS 요청 및 응답 분석

```bash
# 터미널에서 DNS 요청
dig lulu-roh.xyz

# 응답 결과
Name: lulu-roh.xyz
[Time: 4.142000 milliseconds]
```

WireShark에서 캡처된 DNS 패킷 분석 결과:
- **Query**: `lulu-roh.xyz` Type A (Host Address), Class IN
- **Answer**: `lulu-roh.xyz` → `13.125.221.44`
- **TTL**: 408초 (6분 48초)
- **응답 시간**: 4.142ms

### 🗨️ 오늘의 회고

- **Inline과 Out of Path**는 장비가 아니라 배치 방식이라는 점이 핵심이다. 같은 방화벽도 어디에 놓이느냐에 따라 달라진다.
- Inline의 강력함은 곧 취약점이기도 하다. SPOF 문제를 반드시 이중화와 Failover로 보완해야 한다. "강력한 제어 = 높은 책임"이라는 게 인상 깊었다.
- Out of Path는 "탐지와 분석"에 집중한다. Inline이 경찰이라면 Out of Path는 CCTV에 가깝다. 둘은 함께 쓰일 때 진짜 효과를 낸다.
- WireShark로 실제 DNS 패킷을 캡처해서 보니, 눈에 보이지 않던 네트워크 흐름이 실감났다. `dig` 명령어 결과 4ms 응답시간이 WireShark에서도 그대로 확인됐다.
- 브라우저 접속 한 번에 DNS → TCP → TLS → HTTP가 순차적으로 일어난다는 걸 WireShark로 직접 보면 훨씬 와닿는다. 앞으로 네트워크 디버깅 시 WireShark를 적극 활용해야겠다.
- 이번 주 안에 6주차 과제와 위클리 챌린지까지 반드시 마무리하자.
